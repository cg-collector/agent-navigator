# 第303份研究报告：SCUBA — 面向Salesforce CRM工作流的Computer Use Agent基准测试

> **项目名称：** SCUBA: Salesforce Computer Use BenchmArk
> **论文来源：** arXiv:2509.26506（2025年9月30日）
> **研究机构：** Salesforce AI Research（Yutong Dai, Krithika Ramakrishnan, Jing Gu, Caiming Xiong, Ran Xu等）
> **开源地址：** https://github.com/SalesforceAIResearch/SCUBA
> **项目主页：** https://sfrcua.github.io/SCUBA/

---

## 一、项目背景与问题定义

### 1.1 痛点：Enterprise GUI Agent的"最后一公里"缺失

当前GUI Agent领域已有WebArena、OSWorld等知名benchmark，但它们存在一个共同的盲区——**企业级软件生态的自动化能力评估**。具体来说：

- **WebArena**聚焦于消费级网站的导航与操作（购物、社交、信息检索），不涉及复杂的企业SaaS平台。
- **OSWorld**覆盖Ubuntu桌面应用（LibreOffice、GIMP等），但任务偏向通用办公场景，缺乏CRM、ERP这类核心业务系统的深度覆盖。
- **WorkArena**基于ServiceNow平台，但仅限于IT服务管理场景，范围较窄。
- **CRMArena**虽构建在Salesforce上，但其任务设计面向**工具调用型Agent（Tool-Use Agent）**，本质上是信息检索任务，Agent不能真正写入或修改数据记录和系统配置。

这意味着：**一个在OSWorld上表现优异的Agent，到了真实的Salesforce CRM环境中可能完全失效。** 论文实验证实了这一点——在OSWorld上表现出色的开源模型驱动的Computer Use Agent，在SCUBA上的成功率不到5%。

### 1.2 SCUBA要解决的问题

SCUBA的核心目标是填补这一空白，提供一个**真实、可交互、多维度评估**的enterprise CRM benchmark，用于回答以下关键问题：
1. Computer Use Agent能否在复杂的SaaS平台上完成端到端的业务自动化？
2. 不同Agent设计范式（原生GUI Agent vs Agentic Framework）在企业场景下的优劣如何？
3. 开源模型与闭源模型在enterprise task上的差距有多大？
4. Demonstration（人类演示）能否有效提升Agent性能？

## 二、核心架构深度剖析

### 2.1 整体架构：三层设计

SCUBA采用**环境层—任务层—评估层**的三层架构：

```
┌─────────────────────────────────────────────┐
│              评估层 (Evaluation Layer)        │
│  ┌──────────┐ ┌──────────┐ ┌──────────────┐ │
│  │规则评估器 │ │里程碑评分│ │延迟/成本指标  │ │
│  └──────────┘ └──────────┘ └──────────────┘ │
├─────────────────────────────────────────────┤
│              任务层 (Task Layer)              │
│  300个任务实例 × 3种角色 × 5类能力             │
│  + 初始化脚本 + Knowledge Articles           │
│  + Human Demonstrations                     │
├─────────────────────────────────────────────┤
│              环境层 (Environment Layer)       │
│  Salesforce Sandbox Developer Org            │
│  (UI与生产环境一致，免费功能足够支撑所有任务)    │
│  支持Browser Use / Computer Use两种模式       │
└─────────────────────────────────────────────┘
```

### 2.2 任务体系设计

SCUBA包含**300个任务实例**，来源于对三类真实用户的深度访谈：

| 角色类型 | 任务数量 | 典型场景 |
|---------|---------|---------|
| 平台管理员 (Platform Admin) | ~100 | 用户权限管理、安全设置、系统配置 |
| 销售代表 (Sales Rep) | ~100 | 商机管理、报价生成、客户数据维护 |
| 服务代理 (Service Agent) | ~100 | Case处理、知识库查询、工单流转 |

每个任务测试五类核心能力：
1. **Enterprise Software UI导航** — 在复杂的Salesforce多层级菜单中定位目标页面
2. **数据操纵** — 创建、编辑、删除记录（Contact、Opportunity、Case等）
3. **工作流自动化** — 触发Approval Process、Flow、Workflow Rule
4. **信息检索** — 从Dashboard、Report、List View中提取特定数据
5. **故障排查** — 诊断并修复配置错误、权限问题等

### 2.3 两种Agent评估模式

SCUBA支持两种Agent接入方式，对应不同的技术路线：

**Browser Use模式：**
- Agent通过Playwright控制浏览器访问Salesforce Web界面
- 输入为DOM树/截图，输出为浏览器操作（click、type、scroll等）
- 代表Agent：Claude Browser Use、Browser Use框架等
- 优势：无需桌面环境，部署简单；劣势：依赖DOM结构稳定性

**Computer Use模式：**
- Agent通过截图+鼠标键盘操作控制完整桌面环境
- 基于OSWorld的Docker容器方案，在GCP VM上运行
- 输入为纯视觉截图，输出为坐标级点击/输入动作
- 代表Agent：CUA（Claude Computer Use）、OpenCUA等
- 优势：更通用、接近真实用户行为；劣势：需要GPU资源、延迟更高

### 2.4 异步评估流水线

SCUBA的一个关键工程创新是**异步并行评估架构**：

- 每个任务运行在独立的Salesforce sandbox org中，互不干扰
- 通过异步队列调度，支持大规模并行执行
- 全量300任务评估可在**90分钟内完成**（自托管模型）/ 受API限速约束（API模型）
- 评估流程：初始化环境 → Agent执行 → 规则验证 → 里程碑评分 → 结果汇总

### 2.5 细粒度评估指标

不同于传统benchmark仅给出binary success/failure，SCUBA引入**Process Reward（过程奖励）**机制：

- 每个任务分解为多个**里程碑步骤（milestones）**
- 评估器不仅判断最终结果，还追踪每一步的完成情况
- 失败时可精确定位到哪个环节出错（如"成功导航到页面但字段填写错误"）
- 同时报告**延迟指标**（任务完成时间）和**成本指标**（token消耗/API费用）

## 三、技术亮点

### 亮点1：从真实用户访谈中提炼任务——解决"任务真实性"难题

大多数benchmark的任务由研究者人工设计，容易脱离实际业务需求。SCUBA的创新在于**所有300个任务均来自对Salesforce三类真实用户的半结构化访谈**。

**原理说明：** 研究团队访谈了多位平台管理员、销售代表和服务代理，收集他们在日常工作中的高频、高价值操作场景，然后将这些自然语言描述转化为可自动化的、有明确验收标准的任务实例。这种方法确保了任务的**生态效度（ecological validity）**——即benchmark中的任务确实反映了真实世界的需求分布。

**意义：** 这意味着SCUBA不仅是学术工具，更是企业评估Agent ROI的直接参考。一个在SCUBA上得分高的Agent，更有可能在真实的Salesforce自动化项目中创造商业价值。

### 亮点2：Demonstration-Augmented Setting——用"示教"而非"微调"提升性能

SCUBA是首个系统性地提供**Human Demonstration（人类演示轨迹）**作为辅助信息的GUI Agent benchmark。

**原理说明：** 对于每个任务类别，研究团队录制了人类专家完成同类任务的完整操作轨迹（截图序列+动作序列）。在demonstration-augmented模式下，这些轨迹作为few-shot示例注入到Agent的prompt中，引导Agent模仿人类的操作策略。

**关键发现：**
- Demonstration将最佳Agent的成功率从39%提升至**50%**（+11个百分点）
- 同时降低**13%的任务耗时**和**16%的token成本**
- 原因：演示帮助Agent学会了"捷径"操作路径，减少了探索性试错

**深层启示：** 这验证了**In-Context Learning在GUI领域的有效性**——不需要微调模型权重，仅需提供高质量的操作示范即可显著改善性能。这对企业落地极为友好，因为收集演示数据远比训练定制模型成本低。

### 亮点3：领域迁移的性能悬崖（Domain Shift Cliff）——揭示当前Agent的本质局限

SCUBA最令人震撼的发现之一是**跨domain的性能断崖式下跌**。

**数据：** 在OSWorld上表现优秀的开源模型Agent（如基于Qwen2.5-VL的方案），迁移到SCUBA后成功率从~15%-20%暴跌至**<5%**。即便是闭源模型的Agent（如Claude CUA），zero-shot成功率最高也仅为**39%**。

**原因分析：**
1. **UI复杂度跃升**：Salesforce的Lightning UI包含动态组件、嵌套列表、自定义字段，远超OSWorld中标准桌面应用的复杂度
2. **业务逻辑深度**：许多任务要求理解隐含的业务规则（如"审批链路"、"共享规则"），纯视觉模型难以推断
3. **长程依赖**：平均任务步数远超OSWorld，Agent需要在数十步操作中维持目标和上下文
4. **状态空间爆炸**：Salesforce的配置组合空间极大，相似页面的细微差异可能导致完全不同的正确操作

这个发现对整个GUI Agent社区是一个重要的"清醒剂"——**当前的Agent能力离真正的enterprise-ready还有很大距离**。

## 四、基准测试表现

### 4.1 Zero-Shot设置下各Agent的表现

| Agent类型 | 底座模型 | 成功率(约) | 备注 |
|----------|---------|-----------|------|
| Computer Use (Agentic Framework) | 开源VLM | <5% | OSWorld强者在此崩盘 |
| Computer Use (Agentic Framework) | Claude Sonnet | ~30-35% | 闭源模型明显领先 |
| Computer Use (Native GUI) | Claude | ~39% | zero-shot最佳 |
| Browser Use | GPT-4o/Claude | ~25-35% | 浏览器模式略逊于桌面模式 |

### 4.2 Demonstration-Augmented设置下的提升

| 指标 | Zero-Shot最佳 | Demo-Augmented最佳 | 提升幅度 |
|------|-------------|-------------------|---------|
| 任务成功率 | ~39% | **~50%** | +11pp (+28%) |
| 平均耗时 | 基准 | 降低13% | 效率提升 |
| Token成本 | 基准 | 降低16% | 成本优化 |

### 4.3 关键观察

- **闭源 vs 开源差距巨大**：在enterprise场景下，闭源模型的优势比在通用benchmark中更加明显
- **Browser Use vs Computer Use**：两者各有千秋，Browser Use部署更简单但受DOM变化影响大；Computer Use更通用但对算力要求高
- **不同范式差异显著**：论文识别了三种Agent设计范式（基础VLM+Toolbox、原生GUI Agent、Agentic Framework），其中Agentic Framework整体表现最优

### 优劣势总结

**优势：**
1. **最高的企业真实性**：唯一同时满足"真实平台+真实用户访谈+数据写入+多角色"的benchmark
2. **里程碑评估机制**：失败不再是个黑盒，可以诊断具体哪个环节出问题
3. **Demonstration支持**：开创了GUI Agent benchmark提供教学数据的先河
4. **多维指标**：不只看成功率，还关注延迟和成本，更贴近部署现实

**劣势：**
1. **平台锁定**：仅针对Salesforce，泛化性有限（但这是深度优先策略的必然取舍）
2. **准入门槛高**：需要Salesforce Developer Org账号、可能需要GCP资源（Computer Use模式）
3. **任务规模相对较小**：300个任务相比OSWorld的689个偏少
4. **尚未见广泛采用**：作为2025年9月新发布的benchmark，社区复现和研究尚在早期阶段

## 六、对我们的实际启发

### 启发1：Enterprise Agent的评估必须走向"业务对齐"

如果我们未来开发面向企业客户的Agent产品，不应只拿OSWorld/WebArena说事。SCUBA给我们的思路是：**针对目标行业设计专属的评估集**，任务来自真实用户访谈而非研究者想象。对于我们的场景，可以考虑：
- 构建飞书/钉钉环境的mini-benchmark
- 从学长日常使用OpenClaw的工作流中提炼典型任务
- 重点关注"写操作"能力的评估，因为读操作大部分Agent已经做得不错了

### 启发2：Demonstration Pipeline是低成本的Agent增强手段

SCUBA证明了**提供操作演示可以同时提升成功率、降低延迟和成本**。这给我们一个可落地的想法：
- 为楠楠常用的操作流程录制"标准操作轨迹"（如飞书文档批量处理、股票分析报告生成等）
- 在遇到类似任务时，自动检索并注入相关演示作为few-shot参考
- 这比微调或换更强的模型都更经济高效

### 启发3：细粒度里程碑评估值得借鉴

SCUBA的Process Reward机制告诉我们：**知道在哪里失败比知道是否失败更重要**。在我们的Agent开发中可以引入类似的中间检查点：
- 将复杂任务拆解为可验证的子步骤
- 每步完成后进行轻量级校验，及时发现问题
- 失败时回退到最近的成功节点重试，而非从头开始

### 启发4：领域迁移是当前Agent的最大软肋

<5%的开源模型成功率提醒我们：**在通用benchmark上刷分≠在实际场景中好用**。我们在评估和选择Agent方案时，应该：
- 优先在目标领域的数据上进行测试，而不是迷信公开榜单
- 对开源方案保持审慎乐观，enterprise场景下闭源模型的优势可能被低估
- 关注长程任务（multi-step workflow）的能力，这是当前最大的短板
