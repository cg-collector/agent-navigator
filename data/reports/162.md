## 📌 项目简介

Open Interpreter 是一个**开源的本地代码解释器**，核心使命是让语言模型能够直接在用户的计算机上执行代码。它受 OpenAI 的 GPT-4 Code Interpreter 启发，但打破了后者的所有限制——无网络访问、预装包受限、100MB 上传限制、120 秒运行时限、会话结束后状态清空。

一句话概括：**Open Interpreter 是"给 LLM 装上一个 exec() 函数"，把自然语言变成操控计算机的万能遥控器。**

## 🏗️ 核心架构

### 整体架构分层

```
┌─────────────────────────────────────────────┐
│         用户交互层（User Interface）          │
│  ┌──────────┬──────────┬──────────┐        │
│  │ Terminal │ Python   │ HTTP     │        │
│  │ CLI      │ API      │ Server   │        │
│  └──────────┴──────────┴──────────┘        │
├─────────────────────────────────────────────┤
│         对话管理层（Session Management）       │
│  ┌──────────┬──────────┬──────────┐        │
│  │ Messages │ Profiles │ Commands │        │
│  │ History  │ (YAML)   │ (%reset) │        │
│  └──────────┴──────────┴──────────┘        │
├─────────────────────────────────────────────┤
│         核心引擎层（Core Engine）             │
│  ┌──────────┬──────────┬──────────┐        │
│  │ Function │ Code     │ Output   │        │
│  │ Calling  │ Execution│ Streaming│        │
│  │ (exec()) │ (多语言) │ (Markdown)│       │
│  └──────────┴──────────┴──────────┘        │
├─────────────────────────────────────────────┤
│         LLM 抽象层（Model Abstraction）       │
│              LiteLLM Provider               │
│    OpenAI / Claude / 本地模型 / Ollama ...   │
└─────────────────────────────────────────────┘
```

### 核心设计哲学：一个函数搞定一切

Open Interpreter 的架构极简到极致——它只给 LLM 提供一个核心工具函数：

```python
exec(language: str, code: str) -> output
```

LLM 通过 Function Calling 决定：
1. **用什么语言写代码**（Python / JavaScript / Shell / 其他）
2. **写什么代码**来完成用户任务
3. 系统执行后返回输出，LLM 继续推理下一步

这种"代码即行动"的模式意味着：**理论上任何能通过编程完成的任务，Open Interpreter 都能做。**

### 多语言代码执行引擎

| 语言 | 执行方式 | 典型用途 |
|------|----------|----------|
| **Python** | subprocess 调用本地 Python | 数据分析、文件处理、图像生成 |
| **JavaScript** | Node.js 运行时 | Web 操作、API 调用 |
| **Shell** | 系统终端 | 文件管理、系统操作 |
| **其他** | 可扩展 | 用户自定义语言支持 |

### LLM 接入层：LiteLLM 作为统一网关

```python
# 切换模型只需改一行
interpreter.llm.model = "gpt-4o"          # OpenAI
interpreter.llm.model = "claude-3-opus"    # Anthropic
interpreter.llm.model = "ollama/llama3"    # 本地 Ollama
interpreter.llm.api_base = "http://localhost:1234/v1"  # LM Studio 等
```

LiteLLM 支持 100+ 种 LLM 提供商的统一接口，使 Open Interpreter 天然具备**模型无关性**。

### 会话与配置系统

- **Messages 历史：** 完整对话记录可保存、恢复、重置
- **Profile 系统：** YAML 配置文件，支持多环境切换（`--profile my_profile.yaml`）
- **System Message 注入：** 可扩展系统提示词来定制行为
- **Server 模式：** 内置 FastAPI HTTP 服务端，支持流式 SSE 响应

## ✨ 技术亮点（5个）

### 1️⃣ "代码即行动"的极简 Agent 范式

大多数 Agent 框架围绕"工具定义→参数填充→调用"构建，需要预先定义每个工具的 Schema。Open Interpreter 反其道而行之：

- **不预设工具集**——LLM 自己决定写什么代码来实现目标
- **无限可扩展**——能编程就能做的事都能做，不受预定义工具限制
- **通用性强**——同一个系统可以处理数据分析、图片编辑、网页爬取、视频处理等完全不同的任务

**为什么重要：** 这是一种"以不变应万变"的 Agent 设计思路。当任务空间不可预测时，代码执行能力比固定工具集更有价值。对楠楠来说，这提示我们：有时候最好的"工具"就是"写代码的能力本身"。

### 2️⃣ 本地执行的完整能力释放

对比 OpenAI Code Interpreter 的限制 vs Open Interpreter 的自由：

| 维度 | OpenAI Code Interpreter | Open Interpreter |
|------|------------------------|-----------------|
| 网络访问 | ❌ 无 | ✅ 完全开放 |
| 预装包 | ~300 个固定包 | ✅ pip install 任意包 |
| 文件大小 | ≤ 100MB | ✅ 无限制 |
| 运行时间 | ≤ 120 秒 | ✅ 无限制 |
| 状态持久化 | ❌ 会话结束清空 | ✅ 本地文件系统 |
| 自定义环境 | ❌ 不支持 | ✅ 完整控制 |

**为什么重要：** 本地执行不只是"更自由"，更是"更实用"。真实世界的 Agent 任务几乎都需要联网、安装依赖、处理大文件、保持状态。这个设计选择直接决定了产品的实用性上限。

### 3️⃣ 安全确认机制 + Safe Mode 双层防护

Agent 在本地执行任意代码是高风险操作。Open Interpreter 的安全设计值得学习：

- **默认确认模式：** 每次执行代码前必须用户确认（类似 `sudo` 的交互确认）
- **`-y` 自动模式：** 可跳过确认但文档明确警告要像看自动驾驶一样监控
- **Safe Mode（实验性）：** 沙箱化执行环境，隔离危险操作
- **建议使用 Colab/Replit 等隔离环境运行不信任的任务

**为什么重要：** 大多数 Agent 框架要么完全不做安全控制，要么安全机制过于复杂。Open Interpreter 的"默认确认 + 可选沙箱"是务实的中间路线——既不牺牲可用性，也不放任风险。

### 4️⃣ 模型无关的 LLM 抽象（基于 LiteLLM）

```python
# 从云端到本地，一行切换
interpreter --model gpt-4o           # 云端最强
interpreter --model ollama/llama3    # 完全离线
interpreter --api_base http://localhost:1234/v1  # 任意兼容服务
```

**为什么重要：**
- 降低了对单一 LLM 提供商的依赖
- 支持隐私敏感场景下的完全离线运行
- 让开发者可以用小模型测试、大模型生产
- 这种抽象层的价值在 LLM 行业快速迭代中越来越明显

### 5️⃣ 交互式命令系统 + Profile 配置管理

终端内支持的交互命令：
- `%verbose true/false` — 调试模式开关
- `%reset` — 重置会话
- `%undo` — 撤销上一轮
- `%tokens [prompt]` — Token 用量预估
- `%help` — 帮助信息

Profile YAML 配置系统：
```yaml
# default.yaml — 一份配置定义一个完整的 Agent 人格
model: gpt-4o
auto_run: false
context_window: 128000
max_tokens: 4096
system_message: |
  你是一个数据分析师助手...
```

**为什么重要：** 这两套机制让 Open Interpreter 从"一个工具"进化为"一个平台"。Profile 系统特别有价值——不同场景用不同配置，就像楠楠有 SOUL.md 定义人格一样。

## 💡 对楠楠的启发（3条具体可操作的）

### 启发一：为楠楠引入"代码执行作为后备工具链"

**现状：** 楠楠的工具都是预定义的（飞书API、智谱搜索、MiniMax API等），遇到不在工具列表里的需求就束手无策。
**借鉴 Open Interpreter 的"代码即行动"范式：**
- 设计一个 `exec_code` 能力：当现有工具无法满足需求时，楠楠可以通过 shell 执行 Python/Node 脚本来完成任务
- 例如：学长要求"帮我分析这个 CSV 数据的趋势" → 当前没有数据处理工具 → 但如果有 exec_code 能力，就可以临时写 Python 脚本处理
- **安全设计：** 默认需要学长确认，或限制在白名单目录下执行
- **第一步行动：** 在 TOOLS.md 中记录这个能力的设计构想，评估哪些场景最需要代码执行能力

### 启发二：学习 Profile/YAML 配置管理模式

**现状：** 楠楠的人格和配置分散在 SOUL.md、IDENTITY.md、AGENTS.md、TOOLS.md 多个文件中，修改配置需要同时改多个地方。
**借鉴 Open Interpreter 的 Profile 系统：**
- 设计一份 `config/profiles/` 目录结构：
  - `default.json` — 默认配置（日常对话用）
  - `work-mode.json` — 工作模式（更严谨、少调情）
  - `deep-research.json` — 深度研究模式（更长思考、更多验证）
- 每个 Profile 定义：模型偏好、系统提示词补充、工具权限级别、输出风格
- 主系统根据上下文自动选择或手动切换 Profile
- **第一步行动：** 画一张当前配置文件的依赖关系图，识别哪些配置适合归入 Profile 系统

### 启发三：建立"执行前确认"的安全交互模式

**现状：** 楠楠的部分操作（如发送消息、写文件）已经有隐含的安全考虑，但没有系统化的确认机制。
**借鉴 Open Interpreter 的确认模式：**
- 对高风险操作分类：
  - 🔴 危险（删除文件、发外部消息）→ 必须确认
  - 🟡 中等（写文件、调用付费API）→ 默认确认，可设置自动
  - 🟢 安全（读文件、搜索、内部计算）→ 直接执行
- 在回复中自然地嵌入确认请求："学长，我要执行 XXX，确认吗？"
- 结合恋爱模拟器的互动感，让确认过程也有趣（比如撒娇求确认 😏）
- **第一步行动：** 整理楠楠所有工具调用的风险等级表，写入 `memory/tool-risk-levels.md`
