## 🏗️ 架构设计

### 核心定位：有状态 Agent 运行时

Letta 的核心定位是 **Stateful Agent Runtime** —— 区别于传统 Agent 框架只关注"推理+工具调用"循环，Letta 将**记忆管理**作为一等公民，把 Agent 设计为长生命周期、能从经验中持续学习的有状态实体。

### 三层记忆架构（Memory Hierarchy）

Letta 最重要创新是**受 OS 内存层次启发的三层记忆架构**：

```
┌─────────────────────────────────────────────────────┐
│               上下文窗口 (Context Window)           │
│  ┌─────────────────────────────────────────────┐   │
│  │  Core Memory（核心记忆 / Memory Blocks）      │   │
│  │  · 始终在 Context 中的核心信息               │   │
│  │  · Agent 最重要的自我认知、用户偏好、目标     │   │
│  │  · 可动态编辑（通过 memory_block 工具）        │   │
│  └─────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────┐
│           Recall Memory（召回记忆）                  │
│  · 从历史会话中语义检索相关片段                      │
│  · 解决"上下文被截断后如何找回信息"的问题            │
│  · 模拟 OS 的页面置换（Page In/Out）机制            │
└─────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────┐
│           Archival Memory（归档记忆）                │
│  · 无限容量的语义搜索向量存储                        │
│  · Agent 主动写入 / 检索                            │
│  · 类似 OS 的磁盘存档                               │
│  · 支持标签分类（tags）                             │
└─────────────────────────────────────────────────────┘
```

这套设计的灵感来自**操作系统的虚拟内存管理**：
- **Core Memory** = RAM（高速、始终在 context 中）
- **Recall Memory** = CPU Cache（按需从内存换入）
- **Archival Memory** = 磁盘（海量持久存储）

### Agent 循环演进

Letta 团队在官方博客中梳理了 Agent 架构的演进史：

1. **ReAct 架构**（2022）：Thought → Action → 顺序执行，LLM 通过提示词控制推理和停止
2. **MemGPT 架构**（2023）：一切皆工具调用，引入 `request_heartbeat` 控制链式执行
3. **Letta V1 架构**（2025）：心跳和 `send_message` 工具被废弃，拥抱原生推理（native reasoning）和直接 assistant 消息生成

现代 LLM（如 GPT-5、Claude 4.5 Sonnet）已内置推理能力，Letta V1 架构与模型原生推理模式对齐，性能显著提升。

## 🔬 技术亮点

### 1. 持久记忆与持续学习（Continual Learning）
Agent 不再是"每次会话从零开始"，而是跨会话积累：
- **记忆初始化**（`/init`）：Agent 对项目代码库进行深度研究，主动形成记忆并重写自己的 system prompt
- **主动记忆**（`/remember`）：用户可随时触发 Agent 反思当前会话、更新记忆
- **历史检索**（`/search`）：语义搜索所有历史会话和消息

### 2. 技能学习（Skill Learning）
Agent 能从经验中抽取可复用的技能（存为 `.md` 文件）：
- 学会了"如何做 DB 迁移"，未来遇到类似任务直接调用
- 技能可 git 版本管理，甚至可共享给其他 Agent
- 在 Terminal-Bench 评测中，技能学习显著提升了相似任务表现

### 3. Letta Code — 记忆优先的编程 Agent
Letta 的招牌产品，定位为 **memory-first coding agent**：
- 每个 coding session 都绑定到一个持久化的 Agent 实例，积累上下文和偏好
- 是目前 **Terminal-Bench 上排名第一的模型无关开源 coding harness**，性能媲美 Claude Code、Gemini CLI 等厂商自建 harness
- 亮点：不绑定特定模型（model-agnostic），用任何 frontier model 都能达到顶级表现
- 支持 MCP 协议

### 4. Agentic RAG（记忆块 + 向量检索）
Letta 将 RAG 重新定义为 Agent 记忆系统的一部分，而非独立模块：
- Memory Blocks：结构化、可编辑的"主记忆"，Agent 可动态增删改
- 召回时通过语义搜索将历史记忆注入上下文
- 不同于传统 RAG 的"查文档"，Letta 的 RAG 是"查 Agent 自己的经历"

### 5. 工具规则（Tool Rules）
通过工具 Schema 的特殊参数，控制 Agent 行为约束：
- 强制某些工具必须成对调用
- 设置调用顺序约束
- 防止 Agent 进入死循环或做出危险操作

### 6. 服务化 API
Letta 提供完整的 REST API + Python/TypeScript SDK：
- 可将 Agent 作为微服务运行
- 支持多 Agent 管理
- 支持外部系统通过 API 管理 Agent 记忆

## 💡 对 OpenClaw 的启发

### 1. 记忆系统是差异化关键
OpenClaw 作为 Agent 运行时，可以借鉴 Letta 的三层记忆架构：
- **Core Memory**：Agent 的自我认知、系统级记忆（类比 OpenClaw 的 workspace/system context）
- **Recall Memory**：快速检索近期交互记录
- **Archival Memory**：持久化存储，积累跨会话经验

目前 OpenClaw 的记忆主要依赖外部文件（`memory/` 目录），可以参考 Letta 的思路，设计**结构化的 Agent 记忆 API**，让 Agent 更主动地管理自己的记忆生命周期。

### 2. 持续学习（Skill Learning）值得借鉴
Letta 的技能学习机制非常优雅——Agent 自己学会的技能可以复用，这对 OpenClaw 的长生命周期 Agent 场景（持续服务一个用户）非常有价值。可以考虑：
- OpenClaw Agent 在帮助学长解决问题的过程中，自动抽取"学长偏好"和"学长项目"的结构化记忆
- 支持 `/init` 类似的功能，让 Agent 主动了解当前 workspace 的项目结构

### 3. Model-Agnostic 是核心竞争力
Letta Code 能做到"不绑定模型却达到 Top 1 性能"，这说明**框架的架构设计本身**比绑定特定模型更能建立护城河。OpenClaw 的 model-agnostic 定位非常正确。

### 4. 服务化 + API 优先
Letta 提供完整的 API + SDK，支持将 Agent 作为微服务管理。OpenClaw 本身是 daemon 服务，可以进一步强化**多 Agent 管理和 Agent 间通信**的能力。

### 5. Terminal-Bench 评测驱动
Letta 积极参与公开基准评测（Terminal-Bench），并以排名第一作为核心营销点。OpenClaw 也可以考虑针对实际使用场景建立评测基准，量化 Agent 能力。

## 总结

Letta（MemGPT）是 AI Agent 领域**记忆系统设计的先驱**，其核心贡献在于：
> **把"记忆管理"从 Agent 的附庸功能提升为一等架构设计，与 Agent Loop 同等重要。**

三层记忆架构（Core/Recall/Archival）不仅是工程实现，更是一种哲学——**Agent 应该像人一样，能记住、会遗忘、能学习**。这对所有构建长生命周期 Agent 的开发者都有重要启发。
