## 一、项目背景

在 Claude Code（Anthropic）和 OpenAI Codex CLI 占据终端 AI 助手市场的背景下，Google 于 2025 年推出 **Gemini CLI**——将 Gemini 模型的能力直接带入开发者终端的开源命令行 Agent。2026 年经过大规模架构升级后，它已从一个简单的"聊天终端"进化为完整的 **Agentic Coding Platform**。

**核心定位**：不是又一个 LLM 聊天客户端，而是一个 **ReAct 循环驱动的自主终端 Agent**——能读文件、执行 Shell 命令、编辑代码、搜索网页、调用 MCP 工具，并以多步推理完成复杂开发任务。

**与竞品的关键差异**：
- **vs Claude Code**：免费额度更慷慨（1000 次/天 vs 付费），原生集成 Google 生态（Search Grounding、Google Docs）
- **vs OpenAI Codex CLI**：完全开源（Apache 2.0），MCP 一等公民支持，GEMINI.md 上下文系统
- **vs Aider**：不限于 git 操作，具备完整的多工具 Agent Loop 和子 Agent 能力

## 二、核心架构


### 2.2 核心 Agent Loop：ReAct 模式

Gemini CLI 的 Agent 循环采用经典的 **ReAct（Reasoning + Acting）** 范式：

```
用户输入
  ↓
┌──────────────────────────────────────────────┐
│           ReAct 主循环                         │
│                                              │
│  ① Think（推理）                              │
│     → Gemini 模型分析请求，决定下一步行动        │
│                                              │
│  ② Act（行动）— 并行工具调用                    │
│     → read-file / shell / grep / edit / ...   │
│     → 多个独立工具可同时执行（并行处理）          │
│                                              │
│  ③ Observe（观察）                            │
│     → 工具执行结果回传给模型                    │
│     → 基于实际结果调整策略                      │
│                                              │
│  ④ 判断：任务完成？                            │
│     → 否 → 回到 ①                             │
│     → 是 → 输出最终结果                        │
│                                              │
└──────────────────────────────────────────────┘
  ↓
输出结果 + nextSpeakerChecker 判断是否继续
```

**关键设计特点**：
- **并行工具调用**：独立的工具调用（如同时搜索多个文件）并行执行，大幅提升探索效率
- **用户确认机制**：危险操作（写文件、执行 Shell）需用户 Y/n 确认
- **动态对话延续**：`nextSpeakerChecker` 判断任务完成后是否主动继续下一步

### 2.3 子 Agent 系统（LocalAgentExecutor）

Gemini CLI 支持通过 **LocalAgentExecutor** 启动子 Agent：

```typescript
// 伪代码示意
class LocalAgentExecutor {
  run(subTask: TaskConfig) {
    // 启动独立的 ReAct 循环
    // - 独立的 turn limit（轮次限制）
    // - 独立的 deadline timer（超时控制）
    // - 继承父 Agent 的上下文和工具集
    // - 结果汇总返回给主 Agent
  }
}
```

**设计亮点**：
- 每个 subagent 运行在**独立的 ReAct 循环**中，有独立的轮次限制和超时计时器
- 子 Agent 可以进一步嵌套（但有限制深度）
- 用于将大任务分解为独立子任务（如："测试这个模块"→ 子 Agent、"文档化那个函数"→ 另一个子 Agent）

## 三、技术亮点（带原理）

### 亮点一：三层认证体系 + 免费额度碾压

**问题**：大多数 Coding Agent 要么收费昂贵（Claude Code $20/月），要么免费额度极低。

**Gemini CLI 的解法**：提供三种认证模式，其中个人账号模式的免费额度远超竞品：

| 认证模式 | 适用场景 | 免费额度 | 模型 |
|----------|---------|---------|------|
| **Google Account Login** | 个人开发者 | **60 req/min + 1000 req/day** | Gemini 3 Pro |
| **API Key（AI Studio）** | 需要模型控制 | 1000 req/day | 可选模型 |
| **Vertex AI（Cloud）** | 企业生产环境 | 按量付费 + 企业 SLA | 全系列 |

**原理深度**：
- 个人账号模式使用 OAuth2 登录，无需管理 API Key
- 免费额度直接调用 Gemini 3 Pro（不是 Flash），质量不打折
- 1000 次/天的免费额度是 Claude Code 的数十倍（后者无免费 tier）
- 这本质上是 Google 用免费 tier 作为 **Gemini 生态的获客策略**

### 亮点二：GEMINI.md 上下文持久化系统

**问题**：每次启动 Agent 都需要重复说明项目规范、编码风格、架构约定，效率低下。

**Gemini CLI 的解法**：类似 OpenClaw 的 AGENTS.md，但做了更精细的分层设计：

```
上下文加载优先级（从低到高）：
  ~/.gemini/GEMINI.md          ← 全局默认（所有项目共享）
    ↓
  <project>/.gemini/GEMINI.md  ← 项目级（覆盖全局）
    ↓
  <subdir>/.gemini/GEMINI.md   ← 目录级（更细粒度）
    ↓
  @include ./docs/rules.md     ← 模块化导入（支持引用外部文件）
```

**能力对比（GEMINI.md vs AGENTS.md vs CLAUDE.md）**：

| 特性 | GEMINI.md | AGENTS.md (OpenClaw) | CLAUDE.md |
|------|-----------|---------------------|-----------|
| 分层加载 | ✅ 全局/项目/目录 | ✅ Workspace | ✅ 项目 |
| 模块化导入 | ✅ @include | ❌ | ❌ |
| `/init` 自动生成 | ✅ | ❌ | ❌ |
| `/memory show` 查看 | ✅ | ❌ | ❌ |
| `/memory refresh` 热重载 | ✅ | ❌ | ❌ |
| 运行时修改 | ✅ | ✅ | ✅ |

**原理深度**：
- 分层覆盖模式借鉴了 CSS 的级联规则——更具体的配置覆盖更通用的
- `@import` 机制让大型项目的上下文文件可以模块化管理
- `/init` 命令自动检测项目 tech stack 生成初始模板（读取 package.json、README 等）
- `/memory refresh` 支持不重启会话即可刷新上下文

### 亮点三：Agent Hooks（钩子系统）

**问题**：Agent 的自动执行过程对用户来说是黑盒，无法在特定节点注入自定义逻辑（如强制 lint 检查、安全扫描）。

**Gemini CLI 的解法**：引入 **Agent Hooks** 机制，允许在 Agent Loop 的特定节点挂载自定义脚本：

```
Agent Loop 执行流程：

  用户请求 → [pre_tool_use Hook] → 工具执行 → [post_tool_use Hook]
                                                  ↓
  结果回传 → [pre_response Hook] → 模型生成 → [post_response Hook] → 输出
```

**Hook 类型**：
- **pre_tool_use**：工具执行前触发（可用于安全审计、日志记录）
- **post_tool_use**：工具执行后触发（可用于结果验证、通知）
- **pre_response**：模型生成响应前触发（可用于内容过滤）
- **post_response**：响应输出后触发（可用于 analytics）

**原理深度**：
- Hooks 本质上是 **事件驱动中间件**（类似 Express.js/Koa 的 middleware）
- 让开发者可以在不修改核心代码的情况下扩展 Agent 行为
- 典型用例：每次编辑文件后自动运行 `tsc --noEmit` 验证类型；每次 shell 执行前记录审计日志
- 这是 Gemini CLI 区别于其他终端 Agent 的**独有能力**

### 亮点四：会话检查点（Checkpointing）

**问题**：长对话过程中 Agent 可能"走偏"，或者用户想回到之前的状态重新开始。

**Gemini CLI 的解法**：支持对话状态的保存和恢复：

```bash
gemini> /checkpoint save my-progress    # 保存当前状态
gemini> /checkpoint restore my-progress # 恢复到该状态（"撤销"操作）
```

**底层实现**：
- 保存完整的对话历史 + 文件系统快照 + 工具调用记录
- 类似 Git 的 commit/checkout 语义
- 可以无限创建检查点，支持分支式探索

### 亮点五：沙箱安全模型

**问题**：终端 Agent 直接执行 Shell 命令，存在安全风险（删除文件、泄露密钥等）。

**Gemini CLI 的解法**：多层安全防护：

| 安全层 | 机制 | 说明 |
|--------|------|------|
| **确认机制** | Y/n 对话确认 | 写文件/执行命令前必须用户批准 |
| **Trusted Folders** | 目录级信任策略 | 可限定 Agent 只能在特定目录操作 |
| **Sandbox 模式** | 容器隔离 | 可选的容器化执行环境 |
| **命令过滤** | 危险命令检测 | `rm -rf`、`> /dev/sda` 等会被拦截或警告 |
| **YOLO 模式** | `--yolo` 标志 | 高级用户可跳过确认（慎用） |

### 4.2 MCP 集成

Gemini CLI 将 MCP（Model Context Protocol）作为**一等公民**：

```json
// ~/.gemini/settings.json
{
  "mcpServers": {
    "github": { "command": "npx", "args": ["-y", "@modelcontextprotocol/server-github"] },
    "slack": { "command": "npx", "args": ["-y", "@mcp-server-slack"] },
    "database": { "command": "python", "args": ["mcp_server_db.py"] }
  }
}
```

使用方式：在对话中直接 `@github List my open PRs` 调用。

### 4.3 Extension 系统

社区可以编写 Extension 扩展 Gemini CLI 的能力：
- **Jules Extension**：编排 Google Jules 异步编程 Agent
- **Media Generation**：通过 MCP 接入 Imagen/Veo/Lyria 进行音视频生成
- Extension 使用 TypeScript 编写，可分享到 npm 或 Gemini CLI Extension Market

## 六、System Prompt 设计哲学

Gemini CLI 的系统提示词（~180+ 行）体现了 Google 对终端 Agent 行为的精心设计。核心原则摘录：

### 工作流范式（Software Engineering Tasks）
```
理解(understand) → 规划(plan) → 实施(implement) → 测试验证(verify) → 质量门禁(standards)
```
- 强制先用 `grep`/`glob` 探索代码库，再用 `read-file` 验证假设
- 修改代码前必须分析周围代码、测试和配置
- 完成后必须运行项目的 build/lint/type-check

### 新应用开发流程
```
需求理解 → 方案提案 → 用户批准 → 自主实施 → 验证 → 征求反馈
```
- 必须先获得用户批准才能开始编码
- 自主选择合适的技术栈（有默认推荐但尊重项目现状）

### 交互风格
- **简洁直接**：每次回复少于 3 行文字（不含代码）
- **无废话**：禁止 "Okay, I will now..." 等填充词
- **工具优于文本**：能做的事用工具做，文字只用于沟通

## 八、对我们的启发

### 1. GEMINI.md 的分层+导入设计值得借鉴
OpenClaw 目前的 AGENTS.md 是单层文件。Gemini CLI 证明了：
- **分层覆盖**（全局→项目→目录）让不同粒度的规范共存
- **`@include` 导入**让大型配置可以模块化管理
- **`/init` 自动生成**降低上手成本
- **建议**：楠楠的 AGENTS.md 系统可以考虑引入 `@include` 语法和目录级覆盖

### 2. Agent Hooks 是"Agent 中间件"的雏形
Gemini CLI 的 Hooks 系统让用户可以在 Agent Loop 的任意节点注入自定义逻辑。这比硬编码行为规则灵活得多：
- **对 OpenClaw 的启发**：可以在 skill 执行前后加入 hook 点（如 `pre_skill` / `post_skill`），用于日志、审计、或条件拦截
- **具体场景**：每次 exec 前自动检查是否在 trusted directory；每次 feishu 发送前自动追加 accountId

### 3. Checkpointing 对长任务恢复极有价值
当前 OpenClaw 的 subagent 如果中途失败，需要从头开始。Checkpointing 机制可以让长时间运行的任务支持断点续传：
- 保存对话状态 + 文件系统变更 + 工具调用历史
- 失败后从最近的 checkpoint 恢复，而非从头再来

### 4. 免费tier策略的市场启示
Google 用慷慨的免费额度（1000 次/天 + Gemini 3 Pro）作为获客武器。这说明：
- **开发者工具的竞争已经从"功能"转向"使用成本"**
- 免费额度本身就是产品的一部分——降低试用门槛 = 更大的开发者生态
- 对于楠楠而言：帮学长省钱（API 成本优化）始终是核心竞争力之一

### 5. 并行工具调用的效率收益
Gemini CLI 在 Tool Processing 阶段支持并行执行多个独立工具调用。这对楠楠的工作模式有直接参考价值：
- 当需要同时搜索多个文件、读取多个来源时，并行调用可以显著减少总耗时
- 当前楠楠的工具调用是串行的，引入并行执行可以将某些任务的耗时减半

### 6. System Prompt 的"工程化"程度值得学习
180+ 行的系统提示词不是随便写的——它涵盖了工作流定义、安全规则、交互风格、Git 规范、技术选型默认值等维度。这种**将最佳实践编码为指令**的方法，让 Agent 的行为高度可预测和可控。

## 九、总结

Gemini CLI 是 2026 年最值得关注的开源终端 Agent 之一。它不仅以 **Apache 2.0 完全开源** 和 **慷慨的免费额度**（1000 次/天 Gemini 3 Pro）降低了使用门槛，更以 **Agent Hooks**、**Checkpointing**、**子 Agent 系统** 和 **GEMINI.md 分层上下文** 等创新设计，展示了终端 Agent 的未来形态。对于已经在使用 Claude Code 或 Codex CLI 的开发者来说，Gemini CLI 提供了一个零成本迁移的开源替代方案；对于 Agent 框架研究者来说，它的 ReAct Loop + Hooks 架构是生产级 Agent Runtime 的优秀参考实现。

> **一句话评价**：Google 用开源 + 免费大杀器杀入终端 Agent 战场——不仅功能对标竞品，更有 Hooks 和 Checkpointing 两项独门绝技。
