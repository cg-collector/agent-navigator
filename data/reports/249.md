# Skyvern：基于LLM与计算机视觉的开源浏览器Agent

## 一、项目背景与问题定义

Skyvern是由Skyvern AI（YC孵化）开发的开源浏览器自动化Agent框架，核心定位是**用自然语言驱动浏览器完成复杂工作流**。项目灵感来源于BabyAGI和AutoGPT的任务驱动自主Agent设计范式，但关键差异在于——它赋予Agent通过Playwright等浏览器自动化库直接操作网页的能力。

**解决的核心问题：**

1. **传统浏览器自动化的脆弱性**：基于DOM解析和XPath的脚本在网站布局变更时极易断裂，维护成本高昂
2. **跨站泛化能力缺失**：每个目标网站都需要编写定制化脚本，无法"一次编写、多站复用"
3. **复杂多步骤任务的规划困难**：单一Prompt驱动的循环架构难以处理多目标长链路任务（如"去Amazon加购iPhone 16、手机壳和屏幕保护膜"），容易出现重复操作或遗漏

Skyvern的愿景是让用户用一句自然语言描述意图，Agent就能在任意网站上自主导航、交互、提取数据并完成任务。

## 二、核心架构深度剖析

### 2.1 整体架构演进

Skyvern经历了三个版本的架构迭代：

**v1.0 — 单Actor循环架构**
```
用户Prompt → LLM决策 → 浏览器动作 → 截图反馈 → LLM决策 → ...
```
最简单的单循环设计，LLM同时负责规划和执行。WebVoyager得分约45%，仅适合单目标简单任务。

**v2.0 — Planner-Actor-Validator三阶段架构（当前版本）**
```
用户Prompt
    ↓
[Planner] 任务分解为子目标列表 + 维护工作记忆
    ↓
[Actor] 逐个执行子目标（截图→LLM决策→Playwright动作）
    ↓
[Validator] 验证子目标是否真正完成
    ↓ ← 失败则反馈给Planner重新规划
成功 → 下一个子目标
```

这一架构升级将WebVoyager分数从~45%提升至**85.85%**，达到SOTA水平。

### 2.2 核心组件

| 组件 | 职责 | 技术实现 |
|------|------|----------|
| **Planner** | 将复杂Prompt分解为有序子目标列表 | LLM结构化输出，维护completed/pending状态 |
| **Actor (Task Runner)** | 执行具体浏览器操作 | Playwright + Vision LLM截图理解 |
| **Validator** | 确认动作是否达成预期效果 | 截图+页面状态验证，返回bool+错误信息 |
| **Browser Engine** | 底层浏览器控制 | 基于Playwright，支持本地/云端模式 |

### 2.3 SDK设计哲学

Skyvern选择**扩展Playwright而非替代它**的策略。它在标准Page对象上增加四个AI命令：

- `page.act(prompt)` — 自然语言执行动作
- `page.extract(prompt, schema)` — 结构化数据抽取
- `page.validate(prompt)` — 页面状态验证
- `page.prompt(prompt)` — 通用LLM查询

同时支持三种交互模式：传统CSS/XPath选择器、纯AI自然语言、以及**AI Fallback模式**（先尝试选择器，失败后回退到AI视觉定位）。这种渐进式增强的设计极大降低了迁移成本。

## 三、技术亮点

### 亮点1：Planner-Actor-Validator三阶段闭环

**原理：** 这是Skyvern从45%跃升至85.85%的核心架构创新。

- **Planner阶段**引入了显式的工作记忆（working memory），将复杂任务分解为有序子目标队列。这解决了长Prompt导致的幻觉率上升问题——LLM不需要在一次推理中处理所有步骤。
- **Validator阶段**是关键的"监督者"角色。传统单循环架构中，Agent执行"点击添加购物车"后默认认为成功，但实际上可能因为弹窗遮挡、网络延迟等原因导致操作未生效。Validator通过重新截图分析确认真实状态，失败时将错误反馈给Planner触发重试或替代策略。

这一设计的深层意义在于：**将"执行即成功"的隐式假设替换为"验证后才确认"的显式检查**，这与软件测试中的TDD理念异曲同工。

### 亮点2：Vision-first元素定位策略

**原理：** Skyvern不依赖预定义的XPath/CSS选择器来定位页面元素，而是采用**视觉语言模型（VLM）理解截图**的方式。

流程：`页面截图 → VLM识别可交互元素及其语义 → 映射到Playwright坐标/选择器 → 执行动作`

优势：
- **零样本跨站能力**：面对从未见过的网站，VLM能直接从视觉层面理解按钮、表单、链接的含义
- **抗布局变更**：网站改版后无需更新任何代码，VLM会自动适应新布局
- **处理动态内容**：JavaScript渲染的内容对DOM解析器可能是空的，但截图中可见即可交互

这是与传统Selenium/Playwright脚本最本质的区别——从"基于DOM结构的程序化交互"转向"基于视觉理解的语义化交互"。

### 亮点3：全量公开的Benchmark评估平台

**原理：** Skyvern在`eval.skyvern.com`公开了WebVoyager评估中**每一个测试用例的完整运行记录**，包括每一步的思考过程、采取的动作、截图和结果判断。

这在业界极为罕见。大多数Agent论文只公布一个最终数字，而Skyvern允许任何人：
- 查看特定case的失败原因（是规划错误？执行错误？还是验证遗漏？）
- 分析哪些类型的任务表现薄弱
- 追踪网站随时间变化对Agent性能的影响

此外，Skyvern主动清理了过时的测试用例（如已发布产品的发布日期查询），并更新了航班/酒店预订中的过期日期，确保基准测试反映的是**真实的Agent能力而非数据集偏差**。这种对评估严谨性的追求值得整个领域学习。

## 四、基准测试表现

| 基准测试 | Skyvern得分 | 备注 |
|----------|-------------|------|
| **WebVoyager** | **85.85%** | SOTA开源方案，接近Google Mariner等闭源产品 |
| **WebBench** | 64.4% | 复杂多步网页任务基准 |

**关键细节：**
- 所有测试均在**Skyvern Cloud云端环境**运行（非本地浏览器），使用异步云浏览器，更贴近生产环境
- 使用GPT-4o作为主决策模型，GPT-4o-mini辅助
- 相比v1.0的~45%，v2.0的Planner-Actor-Validator架构带来了**超过40个百分点的绝对提升**
- 中间态（仅加Planner，无Validator）约为68.7%，说明Validator贡献了约17个百分点的增益

## 六、对实际工作的启发

1. **Planner-Actor-Validator模式可直接借鉴到OpenClaw的复杂任务处理中**：当学长下达多步骤指令时，先分解→逐步执行→每步验证，比一口气生成完整方案再执行更可靠。特别是Validator环节——每次工具调用后确认结果是否符合预期，可以大幅减少"以为成功了其实没成功"的情况。

2. **AI Fallback模式的渐进式增强思路值得学习**：先用确定性方法（选择器、固定流程），失败后再启用AI能力。这比纯AI方案更可控、更快、更便宜。在楠楠的工具调用设计中也可以应用——优先走确定路径，异常时才启动重推理。

3. **Vision-first的交互范式是趋势**：对于飞书文档操作、网页浏览等场景，当API/SDK不够用时，截图→VLM理解→操作的路径可以作为fallback方案。特别是飞书某些没有开放API的操作，理论上可以通过浏览器自动化+视觉理解来完成。

4. **公开评估过程的做法**：如果未来楠楠需要评估自己的Agent能力（比如恋爱模拟器的对话质量、股票分析的准确率），应该记录完整的中间过程而非只看最终分数，这样才能真正定位问题并持续改进。
