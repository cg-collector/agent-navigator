# AI Agent 学习报告 #109 — PydanticAI

## 项目名
**PydanticAI**

## GitHub 链接
https://github.com/pydantic/pydantic-ai

## 一句话简介
由 Pydantic 团队（FastAPI 创作者）打造的 Python AI Agent 框架，以「把 FastAPI 的开发体验带到 GenAI 领域」为核心理念，强调类型安全、结构化输出和依赖注入。

---

## 架构设计

### 核心模块

```
┌─────────────────────────────────────────────┐
│                  Agent（泛型核心）              │
│  Agent[DepsType, OutputType]                 │
│  ├─ model: Model (LLM provider)             │
│  ├─ deps_type: type (依赖注入类型)            │
│  ├─ output_type: BaseModel (结构化输出)       │
│  ├─ tools: list[FunctionTool]               │
│  ├─ capabilities: list[Capability]           │
│  └─ hooks: list[Hook]                       │
├─────────────────────────────────────────────┤
│              Models & Providers 层           │
│  OpenAI / Anthropic / Gemini / DeepSeek ... │
│  统一接口 → ModelProtocol                   │
├─────────────────────────────────────────────┤
│              Capabilities（可组合能力）        │
│  Thinking / WebSearch / MCP / A2A          │
├─────────────────────────────────────────────┤
│         Durable Execution（持久化执行）        │
│  Temporal / DBOS / Prefect 集成             │
├─────────────────────────────────────────────┤
│     Observability (Logfire / OpenTelemetry) │
│  Tracing / Evals / Cost Tracking            │
└─────────────────────────────────────────────┘
```

### 技术栈
- **语言**: Python 3.10+（大量使用 `match/case`、`type[X]` 泛型、`ParamSpec`）
- **核心依赖**: Pydantic V2（验证层）、`anyio`（异步运行时）、`httpx`（HTTP 客户端）
- **可选集成**: Logfire（可观测性）、Temporal/DBOS（持久执行）、MCP SDK、A2A 协议
- **测试**: pytest + 覆盖率 >90%

### 工作流程

1. **Agent 定义阶段**：通过构造函数或 YAML/JSON Agent Spec 定义 Agent，指定模型、工具、输出类型、依赖类型
2. **依赖注入**：通过 `RunContext[DepsType]` 将外部依赖（数据库连接、配置等）注入到 tool 函数和动态指令中
3. **工具注册**：使用 `@agent.tool` 装饰器注册函数，Pydantic 自动从函数签名提取参数 schema，docstring 作为 LLM 描述
4. **运行循环**：
   - 用户消息 + system prompt + 工具 schema 发送给 LLM
   - LLM 返回文本或工具调用请求
   - 若是工具调用 → 执行工具 → 结果回传给 LLM → 循环直到最终文本响应
   - 最终输出用 `output_type` 的 Pydantic 模型做结构化验证，失败则让 LLM 重试
5. **可观测性**：每一步都产生 OpenTelemetry Span，可接入 Logfire 或其他 OTel 后端

## 技术亮点

### 1. 泛型 Agent 类型系统 — 编译期类型安全

这是 PydanticAI 最核心的设计。Agent 是双参数泛型类：

```python
# 定义依赖和输出类型
@dataclass
class SupportDependencies:
    customer_id: int
    db: DatabaseConn

class SupportOutput(BaseModel):
    advice: str
    block_card: bool
    risk: int = Field(ge=0, le=10)

# Agent[SupportDependencies, SupportOutput]
# IDE 能完整推导 result.output 的类型为 SupportOutput
support_agent = Agent(
    'openai:gpt-5.2',
    deps_type=SupportDependencies,
    output_type=SupportOutput,
)

result = await support_agent.run('查询余额', deps=deps)
result.output.advice    # ✅ IDE 知道这是 str
result.output.risk      # ✅ IDE 知道这是 int
```

**关键点**：`deps_type` 和 `output_type` 不是运行时字典，而是真正的 Python 类型。静态类型检查器（mypy/pyright）能在写代码时就捕获类型错误。这和其他框架用 `dict[str, Any]` 传参形成鲜明对比。

### 2. 依赖注入系统 — RunContext 模式

```python
@support_agent.tool
async def customer_balance(
    ctx: RunContext[SupportDependencies],  # 自动注入，类型匹配 deps_type
    include_pending: bool,                 # 这些参数暴露给 LLM 作为 schema
) -> float:
    # ctx.deps 是完全类型化的 SupportDependencies 实例
    balance = await ctx.deps.db.customer_balance(
        id=ctx.deps.customer_id,
        include_pending=include_pending,
    )
    return balance

@support_agent.instructions
async def add_customer_name(ctx: RunContext[SupportDependencies]) -> str:
    name = await ctx.deps.db.customer_name(id=ctx.deps.customer_id)
    return f'客户姓名: {name}'
```

设计精妙之处：
- `ctx` 参数对 LLM **不可见**（不会出现在工具 schema 中），只对开发者可见
- `ctx` 之后的参数才被提取为工具的输入 schema
- 动态指令（`@agent.instructions`）也能访问依赖，实现上下文感知的 prompt 构建
- 如果 `RunContext` 的类型标注错了，**mypy 直接报错**，不用等到运行时才发现

### 3. 结构化输出 + 自动重试机制

```python
class AnalysisResult(BaseModel):
    summary: str = Field(description='分析摘要')
    confidence: float = Field(ge=0.0, le=1.0)
    tags: list[str]

agent = Agent(
    'anthropic:claude-sonnet-4-6',
    output_type=AnalysisResult,  # 强制结构化输出
)

result = agent.run_sync('分析这段文本')
# result.output 保证是 AnalysisResult 实例
# 如果 LLM 返回不符合 schema → Pydantic 验证失败 → 
# 自动将错误信息反馈给 LLM → 让它修正 → 重试（默认最多3次）
```

这个机制的实现原理：
- 将 Pydantic model 的 JSON schema 注入到 LLM 的 system prompt 或 tool definition 中
- LLM 输出后用 `model_validate()` 做严格校验
- 验证失败时，将 Pydantic 的 `ValidationError` 信息（具体哪个字段、什么原因）翻译成自然语言提示返回给 LLM
- 这比 LangChain 的 OutputParser 方案更优雅——不需要手写 parser，Pydantic 本身就是验证引擎

### 4. Capabilities 可组合能力系统

```python
from pydantic_ai.capabilities import Thinking, WebSearch

agent = Agent(
    'anthropic:claude-sonnet-4-6',
    capabilities=[Thinking(), WebSearch()],
)
```

Capabilities 是可插拔的功能单元，每个 Capability 可以：
- 注册额外的 tools（如 WebSearch 注册搜索工具）
- 修改 system prompt（如 Thinking 添加「请先思考再回答」指令）
- 配置模型参数（如 Thinking 设置 extended thinking）

内置 Capabilities 包括：
| Capability | 功能 |
|---|---|
| `Thinking` | 扩展思维链（extended thinking），支持 Anthropic/Gemini 的原生 thinking |
| `WebSearch` | 根据模型自动选择 Tavolt/Google Search/Perplexity |
| `MCP` | Model Context Protocol 客户端/服务端 |

第三方可以发布自己的 capability 包（类似 npm 包的概念），实现能力的模块化分发。

### 5. Durable Execution — 持久化 Agent 执行

```python
from pydantic_ai import Agent
from pydantic_ai.durable_execution import temporal

agent = Agent('openai:gpt-5.2')
# 用 Temporal Workflow Engine 包装 agent run
# 即使进程崩溃、API 超时，也能从断点恢复
run_id = await temporal.execute(agent, user_message, deps=deps)
```

支持三种持久化后端：
- **Temporal**：专业的工作流引擎，支持长运行任务、定时重试、状态恢复
- **DBOS**：声明式操作系统级持久化
- **Prefect**：数据工作流编排

这对生产环境至关重要——普通 Agent 框架的工具调用如果中途失败，整个状态就丢了。PydanticAI 通过与成熟的工作流引擎集成解决了这个问题。

### 6. Graph 系统 — 类型安全的 DAG 编排

```python
from pydantic_graph import Graph, End

research_step = ResearchStep()
analyze_step = AnalyzeStep()
report_step = ReportStep()

graph = Graph(
    nodes=(research_step, analyze_step, report_step),
    edges=(
        research_step >> analyze_step,
        analyze_step >> [report_step, End],  # 条件分支
    ),
)
```

Graph 系统独立于 Agent 但深度集成：
- Node 之间通过类型化的数据流通信（不是 dict）
- 支持 `>>` 操作符定义边（语法糖）
- 支持并行执行（`*` 操作符）
- 支持 Decision node（条件分支）
- Reducer node 用于聚合多个上游结果

## 对我们（OpenClaw/楠楠）的启发

### 1. 🔑 类型安全是 Agent 框架的下一个竞争维度

PydanticAI 证明了 Agent 开发不一定要在「动态灵活性」和「类型安全」之间做取舍。OpenClaw 的 skill 系统、tool 定义如果能引入类似的泛型+Pydantic 验证模式，可以大幅减少运行时错误。特别是楠楠调用工具时的参数校验，目前基本靠 LLM 自觉，如果有强制的 schema 验证层会可靠得多。

### 2. 📦 依赖注入模式值得借鉴

当前 OpenClaw 的 subagent/task 传递上下文主要靠序列化 JSON 或文件。PydanticAI 的 `RunContext[Deps]` 模式展示了如何用类型化的依赖注入替代隐式全局状态。对于 OpenClaw 来说，这意味着：
- Skill 的执行上下文可以是类型化的（而不是 raw dict）
- Tool 函数能声明式地获取需要的资源（数据库、API client、配置）
- 类型错误在部署前就能被发现

### 3. 🔄 结构化输出的自动重试是刚需

PydanticAI 的「验证失败→反馈LLM→重试」闭环非常实用。楠楠目前生成报告、写文件时，偶尔会出现格式不规范的问题。如果在输出层加一个 Pydantic 验证+自动修正的环节，交付质量会显著提升。

### 4. 🧩 Capabilities 的可组合设计

OpenClaw 目前的 skill 系统已经是插件式的，但缺少「组合」语义——每个 skill 相对独立。PydanticAI 的 Capability 模式展示了如何让功能单元像乐高一样组合：一个 Agent 可以同时拥有 Thinking + WebSearch + MCP 能力，且它们之间不冲突。OpenClaw 可以考虑引入类似的 composition layer。

### 5. ⚡ Durable Execution 对长任务至关重要

楠楠跑 coding-agent 这类可能耗时几十分钟的任务时，如果中间出问题就全白费了。PydanticAI 与 Temporal/DBOS 的集成为 OpenClaw 提供了一个参考：长任务的 checkpoint 和恢复机制应该是一等公民，而不是事后补丁。

### 6. 💡 整体评价

PydanticAI 不是功能最全的 Agent 框架（没有内置 RAG pipeline、没有多 Agent 对话编排），但它在**工程品质**上设立了新标杆。它的哲学是：做好一件事（类型安全的 Agent 抽象），然后通过标准协议（MCP/A2A/OTel）与其他系统对接。这种「小而精 + 标准接口」的设计哲学，和 OpenClaw 的定位有相似之处——都是注重开发者体验和工程质量的项目。
