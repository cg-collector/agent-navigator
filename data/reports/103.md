## 架构设计

### 整体架构演进

OpenHands 经历了从 V0（单体架构）到 V1（模块化 SDK）的重大重构：

| 维度 | V0（旧） | V1（新） |
|------|----------|----------|
| 架构风格 | 单体 Monorepo | 四包模块化 |
| 执行模型 | 强制 Docker 沙箱 | 本地优先，可选沙箱 |
| 状态管理 | 可变配置散落各处 | Event Sourcing + 单一状态源 |
| Agent 与应用 | 紧耦合 | SDK 作为共享库，应用消费 |

### 核心模块（四包设计）

```
openhands/
├── sdk/              # 核心抽象层（Agent, Conversation, LLM, Tool, MCP）
├── tools/            # 具体工具实现（文件操作、终端、浏览器、MCP 客户端）
├── workspace/        # 执行环境（Docker Runtime、Local Runtime、Hosted API）
└── agent_server/     # REST/WebSocket 服务端，供远程调用
```

**1. `openhands.sdk` — 核心抽象**
- **Event Sourced State Management**：所有交互作为不可变事件追加到日志
- **LLM 抽象**：统一 100+ 提供商的接口，model-agnostic 路由
- **Tool 类型系统**：基于 Pydantic 的类型安全工具定义 + MCP 原生集成
- **Agent 接口**：默认只需几行代码即可创建 Agent

```python
from openhands.sdk import LLM, Conversation
from openhands.tools.preset.default import get_default_agent

llm = LLM(model="openhands/claude-sonnet-4-5-20250929", api_key="...")
agent = get_default_agent(llm=llm)
conversation = Conversation(agent=agent, workspace="/path/to/project")
conversation.send_message("Write 3 facts about this project into FACTS.txt.")
conversation.run()
```

**2. `openhands.tools` — 工具层**
- 文件编辑（EditFile、ReadFile、WriteFile）
- 终端执行（ShellCommand）
- 浏览器交互（Browser）
- Git 操作（GitOperations）
- MCP 工具桥接

**3. `openhands.workspace` — 执行环境**
- LocalRuntime：直接在主机执行（开发/调试用）
- DockerRuntime：容器化沙箱执行（生产/多租户用）
- 同一套 Agent 代码可无缝切换运行环境

**4. `openhands.agent_server` — 服务层**
- REST API + WebSocket 双协议
- 支持暂停/恢复、子代理委派、历史恢复
- 内置安全分析器审查 Agent 动作

### 事件驱动工作流

```
用户消息 → MessageEvent(append to EventLog)
    ↓
Agent 推理 → ActionEvent(tool_call) [append]
    ↓
Tool 执行 → ObservationEvent(result) [append]
    ↓
上下文窗口管理 → CondensationRequest (历史压缩)
    ↓
循环直到完成 / 用户中断 / Token 上限
```

**事件层次结构：**
```
Event (基类: ID, timestamp, source)
├── LLMConvertibleEvent (可转为 LLM message)
│   ├── MessageEvent (用户/助手文本)
│   ├── ActionEvent (工具调用 + thought/reasoning)
│   ├── SystemPromptEvent (系统提示 + tool schema)
│   ├── CondensationSummaryEvent (压缩摘要)
│   └── ObservationBaseEvent (工具响应基类)
│       ├── ObservationEvent (成功结果)
│       ├── UserRejectObservation (用户拒绝)
│       └── AgentErrorEvent (错误)
└── Internal Events (不暴露给 LLM)
    ├── ConversationStateUpdateEvent
    ├── CondensationRequest
    └── PauseEvent
```

## 技术亮点

### 1. Event Sourcing + 单一状态源（Single Source of Truth）

这是 OpenHands 最核心的设计决策。所有组件（Agent、Tool、LLM）都是**不可变的**，唯一的状态载体是 `ConversationState`：

```python
class ConversationState(BaseModel):
    # 元数据字段（直接存储）
    agent_status: AgentStatus = AgentStatus.RUNNING
    stats: AgentStats = AgentStats()
    
    # 事件日志（追加-only）
    event_log: EventLog = Field(default_factory=EventLog)
```

**为什么这很重要？**
- **确定性回放**：从任意时间点重放事件日志，可以完美复现 Agent 行为（对调试和审计极其关键）
- **线程安全**：FIFO 锁 + 两路径更新模式（元数据直接改，交互走事件）
- **崩溃恢复**：持久化到磁盘后，断点续传天然支持
- **LLM 上下文管理**：只有 `LLMConvertibleEvent` 子类会被发送给 LLM，内部事件自动过滤

**对比我们的启发**：OpenClaw 的 session/memory 机制如果采用 event sourcing 思路，可以大幅提升可观测性和调试能力。

### 2. 本地优先 + 可选沙箱的双模式执行

V0 强制 Docker 导致的问题：
- 每次对话跨两个进程（agent + sandbox），状态可能分歧
- 本地执行需要绕过沙箱层，导致代码重复（MCP 和 Tool 都有 sandbox/local 两套实现）

V1 的解决方案：

```python
# workspace 抽象统一了本地和远程
class Workspace(ABC):
    @abstractmethod
    async def execute(self, action: Action) -> Observation: ...

class LocalWorkspace(Workspace):
    """直接在主机执行，开发时零延迟"""
    async def execute(self, action):
        return await self._local_executor.run(action)

class DockerWorkspace(Workspace):
    """容器化执行，生产环境安全隔离"""
    async def execute(self, action):
        return await self._container_executor.run(action)
```

**关键洞察**：同一个 Agent 对象，切换 `workspace` 参数就能从本地开发切到云端部署。这对我们很有参考价值——OpenClaw 的 subagent 执行也可以考虑 local/sandbox 双模式。

### 3. Context Window Management（智能上下文压缩）

长时间运行的 Agent 会遇到 token 溢出问题。OpenHands 的方案不是简单截断，而是**基于事件的智能压缩**：

```
正常循环: Event 追加 → 发送给 LLM
    ↓ 当接近 token 上限时
CondensationRequest 触发 → LLM 总结历史事件为 CondensationSummaryEvent
    ↓ 
旧事件标记为 "forgotten"，保留摘要
    ↓
继续循环，总 token 数可控
```

这个设计的精妙之处在于：
- 压缩本身也是由 LLM 完成的（保证语义保真度）
- 原始事件并不删除（只是标记 forgotten），支持审计回溯
- 摘要事件也是 `LLMConvertibleEvent`，自然融入后续上下文

### 4. Security Analyzer（内置安全审计）

OpenHands 在 Tool 执行前后插入安全检查层：

```python
class SecurityAnalyzer:
    """分析 Agent 动作是否安全"""
    
    def analyze(self, action: Action, state: ConversationState) -> SecurityDecision:
        # 1. 检查动作类型风险等级
        # 2. 检查目标路径是否越界
        # 3. 检查命令是否包含危险模式
        # 4. 返回 ALLOW / DENY / CONFIRM_NEEDED
        ...
```

与简单的 allowlist/denylist 不同，OpenHands 的安全分析是**上下文感知**的——它会结合当前任务和会话历史做判断，而不是机械匹配规则。同时支持三种策略：
- **Auto-accept**：全自动信任（CI/CD 场景）
- **Auto-reject**：高风险动作自动拒绝
- **Ask human**：模糊场景交给人确认

### 5. 两层组合性（Two-Layer Composability）

```
部署层组合:
  SDK (core) + Tools (impl) + Workspace (runtime) + Server (API)
  
扩展层组合:
  自定义 Tool → 注册到 Agent
  自定义 Agent → 替换默认实现
  自定义 LLM Router → 多模型策略路由
  自定义 Workspace → 新执行环境（如 K8s）
```

这种设计让 OpenHands 既是开箱即用的产品，又是可深度定制的框架。对比之下，很多 Agent 框架要么太死板（只能按预设流程跑），要么太自由（什么都得自己写），OpenHands 在两者之间找到了好的平衡点。

## 对 OpenClaw/楠楠的启发

### 🔥 高价值启发

**1. Event Sourcing 用于 Session 管理**
- 当前 OpenClaw 的 session 历史是线性增长的，没有结构化的事件模型
- 可以借鉴 `ConversationState` + `EventLog` 的设计：每次工具调用、消息收发都记录为不可变事件
- 直接收益：session 断点恢复、行为审计、token 用量精确统计

**2. Local/Sandbox 双模式执行**
- 我们的 subagent 目前只能在主机上跑。对于需要执行不可信代码的场景（比如 coding-agent 跑用户提供的项目），缺少隔离能力
- 可以引入轻量沙箱模式：Docker 或 firejail，按需切换
- 不需要照搬 OpenHands 的重量级方案，但"接口统一、后端可切换"的思路值得学

**3. 上下文压缩策略**
- 长对话的 context window 问题我们也面临。当前主要靠模型自身的 context length 硬扛
- OpenHands 的 event-level condensation 很优雅：不是截断消息，而是让 LLM 总结"到目前为止做了什么"
- 可以在我们的 heartbeat / 长任务 subagent 中实现类似机制

**4. 安全分析的分层确认**
- 我们有 `/approve` 机制，但比较粗糙（整体批准/拒绝）
- OpenHands 的三级策略（auto-accept / auto-reject / ask human）+ 上下文感知判断更精细
- 特别是"根据任务上下文动态调整安全策略"这个思路，比静态规则强很多

**5. SDK 化的架构思维**
- OpenHands 从 monolith 到 SDK 的演进经验很值钱。V0 的问题是所有东西耦合在一起，改一个功能牵一发动全身
- OpenClaw 如果要长期发展，核心逻辑（agent loop、tool dispatch、memory）应该作为 library 暴露，CLI/GUI/cron 都是这个 library 的 consumer
- 这和我们 AGENTS.md 里提到的"复杂任务派 subagent"方向是一致的，但需要更清晰的模块边界
