## 一、项目背景

2025年，AI Agent 生态呈现爆发式增长——LangChain、CrewAI、AutoGen、Google ADK、OpenAI Agents SDK 等框架百花齐放。然而一个根本性问题日益凸显：**不同框架构建的 Agent 之间无法互通协作**。一个用 LangGraph 构建的代码审查 Agent 和一个用 CrewAI 构建的数据分析 Agent，即使部署在同一网络中，也如同说不同语言的孤岛。

Google 在 2025年4月正式发布 A2A（Agent2Agent）协议，旨在解决这一互操作性问题。与 Anthropic 的 MCP（Model Context Protocol）解决"Agent 如何连接工具和数据源"不同，A2A 聚焦于**"Agent 之间如何像 Agent 一样通信和协作"**。Google 明确定位两者为互补关系：MCP 管纵向的工具集成，A2A 管横向的 Agent 间协调。

该项目已贡献给 Linux Foundation，采用 Apache 2.0 协议开源，获得了 IBM Research 等机构的联合支持，并推出了与 Google Cloud 合作的专业课程。

## 二、核心架构

A2A 采用**三层分离架构**，确保核心语义的一致性和传输层的可扩展性：

### Layer 1：规范数据模型（Canonical Data Model）

基于 Protocol Buffers 定义的核心数据结构，所有实现必须理解：

- **Task（任务）**：Agent 间协作的基本单元，包含状态机（submitted → working → completed/failed/cancelled）
- **Message（消息）**：Agent 间交换的信息载体，支持文本、文件引用、结构化数据
- **AgentCard（Agent 卡片）**：类似"名片"，声明 Agent 的能力、连接信息、支持的技能列表
- **Part（部分）**：消息的内容片段，支持多模态（text、file、data）
- **Artifact（制品）**：任务执行过程中产生的持久化输出
- **Extension（扩展）**：预留扩展点，允许自定义字段

关键设计：**proto 文件是唯一权威来源**（normative source），JSON Schema 等均为派生构建产物，从 proto 自动生成，杜绝规范漂移。

### Layer 2：抽象操作层（Abstract Operations）

定义六大核心操作，与具体传输协议无关：

1. **SendMessage** — 发送消息给目标 Agent
2. **StreamMessage** — 流式发送（实时对话场景）
3. **GetTask** — 查询任务状态和结果
4. **ListTasks** — 列出历史任务
5. **CancelTask** — 取消正在运行的任务
6. **GetAgentCard** — 获取 Agent 能力声明

### Layer 3：协议绑定层（Protocol Bindings）

将抽象操作映射到具体传输协议：
- **JSON-RPC 2.0 over HTTP(S)** — 默认绑定，最广泛支持
- **gRPC** — 高性能 RPC 场景
- **HTTP/REST** — RESTful 风格端点
- **自定义绑定** — 开放扩展

### 通信模式三合一

A2A 原生支持三种交互模式，根据任务特征自动选择：

| 模式 | 适用场景 | 技术实现 |
|------|----------|----------|
| 同步请求/响应 | 短任务、简单查询 | JSON-RPC + 轮询 |
| SSE 流式 | 实时对话、进度更新 | Server-Sent Events |
| 异步推送 | 长时间任务、人机协同 | Push Notification + 回调 |

## 三、技术亮点

### 亮点1："不透明执行"（Opaque Execution）原则

这是 A2A 最具前瞻性的设计理念。传统多 Agent 系统通常要求共享内部状态、思维链或工具定义，这在跨组织协作中既不现实也不安全。A2A 提出：**Agent 仅通过声明的能力（AgentCard）和交换的消息来协作，无需暴露内部记忆、推理过程或工具实现细节**。

这意味着：
- 甲公司的法律审查 Agent 可以调用乙公司的合同分析 Agent，而无需暴露各自的专有逻辑
- 不同框架的 Agent 可以即插即用地组合，无需了解对方 internals
- 安全边界清晰——每个 Agent 仍然是"黑盒"

这一原则直接解决了企业级多 Agent 部署中的**知识产权保护**和**安全隔离**两大痛点。

### 亮点2：异步优先 + 任务状态机设计

A2A 从第一天起就是为**长时间运行任务**设计的。Task 对象拥有完整的状态机：

```
submitted → working → completed
                       → failed
                       → cancelled
                       → (input-required, 人工介入点)
```

特别值得注意的是 `input-required` 状态——它原生支持**人机协同（Human-in-the-Loop）**场景。当 Agent 需要人工确认或补充信息时，任务可以优雅地暂停等待，而非超时失败。这对金融审批、医疗诊断等需要人类把关的场景至关重要。

相比之下，许多早期的 Agent 框架（包括 MCP 的早期设计）都是同步请求/响应模型，处理长任务时只能靠客户端轮询 hack。

### 亮点3：三层解耦 + Proto 为源的规范治理

A2A 的三层架构（数据模型→操作→绑定）实现了真正的关注点分离。更关键的是其**规范治理策略**：

- 以 `.proto` 文件为唯一 normative source
- 所有 SDK/Schema/文档均由 proto 自动生成
- 引入 RFC 2119 需求语言（MUST/SHOULD/MAY）
- 定义了完整的废弃生命周期（引入→标记废弃→下一大版本移除）
- 向后兼容保证：旧名称在移除前至少保留一个大版本

这种工程化程度在 AI 开源项目中极为罕见，更接近 IETF 标准的治理水平，保证了协议在大规模生态中的稳定性。

## 四、基准测试与生态表现

A2A 作为协议标准，不以传统的 accuracy/F1 分数衡量，而是以**生态采纳度**和**互操作性验证**为核心指标：

### 生态采纳（截至2026年初）
- **6种官方 SDK**：Python、Go、JS/TS、Java、.NET，覆盖主流开发语言
- **框架集成**：Google ADK 原生支持、LangGraph 集成示例、BeeAI 支持、CrewAI 社区适配
- **示例仓库**：a2a-samples 提供医疗、电商、客服等多领域 demo
- **教育合作**：与 DeepLearning.AI 联合推出专业课程（Holt Skinner 等主讲）
- **企业背书**：IBM Research 联合参与，Google Cloud 推出 Partner Program

### 与 MCP 的定位对比

| 维度 | MCP (Anthropic) | A2A (Google) |
|------|-----------------|--------------|
| **解决问题** | LLM ↔ 工具/数据源的连接 | Agent ↔ Agent 的协作 |
| **通信模型** | Client-Server（Host-MCP Server） | Peer-to-Peer（Agent-Agent） |
| **典型场景** | Claude 连接数据库/GitHub | 多个专业 Agent 组合完成复杂任务 |
| **核心对象** | Tool、Resource、Prompt | Task、Message、AgentCard |
| **同步模型** | 主要同步 | 异步优先，原生支持长任务 |
| **透明度要求** | Server 内部对 Host 可见 | Agent 内部互相不可见（Opaque） |
| **关系定位** | 纵向集成层 | 横向协调层 |

两者被明确设计为互补关系：MCP 让单个 Agent 变得更强大（有更多工具），A2A 让多个 Agent 变得更协同（可以分工合作）。一个完整的 Agent 系统可能同时使用两者。

### 与其他多 Agent 框架对比

| 框架/协议 | 定位 | 互操作性 | 标准化程度 |
|-----------|------|----------|------------|
| **A2A** | 通用 Agent 通信协议 | ⭐⭐⭐⭐⭐ 跨框架跨语言 | IETF 级规范治理 |
| **CAMEL (#238)** | 角色扮演多 Agent 框架 | ⭐⭐ 框架内协作 | 学术项目 |
| **AutoGen (MS)** | 多 Agent 对话框架 | ⭐⭐⭐ 框架内+有限外部 | 微软生态 |
| **LangGraph** | 基于 Graph 的 Agent 编排 | ⭐⭐⭐ LangChain 生态 | 社区驱动 |
| **CrewAI** | 角色扮演 Agent 团队 | ⭐⭐ 框架内 | 商业公司 |

A2A 的独特价值在于它是**元协议（meta-protocol）**——不绑定任何特定框架或厂商，目标是成为 Agent 界的 "HTTP"。

## 五、局限性与挑战

1. **生态仍处早期**：虽然 SDK 齐全，但生产级大规模部署案例尚少，多数使用仍在 PoC 阶段
2. **发现机制未完善**：AgentCard 是手动配置的，缺乏类似 mDNS 或服务网格的自动发现机制（路线图中有计划）
3. **安全模型待加强**：当前依赖标准 Web 安全实践（OAuth/JWT），但 Agent 特有的授权模型（如权限委托链）尚未标准化
4. **Google 主导嫌疑**：尽管托管于 Linux Foundation，但核心设计和推进力量来自 Google，社区声音权重有待观察
5. **与 MCP 的潜在摩擦**：虽然官方定位为互补，但在实际落地中两者的边界可能模糊，开发者需自行判断何时用哪个

## 六、实际启发

### 对 Agent 开发者的启示

1. **设计 Agent 时考虑 A2A 接口**：新开发的 Agent 应预留 A2A 兼容层，将能力声明（AgentCard）作为一等公民设计，而非事后补丁
2. **Opaque Execution 是正确抽象**：不要试图让 Agent 共享内部状态。声明式的能力描述 + 结构化的消息交换，比共享内存/数据库更安全、更可维护
3. **异步优先的任务建模**：如果你的 Agent 只支持同步请求/响应，就已经落后了。长任务、人机协同是真实场景的常态

### 对 OpenClaw 类系统的启示

OpenClaw 本身就是一个多 Agent 平台（主 Agent + Subagent），A2A 的设计理念值得借鉴：
- **Subagent 通信可以标准化**：当前主-subagent 通信是内部协议，未来可以考虑 A2A 兼容接口，使 subagent 可以被外部系统调用
- **AgentCard 式的能力注册**：每个 skill/agent 声明自己能做什么，比硬编码路由表更灵活
- **Task 状态机的参考价值**：OpenClaw 的 cron/reminder 系统可以参考 A2A 的任务状态机设计，增加 input-required 等中间状态

### 行业趋势判断

A2A 代表了 AI Agent 从"单兵作战"走向"联合作战"的关键基础设施。正如 HTTP 让 Web 服务器互通、SMTP 让邮件服务器互通，A2A 有潜力成为 Agent 世界的通用语。2026-2027 年将是决定其能否真正成为行业标准的关键窗口期——如果能在 Google 之外吸引到足够的独立采纳者（尤其是 Anthropic、OpenAI、Meta 的某种形式支持），A2A 将成为 AI 基础设施的重要一环。
