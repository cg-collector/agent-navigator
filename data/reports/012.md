# AI Agent 开源项目学习报告 #012

> **研究日期:** 2026-03-27
> **项目名称:** Google ADK (Agent Development Kit)
> **GitHub:** https://github.com/google/adk-python
> **语言支持:** Python / TypeScript / Go / Java
> **协议:** Apache 2.0

---

## 一、项目简介

Google ADK 是 Google 于 2025 年推出的开源 Agent 开发工具包，支持 Python、TypeScript、Go、Java 四种语言。它的核心定位是提供一个 **code-first（代码优先）** 的框架，用于构建、评估和部署复杂的 AI Agent 应用。

ADK 的独特之处在于它同时支持三大 Agent 通信协议：**MCP（Model Context Protocol）**、**A2A（Agent-to-Agent）** 和 **Function Calling**，使其成为目前协议兼容性最全面的 Agent 框架之一。

---

## 二、架构概述

### 2.1 核心抽象：BaseAgent

ADK 的所有 Agent 都继承自 `BaseAgent`，框架提供三种开箱即用的 Agent 类型：

| 类型 | 说明 |
|------|------|
| **LlmAgent** | 由 LLM 驱动的标准 Agent，支持 model、instruction、tools 等配置 |
| **Workflow Agent** | 编排型 Agent，自身不执行任务，负责控制子 Agent 的执行流程 |
| **Custom Agent** | 用户自定义 Agent，继承 BaseAgent 实现 `run_async_impl`，可以是纯逻辑无 LLM |

### 2.2 Agent 层级树（Hierarchy）

ADK 采用 **树形层级结构** 组织多 Agent 系统：

- 通过 `sub_agents` 参数建立父子关系
- 单父节点规则：一个 Agent 只能有一个 parent
- 支持 `agent.find_agent(name)` 遍历查找
- LLM Agent 可以通过 delegation 自动将任务委派给子 Agent

```
RootAgent
├── SequentialAgent (流水线编排)
│   ├── FetchAgent
│   └── ProcessAgent
├── ResearchAgent
└── CodeAgent
    ├── PlannerAgent
    └── ExecutorAgent
```

### 2.3 Workflow Agent（工作流 Agent）

这是 ADK 架构的一大亮点，提供三种内置编排模式：

1. **SequentialAgent** — 顺序执行子 Agent，共享 InvocationContext，可通过 `output_key` 将结果存入 state 供下游使用
2. **ParallelAgent** — 并行执行所有子 Agent，等待全部完成后汇总
3. **LoopAgent** — 循环执行子 Agent，直到满足某个条件才退出（适合迭代优化场景）

### 2.4 三协议支持

| 协议 | 用途 | 说明 |
|------|------|------|
| **MCP** | 连接外部工具和数据源 | Agent 作为 MCP Client，调用远端 Tool Server |
| **A2A** | Agent 间通信 | 跨应用、跨框架的 Agent 互操作协议 |
| **Function Calling** | 模型原生工具调用 | 直接调用代码级函数，零延迟 |

ADK 的设计理念是：**MCP 用于连接工具，A2A 用于连接 Agent，Function Calling 用于简单内联逻辑**。

### 2.5 运行时模型

ADK 的运行时基于 `InvocationContext`（调用上下文）：

- 每个 Agent 运行时都会收到一个 InvocationContext
- Context 中包含：session state、user input、tool registry、callback hooks
- 通过 `output_key` 机制，Agent 可以将结果写入共享状态
- 子 Agent 自动继承父 Agent 的 Context，形成数据流

## 三、技术亮点

### 3.1 🏆 Workflow Agent 作为一等公民

大多数框架（LangGraph 除外）的工作流都是事后附加的。ADK 把 SequentialAgent / ParallelAgent / LoopAgent 设计为 **BaseAgent 的子类**，和普通 LLM Agent 完全同级。这意味着：

- 工作流可以嵌套（Sequential 里嵌 Parallel）
- 工作流可以作为更大工作流的子节点
- 工作流本身可以被 LLM Agent 委派调用

这种 **统一抽象** 比 CrewAI 的 Task/Crew 分离、LangGraph 的 Graph/Node 分离都更优雅。

### 3.2 🏆 三协议原生集成

ADK 不是简单封装了 MCP 或 A2A，而是在 Agent 生命周期中 **原生集成了协议支持**：

- Agent 可以同时拥有 MCP 工具、A2A 远程 Agent、本地函数
- LLM 在 tool calling 时，框架透明地路由到正确的后端
- A2A 支持让不同团队、不同框架构建的 Agent 互操作

### 3.3 🏆 多语言 SDK 同步发布

Python / TypeScript / Go / Java 四种语言的 SDK **同步维护**，API 设计保持一致。这对于企业级项目（尤其是 Java/Go 后端）非常实用，是目前多语言覆盖最广的 Agent 框架。

### 3.4 🏆 output_key 状态传递机制

通过 `output_key` 参数，Agent 可以将 LLM 输出自动存入 session state：

```python
step1 = LlmAgent(name="Fetch", output_key="raw_data")
step2 = LlmAgent(name="Summarize", instruction="Summarize: {raw_data}")
```

这种声明式的状态传递，比手动管理 state 更简洁，类似 React 的 props 传递。

### 3.5 🏆 LLM-Driven Delegation

LLM Agent 可以配置子 Agent，LLM 在运行时自动判断是否需要将任务委派给子 Agent。这不是硬编码的 if-else，而是让 LLM **自主决策** 调用哪个子 Agent，实现了真正的智能调度。

## 四、与同类框架对比

| 特性 | Google ADK | LangGraph | CrewAI | AutoGen |
|------|-----------|-----------|--------|---------|
| 多语言 | ✅ 4种 | ❌ Python only | ❌ Python only | ❌ Python only |
| A2A 协议 | ✅ 原生 | ❌ | ✅ v1.10+ | ❌ |
| MCP 协议 | ✅ 原生 | ✅ 社区集成 | ✅ v1.10+ | ⚠️ 有限 |
| 工作流编排 | ✅ 内置 | ✅ 图结构 | ⚠️ Crew 流程 | ⚠️ 对话驱动 |
| 树形层级 | ✅ 原生 | ⚠️ 图节点 | ❌ 扁平 | ❌ 扁平 |
| 模型绑定 | Gemini 偏重 | 模型无关 | 模型无关 | 模型无关 |

**ADK 的核心优势**：协议最全 + 多语言 + 树形层级架构。主要劣势是深度绑定 Google 生态（Gemini 模型、GCP 部署），虽然支持其他模型但最佳体验在 Google 平台上。

## 五、启发与思考

### 5.1 工作流即 Agent 的设计哲学

ADK 把 SequentialAgent、ParallelAgent 设计为 BaseAgent 子类，这个设计非常值得借鉴。**编排逻辑和执行逻辑使用同一套抽象**，消除了"框架层"和"业务层"的界限。在 OpenClaw 的 sub-agent 系统中，也可以借鉴这种思路——让编排本身成为一个可组合的单元。

### 5.2 协议分层思维

ADK 清晰地将三种通信方式分层：Function Calling 用于进程内调用（最快），MCP 用于工具发现（标准化），A2A 用于跨应用互操作（最灵活）。这种分层不是随意选择，而是根据 **延迟、标准化程度、灵活性的 trade-off** 来决定的。在设计 Agent 系统时，应该根据场景选择合适的通信层。

### 5.3 output_key 的声明式数据流

手动在 state 字典中存取 key 容易出错，而 output_key 提供了声明式、类型安全的接口。这种模式在流水线式 Agent 系统中特别有用，值得在 skill 脚本中采用。

### 5.4 多 Agent 不等于多进程

ADK 的多 Agent 系统可以在**单进程内**运行多个 Agent 实例，通过树形结构和委派机制协作。这比 CrewAI 的进程隔离更轻量，比 LangGraph 的图节点更直观。对于资源受限的环境（如我们当前的 VPS），这种设计更实用。

---

## 六、总结

Google ADK 是一个设计精良、架构优雅的 Agent 框架。它的核心价值不在于某一个单独的特性，而在于 **将多种最佳实践统一到一套简洁的抽象中**：树形层级、工作流即 Agent、三协议集成、声明式数据流。虽然深度绑定 Google 生态是其局限，但其架构思想对任何 Agent 系统的设计都有参考价值。

**推荐关注点：**
- Workflow Agent 的统一抽象设计
- A2A 协议的跨框架互操作理念
- output_key 声明式状态传递
- 多语言 SDK 同步维护的工程实践
