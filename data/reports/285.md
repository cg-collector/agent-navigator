# AI Agent 项目研究报告 #285：OSWorld — 真实计算机环境中的多模态智能体基准

## 1. 项目背景与问题定义

**OSWorld**（全称 *OSWorld: Benchmarking Multimodal Agents for Open-Ended Tasks in Real Computer Environments*）由香港大学 XLANG Lab 团队于2024年4月提出，发表于 NeurIPS 2024。它是**首个在真实操作系统环境中对多模态AI Agent进行端到端评估的基准测试平台**。

### 解决的核心痛点

在OSWorld出现之前，GUI Agent / Computer Use领域存在三大根本性问题：

1. **环境不真实**：现有基准（如MiniWoB++、WebArena、Mind2Web）要么使用模拟/简化环境，要么局限于单一应用域（纯网页或单一App），无法反映真实计算机使用的复杂性和多样性。Agent在玩具环境中表现好，一到真实桌面就崩溃。

2. **评估不可靠**：多数基准采用基于文本相似度或规则匹配的"近似评估"，无法精确判断Agent是否真正完成了任务（比如"把文件移到指定文件夹并重命名"——光看最终截图很难准确判定）。

3. **缺乏跨应用能力测试**：真实计算机任务往往涉及多个应用协作（如在浏览器搜索信息→用LibreOffice编辑文档→保存到指定路径），但此前没有任何基准能系统性地评测这种跨应用工作流。

OSWorld的核心主张是：**要评估一个Computer Use Agent，就必须让它在真实的、完整的操作系统中跑，而不是在沙盒里做选择题。**

## 2. 核心架构深度剖析

### 2.1 整体架构

OSWorld采用**虚拟机+控制器+评估器**的三层架构：

```
┌─────────────────────────────────────────────┐
│              Host Machine (宿主机)            │
│  ┌──────────┐  ┌───────────┐  ┌──────────┐ │
│  │ VM Manager│  │ Task      │  │Evaluator │ │
│  │(VMware/   │  │Controller │  │(执行式    │ │
│  │ VBox/AWS) │  │           │  │ 检验脚本) │ │
│  └────┬─────┘  └─────┬─────┘  └─────┬────┘ │
│       │              │              │       │
└───────┼──────────────┼──────────────┼───────┘
        │         VM Screen/API        │
┌───────▼──────────────▼──────────────▼───────┐
│           Guest VM (Ubuntu/Windows/macOS)    │
│  完整桌面环境 + 真实应用 + 文件系统            │
│  浏览器 / LibreOffice / GIMP / VS Code ...  │
└─────────────────────────────────────────────┘
```

**关键组件：**

- **VM Manager**：管理虚拟机生命周期，支持VMware、VirtualBox、Docker和AWS EC2部署。2025年7月的OSWorld-Verified版本新增AWS支持，可将369个任务的并行评估时间压缩到**1小时以内**。
- **Task Controller**：负责任务初始化（初始状态设置）、Agent交互中转（屏幕截图→Agent→动作指令→执行）、以及任务完成后的后处理。
- **Evaluator**：每个任务配备自定义的**执行式评估脚本**（Execution-based Evaluation），通过程序化方式检查最终状态是否满足要求。整个benchmark共包含**134种不同的评估函数**。

### 2.2 数据流与交互循环

Agent与环境的交互遵循标准的**观察-决策-行动（Observe-Decide-Act）**循环：

```
1. 初始化阶段：
   配置文件 → 设置初始状态（创建文件/打开应用/导航到特定页面）
   
2. 交互循环（每步）：
   a) 截取屏幕截图 → 获取当前视觉状态
   b) 可选：获取Accessibility Tree（无障碍树，结构化UI元素）
   c) 将观察输入VLM/LLM → Agent输出动作（鼠标点击/键盘输入/等待）
   d) 通过pyautogui在VM中执行动作
   
3. 评估阶段：
   Agent宣布完成 或 达到步数上限
   → 运行该任务的专属eval脚本
   → 程序化检查文件内容/注册表/应用状态
   → 返回二元结果：成功/失败
```

### 2.3 任务设计体系

OSWorld包含**369个任务**（另有43个Windows专属任务），覆盖以下维度：

| 维度 | 说明 |
|------|------|
| **应用域** | Chrome/Firefox浏览器、LibreOffice Writer/Calc/Impress、GIMP、VS Code、VLC、文件管理器等 |
| **操作类型** | 点击、拖拽、键盘快捷键、文本输入、菜单导航、多窗口切换 |
| **任务复杂度** | 单步简单操作 → 15+步跨应用工作流 |
| **初始状态** | 支持从中间状态开始（非总是从干净桌面），更贴近真实场景 |

每个任务配置文件包含：任务描述、初始状态setup脚本、评估脚本、超时设置、最大步数限制。

## 3. 技术亮点

### 亮点一：执行式评估（Execution-based Evaluation）

这是OSWorld最具创新性的设计。传统GUI Agent评估依赖：

- **视觉相似度**：对比最终截图与参考图（SSIM/PSNR等）——但对布局微小变化敏感，且无法判断功能性正确性
- **HTML DOM比对**：仅适用于网页场景
- **人工标注**：不可扩展

OSWorld的方案是为每个任务编写**程序化评估脚本**，直接检查计算机状态的"事实"：

```python
# 示例：评估"将hello.txt的内容改为'Hello World'"任务
def evaluate():
    with open('/tmp/hello.txt', 'r') as f:
        content = f.read().strip()
    return content == 'Hello World'
```

这种方式的优越性在于：
- **精确性**：不是"看起来像"，而是"确实等于"
- **可复现**：同一脚本在不同运行中给出一致结果
- **可扩展**：新任务只需写新的eval脚本，无需修改框架
- **134种评估函数**覆盖了文件内容检查、图像属性验证、PDF解析、注册表查询等多种场景

### 亮点二：完整真实环境（Full-Fidelity Environment）

OSWorld不在任何层面做简化妥协：

- **真实应用**：使用LibreOffice而非模拟器，Chrome而非Headless Browser
- **真实GUI渲染**：Agent看到的是像素级真实截图，包括阴影、动画、模糊效果
- **真实文件系统**：支持任意文件读写，路径、权限、编码问题全部真实存在
- **多操作系统**：原生支持Ubuntu、Windows、macOS

这意味着Agent必须面对人类使用电脑时的所有挑战：窗口遮挡、加载延迟、弹窗干扰、字体渲染差异等。这些在简化环境中被刻意抹除的因素，恰恰是当前Agent最薄弱的环节。

### 亮点三：可扩展的任务框架与环境可复现性

OSWorld设计了声明式的任务配置格式，使得添加新任务非常便捷：

```yaml
task_id: "task_301"
instruction: "在GIMP中将图片裁剪为200x150并保存为PNG"
setup:
  - command: "cp /assets/sample.jpg ~/Pictures/input.jpg"
  - command: "gimp ~/Pictures/input.jpg &"
evaluation:
  type: "script"
  path: "evals/task_301.py"
max_steps: 30
timeout: 300
```

同时，通过VM快照机制确保每次评估从完全相同的初始状态开始，消除了环境噪声对评估结果的污染。这在学术界首次实现了GUI Agent评估的**实验可复现性**。

## 4. 基准测试表现

### 原始论文结果（NeurIPS 2024, 2024.04）

| Agent模型 | 成功率 | 备注 |
|-----------|--------|------|
| **人类基线** | **72.36%** | 人类专家在相同环境和时间限制下 |
| GPT-4V (纯文本计划) | 2.03% | 仅生成操作计划，不执行 |
| GPT-4V (带截图反馈) | 6.97% | 截图+文本描述作为观察 |
| Claude 3 Opus | ~7-8% | 论文发布时最新模型 |
| **最佳Agent** | **12.24%** | 当时最强组合 |

**核心发现**：即使是最先进的VLM，成功率也不到人类的1/6。主要失败原因：
- **GUI Grounding困难**：无法准确定位屏幕元素对应的操作目标
- **操作知识匮乏**：不知道"保存"应该用Ctrl+S还是点菜单
- **长程规划崩塌**：超过5步后容易迷失目标或陷入循环

### OSWorld-Verified 更新结果（2025.07）

随着模型能力的快速演进，排行榜变化剧烈：

| 模型/系统 | 成功率 | 时间 |
|-----------|--------|------|
| 人类基线 | 72.4% | — |
| AskUI VisionAgent | **66.2%** | 当前开源最高 |
| Claude Opus 4.6 | ~40-50%区间 | 2025下半年 |
| Claude 3.7 Sonnet (CU) | 28% | 2025.02 |
| OpenAI CUA (GPT-4o) | 38.1% | 2025 |
| GPT-5.4 | **~75%** | 2026年初，已超越人类 |

值得注意的是，**GPT-5.4在2026年初成为首个超越人类表现的模型**（约75% vs 72.4%），标志着Computer Use领域进入新阶段。

## 5. 与同类项目横向对比

| 维度 | **OSWorld** | WebArena | Mind2Web | AndroidWorld | Windows Agent Arena |
|------|------------|----------|----------|-------------|-------------------|
| **环境类型** | 完整操作系统 | 网页模拟 | 网页（冻结DOM） | Android模拟器 | Windows 11 |
| **真实性** | ★★★★★ 真实应用 | ★★★☆☆ 自建网站 | ★★☆☆☆ 冻结快照 | ★★★★☆ 真实Android | ★★★★★ 真实Windows |
| **任务数** | 369 | 812 | 2350 | 9802 | ~100+ |
| **跨应用** | ✅ 强项 | ❌ 仅网页 | ❌ 单网站 | ✅ 多App | ✅ 多App |
| **评估方式** | 执行式脚本 | DOM/视觉混合 | 元素定位 | 执行式 | 执行式 |
| **多模态支持** | ✅ 截图+ATree | ✅ | 部分 | ✅ | ✅ |
| **主要局限** | 任务数相对少 | 环境非真实 | 非交互式 | 移动端专用 | 商业OS授权成本 |

**OSWorld的独特定位**：它不是任务数量最多的（AndroidWorld近万任务），也不是唯一做执行式评估的，但它是**唯一同时在"真实完整操作系统"+"跨应用工作流"+"执行式评估"三个维度都做到位的基准**。这使其成为衡量通用Computer Use能力的"金标准"。

对比来看：
- **WebArena/WebVoyager**系列专注于网页场景，环境可控性强但泛化性受限
- **Mind2Web**规模大但基于冻结DOM，Agent无法真正改变页面状态
- **AndroidWorld**是OSWorld理念在移动端的延伸，同团队出品
- **Windows Agent Arena**填补了Windows生态空白，但受限于商业许可

## 6. 对我们的实际启发

结合OpenClaw的browser agent能力和我们的实际场景，OSWorld带来以下启示：

### 6.1 评估体系建设：从"感觉能用"到"数据说话"

我们当前的agent能力缺乏系统性评估。可以借鉴OSWorld的思想，为OpenClaw构建一套轻量级的**执行式评估集**：
- 针对browser agent：设计20-50个典型网页操作任务（登录、填表、抓取、导航）
- 每个任务配eval脚本检查最终状态（页面URL、DOM元素存在性、下载文件内容）
- 形成回归测试集，每次agent代码变更后自动跑一遍

### 6.2 观察模式的选择：截图 vs Accessibility Tree

OSWorld支持两种观察模式，这是一个重要的工程权衡：
- **纯截图**：信息量大但token消耗高，VLM理解成本大
- **Accessibility Tree**：结构化、紧凑，但丢失视觉布局信息

OpenClaw的browser工具已经支持snapshot（类似ATree）。对于我们的场景，建议**默认用ATree模式节省token**，仅在视觉判断关键时（如验证码识别、布局确认）回退到screenshot。

### 6.3 失败模式的工程防御

OSWorld揭示的Agent典型失败模式在我们的场景中也普遍存在：
- **循环行为**：Agent反复点击同一个位置 → 需要实现action deduplication和loop detection
- **迷失目标**：长程任务中途忘记原始目标 → 需要在prompt中持续注入任务描述
- **GUI定位不准**：点击坐标偏移 → 我们的snapshot+ref机制天然缓解此问题

建议在OpenClaw agent层面增加**步骤计数器 + 动作历史去重 + 目标一致性检查**三层防护。

### 6.4 Computer Use是通用Agent的必经之路

OSWorld证明了一个趋势：**能用好电脑的Agent才是真正的通用助手**。OpenClaw的browser control能力本质上就是Computer Use的一个子集。长期来看，向桌面控制扩展（通过已有browser工具+未来的desktop control插件）是一个自然演进方向。

> **一句话总结**：OSWorld不是又一个benchmark——它重新定义了"什么才算一个合格的Computer Use Agent"。它告诉我们：在真实环境中跑通、用执行脚本验证、跨应用完成任务，这三者缺一不可。对于任何想认真做GUI Agent的团队来说，OSWorld既是标尺也是镜子。
