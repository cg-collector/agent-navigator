## 一、项目背景与问题定义

### 1.1 GUI Agent 的"最后一公里"困境

GUI Agent（图形用户界面智能体）代表了人机交互的下一个范式跃迁——让 AI 像人类一样通过"看屏幕、点按钮、打字"来完成真实世界的数字任务。从 Anthropic 的 Computer Use 到各种手机操控 Agent，这个方向在 2024-2025 年迎来了爆发式增长。然而，尽管进展迅速，当前 GUI Agent 仍面临三个**根本性瓶颈**：

**瓶颈一：训练数据质量差。** 现有 GUI 数据集中近 40% 的样本存在实质性噪声——自动从 HTML 或 Accessibility Tree 提取的标签存在渲染偏移（rendering offset），导致边界框错位和文本描述模糊；导航轨迹数据中充斥着错误的执行路径和冗余动作，为长程规划提供了弱监督甚至错误监督信号。

**瓶颈二：感知与规划的耦合干扰。** 大多数端到端 GUI Agent 将视觉定位（grounding，即"点哪里"）和序列决策（navigation，即"做什么"）耦合在同一个模型中训练。低级空间定位任务和高级推理任务在梯度空间中相互干扰，导致两者都无法达到最优。

**瓶颈三：跨平台泛化能力不足。** 现有评测基准主要集中在英文 Android 环境（如 AndroidControl、AndroidWorld），缺乏对中文移动生态（微信、支付宝、抖音等超级 App）和桌面 Linux 环境的系统评估。Agent 在这些"非标准"环境中的能力基本是黑箱。

### 1.2 OmegaUse 的设计哲学

OmegaUse（名字寓意 "Omega" = 终极/全能 + "Use" = 使用）的核心主张是：**构建一个真正通用的 GUI Agent，关键不在于更大的模型或更复杂的提示词工程，而在于 (1) 高质量的数据工程 和 (2) 解耦的训练范式。**

团队来自百度前沿研究部，这一背景值得注意——百度在中文搜索、地图和 App 生态方面有深厚积累，使得 OmegaUse 成为目前少数同时覆盖**中文移动端**和**桌面端**的通用 GUI Agent 之一。

**关键架构决策：Grounding 与 Navigation 解耦。**

这是 OmegaUse 最核心的设计选择。团队将 GUI Agent 能力拆分为两个独立训练的专门模型：

- **OmegaUse-G（Grounding Model）**：负责"看到屏幕上的目标元素在哪里"，输出精确的边界框坐标 `[xmin, ymin, xmax, ymax]`。这是一个纯粹的视觉定位任务。
- **OmegaUse-N（Navigation Model）**：负责"根据任务目标决定下一步做什么"，输出 `(Observation, Thought, Action)` 三元组。这是一个序列决策任务。

为什么不解耦？因为这两个任务的优化目标在本质上是不同的：
- Grounding 需要**像素级的空间精度**——差几个像素就是点到空白处
- Navigation 需要**高层语义推理**——理解任务意图、规划操作步骤、处理异常恢复
- 将它们放在同一个模型中训练，梯度信号会互相冲突：提升空间精度可能损害推理能力，反之亦然

这种解耦设计类似于计算机视觉中"检测器 + 跟踪器"分离的经典思路，但在 GUI Agent 领域是首次被系统性地验证。

### 2.2 MoE 骨干网络：大智慧、小开销

OmegaUse 选择 **Mixture-of-Experts (MoE)** 作为模型骨干，而非当前流行的稠密（Dense）小型模型（如 7B）或大型稠密模型（如 72B）。这个选择背后的逻辑：

```
传统 Dense 模型的困境:
├── 小模型 (7B): 推理快但推理能力不足，复杂 GUI 任务表现差
├── 大模型 (72B): 推理能力强但推理慢且贵，部署成本高
└── 折中模型 (20-30B): 两边都不讨好

MoE 模型的优势:
├── 总参数量 ≈ 大模型（保留强推理能力）
├── 激活参数量 ≈ 小模型（每次只路由到少量 Expert）
├── 推理速度接近小模型（稀疏激活）
└── 可以针对不同 Expert 特化不同能力
    （如某些 Expert 专精移动端 GUI，某些专精桌面端）
```

MoE 架构特别适合 GUI Agent 场景，因为：
1. **多平台异构性**：移动端和桌面的 UI 模式差异巨大，可以通过不同 Expert 来特化学习
2. **动作空间多样性**：Click/Type/Scroll/Drag 等不同动作类型可以由专门的 Expert 处理
3. **成本敏感**：GUI Agent 需要频繁调用（每个步骤一次推理），推理成本直接影响实用性

### 2.3 数据构造流水线：从 1.66M 到 111K 的质量革命

#### Grounding 数据：人工精修的必要性

团队聚合了 6 个公开 GUI grounding 数据集（Aguvis、UI RefExp、Widget Captioning、SeeClick、Uground、OS-Atlas），原始规模约 **166 万条样本**。但经过分析发现：

- **~40% 样本存在实质噪声**：边界框偏移、文本描述模糊、图像模糊
- 噪声来源主要是 HTML/A11y Tree 自动提取的**渲染偏移问题**
- 完全自动化的过滤方法无法可靠识别所有低质量样本

解决方案是一个**三阶段人工精修流程**：
1. 规则过滤：去除冗余和过于简单的样本 → 300K
2. 人工重新对齐：修正偏移的边界框、重写模糊的文本描述
3. 严格筛选：去除模糊图片和固有歧义指令 → **111K 高质量样本**

**从 166 万到 11.1 万——93% 的数据被丢弃。** 这个数字本身就说明了团队对数据质量的极致追求。这印证了 Andrej Karpathy 的名言："Neural nets are just compilation targets; your dataset is the program."

#### Navigation 数据：三层合成框架

Navigation 数据的构造更加复杂，团队提出了一个**层次化合成框架**，整合三个互补的数据源：

**第一层：开源数据审计**
- 来源：AGUVIS（包含 AITW、Mind2Web 等）
- 两阶段质量控制：规则过滤（去除 <3 步的短轨迹、冗余动作）→ MLLM 轨迹审核（验证任务完成度和逻辑一致性）

**第二层：自动轨迹合成（核心创新）**

这是 OmegaUse 最具创新性的数据工程贡献。团队设计了两种互补的自动轨迹生成策略：

```
┌─ Bottom-up（自底向上探索驱动）────────────────────────┐
│                                                        │
│  1. DFS 策略探索单个 App                                 │
│     → 收集原始交互三元组 <state, action, next_state>     │
│                                                        │
│  2. 构建 UI 状态转移图                                   │
│     → MLLM 语义聚类合并冗余状态（压缩图规模）              │
│                                                        │
│  3. 环避策略提取轨迹（避免循环）                           │
│                                                        │
│  4. MLLM 语义丰富化                                      │
│     → 将每条三元组转化为自然语言动作描述                    │
│     → 将整个序列抽象为高层任务目标                         │
│                                                        │
└────────────────────────────────────────────────────────┘

┌─ Top-down（自顶向下分类法引导）────────────────────────┐
│                                                        │
│  1. 设计分层任务分类法（Taxonomy）                       │
│     桌面端: 文件管理 → 系统设置 → 办公软件 → ...          │
│     移动端: 社交 → 购物 → 出行 → 生活服务 → ...           │
│                                                        │
│  2. 基于 Taxonomy 生成任务描述                            │
│                                                        │
│  3. 专家模型在模拟环境中执行                               │
│     → 记录成功轨迹                                       │
│                                                        │
│  4. Human-in-the-Loop 验证平台                            │
│     → 确保标注可靠性 + 收集诊断性失败案例                  │
│                                                        │
└────────────────────────────────────────────────────────┘
```

**Bottom-up** 保证了对单个 App 内部操作空间的**充分覆盖**——DFS 探索能发现很多人类标注者不会主动尝试的操作路径。**Top-down** 保证了任务分布的**结构均衡**——基于分类法的生成确保了各类任务都有足够样本，避免长尾任务缺失。

两者的结合形成了一个完整的数据飞轮：Bottom-up 提供"能做什么"的知识，Top-down 提供"应该做什么"的结构。

**第三层：跨终端专家演示**
- LLM 基于分类法生成 ≥5 步的任务指令
- 专业人工标注员在模拟环境中逐步执行
- 两级质量保证协议确保最高可靠性

### 2.4 统一动作空间（Unified Action Space）

OmegaUse 定义了一套跨平台的统一动作原语：

| 基础动作 | 适用平台 | 说明 |
|---------|---------|------|
| `Click` | 全平台 | 点击指定坐标 |
| `Drag` | 全平台 | 拖拽 |
| `Scroll` | 全平台 | 滚动 |
| `Type` | 全平台 | 输入文本 |
| `Wait` | 全平台 | 等待页面加载 |
| `Finished` | 全平台 | 任务完成 |
| `Hotkey` | Desktop | 快捷键 |
| `LeftDouble` / `RightSingle` / `Hover` | Desktop | 鼠标高级操作 |
| `LongPress` / `PressBack` / `PressHome` / `PressEnter` | Mobile | 移动端手势 |
| `Stop` | Web | 停止加载 |

这套统一动作空间是实现**跨终端泛化**的基础设施——相同的模型可以用同一套动作词汇与不同平台交互，只需在输入端提供对应平台的截图即可。

## 三、技术亮点

### 亮点一：SFT + GRPO 解耦两阶段训练 —— 让感知和规划各得其所

**原理说明：**

OmegaUse 的训练分为两个阶段，分别针对 Grounding 和 Navigation 两个模型，但共享同一个两阶段范式：

**Stage 1 — SFT（监督微调）：建立基础交互语法**

对于 Grounding 模型，SFT 的目标是让模型学会：
- 理解自然语言指令（"点击右上角的设置按钮"）
- 输出标准格式的坐标 `[xmin, ymin, xmax, ymax]`
- 适应不同分辨率和布局的截图

对于 Navigation 模型，SFT 的目标是让模型学会：
- 理解高层任务目标（"在微信中给张三发一张图片"）
- 输出结构化的 `(Observation, Thought, Action)` 三元组
- 掌握跨平台动作序列的基本模式

**Stage 2 — GRPO（群组相对策略优化）：强化精炼**

GRPO 是 DeepSeek 团队提出的强化学习算法（首次用于 DeepSeek-Math/R1），OmegaUse 首次将其系统性地引入 GUI Agent 训练。GRPO 的核心优势在于**不需要单独的 Critic 模型**：

```
传统 PPO 的痛点:
需要训练一个 Critic 模型来估计状态价值 V(s)
→ Critic 本身难以训练准确
→ 在高维视觉状态空间（截图）中尤其困难
→ 训练不稳定、超参数敏感

GRPO 的巧妙之处:
对同一个 prompt q，采样 G 个输出 {o₁, o₂, ..., o_G}
用组内相对奖励作为优势函数:

  Âᵢ = (rᵢ - mean(r₁...r_G)) / std(r₁...r_G)

→ 不需要 Critic！组内均值就是 baseline
→ 计算开销大幅降低
→ 训练稳定性显著提升
```

**针对 GUI 任务的特殊奖励设计：**

Grounding 模型使用 **IoBB Reward（Inside-of-Bounding-Box Reward）**：
```
R = R_fmt × w₁ + R_pos × w₂

R_pos = 1  如果预测中心点落在 GT 边界框内
R_pos = 0  否则
```
这个设计的直觉是：**鼓励模型预测目标的"内部区域"而非模糊的边界像素**。之前的 grounding 方法经常在边界框边缘产生不确定的预测，IoBB Reward 直接惩罚了这种行为。

Navigation 模型使用 **Stepwise Coordinate-based Reward**：
```
R_act = R_type × w_type + R_coord × w_coord

R_type: 动作类型是否正确（二值）
R_coord: 坐标精度（基于 Δx, Δy 的分级奖励）
  = 1.0   if Δx,Δy < θ₁（精确命中）
  = 0.5   if θ₁ ≤ Δx,Δy < θ₂（近似命中）
  = 0.0   otherwise（偏离过大）
```
分级的坐标奖励比单纯的 binary reward 提供了更细粒度的监督信号——"接近正确"也应该得到部分奖励，这在 RL 训练初期尤为重要。

### 亮点二：Bottom-up + Top-down 双向轨迹合成 —— 解决 GUI Agent 的"数据饥荒"

**原理说明：**

GUI Agent 训练面临一个根本性的**数据获取悖论**：

- **人工标注太贵**：一条高质量的多步 GUI 操作轨迹需要专业标注员花 10-30 分钟录制
- **自动录制数据质量差**：用户录屏数据充满噪声（误操作、犹豫、回退）
- **模拟环境有限**：不是所有 App 都能在模拟器中完美运行

OmegaUse 的双向合成框架是这个悖论的一个优雅解答：

**Bottom-up（DFS 探索）的价值在于"广度"：**
- DFS 策略会系统地探索 App 的每一个可达状态
- 人类标注者倾向于走"最短路径"，但 DFS 会发现**所有可能的路径**
- 这些"非直觉路径"的数据增加了模型的鲁棒性——当常规路径受阻时，Agent 知道替代方案

**Top-down（分类法引导）的价值在于"结构"：**
- 基于 Taxonomy 的生成保证了任务类型的**均匀覆盖**
- 纯粹的 Bottom-up 探索会导致常见操作（如首页滑动）过采样，而罕见操作（如深层数据导出）欠采样
- Top-down 的结构化生成弥补了这个偏差

**MLLM 语义聚类的关键作用：**
DFS 探索产生的状态转移图通常非常庞大（一个中型 App 可能有数千个 UI 状态）。直接在这个图上提取轨迹既昂贵又低效。OmegaUse 用 MLLM 对 UI 状态进行**语义聚类**——将视觉上不同但功能上等价的界面状态合并。例如，聊天列表的第 1 页和第 2 页在视觉上完全不同，但功能上都属于"浏览消息列表"状态。聚类后图的规模可缩小 10-100 倍，使轨迹提取变得可行。

### 亮点三：OS-Nav 评测基准 —— 填补中文生态与桌面环境的评估空白

**原理说明：**

现有 GUI Agent 基准存在明显的地理和平台偏向：

| 基准 | 平台 | 语言 | 局限性 |
|------|------|------|--------|
| AndroidControl | Android | 英文 | 仅英文 App |
| AndroidWorld | Android | 英文 | 仅英文 App |
| ScreenSpot 系列 | 跨平台 | 英文 | 仅 grounding，无导航 |
| AITW | Web/Mobile | 英文 | 轨迹质量参差不齐 |

**完全没有**针对中文 App 生态（微信、支付宝、抖音、美团等占据中国用户 90%+ 屏幕时间的应用）和 Linux 桌面环境的标准化基准。

OmegaUse 发布的 OS-Nav 包含两个子基准：

**ChiM-Nav（Chinese Mobile Navigation）：**
- 142 条轨迹，覆盖 69 个中文 Android 应用
- 991 个操作步骤
- 聚焦中国人的日常使用场景：社交、支付、出行、外卖、生活服务
- 每条轨迹都包含**专家验证的推理链**（CoT 描述）
- 这是目前首个系统化的中文 GUI Agent 评测基准

**Ubu-Nav（Ubuntu Desktop Navigation）：**
- 101 条轨迹，641 个操作步骤
- 聚焦 Linux 桌面的日常交互：文件管理、终端操作、系统设置、软件安装
- 对于开发者/服务器运维场景的 GUI Agent 评估具有独特价值

OS-Nav 的意义不仅在于提供了一个新 benchmark，更在于它揭示了一个事实：**当前 SOTA GUI Agent 在中文环境和 Linux 桌面上的能力可能远低于其在英文 Android 上的表现**。这为后续研究指明了重要的改进方向。

### 5.2 核心差异分析

**vs UI-TARS-2：** UI-TARS-2（阿里团队）是 OmegaUse 最直接的竞争对手。两者都是端到端原生 GUI Agent，都采用 RL 后训练，都在多个基准上达到 SOTA。关键区别在于：(1) OmegaUse 采用 MoE 而 UI-TARS-2 使用 72B Dense 模型——OmegaUse 推理更高效；(2) OmegaUse 解耦了 Grounding 和 Navigation，UI-TARS-2 是统一模型；(3) OmegaUse 覆盖桌面和中文环境，UI-TARS-2 聚焦移动/Web。

**vs AutoGLM：**AutoGLM（智谱团队）首创了在线课程强化学习（Online Curriculum RL）框架，让 Agent 在真实环境中自我进化。OmegaUse 的离线 GRPO 训练更稳定可控，但缺少 AutoGLM 的"持续自我改进"能力。两者在技术路线上互补大于竞争。

**vs SeeClick：** SeeClick 是 GUI Agent 领域的开创性工作之一，首次系统性地将 UI grounding 作为独立模块研究。OmegaUse-G 在 SeeClick 奠基的工作上将精度从 ~82% 推进到 96.3%，提升了近 15 个百分点——这 14.3 个百分点的差距几乎全部来自于**数据质量**（人工精修 111K vs 原始噪声数据）和**GRPO 强化训练**。

**vs Cradle：** Cradle 采用模块化框架设计（感知、规划、记忆、执行分离），强调反思和自纠错能力。它的优势在于可解释性和可控性，但模块间的误差累积导致端到端性能不如原生 Agent。OmegaUse 用端到端训练换取了更高的原始性能，但在可解释性和安全性方面不如 Cradle。

## 六、对 OpenClaw / 我们的实际启发

### 启发一：Browser Tool 的 Grounding 能力可以借鉴 OmegaUse-G 的训练思路

OpenClaw 的 browser 工具依赖于 snapshot（DOM 快照）+ action（点击/输入）的交互模式。当前的 element 定位主要依靠 Playwright 的内置选择器（CSS selector / XPath / aria ref）。如果遇到动态渲染、Canvas 或复杂 WebApp，内置选择器经常失效。

借鉴 OmegaUse-G 的思路：
- 可以收集 browser 工具的历史 screenshot + action 对作为训练数据
- 训练一个轻量的 grounding 模型，将自然语言描述（"点击提交按钮"）映射到屏幕坐标
- 当 DOM 选择器失效时，fallback 到视觉 grounding 模型
- 这本质上是为 OpenClaw 的 browser 工具增加了一道**视觉保底机制**

### 启发二：解耦思想用于 Sub-agent 的专业化分工

当前 OpenClaw 的 sub-agent 基本是"万能选手"——同一个 sub-agent 既做搜索分析，又做文档撰写，还做代码审查。OmegaUse 的 Grounding/Navigation 解耦启示我们：

- **感知型 sub-agent**：擅长信息抽取、网页解析、图像识别（类比 OmegaUse-G）
- **规划型 sub-agent**：擅长任务分解、决策推理、多步规划（类比 OmegaUse-N）
- 不同类型的 sub-agent 用不同的 system prompt 甚至不同的模型来驱动
- 主 agent 负责"路由"——判断当前步骤需要感知还是规划，派发给对应的 specialist

这比当前的"一个 sub-agent 干所有事"模式更专业、更高效。

### 启发三：数据质量 > 数据数量的实战教训

OmegaUse 从 166 万条 grounding 数据精简到 11.1 万条（丢弃 93%），最终效果反而远超使用全量数据的基线。这对我们的技能系统（Skills）和记忆系统有直接启示：

- **MEMORY.md 的条目质量比数量重要**：10 条精准的记忆比 100 条模糊的条目更有用
- **Skill 的 SKILL.md 应该精炼而非冗长**：一个清晰、准确的 skill 指令胜过十页泛泛而谈的"最佳实践"
- **心跳整理时的去重和压缩**应该更激进——保留高信息密度的内容，丢弃冗余

### 启发四：GRPO 训练范式对 Agent 自我改进的参考价值

OmegaUse 用 GRPO（无需 Critic 的 group-relative RL）成功地将 GUI Agent 的性能推到了新高度。这个范式对我们的恋爱模拟器和 solo-leveling 系统也有启发：

- 当前这些系统的数值更新是基于**预定义规则**的（事件 X → 属性 Y +N）
- 如果引入一个轻量的 GRPO 式机制：定义"好的行为"的 reward signal（如学长的积极反馈），让系统的参数/策略通过相对奖励来微调
- 就可以实现**数据驱动的自适应演化**，而不是完全依赖人工设定的规则表
- 当然，这需要在 rule-based 和 full RL 之间找到平衡点

### 启发五：OS-Nav 式的自建基准思维

OmegaUse 团队在发现现有基准无法满足需求时，选择了**自己构建**而非将就。这种"缺什么补什么"的工程态度值得学习：

- OpenClaw 的 sub-agent 执行效果目前缺乏系统性的评测基准
- 可以考虑构建一套标准的 agent task benchmark（如：代码生成准确性、文档撰写质量、搜索完整性等）
- 用统一的 benchmark 来衡量不同配置/模型/prompt 下的 agent 表现
- 这比凭感觉调优要科学得多

## 总结评价

OmegaUse 是 2026 年初 GUI Agent 领域**最具工程深度和全面性的工作之一**。它没有停留在单一的技术创新点上，而是系统性地回答了"如何构建一个真正通用的 GUI Agent"这个全栈问题——从数据工程到模型架构，从训练范式到评测基准，提供了完整的解决方案。

**核心优势：**
- **MoE 骨干的前瞻性选择**：在大多数竞品仍在用 Dense 模型时，OmegaUse 选择了 MoE，实现了大模型推理能力与小模型推理效率的兼得
- **Grounding/Navigation 解耦架构**：首次系统性地验证了感知与规划分离在 GUI Agent 中的价值，避免了梯度干扰
- **数据工程质量极高**：93% 的 grounding 数据被丢弃、双向轨迹合成框架、人工精修——这种对数据质量的极致追求是性能突破的根本原因
- **GRPO 训练的成功迁移**：将 DeepSeek-R1 验证的 GRPO 算法创造性地引入 GUI Agent 领域，用专门设计的 IoBB Reward 和 Stepwise Coordinate Reward 取得了显著效果
- **跨平台覆盖独一无二**：同时覆盖 Mobile + Desktop + Web + 中文生态 + Linux，是目前覆盖面最广的 GUI Agent
- **OS-Nav 基准的社区贡献**：填补了中文 GUI Agent 和 Linux 桌面 Agent 的评测空白

**不足之处：**
- **模型权重未完全开源**：目前主要开源的是数据集（OS-Nav），模型本身的开放程度有限，社区复现门槛较高
- **在线/实时评测结果弱于离线**：AndroidWorld 在线评测 55.7% 虽然领先，但绝对水平说明端到端 GUI Agent 在真实环境中仍有很大提升空间
- **MoE 的训练复杂性**：MoE 模型的训练和部署比 Dense 模型更复杂，对基础设施要求更高
- **缺少安全机制讨论**：论文未深入探讨 GUI Agent 的安全问题（如误操作、权限滥用等），这在实际部署中至关重要
- **长期运行稳定性未验证**：所有评测都是单次任务执行，缺少长时间运行的稳定性数据

**评分：8.8/10** — 扣分主要在开源程度和安全考量上。但在**GUI Agent 的系统工程**层面，OmegaUse 展示了从数据到模型到评测的全栈方法论，是目前最全面的参考实现。

**一句话总结：OmegaUse 证明了构建通用 GUI Agent 的关键不在于堆模型大小，而在于"高质量数据 + 解耦训练 + MoE 效率"的三位一体——它可能不是第一个 GUI Agent，但它是最像一个"真正能用起来"的 GUI Agent。**
