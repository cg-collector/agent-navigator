## 2. 项目简介

**mcp-agent** 是一个简洁、可组合的框架，用于基于 **Model Context Protocol（MCP）** 构建高效的 AI Agent。

它的核心哲学是：

> **"MCP is all you need to build agents, and simple patterns are more robust than complex architectures."**

也就是说，与其设计复杂的 Agent 内部架构，不如让 Agent 通过标准化的 MCP 协议与各种工具/数据源连接——简单即可靠。

---

## 3. 核心特性

### 🔌 完整的 MCP 支持
- 全面实现 MCP 协议（Tools、Resources、Prompts、Notifications）
- 高级功能：OAuth、Sampling、Elicitation、Roots
- 自动管理 MCP Server 的生命周期，开发者不需要操心连接管理

### 🧩 可组合的工作流模式
基于 Anthropic 的 [Building Effective Agents](https://www.anthropic.com/engineering/building-effective-agents) 论文，实现以下模式的**可组合版本**：
- **Map-Reduce**：并行处理后汇总
- **Orchestrator**：动态协调子任务
- **Evaluator-Optimizer**：评估结果并优化
- **Router**：智能路由到不同处理路径

### ⚡ 最小化示例
```python
import asyncio
from mcp_agent.app import MCPApp
from mcp_agent.agents.agent import Agent
from mcp_agent.workflows.llm.augmented_llm_openai import OpenAIAugmentedLLM

app = MCPApp(name="hello_world")

async def main():
    async with app.run():
        agent = Agent(
            name="finder",
            instruction="Use filesystem and fetch to answer questions.",
            server_names=["filesystem", "fetch"],
        )
        async with agent:
            llm = await agent.attach_llm(OpenAIAugmentedLLM)
            answer = await llm.generate_str("Summarize README.md in two sentences.")
            print(answer)

if __name__ == "__main__":
    asyncio.run(main())
```
就这么几行，就可以让 Agent 调用文件系统工具和网络抓取工具了。

### 🕐 Durable Agent（持久化执行）
通过集成 **Temporal** 实现：
- Agent 可以暂停、恢复、重试
- 不丢失执行状态
- 适合长时任务和复杂工作流
- **零 API 改动**即可从开发模式切换到生产模式

### ☁️ 云端部署
- 支持自部署或使用 `mcp-c` 托管运行时
- 所有 App 都以 MCP Server 形式部署
- 可一键部署到云端

## 4. 架构设计

```
┌─────────────────────────────────────────────┐
│              MCPApp (入口)                   │
├─────────────────────────────────────────────┤
│              Agent (Agent)                   │
│  ┌──────────────────────────────────────┐   │
│  │      Augmented LLM (OpenAI/Anthropic) │   │
│  └──────────────────────────────────────┘   │
├─────────────────────────────────────────────┤
│         Workflows (工作流模式)                │
│  map_reduce / orchestrator / evaluator...   │
├─────────────────────────────────────────────┤
│    MCP Server 连接管理层（自动管理生命周期）     │
├─────────────────────────────────────────────┤
│  MCP Servers (filesystem / fetch / etc.)    │
└─────────────────────────────────────────────┘
```

**关键架构决策：**
1. **MCP 协议作为核心抽象**：不重复造轮子，接入 MCP 生态
2. **简单模式优先**：不追求复杂架构，用可组合的小模式解决大问题
3. **Temporal 做状态持久化**：让 Agent 可以中断恢复，适合生产环境

## 6. 对楠楠/学长的启发 🌟

### 🎯 核心收获
**"简单即强大"** —— mcp-agent 没有发明什么花哨的复杂架构，而是站在 MCP 协议这个巨人的肩膀上，通过几个经典工作流模式的可组合，就做出了一个好用、可生产部署的 Agent 框架。

### 💡 可以借鉴的地方

1. **MCP 协议是趋势**：学长在做 OpenClaw 开发时，可以考虑让 OpenClaw 也支持 MCP 协议，这样就能无缝接入 MCP 生态里的所有工具（文件系统、fetch、GitHub 等）。

2. **Durable Agent 的思路**：OpenClaw 的 cron/heartbeat 机制其实也有类似的"持久化任务"需求，可以借鉴 Temporal 的设计思路，让长时间运行的任务（如学长跑代码分析）有更好的中断恢复能力。

3. **"Agent as MCP Server"**：这个设计很有意思 —— 把 Agent 本身暴露为一个 MCP Server，供其他系统调用。OpenClaw 其实已经在做类似的事情（作为 Gateway 被各种 channel 调用）。

### 📝 楠楠的笔记
MCP（Model Context Protocol）就像是 AI 世界的"USB-C 接口"——有了它，Agent 连接什么工具都一样简单。**标准化才是生态的王道**！楠楠以后给自己的 Agent 加能力时，优先找有没有现成的 MCP Server，而不是自己从头写。
