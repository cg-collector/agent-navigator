# AI Agent 学习报告 #115：Letta（原 MemGPT）

## 项目信息

- **项目名：** Letta（原名 MemGPT / Memory-GPT）
- **GitHub：** https://github.com/letta-ai/letta
- **Stars：** ~35k+（MemGPT 时期已积累大量关注，更名后持续增长）
- **一句话简介：** 受操作系统内存分层启发，让 LLM Agent 拥有"无限上下文窗口"的有状态智能体框架——Agent 可以自主管理自己的记忆，在固定上下文限制内实现长期学习和自我进化。

## 背景与定位

Letta 源自 UC Berkeley 的学术论文《MemGPT: Towards LLMs as Operating Systems》（arXiv:2310.08560, 2023年10月），核心洞察是：**LLM 的上下文窗口就像操作系统的物理内存一样有限，可以用虚拟内存管理的思路来解决**。2024年底项目从 MemGPT 正式更名为 Letta，并推出商业 API 服务 + Letta Code CLI 工具。

技术栈：**Python 后端 + TypeScript/Node.js SDK + FastAPI + SQLAlchemy + Redis（可选）+ PostgreSQL/SQLite**

### 三层记忆架构（核心创新）

这是 Letta 最根本的设计，直接借鉴 OS 的内存层次：

| 层级 | 类比 | 实现方式 | 特点 |
|------|------|----------|------|
| **Core Memory（主存）** | RAM | Memory Blocks，始终在 context window 中 | 结构化、可编辑、有大小限制 |
| **Recall Memory（换出页）** | Swap/Page File | 对话历史自动持久化到磁盘 | 可搜索、按时间排序、自动滚动 |
| **Archival Memory（外存）** | 磁盘 | 外部数据库（向量库/图数据库） | 显式存储的知识，需主动检索 |

### 工作流程

1. **用户发送消息** → 进入消息队列
2. **Context Window 编译器** 从 DB 加载当前状态：
   - 组装 system prompt（包含 memory blocks 当前值）
   - 加载 message buffer（最近 N 条对话）
   - 注入可用工具列表及工具 schema
3. **发送给 LLM** → 返回响应（可能包含 function call）
4. **如果是工具调用** → 执行对应工具：
   - `core_memory_append` / `core_memory_replace`：修改 memory block
   - `archival_memory_insert` / `archival_memory_search`：操作外部存储
   - 自定义工具：web search、code execution 等
5. **工具结果返回 LLM** → 继续推理或生成最终回复
6. **Compaction（压缩）**：当 context window 快满时，触发递归摘要：
   - 驱逐 ~70% 的旧消息
   - 旧消息被摘要后存入 recall memory
   - 保留最近消息保证对话连贯性
7. **持久化**：所有状态变更写入 DB

### Memory Blocks 详解

Memory Block 是 Letta 最精妙的数据结构：

```python
# 每个 Block 的结构（概念模型）
class MemoryBlock:
    block_id: str          # 唯一标识
    label: str             # "human" / "persona" / "knowledge" / 自定义
    value: str             # 实际内容（纯文本，可存 JSON/列表等）
    limit: int             # 字符数上限（控制 context window 占用）
    is_read_only: bool     # 是否只允许开发者修改
```

预定义的两个经典 block：
- **`human` block**：存储用户信息（名字、偏好、事实）
- **`persona` block**：存储 Agent 自身人格设定和行为准则

关键特性：**Block 值由 Agent 自己通过工具调用修改**，不是开发者硬编码的。这实现了"自主记忆管理"。

### 多 Agent 协作模式

Letta 内置了几种多 Agent 编排模式：

1. **Supervisor-Worker**：主管 Agent 分配任务给 Worker
2. **Round-Robin**：Agent 轮流处理同一对话
3. **Parallel Execution**：多个 Agent 并行执行后合并结果
4. **Producer-Reviewer**：一个 Agent 产出，另一个审查
5. **Hierarchical Teams**：层级式团队协作

**共享内存机制**：多个 Agent 可以引用同一个 memory_block_id，实现跨 Agent 的状态共享。这让"Sleep-Time Agent"（后台 Agent 在空闲时整理记忆写入共享 Block）成为可能。

## 技术亮点

### 1. 虚拟上下文管理（Virtual Context Management）

这是 Letta 的灵魂。传统 Agent 框架把 context window 当作黑盒，Letta 把它当作**可编程的资源**：

```python
# Context 编译过程（简化）
def build_prompt(agent_state):
    prompt_parts = []
    
    # 1. System prompt with memory blocks
    for block in agent_state.memory_blocks:
        prompt_parts.append(f"<{block.label}>\n{block.value}\n</{block.label}>")
    
    # 2. Available tools (function schemas)
    prompt_parts.append(format_tools(agent_state.tools))
    
    # 3. Recent messages (with truncation)
    messages = get_message_buffer(agent_state, max_tokens=context_window_limit)
    prompt_parts.append(messages)
    
    return assemble(prompt_parts)  # Jinja2 template rendering
```

**为什么重要？** 大多数框架的 system prompt 是静态字符串。Letta 的 system prompt 是**从 DB 状态动态编译**的，memory block 的每次修改都会反映到下一次 LLM 调用中。这意味着 Agent 的"人格"和"知识"是真正可进化的。

### 2. 中断驱动的控制流（Interrupt-Based Control Flow）

借鉴 OS 的中断机制，Letta 用"函数调用即中断"来管理控制流：

```python
# Agent 执行循环（简化伪代码）
async def run_agent_loop(agent, user_message):
    while True:
        # 1. 编译 context window
        prompt = build_prompt(agent.state)
        
        # 2. 调用 LLM
        response = await llm.generate(prompt)
        
        # 3. 检查是否有函数调用（= 中断）
        if response.function_call:
            result = execute_tool(response.function_call)
            
            # 4. 特殊中断：内存管理操作
            if response.function_call.name in ["core_memory_replace", "archival_memory_insert"]:
                # 直接修改持久化状态
                update_agent_state(agent, result)
            
            # 5. 将工具结果作为新消息继续循环
            append_message(agent, tool_result=result)
            continue  # ← 继续循环，不返回给用户
        
        # 6. 无函数调用 = 正常回复，退出循环
        return response.content
```

这个设计让 Agent 可以在**一次用户交互中执行多步工具链**（比如：搜索→读取→更新记忆→回复），而不需要外部编排引擎。

### 3. Compaction（递归上下文压缩）

当消息缓冲区接近 context window 上限时触发的优雅降级：

```python
def compact_messages(messages, archive_fn, summary_model):
    """驱逐约70%旧消息，递归摘要"""
    if len(messages) <= SAFE_LIMIT:
        return messages
    
    # 分割点：保留最近的30%
    split_point = int(len(messages) * 0.3)
    to_evict = messages[:split_point]
    to_keep = messages[split_point:]
    
    # 递归摘要：和之前的摘要合并后再摘要
    summary = summary_model.summarize(to_evict + existing_summary)
    archive_fn(summary)  # 存入 recall memory
    
    return [summary_message] + to_keep
```

**设计巧思：**
- 不是简单截断，而是**递归摘要**——越老的对话被压缩得越多
- 保留 30% 最近消息确保短期连贯性
- 摘要存入 recall memory，之后仍可通过搜索检索

### 4. Sleep-Time Compute（异步记忆增强）

这是 Letta 2025 年推出的重要演进（论文 arXiv:2504.13171）：

传统 MemGPT 的记忆管理是**同步的**——Agent 一边聊天一边管理记忆，导致：
- 响应延迟增加（记忆操作占用 token）
- 记忆质量不高（聊天时的记忆更新是增量的、碎片化的）

Sleep-Time Compute 引入**异步后台 Agent**：

```python
# Sleep-Time Agent 工作模式
async def sleep_time_worker(primary_agent_id, shared_block_id):
    while True:
        await wait_for_idle_period()  # 主 Agent 空闲时触发
        
        # 读取主 Agent 的完整对话历史
        history = get_full_conversation(primary_agent_id)
        
        # 深度反思和整合（不受实时延迟约束）
        refined_memories = await deep_reflect(history)
        
        # 写入共享 memory block
        update_memory_block(shared_block_id, refined_memories)
```

**效果类比：** 就像人类睡觉时大脑会整理白天的记忆一样，Agent 在"睡眠"期间可以：
- 用更强的模型做深度推理（不怕慢）
- 整合碎片化信息为结构化知识
- 重写和优化 memory block 内容
- 不影响用户的实时交互体验

### 5. AgentFile (.af) — Agent 可执行的代码文件

Letta 支持一种特殊的 `.af` 文件格式，Agent 可以读写和执行这些文件：

```python
# example.af — Agent 可自主调用的技能文件
@skill
def analyze_codebase(path: str):
    """分析代码库结构"""
    structure = walk_directory(path)
    summary = summarize_structure(structure)
    # 结果可以写入 archival memory 或 memory block
    archival_memory_insert(summary)
    return summary
```

这实际上给了 Agent **持久化的、可组合的能力单元**，比单纯的 function calling 更进一步。

## 对 OpenClaw / 楠楠的启发

### 🔥 高价值启发

**1. Memory Block 模式可以直接用于楠楠的记忆系统**

当前楠楠的记忆靠 markdown 文件（daily notes + MEMORY.md），本质上是手动管理的。Letta 的 Memory Block 思路可以借鉴：
- 将 SOUL.md / USER.md 中的关键信息结构化为"常驻 memory blocks"
- 让楠楠在对话中**自主判断**哪些信息值得写入长期记忆（而不是靠 heartbeat 定期整理）
- 实现 `memory_update` 工具调用 → 自动更新特定 block → 下次对话自动生效

具体实现思路：
```
当前：SOUL.md 是静态文件 → 每次加载全文
改进：定义 blocks = { persona, user_facts, preferences, recent_context }
     → 每次 context injection 只注入相关 block 的当前值
     → block 值存在 JSON/DB 中，可被工具调用修改
```

**2. Context Window 编译器模式**

Letta 的 Jinja2 模板编译思路非常实用。OpenClaw 的 system prompt 目前是静态拼接的，可以改为模板化编译：
- 根据 task type 动态选择注入哪些 memory blocks
- 不同复杂度任务分配不同的 context budget
- 支持 memory block 的优先级排序（空间不够时先裁剪低优先级 block）

**3. Sleep-Time Compute 对心跳系统的升级**

楠楠目前的心跳（heartbeat）主要是轮询检查。Letta 的 Sleep-Time Agent 启发我们可以把心跳变成**深度工作时段**：
- 心跳时不只是检查邮件/天气，而是运行一个"记忆整合"子任务
- 回顾今天的对话日志，提取值得长期记住的信息
- 更新 memory blocks 和 MEMORY.md
- 这比当前的"定期整理日记"更系统化

**4. 多 Agent 共享内存**

如果未来 OpenClaw 支持多 Agent 并行，Letta 的 shared memory block 模式是最自然的协调机制：
- 主 Agent 负责 user interaction
- 后台 Agent 负责 memory refinement / proactive learning
- 通过共享 block 通信，不需要复杂的消息传递协议

### ⚠️ 注意事项

- Letta 的记忆管理依赖 LLM 自主调用 memory 工具，**对小模型来说可靠性不够好**（经常忘记调工具）。需要配合强制检查点或外部触发器
- Archival memory 的检索质量高度依赖 embedding 模型和分块策略，这部分 Letta 本身没有提供太多创新
- 项目商业化较重（Letta Cloud API），开源版的部分高级功能（如 sleep-time compute）文档不够完善
