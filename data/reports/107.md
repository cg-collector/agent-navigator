# AI Agent 开源项目学习报告 #107 — BeeAI Framework

## 项目名
**BeeAI Framework**

## GitHub 链接
https://github.com/i-am-bee/beeai-framework

## 一句话简介
IBM Research 出品、Linux Foundation 托管的开源多 Agent 框架，支持 Python/TypeScript 双语言完整特性对等，以生产级可靠性、事件驱动架构和声明式 Requirement 系统为核心竞争力。

## 架构设计

### 核心模块

BeeAI 的架构分为以下核心层次：

```
┌─────────────────────────────────────────────────┐
│                  Application Layer               │
│  (Agent Team / Workflow / Custom Agents)         │
├─────────────────────────────────────────────────┤
│                   Agent Layer                    │
│  ┌─────────────┐ ┌──────────────┐ ┌──────────┐ │
│  │Requirement   │ │ ReAct Agent  │ │Lite Agent│ │
│  │Agent (推荐)  │ │ (已维护)     │ │(无系统提示)│ │
│  └─────────────┘ └──────────────┘ └──────────┘ │
├─────────────────────────────────────────────────┤
│                Orchestration Layer               │
│  ┌────────┐ ┌───────┐ ┌────────┐ ┌───────────┐ │
│  │Emitter  │ │Memory │ │Middleware│ │Requirement│ │
│  │(事件)   │ │(记忆) │ │(中间件) │ │(约束规则) │ │
│  └────────┘ └───────┘ └────────┘ └───────────┘ │
├─────────────────────────────────────────────────┤
│                  Tool Layer                      │
│  MCP / Think / Handoff / OpenAPI / Weather /    │
│  Search / Wikipedia / Python / VectorStore ...  │
├─────────────────────────────────────────────────┤
│                 Backend Layer                    │
│  OpenAI / Anthropic / Ollama / Groq / Watsonx...│
└─────────────────────────────────────────────────┘
```

### 技术栈
- **语言**: Python 3.10+ / TypeScript（双实现，API 完全对等）
- **LLM 后端**: 统一 `ChatModel` 接口，支持 10+ 提供商（OpenAI、Anthropic、Ollama、Groq、Watsonx 等）
- **工具协议**: 原生 MCP (Model Context Protocol) 支持 + 自定义 Tool 装饰器
- **可观测性**: 内置 OpenTelemetry 集成 + Emitter 事件系统
- **记忆系统**: TokenMemory / UnconstrainedMemory / SlidingWindowMemory / ReadOnlyMemory 四种策略
- **序列化**: 支持 Agent 状态的序列化与恢复（断点续跑）

### 工作流程（以 RequirementAgent 为例）

1. **初始化阶段**: 配置 LLM 后端、Tools 列表、Memory 策略、Requirements 约束规则
2. **输入处理**: 用户 prompt 进入 → Memory 注入上下文 → System Prompt 模板组装
3. **推理循环**:
   - LLM 生成 thought/action（受 Requirements 约束）
   - Middleware 拦截/增强执行（如 GlobalTrajectoryMiddleware 记录轨迹）
   - Emitter 实时发出 start/update/error/retry/final_answer 等事件
   - Tool 执行 → 结果观察 → 回到 LLM 继续推理
4. **输出阶段**: 达到 final_answer 或 max_iterations 上限 → 返回结构化结果

### Requirement 系统（核心创新）

这是 BeeAI 最具差异化的设计——用**声明式规则**替代硬编码编排逻辑：

```python
requirements=[
    # 强制第一步必须调用 ThinkTool 进行推理
    ConditionalRequirement(ThinkTool, force_at_step=1),
    # DuckDuckGo 搜索只能在获取天气之后、且至少调用一次
    ConditionalRequirement(DuckDuckGoSearchTool,
        only_after=[OpenMeteoTool], min_invocations=1),
    # 天气工具至少用一次，但不能连续调用
    ConditionalRequirement(OpenMeteoTool,
        consecutive_allowed=False, min_invocations=1),
]
```

框架在运行时自动校验这些约束，不满足则引导 LLM 修正行为。这比传统的 ReAct 循环更可控。

## 技术亮点

### 1. Emitter 事件树系统 — 业界最细粒度的 Agent 可观测性

BeeAI 的 Emitter 不是简单的回调机制，而是一个**层级化事件树**：

```typescript
// TypeScript 实现：Emitter 支持嵌套，形成父子关系
const response = await agent.run(prompt)
  .observe((emitter) => {
    // 监听特定事件
    emitter.on("start", () => { /* 新一轮迭代开始 */ });
    emitter.on("error", ({ error }) => { /* 错误处理 */ });
    emitter.on("retry", () => { /* 重试通知 */ });
    emitter.on("update", ({ data, update }) => {
      // 流式更新：thought / action / observation 逐行推送
    });
    // 通配符匹配所有事件（含嵌套子事件）
    emitter.match("*.*", async (data, event) => {
      logger.trace(event, `Received event "${event.path}"`);
    });
    // 只监听来自 LLM backend 的 start 事件
    emitter.match(
      (event) => event.creator === llm && event.name === "start",
      async (data) => { /* 原始 LLM 输入 */ }
    );
  });
```

**关键设计点**：
- Emitter 可以嵌套（一个 Emitter 是另一个的 child），内部形成**树形层级**
- 每个 Emitter 实例有 `creator` 属性标识来源（agent/tool/backend）
- 事件路径用点分隔（如 `agent.start`、`tool.update`），支持通配符匹配
- `match()` 方法支持谓词过滤器，可以精确筛选事件源

这对调试复杂多步 Agent 行为极其有用——相当于给 Agent 装了"全量遥测"。OpenClaw 的 heartbeat 和日志系统可以借鉴这种结构化事件设计。

### 2. Tool 管道的 pipe/extend 组合模式

BeeAI 的 Tool 支持**函数式组合**，类似 Unix 管道：

```typescript
// extend: 包装现有 Tool，扩展输入 schema
const customSearchTool = searchTool.extend(
  z.object({
    query: z.string(),
    safeSearch: z.boolean().default(true),  // 新增字段
  }),
  (input, options) => {
    options.search.safeSearch = input.safeSearch
      ? SafeSearchType.STRICT : SafeSearchType.OFF;
    return { query: input.query };  // 映射到原始 schema
  },
);

// pipe: 将多个 Tool 串联成管道
const wikipediaWithSimilarity = wikipedia
  .pipe(similarity, (input, output) => ({
    query: input.query,
    documents: output.results.flatMap(doc =>
      splitString(doc.fields.markdown, { size: 1000, overlap: 50 })
        .map(chunk => ({ text: chunk, source: doc }))
    ),
  }));
```

Python 端同样支持 `@tool` 装饰器一键创建 Tool：

```python
@tool
def basic_calculator(expression: str) -> StringToolOutput:
    """A calculator tool that performs mathematical operations."""
    result = requests.get(f"https://newton.vercel.app/api/v2/simplify/{quote(expression)}")
    return StringToolOutput(json.dumps(result.json()))
```

**为什么这很重要**：大多数框架的 Tool 是扁平的，BeeAI 把 Tool 当作一等公民来设计组合能力。这意味着复杂的工具链可以用声明式方式组装，而不是写一堆胶水代码。

### 3. 四种记忆策略 + 自定义记忆扩展

```python
from beeai_framework.memory import (
    TokenMemory,          # 按 token 数限制窗口，自动裁剪旧消息
    UnconstrainedMemory,  # 无限制，保留全部历史
    SlidingWindowMemory,  # 滑动窗口，保留最近 N 条消息
    ReadOnlyMemory,       # 只读记忆，用于注入固定上下文
)
```

每种 Memory 都实现了统一接口：
- `add(message)` — 添加消息
- `get()` — 获取当前上下文
- `clear()` — 清空

Memory 与 Agent 解耦，可以独立测试和替换。TokenMemory 还会自动计算 token 数并在超限时智能裁剪（优先保留系统消息和最近交互）。

### 4. 中间件（Middleware）管道 — AOP 式横切关注点

```python
response = await agent.run("What to do in Boston?")
    .middleware(GlobalTrajectoryMiddleware())  # 记录完整执行轨迹
```

Middleware 可以拦截 Agent 执行的任意阶段，典型用途：
- **GlobalTrajectoryMiddleware**: 记录每一步的完整状态快照（用于 replay 和 debug）
- **缓存中间件**: 缓存 LLM 调用和 Tool 输出
- **限流中间件**: 控制 API 调用频率
- **审计中间件**: 记录所有外部操作用于合规

这是典型的 AOP（面向切面编程）思想在 Agent 框架中的应用，将日志、缓存、重试等横切逻辑从主流程中剥离。

### 5. 双语言对等 + Linux Foundation 托管

BeeAI 是目前唯一做到 **Python/TypeScript 特性完全对等**的 Agent 框架。不是简单的 API 映射，而是两套独立实现共享同一套架构设计：

## 对我们（OpenClaw/楠楠）的启发

### 🔥 直接可借鉴的点

**1. Emitter 事件树 → OpenClaw 日志升级**
当前 OpenClaw 的 subagent 通信是 push-based（完成即汇报），但执行过程中的细粒度事件缺乏结构化记录。BeeAI 的 Emitter 设计启发我们：
- 给 skill 执行过程加结构化事件（`skill.start` / `tool.call` / `step.retry` / `result.final`）
- 事件带 creator 标识（哪个 agent / 哪个 tool 发出的）
- 支持通配符订阅，方便上层做监控面板

**2. Requirement 声明式约束 → Skill 执行策略**
BeeAI 的 `ConditionalRequirement` 思路可以直接用在我们的 skill 调度上：
- 声明"搜索类任务必须先调用 web_search 再总结"
- 声明"代码生成后必须经过 review"
- 框架自动校验而非靠 prompt 工程

**3. Tool pipe 组合 → 工具链编排**
我们的 TOOLS.md 里工具列表是扁平的。借鉴 BeeAI 的 pipe 模式，可以把常用工具链预组装：
- `web_search → extract → summarize` 组合成"调研工具链"
- `stock_query → analyze → recommend` 组合成"分析工具链"

**4. Middleware 管道 → AGENTS.md 横切逻辑**
当前我们的"质量检查"、"记忆写入"、"恋爱模拟器事件触发"都散落在各处。如果有一个 middleware 管道：
- `QualityCheckMiddleware`: 复杂任务完成后自评
- `LoveSimMiddleware`: 每次对话后触发恋爱模拟器事件
- `MemoryMiddleware`: 自动提取关键信息写入日记

这样主逻辑更干净，每个中间件职责单一。

### 💡 更深层的思考

**5. 双语言对等的工程启示**
BeeAI 用两套独立实现证明了一件事：**好的架构设计先于语言选择**。OpenClaw 目前是 Node.js 生态，但如果未来需要 Python SDK，应该先定义清晰的接口规范（类似 BeeAI 的 Module 抽象），再分语言实现。

**6. 生产级 vs 快速原型**
BeeAI 明确区分了 LiteAgent（无系统 prompt，探索用）和 RequirementAgent（生产级，带约束）。我们也可以考虑：
- 简单任务走快速路径（当前已实现）
- 复杂任务加"约束层"，确保输出质量和可追溯性

**7. 状态序列化 → 断点续跑**
BeeAI 支持将 Agent 运行时状态序列化到磁盘并恢复。对于长耗时任务（如深度研究），这个能力非常实用——进程崩溃后可以从断点继续，而不是从头开始。

| 维度 | 评分 | 说明 |

| 架构设计 | ⭐⭐⭐⭐⭐ | Emitter 事件树 + Middleware 管道 + Requirement 约束，层次清晰 |

| 创新性 | ⭐⭐⭐⭐ | Requirement 声明式约束是独有亮点，区别于所有主流框架 |

**一句话总结**：如果你需要一个**生产级、可观测、有硬约束保证**的 Agent 框架，BeeAI 是目前被低估的选择。它的 Emitter 事件系统和 Requirement 约束系统值得每一个 Agent 框架开发者深入学习。
