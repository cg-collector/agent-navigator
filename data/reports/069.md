## 项目名称
**MetaGPT**（Multi-Agent Framework）

## GitHub 链接
https://github.com/FoundationAgents/MetaGPT

## 一句话简介

MetaGPT 是一个让大语言模型像真实软件公司一样协作的多智能体框架，核心理念是 **"Code = SOP(Team)"**——把标准操作流程（SOP）具象化，让多个 AI Agent 扮演产品经理、架构师、工程师等角色，共同完成复杂任务。

---

## 详细介绍

MetaGPT 由深度之约（DeepWisdom）团队开发，于 2023 年 8 月提出论文（MetaGPT: Meta Programming for a Multi-Agent Collaborative Framework），并于 2024 年初开源。目前 GitHub 星标超过 4 万，是多智能体框架领域最受关注的开源项目之一。

它的核心思想非常有意思：**把一个软件公司的完整流程压缩到一个 Agent 团队里**。用户只需要输入一行需求（例如"写一个 2048 小游戏"），MetaGPT 就会启动一个虚拟软件公司，里面有产品经理（PM）写需求文档，架构师（Architect）做技术设计，项目经理（PM）排期，工程师（Engineer）写代码，测试工程师（Tester）写测试用例——所有角色通过标准化的消息传递协作，最终产出完整的代码仓库。

2025 年 2 月，MetaGPT 团队还推出了 [MGX（MetaGPT X）](https://mgx.dev/) 产品，被称为"世界上第一个 AI Agent 开发团队"，登上了 Product Hunt #1 Product of the Day。

## 架构设计

### 核心哲学：SOP 驱动

MetaGPT 的核心洞察是：**团队协作之所以有效，是因为有 SOP（Standard Operating Procedures）**。真实软件公司里，产品经理不会直接跳到写代码，架构师也不会跳过设计直接写测试——每一步都依赖前一步的输出，并给下一步提供输入。

MetaGPT 把这个机制引入多 Agent 系统：

```
用户需求 → PM（写需求文档）→ Architect（写技术设计）
         → Engineer（写代码）→ Tester（写测试）
         → Reviewer（评审）→ 产出完整代码
```

每个 Agent 只关注自己的 SOP，观察（watch）上游角色的输出消息，并发布（publish）自己的输出给下游角色。

### 关键组件

1. **Role（角色）**：每个 Agent 是一个 Role，有自己的 profile（角色描述）、actions（可执行的动作）、watch（监听的消息类型）。定义一个 Role 只需约 10 行代码。

2. **Action（动作）**：Role 执行的具体操作，比如 `SimpleWriteCode`、`SimpleWriteTest`。Action 通过 LLM 生成内容，返回结构化的消息。

3. **Team（团队）**：多个 Role 组成一个 Team，共享同一个 Environment。Team 负责初始化所有 Role 并驱动它们交互。

4. **Environment（环境）**：团队共享的消息空间，所有 Role 的输出都发布到这里，其他 Role 通过 watch 机制订阅相关消息。

5. **Message（消息）**：Role 之间传递的信息载体，包含内容、发送者、接收者等元数据。

### 消息驱动的工作流

```
Role A 执行 Action → 发布 Message → Environment 存储
Role B watch([ActionA]) → 检测到上游消息 → 执行自己的 Action → 发布新 Message
...
```

这种消息驱动设计让 Agent 之间的依赖关系清晰可见，也便于调试和扩展。

## 技术亮点

### 1. SOP 具象化（Core Innovation）
MetaGPT 最早系统性地将人类公司的 SOP 引入多 Agent 协作，而不是让 Agent 自由对话。这大大提升了任务分解的可靠性和可解释性。

### 2. 极简的多 Agent 开发体验
只需约 10 行代码就能定义一个 Role 并加入团队：

```python
class SimpleCoder(Role):
    name: str = "Alice"
    profile: str = "SimpleCoder"

    def __init__(self, **kwargs):
        super().__init__(**kwargs)
        self._watch([UserRequirement])        # 监听上游消息
        self.set_actions([SimpleWriteCode])   # 设置可执行动作
```

### 3. 支持软件公司全套流程
内置角色包括：ProductManager、Architect、ProjectManager、Engineer、Tester、Reviewer 等，覆盖从需求到代码的全生命周期。

### 4. Data Interpreter（数据解释器）
除了软件开发场景，MetaGPT 还提供了 Data Interpreter 角色，能自主完成数据分析任务——接收自然语言指令，自动写代码、执行、修复错误、生成图表。

### 5. 丰富的 Use Cases
- **Debate**：多 Agent 辩论
- **Researcher**：AI 研究助手
- **Receipt Assistant**：收据识别助手
- **Software Company**：完整软件开发流程

### 6. 学术认可
MetaGPT 的论文被 ICLR 2025 接收（Oral，top 1.8%），并在 ICLR 发表了 AFlow（自动化 Agent 工作流生成）等后续工作。

## 对楠楠/学长的启发

### 1. SOP 思维值得借鉴 ✨

MetaGPT 最重要的启发是：**好的协作不靠自由发挥，靠 SOP**。不管是楠楠和学姐协作完成项目，还是学长管理多个 Agent 服务，有一个清晰的上下游流程定义，执行效果都会大幅提升。

楠楠以后帮学长做事情，也可以先想清楚：
- 任务的第一步是什么？
- 第二步依赖什么输入？
- 最终交付什么？

先 SOP，后执行，事半功倍～

### 2. 多 Agent 分工比单个 Agent 堆能力更优雅

MetaGPT 证明了：与其让一个超级 Agent 什么都做，不如让多个专业化的小 Agent 协作。这对学长做系统设计也很有启发——分布式、专门化、流水线式的工作分解，往往比单体架构更有韧性和可扩展性。

### 3. "虚拟团队" 模式很有前景

MetaGPT 展示了一个很有前景的方向：**一个 AI Agent 团队可以像一个小公司一样运作**。学长如果做 AI 应用开发，可以考虑类似思路——让不同角色的 Agent 分别负责理解需求、生成代码、执行测试、部署上线，形成全自动流水线。

### 4. 成本可控，值得实验

文档提到，用 GPT-4 生成一个完整项目分析+设计大约 **$0.2**，完整项目大约 **$2.0**。这个成本对于探索性开发来说非常值得。学长可以用 MetaGPT 来快速验证想法、生成原型代码，再手动优化～

### 5. 对楠楠写报告的启发

楠楠和桐桐（也就是我！）其实也可以用类似的思路：先定义好每个角色的职责（我写报告、桐桐跑腿搜集资料），建立清晰的工作流程（SOP），这样协作效率会更高哦～ 💪
