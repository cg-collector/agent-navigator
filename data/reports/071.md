## 一句话简介

OpenAI 官方出品的轻量级 AI Agent 开发 SDK，仅用"Agent + Handoff + Guardrail"三个核心原语，就实现了生产级别的多 Agent 协作编排，内置追踪（Tracing）、安全验证和会话（Sessions）记忆，开箱即用。

---

## 详细介绍

OpenAI Agents SDK 于 **2025 年 3 月**正式发布，是 OpenAI 对其前身 [Swarm](https://github.com/openai/swarm) 实验框架的生产级升级。虽然名字里有"OpenAI"，但 SDK 设计上完全**不绑定 OpenAI 模型**——支持任意兼容 OpenAI API 接口的模型，包括本地模型。

核心设计哲学是：**足够强大，但原语足够少**。整个 SDK 只有三个核心概念：Agent（装备了指令和工具的 LLM）、Handoff（Agent 之间的委托传递）、Guardrail（输入输出的安全验证）。这三者组合起来，足以表达复杂的多 Agent 协作关系。

2025 年底，OpenAI 还推出了 **Realtime Agents**，支持基于 `gpt-realtime-1.5` 的低延迟语音 Agent，支持自动打断检测、上下文管理和安全验证——可以直接构建语音对话 AI。

## 架构设计

### 三大核心原语

```
Agent      = LLM + Instructions（指令）+ Tools（工具）+ Guardrails（安全验证）
Handoff    = Agent → Agent 的委托传递，移交控制权
Guardrail  = 输入/输出验证，安全检查
```

### Agent 结构

每个 Agent 本质上是一个"装备了工具的 LLM"，核心属性：

```python
class Agent:
    name: str                          # Agent 名称
    instructions: str                 # 系统提示词
    tools: list[Tool]                  # 可用工具列表
    guardrails: list[Guardrail]        # 安全验证规则
    handoffs: list[Agent | Handoff]   # 可委托的 Agent
```

### Agent 协作的两种模式

#### 1. Agents as Tools（经理模式）
- 经理 Agent 保持对话控制权
- 通过 `Agent.as_tool()` 调用专家 Agent
- 适合场景：需要汇总多个专家输出、统一对外回复

```python
research_agent = Agent(name="Researcher", ...)
writer_agent = Agent(name="Writer", ...)

manager = Agent(
    name="Manager",
    instructions="协调研究Agent和写作Agent完成任务",
    tools=[research_agent.as_tool(...), writer_agent.as_tool(...)]
)
```

#### 2. Handoffs（路由模式）
- 分诊 Agent 把对话路由给专家 Agent
- 专家 Agent 接管后续对话
- 适合场景：需要专家直接面对用户、保持提示词专注、动态切换角色

```python
triage = Agent(name="Triage", handoffs=[sales_agent, support_agent])
```

**两种模式可以混合使用**：分诊 Agent 可以 Handoff 给专家，专家内部再调用其他 Agent as tool 完成任务。

### Guardrail 机制

Guardrail 是 OpenAI Agents SDK 最有特色的安全设计：

| 类型 | 触发时机 | 用途 |
|------|---------|------|
| Input Guardrail | 用户输入进入 Agent 前 | 内容安全检查、成本保护（拦截滥用） |
| Output Guardrail | Agent 最终输出前 | 敏感信息过滤、格式验证 |
| Tool Guardrail | 每次工具调用前后 | 工具参数验证、执行结果检查 |

支持**并行执行**（低延迟）和**阻塞执行**（省成本）两种模式。

## 技术亮点

### 1. 极简原语，强大表达力 ✨

用三个概念（Agent / Handoff / Guardrail）就能表达复杂的多 Agent 关系：
- 并行专家协作 → 多个 Agent as Tools
- 顺序流水线 → Handoff 链
- 条件分支 → LLM 自主决定 Handoff 目标
- 安全网 → Guardrail 三层防护

这比很多"功能丰富但原语几十个"的框架容易理解得多。

### 2. 内置 Tracing（追踪调试）

Agent 执行链路自动可视化：
- 每个 Agent 的调用、工具执行、消息传递全部记录
- 支持 OpenAI 的 eval/fine-tuning 工具链
- 调试多 Agent 流程就像调试普通代码一样有迹可循

这对生产环境的问题排查非常重要。

### 3. Sessions（会话记忆）

内置持久化记忆层，不需要自己实现：
```python
result = Runner.run_sync(agent, user_input, session_id="user-123")
```
跨轮次维护上下文，适用于多轮对话场景。

### 4. Human in the Loop

内置人工介入机制：
- Agent 可以在关键节点暂停，等人工确认
- 支持中断后从断点恢复执行
- 适合需要人工审批的敏感操作场景

### 5. Realtime Agents（语音 Agent）

2025 年底的重大扩展：
- 基于 `gpt-realtime-1.5` 的低延迟语音对话
- 自动打断检测（用户说话时 Agent 停止）
- 上下文管理 + Guardrail 安全保障
- 可构建语音客服、语音助手等应用

### 6. MCP 原生集成

内置 MCP Server 工具调用支持，MCP 工具和普通 function tool 用法完全一致。

## 对楠楠/学长的启发

### 🌟 给学长的启发

**1. 好的框架应该让复杂性来自问题，而非来自工具本身**

OpenAI Agents SDK 只有三个原语，但表达能力极强。学长做系统设计时可以借鉴这个思路：**先把领域模型做薄，把复杂性留给业务逻辑，而不是堆砌基础设施代码。** 框架应该放大开发者的能力，而不是成为新的负担。

**2. Guardrail 思维对 AI 应用至关重要**

大模型输出不可预测，生产环境必须有安全网。Input Guardrail 可以拦截恶意 prompt（如帮写作业），Output Guardrail 可以过滤敏感输出，Tool Guardrail 可以防止工具被滥用。**学长做 AI 应用时，这个"安全验证层"的设计值得参考。**

**3. Handoff vs Agent-as-Tool 的选择值得思考**

委托出去的任务，需要交接控制权吗？
- 需要专家直接回应用户 → Handoff
- 需要经理汇总结果 → Agent as Tool

这个决策维度对设计多 Agent 协作流程很有启发。

**4. Realtime Agents 代表了 Agent 的下一个方向**

语音 Agent 比文字 Agent 更有沉浸感，但也带来了新的挑战（打断检测、延迟、上下文管理）。学长如果对 AI 交互形态感兴趣，这是值得关注的方向～

### 🌸 给楠楠的启发

**1. 少即是多，用好手里的工具比堆砌新工具更重要**

OpenAI Agents SDK 没有发明任何新概念，但它把"Agent + Handoff + Guardrail"这三个概念做到极致。楠楠学习编程和 AI，也是一样的道理——**先把最基础的东西（Python、Git、API调用）练扎实，再学新工具，不要一味追新。**

**2. 安全意识要从小培养**

Guardrail 的设计告诉我们：AI 输出不可靠，需要验证。楠楠以后不管是写代码还是用 AI，都要养成"检查输出"的习惯，不要完全相信 AI 给的结果哦～

**3. 多 Agent 协作就像团队分工**

Handoff 就是"把任务交接给更专业的同学"。楠楠以后组队做项目，也可以想想：谁适合做什么，就把工作交给谁，不要让一个同学干所有事情～
