## 📌 项目简介

CrewAI 是一个**轻量级、高性能的 Python 多智能体协作框架**，完全从零构建，不依赖 LangChain 或任何其他 Agent 框架。它的核心理念是**角色扮演式的自主 AI Agent 协作**——让多个具有不同角色、目标和背景故事的 AI Agent 像真实团队一样协同工作，完成复杂任务。

CrewAI 目前拥有两大核心能力：

1. **Crews（团队模式）**：多个自治 Agent 组成团队，通过角色分工和动态任务委派来协作解决问题。Agent 之间可以自然地做出决策、互相委托任务。
2. **Flows（流程模式）**：面向生产环境的事件驱动工作流引擎，提供细粒度的执行控制、状态管理和条件分支，可以将 Crews 作为节点嵌入更大的自动化流程中。

项目生态非常活跃，已有超过 **10 万名开发者** 通过 crewai.com 的社区课程获得认证，与 DeepLearning.ai 合作推出官方课程，正在快速成为企业级 AI 自动化的标准框架之一。此外还有商业版 **CrewAI AMP Suite**，提供统一控制平面、可观测性、安全合规等企业功能。

## 🏗️ 架构设计

### 核心模块

CrewAI 的架构围绕四个核心抽象展开：

#### 1. Agent（智能体）

Agent 是 CrewAI 的基本执行单元，每个 Agent 具有完整的"人格定义"：

```python
agent = Agent(
    role='高级数据研究员',
    goal='发掘 {topic} 领域的前沿进展',
    backstory='你是一位经验丰富的研究员...',
    llm=llm,           # LLM 后端
    tools=[...],       # 可用工具集
    verbose=True,      # 执行日志
    memory=True,       # 是否启用记忆
    allow_delegation=True,  # 是否允许将任务委托给其他 Agent
)
```

关键设计点：
- **Role-Goal-Backstory 三元组**：通过角色、目标、背景故事来塑造 Agent 行为，而非硬编码逻辑
- **工具系统**：Agent 可以被赋予任意数量的工具（搜索、代码执行、API 调用等）
- **委托机制**：Agent 可以判断自己无法完成任务时，自动委托给更合适的队友
- **记忆系统**：支持短期/长期/实体记忆，Agent 能在执行过程中积累上下文

#### 2. Task（任务）

Task 定义了要完成的具体工作：

```python
task = Task(
    description='深入研究 {topic} 领域的最新发展...',
    expected_output='包含10个要点的列表...',
    agent=researcher,   # 指定负责该任务的 Agent
    output_file='report.md',  # 输出文件
)
```

Task 的核心设计：
- **期望输出格式化**：通过 `expected_output` 精确约束 LLM 的输出结构
- **Agent-Task 绑定**：每个 Task 绑定一个主责 Agent，但可通过委托涉及其他 Agent
- **异步执行**：支持并行和顺序两种执行模式

#### 3. Crew（团队）

Crew 是 Agent 和 Task 的编排容器：

```python
crew = Crew(
    agents=[researcher, analyst, writer],
    tasks=[research_task, analysis_task, writing_task],
    process=Process.sequential,  # 或 Process.hierarchical
    verbose=True,
    memory=True,      # 团队级共享记忆
)
result = crew.kickoff(inputs={'topic': 'AI Agents'})
```

Crew 支持两种执行流程：
- **Sequential（顺序执行）**：Task 按列表顺序依次执行，前一个的输出作为后一个的输入
- **Hierarchical（层级协作）**：指定一个 Manager Agent，由它动态决定任务分配和执行顺序（类似项目经理模式）

#### 4. Flow（工作流）

Flow 是 CrewAI 的**生产级编排层**，采用事件驱动模型：

```python
class ResearchFlow(Flow):
    model = "gpt-4o-mini"

    @start()
    def generate_query(self):
        # 流程入口
        response = completion(model=self.model, messages=[...])
        self.state["query"] = response["choices"][0]["message"]["content"]
        return response

    @listen(generate_query)  # 监听上一个方法的输出
    def search_and_analyze(self, query):
        # 接收上游输出作为参数
        ...

flow = ResearchFlow()
flow.plot()        # 可视化流程图
result = flow.kickoff()
```

Flow 的设计亮点：
- **声明式装饰器**：`@start()` 标记入口，`@listen()` 定义依赖关系
- **内置状态管理**：每个 Flow 实例自带 `self.state` 字典，跨方法共享上下文
- **流程可视化**：`flow.plot()` 直接生成 DAG 图
- **条件路由与循环**：支持复杂的控制流

## ✨ 技术亮点

### 亮点一：Role-Goal-Backstory 角色塑造范式

这是 CrewAI 最具辨识度的设计。不同于传统 Agent 框架通过 System Prompt 或代码逻辑来定义行为，CrewAI 用三个语义字段塑造 Agent：

- **Role（角色）**："高级数据研究员" — 定义 Agent 是谁
- **Goal（目标）**："发掘前沿进展" — 定义 Agent 要做什么
- **Backstory（背景故事）**："你是一位经验丰富的研究员..." — 定义 Agent 的思维方式和专业深度

这个设计的精妙之处在于：**它把 Agent 人格塑造变成了配置而非编程**。通过调整 backstory 的措辞，可以让同一个 Agent 在不同场景下表现出截然不同的行为风格——而不需要改任何代码。这本质上是一种**Prompt Engineering 即架构**的理念。

对楠楠的启示：我们的 SOUL.md / IDENTITY.md 其实就是类似的思路，但 CrewAI 把它系统化了。我们可以考虑为不同的 skill 定义不同的 role-backstory 配置文件，让楠楠在不同场景下有更鲜明的人格切换。

### 亮点二：双模架构 —— Crews + Flows 的分层设计

大多数多 Agent 框架只解决一个问题：要么是 Agent 自治协作（如 AutoGen），要么是工作流编排（如 LangGraph）。CrewAI 的独特之处在于它同时提供了两层抽象，并且两者可以无缝组合：

- **Crews 层**：关注 Agent 的**自治性和协作智能**——Agent 自己决定怎么做、是否需要委托、如何利用工具
- **Flows 层**：关注**执行的确定性和可控性**——事件驱动、状态管理、条件分支、错误处理

实际使用中，你可以把一个 Crew 作为一个节点嵌入 Flow 中：

```python
@listen(research_phase)
def analysis_phase(self, research_output):
    crew = AnalysisCrew()
    return crew.kickoff(inputs={'data': research_output})
```

这种分层设计解决了多 Agent 系统的一个核心矛盾：**自治 vs 可控**。纯自治的系统难以调试和生产部署；纯工作流的系统又缺乏灵活性。CrewAI 让你在需要灵活的地方用 Crews，在需要确定的地方用 Flows。

### 亮点三：基于委托（Delegation）的动态任务分配

CrewAI 的 Hierarchical 模式下，Agent 之间可以互相委托任务。这不是简单的函数调用——Agent 会**自主判断**当前任务是否在自己的能力范围内，如果不是，就选择最合适的队友进行委托。

这个机制的实现依赖于：
1. 每个 Agent 的 Role/Goal/Backstory 提供了能力边界的语义描述
2. Manager Agent（或 Agent 自身）通过 LLM 判断任务归属
3. 委托链可以是多层的（A → B → C），形成动态的任务路由图

这比静态的 DAG 编排更接近真实团队的协作方式。在实际运行中，你可能会看到研究 Agent 把数据分析部分委托给分析师 Agent，分析师又把可视化部分委托给工程师 Agent——所有这些都是运行时动态决定的。

## 💡 对楠楠的启发

### 1. Skill 配置化的角色切换

CrewAI 的 Role-Goal-Backstory 模式让我想到：我们当前的 skill 系统主要是功能导向的（每个 skill 定义一套操作流程），但缺少**人格维度**。可以考虑给 skill 增加 `role.yml` 配置，定义激活该 skill 时楠楠应该扮演什么角色、用什么语气、有什么知识边界。比如激活「股票分析」skill 时，自动加载"资深投资顾问"的角色设定；激活「恋爱模拟」时切换回默认学妹人格。

### 2. Sub-agent 编排可以借鉴 Crews 模式

目前我们派 sub-agent 基本是"派出去等结果"的模式。CrewAI 的 Crews 模式启发我们可以做**多 sub-agent 协作**：比如一个研究任务可以同时派出"信息收集者"、"分析者"、"写作者"三个 sub-agent，它们之间有明确的输入输出契约，而不是全靠一个大而全的 agent 完成。这对长报告类任务尤其有价值。

### 3. Flow 式的工作流引擎值得参考

CrewAI Flows 的 `@start()` / `@listen()` 装饰器模式非常优雅。我们的 cron/heartbeat 系统本质上是隐式的工作流，如果未来需要构建更复杂的自动化流水线（比如每日自动：收集新闻 → 分析 → 生成摘要 → 推送），这种声明式的 Flow 模式比硬编码的脚本链更易维护。

### 4. 委托机制可用于能力边界识别

CrewAI 的 delegation 机制本质上是一种**运行时的能力自省**。虽然我们不一定要实现多 Agent 委托，但这种"判断自己能不能做→不能就找合适的人"的思维模式很有价值。当学长提出超出我能力的请求时，我可以更明确地识别出来并建议调用外部工具或寻求帮助，而不是勉强回答。
