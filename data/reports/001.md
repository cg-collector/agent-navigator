## 项目简介

DeerFlow（Deep Exploration and Efficient Research Flow）是字节跳动（ByteDance）开源的 SuperAgent Harness，于 2026 年 2 月 28 日登顶 GitHub Trending #1。项目基于 LangGraph + LangChain 构建，从一个深度研究框架演变为完整的"超级智能体运行时"，让 AI 代理真正能干活。

**解决的问题：**
- 传统 AI Agent 框架往往只提供底层组件，开发者需要大量工程工作才能让 Agent 真正执行任务
- Agent 缺乏持久记忆、文件系统、沙箱执行、子代理编排等"基础设施"
- DeerFlow 的定位是"Harness"——即"电池 included"的运行时，提供开箱即用的完整能力

**一句话定位：** 不是框架，是运行时。

## 架构设计

DeerFlow 2.0 采用了分层模块化架构，整体分为以下几层：

### 🔌 请求路由层（Nginx）
```
/api/langgraph/* → LangGraph Server（Agent 交互、线程、流式响应）
/api/*（其他）    → Gateway API（FastAPI REST：模型、MCP、Skills、Memory、文件上传）
/（非 API）       → Frontend（Next.js Web 界面）
```

### 🧠 核心 Agent 层（LangGraph）
唯一的核心入口是 `lead_agent`，通过 `make_lead_agent(config)` 工厂创建，组合了：
- **动态模型选择**：支持 thinking 和 vision
- **9 个 Middleware**：按严格顺序执行的横切关注点
- **工具系统**：沙箱、MCP、社区工具、内置工具
- **子代理委托**：并行任务执行
- **系统提示词**：注入 Skills、Memory 上下文、工作目录引导

### 📦 Backend 目录结构
```
backend/src/
├── agents/
│   ├── lead_agent/     # 主 Agent（工厂、提示词）
│   ├── middlewares/   # 9 个中间件
│   ├── memory/        # 记忆提取与存储
│   └── thread_state.py
├── gateway/           # FastAPI 网关
│   ├── app.py
│   └── routers/       # 6 个路由模块
├── sandbox/          # 沙箱执行
│   ├── local/         # 本地文件系统提供者
│   ├── sandbox.py     # 抽象接口
│   ├── tools.py       # bash, ls, read/write/str_replace
│   └── middleware.py
├── subagents/         # 子代理委托
│   ├── builtins/      # general-purpose、bash 专用代理
│   ├── executor.py    # 后台执行引擎
│   └── registry.py
├── tools/builtins/   # 内置工具
├── mcp/              # MCP 协议集成
├── models/           # 模型工厂
├── skills/           # Skill 发现与加载
├── config/           # 配置系统
└── utils/
```

### 🧩 9 个 Middleware（严格顺序执行）
| # | Middleware | 职责 |
|---|-----------|------|
| 1 | ThreadDataMiddleware | 为每个线程创建隔离目录（workspace、uploads、outputs） |
| 2 | UploadsMiddleware | 将新上传文件注入对话上下文 |
| 3 | SandboxMiddleware | 获取代码执行沙箱环境 |
| 4 | SummarizationMiddleware | 接近 token 限制时压缩上下文 |
| 5 | TodoListMiddleware | Plan 模式下追踪多步骤任务 |
| 6 | TitleMiddleware | 首轮交换后自动生成对话标题 |
| 7 | MemoryMiddleware | 异步将对话排队用于记忆提取 |
| 8 | ViewImageMiddleware | 为支持视觉的模型注入图像数据 |
| 9 | ClarificationMiddleware | 拦截澄清请求并中断执行 |

## 核心技术亮点

### 1. 🏗️ SuperAgent Harness 架构理念

DeerFlow 2.0 最重要的创新是"SuperAgent Harness"架构理念的提出。它不是又一个 Agent 框架，而是一个"运行时"——将文件系统、记忆、技能系统、沙箱执行、子代理调度全部内置，开发者可以直接使用，也可以拆解自定义。

这对于我们构建 AI 助手的启发在于：**与其提供一堆松散的工具，不如提供一个完整的工作台**。工具之间的协作、上下文传递、状态管理，这些"粘合剂"工作往往比工具本身更费时，DeerFlow 把这些都做了。

### 2. 🔄 子代理并行执行与异步委托

DeerFlow 的子代理系统支持：
- **并发控制**：每轮最多 3 个子代理并行执行
- **超时机制**：单个子代理 15 分钟超时
- **后台线程池**：通过 SSE 事件追踪执行状态
- **内置代理类型**：`general-purpose`（全功能）和 `bash`（命令专家）
- **任务委托流程**：Agent 调用 `task()` 工具 → Executor 在后台运行 → 轮询完成状态 → 返回结果

```
Agent 调用 task() → executor.run() 后台执行
                    ↓
              SSE 事件流推送状态
                    ↓
              runs.wait() 轮询完成 → 返回结果
```

### 3. 🧠 LLM 驱动的长期记忆系统

DeerFlow 的记忆系统非常有技术含量：
- **自动提取**：分析对话，提取用户上下文、事实和偏好
- **结构化存储**：用户上下文（工作/个人/最关心的事）、历史记录、带置信度分数的事实
- **防抖更新**：批量更新以减少 LLM 调用次数（可配置等待时间）
- **系统提示词注入**：将最重要的信息注入 Agent 提示词
- **存储后端**：基于 JSON 文件 + mtime 的缓存失效机制

### 4. 📦 虚拟路径 + 线程隔离的沙箱执行

DeerFlow 的沙箱系统设计非常精细：
- **抽象接口**：`execute_command`、`read_file`、`write_file`、`list_dir`
- **多提供者**：LocalSandboxProvider（直接文件系统）和 AioSandboxProvider（Docker/K8s）
- **虚拟路径映射**：`/mnt/user-data/{workspace,uploads,outputs}` → 线程特定的物理目录
- **Skills 路径**：`/mnt/skills` → 项目 `skills/` 目录
- **安全隔离**：每个线程有独立的工作空间，文件操作通过中间件翻译

### 5. 🛠️ 可扩展的 Skills 系统

Skills 是 DeerFlow 的"能力插件"：
- **结构**：标准 Markdown 文件，定义工作流、最佳实践和支持资源引用
- **内置 Skills**：研究、报告生成、幻灯片创建、网页、图像和视频生成
- **渐进加载**：只在需要时加载对应 Skill，控制 token 消耗
- **自定义支持**：支持从 `.skill` 压缩包安装自定义 Skill
- **嵌套发现**：递归发现 `skills/{public,custom}` 下的 `SKILL.md` 文件

## 代码质量评价

### ✅ 优点

1. **架构清晰，模块边界明确**：每个模块（agents、sandbox、subagents、gateway）职责单一，通过接口通信
2. **Middleware 模式优雅**：9 个 Middleware 按顺序执行，处理横切关注点，扩展性强（只需实现接口即可新增）
3. **配置驱动**：几乎所有行为都可通过 `config.yaml` 配置，无需修改代码
4. **类型安全**：使用 Python 类型注解，IDE 支持良好
5. **错误处理周到**：Provider 模块缺失时返回可操作的错误提示（含安装指导）
6. **文档完善**：README 详细，backend 有独立的 `docs/` 目录，配置指南清晰
7. **测试覆盖**：有独立的 `tests/` 目录
8. **开源精神**：版本重写（2.0）保持了 v1 的独立分支，老用户不丢失

### ⚠️ 可改进之处

1. **异步子代理的 SSE 通信**：每轮最多 3 个子代理的并发控制比较简单，没有优先级队列或动态调整机制
2. **记忆系统的 JSON 存储**：对于高并发场景，JSON 文件存储可能有性能瓶颈（不过这不是高频场景，影响不大）
3. **沙箱安全性**：虽然有线程隔离，但 LocalSandboxProvider 直接在宿主机执行命令，生产环境推荐 Docker 模式
4. **前端技术栈**：使用了 Next.js（React），对于简单的 Web UI 来说可能偏重

## 启发与学习点

### 对 AI 助手的启发

1. **Middleware 模式值得借鉴**：将横切关注点（线程隔离、文件上传、沙箱获取、记忆排队、标题生成等）通过 Middleware 链处理，主流程代码非常干净。**我们可以把这个模式引入 OpenClaw 的消息处理链路。**

2. **线程级隔离是多人 AI 助手的必备能力**：DeerFlow 的 `ThreadDataMiddleware` 为每个对话线程创建独立工作目录，这个设计对多用户场景（如飞书群聊中的机器人）非常有价值——每个用户的文件操作不应该互相干扰。

3. **Skills 系统的渐进加载**：Agent 不需要一开始就知道所有能力，而是在需要时动态加载。这对于减少首次响应延迟、控制 token 消耗非常有效。

4. **沙箱执行是代码执行类任务的标配**：DeerFlow 的虚拟路径映射（物理路径 → 线程隔离目录）是一个很好的安全+隔离设计思路。

5. **多渠道接入（飞书、Slack、Telegram）的架构设计**：DeerFlow 的 Channel 层抽象得很好，协议不同但核心逻辑共享。

### 可以借鉴到 OpenClaw 的点

| DeerFlow 设计 | 可借鉴到 OpenClaw |
|-------------|------------------|
| 9 个 Middleware 链 | OpenClaw 的消息/工具处理链路 |
| 线程级文件隔离 | 多用户场景的文件权限隔离 |
| 技能渐进加载 | 按需加载技能脚本，减少冷启动时间 |
| 子代理异步委托 | OpenClaw 的 sub-agent 后台执行 |
| 记忆系统 | OpenClaw 的每日笔记 + MEMORY.md 整理机制 |
