## 1. 项目概述

AgentDoG 是一个**专注于 AI Agent 安全与防护的诊断型护栏框架**，由上海人工智能实验室（Shanghai AI Lab）的安全可信 AI 中心（Center for Safe & Trustworthy AI）提出。

不同于传统的 LLM 安全过滤（如 LlamaGuard3、Qwen3Guard），AgentDoG 的核心创新在于：**对 Agent 的执行轨迹（trajectory）进行全流程风险评估**，而不是对单步输出做简单的安全/非安全二元分类。

---

## 2. 核心问题：为什么需要轨迹级护栏？

### 2.1 现有护栏的局限

论文指出了当前主流护栏模型（如 LlamaGuard3、Qwen3Guard、ShieldGemma）的两大核心缺陷：

1. **缺乏 Agentic Risk Awareness（Agent 风险感知）**：传统护栏基于 LLM 输出内容的安全策略，无法理解 Agent 在**工具调用、环境交互、多步推理**中产生的复杂风险。

2. **缺乏溯源性与透明度（Provenance & Transparency）**：传统的 safe/unsafe 二元标签无法准确诊断风险根源，更无法捕捉那些"看起来安全但不合理"的行为。

### 2.2 Agent 风险的独特性

Agent 系统在开放式环境中运行，交互外部工具，执行多步骤任务。与静态 LLM 输出不同，Agent 风险往往源于**输入、推理、工具、动作之间的交互演化**，风险可能在中间步骤才显现，而非最终输出。

---

## 3. 架构设计

### 3.1 三维风险分类体系（Three-Dimensional Taxonomy）

AgentDoG 提出了一个**统一、分层的三维安全分类法**，从三个正交维度对 Agentic 风险进行系统化分类：

#### 维度一：风险来源（Risk Source）— "风险从哪来？"
- 标识风险在 Agent 循环中具体来自哪个环节
- 共 **8 类**：用户输入、环境观察、外部工具/API、Agent 内部推理等

#### 维度二：故障模式（Failure Mode）— "风险如何表现？"
- 描述不安全行为在 Agent 行为中的具体实现方式
- 共 **14 类**：规划缺陷、不安全工具使用、指令优先级混淆、不安全内容生成等

#### 维度三：现实危害（Real-World Harm）— "造成了什么后果？"
- 描述风险的现实世界影响
- 共 **10 类**：隐私泄露、金融损失、物理伤害、安全妥协、社会/心理伤害等

> 这个分类体系是"正交"的，意味着每个风险可以同时用三个维度描述，避免了传统平铺式风险列表的枚举局限。

### 3.2 轨迹级监控（Trajectory-Level Monitoring）

AgentDoG 监控 Agent 的**完整执行轨迹**，包括：
- **Thought（思考过程）**：Agent 的推理步骤
- **Action（工具调用）**：Agent 调用了哪些工具
- **Observation（环境反馈）**：工具返回了什么信息

这与传统的单步输出过滤有本质区别——风险可能出现在中间步骤，而非最终输出。

### 3.3 Agentic XAI 归因框架

AgentDoG 还包含一个**可解释 AI（XAI）模块**，能够：
- 诊断不安全行为的**根本原因**
- 将风险追溯到**具体的规划步骤、工具选择或上下文误解**
- 不仅判断"是否安全"，还解释"为什么不安全"

## 4. ATBench 基准测试

AgentDoG 配套发布了 **ATBench（Agent Trajectory Safety and Security Benchmark）**：

| 指标 | 数值 |
|------|------|
| 总轨迹数 | 500（250安全 / 250不安全） |
| 平均每条轨迹轮次 | ~8.97 轮 |
| 总交互轮次 | ~4486 轮 |
| 轨迹中出现的不重复工具 | 1575 个 |
| 独立未见工具库 | 2292 个工具定义 |
| 标注维度 | 二元安全分类 + 细粒度三维标注 |

ATBench 的关键设计原则：**测试模型对"未见工具"（unseen tools）的泛化能力**，而非仅在训练工具上的记忆。

## 6. 技术亮点总结

### 6.1 核心创新

1. **轨迹级而非输出级**：从监控单步 LLM 输出升级为监控整个 Agent 执行轨迹
2. **三维正交风险分类**：用 Risk Source × Failure Mode × Real-World Harm 三个维度替代单一的 safe/unsafe 标签
3. **根因诊断能力**：不仅判断是否安全，还能溯源到具体的规划步骤或工具选择
4. **未见工具泛化**：ATBench 专门设计了 2292 个训练时未见过的工具，测试护栏的泛化能力

### 6.2 SOTA 性能

AgentDoG 在多个 Agent 安全基准上达到了最优结果：
- **R-Judge**：Agent 安全判断基准
- **ASSE-Safety**：Agent 安全评估基准
- **ATBench**：自研细粒度基准

### 6.3 实用价值

- 可作为**评测基准**使用（ATBench）
- 可作为**轨迹风险分类器**集成到 Agent 管道中
- 可作为**在线护栏模块**实时拦截不安全行为

## 7. 对 OpenClaw 的启发

### 7.1 引入轨迹级安全监控

目前 OpenClaw 的安全护栏主要基于**内容过滤**和**工具权限控制**。AgentDoG 的思路提示我们：

> **与其只判断单步输出是否安全，不如监控整个 Agent 执行轨迹，在中间步骤就发现风险。**

OpenClaw 作为 Agent 框架，可以在执行过程中插入**轨迹级检查点**，对 Thought → Action → Observation 的循环进行周期性安全评估，而非只在最终输出时过滤。

### 7.2 细粒度风险分类指导安全策略

AgentDoG 的三维分类法（Risk Source / Failure Mode / Real-World Harm）可以作为 OpenClaw 安全策略设计的参考：

- **Risk Source** 维度可以帮助 OpenClaw 区分风险来源（用户输入 v.s. 工具响应 v.s. 内部推理），对不同来源的风险采用不同的处理策略
- **Failure Mode** 维度可以帮助定义 OpenClaw 的**拒绝/警告/降级**策略
- **Real-World Harm** 维度可以帮助评估风险严重程度，决定是否需要人工介入

### 7.3 可解释性增强用户信任

AgentDoG 的 XAI 归因框架提示我们：OpenClaw 在拒绝某个请求或拦截某个行为时，不仅要说"不安全"，还要能**解释为什么不安全、风险来自哪一步**。这能帮助用户理解 OpenClaw 的安全边界，提升信任度。

### 7.4 开源模型本地部署

AgentDoG 提供了 4B~8B 的小模型，这些模型足够轻量，可以**在 OpenClaw 中本地部署**作为安全护栏，而不需要调用外部 API。这对于注重隐私和延迟的场景非常有价值。

---

## 8. 局限性与思考

1. **学术项目，尚未大规模验证**：AgentDoG 目前 star 数较少（410），是一个相对早期的研究项目，在生产环境中的大规模验证数据还不足。

2. **性能与延迟的权衡**：轨迹级监控意味着在 Agent 执行的每一步都需要调用护栏模型，这会增加延迟。如何在不显著影响 Agent 响应速度的前提下实现实时护栏，是工程化的挑战。

3. **工具生态的快速演化**：Agent 的工具生态（特别是 MCP 工具）还在快速发展，新的工具类型和交互模式会不断出现，护栏模型需要持续更新。
