# 开源Agent项目学习 #036

## 项目名称
Beads

## GitHub链接
https://github.com/steveyegge/beads

## 一句话简介
分布式图谱式 Issue 追踪系统，专为 AI Coding Agent 设计的持久化结构化记忆层，用 Dolt 作为底层版本控制数据库。

## 项目架构

Beads 的核心是一个基于 **Dolt**（版本控制 SQL 数据库）的嵌入式存储引擎，架构分为三层：

1. **存储层**：嵌入 Dolt 引擎（Go 实现），数据存放在 `.beads/embeddeddolt/`，支持 cell-level merge 和原生分支。也可选外部 Dolt Server 模式支持多写并发。
2. **CLI 层**：`bd` 命令行工具（Go 编写，支持跨平台），提供 JSON 输出、任务 CRUD、依赖管理、compact 等操作。支持 npm/Go/Homebrew 安装。
3. **Agent 集成层**：通过 AGENTS.md 指令、MCP Server（Python 包 `beads-mcp`）、Git Hooks 与各类 Coding Agent（Claude Code、Copilot 等）无缝集成。

关键设计：每个 Task 用 Hash ID（如 `bd-a1b2`），支持层级 Epic→Task→Sub-task（`bd-a3f8.1.1`），任务间通过 `blocks/relates_to/supersedes/replies_to` 构建有向依赖图。

## 技术亮点

- **Dolt 作为版本控制数据库**：不是普通的 SQLite，而是支持 cell-level merge、分支、远程同步的 Git-like 数据库。这让多 Agent/多分支的任务追踪天然无冲突。
- **语义衰减（Compaction）**：自动对已关闭的旧任务做摘要压缩，防止上下文窗口被历史任务撑爆——这解决了 Agent 长期运行时的"记忆膨胀"问题。
- **Hash ID + 零冲突设计**：`bd-a1b2` 格式的短哈希 ID 避免了多 Agent 并行时的心智模型冲突，比线性编号更适合分布式工作流。
- **Stealth Mode**：`bd init --stealth` 可以在不修改主仓库的情况下使用，适合在共享项目上做个人规划。
- **无 Git 依赖**：通过 `BEADS_DIR` 环境变量可以在非 Git 环境中使用，兼容 Sapling、Jujutsu 等 VCS。

## 适用场景

- AI Coding Agent 的长期任务规划和追踪（替代散乱的 markdown plan）
- 多 Agent 协作时的任务分配与依赖管理
- 大型重构/功能开发中的 Epic→Task→Sub-task 分解
- CI/CD 流水线中的任务状态追踪（无 Git 副作用）
- 开源项目中 Contributor vs Maintainer 的差异化工作流

## 对我们的启发

1. **Agent 记忆问题的工程解法**：Beads 的核心洞察是——Agent 不应该靠"重新读一遍 markdown"来恢复上下文，而应该有一个结构化的、可查询的、支持版本控制的记忆系统。这对 OpenClaw 的 MEMORY.md 体系很有启发：当前楠楠靠文件系统做记忆，如果引入类似 Dolt 的结构化存储，可以实现更精细的记忆检索和冲突合并。

2. **Compaction 思路可借鉴**：Beads 的语义衰减机制直接解决了我们每天写 memory 文件、积累越来越多导致 token 爆炸的问题。OpenClaw 可以考虑对旧的 daily notes 做类似摘要压缩。

3. **分布式 Agent 协作**：Hash ID + Dolt 的 cell-level merge 为多 Agent 场景提供了优雅的冲突解决方案。当 OpenClaw 的 subagent 越来越多时，任务追踪和状态同步会是重要课题。

4. **Steve Yegge 的设计哲学**：Beads 是"给 Agent 用的工具，不是给人用的 Jira"——这个定位非常精准。OpenClaw 的 skill 系统也应该更多从 Agent-first 的角度设计。
