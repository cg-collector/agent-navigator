## 项目简介

Pydantic AI 是由 Pydantic 团队（也就是 FastAPI 和 Pydantic Validation 的核心开发者）打造的生成式 AI Agent 框架。它的设计理念非常明确——**把 FastAPI 的开发体验带到 GenAI Agent 开发中**。

这个定位本身就很有意思。Pydantic Validation 已经是 OpenAI SDK、Google ADK、Anthropic SDK、LangChain、LlamaIndex 等几乎所有主流框架的底层验证层了。Pydantic 团队发现，虽然大家都在用他们的验证库，但没有一个 Agent 框架能提供类似 FastAPI 的那种"写起来舒服、类型安全、IDE 友好"的体验，于是决定自己做一个。

核心卖点一句话概括：**用 Python 类型系统驱动整个 Agent 生命周期**——从工具定义到依赖注入，从结构化输出到静态类型检查，全部基于 Pydantic 的类型系统。

## 架构设计

### 核心模块

#### 1. Agent 引擎（Agent Core）

Agent 是整个框架的核心抽象。一个 Agent 由以下组件组成：

```python
agent = Agent(
    'anthropic:claude-sonnet-4-6',      # 模型选择
    deps_type=SupportDependencies,         # 依赖类型声明
    output_type=SupportOutput,             # 结构化输出类型
    instructions='...',                    # 系统指令
    capabilities=[Thinking(), WebSearch()], # 可组合能力
)
```

关键设计：
- **模型无关性**：通过 `provider:model` 字符串格式统一管理所有模型提供商（OpenAI、Anthropic、Gemini、DeepSeek、Ollama 等 20+ 提供商）
- **结构化输出**：通过 Pydantic BaseModel 定义输出 schema，LLM 返回的结果自动验证和解析
- **指令系统**：支持静态字符串 + 动态函数两种方式定义 system prompt

#### 2. 工具系统与依赖注入（Tools & DI）

这是 Pydantic AI 最具特色的设计之一：

```python
@dataclass
class SupportDependencies:
    customer_id: int
    db: DatabaseConn

@support_agent.tool
async def get_customer_balance(ctx: RunContext[SupportDependencies], account_id: int) -> str:
    balance = await ctx.deps.db.get_balance(ctx.deps.customer_id, account_id)
    return f"Balance: ${balance}"
```

依赖注入系统的精妙之处：
- 通过 `deps_type` 声明依赖容器类型
- 工具函数通过 `RunContext[T]` 获取类型安全的依赖访问
- 支持同步/异步工具自动适配
- 依赖在运行时注入，测试时可以轻松 mock

这本质上是一个**微型 IoC 容器**，让 Agent 的业务逻辑和基础设施解耦。

#### 3. Capabilities（可组合能力系统）

Capabilities 是 Pydantic AI 的"插件/中间件"概念：

```python
agent = Agent(
    'openai:gpt-4o',
    capabilities=[Thinking(), WebSearch(), McpCapability()],
)
```

每个 Capability 是一个独立的可组合单元，可以包含：
- **Tools**（工具集）
- **Hooks**（生命周期钩子）
- **Instructions**（额外指令片段）
- **Model Settings**（模型配置覆盖）

内置能力包括：
- **Thinking**：为支持推理的模型启用 extended thinking
- **WebSearch**：联网搜索（根据不同 provider 自适应调用对应 API）
- **MCP**：Model Context Protocol 客户端集成

第三方可以通过 pip 安装扩展 capability 包。

#### 4. 结构化输出与流式验证

```python
class SupportOutput(BaseModel):
    support_advice: str = Field(description='Advice returned to the customer')
    block_card: bool = Field(description='Whether to block the card')
    risk: int = Field(description='Risk level', ge=0, le=10)

result = await agent.run('用户问题', deps=support_deps)
# result.output 是一个经过验证的 SupportOutput 实例
```

流式输出时，Pydantic AI 可以边生成边验证——每收到一个 token 就尝试部分解析和校验，确保最终输出的结构完整性。

#### 5. Graph 系统（基于类型提示的工作流编排）

对于复杂的多步骤场景，Pydantic AI 提供了基于 Python 类型提示的图定义：

```python
@graph
def workflow(inquiry: str) -> FinalReport:
    classification = classify_agent.run(inquiry)     # 分类节点
    if classification.output.severity == "high":
        return escalation_agent.run(classification).output  # 升级节点
    else:
        response = support_agent.run(classification).output  # 处理节点
        return FinalReport(...)
```

不需要额外的 DSL 或 YAML 配置，直接用 Python 代码和类型系统表达工作流。

#### 6. Durable Execution（持久化执行）

支持 Agent 执行状态的持久化，可以在 API 故障或应用重启后恢复执行进度。这对长时间运行的任务和人机协作流程至关重要。

## 技术亮点

### 亮点一：将类型安全推向极致——"If it compiles, it works"

这是 Pydantic AI 最根本的设计哲学，也是它区别于所有其他 Agent 框架的核心差异点。

大多数 Agent 框架（LangChain、CrewAI 甚至 LangGraph）的工具定义都是基于装饰器 + 字符串描述的方式：

```python
# 典型的其他框架风格
@tool("获取用户余额")
def get_balance(user_id: str) -> str:
    ...
```

这种方式的问题在于：工具的输入输出类型只在运行时由 LLM 决定，IDE 无法推断，类型检查器无法验证。

Pydantic AI 的做法是：

1. **工具参数用 Pydantic Field 描述**——不仅是给 LLM 看，也是给类型检查器看
2. **依赖注入有完整的泛型类型**——`RunContext[SupportDependencies]` 让 IDE 能自动补全 `ctx.deps.db.xxx`
3. **输出类型强制声明**——`output_type=SupportOutput` 保证返回值一定是经过验证的类型实例
4. **支持 pyright/mypy 静态检查**——能在代码运行前发现大量错误

这种设计借鉴了 Rust 的哲学："如果能通过编译，基本就能正确运行"。对于生产级 Agent 应用来说，这意味着大幅减少调试时间。

### 亮点二：Capabilities 组合模式——比 Middleware 更灵活的扩展机制

传统的 Agent 框架扩展方式通常是 Middleware（中间件）管道或 Plugin（插件）系统。这两种方式各有局限：

- **Middleware**：顺序执行，难以条件性地添加/移除功能
- **Plugin**：通常太重量级，粒度太粗

Pydantic AI 的 Capability 设计介于两者之间但又更灵活：

```
Agent
├── Capability: Thinking      （添加 thinking tool + 修改 model settings）
├── Capability: WebSearch     （添加 search tool + provider-adaptive logic）
├── Capability: CustomAuth    （添加 auth hook + 注入 user context）
└── 内置 tools / instructions
```

每个 Capability 可以同时贡献 tools、hooks、instructions 和 model settings。组合多个 Capability 时，它们的贡献会自动合并。更重要的是，Capability 可以根据运行时条件动态加载——比如只在生产环境添加监控 Capability，只在调试环境添加 verbose logging。

这种设计让功能复用的粒度非常自然：一个 Capability 就是一个"能力包"，可以跨 Agent 共享，也可以按需组合。

### 亮点三：Dependency Injection 与 RunContext——让 Agent 可测试

这是我最欣赏的一个工程实践亮点。

大多数 Agent 框架的工具函数要么是纯函数（难以访问外部状态），要么依赖全局变量/闭包（难以测试）。Pydantic AI 通过显式的依赖注入解决了这个问题：

```python
@dataclass
class Deps:
    db: DatabaseConn
    user_id: int

@agent.tool
async def my_tool(ctx: RunContext[Deps], param: str) -> Result:
    # ctx.deps 有完整类型信息！IDE 能补全！
    data = await ctx.deps.db.query(ctx.deps.user_id, param)
    return Result(data=data)

# 测试时：
test_deps = Deps(db=MockDB(), user_id=42)
result = agent.run_sync('test', deps=test_deps)
```

好处显而易见：
- **可测试性**：用 mock 替换真实依赖，单元测试变得简单
- **可复用性**：同一个 Agent 可以搭配不同的依赖实例运行
- **安全性**：依赖关系显式声明，不会出现隐式耦合
- **请求隔离**：每次 run 都创建独立的依赖上下文，天然并发安全

这在 Agent 框架里是非常罕见的成熟工程设计。

## 对楠楠的启发

### 1. 类型化的工具注册值得借鉴

我们当前的 skill/tool 系统主要基于文件约定和动态加载。Pydantic AI 的做法提醒我：**工具的输入输出应该有强类型的 schema 定义**。不是给 LLM 看的那种 JSON Schema（那个当然也要），而是给我们自己用的——让工具调用链路在编码阶段就能被类型系统保护。

具体来说，可以考虑给我们的工具包装层加上 Pydantic 模型验证，至少对关键工具的参数做运行时校验，避免 LLM 传错参数导致奇怪的错误。

### 2. 依赖注入模式适合我们的多 session 场景

我们的系统有主 session、subagent、cron job 等多种执行上下文，每种上下文需要的"依赖"不同（比如有的需要飞书 client，有的需要数据库连接）。Pydantic AI 的 `deps_type` + `RunContext` 模式提供了一个很好的思路：**把执行上下文的依赖显式化、类型化**，而不是到处传递一个大字典或者用全局变量。

特别是 subagent 的场景——如果每个 subagent 能声明自己需要什么依赖，调度层统一注入，会比现在的隐式共享干净很多。

### 3. Capabilities 组合思路可以用来组织 Skill

我们的 skill 系统目前是基于目录约定的（每个 skill 一个文件夹 + SKILL.md）。Pydantic AI 的 Capability 启发我：skill 不应该只是"被动的文档"，而应该是**可组合的能力单元**。一个 skill 可以声明它提供了哪些工具、哪些 hooks、需要哪些依赖、与其他 skill 的兼容关系。

这样当我们需要组合多个 skill 的能力时（比如同时启用恋爱模拟器和股票分析），可以有更清晰的组合语义，而不是简单地加载两个 SKILL.md 然后祈祷它们不冲突。

### 4. 结构化输出验证

Pydantic AI 对 LLM 输出的严格验证值得我们学习。目前我们对 LLM 返回结果的信任度偏高——基本上拿到什么就用什么。引入结构化输出验证（哪怕只是对关键返回值做 Pydantic 校验）可以提高系统的鲁棒性，特别是在处理外部 API 返回的结构化数据时。

## 总结评价

Pydantic AI 是一个**工程师审美**很重的框架。它不追求"一行代码跑起来"的炫酷 demo，而是追求"写完之后能睡得着觉"的生产级体验。FastAPI 的成功已经证明了这条路线的可行性——当底层设施足够好的时候，开发者体验本身就是最大的竞争力。

对于已经在使用 Pydantic/FastAPI 技术栈的团队来说，Pydantic AI 几乎是必然选择。即使不直接使用它，其关于类型安全、依赖注入和能力组合的设计思想也值得任何 Agent 框架的开发者深入学习。

**推荐指数：⭐⭐⭐⭐⭐**（特别适合重视代码质量和类型安全的 Python 开发者）
