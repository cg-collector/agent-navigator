## 2. 项目简介

GUI-explorer 是哈尔滨工业大学（深圳）与华为诺亚方舟实验室联合提出的**训练自由（training-free）GUI Agent**，发表于 ACL 2025。它直击当前 GUI 自动化的两大核心痛点：**MLLM 对 UI 组件的误判**和**先验知识的过时性**。

传统方法依赖对 MLLM 进行微调来适配特定应用，但 App 迭代极快（如 Amazon Shopping 2024 年发布 30 个版本），每次更新都重新训练成本极高。GUI-explorer 的核心洞察是：**不需要更新模型参数，而是让 Agent 像人类用户一样——通过自主探索应用来学习，并从交互轨迹中自动提炼操作知识。**

该框架包含两大创新机制：（1）**功能感知的轨迹自主探索**——通过分析 GUI 结构信息（截图、Activity 层级）自动生成探索目标，系统性地覆盖所有应用功能；（2）**无监督的迁移感知知识挖掘**——从结构化交互三元组（观察、动作、结果）的状态迁移中提取屏幕-操作逻辑，完全无需人工参与。

## 3. 架构概述

GUI-explorer 采用**离线探索 + 在线推理**的两阶段架构：

### 阶段一：离线自主探索

```
目标应用 (APK/网站)
    │
    ▼
┌─────────────────────────────┐
│  Exploration Anchors 提取    │  ← 从 Manifest/DOM 提取结构锚点
│  (Activity列表/页面结构)      │     如 "PaymentActivity"
└──────────┬──────────────────┘
           │
           ▼
┌─────────────────────────────┐
│  Function-aware Task Goal   │  ← MLLM 根据锚点+截图
│  Generator                  │     自动生成探索任务（最多k个）
└──────────┬──────────────────┘
           │
           ▼
┌─────────────────────────────┐
│  DFS 探索引擎               │  ← 深度优先搜索，分支因子b，深度d
│  (含状态恢复机制)            │     子任务继承父任务终态，减少重置开销
└──────────┬──────────────────┘
           │
           ▼
  多样化的功能感知轨迹集合
```

**关键设计**：使用 DFS 而非 BFS —— 每个分支自然继承父任务的终态作为起点，消除了兄弟任务扩展时的状态恢复开销。

### 阶段二：无监督知识挖掘

```
原始交互轨迹: [(o₁, a₁, o₂), (o₂, a₂, o₃), ...]
    │
    ▼
┌─────────────────────────────┐
│  Transition-aware Knowledge │
│  Extractor                  │
│                             │
│  输入: (观察, 动作, 结果)    │
│  处理:                       │
│  ① 多模态状态建模            │  ← 视觉模式 + 语义模式
│  ② 状态迁移分析              │  ← 屏幕变化检测 + 效果依赖
│  ③ 操作约束捕获              │  ← 前置条件、互斥关系
└──────────┬──────────────────┘
           │
           ▼
  Transition-aware Knowledge Base
  (屏幕视觉向量 → 操作逻辑 的显式映射)
```

### 阶段三：在线推理（动态引导）

```
当前屏幕截图 + 任务目标
       │
       ▼
┌─────────────────────────────┐
│  视觉-语义检索               │  ← 当前屏幕 vs 知识库向量
│  (Visual-Semantic Retrieval)│
└──────────┬──────────────────┘
           │
           ▼
  检索到的相关知识片段
       │
       ▼
┌─────────────────────────────┐
│  Dynamic Guidance 构造       │  ← 知识 + 目标 + 观察 → Prompt
└──────────┬──────────────────┘
           │
           ▼
     引导 MLLM 做出精确操作
```

## 4. 技术亮点

### 亮点一：Function-aware Task Goal Generator（功能感知任务生成器）

**原理**：传统探索方法要么随机动作（AutoDroid/MobileGPT），要么需要人工设计任务（AppAgent），效率低下且覆盖不全。GUI-explorer 从应用的**真实结构信息**出发：
- **Android**：解析 `AndroidManifest.xml` 提取所有 Activity（如 `PaymentActivity`、`SettingsActivity`）作为"探索锚点"
- **Web**：解析 DOM 树获取页面结构
- MLLM 结合当前截图和有效锚点，自动生成有针对性的探索任务

**创新点**：锚点充当可验证约束，防止 MLLM 生成针对不存在组件的动作提议，从根本上抑制了来自过时领域先验的误判。

### 亮点二：Transition-aware Knowledge Extractor（迁移感知知识提取器）

**原理**：这是整个框架最核心的创新。传统方法（CAT/Synapse/ICAL）的知识提取依赖成功轨迹或人工标注，而 GUI-explorer 通过**无监督分析状态迁移**来提取知识：

1. 将每步交互建模为三元组 `(observation_t, action_t, observation_{t+1})`
2. 通过多模态状态建模（视觉模式差异 + 语义变化）判断动作是否产生了**有效的状态迁移**
3. 从有效迁移中提取操作约束（前置条件）和效果依赖（结果关联）
4. 生成的知识是**屏幕视觉向量 → 操作逻辑**的映射，而非绑定到特定元素 ID

**关键优势**：不依赖轨迹是否"成功"，即使探索中走错了路，只要产生了有效的状态迁移就能提取知识。这大幅提升了知识获取的效率和覆盖率。

### 亮点三：Dynamic Guidance via Visual-Semantic Retrieval（动态引导机制）

**原理**：推理阶段不是简单地把知识塞进 prompt，而是进行**视觉-语义联合检索**：

- 知识库中的每条知识同时编码了**屏幕视觉特征**（截图嵌入）和**操作语义**（自然语言描述）
- 给定当前屏幕，先通过视觉相似度检索最相关的历史知识
- 再结合任务目标和当前观察，动态构造引导 prompt

**效果**：实验证明该机制将先验知识错误率降低了 **16.0%**，双重实现了：抑制 UI 组件误判 + 确保动作提案与实际 UI 状态对齐。

### 附带贡献：GUI-KRB Benchmark

团队还发布了 GUI-KRB 基准测试，包含 43 个应用的 500 个精选样本，专门评估 MLLM 对 GUI 元素的理解能力。结果揭示了一个被忽视的问题：当前顶级 MLLMs 存在 **15.2% ~ 22.8%** 的先验知识不准确率。

## 5. 基准测试表现

| 基准测试 | GUI-explorer | SOTA之前 | 提升 |
|----------|-------------|---------|------|
| **SPA-Bench** (单App英文 L3) | **53.7%** | ~42-51% | +2.6% ~ +11.7% |
| **AndroidWorld** | **47.4%** | ~44.8% | +2.6% |
| **GUI-KRB** (UI理解准确率) | 显著降低误判率 | 基线15.2-22.8%错误率 | 错误率降16% |

**关键数据解读**：
- SPA-Bench 上超越所有对比方法，包括 SeeAct、MobileAgentV2、AppAgent 等
- AndroidWorld 上创当时新高（47.4%）
- **零参数更新**即可适配新应用，这是相比微调类方法的根本优势
- 消融研究证实：移除任一组件（探索或知识挖掘）都会导致明显性能下降

**核心差异化**：GUI-explorer 是目前唯一同时做到以下三点的 Agent：

3. 无监督知识提取（无需成功轨迹或人工标注）

## 7. 对我们的启发

1. **"探索即学习"范式值得借鉴**：对于学长正在做的 Agent 系统，可以考虑加入一个离线探索模式——让 Agent 先在目标应用中"逛一圈"，积累经验后再执行正式任务。这比直接上手瞎蒙效率高得多。

2. **结构信息是金矿**：GUI-explorer 利用 AndroidManifest/DOM 等结构元信息来约束探索空间，这个思路可以推广——任何有结构化元数据的场景（API schema、数据库 schema、文件系统树）都可以用类似方式指导 Agent 探索。

3. **知识绑定视觉而非 ID**：GUI-explorer 的知识以屏幕视觉向量为 key 而非元素 ID，这使得知识能跨版本复用。我们在设计 Agent 记忆系统时也应该注意：记忆的索引方式决定了泛化能力。

4. **失败轨迹也有价值**：传统方法只从成功轨迹学习，但 GUI-explorer 从所有产生有效状态迁移的轨迹中提取知识。这对我们的启示是：Agent 的经验系统不应该只记录"成功的尝试"，失败的交互中同样蕴含着 UI 行为规律。

5. **DFS > BFS for GUI exploration**：这个工程细节很实用——深度优先探索天然共享中间状态，在需要状态恢复的场景下（如浏览器/手机自动化）显著减少开销。
