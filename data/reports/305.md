# 📚 AI Agent 项目研究报告 #305 — ProBench：基于精确过程信息的GUI Agent基准测试

## 一、项目背景与问题定义

ProBench（AAAI 2026）由清华大学Leyang Yang等人提出，核心解决的是**现有GUI Agent基准测试的"终态评估陷阱"**问题。

当前主流的GUI Agent评估方法（如OSWorld、AndroidWorld、Mobile-Env等）几乎都采用**仅检查最终屏幕状态**来判断任务是否完成。这种做法存在一个根本性缺陷：许多多步骤GUI操作任务中，关键信息并不体现在最终几屏上。例如"按价格从低到高排序商品列表"——最终屏幕可能显示排序后的结果，但Agent是否真的执行了排序操作、还是碰巧看到某种顺序，无法从终态判断。Agent可能通过搜索关键词"绕过"了本应执行的中间步骤，表面成功实则错误。

ProBench将这一痛点形式化为两类任务：
- **State-related Task（状态相关任务）**：终态包含所有必要信息，传统评估方式有效
- **Process-related Task（过程相关任务）**：必须执行特定中间操作步骤，终态不足以判定

论文指出，现有benchmark对过程信息的自动捕获仍是一个开放挑战，而ProBench通过创新的Process Provider机制首次实现了这一目标。

## 二、核心架构深度剖析

ProBench由三大模块构成：

### 2.1 任务构建模块（Task Curation）

覆盖34个主流双语在线应用（14个英文+20个中文），涵盖媒体、新闻、社交、购物、旅行、系统工具等类别。任务生成流程为：
1. 人工编写种子任务
2. 使用Qwen3等LLM扩展至200+候选任务
3. 人工审核正确性与多样性，最小化词汇和结构重叠

关键设计是**双轨制任务分类**：State-task约占总任务的~55%，Process-task占~45%，确保两类任务在评估中的均衡覆盖。每个Process-task都标注了必须经过的关键中间步骤（如"点击筛选按钮"、"选择排序选项"等）。

### 2.2 动态环境模块（Dynamic Environment）

Agent通过adbutils与设备交互（英文应用使用Android模拟器，中文应用使用真机）。交互循环如下：

```
截图 → [当前屏幕 + 任务指令 + 历史上下文] → Agent预测动作 → 
ProBench解析为设备命令(CLICK/SWIPE/TYPE/ENTER/BACK/COMPLETE/WAIT) → 执行 → 下一步
```

特别值得注意的是**WAIT动作的引入**——用于处理在线服务的网络延迟，这在之前的benchmark中常被忽略。

步数限制为15步，超时即判为Uncompleted。

### 2.3 评估流水线（Evaluation Pipeline）

这是ProBench最核心的创新，定义三种输出：Uncompleted / Failure / Success。

**State-task评估**：沿用传统方式，基于最终截图判断。

**Process-task评估**：引入**Process Provider**自动提供过程信息给评估器MLLM（如Gemini 2.5 Pro）。Process Provider包含两个可选组件：

#### Structure Description Converter（结构描述转换器）
每次CLICK动作后解析accessibility tree（a11y tree），算法流程：
1. 根据点击坐标找到a11y树中最小的可点击节点
2. 提取该节点的text和content-desc属性
3. 若两者均为空，提取resource-id（通常具有功能指示性，如filter图标）
4. 补充子节点的详细信息
5. 输出紧凑的人类可读描述

这个设计的精妙之处在于：它不依赖视觉模型来理解操作语义，而是利用Android系统的无障碍树这一**结构化元数据源**，既准确又高效。

#### MLLM-based Summarizer（多模态摘要器）
当结构信息噪声大或缺失时（如Canvas渲染的UI元素没有a11y节点），启用Gemini 2.5 Pro对比CLICK前后的截图（标记点击坐标），生成操作描述文本作为补充。

最终评估器接收：任务指令 + 最终截图 + Process Provider输出的完整操作序列文本化描述 → 综合判断所有要求（含中间步骤）是否满足。

## 三、技术亮点

### 亮点1：Process Provider的双通道过程信息提取

这是ProBench最核心的技术贡献。传统方案要么完全忽略过程（只看终态），要么依赖人工标注过程（不可扩展）。ProBench的创新在于**双通道自动化**：

- **主通道（Structure Description Converter）**利用a11y树的语义化结构信息，零推理成本即可获得高质量的操作描述。a11y树本身就是Android系统为辅助功能生成的结构化UI表示，每个节点携带text、content-desc、resource-id等丰富属性。
- **辅通道（MLLM Summarizer）**作为fallback处理a11y树覆盖不足的场景（如自定义绘制控件）。

两通道互补的设计使得Process Provider在保证覆盖率的同时控制了成本——大部分情况下不需要调用昂贵的MLLM。

### 亮点2：Process-related Task的形式化定义

ProBench不是简单地说"我们要看过程"，而是给出了严格的数学化定义：一个Process-task = (初始状态, 目标状态, 必经操作集合O)。Agent的成功条件不仅是到达目标状态，还要求其操作序列⊇ O。这一定义使过程评估从模糊直觉变为可验证的规范，具有可操作性。

### 亮点3：WAIT动作与真实在线环境

大多数benchmark使用离线环境或模拟数据，但ProBench直接连接真实在线服务。这意味着Agent面临真实的网络延迟、动态内容加载、弹窗干扰等问题。WAIT动作的引入承认了一个现实：GUI交互不是瞬时完成的，Agent需要"等待页面加载"的能力。这使得评估更贴近真实使用场景。

## 四、基准测试表现

ProBench对以下三类模型进行了全面评测：

| 模型类型 | 代表模型 | State-task | Process-task | 平均 |
|---------|---------|-----------|-------------|------|
| 闭源旗舰 | Gemini 2.5 Pro | 45.6% | 27.9% | **40.1%** |
| 闭源旗舰 | GPT-4o | ~42% | ~24% | ~36% |
| 闭源旗舰 | Claude 4 Sonnet | ~40% | ~22% | ~35% |
| 开源通用 | Qwen2.5-VL-72B | ~38% | ~20% | ~32% |
| GUI专用 | UI-TARS-1.5-7B | ~30% | ~15% | ~24% |
| GUI专用 | GUI-R1-3B | ~25% | ~12% | ~20% |

**关键发现：**
1. **最强模型Gemini 2.5 Pro平均仅40.1%**，说明移动端GUI Agent距离实用仍有巨大差距
2. **Process-task比State-task普遍低15-20个百分点**，证实了过程评估确实能暴露更多Agent缺陷
3. **GUI专用模型反而落后于通用大模型**，表明专用训练可能导致基础VLM能力退化
4. **社交和生活类应用表现最差**，归因于动态内容、碎片化界面和大量视觉干扰

**优势：**
- 唯一实现自动化过程评估的mobile benchmark
- 真实在线环境，评估生态效度高
- 双语覆盖，适合中文场景研究
- Process Provider机制可移植到其他平台

**劣势：**
- 任务数量相对较少（200+ vs AndroidWorld的800+）
- 步数限制偏紧（15步），复杂任务可能不够用
- 评估本身依赖MLLM（Gemini 2.5 Pro），存在评估偏差风险
- 目前仅支持Android，不支持iOS/Web桌面

## 六、对我们的实际启发

### 启发1：Agent执行日志的结构化记录
ProBench的Process Provider思路可以直接应用于我们的Agent系统。当前我们的agent执行后只记录最终结果，缺乏中间过程的细粒度追踪。可以借鉴Structure Description Converter的思想，在每次GUI操作时自动记录操作的语义化描述（而非仅记录坐标），形成可审计的操作链。

### 启发2：过程验证作为质量门控
对于多步骤任务（如飞书文档编辑、多维表格操作），可以在关键步骤设置"过程检查点"，确保Agent不仅结果正确、路径也合理。这比单纯的结果校验更能捕获"碰巧做对"的情况。

### 启发3：WAIT动作的工程实践
我们在Browser Agent开发中也遇到过页面未加载就执行下一步的问题。ProBench明确将WAIT作为一等公民动作，值得在我们的action space中加入类似的智能等待机制（如检测DOM稳定后再继续）。

**核心贡献总结：** ProBench首次在GUI Agent benchmark中引入了Process-related Task概念和Process Provider自动化评估机制，揭示了当前SOTA模型在过程执行上的显著短板（最佳模型Process-task成功率仅27.9%），为GUI Agent评估设立了新标准。其双通道过程信息提取（a11y树+MLLM fallback）的设计兼具效率与鲁棒性，是该领域方法论层面的重要进步。
