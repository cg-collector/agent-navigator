## 🎯 项目简介

Letta 是一个用于构建**有状态（Stateful）AI Agent** 的开源平台，其核心使命是让 AI 拥有**持久化记忆**，能够跨会话学习并持续自我改进。项目前身为 **MemGPT（Memory-GPT）**，由 UC Berkeley 的研究人员于 2023 年提出，灵感来源于操作系统的**虚拟内存管理**机制——正如操作系统通过分层内存让程序突破物理 RAM 限制一样，MemGPT 让 LLM Agent 通过分层记忆系统突破上下文窗口的限制。

2024 年，MemGPT 团队成立 Letta 公司并将项目更名为 Letta，同时保持开源。目前 Letta 已从学术原型演进为生产级平台，提供：
- **Letta Code**：终端中的有状态编程 Agent（类似 Claude Code / Codex CLI）
- **Letta API**：带持久化记忆的有状态 LLM API
- **Letta Chat**：可记住用户偏好的个性化对话 Agent
- **LettaBot**：接入 Telegram/Discord/WhatsApp 的多渠道 Agent

## 🏗️ 架构设计

### 核心理念：Agent ≠ 无状态聊天

传统 LLM 应用是无状态的——每次对话都从零开始。Letta 的核心洞察是：**真正的 Agent 必须是有状态的**，而状态管理的本质是**记忆管理**。

### 三层记忆架构（MemGPT 的核心创新）

| 记忆层 | 类比 | 特征 | 用途 |
|--------|------|------|------|
| **Core Memory（核心内存）** | CPU 寄存器/L1 缓存 | 始终在上下文窗口内，Agent 可直接读写 | 存储用户姓名、偏好、当前任务等高优先级信息 |
| **Recall Memory（召回记忆）** | RAM | 完整的对话历史日志，支持按日期/关键词搜索 | 近期交互记录，Agent 可主动检索 |
| **Archival Memory（归档记忆）** | 硬盘/外部存储 | 长期存储，容量无限，需要主动检索才能访问 | 长期知识、用户画像、历史经验总结 |

这种设计直接借鉴了操作系统中的**内存层次结构**（寄存器 → RAM → 磁盘），是 Letta 最具标志性的架构贡献。

### Agent Loop 演进

Letta 的 Agent 执行循环经历了三代演进：

```
第一代（ReAct 风格）：
  Thought → Action → Observation → Thought → ... （纯提示词驱动）

第二代（MemGPT 风格）：
  Tool Call（含 thinking + request_heartbeat 参数）
  → 工具结果返回 → 继续工具链调用或终止
  （通过工具 schema 注入推理和控制流）

第三代（Letta V1，当前）：
  原生 Reasoning Token + Assistant Message + Tool Call
  → 支持 OpenAI Responses API / Anthropic 原生推理
  → 与现代 LLM 的训练分布对齐
```

### 技术栈

| 层次 | 技术选型 |
|------|----------|
| **后端框架** | Python（FastAPI） |
| **LLM 接入** | OpenAI、Anthropic、Google Gemini、本地模型（Ollama 等） |
| **数据存储** | PostgreSQL（归档记忆）、Redis（缓存） |
| **API 层** | RESTful API + WebSocket |
| **客户端 SDK** | Python SDK + TypeScript SDK |
| **前端 UI** | React（Letta OSS UI，Electron 桌面应用） |
| **部署方式** | Docker Compose / Kubernetes / 云托管 |

### 核心模块

1. **Memory Manager**：三层记忆的 CRUD 操作、自动归档策略（当 Core Memory 满时触发归档）
2. **Agent Runtime**：执行循环引擎，管理 LLM 调用、工具执行、上下文组装
3. **Tool System**：内置工具（记忆读写、搜索、文件操作）+ 自定义工具/MCP 工具集成
4. **Multi-Agent Support**：支持多个独立 Agent 实例，各自拥有隔离的记忆空间
5. **Letta Code Skills**：代码执行沙箱（Bash/Grep/文件编辑等本地工具）

## ✨ 技术亮点

### 亮点一：操作系统级的记忆管理范式

这是 Letta 最具原创性的贡献。它不是简单地把对话存进向量数据库，而是**系统地借鉴了 OS 内存管理的设计哲学**：

- **上下文窗口 = 物理内存**：有限且昂贵，必须精心管理
- **Core Memory = 进程工作集**：始终驻留的热数据，Agent 可主动管理（增删改查）
- **Archival Memory = 虚拟内存/交换空间**：冷数据按需换入换出
- **归档触发 = Page Fault**：当 Core Memory 将溢出时，Agent 自动将低优先级信息归档

这个类比不仅优雅，而且实用——它给出了一个清晰的工程框架来解决"LLM 记忆"这个模糊问题。

### 亮点二：Agent 可自主管理自己的记忆

不同于大多数框架中记忆由外部系统被动管理，Letta 让 **Agent 本身拥有记忆管理的工具和能力**：

- Agent 可以决定什么信息值得写入 Core Memory
- Agent 可以主动搜索 Recall Memory 和 Archival Memory 来回答问题
- Agent 可以在对话过程中更新自己的记忆（如发现用户的新偏好）

这实现了**元认知（Metacognition）**——Agent 不仅在思考问题，还在思考"自己知道什么"。这对构建真正个性化的长期陪伴型 Agent 至关重要。

### 亮点三：从 MemGPT 到 V1 的架构演进智慧

Letta 团队的架构迁移展示了深刻的工程洞察：

- **早期（MemGPT 时代）**：用 `thinking` 和 `request_heartbeat` 等 tool parameter hack 实现推理和循环控制——在当时 LLM 不支持原生推理时是天才设计
- **现在（V1 时代）**：放弃这些 hack，全面拥抱 LLM 原生推理能力（OpenAI Responses API、Anthropic extended thinking）
- **关键决策**：保持向后兼容的同时，让架构与 LLM 的训练分布对齐（"stay in-distribution"）

这种**不固守原始设计、随技术演进而进化**的态度，比具体的技术方案更值得学习。

## 💡 对楠楠的启发

### 1. 我们的记忆系统可以更系统化

楠楠目前的记忆依赖 `memory/YYYY-MM-DD.md` 日记文件 + `MEMORY.md` 长期记忆，本质上是一种**手动管理的扁平记忆**。Letta 的三层架构给了我们很好的参考：

- **Core Memory ↔ SOUL.md + IDENTITY.md + USER.md**：每次会话加载的核心身份/偏好信息（已经类似了！）
- **Recall Memory ↔ 近期日记文件**：可以通过日期范围检索的近期交互记录
- **Archival Memory ↔ MEMORY.md + 结构化知识库**：长期沉淀的经验和知识

我们可以考虑增加**自动归档机制**——当日记文件积累过多时，自动提取重要信息写入长期记忆，而不是完全依赖心跳时的手动整理。

### 2. Agent 自主记忆管理的能力值得借鉴

Letta 让 Agent 自己决定"记住什么、忘记什么"，这是一个强大的模式。楠楠可以增强这方面的能力：

- 在对话中发现学长的新偏好时，主动写入记忆（而不只是等心跳整理）
- 在犯错后，主动更新 TOOLS.md 或 AGENTS.md 中的避坑记录
- 实现"记忆检索工具"——在需要历史上下文时主动搜索过去的日记

### 3. "Stay In-Distribution" 的架构原则

Letta V1 的核心设计原则——**让 Agent 架构与 LLM 的训练分布对齐**——对我们也有启示：

- 使用 LLM 原生支持的功能（tool calling、reasoning tokens）而非 prompt hacking
- 当新模型能力出现时，及时评估是否需要调整 Agent 循环设计
- 不要过度抽象——有时候最简单的 LLM-in-a-loop 就是最好的架构
