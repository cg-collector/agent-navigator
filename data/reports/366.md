## 一、项目概述

**BeeAI Framework** 是由 IBM Research 发起并开源的 AI Agent 构建框架，旨在提供一套"超越提示词、强制执行规则"的轻量级生产级多智能体系统开发工具链。该项目于 2024 年底以 TypeScript 版本首次亮相，2025 年 2 月推出 Python 版本并实现双语言特性对等，随后于 2025 年正式纳入 **Linux Foundation AI & Data** 项目进行社区化开源治理。

BeeAI 的核心理念可以用一句话概括：**让 Agent 的行为可预测、可约束、可观测**。在当前 AI Agent 领域普遍面临"黑盒行为不可控"的痛点下，BeeAI 提出了独特的 **Requirement Agent（需求/约束型智能体）** 范式——通过声明式的规则定义来强制 Agent 执行确定性流程，同时保留 LLM 的推理灵活性。这一设计哲学使其与 LangChain/LangGraph 的图编排范式、CrewAI 的角色扮演范式、AutoGen 的对话范式形成了鲜明的差异化定位。

从生态角度看，BeeAI 已经发展出两个互补项目：
- **BeeAI Framework**：Agent 开发框架（本项目）
- **Agent Stack**：Agent 部署与共享平台（含即时 UI 生成、多框架兼容运行时）

## 二、项目背景与动机

### 2.1 为什么需要 BeeAI？

IBM Research 团队在投入一年时间将 AI Agent 从原型推向生产环境的过程中，发现了几个关键痛点：

1. **行为不可预测性**：传统 ReAct 或 Tool-Calling Agent 完全依赖 LLM 自主决定下一步行动，在生产环境中经常出现工具调用顺序错误、遗漏关键步骤、无限循环等问题。
2. **可观测性缺失**：大多数框架缺乏内置的结构化事件追踪能力，调试复杂的多步 Agent 流程如同盲人摸象。
3. **跨语言壁垒**：主流框架要么只支持 Python（LangChain/CrewAI），要么只支持 TypeScript（Vercel AI SDK/Mastra），团队技术栈切换成本高。
4. **协议碎片化**：MCP、A2A 等新兴协议快速涌现，但各框架集成程度参差不齐。

### 2.2 设计哲学

BeeAI 的设计遵循以下原则：

| 原则 | 说明 |
|------|------|
| **规则优于提示** | 用声明式约束控制 Agent 行为，而非依赖 system prompt "建议" |
| **事件驱动一切** | 所有组件通过 Emitter 事件系统通信，天然支持中间件和可观测性 |
| **双语言一等公民** | Python 和 TypeScript 不是"主+移植"关系，而是完全对等的实现 |
| **开放治理** | 托管于 Linux Foundation，避免厂商锁定 |

## 三、核心特性深度解析

### 3.1 Requirement Agent — 约束型智能体（核心创新）

这是 BeeAI 最具差异化的特性。传统的 Agent 通过 system prompt 来"建议"模型按某种方式行动，但 LLM 不一定遵守。Requirement Agent 则通过 **Requirements（需求约束）** 系统，在框架层面强制执行确定性行为规则。

#### 核心概念：Requirements（约束）

```python
from beeai_framework.agents.requirement import RequirementAgent
from beeai_framework.agents.requirement.requirements.conditional import ConditionalRequirement
from beeai_framework.tools.think import ThinkTool
from beeai_framework.tools.weather import OpenMeteoTool
from beeai_framework.tools.search.duckduckgo import DuckDuckGoSearchTool
from beeai_framework.backend import ChatModel
from beeai_framework.middleware.trajectory import GlobalTrajectoryMiddleware

agent = RequirementAgent(
    llm=ChatModel.from_name("ollama:granite4:micro"),
    tools=[
        ThinkTool(),             # 推理工具
        OpenMeteoTool(),         # 天气查询
        DuckDuckGoSearchTool()   # 网络搜索
    ],
    instructions="根据天气和活动信息规划行程。",
    requirements=[
        # 强制第一步必须先思考（不是"建议"，是"强制"）
        ConditionalRequirement(ThinkTool, force_at_step=1),
        # 只有获取天气后才能搜索，且至少搜索一次
        ConditionalRequirement(
            DuckDuckGoSearchTool,
            only_after=[OpenMeteoTool],
            min_invocations=1
        ),
        # 天气工具至少用一次，但不能连续调用两次
        ConditionalRequirement(
            OpenMeteoTool,
            consecutive_allowed=False,
            min_invocations=1
        ),
    ]
)

response = await agent.run("波士顿有什么好玩的？").middleware(
    GlobalTrajectoryMiddleware()
)
```

#### 支持的约束类型

| 约束类型 | 功能 | 典型场景 |
|----------|------|----------|
| `ConditionalRequirement` | 条件触发约束（force_at_step / only_after / min_invocations / consecutive_allowed） | 控制工具调用顺序和频率 |
| `HandoffTool` | 将任务转交给子 Agent | 多专家协作场景 |
| `expected_output` | 引导输出格式（支持 Pydantic Model / JSON Schema） | 结构化数据提取 |

**关键洞察**：这种"硬约束 + 软推理"的混合模式，解决了 Agent 生产部署中最头疼的问题——**如何在保留 LLM 创造力的同时确保关键流程不被跳过**。

### 3.2 事件驱动的 Middleware 中间件系统

BeeAI 的中间件系统是其架构的灵魂。所有组件（Agent、Tool、ChatModel）都继承自 `Runnable` 抽象类，每次执行都会创建一个带生命周期的 `Run` 对象，该对象携带自己的 Emitter（事件发射器），中间件可以拦截任何执行阶段的输入/输出。

#### 核心原语层次

```
Emitter（事件发射器）
  └── Run / RunContext（执行上下文，含生命周期）
       └── Runnable[R]（可执行抽象，统一 run/middleware/emitter 接口）
            ├── Agent（智能体）
            ├── ChatModel（语言模型后端）
            └── Tool（工具）
```

#### 内置中间件

**GlobalTrajectoryMiddleware** — 全局轨迹追踪：

```python
# 只记录工具调用事件
agent.run("...").middleware(GlobalTrajectoryMiddleware(included=[Tool]))

# 排除特定工具
agent.run("...").middleware(
    GlobalTrajectoryMiddleware(included=[Tool], excluded=[OpenMeteoTool])
)

# 自定义前缀显示
agent.run("...").middleware(
    GlobalTrajectoryMiddleware(prefix_by_type={BaseAgent: "🐝 "})
)
```

**StreamToolCallMiddleware** — 工具调用流式输出：

```python
middleware = StreamToolCallMiddleware(
    weather_tool,
    key="location_name",      # 监听的工具参数字段
    match_nested=False,
    force_streaming=True,     # 强制开启流式
)

@middleware.emitter.on("update")
def log_update(event, meta):
    print(f"收到更新: {event.delta}, 输出: {event.output_structured}")
```

#### 自定义中间件示例

```python
from beeai_framework.context import RunContext

class SafetyCheckMiddleware:
    """自定义安全检查中间件"""
    
    async def bind(self, ctx: RunContext):
        @ctx.emitter.on("start")
        async def on_start(data, meta):
            # 在执行开始时注入安全上下文
            ctx.context["safety_checked"] = False
        
        @ctx.emitter.on("success")
        async def on_success(data, meta):
            # 执行成功后记录审计日志
            await self.audit_log(meta)

agent.run("...").middleware(SafetyCheckMiddleware())
```

### 3.3 统一 Backend 后端系统

BeeAI 提供了统一的 LLM 后端抽象，支持 10+ 提供商无缝切换：

```python
from beeai_framework.backend import ChatModel

# 本地模型
llm_local = ChatModel.from_name("ollama:granite3.3:8b")
llm_local2 = ChatModel.from_name("ollama:granite4:micro")

# 云端 API
llm_openai = ChatModel.from_name("openai:gpt-5-mini")
llm_anthropic = ChatModel.from_name("anthropic:claude-sonnet-4")
llm_groq = ChatModel.from_name("groq:llama-3.3-70b-versatile")

# IBM 自家
llm_watsonx = ChatModel.from_name("watsonx:ibm/granite-3-8b-instruct")
```

支持的提供商包括：OpenAI、Anthropic、Ollama、Groq、Watsonx.ai、Google Gemini、DeepSeek 等。切换只需改一个字符串，其余代码完全不变。

### 3.4 Memory 记忆管理

BeeAI 内置三种记忆策略，可在不修改 Agent 代码的情况下灵活切换：

```python
from beeai_framework.memory import (
    TokenMemory,          # 基于 token 数限制的滑动窗口记忆
    UnconstrainedMemory,  # 无限制记忆（适合短对话）
    SummaryMemory         # 自动摘要压缩的长对话记忆
)

# 方式一：Token 限制（推荐用于生产环境）
agent = RequirementAgent(
    llm="ollama:granite4",
    memory=TokenMemory(max_tokens=20*1024)  # 20K token 上下文窗口
)

# 方式二：无限制
agent = RequirementAgent(
    llm="ollama:granite4",
    memory=UnconstrainedMemory()
)
```

### 3.5 Workflows 多智能体编排

Workflows 是 BeeAI 的多 Agent 编排模块，支持通过装饰器或 YAML 声明式地构建复杂的执行流程：

```python
from beeai_framework.workflows import Workflow

workflow = Workflow(name="research_pipeline")

@workflow.step()
async def search(query: str):
    """搜索步骤"""
    return await search_tool.execute(query)

@workflow.step(after=["search"])
async def analyze(search_results):
    """分析步骤"""
    return await analysis_agent.run(search_results)

@workflow.step(parallel=True)
async def fact_check(analysis):
    """并行事实核查"""

result = workflow.run("最新 AI 研究趋势")
```

支持的高级模式包括：
- **并行执行**（parallelism）
- **条件重试**（retries with backoff）
- **动态重规划**（replanning）
- **YAML 声明式编排**

### 3.6 Serve 模块 — 协议原生支持

BeeAI 原生支持将 Agent 作为服务暴露，支持多种协议：

| 协议 | 说明 | 状态 |
|------|------|------|
| **MCP (Model Context Protocol)** | Anthropic 主导的工具互操作协议 | ✅ 正式支持 |
| **A2A (Agent-to-Agent)** | Google 主导的 Agent 互操作协议（已并入 Linux Foundation） | ✅ 正式支持 |
| **ACP (Agent Communication Protocol)** | IBM 的 Agent 通信协议（已合并入 A2A） | ⚠️ 迁移至 A2A |

这意味着你用 BeeAI 构建的 Agent 可以：
- 作为 MCP Server 被 Claude Desktop / Cursor 等客户端直接使用
- 通过 A2A 协议与其他框架的 Agent 互操作

### 3.7 可观测性与 OpenTelemetry 集成

BeeAI 原生集成 OpenTelemetry，所有事件自动导出为 OTel Trace：

```python
# 自动集成到现有的 OTel 基础设施
# 无需额外配置，安装 opentelemetry-api 即可
import opentelemetry.trace as trace

tracer = trace.get_tracer(__name__)
# BeeAI 的所有 agent/tool/model 调用会自动出现在你的 tracing 后台
```

配合 `openinference-instrumentation-beeai` 库，可直接接入 Orq.ai、Langfuse、Arize Phoenix 等可观测平台。

### 4.2 事件系统架构（核心创新）

BeeAI 的事件系统采用树形层级传播机制：

```
Root Emitter
├── Agent Emitter (namespace: agents.requirement)
│   ├── Event: start → data={input, config}
│   ├── Event: tool_call → data={tool_name, args}
│   ├── Event: thinking → data={thought}
│   ├── Event: success → data={output}
│   └── Event: error → data={error}
│       └── ChatModel Emitter (namespace: backend.chat.ollama)  [嵌套]
│           ├── Event: start → data={messages, params}
│           └── Event: success → data={response}
│               └── Tool Emitter (namespace: tool.open_meteo)  [嵌套]
│                   ├── Event: start → data={input}
│                   └── Event: success → data={output}
```

每个事件包含三要素：
- **name**：事件标识符（如 start/success/error/final_answer）
- **data payload**：结构化数据（Pydantic Model）
- **metadata**：上下文信息（path、trace_id、run_id、creator 类型）

| **独特优势** | 约束系统 + 中间件 | 图编排 + 状态管理 | 快速角色编排 | 多 Agent 对话 | 类型安全 + 验证 |

### 6.3 我的判断

BeeAI 最独特的价值在于其 **"约束即代码"** 的理念。当前大多数 Agent 框架解决的是"如何让 Agent 更聪明"的问题，而 BeeAI 解决的是"如何让 Agent 更可靠"的问题。在企业级生产环境中，后者往往比前者更重要——一个偶尔给出精彩答案但不稳定的 Agent，远不如一个稳定给出合格答案的 Agent 有价值。

## 七、代码示例：完整的 BeeAI 多 Agent 系统

以下是一个展示 BeeAI 核心能力的完整示例——一个具备天气查询、知识检索、手off 协作的多 Agent 系统：

```python
import asyncio
from beeai_framework.agents.requirement import RequirementAgent
from beeai_framework.agents.requirement.requirements.conditional import ConditionalRequirement
from beeai_framework.backend import ChatModel
from beeai_framework.errors import FrameworkError
from beeai_framework.middleware.trajectory import GlobalTrajectoryMiddleware
from beeai_framework.tools.handoff import HandoffTool
from beeai_framework.tools.search.wikipedia import WikipediaTool
from beeai_framework.tools.think import ThinkTool
from beeai_framework.tools.weather import OpenMeteoTool


async def main() -> None:
    # === 子 Agent 1：知识专家 ===
    knowledge_agent = RequirementAgent(
        llm=ChatModel.from_name("ollama:granite3.3:8b"),
        tools=[ThinkTool(), WikipediaTool()],
        requirements=[ConditionalRequirement(ThinkTool, force_at_step=1)],
        role="Knowledge Specialist",
        instructions="回答关于世界的常识性问题。",
    )

    # === 子 Agent 2：天气预报员 ===
    weather_agent = RequirementAgent(
        llm=ChatModel.from_name("ollama:granite3.3:8b"),
        tools=[OpenMeteoTool()],
        role="Weather Specialist",
        instructions="提供指定目的地的天气预报。",
    )

    # === 主 Agent：协调者 ===
    main_agent = RequirementAgent(
        name="MainAgent",
        llm=ChatModel.from_name("ollama:granite3.3:8b"),
        tools=[
            ThinkTool(),
            HandoffTool(
                knowledge_agent,
                name="KnowledgeLookup",
                description="咨询知识 Agent 回答一般性问题。",
            ),
            HandoffTool(
                weather_agent,
                name="WeatherLookup",
                description="咨询天气 Agent 获取预报。",
            ),
        ],
        requirements=[ConditionalRequirement(ThinkTool, force_at_step=1)],
        middlewares=[GlobalTrajectoryMiddleware(included=[Tool])],
    )

    question = (
        "如果我下周末去罗马旅行，天气会怎么样？"
        "另外告诉我一个著名的历史地标。"
    )
    print(f"User: {question}")

    try:
        response = await main_agent.run(
            question,
            expected_output="有帮助且清晰的回复。",
        )
        print(f"Agent: {response.last_message.text}")
    except FrameworkError as err:
        print(f"Error: {err.explain()}")


if __name__ == "__main__":
    asyncio.run(main())
```

这个示例展示了 BeeAI 的五大核心能力：
1. **Requirement Agent**：声明式约束定义
2. **HandoffTool**：Agent 间任务委派
3. **ConditionalRequirement**：条件化工具调用控制
4. **GlobalTrajectoryMiddleware**：可视化执行轨迹
5. **expected_output**：引导期望输出格式

## 九、总结评价

### 9.1 核心优势

1. **约束系统独一无二**：Requirement Agent 的声明式约束是当前所有主流框架中没有的直接竞品
2. **中间件架构优雅**：Emitter → Run → Runnable 的层次设计兼具灵活性和可维护性
3. **可观测性开箱即用**：不需要额外集成，OTel 支持原生内置
4. **双语言战略前瞻**：PT+TS 对等满足现代全栈团队需求
5. **协议支持领先**：MCP + A2A 双协议原生支持走在行业前列

### 9.2 核心风险

1. **社区规模仍是短板**：GitHub star 数和贡献者数量与头部框架有量级差距
2. **IBM 的"非产品"定位**：官方明确声明不作质量保证，企业用户需谨慎评估
3. **框架竞争白热化**：2025-2026 年 Agent 框架赛道极度拥挤，差异化窗口期有限

### 9.3 推荐指数
