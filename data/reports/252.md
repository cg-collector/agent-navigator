# 📚 AI Agent 项目研究报告 #252 — OS-Copilot：面向通用计算机 Agent 的自改进框架

## 一、项目背景与问题定义

**OS-Copilot** 是由研究团队于 2024 年 2 月提出的**操作系统级通用 Agent 框架**（arXiv:2402.07456），核心目标是构建能够与操作系统中所有元素交互的"通才型"计算机 Agent——包括网页、代码终端、文件系统、多媒体以及各类第三方应用程序。基于该框架实现的 Agent 实例名为 **FRIDAY**（致敬《鲁滨逊漂流记》中自学生存的主人公）。

### 要解决的核心问题

2023-2024 年间，LLM 驱动的数字 Agent 虽然发展迅速，但普遍存在三个结构性缺陷：

1. **领域窄化（Domain Narrowness）**：大多数 Agent 只能操作单一应用或网站（如仅浏览器、仅终端），无法跨应用完成复合任务。用户想"从网页下载文件→用 Excel 分析→用 PPT 制作报告"，需要多个 Agent 手工串联。

2. **泛化能力弱（Poor Generalization）**：Agent 在训练/演示时见过的应用上表现尚可，但面对全新应用时几乎完全失效。本质上是因为缺乏"学会如何学习新应用"的元能力。

3. **无法自我进化**：传统 Agent 的能力在部署时即固定，不会随着使用而提升。每次遇到新任务类型都需要人工介入添加工具或编写 prompt。

OS-Copilot 的回答是：**设计一个统一的 OS 交互接口 + 自指导学习机制 + 多层记忆系统**，让 Agent 像人类新手一样——通过自主探索和试错来掌握新应用的操控能力。

## 二、核心架构深度剖析

### 2.1 整体架构：Sense-Plan-Act 扩展范式

OS-Copilot 在经典的"感知-规划-行动"（Sense-Plan-Act）机器人范式基础上，扩展为五模块架构：

```
┌──────────────────────────────────────────────────────────────┐
│                      用户请求 (自然语言)                        │
└─────────────────────┬────────────────────────────────────────┘
                      ▼
┌──────────────────────────────────────────────────────────────┐
│              Planner (计划器) — DAG 任务分解                    │
│   将复杂请求分解为有向无环图(DAG)结构的子任务                     │
│   支持并行执行无依赖关系的子任务                                 │
└─────────────────────┬────────────────────────────────────────┘
                      ▼
┌──────────────────────────────────────────────────────────────┐
│           Configurator (配置器) — 多层记忆检索                   │
│   ┌──────────────┬──────────────┬──────────────┐              │
│   │ Declarative  │ Procedural   │   Working    │              │
│   │ Memory       │ Memory       │   Memory     │              │
│   │ (声明性记忆)  │ (程序性记忆)  │  (工作记忆)   │              │
│   │ 事实/知识     │ 工具库/技能集  │ 短期上下文    │              │
│   └──────────────┴──────────────┴──────────────┘              │
└─────────────────────┬────────────────────────────────────────┘
                      ▼
┌──────────────────────────────────────────────────────────────┐
│            Actor (执行器) — 执行 + 自我批评循环                  │
│   ┌──────────────┐    ┌──────────────┐    ┌──────────────┐    │
│   │   Executor   │───▶│   Critic     │───▶│   Refiner    │    │
│   │ (执行操作)    │    │ (评估结果)    │    │ (修正改进)    │    │
│   └──────────────┘    └──────────────┘    └──────────────┘    │
└─────────────────────┬────────────────────────────────────────┘
                      ▼
┌──────────────────────────────────────────────────────────────┐
│          统一 OS 接口层 (Unified OS Interface)                 │
│   Python运行时 | Bash运行时 | API调用 | 鼠标/键盘控制           │
└──────────────────────────────────────────────────────────────┘
```

### 2.2 核心组件详解

#### Planner：基于 DAG 的并行任务规划

这是 OS-Copilot 最具工程价值的设计之一。传统 Agent 的 Planner 输出线性任务序列（步骤1→步骤2→步骤3），但实际场景中大量子任务彼此独立。

OS-Copilot 让 LLM 将任务分解为**有向无环图（DAG）**：
- 每个**节点**代表一个原子子任务
- 每条**边**表示依赖关系（A 的输出是 B 的输入）
- 无依赖关系的节点可以**并行分配给不同子 Agent**

例如用户请求"分析销售数据并生成月度报告"：
```
[下载数据] → [数据清洗] → [Excel分析] → [生成图表]
                          ↗
                [收集历史数据] ─┘
                                  → [PPT整合] → [发送邮件]
```

DAG 结构使得 FRIDAY 可以同时派发多个子 Agent 并行工作，大幅缩短端到端执行时间。这在多步复杂任务中是关键的性能优势。

#### Configurator：三层记忆架构

受认知心理学中人类记忆系统的启发，Configurator 实现了三层记忆：

| 记忆层 | 类比 | 存储内容 | 技术实现 | 生命周期 |
|--------|------|----------|----------|----------|
| **Declarative Memory** | 长期语义记忆 | 应用特性、用户偏好、领域知识 | 向量数据库 + 键值存储 | 永久 |
| **Procedural Memory** | 长期程序性记忆 | 可复用的工具函数、技能代码 | Python 函数库 / 代码仓库 | 永久，可增长 |
| **Working Memory** | 工作记忆 | 当前任务的中间状态、执行上下文 | 动态 Context Window | 任务级临时 |

**关键设计——程序性记忆的增长机制**：当 FRIDAY 通过自指导学习掌握了新应用的操作方法后，会将验证通过的代码片段封装为可复用函数存入 Procedural Memory。下次遇到类似需求时直接调用，无需重新探索。这与 Voyager 的技能库思想一脉相承，但 OS-Copilot 的程序性记忆更强调**跨应用的通用工具抽象**（如"读取表格""创建图表"而非 Minecraft 特定技能）。

#### Actor：执行-批评-精炼闭环

Actor 不是简单的"生成代码→执行"单次流程，而是包含三阶段的迭代循环：

1. **Executor**：根据 Configurator 提供的工具和上下文，生成可执行的 Python/Bash 代码并运行
2. **Critic**：分析执行结果（返回值、错误信息、环境状态变化），判断是否达成预期目标；若失败则生成结构化错误诊断
3. **Refiner**：根据 Critic 的反馈修改代码或调整策略，重新提交执行

这个循环最多迭代 N 次（论文中默认 5 次），本质上是将**软件调试过程自动化**——写代码→跑测试→看报错→改 bug→再测。Critic 模块使用 LLM 作为评判者，通过精心设计的 prompt 引导模型关注关键失败信号（异常栈、空返回值、状态未变更等）。

#### 统一 OS 接口层

OS-Copilot 底层封装了四种与操作系统交互的方式：

| 接口类型 | 适用场景 | 示例 |
|----------|----------|------|
| **Python 运行时** | 结构化数据处理 | openpyxl 操作 Excel、python-pptx 操作 PPT |
| **Bash 运行时** | 系统管理、文件操作 | git、curl、文件移动/复制 |
| **API 调用** | Web 服务交互 | Google Search API、天气 API |
| **鼠标/键盘控制** | GUI 自动化 | PyAutoGUI 操作任意桌面应用 |

这种多模态接口设计意味着 FRIDAY 不局限于某一类操作方式——对有 SDK 的应用优先用 API，没有 SDK 的应用回退到 GUI 自动化，实现了**最大化的应用覆盖范围**。

### 2.3 自指导学习（Self-Directed Learning）机制

这是 FRIDAY 区别于其他 Agent 的核心竞争力。其流程如下：

```
给定学习目标（如"掌握 Excel 数据透视表操作"）
        ↓
FRIDAY 自主生成任务课程（Self-Instruct）
  - 简单：创建表格、写入数据
  - 中等：使用公式计算、排序筛选
  - 复杂：创建数据透视表、条件格式
        ↓
逐个解决课程任务（Actor 循环）
        ↓
成功 → 将验证通过的代码封装为工具 → 存入 Procedural Memory
失败 → Critic 反馈 → Refiner 修正 → 重试
        ↓
课程完成后，FRIDAY 已积累一套针对该应用的专属工具集
```

**关键洞察**：这个过程不需要任何人工标注或演示。FRIDAY 像一个自学的程序员——给自己布置练习题、自己写代码调试、把写好的函数存入库中以备后用。论文在 Excel 和 PowerPoint 两个应用上定量证明了这一机制的有效性。

## 三、技术亮点

### 亮点一：DAG 并行任务规划——突破线性执行瓶颈

**原理**：传统 Agent 框架（包括 GPT-Researcher、早期 AutoGPT）的任务规划器输出的是线性步骤列表。这意味着即使步骤 A 和步骤 B 完全独立，Agent 也必须串行执行。对于涉及 10+ 步骤的复杂任务，这造成严重的时间浪费。

OS-Copilot 的 DAG 规划器让 LLM 理解任务间的**逻辑依赖关系**而非仅仅顺序关系。实现上通过 prompt engineering 引导 LLM 输出结构化的 DAG 表示（JSON 格式的节点+边集合），然后使用拓扑排序算法确定可并行执行的节点组。

**深层意义**：这不仅是一个工程优化，更是 Agent 从"单线程思维"走向"分布式思维"的关键一步。真实世界中的人类就是并行处理多任务的（一边下载文件一边整理文档），DAG 规划让 Agent 的行为模式更接近人类。后续的 multi-agent 编排框架（如 CrewAI、LangGraph）都借鉴了这一思想。

### 亮点二：三层记忆架构——认知科学驱动的 Agent 记忆系统

**原理**：大多数 Agent 框架只维护一种记忆（通常是 context window 或简单的向量数据库）。OS-Copilot 明确区分了三种不同性质的记忆：

- **声明性记忆**存储"知道什么"（事实性知识）：Excel 支持数据透视表、PPT 中动画在"动画"选项卡下
- **程序性记忆**存储"会做什么"（操作性知识）：`create_pivot_table(df, rows, values)` 函数
- **工作记忆**存储"正在处理什么"（当前上下文）：当前打开的文件名、已完成的子任务列表

这种分层不是随意设计，而是对应了认知心理学中经过验证的人类记忆模型。声明性记忆和程序性记忆的分离尤其重要——它让 Agent 能够**独立地获取知识和技能**：可以通过阅读文档增加声明性记忆，也可以通过练习增加程序性记忆，两者互不干扰又相互配合。

**与其他项目的对比**：Voyager 只有单一技能库（类似程序性记忆），OpenHands 用文件系统做粗糙的记忆，Computer Use 完全依赖 context window。OS-Copilot 是少数明确设计多层记忆的框架之一。

### 亮点三：零样本应用掌控——自指导学习的泛化奇迹

**原理**：FRIDAY 最令人印象深刻的实验结果是在 **SheetCopilot-20**（电子表格任务基准）上的表现。在没有见过任何 Excel 示例的情况下，仅通过自指导学习：

1. FRIDAY 给自己布置了约 20 个难度递增的 Excel 学习任务
2. 通过 Actor 循环反复试错，逐步掌握 openpyxl 的用法
3. 最终在 SheetCopilot-20 上达到 **60% 成功率**，甚至超过了专门为电子表格设计的专用模型

这说明了一个深刻的事实：**只要给予足够强的 LLM 推理能力和合理的自学习框架，Agent 可以在不依赖任何领域特定训练数据的情况下，达到接近专家的水平。** 这对 GUI Agent 领域具有重大启示——我们可能不需要为每个应用收集海量演示数据，而是应该投资于更好的自学习框架。

**技术前提**：自指导学习能奏效依赖于两个条件：(1) 目标应用有可编程接口（如 Python SDK），(2) LLM 具备足够的代码生成和调试能力。对于纯 GUI 应用（无可编程接口），这一方法的适用性受限。

## 四、基准测试表现

### GAIA 基准测试（General AI Assistants Benchmark）

GAIA 包含 466 个高难度的问答式任务，要求 Agent 在多应用环境中搜索、推理并给出答案。

| 方法 | Level-1 成功率 | 相对提升 |
|------|---------------|---------|
| **FRIDAY (OS-Copilot)** | **40.86%** | **+35%** |
| ReAct + GPT-4 | ~30% | 基线 |
| AutoGPT | ~15% | - |
| BabyAGI | ~10% | - |

FRIDAY 在 GAIA 上相比之前最佳方法提升了 **35 个百分点**，证明其在跨应用通用任务上的显著优势。

### SheetCopilot-20（电子表格专项）

| 方法 | 成功率 | 备注 |
|------|--------|------|
| **FRIDAY（自指导学习后）** | **60%** | 零样本起点，自主学习 |
| SheetCopilot（专用模型） | ~55%** | 专门为表格任务训练 |

*注：FRIDAY 甚至超过了专用模型，展示了自指导学习的强大潜力。*

### PowerPoint 自学实验

论文还报告了 FRIDAY 在 PowerPoint 上的自学效果：通过自指导学习，FRIDAY 积累了 30+ 个 PPT 操作工具函数，能够完成创建幻灯片、插入图表、设置格式、添加动画等多样化任务。虽然未给出量化分数，但定性案例展示了完整的自学到应用的全流程。

| 维度 | OS-Copilot/FRIDAY | Voyager | UI-TARS-2 | Skyvern | OpenHands |
|------|-------------------|---------|-----------|---------|-----------|
| **定位** | OS 级通用 Agent | 游戏内具身 Agent | 原生 GUI 模型 | 浏览器 Agent | 通用开发 Agent |
| **交互方式** | 代码/API/GUI 多模态 | 游戏代码 API | 截图→坐标 | Playwright+CV | 代码终端 |
| **任务规划** | **DAG 并行规划** | 自动课程 | System-2 推理 | Planner-Actor | 多框架可选 |
| **记忆系统** | **三层记忆（声明/程序/工作）** | 向量技能库 | 隐式（模型权重） | 工作记忆 | 文件系统 |
| **学习能力** | **自指导学习（核心）** | 自动课程+技能库 | RL 数据飞轮 | 无显式学习 | 有限 |
| **跨应用能力** | ✅ 强（Web+Office+终端+GUI） | ❌ 仅 Minecraft | ✅ Win/Android/Web | ❌ 仅浏览器 | ⚠️ 主要代码 |
| **开源** | ✅ 开源 | ✅ 开源 | ✅ 开源 | ✅ 开源 | ✅ 开源 |
| **最强基准** | GAIA 40.86% | Minecraft 63物品 | OSWorld 47.5 | WebVoyager 85.85% | SWE-bench ~30% |

**核心差异化总结**：

1. **vs Voyager**：两者都有"自学习"能力，但 Voyager 的环境是封闭游戏世界（Minecraft），OS-Copilot 面向真实 OS 的开放环境，挑战更大且实用价值更高
2. **vs UI-TARS**：UI-TARS 走原生模型路线（端到端训练），OS-Copilot 走框架路线（LLM + 工程设计）。前者在感知精度上有优势，后者在灵活性和可解释性上更强
3. **vs Skyvern**：Skyvern 是浏览器垂直领域的深度优化方案，OS-Copilot 是横跨多应用类型的水平方案。两者互补而非替代
4. **独特优势**：OS-Copilot 是唯一同时具备 **DAG 并行规划 + 三层记忆 + 自指导学习** 三大特性的开源框架

## 六、对实际工作的启发

### 1. DAG 并行规划可直接应用于 OpenClaw 的复杂任务编排

当学长下达涉及多个独立子任务的需求时（如"调研 A 项目、分析 B 股票、同时准备 C 文档"），可以用 DAG 思想分解任务并并行派发 subagent，而不是串行排队。这将显著缩短复杂任务的交付时间。

### 2. 三层记忆架构值得借鉴到楠楠的记忆系统设计

当前楠楠的记忆主要靠 daily notes + MEMORY.md 文件，比较扁平。参考 OS-Copilot 的分层思路：
- **声明性记忆** = MEMORY.md（学长偏好、领域知识、长期事实）
- **程序性记忆** = scripts/ 目录 + skills/（可复用的工具函数和技能脚本）
- **工作记忆** = 当前对话上下文 + session 状态

这种显式分类可以让记忆的管理和检索更有针对性，避免所有东西混在一起导致检索效率下降。

### 3. 自指导学习对 Agent 能力增长的启示

FRIDAY 证明了 Agent 可以通过"自己给自己出题→自己做→把经验存起来"的方式来掌握新技能。这对楠楠意味着：当遇到不熟悉的新工具或新场景时，可以主动进行小规模的自学实验（比如尝试几次飞书 API 调用，把成功的模式记录下来），而不是每次都从零开始摸索。

### 4. "代码即动作"的多模态接口策略

OS-Copilot 对有 SDK 的应用用代码操作、对没 SDK 的应用用 GUI 自动化的混合策略非常务实。在实际工作中，优先使用飞书 API（代码接口），当 API 不满足需求时回退到浏览器自动化（GUI 接口），这与 OS-Copilot 的统一接口层思想一致。

### 5. Critic-Refiner 循环的质量保障价值

每次工具调用后加入自动化的结果检查和错误修复环节，比"调用完就默认成功"可靠得多。特别是在飞书文档操作、消息发送等场景中加入隐式验证，可以提前发现并修复问题。
