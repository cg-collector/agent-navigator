## 架构设计

### 核心哲学：Less is More

Swarm 的设计哲学可以用一句话概括：**把多 Agent 编排的复杂度压到最低**。整个框架只有 **~300 行核心代码**（`core.py` 292行），不依赖任何重型中间件。

### 技术栈

```
Python 3.10+
├── openai (Chat Completions API) — 唯一的外部依赖
└── 纯 Python 原生类型 — 无框架绑定
```

### 两大原语

#### 1. Agent — 最小化智能体单元

```python
class Agent:
    name: str = "Agent"
    model: str = "gpt-4o"
    instructions: str | Callable = "You are a helpful agent."
    functions: List[Callable] = []
    tool_choice: Optional[str] = None
```

Agent 本质上就是 **system prompt + tool set** 的封装：
- `instructions` 可以是字符串或函数（支持 `context_variables` 动态注入）
- `functions` 是普通 Python 函数列表，自动转为 JSON Schema 传给 Chat Completions API 的 `tools` 参数
- **完全无状态** — 不持有任何运行时状态

#### 2. Handoff — 显式控制权转移

```python
def transfer_to_sales():
    return sales_agent  # 返回 Agent 实例即触发 handoff

# 更精细的控制：Result 对象
def talk_to_sales():
    return Result(
        value="Done",
        agent=sales_agent,
        context_variables={"department": "sales"}
    )
```

Handoff 是 Swarm 最精妙的设计——**函数返回值决定控制流**：
- 返回 `str` → 普通工具调用结果
- 返回 `Agent` 实例 → 触发 Agent 切换
- 返回 `Result(value, agent, context_variables)` → 同时返回值、切换 Agent、更新上下文

### 工作流程（core.py 核心循环）

```
client.run(agent, messages, context_variables)
    │
    ▼
┌─────────────────────────────────┐
│  WHILE True (max_turns 限制):     │
│                                   │
│  ① 调用当前 Agent 的              │
│     chat.completions.create()     │
│     (instructions → system prompt) │
│                                   │
│  ② 解析 response:                 │
│     ├─ 无 tool_calls → RETURN     │
│     └─ 有 tool_calls ↓            │
│                                   │
│  ③ 执行每个 tool_call:            │
│     ├─ 函数返回 str → 追加消息    │
│     ├─ 函数返回 Agent → 切换Agent  │
│     └─ 函数返回 Result → 三合一    │
│                                   │
│  ④ 更新 context_variables         │
│  ⑤ 继续循环 (当前或新 Agent)       │
└─────────────────────────────────┘
    │
    ▼
Response(messages, agent, context_variables)
```

关键点：
- **每次循环只有一个 Agent 处于活跃状态**（system prompt 随之切换）
- **聊天历史完整保留**（只换 system prompt，不丢上下文）
- **无服务端状态** — 所有状态通过 `context_variables` 在调用间显式传递

### 源码结构（极简）

```
swarm/
├── __init__.py      # 导出 Swarm, Agent, Result
├── core.py          # ★ 核心：run() 循环、tool 执行、handoff 逻辑 (~292行)

```

## 技术亮点

### 1. 函数自动 Schema 化（util.py）— 零样板 Tool 定义

Swarm 不要求手写 JSON Schema，而是从 Python 函数签名**自动推断**：

```python
def greet(name, age: int, location: str = "New York"):
    """Greets the user.
    
    Args:
        name: Name of the user.
        age: Age of the user.
        location: Best place on earth.
    """
    print(f"Hello {name}, glad you're {age} in {location}!")
```

自动生成：

```json
{
  "type": "function",
  "function": {
    "name": "greet",
    "description": "Greets the user.\n\nArgs:\n name: ...\n age: ...",
    "parameters": {
      "type": "object",
      "properties": {
        "name": {"type": "string"},
        "age": {"type": "integer"},
        "location": {"type": "string"}
      },
      "required": ["name", "age"]
    }
  }
}
```

映射规则：
- **docstring** → `description`
- **无默认值的参数** → `required`
- **type hints** → JSON 类型映射（`int` → `integer`, `str` → `string`）
- 使用 jinja2 模板引擎做 schema 渲染

**启发意义**：这比 LangChain/PydanticAI 的装饰器方案更简洁。OpenClaw 的 skill/tool 注册可以借鉴这种"写普通 Python 函数就自动变成 LLM tool"的模式。

### 2. 动态 Instructions — Context-Aware System Prompt

```python
def instructions(context_variables):
    user_name = context_variables["user_name"]
    language = context_variables.get("language", "English")
    return f"""You are a helpful assistant for {user_name}.
Always respond in {language}.
Today's date is {context_variables.get('date', 'unknown')}."""

agent = Agent(instructions=instructions)

response = client.run(
    agent=agent,
    messages=[{"role": "user", "content": "Hi!"}],
    context_variables={"user_name": "John", "language": "Spanish", "date": "2026-04-04"}
)
```

`instructions` 作为 callable 接收 `context_variables`，每次 run 时动态生成 system prompt。这意味着：

- **同一个 Agent 可以根据用户/场景自适应**
- 无需创建大量 Agent 子类，一个 Agent + 动态指令即可
- `context_variables` 在 handoff 过程中累积更新，形成全局上下文传播链

**代码级洞察**：在 `core.py` 中，每次循环开始时都会重新调用 `instructions(context_variables)` 生成最新的 system prompt，确保 prompt 始终反映最新上下文。

### 3. Result 对象 — 原子化的三合一返回值

```python
@dataclass
class Result:
    value: str = ""           # 返回给用户的文本
    agent: Optional[Agent] = None  # 要切换到的 Agent
    context_variables: dict = field(default_factory=dict)  # 要更新的变量
```

这个设计的优雅之处在于**一次函数调用同时完成三个动作**：

```python
def escalate_to_human(context_variables, issue):
    # 一条 return 完成：回复用户 + 切换到人工客服 + 记录工单号
    return Result(
        value=f"I've escalated your issue ({issue}). A human will contact you soon.",
        agent=human_agent,
        context_variables={"ticket_id": generate_ticket(), "escalated": True}
    )
```

对比其他框架的实现方式：
- **LangGraph**: 需要定义条件边（conditional edge）+ 节点状态机
- **CrewAI**: 需要 Task 委托机制
- **Swarm**: 一个 `return` 语句搞定

### 4. 完全客户端执行 — 零服务端依赖

Swarm **100% 基于 Chat Completions API**，不使用 Assistants API、不存储线程状态、不需要 Vector Store。

这意味着：
- **完全可调试** — 每一步都是可见的 API 调用
- **完全可测试** — 可以 mock OpenAI client 做单测
- **模型无关** — 理论上任何支持 function calling 的模型都能跑
- **成本透明** — 每个 token 都在你的控制之下

```python
# 测试示例（来自官方 airline eval）
def test_agent_handoff():
    client = Swarm()
    response = client.run(
        agent=triage_agent,
        messages=[{"role": "user", "content": "I lost my luggage"}],
        execute_tools=True,
    )
    assert response.agent.name == "Loss Agent"  # 断言被正确路由
```

### 5. 流式输出中的 Agent 切换事件

Swarm 在标准 SSE streaming 之外增加了两种自定义事件：

```json
{"delim": "start"}   // Agent 开始处理新消息
{"delim": "end"}     // Agent 处理完毕
{"response": Response} // 最终聚合结果
```

这让前端可以实时感知 Agent 切换时机，对 UI 展示非常有价值（比如显示"正在转接给销售专家..."）。

## 对 OpenClaw / 楠楠的启发

### 🔑 启发一：Handoff 模式可以直接用于 Skill 路由

OpenClaw 当前的 skill 选择逻辑是基于 description 匹配。Swarm 的 handoff 思路提供了一个更优雅的替代方案：

**现状**：主 Agent 收到任务 → 匹配 skill description → 执行
**Swarm 启发**：主 Agent 收到任务 → 调用 `transfer_to_coding_skill()` → 控制权交给 coding sub-agent → 完成后返回结果

具体来说，可以把每个 OpenClaw Skill 包装成一个 Swarm-style Agent，让 LLM 通过 function calling **自主选择**何时转移控制权，而不是靠前置的描述匹配。这样更灵活，也更容易处理"需要多个 skill 协作"的场景。

### 🔑 启发二：动态 Instructions 用于 Persona 切换

楠楠有多个 persona（乖巧学妹 / 骚母狗 / 能干助手），当前的切换可能基于规则。Swarm 的 `callable instructions` + `context_variables` 模式提供了一种更自然的方式：

```python
def get_persona_instructions(cv):
    mode = cv.get("persona_mode", "default")
    if mode == "flirty":
        return "你是学长的小荡妇..."
    elif mode == "work":
        return "你是专业的技术助手..."
    return default_instructions
```

结合 `context_variables` 中的对话历史情感分析，可以实现**平滑的 persona 渐变**而非硬切换。

### 🔑 启发三：Result 三合一模式用于 Sub-agent 协作

当前 OpenClaw 的 sub-agent 模式是 spawn-and-forget（派生后等结果）。Swarm 的 `Result(value, agent, context_variables)` 启发了更精细的模式：

- **value**: sub-agent 的产出（报告、代码等）
- **agent**: 是否需要链式调用下一个 sub-agent（比如先研究再写代码）
- **context_variables**: 积累的研究发现、决策依据等

这样可以构建 **multi-hop sub-agent pipeline**，一个 sub-agent 的输出直接驱动下一个的选择。

### 🔑 启发四：极简架构的设计哲学

Swarm 用 300 行代码解决了其他框架几千行才解决的问题。核心教训：

1. **不要过度抽象** — Agent 就是 prompt + tools，不要搞复杂的基类继承体系
2. **显式优于隐式** — handoff 是显式函数调用，不是隐式的图遍历
3. **状态外置** — 不在框架内管理状态，让调用方自己管理
4. **可测试性是第一优先级** — 纯函数、无副作用、容易 mock

对于 OpenClaw 的 skill 系统，可以考虑是否有些模块可以大幅简化而不损失功能。

### 🔑 启发五：Tool Schema 自动推断

Swarm 从 Python 函数签名自动生成 JSON Schema 的做法值得借鉴。当前 OpenClaw 的 tool 定义可能需要手动编写 schema。如果采用类似方案：

```python
# 开发者只需写普通函数
def search_web(query: str, count: int = 5) -> str:
    """Search the web for information."""
    ...

# 框架自动变为 LLM 可调用的 tool
```

这能显著降低开发新 skill/tool 的门槛。

## 总结评价

**Swarm 是一把手术刀** — 它不做所有事，但把它做的事做到了极致。作为 OpenAI 官方的教育性框架，它展示了多 Agent 系统可以有多简单。

虽然已被 Agents SDK 取代，但 Swarm 的**设计思想**远超其"实验项目"的定位：
- Handoff 原语比图编排更直觉
- 无状态设计比内置记忆更可控
- 300 行代码的可维护性是所有 Agent 框架的标杆

**推荐阅读顺序**：`types.py` → `core.py`（run 循环）→ `util.py`（schema 转换）→ examples 目录

> ⚠️ 注意：生产环境请使用 [OpenAI Agents SDK](https://github.com/openai/openai-agents-python)，它是 Swarm 的正式继任者，增加了 tracing、guardrails 等企业级特性。
