## 一、一句话简介

SWE-agent 是普林斯顿大学提出的**软件工程自主修复 Agent**，能读取 GitHub Issue 并自动在真实代码仓库中定位和修复 Bug；其精简版 **Mini-SWE-Agent** 仅用 ~100 行 Python 核心代码就达到了 SWE-bench Verified **74%+** 的修复率，是"极简主义 Agent 设计"的标杆项目。

### 2.2 Mini-SWE-Agent 极简架构（v2）

```
┌─────────────────────────────────────────┐
│           DefaultAgent.run()             │
│                                         │
│  messages = []                          │
│  ├── system: 渲染 system_template        │
│  ├── user:   渲染 instance_template      │
│  │                                     │
│  └── while not exit:                    │
│       ├── query() → 调 LLM → 解析 actions│
│       └── execute_actions() → subprocess │
│           └── 追加 observation → messages │
│                                         │
│  return {exit_status, submission}        │
└─────────────────────────────────────────┘
         │           │           │
    ┌────▼──┐   ┌────▼────┐  ┌▼──────────┐
    │ Model │   │  Env    │  │  Config    │
    │litellm│   │local/docker│  │(Pydantic) │
    └───────┘   └─────────┘  └───────────┘
```

**核心设计哲学（3 条）：**
1. **唯一工具 = Bash** — 不定义任何专用工具，LM 直接用 shell 命令完成一切操作
2. **无状态执行 = `subprocess.run`** — 每次 action 独立执行，不维护持久 shell 会话
3. **线性历史 = Messages 列表** — trajectory 就是 messages 本身，零转换开销

## 三、技术亮点（代码级深度分析）

### 亮点 1：ACI（Agent-Computer Interface）—— "给 LM 设计 UI"

这是 SWE-agent 论文最核心的贡献。团队发现：**Agent 的表现不取决于模型能力，而取决于你给它设计的"人机界面"好坏。**

具体设计决策：

| ACI 元素 | 设计 | 效果 |
|---------|------|------|
| **文件查看器** | 自定义 `view` 命令，每次只显示 100 行 + 支持滚动/搜索 | 避免长文件撑爆 context |
| **全局搜索** | 只列出匹配的**文件名**，不显示匹配内容上下文 | 减少信息噪音 |
| **编辑命令带 Lint** | 执行 edit 前先跑语法检查，失败则拒绝 | 阻止错误代码进入代码库 |
| **空输出处理** | 命令无输出时返回 `"Your command ran successfully..."` | 避免 LM 对空输出困惑 |

**代码级体现（完整版 SWE-agent 的 ACI 工具定义）：**

```python
# SWE-agent 通过 YAML 定义工具，每个工具有 name/description/arguments
# 例如 edit 工具会在执行前调用 Python AST linter
def _check_syntax(code: str) -> bool:
    try:
        ast.parse(code)
        return True
    except SyntaxError as e:
        return False
```

**启发：** OpenClaw 的 tool 设计也可以借鉴这个思路——不是工具越多越好，而是每个工具的**输出格式**要为 LLM 的理解优化。比如文件读取应该分页、搜索结果应该精简。

### 亮点 2：极简 Agent 循环 —— DefaultAgent 核心代码

这是整个 mini-swe-agent 的灵魂，**仅约 120 行 Python**：

```python
class DefaultAgent:
    def __init__(self, model: Model, env: Environment, *, config_class=AgentConfig, **kwargs):
        self.config = config_class(**kwargs)
        self.messages: list[dict] = []
        self.model = model
        self.env = env
        self.cost = 0.0
        self.n_calls = 0

    def run(self, task: str = "", **kwargs) -> dict:
        """主循环：初始化 → 循环 step → 直到退出"""
        self.extra_template_vars |= {"task": task, **kwargs}
        self.messages = []
        # 1. 构建初始消息：system prompt + task 描述
        self.add_messages(
            self.model.format_message(role="system", 
                content=self._render_template(self.config.system_template)),
            self.model.format_message(role="user",
                content=self._render_template(self.config.instance_template)),
        )
        # 2. 循环执行直到 agent 输出 exit 角色
        while True:
            try:
                self.step()
            except InterruptAgentFlow as e:
                self.add_messages(*e.messages)
            except Exception as e:
                self.handle_uncaught_exception(e)
                raise
            finally:
                self.save(self.config.output_path)
            if self.messages[-1].get("role") == "exit":
                break
        return self.messages[-1].get("extra", {})

    def step(self):
        """单步：查询模型 → 执行动作"""
        return self.execute_actions(self.query())

    def query(self):
        """调用 LLM API，解析返回的动作"""
        if 0 < self.config.step_limit <= self.n_calls or 0 < self.config.cost_limit <= self.cost:
            raise LimitsExceeded(...)
        self.n_calls += 1
        message = self.model.query(self.messages)
        self.cost += message.get("extra", {}).get("cost", 0.0)
        self.add_messages(message)
        return message

    def execute_actions(self, message):
        """执行模型返回的所有 bash 动作"""
        outputs = [self.env.execute(action) 
                   for action in message.get("extra", {}).get("actions", [])]
        return self.add_messages(
            *self.model.format_observation_messages(message, outputs, self.get_template_vars()))
```

**关键设计细节：**
- **Jinja2 模板系统**：system prompt 和 task 描述都用 Jinja2 渲染，支持注入环境变量、模型信息等动态内容
- **异常即控制流**：`InterruptAgentFlow`（正常中断如任务完成）、`LimitsExceeded`（超限）、`Submitted`（提交结果）——用异常实现优雅的状态跳出
- **每步保存轨迹**：`finally` 块中调用 `save()`，即使崩溃也能保留完整历史

### 亮点 3：无状态 Bash 执行 —— LocalEnvironment

```python
class LocalEnvironment:
    def execute(self, action: dict, cwd="", *, timeout=None) -> dict:
        command = action.get("command", "")
        cwd = cwd or self.config.cwd or os.getcwd()
        try:
            result = subprocess.run(
                command,
                shell=True,          # 直接 shell 执行，LM 可以用管道、重定向等
                text=True,
                cwd=cwd,
                env=os.environ | self.config.env,
                timeout=timeout or self.config.timeout,
                encoding="utf-8",
                errors="replace",     # 编码容错
                stdout=subprocess.PIPE,
                stderr=subprocess.STDOUT,  # 合并 stderr 到 stdout
            )
            output = {"output": result.stdout, 
                      "returncode": result.returncode, 
                      "exception_info": ""}
        except Exception as e:
            output = {"output": raw_output, "returncode": -1, ...}
        
        self._check_finished(output)  # 检测 COMPLETE_TASK_AND_SUBMIT
        return output
```

**为什么 `subprocess.run` 比 shell session 好？**
1. **天然沙箱兼容**：把 `subprocess.run` 替换为 `docker exec` 就实现了容器隔离，无需改 Agent 逻辑
2. **无状态 = 无副作用**：不会因为上一次命令的 cd/环境变量影响本次执行
3. **可重现性**：每条命令独立可审计
4. **并行潜力**：多个 action 理论上可以并行执行

### 亮点 4：Model 层的统一抽象与 Action 解析

```python
class LitellmModel:
    def query(self, messages, **kwargs) -> dict:
        # 1. 重试机制（指数退避）
        for attempt in retry(logger=logger, abort_exceptions=self.abort_exceptions):
            with attempt:
                response = litellm.completion(
                    model=self.config.model_name,
                    messages=self._prepare_messages_for_api(messages),
                    tools=[BASH_TOOL],  # ← 只注册一个 bash 工具
                    **(self.config.model_kwargs | kwargs),
                )
        # 2. 成本追踪
        cost_output = self._calculate_cost(response)
        # 3. 解析 tool_call 为结构化 actions
        message = response.choices[0].message.model_dump()
        message["extra"] = {
            "actions": self._parse_actions(response),  # 从 tool_calls 提取
            "response": response.model_dump(),
            **cost_output,
            "timestamp": time.time(),
        }
        return message
```

**Action 解析链路：**

```
LLM Response (tool_calls)
  → parse_toolcall_actions()     # 提取 function name + arguments
    → 仅接受 BASH_TOOL ("bash")
      → {"command": "grep -r 'bug' src/"}
        → env.execute(action)     # subprocess.run
          → {"output": "...", "returncode": 0}
            → format_observation_messages()  # 用 Jinja2 模板格式化
              → <returncode>0</returncode><output>...</output>
                → 追加到 messages → 下轮 query
```

**Observation 格式化模板（关键！）：**

```python
observation_template = """
{% if output.exception_info %}<exception>{{output.exception_info}}</exception>{% endif %}
<returncode>{{output.returncode}}</returncode>
<output>
{{output.output}}
</output>
"""
```

用 XML 标签包裹结构化信息，让 LLM 能清晰区分异常、退出码和标准输出。这比纯文本拼接聪明得多。

### 亮点 5：配置驱动的 Prompt 工程

```python
class AgentConfig(BaseModel):
    system_template: str      # System prompt 模板
    instance_template: str    # Task 描述模板
    step_limit: int = 0       # 步数上限（0=无限）
    cost_limit: float = 3.0   # 成本上限（美元）
    output_path: Path | None  # 轨迹保存路径
```

所有行为通过 **YAML + Jinja2 模板** 控制，不改代码就能调整：
- 切换 system prompt 风格
- 调整成本/步数限制
- 改变 observation 格式
- 注入自定义变量

这意味着 **prompt engineering 和 code engineering 完全解耦**。

## 五、对 OpenClaw 的启发

### 5.1 ✅ 已做得好的（确认现有优势）
- OpenClaw 的 subagent 机制本质上就是类似 mini-swe-agent 的 `run()` 循环
- Tool call → execute → observe → 回传的闭环已经成熟
- 配置化设计（skills、yaml config）思路一致

### 5.2 🔥 值得借鉴的方向

#### ① ACI 思维：Tool Output 格式要为 LLM 优化
**现状：** OpenClaw 的 tool 返回值多为 raw JSON/text，没有统一的"展示层"。  
**建议：** 参考 SWE-agent 的 `<returncode><output>` XML 包装，给 tool output 加一层 **LLM-friendly formatting**：
- 文件内容自动分页（超过200行时折叠中间部分）
- 搜索结果只显示文件名+行号，默认不展开上下文
- 错误信息用 `<error>` 标签高亮
- 空操作明确返回成功提示

#### ② 无状态执行 > 有状态 Session
**现状：** OpenClaw exec 支持 PTY session（用于交互式 CLI）。  
**建议：** 对于非交互场景，优先使用 `subprocess.run` 式的无状态执行。好处：
- 天然可序列化/重现
- 更容易迁移到沙箱/容器
- 并行执行多个独立命令时不互相干扰

#### ③ 异常即控制流模式
**现状：** OpenClaw subagent 主要靠返回值判断完成/失败。  
**建议：** 引入 SWE-agent 的结构化异常体系：
```python
class TaskComplete(Exception): pass   # 正常完成
class BudgetExhausted(Exception): pass  # token/成本超限
class UserInterrupt(Exception): pass    # 用户打断
```
让 subagent 的生命周期管理更清晰，避免 `if/else` 地狱。

#### ④ Trajectory 即 Messages
**现状：** OpenClaw subagent 的历史管理和消息传递有额外抽象层。  
**建议：** 对于轻量 subagent，考虑"trajectory = messages list"的最简设计。这使：
- 调试时直接打印 messages 就是完整轨迹
- 微调数据生成零成本（直接 dump messages）
- 缓存/断点续跑更简单

#### ⑤ 成本门控内置
**现状：** 成本监控主要在 API 层面。  
**建议：** 在 Agent 循环层面加 `cost_limit` 和 `step_limit` 双重门控，像 mini-swe-agent 那样在每次 `query()` 前检查。防止一个 runaway agent 烧光预算。

#### ⑥ Observation Template 可配置
**现状：** Tool output 格式基本固定。  
**建议：** 允许 per-tool 或 per-agent 配置 observation 渲染模板。不同模型对不同格式的敏感度差异很大（比如 XML 标签对某些模型效果更好，Markdown 对另一些更好）。
