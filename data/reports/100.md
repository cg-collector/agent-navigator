# AI Agent 学习报告 #100 — LangGraph：生产级有状态图执行引擎

## 项目基本信息

- **项目名**：LangGraph
- **GitHub 链接**：<https://github.com/langchain-ai/langgraph>
- **Star 数**：~25k+（持续快速增长中）
- **一句话简介**：LangChain 团队打造的低抽象、生产级 AI Agent 图执行框架，以 **StateGraph（状态图）** 为核心，将 Agent 工作流建模为带循环的有向图，解决 LLM Agent 的延迟、可靠性、非确定性三大生产难题。

## 一、架构设计

### 1.1 核心设计哲学

LangGraph 的设计出发点非常清晰——**"我们不知道 AI 的未来长什么样，所以假设越少越好"**。它只做了两个核心假设：

1. **LLM 很慢** → 需要并行化 + 流式输出
2. **LLM 不稳定且非确定性** → 需要检查点 + 人工介入 + 可观测性

基于这两个假设，LangGraph 提炼出 **6 大生产级特性**：并行化（Parallelization）、流式输出（Streaming）、任务队列（Task Queue）、检查点（Checkpointing）、人机协作（Human-in-the-Loop）、追踪（Tracing）。

### 1.4 工作流程

```
用户输入
    ↓
写入 Input Channels → 触发订阅节点
    ↓
PregelLoop 主循环:
    ├── 1. 扫描所有 Node，对比 Channel 版本号
    │   └── 找出"有新数据的节点"作为候选
    ├── 2. 并行执行候选节点（每个节点获得 state 的独立副本）
    │   ├── Node A: LLM 调用 → 产生决策
    │   ├── Node B: 工具调用 → 获取外部数据
    │   └── Node C: 自定义逻辑 → 处理业务规则
    ├── 3. 所有节点完成后，确定性合并状态更新
    │   └── 按 Channel 定义顺序 apply，避免竞态条件
    ├── 4. 写入 Checkpoint（如果配置了）
    ├── 5. 发送流式事件给客户端
    └── 6. 回到步骤 1，直到无节点需要执行或达到迭代上限
    ↓
从 Output Channels 读取最终结果
```

**关键区别于 DAG 框架**：LangGraph 支持**边形成环（Cycle）**，即 Agent 可以循环执行（如 ReAct 模式的 思考→行动→观察→思考 循环）。这是传统 DAG 工作流框架（Airflow 等）做不到的。

## 二、技术亮点（代码级别）

### 亮点 1：Pregel/BSP 执行算法 — 确定性并发无竞态

这是 LangGraph 最核心的创新。传统并发最大的问题是**非确定性**——哪个线程先完成会影响结果。LangGraph 用 Pregel 算法彻底解决了这个问题：

```python
# 核心执行伪代码（简化版）
class PregelLoop:
    def run(self, input_values):
        # 1. 初始化 channels
        self.channels.initialize(input_values)
        
        while True:
            # 2. 选择待执行节点：对比 channel 当前版本 vs 节点上次看到的版本
            pending_nodes = []
            for node in self.nodes:
                if node.has_new_data(self.channels):
                    pending_nodes.append(node)
            
            if not pending_nodes:
                break  # 终止条件：没有节点需要执行
            
            # 3. 并行执行，每个节点获得 state 的独立副本
            local_states = {}
            results = await asyncio.gather(*[
                node.run(self.channels.snapshot()) 
                for node in pending_nodes
            ], return_exceptions=True)
            
            # 4. 确定性合并！按固定顺序 apply 更新
            for node, result in zip(pending_nodes, results):
                self.channels.apply_update(node, result)
                self.channels.bump_version()  # 单调递增版本号
            
            # 5. 检查点
            if self.checkpointer:
                self.checkpointer.save(self.channels.snapshot())
```

**为什么这很重要？**
- 并行节点的执行顺序**不影响最终结果**（确定性合并）
- LLM 本身是非确定性的，但框架层面不会引入额外的不确定性
- 调试时可以精确复现问题

### 亮点 2：TypedDict 状态 Schema — 类型安全的状态管理

LangGraph 使用 Python `TypedDict` 或 `pydantic.BaseModel` 定义状态结构，实现**细粒度读写控制**：

```python
from typing import TypedDict, Annotated
from operator import add

class AgentState(TypedDict):
    # 每个 key 对应一个 channel
    messages: Annotated[list, add]  # add reducer：新值追加到旧值
    current_user: str               # 默认 reducer：覆盖
    loop_count: int                 # 默认 reducer：覆盖

# 构建状态图
graph = StateGraph(AgentState)

# 节点函数接收完整 state，返回需要更新的字段
def chatbot(state: AgentState):
    messages = state["messages"]
    response = llm.invoke(messages)
    return {"messages": [response]}  # 只更新 messages，通过 add 追加

def human_check(state: AgentState):
    # 可以读取完整 state，选择性更新
    return {"current_user": "approved", "loop_count": state["loop_count"] + 1}

graph.add_node("chatbot", chatbot)
graph.add_node("human_check", human_check)

# 条件边：根据状态动态决定下一步
def should_continue(state):
    if state["loop_count"] > 3:
        return END
    return "human_check"

graph.add_conditional_edges("chatbot", should_continue)
```

**Reducer 模式的精妙之处**：
- `add` reducer：适合消息列表、日志等追加型数据
- 默认覆盖：适合当前状态、计数器等
- 自定义 reducer：可以实现任意合并策略
- 这让**并行节点写同一个字段时不会冲突**

### 亮点 3：检查点驱动的 Human-in-the-Loop

这不是简单的 `input()` 暂停，而是**真正的持久化中断**：

```python
from langgraph.checkpoint.memory import MemorySaver
from langgraph.graph import StateGraph, START, END

# 配置检查点存储
checkpointer = MemorySaver()
graph = StateGraph(AgentState, checkpointer=checkpointer)

# 在特定节点后设置中断点
graph.add_node("dangerous_action", call_api)
graph.add_edge(START, "agent")
graph.add_conditional_edges("agent", route_next_step)

# 编译时指定中断策略
app = graph.compile(
    interrupt_before=["dangerous_action"],  # 在执行前暂停
    interrupt_after=["research"],           # 在执行后暂停
)

# 第一次运行 — 会在 interrupt 点自动停止
config = {"configurable": {"thread_id": "user-123"}}
result = app.invoke({"messages": [user_input]}, config)
# → 运行到 dangerous_action 前自动暂停，保存完整 checkpoint

# 用户审核后恢复执行 — 从断点继续，不重跑之前的步骤
update_result = app.invoke(
    None,  # None 表示继续，不需要重新输入
    config,
    # 可选：在恢复前修改状态！
    # as_node_updates={"dangerous_action": {"api_params": {...}}}
)
```

**关键能力**：
- 中断后进程可以完全退出，**几天甚至几个月后恢复**
- 恢复可以在**不同的机器**上进行（checkpoint 是序列化的）
- 恢复前可以**修改状态**（编辑下一步动作）
- 支持时间旅行（Time Travel）：回到任意历史 checkpoint 重新执行

### 亮点 4：6 种流式输出模式

LangGraph 不是简单地把 LLM token stream 出来，而是在图的每个层级提供不同粒度的流式信息：

```python
# 模式 1: values — 每步执行后的完整状态快照
async for chunk in app.stream(input, config, stream_mode="values"):
    print(chunk)  # {"messages": [...], "current_user": ...}

# 模式 2: updates — 仅该步产生的增量更新（节省带宽）
async for chunk in app.stream(input, config, stream_mode="updates"):
    print(chunk)  # ("chatbot", {"messages": [AIMessage(...)]})

# 模式 3: messages — 专门针对聊天场景，stream LLM tokens
async for chunk in app.stream(input, config, stream_mode="messages"):
    print(chunk.content, end="", flush=True)  # 逐 token 输出

# 模式 4: tasks — 子任务级别的进度（多 agent 场景）
# 模式 5: checkpoints — 每次 checkpoint 保存时的通知
# 模式 6: custom — 开发者自定义的流式事件
```

这种分层设计让前端可以根据场景选择最合适的粒度：
- 聊天机器人用 `messages` 模式（逐 token 显示）
- 长任务 Agent 用 `updates` 模式（显示每步进展）
- 调试用 `values` 模式（看完整状态变化）

## 三、对我们（OpenClaw/楠楠）的启发

### 启发 1：结构化执行 > 单体大循环

OpenClaw 当前的 agent 执行模型偏向"一个函数里做所有事"。LangGraph 证明了**将 Agent 拆分为离散的结构化步骤**是获得检查点、HITL、并行化等能力的**前提条件**。即使不做完整的图系统，至少可以将 agent 的生命周期划分为明确的阶段（输入解析 → 工具选择 → 执行 → 结果验证 → 输出格式化），每个阶段作为独立的"节点"。

**具体建议**：考虑为 OpenClaw 的 skill 执行流程引入类似的概念——将一次复杂的 agent 调用分解为可观测、可中断、可重试的阶段。

### 启发 2：Reducer 模式管理共享状态

当前 OpenClaw 的 memory 文件系统本质上是扁平的键值存储。LangGraph 的 Reducer 模式（`add` vs 覆盖 vs 自定义合并）给了很好的启发：

- **日志类数据**（每日记录、对话历史）应该用 append-only 模式
- **配置类数据**（用户偏好、身份信息）应该用覆盖模式
- **统计类数据**（恋爱模拟器的亲密度、经验值）应该用自定义聚合

这比简单地读写 JSON 文件更安全，特别是在并发场景下。

### 启发 3：检查点是长任务 Agent 的刚需

楠楠经常执行耗时较长的任务（研究分析、代码生成等）。当前的 subagent 机制虽然有后台执行能力，但**失败后只能从头开始**。引入轻量级检查点机制可以让长任务在中断后从最近的成功步骤恢复，而不是全部重来。

**最小可行实现**：在关键 subagent 的每个主要阶段完成后，将中间结果写入临时文件；失败时检测是否有可用的检查点文件并从中恢复。

### 启发 4：流式模式的分层设计值得借鉴

LangGraph 的 6 种流式模式说明了一个道理：**不同场景需要的流式粒度完全不同**。OpenClaw 目前主要通过 message tool 直接发送消息给用户，可以考虑更丰富的反馈层次：

- **Token 级别**：LLM 正在生成时的逐字显示（已有 TTS 能力）
- **步骤级别**："正在搜索..."、"正在分析第 2 个数据源..."
- **结果级别**：最终报告/答案

特别是对于复杂任务，中间进度的展示能大幅降低用户的等待焦虑。

### 启发 5：Pregel 算法的确定性保证

虽然 OpenClaw 不一定需要完整的图执行引擎，但**"并行执行的顺序不影响结果"**这一原则非常有价值。当未来 OpenClaw 支持并行调用多个工具/skill 时，可以用类似的思路：每个并行分支操作状态的独立副本，最后按固定顺序合并，避免隐含的竞态条件。
