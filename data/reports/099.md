## 一、架构设计总览

### 1.1 核心哲学：Code = SOP(Team)

MetaGPT 的核心设计理念是 **将人类软件公司的 SOP（Standard Operating Procedure）编码进多智能体协作系统**。不同于 ChatDev 等框架让 Agent 自由对话（容易产生"吃了吗"之类的无效闲聊），MetaGPT 强制要求：

1. 每个 Role 有明确的**结构化输出**（PRD、API 设计、数据结构、流程图）
2. 角色间通过**标准文档**交接，而非自由聊天
3. 中间产物可被下游角色**验证和纠错**

### 1.2 核心模块

### 1.3 四大抽象层

| 层级 | 类 | 职责 |
|------|-----|------|
| **Role** | `Role` | Agent 的身份、目标、约束、状态机 |
| **Action** | `Action` | Agent 能执行的具体操作单元 |
| **Environment** | `ExtEnv` / `Team` | 消息总线、共享记忆、角色调度 |
| **Memory** | `Memory` | 角色私有记忆 + 环境全局记忆 |

### 1.4 工作流程（软件公司场景）

```
用户输入: "Create a 2048 game"
        │
        ▼
   ┌─────────────┐
   │  TeamLeader  │ ← 任务分解、需求澄清
   │  (团队负责人)  │
   └──────┬───────┘
          │ publish_message(需求文档)
          ▼
   ┌──────────────┐

   │   (产品经理)    │   输出: 结构化 PRD 文档
   └──────┬────────┘
          │ publish_message(PRD)
          ▼
   ┌──────────────┐
   │   Architect    │ ← 技术选型、API设计、数据结构、流程图
   │    (架构师)     │   输出: 系统设计文档 + API specs
   └──────┬────────┘
          │ publish_message(设计文档)
          ▼
   ┌──────────────┐
   │   Engineer     │ ← 根据设计写代码
   │    (工程师)     │   输出: 完整项目源码
   └──────┬────────┘
          │ publish_message(代码)
          ▼
   ┌──────────────┐
   │  DataAnalyst   │ ← 数据分析支持（可选）
   │  (数据分析师)   │
   └──────────────┘
```

每个角色通过 `publish_message()` 将**结构化输出**发布到 Environment（消息总线），下游角色通过 `subscribed_tags` 订阅并接收。

## 二、技术亮点（代码级别）

### 亮点 1：Role 状态机驱动的行为控制

MetaGPT 的 Role 不是简单的"接收消息→调用LLM→回复"，而是内置了**有限状态机（FSM）**：

```python
# metagpt/roles/role.py — 核心状态模板
STATE_TEMPLATE = """Here are your conversation records. You can decide which stage 
you should enter or stay in based on these records.
...
Your previous stage: {previous_state}
Now choose one of the following stages you need to go to in the next step:
{states}
Just answer a number between 0-{n_states}...
If you think you have completed your goal, return -1.
"""
```

**工作原理：**
- 每个 Role 定义 `states: list[str]`（如产品经理的 states = ["需求分析", "竞品调研", "编写PRD", "评审"]）
- 每次 `_think()` 时，LLM 根据历史记录选择下一个状态编号
- `_act()` 根据 current state 执行对应的 Action
- 返回 `-1` 表示任务完成

**三种 React 模式：**

```python
class RoleReactMode(str, Enum):
    REACT = "react"           # 观察→思考→行动 循环（ReAct模式）
    BY_ORDER = "by_order"     # 按顺序执行所有 Actions
    PLAN_AND_ACT = "plan_and_act"  # 先规划再执行（Planner策略）
```

这对 OpenClaw 的启发：**Agent 行为不应是平铺的函数调用链，而应有明确的状态阶段和转换逻辑。**

### 亮点 2：ActionNode — 结构化输出保证

这是 MetaGPT 最精巧的设计之一。普通的 LLM 调用返回自由文本，难以解析和验证。MetaGPT 通过 `ActionNode` 强制**结构化输出**：

```python
# metagpt/actions/action_node.py (概念示意)
class ActionNode:
    def __init__(self, key, expected_type, instruction, example, schema):
        self.key = key              # 输出字段名
        self.expected_type = expected_type  # 期望类型（str/int/list/dict/object）
        self.instruction = instruction      # 填充指令
        self.example = example              # 示例
        self.schema = schema                # 输出格式（json/markdown）
    
    async def fill(self, req, llm):
        # 构造包含 schema 约束的 prompt
        prompt = self._build_prompt(req)
        output = await llm.aask(prompt)
        return self._parse(output)  # 解析为结构化对象
```

**实际效果：** 产品经理的 `WritePRD` Action 输出的不是一段话，而是一个包含以下字段的**结构化文档**：
- `user_stories`: List[UserStory]
- `requirements`: List[Requirement]
- `competitive_analysis`: Dict
- `ui_draft`: str

这保证了下游架构师拿到的输入是**可解析、可验证的**，大幅减少幻觉传播。

### 亮点 3：基于标签的消息订阅与路由系统

```python
# metagpt/roles/role.py — RoleContext
class RoleContext(BaseModel):
    msg_buffer: MessageQueue = Field(default_factory=MessageQueue)
    watch: set[str] = Field(default_factory=set)  # 订阅的消息标签
    memory: Memory = Field(default_factory=Memory)
    state: int = Field(default=-1)
    todo: Action = Field(default=None)

# metagpt/schema/Message — 消息携带 cause_by 标签
class Message(BaseModel):
    content: str
    role: str
    cause_by: Type[Action]  # 由哪个 Action 产生 → 用于路由匹配
    
# Environment 中的消息分发逻辑（简化）
def publish_message(self, message):
    self.memory.add(message)
    for role in self.roles:
        if message.cause_by in role.rc.watch:  # 标签匹配
            role.rc.msg_buffer.push(message)
```

**关键设计：** 消息不是发给特定角色的，而是发布到环境后由**标签匹配机制**自动路由。`cause_by` 字段标记消息来源的 Action 类型，角色的 `watch` 集合声明自己关心哪些类型的消息。

对比 OpenClaw 当前的消息机制：我们主要靠 session 上下文传递，可以借鉴这种**声明式订阅+标签路由**来解耦模块间通信。

### 亮点 4：Team 编排器与投资模型

```python
# metagpt/team.py (概念)
class Team:
    def hire(self, roles): ...        # 招聘角色
    def invest(self, investment): ... # 设置投资额度（控制 token 预算）
    
    async def run(self, n_round, idea):
        for i in range(n_round):
            for role in self.roles:
                # 1. 观察环境中的新消息
                role._observe()
                # 2. 思考（状态机决策）
                if await role._think():
                    # 3. 执行动作
                    msg = await role._act()
                    if msg:
                        self.publish_message(msg)
```

**投资模型（investment）：** 这是一个独特的经济约束机制——模拟公司融资，投资额越高，允许使用的 token 越多，能完成更复杂的项目。这本质上是一种**资源预算控制系统**。

### 亮点 5：序列化与断点恢复

```python
# software_company.py — 支持从序列化状态恢复
def generate_repo(..., recover_path=None):
    if recover_path:
        company = Team.deserialize(stg_path=recover_path, context=ctx)
        idea = company.idea  # 恢复原始需求
    else:
        company = Team(context=ctx)
        company.hire([TeamLeader(), ProductManager(), Architect(), Engineer2(), DataAnalyst()])
    
    company.invest(investment)
    asyncio.run(company.run(n_round=n_round, idea=idea))
```

整个 Team 状态（所有 Role 的记忆、状态机位置、中间产物）都可以序列化到磁盘并恢复。这对于长运行任务和错误恢复至关重要。

## 三、最新进展（2025年）

### AFlow — 自动工作流生成（ICLR 2025 Oral）

MetaGPT 团队在 ICLR 2025 发表了 **AFlow（Agentic Workflow Generation）**，排名 LLM-based Agent 类别第 2：

- **问题：** 手工设计多 Agent 工作流很困难且次优
- **方案：** 用 LLM 自动搜索最优工作流拓扑
- **方法：** 将工作流定义为 DAG，通过 LLM 生成候选图 → 执行评估 → 迭代优化
- **代码位置：** `examples/aflow/`
- **意义：** 从"人工编排 SOP"进化到"自动发现最优 SOP"

### SPO & AOT — 新的 Prompt 策略

- **SPO（Structured Prompt Optimization）：** 自动优化 ActionNode 的 prompt 模板
- **AOT（Auto-Orchestration of Tools）：** 自动选择和组合工具链

## 四、对我们（OpenClaw / 楠楠）的启发

### 可直接借鉴的点

| 启发 | 具体应用 | 优先级 |
|------|----------|--------|
| **SOP 编码思想** | 将楠楠的日常行为（学习→研究→写报告→反思）编码为显式 SOP，而非隐式在 system prompt 里 | ⭐⭐⭐ |
| **ActionNode 结构化输出** | 子 agent 返回结果时用固定 schema（如研究报告必须包含：数据来源、结论、置信度、风险提示），便于主 agent 自动校验 | ⭐⭐⭐ |
| **Role 状态机** | 恋爱模拟器的亲密度等级系统可以参考 FSM 设计——不同等级触发不同的行为集合和状态转换规则 | ⭐⭐ |
| **标签消息路由** | OpenClaw 的 skill 触发可以用类似 `watch` + `cause_by` 的机制替代当前的 if-else 匹配 | ⭐⭐ |
| **投资/预算模型** | API token 消耗控制——给每个任务分配预算，超预算时自动降级或停止 | ⭐ |
| **Team 序列化** | 长任务（如股票分析的多轮进化）支持断点续跑 | ⭐⭐ |

### 架构层面的思考

1. **MetaGPT 证明了"结构化中间产物"是多 Agent 协作成功的关键。** 自由文本对话在单轮交互中 OK，但在多步骤协作中会产生严重的误差累积。OpenClaw 在 subagent pipeline 中应强制结构化输入/输出协议。

2. **Role = 身份 + 状态机 + 动作集** 这个三元组非常清晰。我们的 SOUL.md 定义了身份，但缺少显式的状态机和动作集定义。可以考虑为楠楠定义一套正式的 `states` 和 `actions`。

3. **AFlow 的方向值得关注** —— 未来 Agent 框架的竞争点可能从"谁的角色定义更好"转向"谁能自动发现最优工作流"。OpenClaw 的 skill 系统未来可以考虑加入自动组合能力。

## 五、总结评价

**优势：**
- 学术背景深厚（ICLR 论文），设计有理论支撑
- SOP 编码思想独特且有效，100% 任务完成率证明了鲁棒性
- 结构化输出机制（ActionNode）是业界最佳实践之一
- 生态丰富：Data Interpreter、Researcher、Debate 等多种使用场景

**局限：**
- 主要面向软件工程场景，通用性不如 CrewAI/LangGraph
- Python 重度绑定，不支持其他语言的原生 SDK
- 单次运行成本较高（需要多个 GPT-4 级别的调用串联）
- 状态机的 LLM 决策本身也有不确定性（偶尔会跳到错误状态）

**适合学习什么：** 多 Agent 协作的结构化方法论、如何用 Pydantic 建模复杂 Agent 系统、SOP 如何转化为代码实现。
