### 1.2 项目起源与设计理念

GNAP（Git-Native Agent Protocol）由 Farol Labs 设计，是一种**极简的智能体协调协议**，其核心理念令人眼前一亮：**用 Git 作为智能体的协调基础设施，不需要任何服务器，不需要数据库，不需要供应商锁定**。

传统多智能体系统需要：
- 消息队列或消息总线（如 RabbitMQ、Kafka）
- 共享数据库（如 PostgreSQL、Redis）
- API 网关或服务发现机制
- 专门的部署和维护

GNAP 全部不需要。它只需要一个 Git 仓库。

Farol Labs 在生产环境中使用 GNAP 协调 **4 个智能体（2个AI + 2个人类）跨 50+ 任务**，零基础设施开销。这证明了该协议的实用性。

### 1.3 项目定位

GNAP 不是一个 AI Agent 框架，而是一个**协议规范（RFC Draft）**。它定义了一种极简的协调语言，让任何能执行 `git push/pull` 的实体（AI Agent、人类、甚至 CI/CD 系统）都能参与协作。

### 1.4 与 OpenClaw 的直接关联

值得注意的是，GNAP 的 README 明确提到 OpenClaw 作为参与者的一个运行时选项：

> "Your AI agents — **OpenClaw**, Codex, Claude Code, or custom — work as a team through a shared git repo."

这意味着 **OpenClaw 已被设计为 GNAP 生态的一等公民**。OpenClaw 可以通过 GNAP 与其他 Agent（Claude Code、GitHub Copilot 等）协作，形成 Git-native 的多 Agent 团队。

## 二、核心技术架构

### 2.1 架构哲学：Git-as-Backbone

GNAP 的架构极度简洁，核心思想是：**Git 仓库 = 任务面板 + 消息总线 + 审计日志**。

```
┌──────────────────────────────────────────────────────┐
│  Application Layer（可选）                            │
│  budgets, dashboards, workflows, governance           │
├──────────────────────────────────────────────────────┤
│  GNAP Protocol（协议规范）                            │
│  agents · tasks · runs · messages                  │
├──────────────────────────────────────────────────────┤
│  Git（传输层 + 存储层）                                │
│  push/pull · merge · history · distribution        │
└──────────────────────────────────────────────────────┘
```

### 2.2 核心数据结构：仅4个实体，4个JSON文件

GNAP 仅定义了4种核心实体，存储在 `.gnap/` 目录中：

```
.gnap/
 ├── version          ← 协议版本（如 "4"）
 ├── agents.json      ← 团队成员（人类+AI）
 ├── tasks/           ← 任务目录（每个任务一个JSON文件）
 │    ├── FA-1.json
 │    └── FA-2.json
 ├── runs/            ← 执行记录（每次尝试一个文件）
 └── messages/        ← 消息目录（Agent间通信）
```

#### 实体1：Agent（agents.json）

定义团队中的每个参与者（AI Agent 或人类）：

```json
{
  "agents": [
    {
      "id": "carl",
      "name": "Carl",
      "role": "CRO",
      "type": "ai",
      "status": "active",
      "runtime": "openclaw",
      "reports_to": "leo",
      "heartbeat_sec": 300,
      "capabilities": ["coding", "research"]
    }
  ]
}
```

**关键字段：**
- `type`: `"ai"` 或 `"human"` — 人类和AI均为平等参与者
- `runtime`: 运行时标识（`openclaw` / `codex` / `claude` / `custom`）
- `reports_to`: 组织树，用于构建层级汇报关系
- `heartbeat_sec`: 心跳间隔（默认300秒）

#### 实体2：Task（tasks/*.json）

每个任务是一个独立的 JSON 文件，定义工作项：

```json
{
  "id": "FA-1",
  "title": "Set up Stripe billing",
  "assigned_to": ["leo"],
  "state": "in_progress",
  "priority": 0,
  "created_by": "ori",
  "created_at": "2026-03-12T11:40:00Z",
  "parent": "FA-0",
  "blocked": false,
  "reviewer": "carl",
  "tags": ["billing", "finance"]
}
```

**状态机：**

```
backlog → ready → in_progress → review → done
   ↓        ↑          │
   ↓        └──────────┘（reviewer拒绝，返工）
   ↓
blocked → ready（解除阻塞）
   ↓
cancelled
```

#### 实体3：Run（runs/*.json）

每次执行尝试的记录，支持成本追踪和重试历史：

```json
{
  "id": "FA-1-1",
  "task": "FA-1",
  "agent": "carl",
  "state": "completed",
  "attempt": 1,
  "started_at": "2026-03-12T12:30:00Z",
  "finished_at": "2026-03-12T12:35:00Z",
  "tokens": { "input": 12400, "output": 3200 },
  "cost_usd": 0.08,
  "result": "Stripe account created, test mode live",
  "commits": ["a1b2c3d"],
  "artifacts": ["docs/stripe-setup.md"]
}
```

**Run 的价值：**
- 成本追踪（所有 Run 的 cost_usd 之和 = 总成本）
- 重试历史（一个 Task 可以有多个 Run，失败不意味着 Task 失败）
- 审计日志（谁、什么时候、做了什么、用了多少 token、花了多少钱）
- 性能对比（按 Agent 维度对比 speed/cost/success）

#### 实体4：Message（messages/*.json）

Agent 间通信的消息：

```json
{
  "id": "1",
  "from": "ori",
  "to": ["carl"],
  "at": "2026-03-12T09:30:00Z",
  "type": "directive",
  "text": "Focus on billing first. Everything else can wait."
}
```

**消息类型：**
- `directive`: 指令/命令
- `report`: 执行结果报告
- `question`: 请求指导
- `approval`: 审批请求（用于人类-in-the-loop）

### 2.3 Agent 心跳循环

每个 GNAP Agent 运行相同的心跳循环：

```
1. git pull（同步最新状态）
2. 读取 agents.json → 我是否 active？
3. 读取 tasks/ → 有分配给我的任务吗？
4. 读取 messages/ → 有新消息吗？
5. 执行工作 → commit → git push
6. sleep(heartbeat_sec)
```

这个循环的巧妙之处在于：**它是完全分布式的，没有中央协调器**。每个 Agent 都是独立的进程，只需定期与 Git 仓库同步。

### 2.4 与其他协调方案的对比

| 特性 | GNAP | AgentHub | Paperclip | Symphony | CrewAI/LangGraph |
|------|------|----------|-----------|----------|------------------|
| 需要服务器 | **No** | Yes (Go) | Yes (Node.js) | Yes | Yes (Python) |
| 数据库 | **None (git)** | SQLite | PostgreSQL | In-memory | In-memory |
| 供应商锁定 | **None** | None | None | Linear+Codex | LangChain/OpenAI |
| 启动时间 | **30秒** | 5分钟 | 30分钟 | 30分钟 | 15分钟 |
| 任务追踪 | **Yes** | No | Yes | External | No |
| 成本追踪 | **Yes (runs)** | No | Yes | Yes | No |
| A2A消息 | **Yes** | Yes (channels) | Limited | No | No |
| 人类+AI协作 | **Yes** | Yes | Yes | No | No |
| 离线工作能力 | **Yes** | No | No | No | No |

## 三、核心特性亮点

### 3.1 零基础设施开销

GNAP 最大的亮点是**不需要任何服务器或数据库**。任何能执行 Git 操作的 Agent 都可以参与协调。这意味着：

- **零部署成本**：不需要配置消息队列、数据库、API 网关
- **零维护成本**：不需要升级服务器、不需要备份数据库
- **即时启动**：创建一个 `.gnap/` 目录，初始化4个文件，30秒内开始工作

### 3.2 Git 原生审计日志

GNAP 利用 Git 的天然特性，实现了**完美的审计追踪**：

- 每个 Task 的创建、分配、状态变更都有对应的 commit
- 每个 Run 的执行记录、token 消耗、成本都有 commit 历史
- 每个 Message 的发送和接收都在 Git 历史中
- `git log` 就是完整的审计日志，不需要额外的 SIEM 系统

### 3.3 离线工作能力

由于每个 Agent 只需要本地 Git 仓库和互联网连接（推送/拉取），Agent 可以在**完全离线**的状态下工作，之后再同步。这对于：

- 网络不稳定的远程环境
- 需要数据主权的企业（数据不经过任何中间服务器）
- 离线开发和 CI/CD 场景

### 3.4 人类和 AI Agent 平等协作

GNAP 的 `agents.json` 中，人类和 AI Agent 都是一等公民。人类可以：
- 接收 AI Agent 的 `approval` 类型消息并审批
- 直接在 `tasks/` 目录创建任务
- 通过 `messages/` 向 AI Agent 发送指令
- 审查 AI Agent 的执行结果（`review` 状态）

### 3.5 多运行时兼容性

GNAP 协议规范是**完全无语言绑定的**。任何 Agent 运行时（OpenClaw、Claude Code、GitHub Copilot、Codex、自定义 Agent）只要能执行 Git 操作，就能参与 GNAP 协作。

### 3.6 成本追踪与预算控制

通过 `runs/` 目录，GNAP 提供**细粒度的成本追踪**：

```bash
# 计算某个 Task 的总成本
cat .gnap/runs/FA-1-*.json | jq '[.cost_usd] | add'

# 计算某个 Agent 的总成本
grep '"agent": "carl"' .gnap/runs/*.json | jq -s '[.[].cost_usd] | add'
```

### 3.7 可组合的应用层

GNAP 只定义了4个核心实体和心跳循环，**所有高级功能都留给应用层**：

- 仪表板和可视化（读取 `.gnap/` 数据渲染）
- 预算治理（读取 `runs/` 计算成本，设置告警）
- 工作流引擎（读取 `tasks/` 状态，触发自动化）
- AI Agent 编排（基于 `agents.json` 的 `reports_to` 构建层级）

## 四、对OpenClaw的借鉴价值（分级P0-P3）

### 4.1 P0 — 直接借鉴（最高优先级）

#### P0.1：集成 GNAP 作为多 Agent 协调协议

**借鉴价值：★★★★★**

OpenClaw 被 GNAP 明确列为支持的运行时（`"runtime": "openclaw"`）。这意味着 OpenClaw 可以：

1. **开箱即用的多 Agent 协作**：OpenClaw Agent 可以通过 GNAP 与 Claude Code、GitHub Copilot 等其他 Agent 协作，无需额外的消息总线或数据库
2. **零成本的多 Agent 编排**：利用 GNAP 的心跳循环和任务分配机制，OpenClaw 可以实现多 Agent 任务分发
3. **Git-native 的审计和追溯**：所有 OpenClaw Agent 的操作都有 Git 历史记录

**具体实现路径：**
- 在 OpenClaw 中实现 GNAP Agent 运行时适配器
- 支持 `.gnap/agents.json` 的读取和写入
- 实现 GNAP 兼容的心跳循环（定期 pull/push）
- 支持 GNAP Task 状态变更的自动 commit/push

#### P0.2：Git History 作为 Agent 记忆系统

**借鉴价值：★★★★★**

OpenClaw 的记忆系统目前依赖文件存储。GNAP 的思路可以借鉴：

- 将 Agent 的操作历史编码为 Git commits
- 利用 Git 的分支机制实现不同"记忆视角"
- 利用 Git 的 diff 功能追踪 Agent 知识的变化
- 利用 Git 的 blame 功能追踪某个知识的来源

### 4.2 P1 — 重要借鉴

#### P1.1：极简协议设计哲学

**借鉴价值：★★★★☆**

GNAP 的设计哲学是"**少即是多**"：仅用4个 JSON 文件定义了整个协调协议。这种极简主义值得学习：

- 避免过度设计：不要为了"将来可能的需求"增加复杂性
- 协议先行：先定义清楚数据和交互规范，再实现代码
- 无服务器优先：能用 Git 解决的就不要引入额外基础设施

对于 OpenClaw 的工具调用协议、MCP 协议扩展，可以借鉴这种思路。

#### P1.2：Run-based 成本追踪

**借鉴价值：★★★★☆**

GNAP 的 `runs/` 实体提供了细粒度的成本追踪（input tokens、output tokens、cost_usd）。OpenClaw 可以借鉴：

- 为每次工具调用创建执行记录
- 记录 token 消耗和 API 调用成本
- 支持按时间、Agent 类型、任务类型聚合成本
- 为 Agent 的"算力预算"提供数据基础

#### P1.3：Human-in-the-Loop 机制

**借鉴价值：★★★★☆**

GNAP 的 `approval` 消息类型和 `review` 状态提供了原生的人类审批机制。OpenClaw 可以借鉴：

- 在关键决策点引入人工审批节点
- 支持 AI Agent 向人类请求确认后再执行
- 将人类的反馈作为 Agent 学习的信号

### 4.3 P2 — 参考借鉴

#### P2.1：分层架构（Protocol Layer vs Application Layer）

**借鉴价值：★★★☆☆**

GNAP 明确区分了协议层和应用层：
- 协议层：极简、稳定、向后兼容
- 应用层：灵活、创新、可替换

OpenClaw 可以借鉴这个思路：
- 核心协议（工具调用、消息格式）保持稳定
- 高级功能（记忆、规划、协作）作为应用层可插拔

#### P2.2：Task 的状态机设计

**借鉴价值：★★★☆☆**

GNAP 的 Task 状态机（backlog → ready → in_progress → review → done）清晰、实用。OpenClaw 的任务管理可以参考：

- 明确的任务生命周期
- 支持 blocked 状态（等待依赖或人工输入）
- 支持 review 状态（结果需要审核）

#### P2.3：离线优先设计

**借鉴价值：★★★☆☆**

GNAP 的离线工作能力对于 OpenClaw 的边缘部署场景有价值：
- 支持网络不稳定的环境

### 4.4 P3 — 启发思考

#### P3.1：Git-as-Coordination 的范式创新

**借鉴价值：★★☆☆☆**

GNAP 最深刻的启发不是具体技术，而是**范式思维**：用成熟的工具（Git）解决新问题（Agent 协调）。这种"不要重复造轮子"的思路值得学习：

- 考虑用已有的成熟系统（如文件系统、Git、SQLite）作为 Agent 的基础设施
- 而不是引入新的专用中间件

#### P3.2：协议规范先于实现

**借鉴价值：★★☆☆☆**

GNAP 先有协议规范（Markdown 文档），后有具体实现。这种"协议先行"的开发模式值得参考：

- 先明确接口和数据格式，再实现代码
- 有利于多运行时支持和互操作性

## 五、风险分析

### 5.1 高频 Git 操作的性能瓶颈

**风险等级：中等**

GNAP 的心跳循环依赖频繁的 `git pull/push`。对于：
- 慢速网络（如卫星互联网、移动网络）
- 大型仓库（Git history 很大）
- 高频心跳（短 heartbeat_sec）

Git 操作可能成为性能瓶颈。

**缓解措施：**
- 使用 Git 的 shallow clone 和单分支 clone 减少数据传输
- 增加 heartbeat_sec 间隔减少操作频率
- 使用 Git LFS 管理大文件

### 5.2 Git Merge 冲突

**风险等级：中等**

多个 Agent 同时修改 `.gnap/` 目录中的文件，可能产生 Git merge 冲突。如果两个 Agent 同时：
- 修改同一个 Task 的状态
- 向同一个 Task 添加 Run 记录
- 发送消息到同一个 Message ID

**缓解措施：**
- Agent 使用独立的文件命名空间减少冲突（如 Agent ID 前缀）
- 实现乐观锁机制（先读后写，检查文件哈希）
- 冲突时采用"先拉取再重试"的策略

### 5.3 协议成熟度风险

**风险等级：中等偏高**

GNAP 目前是 RFC Draft 状态（version 4），尚未成为稳定标准。协议可能在未来版本中发生不兼容变更。考虑到其 Star 数较少（45），社区活跃度和长期维护存在不确定性。

**缓解措施：**
- 关注协议版本字段（`.gnap/version`），拒绝运行高于支持版本的协议
- 为协议变更预留升级路径
- 考虑参与 GNAP 社区，推动协议稳定化

### 5.4 不适合实时性要求高的场景

**风险等级：低**

GNAP 的协调延迟取决于心跳间隔（默认300秒）。对于需要秒级响应的场景（如实时对话），GNAP 不适合。

**适用场景：**
- 异步任务协调（小时/天级）
- 后台批处理任务
- 需要人工审批的长周期工作流

### 5.5 安全性考量

**风险等级：低**

由于所有数据都存储在 Git 仓库中：
- Git 仓库需要适当的访问控制（否则所有 Agent 都能读写所有数据）
- `.gnap/` 目录可能包含敏感信息（Agent ID、任务描述、成本数据），需要加密或私有仓库
- Git 历史永久保存，早期版本的敏感数据不会被自动清除

## 六、竞争格局与相关项目

### 6.1 GNAP 的独特位置

GNAP 处于一个独特的细分市场：**无服务器、多运行时、Git-native 的 Agent 协调**。

- vs **AutoGen/CrewAI/LangGraph**：这些是有服务器的、需要绑定特定语言的框架，GNAP 与它们是正交关系（可以互补而非竞争）
- vs **MCP（Model Context Protocol）**：MCP 定义的是 Agent-to-Tool 的接口，GNAP 定义的是 Agent-to-Agent 的协调，定位不同
- vs **A2A（Agent-to-Agent）**：A2A 是另一种 Agent 协调协议，但需要服务器；GNAP 提供了无需服务器的替代方案

### 6.2 相关协议

| 协议 | 定位 | 服务器需求 | 与 GNAP 关系 |
|------|------|-----------|-------------|
| MCP | Agent ↔ Tools | 无（本地 MCP Server） | 正交（工具 vs 协调） |
| A2A | Agent ↔ Agent | 有（Agent Server） | 竞争（协调协议） |
| GNAP | Agent ↔ Agent | **无**（仅 Git） | 互补（轻量级替代） |
| ACP (Agent Communication Protocol) | Agent ↔ Agent | 待定 | 潜在竞争 |

### 6.3 已被采纳情况

GNAP 已被多个项目提及/集成：
- OpenHands（GitHub Issue #13416）
- AutoGen（GitHub Issue #7398）
- VoltAgent（GitHub Issue #1153）
- oTTomator（社区项目）
- OpenAgents（GitHub Issue #145）
- BotSharp（GitHub Issue #1308）
- Agent Zero（GitHub Issue #1267）
- Composio（OpenClaw 插件生态）

这表明 GNAP 的设计思路被主流 Agent 框架认可，有一定的生态影响力。

## 七、OpenClaw 集成路线图建议

基于 GNAP 的特点和 OpenClaw 的架构，以下是推荐的集成路线：

### Phase 1：轻量级集成（1-2周）

2. **支持 GNAP Task 读取**：OpenClaw 可以读取 `.gnap/tasks/` 目录，将 GNAP 任务作为 OpenClaw 的工作项
3. **支持 GNAP 消息**：OpenClaw 可以向 `.gnap/messages/` 写入消息，实现与其他 Agent 的通信

3. **Run 记录生成**：每次工具调用自动生成 GNAP Run 记录

### Phase 3：生态建设（长期）

2. **GNAP 应用层工具**：构建 GNAP 仪表板、成本分析、任务可视化等工具
