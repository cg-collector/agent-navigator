# 开源Agent项目学习 #044

## 项目名称
Browser Use

## GitHub链接
https://github.com/browser-use/browser-use

## 一句话简介
让 AI Agent 能够理解并操控网页的开源框架——通过 DOM 提取而非截图，实现高效、低成本的浏览器自动化。

## 项目架构
Browser Use 的核心架构分为以下几个层次：

1. **浏览器控制层**：通过 Playwright 启动和管理 Chromium 浏览器，但已从 Playwright 的 Node.js API 层迁移到直接通过 CDP（Chrome DevTools Protocol）通信，使用自研的 `cdp-use` 库。Playwright 仅负责浏览器安装和启动，页面交互直接走 CDP，延迟更低。

2. **DOM 提取层**：替代传统的截图方案，直接提取页面的 DOM 结构，将可交互元素（按钮、输入框、链接等）序列化为结构化文本数据，包含标签名、文本内容、placeholder、CSS 选择器等元信息，发送给 LLM。

3. **Agent 决策层**：LLM 接收 DOM 快照 + 用户指令，返回结构化的动作指令（点击、输入、导航等）。支持循环检测（通过追踪近期动作防止陷入死循环）和策略升级（连续失败后切换到更宽泛的选择器或无障碍树）。

4. **视觉备用层**：对于 DOM 难以处理的复杂页面（Canvas、Shadow DOM），可切换到视觉模式，将截图与 DOM 数据同时发送给模型。

5. **云端服务层**：提供 Browser Use Cloud，包含隐形浏览器（反检测）、CAPTCHA 解决、195+ 国家代理池，以及 Skill API（将任何网站变为可靠的 API 端点）。

## 技术亮点
- **DOM-first 而非 Vision-first**：不同于 OpenAI CUA、Anthropic Computer Use 等基于截图的方案，Browser Use 直接提取 DOM 语义信息，token 消耗大幅降低（大多数任务 $0.01-$0.05），速度更快，可靠性更高
- **自研 BU 2.0 模型**：30B MoE 架构，推理时仅激活 3B 参数，可单 GPU 运行，专为浏览器任务优化，无需调用外部 API
- **CDP 直连架构**：抛弃 Playwright 的 Node.js 中间层，通过自研 cdp-use 库直接与浏览器通信，减少延迟
- **Human-in-the-loop 机制**：遇到验证码、认证、歧义选择时自动暂停求助人类，不追求完全自主
- **循环检测与策略升级**：Agent 追踪自身动作，检测到重复行为后自动切换选择策略，提高鲁棒性
- **跨平台 SDK**：提供 Python 和 TypeScript SDK，也提供 Web UI 前端

## 适用场景
- 网页数据抓取和信息提取
- 表单自动填写和批量操作
- Web 应用自动化测试
- 网站监控和内容追踪
- RAG 系统的网页数据采集管线
- 任何需要 "让 AI 上网" 的场景

## 对我们的启发
1. **DOM vs Screenshot 的取舍**：OpenClaw 的 browser 工具主要基于 Playwright 的 snapshot/screenshot 机制，类似 Vision-first 思路。Browser Use 证明 DOM-first 在大多数网页操作场景下效率更高。可以考虑为 OpenClaw 的 browser 工具增加轻量级 DOM 提取模式，用于简单的表单填写和信息提取，减少 token 消耗。

2. **自研小模型降成本**：Browser Use 用 3B 激活参数的 MoE 模型处理浏览器任务，成本极低。这启发我们在高频、模式化的任务（如网页抓取、消息处理）上，可以用小模型替代大模型，大模型只在复杂决策时介入。

3. **Human-in-the-loop 设计哲学**：Browser Use 不追求完全自主，遇到不确定情况就问人。这与 OpenClaw 的审批机制（exec elevated）理念一致——安全性和可靠性比完全自动化更重要。

4. **CDP 直连**：OpenClaw 的 browser 工具如果也在考虑性能优化，CDP 直连是一个值得参考的方向。
