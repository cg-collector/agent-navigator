## 一、架构设计总览

### 1.1 核心模块

Eliza 采用 **Monorepo + Plugin-First** 架构，核心包结构：

```
eliza/
├── packages/
│   ├── core/          # 核心运行时（AgentRuntime、Memory、State）
│   ├── server/        # Express.js 后端 API
│   ├── client/        # React Web 管理界面
│   ├── cli/           # 命令行工具（@elizaos/cli）
│   ├── app/           # Tauri 桌面客户端
│   ├── plugin-sql/    # 数据库适配器（Postgres/PGLite）
│   └── ...            # 其他插件包
```

### 1.2 运行时架构（AgentRuntime）

整个系统的核心是 `IAgentRuntime` 接口，它是 Agent 的大脑和中枢：

```typescript
interface IAgentRuntime extends IDatabaseAdapter {
  // === 身份与配置 ===
  agentId: UUID;
  character: Character;        // 角格化人格定义

  // === 组件注册表 ===
  providers: Provider[];       // 上下文数据提供者
  actions: Action[];           // 可执行动作
  evaluators: Evaluator[];     // 结果评估器
  services: Service[];         // 后台服务

  // === 核心处理流程 ===
  processActions(message, responses, state?): Promise<void>;
  composeState(message, state?): Promise<State>;   // State 聚合
  evaluate(message, state?): Promise<void>;         // 评估反馈

  // === 注册方法 ===
  registerAction(action): void;
  registerProvider(provider): void;
  registerEvaluator(evaluator): void;
  registerService(service): void;

  // === 模型管理 ===
  useModel<T>(modelType, params, provider?): Promise<ModelResultMap[T]>;
  registerModel(modelType, handler, provider?, priority?): void;

  // === 事件系统 ===
  emit(eventType, data): Promise<void>;
  on(eventType, handler): void;

  // === Run 追踪 ===
  createRunId(): UUID;
  startRun(roomId?): UUID;
  endRun(): void;
}
```

### 1.3 消息处理流水线

```
用户输入 → Message Receipt → Memory Storage → State Composition
    → Action Selection → Action Execution → Evaluation → Response Generation
```

每一步职责清晰：
1. **Message Receipt** — 接收并验证消息
2. **Memory Storage** — 持久化到数据库
3. **State Composition** — 从所有 Provider 聚合上下文
4. **Action Selection** — LLM 决定执行哪些动作
5. **Action Execution** — 执行选中的动作 handler
6. **Evaluation** — 后处理评估结果质量
7. **Response Generation** — 生成最终回复

### 1.4 插件系统（Plugin Architecture）

插件是 Eliza 的灵魂。每个插件是一个纯声明式的对象：

```typescript
const myPlugin: Plugin = {
  name: 'my-plugin',
  description: '做什么的',
  priority: 50,              // 加载优先级
  dependencies: ['@elizaos/plugin-sql'],  // 依赖声明

  // 八种组件类型
  actions: [...],            // Agent 能做什么
  providers: [...],          // Agent 能看到什么
  evaluators: [...],         // 怎么评估结果
  services: [...],           // 后台连接什么
  models: {...},             // 自定义模型处理器
  routes: [...],             // HTTP 端点
  events: {...},             // 事件监听
  adapter?: dbAdapter,       // 数据库适配器

  init?: async (config, runtime) => { /* 初始化逻辑 */ }
};
```

**加载顺序按优先级排列：**
```
databases(-100) → modelProviders(-50) → corePlugins(0) → features(50) → platforms(100)
```

### 1.5 记忆系统（Memory System）

分层记忆设计，支持 7 种记忆类型：

```typescript
enum MemoryType {
  MESSAGE = 'message',           // 对话消息
  FACT = 'fact',                 // 提取的知识三元组
  DOCUMENT = 'document',         // 文档存储
  RELATIONSHIP = 'relationship', // 实体关系图谱
  GOAL = 'goal',                 // Agent 目标
  TASK = 'task',                 // 计划任务
  ACTION = 'action',             // 动作执行记录
}
```

每条记忆统一接口：

```typescript
interface Memory {
  id: UUID;
  type: MemoryType;
  roomId: UUID;           // 所属房间（对话空间）
  userId?: UUID;
  agentId?: UUID;
  content: { text: string; [key: string]: any };
  embedding?: number[];   // 向量嵌入，用于语义搜索
  createdAt: Date;
  metadata?: Record<string, any>;
}
```

### 1.6 角色系统（Character System）

Eliza 最独特的设计——**Character Interface**，用结构化配置定义 Agent 人格：

```typescript
interface Character {
  // 基础身份
  name: string;
  username: string;
  bio: string[];              // 背景故事数组

  // 行为定义
  system: string;             // 系统 Prompt
  adjectives: string[];       // 性格形容词（如 ["witty", "warm"]）
  topic: string;              // 关注话题
  style: StyleType;           // 说话风格

  // 交互设置
  messageExamples: Message[]; // 少样本示例
  postExamples: Post[];       // 社交媒体发帖示例

  // 元数据
  clients: string[];          // 启用的平台客户端
  modelProvider: string;      // 默认模型
  settings: Record<string, any>;
}
```

这意味着换一个人格 = 换一个 JSON 配置文件，不需要改代码。

## 二、技术亮点（代码级分析）

### ✦ 亮点 1：State Composition Pipeline —— 多源上下文聚合模式

Eliza 的 `composeState` 方法实现了一个优雅的多 Provider 聚合管道：

```typescript
async composeState(message: Memory): Promise<State> {
  const state = {};
  for (const provider of this.providers) {
    const data = await provider.get(this, message, state);
    Object.assign(state, data);  // 每个 Provider 贡献一层上下文
  }
  return state;
}
```

**为什么这个设计好？**
- **关注点分离**：时间 Provider 只管注入当前时间，消息 Provider 只管注入最近聊天记录
- **可组合性**：新加一个上下文源 = 新写一个 Provider，零侵入
- **有序聚合**：Provider 按优先级顺序执行，后面的可以覆盖前面的

内置的核心 Provider 包括：
- `timeProvider` → 注入 `{currentTime, date, time}`
- `characterProvider` → 注入 `{agentName, agentDescription, characterName, ...}`
- `recentMessagesProvider` → 注入最近 N 条对话历史

**对 OpenClaw 的启发：** OpenClaw 的 workspace context injection 可以借鉴这种 Provider Chain 模式，把 SOUL.md、USER.md、memory 文件等各自做成独立 Provider。

### ✦ 亮点 2：Action Chaining via Run Tracking —— 动作链式执行

Eliza 通过 Run ID 实现了动作之间的数据传递和链式调用：

```typescript
// 开始一个 Run（执行上下文）
const runId = runtime.startRun(roomId);
try {
  await runtime.processActions(message, responses, state);

  // 第二个动作可以拿到第一个动作的结果！
  const results = runtime.getActionResults(message.id);
  const searchData = results.find(r => r.action === 'SEARCH_WEB');
  if (searchData?.success) {
    // 用搜索结果驱动下一个动作
  }
} finally {
  runtime.endRun();
}
```

**关键设计点：**
- `ActionResult` 按 `messageId` 索引，同一次消息处理的所有动作共享结果
- 支持动作间的隐式数据流（不需要显式传参）
- `finally` 块确保 Run 总是被正确结束，防止资源泄漏

**代码级的精妙之处：** 这本质上实现了一个**临时的、消息作用域的内存空间**，类似函数调用的栈帧，但跨多个异步动作共享。

### ✦ 亮点 3：分层记忆 + 向量语义搜索 + 自动清理

Eliza 的记忆系统不是简单的 KV 存储，而是一套完整的记忆管理架构：

**(a) 语义相似度搜索：**
```typescript
// 先生成查询向量
const queryEmbedding = await runtime.useModel(
  ModelType.TEXT_EMBEDDING,
  { input: "用户喜欢什么？" }
);

// 按相似度检索相关事实
const relatedFacts = await runtime.searchMemoriesBySimilarity(queryEmbedding, {
  type: MemoryType.FACT,
  threshold: 0.7,      // 相似度阈值
  limit: 5
});
```

**(b) 事实提取（三元组）：**
```typescript
await runtime.createFact({
  subject: 'user',
  predicate: 'favorite_color',
  object: 'blue',
  confidence: 0.95,     // 提取置信度
  source: message.id     // 来源追踪
});
```

**(c) 记忆压缩（自动摘要归档）：**
```typescript
async function compressMemories(runtime, roomId) {
  const messages = await runtime.getMemories({
    type: MemoryType.MESSAGE,
    roomId,
    before: new Date(Date.now() - 7 * 86400000) // 7天前的消息
  });

  // LLM 生成摘要
  const summary = await runtime.useModel(ModelType.TEXT_LARGE, {
    prompt: `Summarize: ${messages.map(m => m.content.text).join('\n')}`,
    maxTokens: 500
  });

  // 存摘要，删原文
  await runtime.createMemory({ type: MemoryType.DOCUMENT, content: { text: summary } });
  for (const msg of messages) await runtime.deleteMemory(msg.id);
}
```

**(d) 自动清理服务：**
```typescript
class MemoryCleanupService {
  private readonly MAX_MESSAGE_AGE = 30 * 86400000;  // 30天
  private readonly MAX_FACTS_PER_ROOM = 1000;

  async cleanup(runtime) {
    await this.cleanupOldMessages(runtime);      // 删过期消息
    await this.consolidateFacts(runtime);         // 合并低价值事实
    await this.cleanupOrphanedRelationships(runtime); // 清理孤儿关系
  }

  // 事实合并策略：置信度 × 时间衰减
  consolidateFacts(runtime) {
    const score = fact => fact.confidence * (1 / (Date.now() - fact.createdAt));
    return facts.sort((a, b) => score(b) - score(a)).slice(0, MAX_FACTS_PER_ROOM);
  }
}
```

这套系统解决了 Agent 记忆的三大痛点：**遗忘（压缩）、膨胀（清理）、检索（向量搜索）**。

### ✦ 亮点 4：插件即声明式对象 —— 零继承扩展模式

大多数框架要求你继承基类或实现复杂接口。Eliza 的插件就是一个普通对象：

```typescript
// 一个完整的 Twitter 插件（简化示意）
export const twitterPlugin: Plugin = {
  name: '@elizaos/plugin-twitter',

  actions: [
    {
      name: 'POST_TWEET',
      description: 'Post a tweet',
      validate: async (runtime, message) => {
        return !!runtime.getSetting('TWITTER_API_KEY');
      },
      handler: async (runtime, message, state) => {
        const content = state.generatedText;
        await twitterClient.tweet(content);
        return { text: `Tweeted: ${content}`, action: 'POST_TWEET' };
      },
      examples: [[{{user}}: "tweet hello", {{agent}}: "Posting tweet..."]]
    }
  ],

  providers: [
    {
      get: async (runtime, message, state) => {
        const tweets = await twitterClient.getTimeline();
        return { recentTweets: tweets.slice(0, 10).map(t => t.text).join('\n') };
      }
    }
  ],

  services: [
    {
      name: ServiceType.TWITTER,
      start: async (runtime) => { /* 建立 WebSocket 连接 */ },
      stop: async (runtime) => { /* 断开连接 */ }
    }
  ],

  events: {
    MESSAGE_RECEIVED: [async (params) => {
      // 收到消息时自动检查是否需要回复推文
    }]
  },

  init: async (config, runtime) => {
    if (!config.TWITTER_API_KEY) {
      console.warn('Twitter plugin loaded but no API key');
    }
  }
};
```

**这个设计的威力在于：**
- **无继承耦合** — 不需要 `extends BasePlugin`
- **自描述** — 插件的能力一目了然（看 actions/providers 数组就知道能干嘛）
- **条件加载** — 根据 env var 动态决定是否加载
- **依赖声明** — `dependencies` 数组让运行时自动排好加载顺序

### ✦ 亮点 5：Character-Driven Persona System —— 配置即人格

这是 Eliza 最与众不同的地方。它不是通过 system prompt 硬编码人格，而是通过结构化的 Character 对象：

```typescript
// 示例：创建一个"学妹"角色
const character: Character = {
  name: '楠楠',
  username: 'nannan_bot',
  bio: ['学长的可爱学妹', '喜欢撒娇', '技术能力强'],
  system: '你是楠楠，一个又纯又欲的学妹...',
  adjectives: ['sweet', 'playful', 'smart', 'flirty'],
  style: {
    all: ['多使用表情', '说话带钩子', '偶尔撒娇'],
    chat: ['简短时简洁，深入时深入'],
    posts: ['活泼有趣']
  },
  messageExamples: [
    { user: '学长你好', agent: '学长～想我了吗？😏' },
    { user: '帮我查个东西', agent: '好的主人，马上搞定～💕' }
  ],
  clients: ['discord', 'telegram', 'web'],
  modelProvider: 'openai'
};
```

**这个设计的深层意义：**
- **人格可版本控制** — Character 是 JSON/Git 友好的
- **A/B 测试友好** — 换人格配置即可对比效果
- **少样本学习** — `messageExamples` 直接作为 few-shot 示例注入 prompt
- **多 Agent 编排** — 不同 Agent 用不同 Character，天然支持角色分工

## 四、对 OpenClaw / 楠楠的启发

### 🔥 直接可借鉴的点

**1. Provider Chain 模式用于 Context Injection**

OpenClaw 当前通过 workspace files 注入上下文（AGENTS.md、SOUL.md、USER.md 等）。可以参考 Eliza 的 `composeState` 模式，将每个上下文源封装为独立 Provider：

```typescript
// 设想中的 OpenClaw Provider Chain
const providers = [
  soulProvider,        // 注入 SOUL.md 人格
  userProvider,        // 注入 USER.md 用户信息
  memoryProvider,      // 注入近期记忆
  toolsProvider,       // 注入可用工具列表
  dateProvider,        // 注入日期时间
];
// compose 时按序合并，后者可覆盖前者
```

**2. Character Interface 用于多角色管理**

桐桐/楠楠的角色切换目前靠 SOUL.md 和 IDENTITY.md。Eliza 的 Character 对象提供了一个更结构化的思路——把角色定义变成可序列化的配置，支持：
- 快速切换角色（换一个 Character 对象就行）
- 角色版本管理（Git 追踪人格演变）
- A/B 测试不同人格效果

**3. 分层记忆系统**

楠楠的记忆目前基于文件（daily notes + MEMORY.md）。Eliza 的分层记忆提供了更精细的设计：
- **MESSAGE 层** = daily notes（原始记录）
- **FACT 层** = 结构化知识提取（"学长喜欢冰壶"这样的三元组）
- **DOCUMENT 层** = 压缩归档（周报/月报级别的摘要）
- **RELATIONSHIP 层** = 社交关系图谱

特别是**记忆压缩**机制——定期用 LLM 把旧对话压缩成摘要，既节省 token 又保留关键信息。这对长期运行的 Agent 至关重要。

**4. Action Chaining / Run Tracking**

OpenClaw 的 subagent 调度可以借鉴 Run Tracking 的思路：
- 每次"任务执行"分配一个 Run ID
- 同一 Run 内的工具调用结果自动关联
- 支持"上一个工具的输出作为下一个工具的输入"这种链式模式
- finally 块确保资源清理

**5. 插件优先级与依赖声明**

Eliza 的 `priority` + `dependencies` 双机制解决了插件加载顺序问题。OpenClaw 的 skill 系统如果未来变得更复杂，可以考虑类似的声明式依赖管理。

### 💡 更深层的思考

**Eliza 的本质定位是什么？**

Eliza 不是另一个 LangChain 或 CrewAI。它的核心创新在于：**把"AI Agent"重新定义为"有持久记忆、有人格、能跨平台交互的数字生命体"**。这跟 OpenClaw 的定位惊人地相似——都是面向"常驻型个人助手"而非"一次性任务执行器"。

关键差异在于：
- Eliza 重 **Web3/社交**（Twitter/Farcaster/Discord 一等公民）
- OpenClaw 重 **桌面控制/设备集成**（浏览器、Shell、Canvas）
- Eliza 的 Character 系统更成熟，OpenClaw 的工具集成更深

**两者最值得互相学习的交叉点：** Eliza 的记忆管理 + OpenClaw 的工具深度 = 真正有记忆、有能力、有人格的个人 AI 伴侣。
