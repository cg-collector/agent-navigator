# Voyager：基于大语言模型的开放式具身Agent研究

> 研究报告 #425 | 2026-04-19
> 主题方向：具身Agent + 终身学习
> 研究对象：Voyager — Minecraft开放式终身学习Agent

---

## 一、项目背景

### 1.1 问题域：开放式世界的通用Agent

传统AI方法（强化学习/模仿学习）在开放式世界（如Minecraft）中面临根本性挑战：
- **动作空间原始**：RL通常基于原子动作，难以表达时序扩展行为
- **探索效率低下**：无结构引导时，探索空间呈指数级爆炸
- **灾难性遗忘**：学习新技能时覆盖已有能力
- **缺乏终身学习**：无法持续获取、积累、迁移知识

### 1.2 Voyager的诞生

Voyager由UC Berkeley、Caltech、UT Austin、Stanford、ASU等机构联合提出（2023年5月），是**首个基于LLM的具身终身学习Agent**，在Minecraft中实现了自主探索、技能习得和持续发现。

**核心论文**：*Voyager: An Open-Ended Embodied Agent with Large Language Models* (arXiv:2305.16291)

**核心团队**：Guanzhi Wang等人，导师为Jim Fan（NVIDIA Research）

### 1.3 核心成绩（Empirical Results）

| 指标 | Voyager vs SOTA |
|------|-----------------|
| 独特物品数量 | 3.3x more |
| 旅行距离 | 2.3x longer |
| 木制工具解锁速度 | 15.3x faster |
| 石制工具解锁速度 | 8.5x faster |
| 铁制工具解锁速度 | 6.4x faster |
| 钻石级工具 | **唯一解锁成功的Agent** |

### 2.2 核心设计哲学

**关键决策：用代码作为动作空间**

> "We opt to use code as the action space instead of low-level motor commands because programs can naturally represent temporally extended and compositional actions."

- 不使用原子动作（移动、攻击等原始RL动作）
- 使用**可执行代码程序**作为Agent的动作表示
- 程序天然具备时序扩展性和组合性

### 2.3 三大核心组件

#### 组件1：自动课程（Automatic Curriculum）

**角色**：系统的罗盘，决定"做什么"

**设计原理**：
- 课程由GPT-4根据当前探索进度和Agent状态**自动生成**
- 顶层目标："发现尽可能多的不同事物"（in-context form of novelty search）
- 根据Agent当前技能等级和世界状态动态调整任务难度

**课程生成Prompt示意**：
```
You are a helpful mentor...
Based on the agent's current state and world exploration progress,
propose a task that maximizes exploration...
```

特点：
- 无需人工设计课程
- 任务难度动态适配
- 引导Agent去往技能尚未覆盖的领域

#### 组件2：技能库（Skill Library）

**角色**：系统的记忆，存储"怎么做"

**存储结构**：
```
每个技能 = {
    description_embedding: 向量嵌入,
    code: 可执行Python代码,
    success_rate: 历史成功率
}
```

**技能检索流程**（新任务来临时）：
1. 将新任务描述转为embedding
2. 向量相似度检索top-5相关技能
3. 组合简单技能形成复杂行为
4. 执行组合后的程序

**关键特性**：
- **Compositionality（组合性）**：复杂技能由简单技能合成
- **缓解灾难性遗忘**：已学会的技能永不丢失
- **能力复利**：技能库持续扩大，能力加速增长

#### 组件3：迭代提示机制（Iterative Prompting Mechanism）

**角色**：系统的执行器，解决"如何做"

**三层反馈循环**：

```
GPT-4生成代码
      │
      ▼
┌─────────────┐
│ 1.执行错误  │◀──代码语法/库调用错误
│ (Execution  │
│   Errors)   │
└─────────────┘
      │ 反馈
      ▼
┌─────────────┐
│ 2.环境反馈  │◀──Mineflayer: "I cannot make stoneshovel 
│(Environment │    because I need: 2 more stick"
│  Feedback)  │
└─────────────┘
      │ 反馈
      ▼
┌─────────────┐
│ 3.自我验证  │◀──另一个GPT-4实例作为评论者
│(Self-       │    判断任务是否完成，失败时给出批评建议
│ Verification)│
└─────────────┘
      │
      └───▶ 若通过 → 加入Skill Library
      └───▶ 若失败 → 回到GPT-4重新生成
```

**自我验证（Self-Verification）**：
- 使用第二个GPT-4实例作为验证者
- 输入：当前状态 + 任务描述
- 输出：任务是否完成 + 失败时的改进建议
- 避免了人工编写验证程序

### 2.4 与GPT-4的交互方式

**纯Black-box Prompting，无梯度更新**：
- 无需访问模型参数
- 无需模型微调
- 仅通过in-context learning和prompt engineering驱动

## 三、核心机制详解

### 3.1 代码作为动作空间

传统Agent使用原子动作（left, right, attack...），Voyager使用**可执行程序**：

```python
# 一个简单技能示例
async def craft_stone_axe(bot):
    # 寻找石头
    stone = bot.findBlock...
    # 制作
    await bot craft stone_axe
    # 返回状态
    return "stone_axe_crafted"
```

**为什么代码优于原子动作？**

| 特性 | 原子动作 | 代码动作 |
|------|---------|---------|
| 时序扩展 | 需手动管理状态 | 程序流程自然表达 |
| 组合性 | 困难 | 简单函数组合 |
| 可解释性 | 低 | 高（代码即文档） |
| 复用性 | 差 | 好 |
| 长度 | 短 | 长但有意义 |

### 3.2 终身学习的实现

Voyager的终身学习通过**技能库的累积**实现，而非模型权重的更新：

```
时间步T：
  Skill Library: [技能1, 技能2, ..., 技能N]
  
新任务到达：
  1. 检索相关技能 → top-5相似技能
  2. 组合简单技能 → 复杂行为
  3. 迭代优化 → 通过三层反馈
  4. 存储新技能 → Library增大
  
结果：
  - 模型权重不变（无遗忘）
  - 技能库持续增长（知识积累）
  - 能力复利（Compositionality）
```

### 3.3 与其他Agent范式对比

| Agent | 推理方式 | 动作空间 | 反馈来源 | 终身学习 |
|-------|---------|---------|---------|---------|
| ReAct | Thought + Action交替 | 原子动作 | 环境 | ❌ |
| AutoGPT | 递归任务分解 | 工具调用 | 人工 | ❌ |
| Reflexion | 自我反思 | 原子动作 | 语义反馈 | ❌ |
| **Voyager** | 自动课程+代码生成 | **可执行代码** | **三层反馈** | ✅ |

### 4.2 核心循环（伪代码）

```python
class VoyagerAgent:
    def __init__(self):
        self.skill_library = SkillLibrary()
        self.llm = GPT4()
    
    def run(self):
        while not terminated:
            # Step 1: 自动课程提出任务
            task = self.curriculum_agent.propose(
                exploration_progress=self.get_progress(),
                agent_state=self.get_state()
            )
            
            # Step 2: 检索相关技能
            relevant_skills = self.skill_library.retrieve(task)
            
            # Step 3: 迭代生成+反馈优化
            skill_code = self.iterative_prompting(
                task=task,
                context=relevant_skills
            )
            
            # Step 4: 存入技能库
            self.skill_library.add(skill_code)
            
    def iterative_prompting(self, task, context):
        code = ""
        for iteration in range(MAX_ITERATIONS):
            # 生成代码
            code = self.llm.generate(
                prompt=build_prompt(task, context, code)
            )
            
            # 执行并收集反馈
            try:
                execute(code)
                env_feedback = get_env_feedback()
            except ExecutionError as e:
                execution_error = str(e)
                env_feedback = None
            
            # 自我验证
            verified = self.critic.verify(task, code, env_feedback)
            
            if verified:
                return code
            else:
                # 准备下一轮迭代
                context = update_context(code, env_feedback, execution_error)
```

### 4.3 技能库的数据结构

```python
class SkillLibrary:
    def __init__(self):
        self.skills = []  # List[Skill]
        self.embedder = OpenAIEmbedder()
    
    def add(self, skill_code, description):
        embedding = self.embedder.embed(description)
        self.skills.append({
            "code": skill_code,
            "description": description,
            "embedding": embedding,
            "success_count": 1
        })
    
    def retrieve(self, task_description, top_k=5):
        query_emb = self.embedder.embed(task_description)
        # 向量相似度检索
        scores = [cosine_sim(query_emb, s["embedding"]) 
                  for s in self.skills]
        top_indices = np.argsort(scores)[-top_k:]
        return [self.skills[i] for i in top_indices]
```

### 4.4 Prompt工程细节

**课程生成Prompt结构**：
1. 格式Schema说明（ priming LLM接收信息的方式）
2. 推理步骤指导（高度具体的Minecraft世界知识）
3. Few-shot示例（展示如何响应）

**关键技术**：Persona Prompting（告诉LLM它是"mentor"），使其回答更果断，避免"分析瘫痪"。

## 五、优缺点分析

### 5.1 优点

#### ✅ 真正的终身学习
- 技能库持续增长，能力随时间累积
- 无需重新训练模型，规避灾难性遗忘
- 在新世界中可直接复用已有技能库（zero-shot generalization）

#### ✅ 代码作为动作空间的优雅设计
- 组合性使能力呈指数增长
- 可解释性强（代码即文档）
- 时序扩展行为自然表达

#### ✅ 三层反馈机制
- 执行错误 → 捕捉代码级错误
- 环境反馈 → 来自Minecraft的真实状态
- 自我验证 → 无需人工编写验证器
- 三者互补，覆盖错误来源全面

#### ✅ 超越人类的探索效率
- 15.3x faster解锁木制工具
- 唯一解锁钻石级工具的Agent
- 3.3x更多独特物品发现

### 5.2 缺点/局限性

#### ❌ 极度依赖GPT-4的能力
- 黑盒调用，无法微调模型本身
- GPT-4的推理能力上限决定了Agent上限
- 若换成较弱模型，性能可能大幅下降

#### ❌ 反馈循环的成本问题
- 每个技能迭代多次LLM调用
- 自我验证需两个GPT-4实例
- 成本随技能库增大而累积

#### ❌ 技能库不更新
- 论文未讨论已存储技能的更新/改进机制
- 技能一旦存入，保持原状
- 可能导致过时技能堆积

#### ❌ Minecraft特定性
- 高度定制化的Minecraft环境
- 迁移到其他开放世界需要大量适配
- Mineflayer API是强依赖

#### ❌ 自我验证的可靠性
- 依赖GPT-4作为评论者，非严格验证
- 可能产生"自我欺骗"——技能声称成功但实际未达成
- 论文中"3次试验"的设计暗示验证存在不稳定性

#### ❌ 探索策略的隐含风险
- 自动课程基于"最大探索"目标，可能产生低效行为
- 无明确利用已知的有效路径
- 可能反复尝试相似任务

## 六、启示与思考

### 6.1 对AI Agent设计的启示

**1. 代码作为动作空间可能是通用Agent的关键**

Voyager最重要的贡献之一是证明了**代码作为动作表示**的优越性。这与传统的原子动作空间相比，提供了：
- 自然的长度/复杂性管理
- 内置的组合性（函数调用链）
- 完美可解释性

这对通用Agent设计具有深远意义：未来的Agent或许应该"写出程序"而非"执行动作"。

**2. 外部记忆是终身学习的关键基础设施**

Voyager的技能库本质上是一个**外部化、可查询的记忆系统**。与模型权重存储知识相比，外部记忆：
- 不会遗忘
- 可以持续增长
- 可被快速检索

这提示我们：通用Agent可能需要"记忆模块"作为核心组件，而非依赖模型本身的知识压缩。

**3. 三层反馈是迭代优化的基础**

执行错误 + 环境反馈 + 自我验证的三层架构，构成了一个完整的**错误发现-修复循环**。这比单一反馈源更健壮：
- 执行错误：捕获语法/类型错误
- 环境反馈：捕获语义/逻辑错误
- 自我验证：捕获任务级别错误

**4. 自动课程优于人工设计课程**

让LLM自己生成课程任务，避免了人工设计课程的瓶颈，且能动态适配Agent能力。这启发我们：**元认知（知道自己该学什么）可以外包给LLM**。

### 6.2 对具身AI研究的启示

**1. Minecraft是具身AI的理想试验场**

Minecraft提供了：
- 开放式、无固定目标的真实环境
- 丰富的工具使用和合成系统
- 可量化的进度指标（物品收集、技能解锁）
- 安全且可重复的实验环境

**2. "具身"不仅是物理身体，还包括"感知-动作循环"**

Voyager的具身体现为：LLM生成代码 → 代码控制bot → bot在Minecraft中行动 → 观察结果反馈。这种循环让Agent的"身体"成为可操控的环境接口。

### 6.3 对通用人工智能的启示

**1. 组合性是智能的核心**

Voyager的能力复利来自技能的可组合性。这印证了Bommasani等人关于"涌现能力"的观点：**组合性可能是通用智能的关键机制**。

**2. 终身学习不等于持续训练**

传统观点认为终身学习需要持续更新模型权重。Voyager证明了：**在固定模型下，通过外部记忆的累积可以实现终身学习**。这大幅降低了持续训练的必要性。

**3. "学习"可以完全通过Prompting实现**

Voyager没有使用任何梯度更新，仅通过prompt engineering和in-context learning实现复杂技能习得。这挑战了"学习=参数更新"的传统观念。

---

## 七、总结

Voyager是具身Agent领域的一项里程碑式工作，它通过三大创新组件（自动课程、技能库、迭代提示）实现了首个真正具备终身学习能力的LLM驱动的具身Agent。

**最核心的洞见**：
> 用代码作为动作空间 + 外部技能库作为记忆 + 三层反馈驱动迭代 = 可持续增长的通用Agent

这为通用人工智能的研究指明了一条新路径：**不一定需要更大的模型，而需要更好的架构来累积和复用知识**。
