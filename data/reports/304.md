# 📚 AI Agent 项目研究报告 #304 — UFO²：Windows 桌面 AgentOS

## 一、项目背景与问题定义

UFO²（The Desktop AgentOS）是微软研究院于2025年4月发布的开源桌面自动化框架，代表了从"概念原型"向"系统级自动化"的范式跃迁。其核心论文发表于 arXiv:2504.14603，代码以 MIT 协议开源于 github.com/microsoft/UFO。

**解决的核心痛点有三个：**

1. **浅层OS集成问题**：现有 Computer-Using Agents（CUAs，如 Claude、Operator）本质上只是"截图→模拟鼠标键盘"的循环，完全绕过了操作系统提供的结构化能力（UI Automation API、应用程序内部接口），导致操作脆弱且效率低下。

2. **执行干扰问题**：传统 CUA 在执行任务时会独占用户的鼠标和键盘，用户无法同时使用电脑——这对实际落地是致命的 UX 障碍。

3. **单步推理瓶颈**：现有 agent 采用"观察→思考→执行一步→再观察"的串行模式，对于需要连续填写10个表单字段的简单任务，也要调用10次 LLM，延迟和成本极高。

UFO² 的定位不是又一个 GUI Agent，而是一个**操作系统级的 Agent 抽象层（AgentOS）**——将桌面自动化提升为与文件系统、进程管理同等级别的一等公民。

## 二、核心架构深度剖析

### 2.1 双层多Agent层级结构

UFO² 采用 HostAgent + AppAgents 的两层架构，通过 Blackboard 模式解耦：

```
用户请求
    ↓
HostAgent（桌面编排器）
    ├── 任务分解（Task Decomposition）
    ├── 应用选择（Application Selection）
    ├── 跨应用协调（Cross-app Coordination）
    └── AppAgent 生命周期管理
        ↓
AppAgent₁（Excel专用）  AppAgent₂（Word专用）  AppAgent₃（Outlook专用）
    ├── UI元素感知（截图 + UIA树）
    ├── 混合动作执行（GUI / Native API）
    └── 结果回传至Blackboard
```

**HostAgent** 负责 WHAT 和 WHEN——选择哪个应用、什么顺序执行。它实现了一个 **7状态有限状态机（FSM）**：INIT → DESKTOP_ANALYSIS → APP_SELECTION → SUBTASK_DELEGATION → MONITORING → VERIFICATION → COMPLETION。关键设计决策是 HostAgent 不直接操作 UI，而是将子任务委派给专业化的 AppAgent。

**AppAgent** 负责 HOW 和 WHERE——具体怎么操作、点哪里。每个 AppAgent 是一个 ReAct 循环，拥有 **6状态 FSM**：INIT → OBSERVATION → PLANNING → ACTION → VALIDATION → REPORTING。每个 AppAgent 针对特定应用加载领域知识（如 Excel 的 xlwings API、Word 的 python-docx 接口）。

### 2.4阶段处理流水线

两个 Agent 共享相同的4阶段处理周期：

| 阶段 | HostAgent策略 | AppAgent策略 |
|------|--------------|-------------|
| 数据采集 | 桌面截图 + 应用列表 | 应用截图 + UIA树 + 控制标注 |
| LLM交互 | 选择应用、规划子任务 | 选择控件、规划动作序列 |
| 动作执行 | 启动应用、创建AppAgent | 执行GUI/API混合动作 |
| 记忆更新 | 保存编排步骤 | 保存交互步骤、更新Blackboard |

### 2.3 统一GUI–API动作层（Puppeteer引擎）

这是 UFO² 最核心的工程创新。传统 CUA 只有 click/type/scroll 三种原语。UFO² 通过 Puppeteer 引擎动态选择最优执行路径：

**Native API 路径（优先）：**
- Excel → xlwings 直接操作单元格/图表
- Outlook → win32com 发送邮件
- PowerPoint → python-pptx 编辑幻灯片
- 文件系统 → shutil/os 原生操作

**GUI Action 路径（降级）：**
- 当目标应用没有可用 API，或 API 操作失败时，自动降级为坐标点击/文本输入

选择逻辑由一个轻量级路由器决定：先检查 MCP Server 是否注册了该应用的 API 能力 → 有则走 API → 否则走 GUI。这种设计使得新增应用支持只需编写对应的 MCP Server，无需修改 Agent 核心代码。

### 2.4 混合控制检测管道

UFO² 的控制元素检测融合了两条路径：

**路径A - UI Automation（UIA）：** 利用 Windows 内置的无障碍树（Accessibility Tree）获取标准控件的语义信息（名称、类型、边界矩形、状态）。优势是准确、稳定；劣势是无法识别自定义绘制的非标准控件（如 Canvas 渲染的按钮）。

**路径B - 视觉解析（OmniParser）：** 对截图运行视觉模型（基于 OmniParser 的 icon detection + text detection），检测非标准 UI 元素。优势是通用；劣势是有噪声、速度慢。

**融合策略：** 先尝试 UIA 匹配 → 若置信度低于阈值或匹配失败 → 启动视觉解析作为补充 → 合并两组候选 → 按置信度排序输出。实验表明，纯 UIA 在标准 Win32 应用上覆盖率达 85%，加入视觉后提升至 96%+。

## 三、技术亮点

### 亮点1：推测性多动作执行（Speculative Multi-Action Execution）

**原理：** 传统 CUA 的"一步一调"模式在重复性操作上浪费大量 LLM 推理。UFO² 借鉴 CPU 分支预测的思想，让 LLM 在一次推理中输出一个**动作序列**而非单个动作，然后用轻量级验证器逐条校验。

**具体机制：**
1. LLM 输出格式变为 JSON 数组：`[{"action": "click", "target": "A1"}, {"action": "type", "value": "1"}, {"action": "click", "target": "A2"}, ...]`
2. Puppeteer 引擎在执行每步前检查前置条件（控件是否可见、是否可交互、前序动作是否成功）
3. 若某步验证失败，丢弃后续所有未执行动作，回滚到失败点重新规划

**效果：** 论文报告 **LLM 调用减少 51%**。以"在 Excel 中填充 A1-A10 单元格"为例：传统方式需 10 次 LLM 调用（~30秒），UFO² 仅需 1 次（~8秒）。这不仅是速度提升，更是成本的大幅降低。

### 亮点2：Picture-in-Picture（PiP）虚拟桌面

**原理：** 利用 Windows Remote Desktop 的 loopback 特性创建嵌套虚拟桌面。Agent 在 PiP 桌面中执行自动化任务，用户在主桌面正常工作，两者完全隔离。

**技术实现：**
- 基于 Windows RDP ActiveX 控件的本地回环连接
- PiP 桌面作为独立 Session 运行，拥有自己的窗口管理器和输入队列
- Agent 的鼠标/键盘事件被路由到 PiP Session，不会泄漏到主桌面
- 用户可通过悬浮窗口观察 Agent 执行过程（可关闭）

**意义：** 这解决了 CUA 落地的最大 UX 障碍——用户不需要"让出电脑"。Agent 可以在后台安静地生成 Excel 报告，而用户同时在主桌面上写邮件、浏览网页。这个设计使得 UFO² 可以部署到真实办公环境而非仅限实验室。

### 亮点3：持续知识基底（Knowledge Substrate）

**原理：** UFO² 实现了一个三层 RAG 系统，Agent 在不重新训练的情况下持续学习：

| 知识源 | 内容 | 接入方式 |
|--------|------|---------|
| 应用帮助文档 | 官方API参考、操作手册 | 向量化检索（Embedding + Vector DB） |
| Bing Search | 实时网络知识（新功能、更新日志） | 动态查询扩展 |
| 执行历史 | 过去成功/失败的动作序列 | 经验回放 + 模式挖掘 |

**工作流：** 当 AppAgent 面临不确定的操作时（例如"如何在 Excel 中创建数据透视表"），系统自动从知识基底中检索相关文档片段和历史成功案例，注入到 LLM 的上下文窗口中作为 few-shot 示例。这意味着随着使用时间增长，UFO² 会变得越来越熟练——类似于人类员工的"经验积累"。

## 四、基准测试表现

UFO² 在两个主流 Windows 自动化基准上进行了评估：

### Windows Agent Arena（WAA）

| Agent | 成功率(SR) | 相对Operator提升 |
|-------|-----------|----------------|
| **UFO²** | **~28.5%** | **+99.3%（接近翻倍）** |
| Operator (OpenAI) | 14.3% | baseline |
| Claude (Anthropic) | ~18-20% | +26-40% |

UFO² 在 WAA 上的成功率几乎是 Operator 的两倍，证明了深度 OS 集成的显著优势。

### OSWorld-Windows 子集

在 OSWorld 的 Windows 任务子集上，UFO² 同样展现出领先表现：
- 平均完成步骤数减少约 40%（得益于 API 直连和推测性执行）
- 任务成功率比纯视觉基线高 10-15 个百分点
- 在 Office 套件任务（Excel/Word/PPT）上的优势尤为明显（API直连+领域知识的组合效应）

### 关键发现

论文揭示了一个重要洞察：**架构设计比模型规模更重要**。UFO² 使用 GPT-4o 即可实现超越使用更强模型但采用纯截图方案的 CUA。这说明当前 GUI Agent 的瓶颈不在模型能力，而在如何有效利用操作系统提供的结构化信息。

## 五、横向对比分析

| 维度 | UFO² | OpenAI Operator | Claude Computer Use | CUA (Anthropic) | OmegaUse |
|------|------|----------------|-------------------|-----------------|----------|
| **平台** | Windows only | Web + Desktop | Web + Desktop | macOS/iOS/Win/Linux | Mobile + Desktop |
| **OS集成深度** | ⭐⭐⭐⭐⭐ UIA+Win32+COM | ⭐⭐ 截图为主 | ⭐⭐⭐ 截图+部分API | ⭐⭐⭐ 截图+工具调用 | ⭐⭐⭐ 视觉为主 |
| **执行方式** | 混合GUI+API | 模拟输入 | 模拟输入+工具 | 模拟输入 | 模拟输入 |
| **多App协作** | ✅ HostAgent编排 | ❌ 弱 | ⚠️ 有限 | ⚠️ 有限 | ✅ 支持 |
| **用户并行使用** | ✅ PiP桌面 | ❌ 独占 | ❌ 独占 | ❌ 独占 | ❌ 独占 |
| **持续学习** | ✅ RAG知识基底 | ❌ | ❌ | ❌ | ❌ |
| **推测执行** | ✅ 51%减少调用 | ❌ | ❌ | ❌ | ❌ |
| **开源** | ✅ MIT | ❌ 商业闭源 | ❌ 商业闭源 | ❌ 商业闭源 | ✅ 开源 |

**UFO² 的优势：**
- 工程成熟度最高：微软出品，文档完善，MCP生态集成
- 效率最优：推测执行+API直连大幅降低延迟和成本
- 唯一支持用户并行的方案：PiP桌面是独有特性
- 可扩展性强：MCP Server 机制使新增应用只需写插件

**UFO² 的劣势：**
- 平台锁定：仅支持 Windows，无法跨平台
- 设置复杂度较高：需要配置多个组件（RDP、UIA权限、各应用MCP Server）
- 依赖 GPT-4o 级别模型：弱模型上效果下降明显
- 仅覆盖桌面场景：不支持移动端/Web端任务

## 六、对我们的实际启发

### 可落地方向1：混合执行引擎的设计模式

UFO² 的 Puppeteer 引擎思路可以直接借鉴到我们的 browser agent 中。当前我们的浏览器操作全部走 screenshot → click 的路径，但对于某些高频操作（如表单填写、导航跳转），可以预定义"快捷路径"——直接通过 CDP（Chrome DevTools Protocol）执行 JavaScript 或 DOM 操作，仅在快捷路径失败时降级为视觉点击。预期可以减少 30-50% 的操作步骤。

### 可落地方向2：推测性批量执行

对于"打开页面→填表→提交"这类固定流程的任务，可以让 LLM 一次性输出完整动作列表，然后逐条验证执行。这在飞书文档批量编辑、表格数据处理等场景尤其有价值。实现上只需要修改 prompt 模板让 LLM 输出 JSON array 格式的动作序列即可。

### 可落地方向3：知识基底的轻量实现

不需要完整的 RAG 管道，但可以为常用操作维护一个 Markdown 知识库（类似我们现有的 SKILL.md 体系），在 Agent 执行相关任务时自动注入上下文。比如"如何在飞书多维表格中创建视图"这类操作指南，可以作为 system prompt 的动态附件按需加载。
