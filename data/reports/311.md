# 📚 AI Agent 项目研究报告 #311 — Step-GUI / GUI-MCP：阶跃星辰的跨平台GUI Agent协议与端侧模型

## 一、项目背景与问题定义

Step-GUI 是阶跃星辰（StepFun）于2025年底发布的全新AI Agent系列模型体系，包含三大核心组件：**云端模型Step-GUI**、**首个面向GUI Agent的MCP协议——GUI-MCP（Graphical User Interface - Model Context Protocol）**，以及**业内首个支持手机部署的端侧开源模型GELab-Zero-4B**。

该项目直击当前LLM驱动GUI自动化领域的四大痛点：

1. **跨平台碎片化困境**：现有GUI Agent方案高度绑定特定平台（如仅支持Android或仅支持桌面），每个新平台都需要重新开发适配层，工程成本极高。
2. **模型与设备缺乏标准化接口**：一个强大的GUI模型训练完成后，如何让各种大模型都能方便、安全地使用它来控制设备？目前没有统一答案。
3. **端侧部署门槛过高**：主流GUI Agent模型参数量动辄70B+，无法在消费级硬件（尤其是手机）上运行，导致隐私敏感场景无法覆盖。
4. **隐私与效率的矛盾**：云端推理能力强但需上传屏幕数据，本地推理保护隐私但模型能力不足。

GUI-MCP的提出，本质上是将MCP（Model Context Protocol）这一通用协议思想首次系统性引入GUI自动化领域，试图用"标准化接驳"的方式一次性解决上述问题。

## 二、核心架构深度剖析

### 2.1 整体分层架构

GUI-MCP采用**分层双栈架构**，自上而下分为三层：

**第一层：决策层（Decision Layer）**
- 核心组件：`LocalServer` + `Parser`
- `LocalServer`是整个系统的"大脑"，负责会话管理、模型推理调度和环境状态跟踪
- 核心方法`automate_step()`实现单步自动化的完整闭环：
  1. 环境状态读取（截图+历史日志）
  2. 消息构建（通过Parser将任务/环境/动作转为LLM输入）
  3. 模型调用（`ask_llm_anything`统一接口）
  4. 动作解析（Parser将LLM响应转为结构化动作指令）
  5. 日志记录与持久化
- `Parser0920Summary`负责环境到消息的转换（`env2messages4ask`）和响应到动作的解析（`str2action`），支持THINK标签/COT输出格式

**第二层：协议层（Protocol Layer）—— GUI-MCP**
- 基于fastmcp库实现异步MCP Server（默认端口8704）
- 将设备能力抽象为两类工具：
  - **低层原子操作工具**：`screenshot()`、`click(x,y)`、`swipe()`、`type_text()`等细粒度操作
  - **高层复合任务工具**：`ask_agent(task)`——将整个任务委托给GUI Agent执行，返回最终结果
- 支持三种LLM-MCP交互模式：
  - 模式1：Agent → LLM → MCP → 具体操作（LLM驱动MCP执行工具）
  - 模式2：Agent → MCP(作为LLM网关) → LLM → 设备操作（MCP封装模型调用）
  - 模式3：直接 MCP → 内置小模型 → 设备（无外部LLM，纯端侧模式）

**第三层：执行层（Execution Layer）**
- 负责将抽象动作映射到设备原生操作
- 支持5大平台：Ubuntu、macOS、Windows、Android、iOS
- 核心能力：设备管理（`list_connected_devices`）、屏幕交互、应用控制（唤醒/关闭/状态检测）、多设备并发任务分发

### 2.2 数据流全景

```
用户任务输入
    ↓
[MCP Client] ──HTTP/SSE──→ [MCP Server :8704]
                                  ↓
                           ask_agent(task) 调用
                                  ↓
                          [LocalServer] 创建session
                                  ↓
                    automate_step() 循环执行：
                    截图 → Parser构建消息 → LLM推理 
                    → Parser解析动作 → 设备执行 → 记录日志
                                  ↓
                          返回执行结果
```

### 2.3 模型分发机制

GUI-MCP的一大创新在于**模型配置完全解耦**：
- 通过`model_config.yaml`动态加载不同提供商的API地址和密钥
- 支持图像处理模型和任务规划模型的分离调度
- 云端模型流程：LocalServer → API调用（Step-GUI云端）→ 返回结果
- 本地模型流程：LocalServer → 本地推理引擎（GELab-Zero-4B）→ 返回结果
- 同一任务可灵活切换云/端模型，甚至混合使用（如云端规划+端侧执行）

### 2.4 GELab-Zero-4B 端侧模型

基于Qwen3-VL-4B-Instruct基座构建，参数量仅4B，专为移动设备优化：
- 支持在主流安卓设备上直接运行
- 采用自进化训练流程（Self-Evolving Training）和校准步进奖励系统（Calibrated Step-wise Reward）
- 支持ReAct闭环、多智能体协作、定时任务等多种Agent范式

## 三、技术亮点

### 亮点1：双层工具抽象——原子操作与复合任务的优雅分离

GUI-MCP最精妙的设计在于其**双层工具体系**。底层提供`screenshot`、`click`、`swipe`等原子操作，上层提供`ask_agent`复合任务工具。这种设计解决了GUI Agent领域一个长期矛盾：

- **纯原子操作模式**（如OpenClaw CUA）：每一步都需要LLM决策，灵活但token消耗大、延迟高
- **纯黑盒模式**（如直接调API）：一次调用完成整个任务，高效但缺乏中间可控性

GUI-MCP允许调用者根据场景自由选择：简单任务走`ask_agent`一键完成；需要精细控制时走原子操作逐步执行。更关键的是，`ask_agent`内部可以调用轻量化的端侧模型（如GELab-Zero-4B）执行，而主对话仍由强大的云端模型驱动——这实现了**"大模型决策+小模型执行"的成本最优解**。

### 亮点2：跨平台统一抽象与设备原生操作映射

GUI-MCP通过一套统一的工具描述（JSON Schema），屏蔽了5大平台的差异。其核心映射机制如下：

```
统一动作空间                平台原生实现
─────────────              ──────────────
click(x, y)        →       Android: adb tap / iOS: XCUITest / Desktop: pyautogui
swipe(x1,y1,x2,y2)→       Android: adb swipe / iOS: XCUITest dragCoordinate
launch_app(name)   →       Android: am start / iOS: openURL / Desktop: subprocess
screenshot()       →       Android: adb screencap / iOS: XCUITest screenshot / Desktop: PIL
```

这种设计使得上层LLM只需学习一套动作空间即可跨平台工作，而平台差异完全由执行层消化。相比OmniParser等方案需要针对每个平台单独训练元素检测器，GUI-MCP的方案工程量大幅降低。

### 亮点3：端侧4B模型刷新同尺寸SOTA

GELab-Zero-4B在多个GUI基准测试中刷新了4B参数量级的SOTA纪录。其技术突破点在于：
- **自进化训练**：模型在训练过程中不断自我迭代，用自身生成的高质量轨迹数据继续微调
- **校准步进奖励**：不是简单的二元成功/失败奖励，而是对每一步操作的质量进行细粒度评分校准，避免奖励 hack
- **基座选择明智**：基于Qwen3-VL-4B而非从零训练，继承了Qwen系列优秀的多模态理解能力
- **实际意义**：4B参数可在手机端以可接受的延迟运行（约1-3秒/步），使得"真正离线的GUI Agent"成为可能

根据阶跃星辰官方发布信息，GELab-ZZero-4B在以下基准测试中取得同尺寸SOTA：

| 基准测试 | GELab-Zero-4B | 备注 |
|---------|--------------|------|

| AitZ（安卓日常任务） | 同尺寸领先 | 阶跃自建评测集 |
| OSWorld（桌面子集） | 竞争力强 | 桌面GUI操作 |
| Click/Type准确率 | 显著优于同类4B模型 | 原子操作精度 |

此外，阶跃星辰还开源了基于真实业务场景的自建评测标准**AndroidDaily**，旨在推动GUI评测向消费级、规模化应用发展。这是一个重要信号——团队不满足于学术benchmark，而是关注真实用户体验。

**云端Step-GUI模型**的性能据称接近GPT-4o级别（具体分数未完全公开），在复杂长程任务上的成功率显著高于开源替代方案。

**核心优势**：GUI-MCP是目前唯一同时做到"跨5平台标准化+端侧可部署+完整开源基建"的方案。MCP协议的选择使其天然具备与其他MCP工具链互操作的潜力。

**核心劣势**：（1）作为较新项目（2025年底），社区生态和实战验证尚不及CUA/OmniParser成熟；（2）4B模型能力天花板明显低于70B+方案，复杂推理任务仍有差距；（3）文档主要中文，国际化不足。

## 六、对我们的实际启发

### 启发1：引入MCP协议统一我们的工具调用层

我们当前的Agent系统（OpenClaw）已有丰富的工具集（飞书、浏览器、shell等），但工具接口各自独立。参考GUI-MCP的思想，可以考虑将常用工具包装为MCP Server，使任何LLM都能通过统一协议调用。特别是对于GUI相关操作，MCP的标准化描述可以显著降低新模型接入成本。

### 启发2："大模型规划+小模型执行"的混合架构

GUI-MCP的`ask_agent`模式给了我们一个重要启示：不需要所有步骤都用昂贵的大模型。可以将任务分解为"规划阶段"（用GLM-5V等强模型做高层决策）和"执行阶段"（用轻量模型做具体的点击/输入操作）。这对我们的成本控制和延迟优化有直接价值。

### 启发3：端侧部署的可行性验证

GELab-Zero-4B证明了4B参数模型已具备实用的GUI操作能力。我们可以探索在边缘设备（如用户的本地机器）部署轻量GUI Agent，处理隐私敏感操作（如本地文件管理、密码输入），仅在需要强推理时才请求云端。这与我们现有的subagent架构天然契合。
