# 开源Agent项目学习 #042

## 项目名称
MetaGPT

## GitHub链接
https://github.com/FoundationAgents/MetaGPT

## 一句话简介
模拟真实软件公司运作流程的多智能体框架——给一行需求，输出完整的软件项目（PRD、架构设计、代码、文档）。

## 项目架构

MetaGPT 的核心理念是 **Code = SOP(Team)**，即把标准操作流程（SOP）具象化并应用到由 LLM 组成的协作团队中。

### 整体架构

1. **Role（角色层）**：每个 Agent 扮演一个专业角色（产品经理、架构师、项目经理、工程师、QA工程师等），每个角色有独立的 Prompt、工具集和责任范围。

2. **Action（动作层）**：角色通过执行具体的 Action 来完成工作，比如 WritePRD、WriteDesign、WriteCode、RunCode 等。

3. **Memory（记忆层）**：每个角色拥有独立的记忆系统（短期 + 长期），支持从历史交互中学习和恢复上下文。

4. **Environment（环境层）**：提供消息池（Message Pool）作为通信总线，采用**发布-订阅模式**而非直接的 Agent 间调用，保证通信的可追溯性和可扩展性。

5. **SOP（流程引擎）**：定义角色之间的协作顺序和标准输入输出格式，确保每个环节产出结构化文档（PRD → 系统设计 → 任务列表 → 代码 → 测试）。

### 协作流程

```
用户需求 → 产品经理(PRD) → 架构师(系统设计) → 项目经理(任务拆分) → 工程师(编码) → QA(测试审查)
```

每个角色的输出通过共享消息池传递给下一个角色，形成流水线式协作。

## 技术亮点

- **SOP 驱动的结构化输出**：不同于其他框架的自由对话式多 Agent 协作，MetaGPT 强制每个角色按照标准流程产出结构化文档（PRD、数据结构、API 接口文档等），大幅降低了幻觉和遗漏。

- **共享消息池架构**：采用发布-订阅模式而非点对点通信，解决了多 Agent 系统中消息混乱、死锁、循环依赖等经典问题，通信链路清晰可审计。

- **人类工作流的精确模拟**：不是简单地把任务分给多个 Agent 并行处理，而是真正模拟软件公司的分工协作流程，每个角色的职责边界清晰，产出物可预期。

- **学术底蕴深厚**：ICLR 2025 Oral（top 1.8%），SPO（Sequential Process Optimization）和 AOT（Action-Oriented Thinking）等前沿论文支撑，2025年推出商业化产品 MGX（MetaGPT X）。

- **Data Interpreter 模式**：除了软件开发流水线，还提供数据分析角色的交互式编程能力，支持探索性数据分析和代码执行。

## 适用场景

- **软件项目快速原型**：输入一句需求描述，自动生成包含 PRD、架构设计、代码、文档的完整项目骨架
- **多角色协作建模**：需要模拟不同专业角色（PM、架构师、工程师）协作流程的场景
- **结构化文档生成**：从需求自动产出标准化的产品文档和技术文档
- **AI Agent 流程编排研究**：研究 SOP 驱动的多 Agent 协作范式
- **数据分析自动化**：通过 Data Interpreter 角色进行探索性数据分析

## 对我们的启发

1. **SOP 思想可借鉴到 OpenClaw 的 Skill 系统中**：当前 OpenClaw 的 SKILL.md 已经是 SOP 的雏形——定义了何时触发、执行什么步骤。MetaGPT 的做法提醒我们可以进一步将 Skill 的输出标准化，比如让某些 Skill 强制产出结构化结果（JSON/Markdown），便于下游消费。

2. **发布-订阅消息模式 vs 直接调用**：OpenClaw 当前的 subagent 机制更像是直接调用模式。对于更复杂的多 Agent 协作场景（比如同时跑多个 subagent 并汇总结果），可以借鉴 MetaGPT 的共享消息池思路，引入任务队列和结果聚合机制。

3. **角色专业化**：楠楠可以不只是"一个全能助手"，而是根据场景切换专业角色。比如写代码时进入"工程师模式"，分析需求时进入"产品经理模式"，每个模式有独立的 prompt 和工具集。这可以通过扩展 Skill 系统来实现。

4. **结构化输出减少幻觉**：MetaGPT 证明约束 Agent 的输出格式（而非让它们自由发挥）反而能提高质量。这对楠楠的实践也有启发——在执行复杂任务时，先定义好输出模板再执行，比自由发挥更可靠。
