## 项目架构

### 核心抽象层（从底到顶）

```
┌─────────────────────────────────────────┐
│            Application Layer            │
│   RAG / 分类 / 信息抽取 / Agent 循环     │
├─────────────────────────────────────────┤
│           Optimizer Layer               │
│  MIPROv2 / GEPA / SIMBA / Bootstrap    │
│  （自动搜索最优 prompt + few-shot 组合） │
├─────────────────────────────────────────┤
│           Module Layer                  │
│  Predict / ChainOfThought / ReAct /     │
│  ProgramOfThought / 自定义 Module       │
├─────────────────────────────────────────┤
│          Signature Layer                │
│  类型化 I/O 契约                         │
│  "question, context -> answer"          │
├─────────────────────────────────────────┤
│         Adapter + LM Layer             │
│  ChatAdapter / JSONAdapter             │
│  OpenAI / Anthropic / Gemini / Ollama  │
└─────────────────────────────────────────┘
```

### 五大核心概念

1. **Signature（签名）**：声明式 I/O 契约。用 Python 类型系统定义输入输出字段，支持 str / list / dict / Literal 枚举 / Pydantic 模型。例如 `"question -> answer: float"` 或完整的 class 定义。

2. **Module（模块）**：执行策略。`Predict`（直接预测）、`ChainOfThought`（思维链）、`ReAct`（推理+行动循环）、`ProgramOfThought`（生成代码执行）。可自由组合嵌套，类似 PyTorch 的 `nn.Module`。

3. **Adapter（适配器）**：控制 LLM 通信格式。`ChatAdapter` 通用，`JSONAdapter` 强制 JSON 输出（API 集成神器）。一处配置全局生效。

4. **Optimizer（优化器）**：**DSPy 的灵魂**。不是手调 prompt，而是用算法自动搜索最优的指令 + few-shot 示例组合：
   - **MIPROv2**：贝叶斯优化，联合搜索指令和示例，支持 light/medium/heavy 三档
   - **GEPA**：反馈驱动的 prompt 进化，维护 Pareto 前沿，用错误反馈迭代突变最弱模块
   - **SIMBA**：黑盒搜索，暴力探索指令空间，适合其他优化器碰到瓶颈时
   - **BootstrapFewShot / Finetune**：快速收集 demo，可进一步微调模型权重

5. **Evaluate（评估）**：内置评估器 + 指标（exact match、F1、passage match），支持 trace 级别的细粒度诊断。

### 工作流程

```
定义 Signature → 组装 Module → 写 Metric → 收集 trainset
       ↓
Optimizer.compile(program, trainset, metric)
       ↓
   自动 Bootstrap（收集高分轨迹）
       ↓
   自动 Propose（生成指令候选）
       ↓
   自动 Search（贝叶斯/进化搜索最优组合）
       ↓
   输出 Compiled Program（优化后的 pipeline）
       ↓
   Evaluate + 部署
```

## 技术亮点

### 1. 编程范式革命：Prompt 即代码

传统做法：手写 prompt → 换模型就崩 → 手动调 → 再崩。
DSPy 做法用类型化的 Python 代码描述任务，让 optimizer 自动生成最优 prompt。**换模型只需改一行 `dspy.configure(lm=...)`，pipeline 无需任何改动。**

### 2. MIPROv2 优化器：贝叶斯搜索 Prompt 空间

这个是真正硬核的：
- 先 bootstrap：用当前程序跑训练集，收集高分的执行轨迹作为 demo 候选
- 再 propose：用 LLM 生成多条候选指令（基于 demo 总结）
- 最后 search：贝叶斯优化评估指令+demo 的组合，找到全局最优解

论文发表在 ICLR 2024，有扎实的学术基础。

### 3. GEPA 优化器：反馈驱动的 Prompt 进化

Metric 不只返回分数，还返回文本反馈（"答案不匹配""推理太长"）。GEPA 据此反思并突变最弱模块，维护多个候选的 Pareto 前沿，周期性地跨候选合并优秀模块。**这是真正的"AI 优化 AI"。**

### 4. 类型安全的 LLM 输出

通过 JSONAdapter + Pydantic 类型签名，强制 LLM 输出结构化 JSON。不再需要正则解析、不再害怕输出格式崩溃。

### 5. 与 PyTorch 的设计类比

| PyTorch | DSPy |
|---|---|
| `nn.Module` | `dspy.Module` |
| 前向传播 `forward()` | 前向传播 `forward()` |
| 损失函数 | Metric 函数 |
| `optimizer.step()` | `optimizer.compile()` |
| 梯度下降 | 指令/示例搜索 |

这个类比非常精准。DSPy 把 prompt engineering 变成了一种"编译"过程。

## 对楠楠/学长的启发

### 🎯 核心启发：Prompt 不是资产，代码才是

DSPy 的哲学彻底颠覆了"精心调教 prompt"的思路。prompt 是编译产物，不是源码。源码是 Signature + Module + Metric。这对 OpenClaw 的 Skill 系统设计有直接参考价值：

1. **Skill 的稳定性**：OpenClaw 的 SKILL.md 本质上就是 prompt。如果用 DSPy 的思路，可以把 Skill 定义成结构化的 Signature + Metric，然后用 optimizer 自动生成最优的 SKILL.md。

2. **自动优化循环**：楠楠已有的 `self-improving` skill 是手动记录错误和改进。DSPy 的 optimizer 提供了一个程序化的替代方案——用 metric + trainset 自动搜索最优策略。

3. **模型无关性**：DSPy 一行代码切模型，pipeline 不变。OpenClaw 也有多模型路由（MiniMax / 智谱），可以借鉴这种"逻辑与模型解耦"的设计。

4. **评估驱动开发**：DSPy 强制你写 metric，用数据说话。楠楠在调 Skill 质量时可以引入类似的评估机制，不再靠"感觉还行"。

### 💡 可落地的想法

- 给 OpenClaw Skill 加一层"Signature"抽象，定义输入输出类型和成功标准
- 用 MIPROv2 思路自动优化 Skill 的 instruction 文本
- 建立一个小型 eval set，量化 Skill 质量改进
