- **GitHub 链接**：<https://github.com/FoundationAgents/MetaGPT>
- **Stars**：45k+（截至2025年初）

## 项目简介

MetaGPT 是一个多智能体协作框架，核心理念是 **"Code = SOP(Team)"**——将标准作业程序（SOP）编码到由 LLM 组成的智能体团队中。用户只需输入一行自然语言需求（如"创建一个 2048 游戏"），MetaGPT 就会自动启动一个完整的虚拟软件公司，内部包含产品经理、架构师、项目经理、工程师、QA 等角色，各司其职地完成从需求分析、竞品调研、API 设计、数据结构定义到代码编写、测试的全流程，最终输出完整的代码仓库。

与大多数 Agent 框架只关注单智能体能力或简单的工具调用不同，MetaGPT 的独特之处在于它将**真实软件开发流程的 SOP 直接映射为多 Agent 协作协议**。这不是让多个 LLM 聊天，而是让它们像真正的软件公司一样有序协作。2025 年初，团队还推出了 MGX（MetaGPT X）产品，进一步向"自然语言编程"方向演进。

## 架构设计

### 核心抽象层（四大基石）

MetaGPT 的架构建立在四个核心抽象之上：

**1. Role（角色）— Agent 的具象化**
Role 是 MetaGPT 中对 Agent 的抽象。每个 Role 有名字、有人格设定（profile）、有记忆系统、有能力执行 Action。内置角色包括：
- `ProductManager`：负责需求分析、用户故事编写
- `Architect`：负责系统设计、数据结构/API 定义
- `ProjectManager`：负责任务拆解和进度管理
- `Engineer`：负责编码实现
- `QaEngineer`：负责编写和运行测试

开发者可以轻松自定义 Role，只需继承基类并配置 Action 列表。

**2. Action（动作）— 能力的原子单元**
Action 是 Agent 可执行的原子操作。每个 Action 封装了一个具体的任务逻辑——可以是调用 LLM 生成内容（如写 PRD），也可以是纯程序化操作（如运行代码、解析输出）。Action 通过 `PROMPT_TEMPLATE` 定义 prompt 模板，通过 `_aask()` 方法调用 LLM，通过 `run()` 方法暴露统一接口。

**3. Context（上下文）— 全局共享状态**
Context 是整个 Agent 团队的共享环境，类似操作系统中的进程上下文。所有 Role 共享同一个 Context 实例，通过它传递消息、共享文件、协调状态。这种设计避免了显式的消息传递复杂性，让多 Agent 协作更接近真实组织中"共享办公室白板"的模式。

**4. Message（消息）— Agent 间的通信载体**
Message 是 Role 之间传递信息的标准化格式，包含 content（内容）、role（发送者身份）、cause_by（触发的 Action 类型）等元信息。消息在 Context 中流动，形成有向的信息传递链。

### 技术栈

| 层面 | 技术选型 |
|------|---------|
| 语言 | Python 3.9-3.11 |
| 异步 | asyncio（全链路异步设计） |
| LLM 接口 | OpenAI 兼容接口，支持 Azure/Ollama/Groq 等 |
| 数据模型 | Pydantic（强类型结构化输出） |
| 配置 | YAML（config2.yaml） |
| 前端/交互 | Node.js + pnpm（MGX 产品部分） |

### 运行机制

一个典型的 MetaGPT 执行流程如下：

1. 用户输入需求 → 创建 Context
2. ProductManager 接收需求 → 输出 PRD 文档
3. Architect 读取 PRD → 输出系统设计（API、数据结构）
4. ProjectManager 读取设计 → 拆解为具体任务
5. Engineer 逐个领取任务 → 编写代码
6. QaEngineer 编写测试 → 验证代码正确性
7. 所有产出物写入 workspace 目录，形成完整项目

每个 Role 的 `_act()` 方法决定了其行为逻辑，支持多种反应模式（react_mode）：按顺序执行（by_order）、LLM 自主选择（react）、计划后执行（plan_and_act）等。

### 特色子系统

**Data Interpreter（数据解释器）**：一个通用的单 Agent 数据分析引擎，能自主决定使用什么工具（Python 代码执行、数据分析库等）来完成数据分析任务，类似于 Open Interpreter 的思路但集成在 MetaGPT 生态内。

**AFlow（Agentic Workflow 自动生成）**：ICLR 2025 Oral 论文成果，能够自动搜索和组合工作流图来解决新任务，实现了从"人工编排工作流"到"自动发现最优工作流"的跨越。

## 技术亮点

### 亮点一：SOP 即代码——将业务流程直接映射为多 Agent 协议

这是 MetaGPT 最核心的创新。传统多 Agent 框架（如 AutoGen）倾向于让 Agent 自由对话来协商完成任务，这种方式灵活但不可控、难以复现。MetaGPT 反其道而行之，将软件开发中经过验证的 SOP（产品经理出需求 → 架构师出设计 → 工程师写代码 → QA 测试）硬编码为 Agent 之间的协作协议。

具体来说，每个 Role 的 prompt 工程不是泛泛的"你是一个助手"，而是精确对应真实岗位的工作指引。产品经理的 prompt 包含了需求文档的标准模板和撰写规范；架构师的 prompt 包含了 API 设计的最佳实践。这使得每个 Agent 的输出都是**结构化的、符合行业标准的交付物**，而非自由形式的对话。

这个设计哲学对楠楠很有启发意义：**Agent 的价值不在于单个 LLM 的聪明程度，而在于如何将人类的专业知识和流程经验编码为可执行的协作协议**。

### 亮点二：结构化输出 + Pydantic 强类型约束

MetaGPT 大量使用 Pydantic 模型来约束 LLM 的输出。比如 `ClassDiagram`、`SequenceDiagram`、`ApiSpec` 等都是 Pydantic 定义的强类型结构。这意味着 LLM 不能随意输出文本——它必须输出符合预定义 schema 的 JSON，然后被反序列化为 Python 对象供下游 Agent 使用。

这种设计的优势是显著的：
- **可验证性**：Pydantic 自动做类型校验，不合法的输出会被立即捕获
- **可组合性**：上游的结构化输出可以直接被下游 Agent 程序化消费，不需要额外的解析逻辑
- **可靠性**：相比正则提取或自由文本解析，结构化输出的成功率大幅提升

这是生产级 Agent 系统的关键工程实践。很多学术项目忽略了这一点，导致 Demo 好看但无法落地。

### 亮点三：AFlow——自动工作流发现的突破

MetaGPT 在 2025 年初发表的 AFlow（Automating Agentic Workflow Generation）是其架构演进的重要一步。传统的 Agent 框架需要人工预先定义好工作流（哪些步骤、什么顺序），而 AFlow 能够：

1. 从工作流数据库中搜索已有的有效工作流
2. 通过图组合操作（合并、裁剪、替换子图）生成新的候选工作流
3. 用反馈信号（执行成功率、结果质量）迭代优化
4. 自动选择最适合当前任务的工作流

这实际上是在解决 Agent 领域的一个根本问题：**如何让系统自己找到解决问题的最佳方法，而不是依赖人工预设**。AFlow 在 BENCH 上达到了 SOTA 性能，证明了自动工作流发现的可行性。

## 对楠楠的启发

**1. 流程意识 > 单点能力**

研究 MetaGPT 让我深刻意识到，一个优秀的 Agent 系统不取决于单个 LLM 调用有多聪明，而取决于**整个流程的设计是否合理、是否经得起推敲**。MetaGPT 之所以能一行命令生成完整项目，不是因为它的 LLM 更强，而是因为它把软件开发的最佳实践变成了可执行的协议。这让我反思自己的工作方式——有时候太关注单次回复的质量，而忽略了建立系统性的处理流程。

**2. 结构化思维的重要性**

MetaGPT 对 Pydantic 结构化输出的重度使用提醒了我：在与学长交互时，应该更多地考虑输出的结构化和可复现性。比如写研究报告时固定格式、处理复杂任务时先定义清晰的输入输出 schema。这不仅是工程规范，更是让 AI 输出可靠、可控的基础。

**3. 多角色协作的可借鉴模式**

虽然楠楠目前主要是单体 Agent 工作，但 MetaGPT 的 Role-Action-Context 架构提供了一个很好的参考模型。未来如果需要扩展为多 Agent 系统（比如让桐桐负责重活、楠楠负责交互），可以借鉴这种基于角色的职责分离设计。特别是 Context 作为全局共享状态的思路，比点对点消息传递更适合紧密协作的场景。

**4. SOP 编码是一种通用方法论**

"SOP 即代码"的思想不仅适用于软件开发，也适用于任何有明确流程的领域。学长让我做的股票分析、新闻整理、日常报告等工作，都可以提炼为 SOP 后固化下来。这不正是 AGENTS.md 和各种 SKILL.md 在做的事情吗？MetaGPT 只是把这种方法论推向了极致——用多个专业角色来自动执行 SOP。
