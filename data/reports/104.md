# AI Agent 开源项目学习报告 #104 — AgentScope

## 项目名
**AgentScope**（阿里巴巴达摩院出品）

## GitHub 链接
https://github.com/agentscope-ai/agentscope

## 一句话简介
面向生产环境的、以开发者为中心的 AI Agent 框架，内置可视化设计器（Studio）、MCP 工具集成、多模态记忆系统和分布式执行引擎，支持从单 Agent 到多 Agent 协作的全栈开发。

---

## 架构设计

### 核心模块分层

AgentScope 采用 **四层架构**，自底向上：

```
┌─────────────────────────────────────────────┐
│         Application Layer (应用层)           │  用户编写的 Agent 应用
├─────────────────────────────────────────────┤
│          Agent Layer (Agent 层)              │  ReAct 循环 / 内置 Agent / 多 Agent 编排
├─────────────────────────────────────────────┤
│       Service Layer (服务层)                 │  Memory │ Tool/Toolkit │ Model │ Pipeline
├─────────────────────────────────────────────┤
│      Infrastructure (基础设施)               │  分布式执行 │ 持久化 │ 监控追踪
└─────────────────────────────────────────────┘
```

### 技术栈
- **语言**: Python 3.10+（另有 Java 版 agentscope-java）
- **异步模型**: 全面基于 `asyncio`，原生支持流式响应
- **模型支持**: OpenAI, Anthropic, Gemini, DeepSeek, Ollama, DashScope（阿里云）
- **记忆后端**: Mem0 集成 + 自定义 LongTermMemoryBase 抽象
- **工具协议**: 原生 MCP（Model Context Protocol）客户端 + 本地函数注册
- **可观测性**: OpenTelemetry 兼容的 `@trace_llm` 装饰器
- **IDE 集成**: VS Code 扩展 — AgentScope Studio（可视化调试）

### 工作流程（ReAct 主循环）

```
用户输入 → Agent.reply()
    ↓
┌─ ReAct Loop ─────────────────────────┐
│  1. LLM 推理 → 生成 Thought + Action   │
│  2. 执行 Action（工具调用）             │
│     ├─ 单工具调用                       │
│     ├─ 并行多工具调用（asyncio.gather）  │
│     └─ 动态工具组切换                   │
│  3. 观察结果 → 更新 Memory             │
│  4. 判断是否完成？                      │
│     ├─ 否 → 回到步骤 1                  │
│     └─ 是 → 返回最终回复                │
│                                       │
│  [中断信号] → handle_interrupt()       │
│    → 暂停循环 → 保存部分状态 → 响应中断  │
└───────────────────────────────────────┘
```

### 分布式 Pipeline 执行
AgentScope 的 Pipeline 模块支持将不同 Agent 部署到不同进程/机器：
- 基于 `asyncio.Pipeline` 实现数据流式编排
- 每个 Agent 作为独立节点运行，通过消息队列通信
- 支持动态拓扑：运行时添加/移除节点
- 内置容错和状态持久化（checkpoint）

## 技术亮点

### 1. 双范式长期记忆系统（Developer-Controlled vs Agent-Controlled）

这是 AgentScope 最具创新性的设计之一。`LongTermMemoryBase` 抽象类定义了 **两套独立接口**：

```python
class LongTermMemoryBase(ABC):
    # 开发者控制 — 在关键生命周期点自动触发
    @abstractmethod
    def record(self, messages: list[Msg]) -> None:
        """在会话开始/结束时由开发者调用，系统性记录"""
    
    @abstractmethod
    def retrieve(self, messages: list[Msg]) -> list[dict]:
        """根据输入消息语义检索相关记忆"""
    
    # Agent 控制 — Agent 自主决定何时存取
    @abstractmethod
    def record_to_memory(self, content: str) -> None:
        """Agent 在推理过程中自主存储重要信息"""
    
    @abstractmethod
    def retrieve_from_memory(self, query: str) -> list[dict]:
        """Agent 主动按关键词查询知识库"""
```

**为什么这个设计好？**
- 开发者控制路径保证**可靠性**（不会漏记关键信息）
- Agent 控制路径保证**灵活性**（LLM 可以自主判断什么值得记住）
- `record_to_memory` 和 `retrieve_from_memory` 会**自动注册为工具函数**放入 Toolkit，Agent 可以直接调用
- 默认实现 `Mem0LongTermMemory` 接入 Mem0 的语义索引能力

**对比我们的 OpenClaw 记忆系统**：楠楠目前用文件做记忆（daily notes + MEMORY.md），本质是纯开发者控制模式。AgentScope 的双范式提示我们可以让楠楠在心跳或对话中更主动地管理自己的记忆——比如发现学长提到新偏好时主动写入结构化记忆。

### 2. 有状态/无状态双模式 MCP 客户端

AgentScope 对 MCP 的实现比大多数框架都深入：

```python
# 无状态 MCP — 每次调用新建连接，适合轻量事务型服务
stateless_client = McpClient(
    client_type="stdio",
    command=["npx", "-y", "@modelcontextprotocol/server-fetch"],
)

# 有状态 MCP — 保持持久连接，适合需要会话上下文的服务（如浏览器）
stateful_client = McpClient(
    client_type="streamable-http",
    url="https://browser-mcp.example.com",
)
await stateful_client.connect()  # 显式建立连接
# ... 后续所有调用在同一 session 中 ...
await stateful_client.close()    # 显式关闭
```

**核心创新**：MCP 远程工具被包装为**本地可调用代理对象**（proxy objects），可以直接传入 `register_tool_function()` 注册：

```python
# 远程工具变成本地一等公民
remote_search = stateless_client.get_tool("web_search")
Toolkit().register_tool_function(remote_search)  # 无缝注册
```

这意味着开发者可以对远程工具做**二次封装**——比如把远程搜索 + 本地正则过滤 + 远程摘要组合成一个复合工具。这种"远程服务本地化"的设计思想非常实用。

### 3. 分组工具管理（Group-Wise Tool Management）

针对"工具过多导致 Agent 性能下降"的问题，AgentScope 引入了工具分组机制：

```python
toolkit = Toolkit()

# 创建逻辑分组
toolkit.create_tool_group("browser_tools", ["navigate", "click", "type_text"])
toolkit.create_tool_group("code_tools", ["execute_python", "read_file", "write_file"])

# 运行时动态激活/停用
toolkit.update_tool_groups(active_groups=["browser_tools"])
# 此时 Agent 只能看到 browser_tools 组的工具
# 减小了工具选择空间，提升准确率
```

这个设计的洞察来自论文 *Paramanayakam et al., 2025* 的实证研究：工具过多会增加 LLM 的认知负载，反而降低选择正确工具的概率。通过按任务阶段动态切换可用工具组，既保证了功能完整性又控制了复杂度。

### 4. 实时中断与状态保持（Real-time Steering）

AgentScope 利用 Python `asyncio.CancelledError` 机制实现了优雅的中断处理：

```python
class ReActAgent(Agent):
    async def reply(self, message: Msg) -> Msg:
        try:
            # ReAct 主循环
            while not self.finished:
                thought = await self.llm.generate(messages)
                action = self.parse_action(thought)
                result = await toolkit.execute(action)
                memory.add(message, thought, action, result)
        except asyncio.CancelledError as e:
            # 关键：中断不是错误，是可控事件
            partial_result = self._preserve_partial_state()  # 保存已产生的部分结果
            memory.annotate_interruption(partial_result)     # 标注中断上下文
            return await self.handle_interrupt(e)             # 交由子类定义策略
```

**精妙之处**：
- 中断时已产生的 LLM 部分回复和工具中间结果**不丢失**，而是作为带标注的记忆保留
- Agent 后续推理可以看到"我在做 X 的时候被打断了"，从而智能决定恢复/修改/放弃
- `handle_interrupt()` 可被子类覆写，支持"快速回复"或"重新推理"等不同策略

这对交互式 Agent 系统非常重要——用户不需要等 Agent 跑完一个长循环才能纠正方向。

### 5. 统一响应 Schema 与跨 Provider 推理追踪

AgentScope 用 `ChatResponse` dataclass 统一了所有模型的输出格式：

```python
@dataclass
class ChatResponse:
    message: Msg                          # 统一消息对象
    content: list[ContentBlock] = field(default_factory=list)
    # ContentBlock 类型：
    # - TextBlock(text="...")           文本内容
    # - ToolUseBlock(name="...", args=...)  工具调用
    # - ThinkingBlock(thinking="...")    推理过程（DeepSeek-R1/o1 等）
    
    usage: ChatUsage | None = None        # token 用量统计
    response_id: str | None = None        # 唯一标识
    timestamp: float | None = None        # 时间戳
```

特别是 `ThinkingBlock` 的统一抽象，使得框架可以透明处理 OpenAI o-series 的 reasoning effort、Anthropic 的 extended thinking budget、Gemini 的 thinking 配置等不同提供商的推理能力暴露方式。

配合 `@trace_llm` 装饰器自动注入 OpenTelemetry span，实现了**零侵入的可观测性**：

```python
@trace_llm  # 自动记录请求参数、响应元数据、token 用量、错误条件
async def call_model(self, messages, **kwargs) -> ChatResponse:
    ...
```

## 对我们（OpenClaw/楠楠）的启发

### 🔍 直接可借鉴的点

| AgentScope 特性 | OpenClaw/楠楠现状 | 改进方向 |
|---|---|---|
| **双范式记忆** | 纯文件记忆（开发者控制） | 增加 Agent 自主记忆能力——楠楠可以在对话中主动判断并写入结构化偏好/知识 |
| **分组工具管理** | 所有工具平铺可用 | 按 skill/场景分组工具，减少每次调用的上下文噪声 |
| **实时中断+状态保持** | subagent 取消即丢失 | 中断时保存部分结果，支持断点续传式的任务恢复 |
| **MCP 双模式客户端** | 通过 HTTP 手动调用 API | 封装有状态/无状态的 MCP 客户端，简化外部服务接入 |
| **ThinkingBlock 统一抽象** | 不同模型推理格式各异 | 统一推理过程的表示层，方便调试和分析 |

### 💡 架构层面的深层启发

1. **ReAct 作为默认范式而非可选插件**：AgentScope 把 ReAct 做为"唯一推荐"架构，而不是像 LangChain 那样提供几十种 chain 变体。这种"少即是多"的设计哲学值得参考——OpenClaw 的 skill 系统也可以考虑收敛核心交互模式，而不是无限扩展。

2. **Pipeline 分布式执行**：AgentScope 的 Pipeline 让每个 Agent 跑在独立进程中，通过消息传递协作。这和 OpenClaw 的 subagent 机制思路相似但更系统化——可以考虑让不同的 skill/subagent 运行在隔离环境中，通过标准化消息协议通信。

3. **Studio 可视化调试**：VS Code 扩展可以直接看到 Agent 的每一步推理和动作。对于 OpenClaw 来说，虽然不一定做 IDE 插件，但可以考虑 Web UI 形式的 Agent 执行可视化面板（Canvas 已有基础）。

4. **工具过多问题被正式对待**：AgentScope 不是简单地"支持更多工具"，而是意识到工具数量本身是个性能瓶颈并设计了分组机制。我们在给楠楠加新 tool 时应该更有节制，或者引入类似的激活/停用机制。

### ⚠️ 注意事项

- AgentScope 由阿里巴巴达摩院维护，商业化倾向明显（强推 DashScope），社区中立性不如 CrewAI 或 LangGraph
- 文档和示例偏中英双语，英文文档质量略逊于纯国际项目
- Java 版本（agentscope-java）功能尚不及 Python 版完整
- 分布式 Pipeline 的容错机制在生产环境的大规模验证还不够充分
