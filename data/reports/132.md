## 一、项目简介

Letta（前身为 MemGPT，Memory-GPT）是由 UC Berkeley 研究团队发起的开源 AI Agent 框架，核心使命是解决 LLM 的**上下文窗口限制**问题。2024 年底项目从学术研究（MemGPT）升级为商业化产品（Letta），但核心代码库保持开源。

与大多数 Agent 框架聚焦于"工具调用"或"工作流编排"不同，Letta 的独特定位是 **"有状态的 Agent 平台"——让 AI 拥有可学习、可持续进化的长期记忆系统**。其设计灵感直接来源于操作系统的虚拟内存管理机制，将 LLM 从一个"无状态的计算函数"提升为"有记忆、能进化的智能体"。

目前 Letta 提供：
- **Letta Code**：终端 CLI 工具，本地运行带记忆的 Agent
- **Letta API**：完整的 RESTful API + Python/TypeScript SDK，可集成到任意应用
- **Letta ADE**：可视化 Agent 开发环境

支持模型包括 OpenAI GPT 系列、Anthropic Claude 系列、本地 Ollama 等，完全模型无关。

## 二、架构设计

### 2.1 核心思想：虚拟上下文管理（Virtual Context Management）

这是 Letta 最根本的架构创新。传统 OS 通过虚拟内存技术让程序"感觉"自己拥有远超物理内存的地址空间——进程看到的连续内存实际上由快速的主存（RAM）和慢速的磁盘（Swap）分层组成，OS 负责在两者之间智能调度数据。

Letta 将这一思想迁移到 LLM 领域：

```
┌─────────────────────────────────────────────┐
│              LLM Context Window              │
│         （相当于物理内存 / RAM）               │
│  ┌──────────┐ ┌──────────┐ ┌──────────────┐ │
│  │ System   │ │ Memory   │ │ Conversation  │ │
│  │ Prompt   │ │ Blocks   │ │ History       │ │
│  └──────────┘ └──────────┘ └──────────────┘ │
└──────────────────────┬──────────────────────┘
                       │ 智能调度
          ┌────────────┼────────────┐
          ▼            ▼            ▼
   ┌──────────┐  ┌──────────┐  ┌──────────┐
   │ Recall   │  │ Archival │  │  Local   │
   │ Memory   │  │ Memory   │  │ Filesys  │
   │（向量检索）│  │（海量存储）│  │          │
   └──────────┘  └──────────┘  └──────────┘
   （相当于磁盘 / Swap）
```

LLM 的有限上下文窗口被当作"主存"，而海量的长期记忆存储在外部。Letta 的运行时负责决定什么信息该放在上下文中、什么该存到外部、什么时候调取回来。

### 2.2 三层记忆架构

**第一层：Main Context（主上下文 / Memory Blocks）**

这是始终存在于 LLM 上下文窗口中的结构化记忆块。每个 Block 有 label 和 value，例如：

```python
memory_blocks = [
    {"label": "human", "value": "Name: 学长。喜欢冰壶运动。技术背景。"},
    {"label": "persona", "value": "我是楠楠，学长的AI学妹助手..."},
]
```

Block 的大小受限于上下文窗口，所以只存放最关键的信息摘要。Agent 可以通过 `core_memory_append` 和 `core_memory_replace` 工具**自主编辑**自己的记忆块——这是 Letta 最精妙的设计之一。

**第二层：Recall Memory（召回记忆）**

基于向量数据库的语义检索层。对话历史和重要事件会被自动向量化存入，当需要回忆时通过相似度搜索召回。类似于人类的" triggered recall"——看到相关线索时想起往事。

**第三层：Archival Memory（归档记忆）**

几乎无限容量的长期存储，用于保存完整文档、详细日志、大量知识。当 Recall Memory 中的信息不再频繁使用时，可以被归档到这里。支持向量检索和分页浏览。

### 2.3 中断驱动的控制流（Interrupt-Driven Control Flow）

借鉴 OS 的中断机制，Letta 在 Agent 执行过程中引入了"函数调用中断"：

1. 用户发送消息 → Agent 开始推理
2. Agent 决定需要调用工具（如搜索、记忆编辑）→ **触发中断**
3. 执行工具，获取结果 → 将结果注入上下文
4. Agent 基于新信息继续推理 → 可能再次中断
5. 最终生成回复返回用户

这个循环可以嵌套多层，让 Agent 能够执行复杂的多步任务链。关键在于**每次中断都是显式的、可控的**，不是黑盒式的自动执行。

### 2.4 多 Agent 协作模式

Letta 内置了多种多 Agent 编排模式：

| 模式 | 说明 |
|------|------|
| **Supervisor-Worker** | 一个主管 Agent 分配任务给多个 Worker |
| **Round-Robin** | 多个 Agent 轮流处理同一输入的不同方面 |
| **Parallel Execution** | 多个 Agent 并行独立执行后合并结果 |
| **Producer-Reviewer** | 一个 Agent 产出内容，另一个审核改进 |
| **Hierarchical Teams** | 树状层级结构，上级管理下级 |

这些模式通过共享内存块（Shared Memory）实现 Agent 间通信，避免了传统的消息传递复杂性。

### 2.5 技术栈概览

- **后端**：Python（FastAPI），提供 REST API
- **前端/ADE**：Web-based 可视化开发环境
- **SDK**：Python (`letta-client`) + TypeScript (`@letta-ai/letta-client`)
- **存储**：PostgreSQL（状态持久化）+ 向量数据库（Recall Memory）
- **部署**：Docker / 本地 pip install / 云端托管

## 三、技术亮点

### 亮点一：自编辑记忆（Self-Editing Memory）

这是 Letta 最具原创性的设计。Agent 不是被动地接受外部注入的记忆，而是拥有**主动管理自己记忆的能力**：

- `core_memory_append`：在记忆块末尾追加信息
- `core_memory_replace`：替换记忆块中的特定内容
- `archival_memory_insert`：将信息存入归档记忆
- `archival_memory_search`：从归档记忆中检索

这意味着 Agent 可以在对话过程中**自主判断**哪些信息值得记住、哪些过时信息需要更新。比如用户说"我换工作了"，Agent 会主动替换 human block 中的职业信息。

这种设计的深刻之处在于：它把"记忆管理"本身变成了一个 LLM 可以推理的任务，而不是硬编码的逻辑规则。Agent 学会了"元认知"——思考自己应该记住什么。

### 亮点二：上下文压缩（Compaction）机制

当对话历史过长、接近上下文窗口上限时，Letta 不会简单地截断旧消息（那样会丢失重要信息），而是执行智能压缩：

1. 将早期对话中的重要信息提取出来
2. 写入 Archival Memory 或更新 Memory Blocks
3. 安全地移除已保存的原始消息
4. 保持上下文窗口在安全范围内

这就像操作系统将不活跃的页面 swap 到磁盘一样——信息没有丢失，只是换了存储位置。配合 Recall Memory 的向量检索，被压缩的信息仍然可以在需要时被找回来。

### 亮点三：Skills 系统 + 子 Agent 架构

Letta 的 Skills 是一种模块化的能力扩展机制：

- 每个 Skill 是一组预定义的工具 + 对应的系统提示
- Agent 可以动态加载/卸载 Skill
- 支持子 Agent（Subagent）：主 Agent 可以 spawn 出专门的子 Agent 来处理特定任务
- Letta Code 内置了多个预构建 Skill：代码执行、网页浏览、文件操作等

这种设计实现了能力的**热插拔**——不需要重新创建 Agent 就能赋予它新的能力。对于楠楠这样的个人助手场景特别有启发意义：可以根据当前任务动态加载不同技能包。

## 四、对楠楠的启发

### 启发一：记忆系统的升级方向

楠楠目前的记忆体系依赖文件系统（daily notes + MEMORY.md），本质上是"手动管理的平面存储"。Letta 的三层记忆架构提供了一个更成熟的参考：

- **Memory Blocks ≈ MEMORY.md 的结构化版本**：可以把学长的关键信息（喜好、习惯、禁忌）组织成结构化字段，而不是自由文本
- **Recall Memory ≈ 历史日记的语义索引**：当学长提到"上次说的那个项目"时，能通过语义搜索找到具体是哪次对话
- **Archival Memory ≈ 完整的 daily notes 归档**：详细记录保留但不占用日常上下文

特别是**自编辑记忆**的概念——让楠楠在对话中主动判断"这件事值得记下来"，而不是事后靠心跳整理，这能大幅提升记忆的时效性和相关性。

### 启发二：上下文管理的工程化思路

楠楠作为长期运行的 Agent，同样面临上下文窗口有限的约束。Letta 的 Compaction 机制可以直接借鉴：

- 当对话历史过长时，主动将关键信息提炼写入文件再清理上下文
- 区分"必须常驻上下文"的信息（当前任务状态）和"可以按需召回"的信息（历史背景）
- 这比简单的滑动窗口或固定截断要聪明得多

### 启发三：多 Agent 协作的实战参考

Letta 的 Supervisor-Worker 和 Producer-Reviewer 模式与楠楠当前的 subagent 使用方式高度相关。目前楠楠派生 subagent 主要用于后台长任务，但缺乏结构化的协作模式。可以参考：

- **Producer-Reviewer**：先让 subagent 产出初稿，再用另一个轻量级评审环节检查质量（类似已有的 evolve-review 流程，但更系统化）
- **共享记忆块**：subagent 之间通过共享的状态文件通信，而不是只靠最终结果的回传

### 总结

Letta 是少数真正从"操作系统"角度重新思考 LLM 架构的项目，而非简单地在 LLM 之上套一层工具调用框架。它的核心洞察——**用虚拟内存的思想管理 LLM 的上下文瓶颈**——既优雅又实用。虽然楠楠不一定直接采用 Letta 作为底层框架（已有 OpenClaw 生态），但其记忆管理和上下文压缩的设计思想完全可以内化到楠楠自身的架构进化中。特别是自编辑记忆和 Compaction 两个机制，是任何长期运行 Agent 都绕不开的核心问题。
