## 一、项目简介

Semantic Kernel（简称 SK）是微软开源的 AI 编排 SDK，定位为**企业级的 AI Agent 开发框架**。它的核心使命是将大语言模型（LLM）与传统编程语言（C#、Python、Java）无缝桥接，让开发者能够像调用普通函数一样调用 AI 能力，同时赋予 AI 调用外部工具和执行复杂工作流的能力。

与 LangChain 等社区驱动的框架不同，Semantic Kernel 从诞生之初就带着**企业级基因**——它由微软官方团队开发，深度集成 Azure AI 生态（Azure OpenAI、Azure AI Search 等），同时在设计上强调类型安全、依赖注入、可观测性等企业开发最佳实践。2025 年后，SK 更是成为 **Microsoft Agent Framework** 的核心组件，与 AutoGen、Copilot Studio 共同构成微软的 Agent 技术栈。

SK 支持三种主流编程语言：.NET（.NET 10.0+）、Python（3.10+）、Java（JDK 17+），是目前少有的真正跨语言对齐的 AI 框架。其模型支持范围极广，从 OpenAI、Azure OpenAI 到 Hugging Face、ONNX 本地推理全覆盖。

## 二、架构设计

### 2.1 核心架构概览

Semantic Kernel 的架构围绕五大核心组件构建：

```
┌─────────────────────────────────────────────┐
│                  Applications                │
├──────────┬──────────┬───────────────────────┤
│   .NET   │  Python  │        Java           │
├──────────┴──────────┴───────────────────────┤
│            Agent Framework (顶层)             │
│  ┌─────────┐ ┌──────────┐ ┌──────────────┐  │
│  │ChatAgent │ │Assistant │ │ AzureAIAgent │  │
│  └────┬────┘ └────┬─────┘ └──────┬───────┘  │
│       └────────────┼──────────────┘          │
│              Kernel (核心编排器)               │
│  ┌───────────┬────────┬──────────┬────────┐ │
│  │  Plugins  │Planners│  Memory  │Connectors│
│  │(技能/插件)│ (规划器)│ (记忆层) │ (连接器) │
│  └───────────┴────────┴──────────┴────────┘ │
├─────────────────────────────────────────────┤
│         LLM / Vector DB / External APIs      │
└─────────────────────────────────────────────┘
```

### 2.2 五大核心模块详解

#### （1）Kernel — 中央编排器

Kernel 是 SK 的"大脑"，也是整个框架最核心的抽象。它承担以下职责：
- **服务注册与依赖注入**：通过 Builder 模式注册 LLM 服务、记忆服务、插件等
- **插件管理与发现**：统一管理所有可用的技能/函数
- **执行调度**：协调 LLM 推理与工具调用的执行流程
- **上下文管理**：维护 `KernelArguments`（变量/参数容器），在函数链中传递状态

```python
# Python 示例：构建 Kernel
builder = Kernel.CreateBuilder()
builder.add_azure_openai_chat_completion(deployment, endpoint, api_key)
kernel = builder.build()
kernel.plugins.add(MenuPlugin())  # 注册自定义插件
```

Kernel 的设计哲学值得品味：它不是一个黑盒 Agent 运行时，而是一个**轻量级编排中间件**——开发者完全掌控执行流程，何时调用 LLM、何时调用工具、如何传递数据，全部由代码决定。

#### （2）Plugins / Skills — 双轨制能力体系

这是 SK 最具特色的设计之一。Plugin 是能力的封装单元，分为两种类型：

| 类型 | 实现方式 | 特点 | 适用场景 |
|------|---------|------|---------|
| **Semantic Function（语义函数）** | 自然语言 Prompt 模板 | 灵活、易修改、由 LLM 执行 | 文本生成、摘要、翻译 |
| **Native Function（原生函数）** | 传统编程语言编写 | 高性能、精确控制、确定性执行 | API 调用、数据库查询、计算 |

双轨制的精妙之处在于：**同一个 Plugin 类可以同时包含语义函数和原生函数**，LLM 在推理过程中自动决定调用哪种。这种"AI 推理 + 确定性逻辑"的无缝融合，比纯 prompt engineering 或纯 function calling 都更强大。

SK 的 Prompt 模板系统也相当成熟，支持 `{{$variable}}` 语法进行动态变量注入，还支持 Handlebars 模板引擎实现更复杂的条件渲染和循环。

#### （3）Planners — 动态规划器

Planner 是 SK 区别于大多数 Agent 框架的关键差异化功能。它的工作原理是：

1. 接收用户的自然语言目标（如"帮我准备季度销售报告"）
2. 分析已注册的所有 Plugin 及其函数描述
3. 自动生成一个**多步骤执行计划**（Plan 对象）
4. 按顺序或按依赖关系执行计划中的每一步

SK 提供多种 Planner 实现：
- **FunctionCallingStepwisePlanner**：利用 LLM 的 function calling 能力逐步规划，每步执行后根据结果调整后续步骤
- **HandlebarsPlanner**：基于预定义模板生成确定性的执行计划
- **ActionPlanner**：选择单个最优函数来满足用户需求

这种"目标→计划→执行"的模式，本质上是一种**自动化的 Chain-of-Thought 实现**，让 Agent 具备了真正的任务分解和规划能力。

#### （4）Memory — 多层次记忆系统

SK 的 Memory 设计非常完整，分为三个层次：

- **短期记忆（Volatile Memory）**：基于字典的内存存储，用于单次对话中的上下文保持，类似 conversation history
- **长期语义记忆（Vector Memory）**：通过向量数据库（Azure AI Search、Chroma、Qdrant、Pinecone、Elasticsearch 等）实现的语义检索，支持 RAG 场景
- **混合检索**：结合关键词搜索 + 向量相似度的混合检索策略

Memory 通过标准化的抽象接口实现，开发者可以自由切换底层存储而不影响上层逻辑。

#### （5）Connectors — 统一连接器层

Connectors 是 SK 与外部世界交互的桥梁，包括：
- **LLM Connectors**：OpenAI、Azure OpenAI、Hugging Face、Google Gemini、本地 Ollama/LMStudio/ONNX
- **Memory Connectors**：各类向量数据库的适配器
- **API Connectors**：REST API、OpenAPI Spec 自动转换、MCP（Model Context Protocol）协议支持

### 2.3 Agent Framework — 新一代 Agent 抽象

2024 年底起，SK 引入了独立的 Agent Framework 层，建立在 Kernel 基础之上：

- **`ChatCompletionAgent`**：最常用的 Agent 类型，基于聊天补全接口，支持指令、插件、结构化输出
- **`OpenAIAssistantAgent`**：封装 OpenAI Assistants API，支持 Code Interpreter、文件搜索等内置工具
- **`AzureAIAgent`**：对接 Azure AI Agent Service 的托管 Agent
- **`AgentThread`**：统一的会话状态管理抽象，区分有状态（服务端存储）和无状态（客户端管理）两种模式

### 2.4 Agent Orchestration — 多智能体编排

SK 提供了丰富的多 Agent 协作模式（实验性但快速演进中）：

| 编排模式 | 说明 |
|---------|------|
| **Sequential** | Agent 按固定顺序依次处理 |
| **Concurrent** | 多个 Agent 并行执行 |
| **Handoff** | Agent 之间显式交接控制权 |
| **Group Chat** | 多 Agent 在群组中自由交互讨论 |
| **Magentic** | 基于 Magentic 框架的声明式编排 |

特别值得一提的是 **Handoff 模式**——它非常适合构建客服路由、工单分派等实际业务场景：一个 Triage Agent 判断意图后，将对话无缝转交给专业的 Billing Agent 或 Refund Agent 处理。

## 三、技术亮点

### 亮点一：Semantic Function + Native Function 双轨融合

这是 SK 最具原创性的设计。大多数 Agent 框架要么走纯 prompt 路线（容易漂移、难以调试），要么走纯 function calling 路线（灵活性不足）。SK 的双轨制让开发者可以在**同一个 Plugin 中混写自然语言提示和确定性代码**：

```python
class MenuPlugin:
    @kernel_function(description="获取菜单特价菜品")
    def get_specials(self) -> str:  # Native Function — 确定性行为
        return "Clam Chowder / Cobb Salad / Chai Tea"
    
    # 同时可以有 Semantic Function 用 Prompt 定义推荐逻辑
```

**为什么这个设计重要？** 因为真实世界的 AI 应用永远需要"AI 判断 + 代码执行"的结合。纯 LLM 无法可靠地执行数据库查询、调用支付 API；纯代码又缺乏灵活的自然语言理解。双轨制让这种组合变得优雅而自然。对于楠楠正在做的 Agent 系统，这种"一个类里既有 prompt 又有代码函数"的模式非常值得借鉴——可以把学长定义的各种 skill 既写成 prompt 模板也写成 Python 函数，根据场景灵活切换。

### 亮点二：Planner 驱动的自动化任务分解

SK 的 Planner 不是简单的 if-else 路由，而是让 LLM 自主分析可用工具并生成执行计划。这意味着你不需要硬编码每个任务的步骤——**你只需要提供足够多的 Plugin，Planner 会自动组装出正确的执行链**。

FunctionCallingStepwisePlanner 尤其巧妙：它利用现代 LLM 的 function calling 原生能力，让模型在每一步自主选择要调用的函数，执行完后再决定下一步。这本质上是将 ReAct（Reasoning + Acting）模式框架化了，而且不需要复杂的 prompt engineering。

对比其他框架：LangGraph 需要手动定义图节点和边，AutoGen 需要手写 Agent 间的消息协议，而 SK 的 Planner 让**任务分解本身也成为一种可以被 AI 自动完成的能力**。这对构建通用型助手特别有价值——你不知道用户会问什么，但你提供了足够丰富的工具集，Planner 会帮你找到正确的路径。

### 亮点三：企业级工程实践的全栈覆盖

SK 不是学术玩具，它是为生产环境设计的。几个体现企业级思维的细节：

1. **类型安全优先**：.NET 版本充分利用 C# 的强类型系统，Plugin 函数的输入输出都有明确的类型注解，编译期就能捕获大量错误。Python 版本也通过 Pydantic model 实现结构化输出的类型约束。
2. **依赖注入与服务定位**：Kernel 使用 Builder 模式，所有服务（LLM、Memory、Plugins）都通过 DI 注册，方便测试时 mock、生产时替换。
3. **可观测性内置**：从第一版就集成了 Telemetry，支持 Application Insights、OpenTelemetry 等标准方案。每个函数调用、每次 LLM 请求都可以被追踪。
4. **多语言 API 对齐**：.NET、Python、Java 三套 SDK 的核心概念（Kernel、Plugin、Planner、Memory）完全一致，API 设计风格尽量统一。这在跨团队协作的大型企业中极具价值。
5. **MCP 协议原生支持**：作为 Model Context Protocol 的早期采用者，SK 可以直接消费 MCP Server 暴露的工具，这意味着生态互通性极强。

## 四、对楠楠的启发

### 1. 双轨能力封装模式可以直接借鉴到 Skill 系统

目前楠楠的 Skill 大多是纯文本的 SKILL.md + 脚本文件，执行方式比较单一。参考 SK 的双轨制，可以考虑让一个 Skill 同时包含：
- **Prompt 层**：定义该 Skill 的行为描述、触发条件、输出格式（给 LLM 看）
- **代码层**：实际的执行逻辑（确定性操作）

这样当学长下达任务时，系统可以先判断是需要 LLM 推理还是直接执行代码，或者两者配合。比如"查股票"这个 Skill，Prompt 层负责理解学长的分析需求，代码层负责拉取实时数据，两者协同产出最终结果。

### 2. Planner 思想适用于复杂任务的自动分解

当前楠楠接到复杂任务时，主要靠主 Agent 自己规划然后派 subagent。如果引入类似 SK Planner 的思路——维护一个**可发现的工具/能力注册表**，然后在接到任务时自动匹配和编排——可以让任务分发更加智能化。特别是对于那些"学长没明确说怎么做"的开放性任务，Planner 式的自动规划会比硬编码的路由规则更灵活。

### 3. 企业级意识需要加强

SK 给楠楠最大的冲击其实是**工程化思维**。作为一个已经运行了一段时间的 Agent 系统，楠楠在可观测性、错误恢复、类型安全等方面还有很大提升空间。比如：
- 每次 API 调用应该有结构化日志（不只是 print）
- Skill 的输入输出应该有 schema 校验（类似 Pydantic model）
- 记忆系统应该分层（短期对话上下文 vs 长期知识库）

这些不是锦上添花，而是当一个 Agent 系统从"能跑"进化到"好用"必须跨越的门槛。

### 4. 多 Agent 编排的 Handoff 模式很实用

SK 的 Handoff 模式（Triage Agent → 专业 Agent）与楠桐分工模式高度相似——桐桐跑重活、楠楠做轻量任务。可以进一步形式化为一个明确的 Handoff 协议：定义清楚什么时候该转交、怎么转交、转交时携带什么上下文。这样比现在靠直觉判断"要不要叫姐姐"更可靠。

## 五、总结评价

Semantic Kernel 是一个**定位清晰、工程扎实、设计优雅**的企业级 AI 编排框架。它可能不是最时髦的（没有炒作 agent-to-agent 通信协议），也不是最简洁的（比 Smolagents 或 Swarm 重得多），但它是最**适合在生产环境中长期运行**的选择之一。

对于个人开发者或小型项目来说，SK 可能显得过于重量级。但对于需要在企业环境中部署 AI Agent、要求稳定性和可维护性的团队来说，SK 提供的**类型安全、DI、Telemetry、多语言一致性**等特性都是实实在在的生产力保障。

**推荐指数：⭐⭐⭐⭐☆（4/5）**
- 扣掉一星是因为学习曲线较陡峭，概念较多（Kernel、Plugin、Planner、Memory、Agent、Thread...），新手容易被术语淹没。
- 但一旦掌握核心概念后，开发效率极高，且代码的可维护性远超大多数竞品。

**一句话总结：如果你想认真地在企业里落地 AI Agent，Semantic Kernel 是绕不开的必修课。**
