## 一、项目背景与问题定义

### 1.1 从 Claude Code 到 Claude Agent SDK

2025年初，Anthropic 发布了 **Claude Code**——一个命令行 AI 编程助手，能够自主阅读代码库、编辑文件、运行命令、提交代码。Claude Code 迅速成为开发者社区的热门工具。

但 Claude Code 最初只是一个 CLI 工具——你只能在终端里使用它。开发者很快意识到：**Claude Code 内部的 Agent 循环（Agent Loop）和工具集是一个通用的、强大的 Agent 运行时**，不应该被限制在 CLI 里。

**2025年9月29日**，Anthropic 发布了 **Claude Agent SDK**，将 Claude Code 的完整能力封装为可编程的 Python/TypeScript SDK。这不是一个全新的框架——它是 Claude Code 运行时的"产品化"。

### 1.2 核心命题："Give Your Agent a Computer"

> **与其他所有 Agent 框架的根本差异：Claude Agent SDK 不是"给 Agent 工具"，而是"给 Agent 一台完整的计算机"。**

传统 Agent 框架的工具模式：
```
用户请求 → LLM → 选择工具(函数调用) → 应用执行 → 结果返回 → LLM → ...
```

Claude Agent SDK 的计算机模式：
```
用户请求 → LLM → 直接操作文件系统(Bash/Read/Write/Edit) → LLM → 直接搜索代码(Glob/Grep) → LLM → ...
```

**区别不是语义上的，而是架构上的。** Claude 不需要通过 API 调用去"读取文件"——它直接有 Read 工具。不需要通过函数调用去"执行命令"——它直接有 Bash 工具。这种设计使 Claude Agent 能够像人类程序员一样与计算机交互：打开文件、修改代码、运行测试、查看结果、再修改。

### 1.3 为什么这很重要

**通用性突破**：大多数 Agent 框架是为特定场景设计的（对话机器人、客服系统、数据分析）。Claude Agent SDK 面向的是**任何需要在计算机上完成多步骤任务的场景**——从代码开发到系统管理到数据处理。

**上下文工程优势**：当 Agent 拥有完整的文件系统和 Shell 访问时，它可以自行决定如何组织信息、如何存储中间结果、如何调试问题。这比预定义的工具集更具表达力。

**生产验证**：这个 Agent 循环已经 powering Claude Code 本身——每天处理数百万次真实的编程任务。它不是实验室原型，而是经过大规模生产验证的系统。

### 2.2 两种使用模式

#### 模式一：`query()` — 无状态一次性查询

```python
from claude_agent_sdk import query, ClaudeAgentOptions, AssistantMessage, TextBlock

options = ClaudeAgentOptions(
    system_prompt="你是一个专业的研究助手",
    max_turns=10,
)

async for message in query(prompt="分析当前目录下的代码质量", options=options):
    if isinstance(message, AssistantMessage):
        for block in message.content:
            if isinstance(block, TextBlock):
                print(block.text)
```

**特点**：
- 最简 API，一行启动
- 自动获得所有内置工具（Bash/Read/Edit/Grep 等）
- 适合脚本化任务、批量处理
- 无需管理会话状态

#### 模式二：`ClaudeSDKClient` — 有状态交互式会话

```python
from claude_agent_sdk import (
    ClaudeSDKClient, ClaudeAgentOptions,
    tool, create_sdk_mcp_server
)

# 定义自定义工具
@tool("check_policy", "检查费用是否合规", {"amount": float, "category": str})
async def check_policy(args):
    limit = 75 if args["category"] == "meals" else 500
    ok = args["amount"] <= limit
    return {"content": [{"type": "text", "text": f"{'✅' if ok else '❌'}"}]}

# 创建 in-process MCP 服务器
server = create_sdk_mcp_server(
    name="expense-tools",
    version="1.0.0",
    tools=[check_policy]
)

options = ClaudeAgentOptions(
    mcp_servers={"expense": server},
    allowed_tools=["Read", "Bash", "mcp__expense__check_policy"],
    permission_mode="acceptEdits",
    cwd="/path/to/project"
)

async with ClaudeSDKClient(options=options) as client:
    # 第一次查询
    await client.query("审核费用报告: 午餐 $47")
    async for msg in client.receive_response():
        print(msg)
    
    # 第二次查询（共享上下文！）
    await client.query("那机票 $650 呢？")
    async for msg in client.receive_response():
        print(msg)
```

**特点**：
- 支持自定义工具（in-process MCP）
- 支持 Hooks（拦截钩子）
- 多轮对话自动维护上下文
- 支持 Subagent（并行子代理）

### 2.3 核心原语详解

#### 原语一：内置工具集（Built-in Tools）

这是 Claude Agent SDK 与其他框架最大的差异化优势：

| 工具名 | 能力 | 类比 |
|--------|------|------|
| **Bash** | 执行任意 Shell 命令 | 人类的手在键盘上 |
| **Read** | 读取文件内容 | 人类的眼睛看屏幕 |
| **Write** | 创建/覆写文件 | 人类的手写文档 |
| **Edit** | 精确编辑文件（基于 diff） | 人类的精确修改 |
| **Glob** | 文件名模式匹配搜索 | 人类的 `find` 命令 |
| **Grep** | 文件内容正则搜索 | 人类的 `grep` 命令 |
| **WebFetch** | 抓取网页内容 | 人类的浏览器 |
| **WebSearch** | 网络搜索 | 人类的 Google |

**关键洞察**：这些工具的组合覆盖了人类在计算机上做几乎所有事情的原语操作。Read+Edit 覆盖了编程；Bash 覆盖了系统管理；Glob+Grep 覆盖了信息检索；WebFetch+WebSearch 覆盖了网络调研。

#### 原语二：自定义工具（In-Process MCP）

```python
from claude_agent_sdk import tool, create_sdk_mcp_server

@tool("tool_name", "工具描述", {"param1": str, "param2": int})
async def my_tool(args):
    # 处理逻辑
    result = do_something(args["param1"], args["param2"])
    return {
        "content": [
            {"type": "text", "text": f"结果: {result}"}
        ]
    }

server = create_sdk_mcp_server(
    name="my-server",
    version="1.0.0",
    tools=[my_tool]
)
```

**设计亮点 —— In-Process MCP**：
- 传统 MCP 需要**独立的子进程**（stdio 通信）或**网络服务**（SSE 通信）
- SDK MCP 在**同一进程内**运行——零 IPC 开销
- 工具就是普通的 Python async 函数——可以直接访问应用的状态、数据库连接等
- 无需序列化/反序列化复杂对象

**这意味着什么？** 你可以把任何已有的 Python 函数暴露为 Agent 工具，无需重构、无需网络层、无需进程管理。

#### 原语三：Hooks（拦截钩子）

Hooks 是 Agent Loop 中确定性的拦截点，让你在不修改 Agent 行为逻辑的前提下注入控制逻辑：

```python
from claude_agent_sdk import HookMatcher

async def block_dangerous_commands(input_data, tool_use_id, context):
    tool_name = input_data["tool_name"]
    tool_input = input_data.get("tool_input", {})
    
    if tool_name == "Bash":
        cmd = tool_input.get("command", "")
        dangerous = ["rm -rf /", "sudo", "> /dev/sda"]
        for pattern in dangerous:
            if pattern in cmd:
                return {
                    "hookSpecificOutput": {
                        "hookEventName": "PreToolUse",
                        "permissionDecision": "deny",
                        "permissionDecisionReason": f"危险命令: {pattern}",
                    }
                }
    return {}

options = ClaudeAgentOptions(
    hooks={
        "PreToolUse": [
            HookMatcher(matcher="Bash", hooks=[block_dangerous_commands]),
        ],
    }
)
```

**可用的 Hook 点**：

| Hook 点 | 触发时机 | 典型用途 |
|---------|---------|---------|
| **UserPromptSubmit** | 用户消息发送前 | 输入过滤/审查 |
| **PreToolUse** | 工具执行前 | 权限控制/命令审计 |
| **PostToolUse** | 工具执行后 | 日志记录/结果过滤 |
| **Stop** | Agent 完成后 | 输出校验/格式修正 |

#### 原语四：Subagents（并行子代理）

```python
from claude_agent_sdk import AgentDefinition

researcher = AgentDefinition(
    name="researcher",
    instructions="深入调研给定主题",
    # 继承主 agent 的部分工具
    allowed_tools=["Read", "WebFetch", "Grep"],
)

writer = AgentDefinition(
    name="writer",
    instructions="基于研究结果撰写报告",
    allowed_tools=["Read", "Write", "Edit"],
)

options = ClaudeAgentOptions(
    subagents=[researcher, writer],
    # 主 agent 可以把任务委派给 subagent
    allowed_tools=["Agent"],  # Agent 工具用于调用 subagent
)
```

**Subagent vs Handoff（vs OpenAI Agents SDK）**：
- OpenAI 用 Handoff 做 Agent 间**控制权转移**
- Claude 用 Subagent 做**任务委派**——主 Agent 始终保持控制权
- Subagent 可以**并行执行**多个
- 每个 Subagent 有独立的工具白名单和指令

#### 原语五：权限系统（Permission System）

```
权限评估顺序:
allowed_tools (白名单) 
    → 工具是否在列表中？
        → 是: permission_mode 决定是否自动批准
        → 否: canUseTool 回调决定
            → disallowed_tools (黑名单): 直接拒绝
            → 其他: 根据 permission_mode 弹出确认或自动批准
```

**permission_mode 选项**：

| 模式 | 行为 |
|------|------|
| `plan` | 只规划不执行（安全审计模式） |
| `default` | 写操作需确认，读操作自动批准 |
| `acceptEdits` | 文件编辑自动批准，Bash 仍需确认 |
| `bypassPermissions` | 全部自动批准（完全自治） |

**成本控制**：`max_budget_usd` 参数限制单次会话的最大花费，超限自动终止。

### 2.4 Agent Loop 工作原理

```
                    ┌──────────────────────┐
                    │   用户发送 Prompt     │
                    └──────────┬───────────┘
                               ▼
                    ┌──────────────────────┐
                    │  UserPromptSubmit Hook│ ← 可拦截/阻止
                    └──────────┬───────────┘
                               ▼
              ┌────────────────────────────────┐
              │   构建 API Request              │
              │  • System Prompt                │
              │  • 对话历史（含缓存）            │
              │  • Tool Definitions             │
              └──────────────┬─────────────────┘
                             ▼
              ┌────────────────────────────────┐
              │   调用 Claude API               │
              │  (Claude Sonnet / Opus)        │
              └──────────────┬─────────────────┘
                             ▼
              ┌────────────────────────────────┐
              │   解析 Response                 │
              │  • TextBlock → 累积输出          │
              │  • ToolUseBlock → 执行工具       │
              └──────────────┬─────────────────┘
                             ▼
                   ┌──────────────────┐
                   │  有 Tool Use？     │
                   └────┬─────────┬───┘
                       是│         │否
                        ▼         ▼
              ┌──────────────┐  ┌──────────────┐
              │ PreToolUse   │  │  Stop Hook   │
              │ Hook         │  │  + 返回结果   │
              └──────┬───────┘  └──────┬───────┘
                     ▼                 ▼
              ┌──────────────┐  ┌──────────────┐
              │ 执行工具      │  │ResultMessage │
              │ PostToolUse  │  │ .result      │
              │ Hook         │  │ .usage       │
              └──────┬───────┘  │ .total_cost  │
                     │          │ .session_id  │
                     ▼          └──────────────┘
              ┌──────────────┐
              │ 结果送回 API  │
              │ (继续循环)    │
              └──────┬───────┘
                     ▼
              (直到 max_turns 或 Agent 说完成)
```

**关键特性：Prompt Caching（提示缓存）**

System Prompt 和 Tool Schema 在首次调用后**自动缓存**。后续相同前缀的请求只需支付约 **10%** 的输入 token 成本。对于多轮工具调用的长会话，这可以节省大量成本。

## 三、技术亮点深度解析

### 亮点1："Computer Use"范式 —— 重新定义 Agent 与环境的交互

**原理**：传统 Agent 框架将"能力"抽象为离散的工具函数（`search_web(query)`, `read_file(path)`, `run_command(cmd)`）。每个工具都是一个黑盒——Agent 不知道工具内部发生了什么，只能看到输入输出。

Claude Agent SDK 反其道而行之：**不给 Agent 抽象工具，给它原语操作**。Bash 不是"执行命令的工具"——它是 Agent 直接操控计算机的方式。Read 不是"读取文件的API"——它是 Agent 看世界的方式。

**为什么这更好？**

1. **组合爆炸的表达力**：Bash + Read + Write 三个原语的组合可以表达几乎任何计算机操作。预定义 100 个专用工具也不如这三个原语灵活。
2. **LLM 的 Shell 知识可直接复用**：Claude 在训练数据中见过海量 Shell 命令、文件操作、Git 工作流。当它拥有 Bash 工具时，这些知识直接可用——无需在 tool description 里教它怎么用。
3. **调试透明度**：Agent 执行的每一步都是人类可读的 Shell 命令或文件操作。你可以精确看到它在做什么、为什么这么做。

**风险**：强大的原语也意味着更大的攻击面。Bash 可以做任何事情——包括破坏性操作。这就是为什么权限系统如此重要。

### 亮点2：In-Process MCP —— 零开销的自定义工具

**原理**：Model Context Protocol (MCP) 是 Agent 工具的标准协议。传统实现方式是 stdio（子进程）或 SSE（HTTP 服务）。两者都有开销：进程管理、序列化、网络延迟。

SDK MCP 将 MCP server **内嵌到你的 Python 进程中**：

```
传统 MCP 调用链:
你的代码 → 序列化(JSON) → IPC/HTTP → MCP Server 进程 → 反序列化 → 执行 → 序列化 → IPC/HTTP → 你的代码

SDK MCP 调用链:
你的代码 → 直接函数调用 → 执行 → 返回值
```

**影响**：
- **性能**：消除 IPC 开销，工具调用延迟降低 1-2 个数量级
- **简单性**：工具就是普通 Python 函数，可以用 debugger 单步调试
- **状态共享**：工具可以直接访问应用的内存中状态（数据库连接、缓存、配置）
- **部署**：单进程部署，无需管理额外的服务

### 亮点3：事件驱动的流式响应

**原理**：Claude SDK 的所有交互都基于异步事件流：

```python
async with ClaudeSDKClient(options) as client:
    await client.query("分析这个代码库")
    async for msg in client.receive_response():
        if isinstance(msg, SystemMessage):
            print(f"[系统] 会话ID: {msg.session_id}")
        elif isinstance(msg, AssistantMessage):
            for block in msg.content:
                if isinstance(block, TextBlock):
                    print(block.text, end="", flush=True)  # 流式输出
                elif isinstance(block, ToolUseBlock):
                    print(f"\n[工具调用] {block.name}({block.input})")
        elif isinstance(msg, UserMessage):
            for block in msg.content:
                if isinstance(block, ToolResultBlock):
                    print(f"[工具结果] {block.content[:100]}")
        elif isinstance(msg, ResultMessage):
            print(f"\n[完成] 成本: ${msg.total_cost_usd:.4f}")
```

**事件类型**：

| 事件类型 | 包含内容 | 触发时机 |
|----------|---------|---------|
| `SystemMessage` | 会话元数据 | 会话开始 |
| `AssistantMessage` | TextBlock / ToolUseBlock | Claude 每次响应 |
| `UserMessage` | ToolResultBlock | 工具执行结果 |
| `ResultMessage` | 最终结果/用量/成本 | 会话结束 |

**价值**：你可以构建实时进度条、中间结果展示、成本监控面板——一切都在 Agent 运行时实时可见。

### 亮点4：Context Compaction（上下文压缩）

**原理**：长运行的 Agent 会话会积累大量对话历史，最终超出上下文窗口。Claude SDK 自动处理这个问题：

- 当对话历史接近上下文窗口限制时，自动触发**摘要压缩**
- 早期对话被压缩为结构化摘要，保留关键决策和结论
- 近期对话保持原始细节
- Agent 可以感知到压缩发生，并在需要时主动重新读取文件恢复上下文

**对比其他方案**：
- OpenAI Agents SDK：依赖 Responses API 的服务端 Store 模式（数据存于 OpenAI 服务器）
- LangGraph：需要手动实现 state summarization
- Claude SDK：**本地磁盘持久化 + 自动压缩**，隐私友好

### 亮点5：多模型后端支持

虽然名为"Claude" Agent SDK，但它并非只能用 Claude 模型：

| 后端 | 配置方式 | 说明 |
|------|---------|------|
| **Anthropic API** | 默认 | Claude Sonnet/Opus/Haiku 全系列 |
| **AWS Bedrock** | 环境变量/配置 | 企业级托管 |
| **Google Vertex AI** | 环境变量/配置 | GCP 集成 |
| **Microsoft Azure AI Foundry** | 环境变量/配置 | Azure 集成 |

这使得企业可以在现有云基础设施上部署，无需直接与 Anthropic 建立商业关系。

## 四、基准测试与性能表现

### 4.1 三大 Agent SDK 横向对比

| 维度 | Claude Agent SDK | OpenAI Agents SDK | Google ADK |
|------|------------------|-------------------|------------|
| **核心哲学** | 给 Agent 一台计算机 | 三个原语（Agent/Handoff/Guardrail） | 工程化原则（版本/测试/模块化） |
| **OS/系统访问** | ✅ **原生支持**（Bash/文件系统） | ❌ 无（依赖外部工具） | ❌ 无（依赖外部连接器） |
| **内置工具** | ✅ 8+ 个（Read/Write/Edit/Bash/Glob/Grep/Web...） | ⚠️ 函数调用 + 内置搜索/文件检索 | ⚠️ 通过连接器 |
| **自定义工具** | ✅ In-Process MCP（零IPC） | ✅ @function_tool（类型推导） | ✅ 连接器/工具注册 |
| **多 Agent 编排** | ✅ Subagents（并行/嵌套） | ✅ Handoff + as_tools | ✅ Sequential/Parallel/Loop + Graph(2.0) |
| **权限系统** | ✅ **分层精细**（allowed/disallowed + permission_mode + Hooks） | ✅ Guardrails（I/O 并行） | ✅ Cloud API Registry |
| **状态管理** | ✅ 内存/磁盘双模式 + 自动压缩 | ✅ Sessions（客户端/服务端） | ✅ 高级（故障恢复/回放） |
| **语音/实时** | ❌ 需外部 API | ✅ **内置 TTS + Realtime** | ✅ 双向音视频 |
| **可观测性** | ✅ OTel 原生 + 用量/成本 API | ✅ Logfire/AgentOps/OTel | ✅ ADK Web UI + Cloud Monitoring |
| **语言支持** | Python, TypeScript | Python, TypeScript | **Python, TS, Java, Go** |
| **模型锁定** | ⚠️ Claude 优先（但支持多云） | ⚠️ OpenAI 优先（但支持100+LLM） | ⚠️ Gemini 优先（但模型无关） |
| **Prompt Caching** | ✅ **自动**（~90%成本节省） | ✅（Responses API 缓存） | ✅ |
| **学习曲线** | 中等（需理解 Agent Loop 概念） | **极低**（3个原语） | 中高（企业级概念多） |
| **Hello World** | ~10行 | **~5行** | ~50-100行 |
| **最佳场景** | 编程 Agent / OS 自动化 / 开发者工具 | 对话 Agent / 语音 Agent / 快速原型 | 企业级系统 / Google Cloud / 多语言团队 |
| **GitHub Stars** | 快速增长（2025.9发布） | 19K+ | 快速增长 |

### 4.3 已知局限与问题

| 问题 | 影响 | 缓解方案 |
|------|------|---------|
| **模型锁定倾向** | 设计围绕 Claude 优化，非 Claude 模型体验下降 | 通过 Bedrock/Vertex/Azure 使用 Claude |
| **无内置语音** | 不能像 OAI SDK 那样开箱做语音 Agent | 组合外部 TTS/STT 服务 |
| **Subagent 工具受限** | Subagent 只能用主 Agent 工具集的子集，无法添加独有工具 | 等待官方改进（Issue #314） |

| **成本** | Claude Opus 级别的 Agent 循环成本不低 | 用 Sonnet 做大部分工作，Opus 做关键决策；利用 Prompt Caching |
| **权限复杂性** | permission_mode + allowed_tools + disallowed_tools + Hooks 的组合可能令人困惑 | 从 default 模式开始，按需逐步放开 |
| **无 GUI 操作** | 虽然叫"Computer Use"但不能操控图形界面（点击按钮等） | 需要结合 Browser Use 等工具 |

## 五、代码示例：完整的多 Agent 系统

### 5.1 开源贡献 Agent（Fork + Fix + PR）

这是一个真实的使用案例——Agent 自主完成开源贡献的全流程：

```python
import asyncio
from claude_agent_sdk import (
    ClaudeSDKClient, ClaudeAgentOptions,
    tool, create_sdk_mcp_server, AgentDefinition
)

# === 自定义工具：GitHub 操作 ===

@tool(
    "fetch_issues",
    "获取仓库的 open issues 列表",
    {"repo_owner": str, "repo_name": str, "labels": list}
)
async def fetch_issues(args):
    """调用 GitHub API 获取 issues"""
    import httpx
    resp = httpx.get(
        f"https://api.github.com/repos/{args['repo_owner']}/{args['repo_name']}/issues",
        params={"state": "open", "per_page": 10}
    )
    issues = [
        {"number": i["number"], "title": i["title"], "labels": [l["name"] for l in i["labels"]]}
        for i in resp.json()
    ]
    return {"content": [{"type": "text", "text": str(issues)}]}

@tool(
    "create_pull_request",
    "创建 Pull Request",
    {"repo": str, "branch": str, "title": str, "body": str}
)
async def create_pr(args):
    """调用 GitHub API 创建 PR"""
    # ... GitHub API 调用实现
    return {"content": [{"type": "text", "text": f"PR created: {args['title']}"}]}

# === 创建工具服务器 ===
github_server = create_sdk_mcp_server(
    name="github-tools",
    version="1.0.0",
    tools=[fetch_issues, create_pr_request]
)

# === Subagent 定义 ===
researcher = AgentDefinition(
    name="researcher",
    instructions="""你是一个开源研究者。
    1. 读取仓库的 CONTRIBUTING.md 了解贡献规范
    2. 分析 issue 的代码位置和根因
    3. 输出修复方案""",
    allowed_tools=["Read", "Grep", "Glob", "WebFetch", "mcp__github-tools__fetch_issues"]
)

coder = AgentDefinition(
    name="coder",
    instructions="""你是一个代码修复专家。
    1. 基于研究者的方案编写修复代码
    2. 确保风格与仓库一致
    3. 运行测试验证""",
    allowed_tools=["Read", "Write", "Edit", "Bash", "Grep"]
)

# === 主 Agent 配置 ===
options = ClaudeAgentOptions(
    system_prompt="""
    你是一个开源贡献 Agent。目标：为指定的 GitHub 仓库找到合适的 issue，
    完成代码修复并提交 PR。
    
    工作流程：
    1. 用 researcher 分析 issue
    2. 用 coder 编写和测试修复
    3. 创建 PR
    """,
    mcp_servers={"github": github_server},
    subagents=[researcher, coder],
    allowed_tools=[
        "Read", "Write", "Edit", "Bash", "Glob", "Grep",
        "Agent",  # 用于调用 subagent
        "mcp__github-tools__fetch_issues",
        "mcp__github-tools__create_pull_request",
    ],
    permission_mode="acceptEdits",
    cwd="/tmp/contribution-workspace",
    max_turns=50,
    max_budget_usd=5.00,
)

# === 运行 ===
async def main():
    async with ClaudeSDKClient(options=options) as client:
        await client.query("为 anthropics/claude-agent-sdk-python 找一个 good-first-issue 并修复")
        async for msg in client.receive_response():
            if isinstance(msg, AssistantMessage):
                for block in msg.content:
                    if hasattr(block, 'text'):
                        print(block.text, end="", flush=True)
            elif isinstance(msg, ResultMessage):
                print(f"\n\n总成本: ${msg.total_cost_usd:.4f}")
                print(f"Token 用量: {msg.usage}")

asyncio.run(main())
```

## 六、同类项目横向对比

### 6.1 与 OpenAI Agents SDK 的根本哲学差异

这是两个最值得对比的框架，因为它们代表了两种截然不同的 Agent 设计哲学：

| 维度 | OpenAI Agents SDK | Claude Agent SDK |
|------|-------------------|-----------------|
| **世界观** | Agent 是**对话者**——通过工具了解世界 | Agent 是**计算机使用者**——直接操作环境 |
| **工具观** | 工具是**能力扩展**——越多越好，标准化 | 工具是**原语操作**——少而强大，组合表达一切 |
| **安全观** | Guardrails 作为**声明式策略**——并行执行 | Permission + Hooks 作为**分层过滤器**——确定性拦截 |
| **编排观** | Handoff（转移控制权） | Subagent（委派任务，保持控制） |
| **状态观** | 服务端管理（Store）或客户端传递 | 本地磁盘持久化 + 自动压缩 |
| **适用场景画像** | 客服/对话/语音/内容生成 | 编程/运维/数据分析/任何"坐电脑前做的事" |

**类比**：
- OpenAI Agents SDK 像 **iOS**——精心设计的 App 生态系统，每个 App（工具）做一件事，做得很好
- Claude Agent SDK 像 **Linux Terminal**——给你原语（命令），你自己组合，能做任何事

### 6.2 与 Browser Use 的关系

Browser Use 让 Agent 操控浏览器（点击/输入/滚动），Claude Agent SDK 让 Agent 操控终端（Bash/文件）。它们是互补的：

- **需要操控 Web UI**（没有 API 的网站）→ Browser Use
- **需要操控本地环境**（代码/文件/系统）→ Claude Agent SDK
- **两者结合**：Claude Agent SDK 启动浏览器 + Browser Use 操控页面

### 6.3 适用场景判断

```
你应该选 Claude Agent SDK，如果：
├── 你的 Agent 需要读写文件、执行命令、操作代码
├── 你在构建开发者工具、编程助手、DevOps 自动化
├── 你希望 Agent 像人类一样使用计算机（而非通过受限 API）
├── 你需要细粒度的权限控制和审计
├── 你重视 prompt caching 带来的成本节省
└── "Give the agent a computer" 这个理念 resonates with you

你应该选 OpenAI Agents SDK，如果：
├── 主要构建对话/语音/内容生成类 Agent
├── 需要最低的学习成本和最快的 Hello World
├── 需要内置的 TTS 和 Realtime 语音能力
├── 团队已有 OpenAI 生态经验
└── 需要在 100+ LLM provider 之间自由切换

你应该选 Google ADK，如果：
├── 企业级部署在 Google Cloud 上
├── 需要多语言支持（Java/Go）
├── 需要复杂的图级别工作流编排
└── 需要高级状态管理（故障恢复/回放）
```

## 七、对我们的实际启发

### 7.1 技术层面

1. **"Computer Use"范式对 OpenClaw 有直接参考价值**：我们的 subagent 体系本质上也是"给 Agent 一个环境"——文件系统、Shell、工具调用。Claude SDK 的内置工具集设计（Read/Write/Edit/Bash/Glob/Grep）可以作为我们工具设计的标杆。特别是 Edit 工具（基于 diff 的精确编辑）比全量 Write 更安全高效

2. **In-Process MCP 思想值得借鉴**：当前我们的自定义工具通过 HTTP/API 调用。如果高频工具能在同一进程内执行，延迟和复杂度都会显著降低。考虑为常用工具提供 in-process 扩展点

3. **Hook 系统的设计模式**：PreToolUse/PostToolUse/UserPromptSubmit/Stop 这四个拦截点的抽象非常清晰。我们在消息发送、工具调用等环节可以引入类似的钩子系统，让安全检查和日志记录变成可插拔的

4. **事件流式响应**：当前我们的 subagent 结果是"跑完再返回"。如果改为事件流模式，学长可以实时看到 Agent 在做什么——这对长时间运行的任务体验提升巨大

### 7.2 架构层面

5. **权限分层思想**：`allowed_tools`（白名单）→ `permission_mode`（行为模式）→ `canUseTool`（动态回调）→ `disallowed_tools`（黑名单）的四层权限模型，比简单的 allow/deny 更适合生产环境。我们的 skill/cron/subagent 权限体系可以参考这个分层设计

6. **Context Compaction 的必要性**：随着楠楠的记忆文件越来越多，上下文窗口压力会增大。Claude 的自动摘要压缩思路——近期保持原始细节、早期压缩为摘要——可以直接应用于我们的记忆管理系统

7. **Subagent 的工具继承模型**：Claude 的 Subagent 从主 Agent 继承工具子集，且有独立白名单。这比"所有 subagent 共享全部工具"更安全。我们的 subagent 体系目前没有工具级别的隔离，这是一个可以改进的方向

### 7.3 战略层面

8. **三大厂商 SDK 三足鼎立格局形成**：OpenAI（对话/语音）、Claude（计算机操作）、Google（企业/多云）。每个厂商都在用自己的模型优势定义 Agent 范式。理解这三个 SDK 的设计哲学，比掌握任何一个框架的语法更有长期价值

9. **Agent 框架正在从"抽象竞赛"转向"原语竞争"**：早期的 LangGraph/CrewAI 比拼谁能提供更多抽象（图/角色/任务）。现在的趋势是反过来的——谁的原语更少、更强大、更通用。Claude SDK 的"Bash+Read+Edit"三个工具胜过 100 个专用工具，就是这个趋势的体现

10. **"Agent 即开发者"是不可逆的趋势**：Claude Code/Cursor/Windsurf 等编程 Agent 的爆发说明，让 AI 像开发者一样操作计算机是最自然的 Agent 形态。OpenClaw 作为一个 Agent 平台，应该在这个方向上持续投入
