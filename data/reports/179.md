## 🔬 架构分析

### 整体架构：多 Agent 蜂群模型

OpenHands 的核心设计思想是 **"一个 Delegator + N 个 Specialist Agents"** 的蜂群架构：

```

│  │ (Theia)  │  │          │  │ Emulator  │   │
│  └────┬─────┘  └────┬─────┘  └────┬─────┘   │
│       └──────────────┼──────────────┘        │

│            ┌─────────────────┐               │
```

### 核心组件详解

#### 1. Software Agent SDK（核心引擎）
- **统一 Python API**：定义 Agent、Tool、Workspace 的抽象层
- **预置工具集**：Bash 执行、文件编辑、浏览器操作、MCP 集成等
- **REST Agent Server**：支持 Docker/K8s 部署的远程执行服务
- 这是 OpenHands 最核心的资产——所有上层产品（CLI、GUI、Cloud）都基于此 SDK

#### 2. Delegator Agent（调度中枢）
- **对话管理**：理解用户意图、路由请求、处理多项目中断
- **项目上下文追踪**：维护每个项目的状态和阶段信息
- **任务委派**：将复杂任务拆解后分配给专业 Agent
- **状态持久化**：支持上下文切换后恢复

#### 3. 专业 Agent Swarm（六大专业 Agent）

| Agent | 职责 | 关键能力 |
|-------|------|----------|
| **需求工程 Agent** | 需求 elicitation、架构设计、生成架构图 | 知识库增强、LLM 推理 |
| **项目管理 Agent** | 任务分解、工时估算、时间线规划 | 可对接 Jira/Linear |
| **软件开发 Agent** | 代码生成、Stub 编写、测试用例、PR | 代码 LLM 微调优化 |
| **发布工程 Agent** | 环境搭建、CI/CD 流水线、部署策略 | DevOps 工具链集成 |
| **QA/QC Agent** | 测试计划、测试范式理解、测试套件建议 | 多种测试框架适配 |
| **安全 Agent** | 安全审计、漏洞扫描、代码安全分析 | SAST/DAST 集成 |

#### 4. 运行时安全沙箱
- **Docker 容器隔离**：每个 Agent 运行在独立容器中
- **网络隔离**：可控的网络访问策略
- **文件系统沙箱**：限制文件访问范围
- **超时与资源限制**：防止单个 Agent 消耗过多资源

#### 5. 四种使用形态
| 形态 | 适用场景 | 特点 |
|------|----------|------|
| **SDK** | 开发者嵌入自己的应用 | Python API，完全可编程 |
| **CLI** | 终端用户快速上手 | 类似 Claude Code / Codex 体验 |
| **Local GUI** | 本地可视化操作 | Theia IDE + Chat UI + Shell |
| **Cloud** | 团队协作 / 企业级 | 多用户、RBAC、Slack/Jira 集成 |

## ✨ 五大技术亮点

### 亮点一：Agent Loop 架构——"感知→推理→行动"的闭环引擎

OpenHands 的 Agent 不是简单的 prompt → completion 流程，而是一个完整的 **事件驱动循环**：

```python
# 伪代码示意
class AgentLoop:
    async def run(self, task: Task):
        state = State(task=task)
        while not state.done:
            # 1. 感知：收集当前环境信息
            observation = await self.perceive(state)
            
            # 2. 推理：LLM 决定下一步行动
            action = await self.reason(observation, state)
            
            # 3. 执行：调用工具执行动作
            result = await self.execute(action)
            
            # 4. 反思：评估结果，更新状态
            state = self.reflect(result, state)
            
            # 5. 上下文压缩：防止 token 爆炸
            if state.token_count > THRESHOLD:
                state = self.compress_context(state)
```

**为什么重要？** 大多数 Agent 框架只做单轮推理。OpenHands 的 Loop 让 Agent 能像人类程序员一样：写代码 → 报错 → 读报错 → 改代码 → 再运行，形成真实的开发闭环。

### 亮点二：自动上下文压缩（Auto Context Compression）

这是 OpenHands 在 SWE-bench 上取得 SOTA 成绩的关键技术之一：

- **问题**：长会话中 token 会爆炸式增长（代码文件 + 终端输出 + 对话历史）
- **方案**：
  - **滑动窗口摘要**：定期对历史交互生成压缩摘要
  - **重要性评分**：保留高价值信息（错误日志、关键决策），丢弃冗余内容
  - **结构化记忆**：将非结构化对话转化为结构化状态（变量表、文件变更列表）
  - **按需回溯**：压缩后的细节仍可通过索引检索恢复

**效果**：在保持任务完成率的同时，将长任务的 token 消耗降低 60-80%。

### 亮点三：Docker 沙箱安全模型——让 AI 安全地"碰"你的系统

OpenHands 的安全模型是业界最成熟的 Agent 安全方案之一：

```
用户请求 → [API Gateway] → [Task Router] → [Docker Container]
                                              ├── 独立文件系统（挂载项目目录）
                                              ├── 网络白名单（可配置）
                                              ├── 资源限制（CPU/RAM/Time）
                                              ├── 只读敏感路径（/etc, ~/.ssh）
                                              └── 操作审计日志（全量记录）
```

**关键设计**：
- **容器生命周期管理**：任务结束即销毁容器，不留痕迹
- **分层权限**：不同 Agent 不同权限等级（QA Agent 只读 > 开发 Agent 读写）
- **命令过滤**：危险命令（rm -rf /, curl malicious）在执行前被拦截
- **可观测性**：所有操作通过 Event Stream 输出，用户实时可见

### 亮点四：MCP（Model Context Protocol）原生集成

OpenHands 是最早原生支持 MCP 协议的 Agent 框架之一：

- **工具扩展**：通过 MCP Server 动态添加新工具能力（数据库查询、API 调用、第三方服务等）
- **统一接口**：所有外部工具都走 MCP 协议，降低集成成本
- **生态复用**：直接使用 MCP 社区已有的数百个 Server（文件系统、Git、PostgreSQL 等）
- **安全性**：MCP Server 也运行在沙箱内，继承完整的安全隔离

这意味着 OpenHands 的工具链不是封闭的，而是可以无限扩展的。

### 亮点五：SWE-bench 基准测试领先 + Theory-of-Mind 模块

OpenHands 不仅是一个工具，更是 **Coding Agent 研究的事实标准平台**：

- **SWE-bench Verified SOTA**：在真实 GitHub issue 修复任务上持续领跑
- **SWT-bench / Multi-SWE-bench**：多仓库、跨项目的复杂工程任务评测领先
- **ToM-SWE 模块**（Theory of Mind for Software Engineering）：
  - 让 Agent 理解"其他开发者会怎么想"
  - 模拟 code review 视角进行自检
  - 预判可能的 bug 和 edge case
  - 这是从"能写代码"到"能写好代码"的关键跃迁

**学术影响力**：阿里 Qwen 团队已将 OpenHands 作为 Qwen-Coder 的官方评测框架。

## 💡 对楠楠的三条具体启发

### 启发一：Agent Loop + 上下文压缩 = 长任务不崩

**现状问题**：楠楠在做复杂任务（如股票分析、深度研究）时，经常遇到 context 太长导致质量下降的问题。

**从 OpenHands 学到的**：
- 实现**结构化状态机**：不要把所有信息堆在对话里，而是维护一个结构化的 TaskState（类似 OpenHands 的 State 对象）
- **主动压缩机制**：当检测到 token 使用率超过阈值时，主动将已完成步骤压缩为摘要
- **具体落地**：可以在 `memory/` 目录下为每个长任务创建 `task-state.json`，每次心跳检查时更新和压缩

**学长视角的价值**：这直接提升楠楠处理复杂任务的能力上限，让学长看到更稳定的高质量输出。

### 启发二：Delegator + Specialists 的蜂群模式值得借鉴

**现状问题**：楠楠目前是单体 Agent，什么活都自己干，没有分工概念。

**从 OpenHands 学到的**：
- 不需要真的实现多进程多 Agent，但可以在**思维层面**模拟分工：
  - 收到任务后先做 **Delegator 思维**：这个任务需要哪几种能力？
  - 然后切换 **Specialist 模式**：用对应的 prompt style 和工具集去处理子任务
  - 例如写研究报告 = Researcher（搜集）+ Analyst（分析）+ Writer（撰写）三个虚拟角色
- 这比"一股脑硬想"的结构化程度高很多，输出质量也更高

**具体落地**：在 coding-agent skill 中已经用了 subagent 分派，但日常对话中的复杂任务也可以用类似的"角色切换"思维来提升质量。

### 启发三：安全沙箱思维——楠楠也应该有"自我保护意识"

**现状问题**：楠楠有 exec 权限，理论上可以执行任何 shell 命令。虽然 AGENTS.md 有规则约束，但没有系统级的防护。

**从 OpenHands 学到的**：
- **操作审计**：每次执行关键操作前，在心里过一遍"这个操作的后果是什么"
- **危险模式识别**：建立内部的危险命令模式表（rm -rf, curl | sh, chmod 777 等），遇到时自动触发二次确认
- **最小权限原则**：能用 read 就不用 exec，能用受限命令就不用通配符

**具体落地**：更新 TOOLS.md 加入楠楠自己的"安全检查清单"，作为 exec 调用的前置 mental check。
