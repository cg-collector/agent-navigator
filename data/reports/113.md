## 架构设计

### 整体架构（六大核心模块）

Browser Use 采用经典的 ReAct（Reason + Act）循环架构，属于「交错分解」范式——边执行边规划，根据环境反馈动态调整下一步动作。整体分为六个模块：

```
┌─────────────────────────────────────────────┐
│                  Agent Core                   │
│         （任务编排 / 状态管理 / 流程控制）       │
│                                              │
│  ┌──────────┐  ┌──────────────┐  ┌────────┐ │
│  │Message   │◄─►│    Memory     │◄─►│   LLM   │ │
│  │Manager   │  │ (mem0压缩)    │  │Interface│ │
│  └────┬─────┘  └──────────────┘  └────┬───┘ │
│       │                                │      │
│  ┌────▼────────────────────────────────▼───┐ │
│  │              Controller                  │ │
│  │        （动作注册表 / 执行引擎）           │ │
│  └────────────────────┬────────────────────┘ │
│                       │                       │
│  ┌────────────────────▼────────────────────┐ │
│  │            BrowserContext                │ │
│  │  （Playwright / DOM提取 / 截图 / 多标签） │ │
│  └─────────────────────────────────────────┘ │
└─────────────────────────────────────────────┘
```

### 各模块职责

| 模块 | 职责 | 关键文件 |
|------|------|----------|
| **Agent Core** | 协调所有组件，管理主循环 `while not finished: step()`，状态转换 | `browser_use/agent/agent.py` |
| **MessageManager** | 管理与 LLM 的全部通信——系统提示词、用户消息、对话历史、工具输出 | `browser_use/agent/message_manager.py` |
| **Memory** | 基于 mem0 的历史记忆压缩，每 N 步总结一次，防止上下文窗口爆炸 | `browser_use/agent/memory.py` |
| **LLM Interface** | 统一 LLM 调用层，支持 OpenAI / Anthropic / Google / Ollama / 自定义 ChatBrowserUse 模型 | `browser_use/agent/llm_manager.py` |
| **Controller** | 动作注册表 + 执行引擎，将 LLM 输出的 JSON 动作映射到 Playwright 操作 | `browser_use/controller/service.py` |
| **BrowserContext** | 浏览器生命周期管理，DOM 元素提取、截图、多标签页、页面状态维护 | `browser_use/browser/context.py` |

### 技术栈

- **语言**：Python ≥ 3.11（大量使用 type hints 和 async/await）
- **浏览器引擎**：Playwright（支持 Chromium / Firefox / WebKit）
- **LLM 集成**：LangChain 的 ChatModel 接口（统一多模型调用）
- **结构化输出**：Pydantic BaseModel（强制 JSON Schema 校验）
- **记忆压缩**：Mem0（可选）
- **异步运行时**：asyncio（全链路异步）

### 工作流程（单步循环）

```python
# 伪代码还原核心循环
async def step():
    # 1. 从 BrowserContext 提取当前页面状态
    dom_elements = await browser_context.get_interactive_elements()  # DOM解析
    screenshot = await browser_context.take_screenshot()              # 视觉截图
    
    # 2. MessageManager 构造 prompt（DOM + 截图 + 历史记忆 + 系统提示词）
    current_prompt = message_manager.build_prompt(
        elements=dom_elements, 
        image=screenshot,
        history=memory.get_summary()
    )
    
    # 3. 调用 LLM，获取结构化输出
    response = await llm_interface.chat(current_prompt)
    
    # 4. Pydantic 校验 → 解析为 AgentOutput
    agent_output = AgentOutput.model_validate(response)
    # 包含: current_state{evaluation, memory, next_goal} + action[...]
    
    # 5. Controller 执行动作序列
    for action in agent_output.actions:
        result = await controller.execute(action)  # click/input_text/scroll...
        
    # 6. Memory 压缩（每 N 步触发）
    if step_count % memory_interval == 0:
        memory.compress(message_manager.history)
    
    # 7. 判断是否 finished
    return agent_output.current_state.finished
```

## 技术亮点

### 1. DOM + 视觉双通道融合的元素感知方案

这是 Browser Use 最核心的创新。它不单纯依赖 DOM 树或纯视觉模型，而是将两者融合：

```python
# browser_use/browser/context.py — DOM元素提取
async def get_interactive_elements(self) -> list[DOMElement]:
    # 使用 JavaScript 在浏览器内执行，提取可交互元素
    elements = await self.page.evaluate('''() => {
        const allElements = document.querySelectorAll(
            'a, button, input, [onclick], [role="button"], select, textarea, [tabindex]'
        );
        return Array.from(allElements).map((el, idx) => ({
            index: idx,
            tag: el.tagName.toLowerCase(),
            type: el.type || '',
            text: el.innerText?.trim().substring(0, 50),  // 截断防止token爆炸
            xpath: get_xpath(el),                          // XPath用于精确定位
            bbox: el.getBoundingClientRect(),               // 视觉边界框
            visible: is_visible(el),
            interactable: !el.disabled && !el.readOnly
        })).filter(el => el.visible && el.interactable);
    }''')
    return [DOMElement(**e) for e in elements]
```

**设计巧思：**
- 用 `[index]<type>text</type>` 格式呈现元素，让 LLM 能用数字索引引用（比 XPath 更自然）
- 非交互元素作为「上下文文本」提供但不带索引，避免 LLM 误操作
- 同时提供截图 + bounding box 标注，视觉和文本互补验证
- 元素文本截断至 50 字符，严格控制 token 消耗

### 2. 三层保障的结构化输出机制

Browser Use 对 LLM 输出格式稳定性做了三层防护：

**第一层：SystemPrompt 格式约束**
```json
{
  "current_state": {
    "evaluation_previous_goal": "Success|Failed|Unknown",
    "memory": "已执行X次，剩余Y次...",
    "next_goal": "下一步要做什么"
  },
  "action": [{"action_name": {"param": value}}]
}
```
每个字段都有详细说明、枚举值约束和使用示例。

**第二层：One-shot 示例注入**
```python
# 初始化时注入标准示例
placeholder_message = HumanMessage(content='Example output:')
example_tool_call = AIMessage(
    content='',
    tool_calls=[{
        'name': 'AgentOutput',
        'args': {
            'current_state': {
                'evaluation_previous_goal': 'Success',
                'memory': '0 out of 10 websites analyzed',
                'next_goal': 'Click the search button'
            },
            'action': [{'click_element': {'index': 5}}]
        }
    }]
)
message_manager.messages.extend([placeholder_message, example_tool_call])
```

**第三层：Pydantic 强校验**
```python
class AgentOutput(BaseModel):
    current_state: CurrentState
    action: list[Action]

class CurrentState(BaseModel):
    evaluation_previous_goal: Literal["Success", "Failed", "Unknown"]
    memory: str
    next_goal: str

# LLM 输出不符合 schema 时直接抛 ValidationError
# 配合重试机制：最多重试 3 次，修改 prompt 后重新请求
```

这种三层设计使得即使是非强推理模型也能保持较高的格式遵循率。

### 3. 动作空间（Action Space）的精心设计

Controller 注册的动作集是经过大量实战迭代的结果：

| 动作 | 参数 | 设计考量 |
|------|------|----------|
| `click_element` | `index` | 最基础操作，通过 DOM index 定位 |
| `input_text` | `index, text` | 自动处理 focus → clear → type → blur 全流程 |
| `scroll` | `direction, amount` | 解决长页面元素不可见问题 |
| `go_to_url` | `url` | 支持新标签页打开 |
| `extract_content` | `goal` | 让 LLM 自己决定提取什么，而非固定字段 |
| `done` | `text, success` | 必须作为最后一步显式调用 |
| `wait` | `seconds` | 处理动态加载页面 |
| `tab_open/close` | `url or index` | 多标签并行任务 |
| `key_press` | `key` | 快捷键支持（Ctrl+S 等） |
| `save_file` | `path` | 下载文件到本地 |

关键设计：
- **动作序列批执行**：一次 LLM 输出可以包含多个动作按序执行（如连续填表单），减少交互轮数
- **页面跳转中断**：如果某个动作导致页面导航，后续动作自动取消并重新获取状态
- **自定义动作扩展**：通过 `@tools.action` 装饰器注册任意 Python 函数作为新动作

```python
tools = Tools()

@tools.action(description='获取当前页面URL')
def get_current_url() -> str:
    return browser_context.current_url

agent = Agent(task="...", llm=llm, browser=browser, tools=tools)
```

### 4. 记忆管理与上下文窗口保护

Browser Use 的记忆系统解决了 Agent 运行中最大的工程痛点——上下文爆炸：

```python
# browser_use/agent/memory.py
class Memory:
    def __init__(self, use_mem0: bool = True):
        self.summaries = []  # 压缩后的历史摘要列表
    
    async def compress(self, messages: list[ManagedMessage]):
        """每 N 步触发，将历史对话压缩为摘要"""
        # 过滤掉系统提示词（type='init'），只压缩实际交互历史
        history_msgs = [m for m in messages if m.metadata.type != 'init']
        
        # 使用 mem0 或 LLM 自身进行总结
        summary = await self._summarize(history_msgs)
        self.summaries.append({
            'step_range': (start_step, current_step),
            'content': summary
        })
        
        # 移除已被压缩的原始消息，释放 token 空间
        messages.remove_old(before=start_step)
```

**亮点：**
- 系统提示词标记为 `type='init'`，永远不会被裁剪（保证 Agent 人格稳定）
- 裁剪策略是「移除最早的非系统消息」，简单但有效
- 压缩摘要以 `procedural memory` 形式重新注入 prompt，让 Agent 知道"之前做过什么"
- 支持 Mem0 作为外部记忆引擎，也支持纯 LLM 自压缩（无额外依赖模式）

### 5. 多模型适配策略 —— Function Calling vs Prompt 注入双轨制

不同模型对工具调用的支持能力差异巨大，Browser Use 做了优雅的双轨适配：

```python
# 对于支持 function calling 的模型（GPT/Claude/Qwen）：
# 工具以 Pydantic schema 注册，由 LangChain 绑定到 API 的 tools 参数
agent = Agent(
    task=task,
    llm=ChatOpenAI(model="gpt-4o"),
    # 工具自动转为 function_calling schema
)

# 对于不支持 function calling 的模型（DeepSeek-R1 等）：
# 工具描述以文本形式注入 system prompt
# 模型需要在文本输出中自行生成 JSON 格式的 tool call
agent = Agent(
    task=task,
    llm=ChatDeepSeek(model="deepseek-r1"),
    # 内部检测到不支持 FC → 自动切换到 prompt-inject 模式
)
```

这种设计让同一个 Agent 代码可以跑在任何模型上，无需用户关心底层差异。

## 对我们（OpenClaw/楠楠）的启发

### 🔥 直接可借鉴的点

**1. DOM + 视觉双通道的元素表示方法**
- OpenClaw 已有 `browser` 工具（基于 Playwright），但在向 LLM 呈现页面信息时可以参考 Browser Use 的 `[index]<type>text</type>` 格式
- 当前我们的 snapshot 返回的是 aria refs，可以考虑增加 DOM 文本摘要层，让非视觉模型也能更好理解页面
- **具体改进**：在 `browser snapshot` 结果中加入简化的可交互元素列表（index + 类型 + 文本），作为纯文本 fallback

**2. 结构化输出的三层保障**
- 我们的 subagent 返回结果目前依赖自然语言，如果改为 Pydantic schema 约束的 JSON 输出，可以大幅提升结果解析的可靠性
- 特别是 coding-agent 的子任务结果，可以用类似 `TaskOutput(current_status, result, next_steps)` 的格式规范

**3. 动作序列批执行**
- 当前 `browser act` 每次只能执行一个动作。参考 Browser Use 的设计，支持一次传入多个动作（如连续填表单）可以显著减少 LLM 调用轮数
- 这对飞书文档操作等需要多步完成的场景尤其有价值

**4. 记忆压缩的分层裁剪策略**
- 我们的心跳/日常记忆也可以采用类似的分层策略：核心身份信息（SOUL.md 内容）永不裁剪，日常对话按时间衰减
- `type='init'` 标记系统消息的做法很优雅，可以在我们的 message 管理中引入

**5. 自定义动作的装饰器模式**
- `@tools.action(description=...)` 这种注册方式非常简洁，我们的 skill/tool 扩展机制可以参考
- 降低用户扩展能力的门槛

### 💡 更深层的思考

**Browser Use 的本质启示**：它证明了**把 Agent 放进真实环境里做闭环反馈**，比纯粹的 chat agent 有价值得多。OpenClaw 已经有了 browser、exec、feishu 等工具——我们其实已经在做这件事了。关键差距在于：

1. **状态呈现的质量**：Browser Use 花了大量工程精力在「如何让 LLM 理解当前页面状态」上（DOM 提取算法、元素过滤、文本截断、视觉标注）。我们在每个工具的状态反馈上都可以更精细化。
2. **自纠错的工程化**：Browser Use 的 `evaluation_previous_goal` 字段强制 LLM 每一步都评估上一步是否成功，形成隐式的 Reflexion 循环。我们的 agent 也可以在每次工具调用后加入类似的自我评估环节。
3. **特定领域优化出 SOTA**：Browser Use 没有做通用 Agent，而是死磕浏览器自动化这一个场景，做到了 89.1% 的基准成功率。**专注 > 泛化**。
