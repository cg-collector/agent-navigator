## 一、项目概述

Skyvern是一款**开源AI驱动浏览器自动化平台**，核心理念是用自然语言描述任务，Skyvern自动打开真实Chromium浏览器、视觉理解页面、完成复杂多步操作。其最大技术特色：**不依赖任何XPath或CSS选择器，完全靠视觉大模型（Vision LLM）理解和操作任意网站。**

在Skyvern出现之前，浏览器自动化需要针对每个网站编写定制脚本，依赖DOM解析和XPath交互——一旦网站改版，脚本就会失效。Skyvern从根本上改变了这一范式：LLM视觉理解网页，像人类一样"看屏幕→找元素→做操作"，因此可以操作从未见过的网站，且对网站改版天然免疫。

**一句话定位：** AI时代的通用浏览器RPA——用自然语言驱动真实浏览器完成任意网页操作。

**行业背景与动机（2024-2026演进）：**

浏览器自动化是企业数字化运营的基础需求，典型场景包括：
- 登录供应商门户，自动下载发票
- 批量填写job application表格
- 自动从多个电商网站比价并下单
- 定期抓取竞品网站价格数据
- 自动化测试Web应用的用户流程

传统解决方案（Selenium、Playwright、Puppeteer）本质上是"编程式自动化"——需要开发者精确描述每个操作步骤，网站改版即失效。据Skyvern官方数据，企业平均每个网站自动化脚本每年需要维护4-6次，改版成本极高。

Skyvern瞄准的是一个更大胆的目标：**零脚本、零维护的通用浏览器Agent**。

**版本演进：**
- **Skyvern 1.0**（2023-2024）：单Actor Prompt循环，WebVoyager基准约45%
- **Skyvern 2.0**（2024-2025）：引入Planner-Agent-Validator架构，WebVoyager达**85.85% SOTA**，全面对标甚至超越Google Mariner等闭源方案
- **2026**：持续迭代，聚焦企业级可靠性、Cloud规模化部署和多云支持

### 2.2 Skyvern 1.0 vs 2.0 架构演进

Skyvern 2.0的核心架构升级解决了1.0的两个关键瓶颈：

**1.0的问题：单Actor Prompt循环**

```
用户Prompt → LLM决定操作 → Playwright执行 → 结果反馈 → 循环
```

这种设计的问题：
- 复杂任务（如"登录→找订单→下载3张发票"）中，LLM容易在多个子目标间顾此失彼
- 无法追踪已完成的子目标和剩余目标，导致重复操作或遗漏
- WebVoyager得分仅约45%

**2.0解法：Planner-Agent-Validator三角架构**

```
用户Prompt
     │
     ▼
┌─────────────┐
│   Planner   │ ← 将复杂任务分解为有序子任务序列
│  (规划器)   │   例："登录→找发票→下载3张"
└──────┬──────┘
       │ 子任务队列
       ▼
┌─────────────┐
│  Task Agent │ ← 每个子任务独立执行Agent Loop
│  (任务执行) │   screenshot → extract DOM → LLM决策 → Playwright执行
└──────┬──────┘
       │ 执行结果
       ▼
┌─────────────┐
│  Validator  │ ← 闭环验证：子任务是否真正达成？
│  (验证器)   │   若失败→反馈Planner重新规划
└──────┬──────┘
       │
       └── 成功 → 返回结果
```

**Validator是2.0的灵魂创新**：它解决了"操作声称成功但实际失败"的问题（如点击了"加入购物车"按钮但商品实际没加进去）。Validator通过再次截图+LLM判断状态来确认目标达成，形成真正的闭环反馈。

**性能提升数据：**
- V1.0（单Actor）：WebVoyager ~45%
- V1.0 + Planner：WebVoyager ~68.7%
- V2.0（+ Validator）：WebVoyager **85.85% SOTA**

## 三、核心技术深度剖析

### 3.1 Agent Loop核心机制

每一步Agent循环包含5个精确步骤：

```
Step N:
┌─────────────────────────────────────────────────────┐
│ 1. Screenshot         截取当前视口截图               │
│    ↓                                           │
│ 2. Extract DOM        提取简化DOM树               │
│    (交互元素: inputs, buttons, links, dropdowns)  │
│    ↓                                           │
│ 3. LLM Reasoning       screenshot + DOM + prompt  │
│    → 决定下一步操作(click/fill/select/scroll)    │
│    ↓                                           │
│ 4. Playwright Execute  在真实Chromium中执行操作   │
│    ↓                                           │
│ 5. Goal Check         验证目标是否达成             │
│    └─ 达成 → 返回结果                           │
│    └─ 未达成 → 回到Step1继续循环                │
└─────────────────────────────────────────────────────┘
```

**关键设计洞察：**

LLM同时接收截图（视觉信息）和DOM树（结构信息）：
- **截图**：提供布局、按钮位置、模态框等视觉上下文
- **DOM**：提供精确的元素标识符，Playwright据此定位元素

这避免了纯视觉模型的"幻觉点击"问题——LLM先用视觉理解页面布局，再结合DOM中的`id`、`name`、`label`等标识符精确定位。

**典型任务步数：** 2-10步/任务，可通过`max_steps`参数控制成本。

### 3.2 Vision LLM与多引擎支持

Skyvern不绑定特定模型，采用**引擎抽象层**：

| 引擎选项 | 说明 |
|---------|------|
| GPT-4o | 默认主力模型，视觉能力强 |
| GPT-4o-mini | 成本优化版，适合简单任务 |
| Claude Sonnet | 备选引擎 |
| 自定义LLM | 支持接入任意兼容API |

Skyvern Cloud版本默认使用GPT-4o + GPT-4o-mini组合（根据任务复杂度自动选择），自托管版本可完全控制模型选择。

### 3.3 Playwright双模交互

Skyvern的核心是Playwright扩展，同时支持三种交互模式：

**模式一：传统Playwright（精确控制）**
```python
await page.click("#login-button")           # 精确XPath
await page.fill("#email", "user@example.com") # 精确选择器
```
适合已知固定元素，已完成初始登录等场景。

**模式二：AI增强（自然语言定位）**
```python
await page.click(prompt="Click the login button")       # AI找按钮
await page.fill(prompt="Email field", value="a@b.com")  # AI找输入框
```
适合元素ID不固定但界面相对稳定的场景。

**模式三：AI原生（完全自然语言）**
```python
await page.act("Click the login button and wait for dashboard")
```
Skyvern内部执行完整Agent Loop，适合复杂多步操作。

**SDK四大核心命令：**

| 命令 | 功能 | 说明 |
|------|------|------|
| `page.act(prompt)` | 自然语言执行操作 | 最常用，复杂多步操作 |
| `page.extract(prompt, schema)` | 提取结构化数据 | 支持JSON Schema |
| `page.validate(prompt)` | 验证页面状态 | 返回bool |
| `page.prompt(prompt, schema)` | 任意LLM查询 | 自定义prompt |

### 3.4 凭证与安全管理

Skyvern设计了专门的**Credentials API**，解决浏览器自动化中最敏感的问题——凭证管理：

```
凭证注入流程（LLM全程不可见真实值）：
1. 用户通过API或Dashboard存储：密码/TOTP/信用卡
2. 凭证加密存储（AES-256）
3. 执行时直接注入浏览器DOM元素
4. LLM只负责找到输入框位置，凭证值完全不经过LLM
```

**支持的凭证类型：**
- 静态密码（Password）
- TOTP动态验证码（2FA）
- 信用卡信息
- 外部集成：Bitwarden、1Password、Azure Key Vault

**对比OpenClaw的密钥管理（SecretRegistry）：**

两者思路相似——都强调"工具执行时才访问密钥、输出时屏蔽密钥值"。但Skyvern更进一步，将凭证注入浏览器DOM，完全绕过LLM的记忆层；OpenClaw的SecretRegistry则侧重于API调用场景的运行时注入。

### 3.5 进阶原生能力

**CAPTCHA自动处理：** 内置CAPTCHA检测与解决，无需用户配置。官方称已覆盖主流CAPTCHA类型。

**住宅代理与地理定位：** 内置住宅代理网络，覆盖30+国家，可按`proxy_location`设置任务执行地区。

**浏览器状态持久化：**
- **Browser Sessions**：跨任务保持浏览器实例（最长24小时），适合需要串联登录态的任务链
- **Profiles**：将Cookie/Token/LocalStorage打包为可复用配置文件，新任务无需重新登录

**文件操作：**
- 从网站下载文件（返回签名URL + checksum）
- 上传文件到表单字段
- 解析PDF/CSV/Excel

**Webhook与异步执行：** 任务可配置`webhook_url`，完成后回调通知，支持长时任务的异步编排。

## 四、Workflow系统——可编排的自动化蓝图

Skyvern 2.0的Workflow系统将一次性Task升级为**可复用、可版本化、可视化编排的自动化蓝图**。

### 4.1 Block体系

Workflow由多个Block串联构成，每个Block完成特定原子操作：

**导航与交互类：**
| Block | 功能 |
|-------|------|
| Navigation | AI自主导航（输入目标，AI决定点击顺序） |
| Action | 执行单一操作（click/fill/select/upload） |
| Go to URL | 直接跳转 |
| Login | 使用存储凭证认证 |
| Wait | 暂停指定时间 |
| Human Interaction | 暂停等待人工介入 |

**数据与逻辑类：**
| Block | 功能 |
|-------|------|
| Extract | 结构化数据提取 |
| Loop | 遍历列表（自动化翻页/批量操作） |
| Condition | 条件分支（IF/ELSE） |
| Code | 执行任意Python代码 |

**集成类：**
| Block | 功能 |
|-------|------|
| Email | 发送结果邮件 |
| Slack | 通知到Slack频道 |
| Webhook | 触发外部API |

### 4.2 工作流编排示例

```
[Navigate Block] → [Login Block] → [Loop Block]
                                          │
                    ┌─────────────────────┼─────────────────────┐
                    ▼                     ▼                     ▼
              [Extract Invoice]     [Extract Invoice]    [Extract Invoice]
                    │                     │                     │
                    └─────────────────────┼─────────────────────┘
                                          ▼
                                   [Download Block]
                                          │
                                          ▼
                                   [Webhook Block] → 发送结果
```

Workflow支持Jinja2模板：
- 参数引用：`{{search_query}}`
- 前置Block输出引用：`{{extract_block.order_id}}`

Workflow可在Dashboard可视化构建，或通过API以代码方式定义。

## 六、对OpenClaw的借鉴价值

### 可落地的想法

#### 💡 启发1：Validator闭环验证机制

OpenClaw在执行工具调用后，缺乏对"操作是否真正成功"的闭环验证。如果引入类似Validator的设计：
- 每个关键操作后截图/获取结果状态
- LLM判断目标是否达成
- 未达成则重新规划操作路径

这对楠楠的`agent-study`任务尤其有用——当执行搜索/读取等关键操作时，先验证结果是否符合预期，再决定下一步。

**实现路径：** 在OpenClaw的Agent循环中，工具执行后插入一个轻量级"验证prompt"，让LLM判断当前状态是否满足任务目标。

#### 💡 启发2：双模交互（AI + 精确控制fallback）

Skyvern的"AI失败后fallback到精确选择器"设计值得借鉴：
```python
# 先试AI，AI失败后用精确XPath兜底
await page.click("#submit-btn", prompt="Click Submit")
```
OpenClaw的工具调用可以采用类似策略：
- 先尝试AI理解的灵活方式
- 失败时降级到精确的参数规范

#### 💡 启发3：Browser-based工具能力

OpenClaw目前没有原生的浏览器自动化能力。引入Playwright-based的browser工具可以：
- 自动填写网页表单
- 从Web界面提取数据
- 完成需要登录态的Web操作

这比现有的`web_fetch`（纯文本提取）强大得多，因为可以处理JavaScript渲染页面和交互式Web应用。

**实现路径：** 参考Skyvern的`page.act()`接口封装一个`browser_tool`，基于Playwright CDP协议实现。

#### 💡 启发4：Workflow引擎的模块化设计

Skyvern的Block系统（Navigation/Extract/Loop/Condition）是一个设计精巧的工作流引擎。OpenClaw的sub-agent委托机制可以借鉴这种Block化思维：
- 将复杂任务分解为标准化的"Block"（工具块）
- Block之间通过输出/输入参数连接
- 支持条件分支和循环
