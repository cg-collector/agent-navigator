## 2. 项目简介

随着 LLM Agent 从研究原型走向真实产品（邮件助手、HR 聊天机器人、文件管理器等），其行为可靠性和安全性变得至关重要。然而，现有 Agent 评估方法存在三大短板：一是高度依赖人工设计测试用例（如 OSWorld 耗时 1800+ 人时设计 369 个任务）；二是在模拟沙箱中运行，与真实交互环境存在差距且模拟器可能产生幻觉；三是对简单文本 Agent 关注过多，缺乏对多模态、UI 导航、工具调用等产品级 Agent 的测试能力。

SpecOps 的核心创新在于提出了一套**完全自动化的端到端 Agent 测试管线**，仅需输入 Agent 的功能列表和规格说明，即可自主生成测试用例、搭建测试环境、执行被测 Agent、验证结果并输出缺陷报告——全程无需人工介入。它在 5 个真实产品级 Agent 上发现了 164 个真实 Bug，F1 达 0.89，单次测试成本低于 $0.73，运行时间不到 8 分钟。

## 3. 核心技术深度剖析

### 3.1 四阶段专家 Agent 架构

SpecOps 将测试过程解构为四个阶段，每个阶段由一个专门的 LLM Agent 负责：

**阶段一：测试用例生成（Test Spec Generation）**
采用**双专家协作**机制。Test Architect 接收 Agent 的功能列表和规格说明，生成结构化测试用例，包含三要素：(1) 环境配置需求（如创建特定文件、发送种子邮件）；(2) 自然语言提示词（模拟真实用户输入）；(3) 期望行为描述。Test Analyst 随后对测试用例进行审查，评估可行性、提示词完整性和验证预言（oracle）的泛化能力。这种双人审核机制确保生成的测试用例既覆盖真实用户场景，又在执行层面切实可行。

**阶段二：环境搭建（Environment Setup）**
Infrastructure Manager 将高层配置需求转化为具体的工具调用——通过 `pyautogui` 操作 GUI、通过 API 发送邮件、通过 shell 命令创建文件等。关键设计是其**自适应策略**：当实际搭建过程中发现约束（如某平台不支持特定功能）时，自动回溯调整测试规格，而非盲目执行失败。

**阶段三：测试执行（Test Execution）**
Engineer Specialist 通过**通用 UI 原语**（键盘/鼠标操作）与被测 Agent 交互，不依赖特定平台的自动化 API。它采用**类人视觉监控**方案——通过屏幕截图实时记录 Agent 行为，监控完成指标和错误信息。这种设计使其能够跨平台工作：CLI 工具（Open Interpreter）、Web 应用（ProxyAI）、浏览器扩展（TaxyAI）均可用同一套机制测试。

**阶段四：验证与审计（Validation & Auditing）**
该阶段由两个专家协作完成。Investigator Specialist **独立探测环境变化**（检查文件系统、浏览已发送邮件等），与测试执行阶段完全解耦，避免执行阶段的信息污染验证结果。Judge 随后汇总所有证据——截图、执行记录、Investigator 报告——使用 **Meta Chain-of-Thought 提示策略**进行推理判断，产出最终缺陷报告，包含 Bug 描述、复现步骤和严重程度评估。

### 3.2 数据流与执行流程

整体流程为线性管线：功能列表/规格说明 → 测试用例（JSON） → 环境状态 → 执行日志+截图 → 缺陷报告。每个阶段的输出通过结构化 JSON 传递给下一阶段，确保端到端的**任务一致性**——验证阶段的评判标准与测试用例的期望行为严格对齐。

底层实现基于 AWS Bedrock Converse API，通过 `converse.py` 抽象 LLM 调用，`operate.py` 处理工具执行循环和 JSON 提取，`screen_utils.py` 和 `setup_utils.py` 分别封装 GUI 操作工具和环境配置工具。

### 3.3 跨平台适配机制

SpecOps 不依赖平台特定的自动化框架（如 Selenium、Appium），而是统一通过屏幕截图 + 键盘/鼠标输入的原始 UI 原语与被测 Agent 交互。这使得单个框架能覆盖 CLI、Web、浏览器扩展等异构平台，避免了为每种 Agent 类型开发专用适配器的工作量。

## 4. 技术亮点

### 亮点一：双专家审查机制（Dual-Specialist Review）
测试用例生成阶段采用 Architect + Analyst 双人审核，类似软件开发中的代码评审。Architect 负责"创造"，Analyst 负责"质疑"——检查提示词是否存在歧义、环境配置是否实际可行、验证预言是否过于严格或宽松。这大幅提升了测试用例质量和缺陷检测的有效性。

### 亮点二：自适应环境搭建（Adaptive Environment Setup）
传统测试框架在环境搭建失败时会直接中止。SpecOps 的 Infrastructure Manager 能在发现约束时动态调整测试规格——比如发现目标平台不支持某种文件格式，就替换为平台支持的格式继续测试。这种"边做边调"的策略显著提升了测试成功率。

### 亮点三：独立验证与 Meta Chain-of-Thought
Investigator 在完全独立于执行管线的情况下探测环境，避免了确认偏差（执行阶段认为"成功了"会影响验证判断）。Judge 使用 Meta Chain-of-Thought，即让 LLM 先构建评估推理链，再基于推理链做出判断，而非直接给出结论。这种结构化推理使缺陷判定更加可靠和可解释。

## 5. 基准测试表现

在 5 个真实产品级 Agent（覆盖 Email、File System、HR Q&A 三大领域）上的评测结果：

| 指标 | SpecOps | AutoGPT | LLM Script |
|------|---------|---------|------------|
| **提示成功率** | 100% | 11-49.5% | ~50% |
| **规划步骤执行率** | 100% | 较低 | 易崩溃 |
| **Bug 检测 F1** | 0.89 | — | — |
| **发现真实 Bug 数** | 164 | — | — |
| **单次测试成本** | <$0.73 | — | — |
| **单次测试时间** | <8 分钟 | — | — |

SpecOps 发现的 Bug 类型包括：邮件发送到错误收件人、文件被静默覆盖、命令在错误目录执行、占位符未替换等——都是核心功能缺陷，而非边缘情况。

## 7. 对我们的实际启发

1. **专家 Agent 分工模式**：SpecOps 的四阶段专家架构验证了"一个 Agent 做所有事"不如"多个专家 Agent 各司其职"有效的理念。在我们设计多 Agent 系统时，应优先考虑按职责拆分而非按任务拆分。

2. **独立验证是关键**：执行者和验证者必须解耦。我们在评估 Agent 输出质量时，评审 Agent（如 evolve-review.py）应完全独立于生成过程，避免信息污染。

3. **自适应错误恢复**：Agent 系统不能"一条路走到黑"。当中间步骤失败时，应具备回溯和调整策略的能力，这比简单的重试机制更优雅。

4. **低成本高效能**：每次测试 <$0.73 的成本证明，用 LLM 测试 LLM 在经济上是完全可行的。我们可以借鉴这套思路构建 Agent 自我审查机制。
