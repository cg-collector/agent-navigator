## 架构设计

### 整体架构：Monorepo + 分层引擎 + 实时协作

Sim 采用 **Turborepo Monorepo** 管理，核心分为以下层次：

```
┌─────────────────────────────────────────────┐
│  前端层 (Next.js App Router + ReactFlow)     │
│  ┌───────────┐ ┌──────────┐ ┌──────────────┐ │
│  │ Canvas画布 │ │ Copilot  │ │ Knowledge    │ │
│  │ 拖拽编辑器 │ │ NL→Node  │ │ 向量知识库   │ │
│  └─────┬─────┘ └────┬─────┘ └──────┬───────┘ │
│        │ Socket.io(实时)           │          │
├────────┼───────────────────────────┼──────────┤
│  引擎层 (executor/)                │          │
│  ┌────┴───────────────────────────┴──────┐   │
│  │ Workflow Engine                      │   │
│  │ ├─ PathTracker (依赖解析/拓扑排序)    │   │
│  │ ├─ BlockHandler[] (策略模式块处理器)   │   │
│  │ ├─ LoopManager / ParallelManager      │   │
│  │ └─ InputResolver (动态输入绑定)       │   │
│  └──────────────────────────────────────┘   │
│                                             │
│  Agent层 (executor/handlers/agent/)         │
│  ┌──────────────────────────────────────┐   │
│  │ AgentBlockHandler                    │   │
│  │ ├─ buildMessages() → 消息序列构建    │   │
│  │ ├─ formatTools() → 工具格式化+调用    │   │
│  │ ├─ executeProviderRequest() → LLM调用 │   │
│  │ └─ StreamingExecution → 流式响应      │   │
│  └──────────────────────────────────────┘   │
├─────────────────────────────────────────────┤
│  基础设施层                                  │
│  PostgreSQL(pgvector) │ BullMQ Worker       │
│  Better Auth         │ E2B(远程代码执行)    │
│  Trigger.dev(后台任务)│ Drizzle ORM          │
└─────────────────────────────────────────────┘
```

### 核心模块

#### 1. 工作流引擎 (`apps/sim/executor/`)

这是整个系统的**神经中枢**，负责：

- **流程解析**：将序列化的 `SerializedWorkflow` 转换为可执行的块序列
- **执行调度**：通过 `PathTracker` 做依赖分析 + 拓扑排序，决定下一批可执行的块
- **状态管理**：`ExecutionContext` 贯穿整个生命周期，用 `Map` 结构高效管理块状态
- **流程控制**：支持条件分支（Condition）、路由分发（Router）、循环（Loop）、并行（Parallel）

#### 2. 块处理器体系（策略模式）

```typescript
// executor/index.ts — 处理器注册表
this.blockHandlers = [
  new TriggerBlockHandler(),      // 触发器
  new AgentBlockHandler(),        // AI Agent 核心
  new RouterBlockHandler(this.pathTracker),   // 路由
  new ConditionBlockHandler(this.pathTracker, this.resolver), // 条件
  new EvaluatorBlockHandler(),    // 表达式求值
  new FunctionBlockHandler(),     // 自定义函数
  new ApiBlockHandler(),          // HTTP API 调用
  new LoopBlockHandler(...),      // 循环
  new ParallelBlockHandler(...),  // 并行
  new ResponseBlockHandler(),     // 响应输出
  new WorkflowBlockHandler(),     // 子工作流嵌套
  new GenericBlockHandler(),      // 通用兜底
];
```

每种块类型一个 Handler 类，统一实现 `execute(block, inputs, context) -> BlockOutput` 接口。**策略模式**让新增块类型只需添加一个 Handler 类，符合开闭原则。

#### 3. 执行上下文（ExecutionContext）

```typescript
export interface ExecutionContext {
  workflowId: string;
  executionId?: string;
  blockStates: Map<string, BlockState>;     // 每个块的执行状态
  blockLogs: BlockLog[];                     // 执行日志
  metadata: { startTime, duration, endTime };
  environmentVariables: Record<string, string>;
  workflowVariables: Record<string, any>;
  decisions: {                                // 条件/路由决策结果
    router: Map<string, string>;
    condition: Map<string, boolean>;
  };
  loopIterations: Map<string, number>;       // 循环迭代计数
  loopItems: Map<string, any[]>;             // 循环项数据
  completedLoops: Set<string>;               // 已完成循环
  executedBlocks: Set<string>;               // 已执行块集合
  activeExecutionPath: Set<string>;          // 当前活跃路径
  workflow: SerializedWorkflow;              // 完整工作流定义
}
```

这个设计非常关键——**一个 Context 对象贯穿整个执行过程**，所有 Block 共享同一个状态容器，天然支持跨块数据传递。

#### 4. Agent 执行流水线

Agent 块的执行遵循经典的 **ReAct（Reasoning + Acting）模式**：

```
用户输入 → buildMessages() → 格式化消息序列
         → formatTools()   → 将工具转换为 LLM 可调用的 function calling 格式
         → LLM Provider    → 发送请求（支持流式）
         → 解析响应        → 判断是最终回答还是工具调用
         ├── 工具调用 → 执行工具 → 结果注入下轮消息 → 回到LLM
         └── 最终回答 → 返回 BlockOutput
```

## 技术亮点

### 亮点1：基于 ReactFlow 的可视化 DAG 编排——"Figma 式"体验

Sim 最大的差异化就是**把 Agent 开发从写代码变成了画图**。底层依赖 ReactFlow 库实现：

- **自定义节点系统**：每种块类型（Agent、API、Condition、Loop 等）对应一个 ReactFlow 自定义节点组件
- **边连接 = 数据流**：节点间的连线不仅是视觉关系，更定义了数据的流向和依赖
- **序列化/反序列化**：画布上的 DAG 图直接序列化为 `SerializedWorkflow` JSON 存入数据库，执行时反序列化为引擎的输入

这比纯代码框架（如 LangGraph 的 Python DSL）降低了**一个数量级的使用门槛**。非开发者也能通过拖拽搭建 Agent。

**代码级洞察**：ReactFlow 的 `onConnect` 回调捕获边的创建事件，更新内部 graph state；`applyNodeChanges` / `applyEdgeChanges` 处理拖拽交互。整个画布状态通过 Zustand store 管理，变更自动持久化。

### 亮点2：PathTracker 拓扑调度引擎——智能执行路径计算

```typescript
// PathTracker 核心算法（简化）
public getNextExecutionLayer(context: ExecutionContext): string[] {
  const availableBlocks = new Set<string>();

  for (const block of context.workflow.blocks) {
    if (context.executedBlocks.has(block.id)) continue;

    const dependenciesMet = this.checkBlockDependencies(block.id, context);
    const loopConstraintsMet = this.checkLoopConstraints(block.id, context);
    const parallelConstraintsMet = this.checkParallelConstraints(block.id, context);

    if (dependenciesMet && loopConstraintsMet && parallelConstraintsMet) {
      availableBlocks.add(block.id);
    }
  }

  return this.sortBlocksByPriority(Array.from(availableBlocks), context);
}
```

这个设计的精妙之处：
- **不是简单的线性执行**，而是真正的 **DAG 拓扑排序**
- 每一轮只执行「所有依赖已满足」的块，天然支持**并行分支同时执行**
- 循环约束检查防止无限循环（结合 `loopIterations` 计数器）
- `activeExecutionPath` 跟踪当前活跃路径，用于调试时高亮显示

对比：LangGraph 用 Python 的 StateGraph 做 DAG 编排，Sim 用 TypeScript 实现了同等能力但加了**可视化的前端映射**。

### 亮点3：策略模式 BlockHandler 体系——极致的可扩展性

12 种块类型，每个都是独立的 Handler 类，共享统一接口：

```typescript
// 所有 Handler 遵循的隐式接口
interface BlockHandler {
  execute(block: SerializedBlock, inputs: any, context: ExecutionContext): Promise<BlockOutput>;
}
```

扩展新块类型的步骤：
1. 在 `blocks/types.ts` 定义新的块配置 schema
2. 创建 `XXXBlockHandler` 实现 `execute()` 方法
3. 在处理器注册表中注册
4. 前端创建对应的 ReactFlow 自定义节点组件

**这种设计让 Sim 的块生态可以无限扩展**。社区贡献者可以开发自定义块（如数据库查询块、邮件发送块），作为插件接入。比 LangChain 的 Tool/Chain 继承体系更清晰——因为 Sim 把"流程控制结构"（循环、并行、条件）也做成了块，而不只是"工具调用"。

### 亮点4：双模式执行——Inline + Queue（BullMQ）

Sim 支持两种执行路径：

- **Inline 模式**（默认）：HTTP 请求内同步执行工作流，适合快速原型和简单流程
- **Queue 模式**（需 Redis）：通过 BullMQ 将执行任务投递到后台 Worker，适合长时间运行的工作流、Webhook 触发、定时调度

```bash
# docker-compose.prod.yml 默认启动 worker
# 但如果没有 REDIS_URL，worker 会 idle，走 inline 路径
```

这个设计很务实：**简单场景零配置启动，复杂场景加 Redis 就能获得生产级可靠性**。Worker 进程与 Next.js 主进程分离，避免长任务阻塞 HTTP 服务。

### 亮点5：Copilot NL→Node —— 自然语言生成工作流节点

Sim 内置了 Copilot 功能，用户可以用自然语言描述想要的功能，Copilot 自动生成对应的节点和连接：

- "帮我做一个先搜索网页再总结内容的 Agent" → 自动生成：Trigger → Agent(Search) → Agent(Summarize) → Response
- "这里报错了帮我修复" → 分析错误信息并修正节点配置

底层原理：用户的自然语言描述被发送到 Copilot API（Sim 托管服务），返回结构化的节点创建指令，前端通过 ReactFlow API 动态添加节点和边。自托管实例可通过 `COPILOT_API_KEY` 接入。

**这实质上是 "LLM-as-Compiler" 模式在可视化编程中的应用**——用 LLM 把自然语言编译成 DAG 图。

## 对 OpenClaw / 楠楠的启发

### 1. 可视化工作流编排的参考价值 ⭐⭐⭐⭐⭐

OpenClaw 目前主要靠代码/SKILL 定义 Agent 行为。Sim 证明了**可视化 DAG 编排对降低使用门槛有巨大价值**。启发点：

- 可以考虑为 OpenClaw 的 skill/cron/heartbeat 配置增加**可视化编辑界面**（类似 Sim 的 ReactFlow 画布）
- 用户拖拽节点定义"收到消息 → 调用技能 → 发送回复"的流程，而非手写 YAML/Markdown
- 这对非技术用户（学长的朋友、团队成员）特别有价值

### 2. ExecutionContext 全局状态容器模式 ⭐⭐⭐⭐

Sim 的 `ExecutionContext` 设计值得借鉴——**一个对象贯穿任务全生命周期，所有子模块读写同一份状态**。OpenClaw 的 subagent 执行也可以引入类似概念：

- 统一的 TaskContext 替代散落的参数传递
- 内置 `executedSteps`、`decisions`、`loopIterations` 等字段
- 天然支持任务断点续跑和调试回放

### 3. 策略模式 + 插件化块体系 ⭐⭐⭐⭐

OpenClaw 的 Skill 体系已经有一定插件性，但 Sim 的 BlockHandler 更进一步——**连流程控制结构（循环/并行/条件）都可插件化**。启发：

- Skill 可以不只是"工具"，还可以是"流程控制原语"
- 比如：循环重试 Skill、条件分支 Skill、并行派发 Skill
- 统一接口 `execute(context) → Result`，组合出任意复杂工作流

### 4. 双模式执行（Inline vs Queue）⭐⭐⭐⭐

OpenClaw 目前的 subagent 已经有类似设计（直接执行 vs 后台执行），但 Sim 的实现更优雅：

- 简单任务 inline 同步返回（低延迟）
- 复杂任务进队列 + Worker 异步处理（高可靠）
- 通过环境变量无缝切换，无需改代码
- **建议 OpenClaw 的 cron/subagent 也明确区分这两种模式并在文档中强调**

### 5. Copilot NL→Graph 的交互范式 ⭐⭐⭐

"用自然语言描述需求 → 自动生成工作流图"这个交互模式很有前瞻性。OpenClaw 可以考虑：

- 让学长说"帮我设一个每天早上9点查天气然后发消息的流程"
- 自动生成对应的 cron + skill 组合配置
- 比手动写 cron 表达式友好得多

### 总结评价

| 维度 | 评分 | 说明 |
|------|------|------|
| **创新性** | ★★★★☆ | 可视化 + Copilot 的组合在开源 Agent 框架中独树一帜 |
| **架构质量** | ★★★★☆ | 分层清晰，策略模式+DAG引擎设计成熟 |
| **实用性** | ★★★★★ | 真正降低了 Agent 开发门槛，22k Stars 是证明 |
| **代码质量** | ★★★☆☆ | TypeScript 全栈，但部分逻辑耦合较深 |
