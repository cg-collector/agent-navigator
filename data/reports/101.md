## 架构设计

### 核心模块

CrewAI 的架构围绕四个核心抽象展开：

```
┌─────────────────────────────────────────────┐
│                  CrewAI                      │
├──────────┬──────────┬──────────┬────────────┤
│  Agent   │   Task   │   Crew   │   Flow     │
│ (角色)   │ (任务)   │ (团队)   │ (工作流)   │
└────┬─────┴────┬─────┴────┬─────┴────┬───────┘
     │          │          │          │
     ▼          ▼          ▼          ▼
 ┌───────┐ ┌────────┐ ┌─────────┐ ┌──────────┐
 │ role  │ │ desc   │ │ agents  │ │ @start() │
 │ goal  │ │ expect │ │ tasks   │ │ @listen()│
 │ tools │ │ output │ │ process │ │ state    │
 │ memory│ │ agent  │ │ verbose │ │ plot()   │
 └───────┘ └────────┘ └─────────┘ └──────────┘
```

**1. Agent（智能体）**
- 每个 Agent 是一个自治单元，拥有 `role`（角色）、`goal`（目标）、`backstory`（背景故事）三元组定义人格
- 可配置 LLM、Tools、Memory、Knowledge Sources 等 20+ 属性
- 支持委托（delegation）：Agent 可以将子任务分配给其他 Agent
- 内置 Reasoning 模式：执行前先反思规划

**2. Task（任务）**
- 描述性任务定义：包含 `description`（做什么）、`expected_output`（期望输出）
- 绑定到特定 Agent 执行
- 支持输出到文件（`output_file`）

**3. Crew（团队）**
- 将 Agents 和 Tasks 组合成一个执行单元
- 支持两种执行模式：
  - `Process.sequential`：顺序执行
  - `Process.hierarchical`：层级式（一个 Manager Agent 协调其他 Agent）
- 使用 `@CrewBase` + `@agent`/`@task`/`@crew` 装饰器模式组织代码

**4. Flow（工作流）— v2.0 核心创新**
- 事件驱动的有向无环图（DAG）编排层
- 通过 `@start()` 和 `@listen()` 装饰器声明式定义执行流
- 支持结构化状态管理（Pydantic BaseModel）和非结构化状态
- 内置 `plot()` 方法生成流程可视化图
- 可与 Crew 原生嵌套：Flow 中调用 Crew，Crew 中嵌入 Flow

### 技术栈

| 层面 | 技术 |
|------|------|
| 语言 | Python 3.10 - 3.13 |
| 包管理 | UV（Astral 出品，比 pip/poetry 更快） |

| 企业功能 | AMP Suite（Control Plane + Tracing + 部署） |

### 工作流程

```
用户输入 → kickoff(inputs)
    │
    ▼
┌──────────────────┐
│  Crew 初始化      │  ← 加载 agents.yaml + tasks.yaml
│  解析 Agent/Task  │
└────────┬─────────┘
         │
         ▼
┌──────────────────┐     ┌──────────────────┐
│  Sequential:     │     │  Hierarchical:   │
│  Task1→Task2→... │     │  Manager 分配任务  │
│  按顺序执行       │     │  给各 Agent       │
└────────┬─────────┘     └────────┬─────────┘
         │                       │
         ▼                       ▼
┌──────────────────────────────────────┐
│  每个 Task 执行循环：                  │
│  1. 构建 System Prompt（role+goal+   │
│     backstory + task description）    │
│  2. Agent 调用 LLM（带工具定义）       │
│  3. LLM 决定：回答 / 调用工具 / 委托   │
│  4. 工具执行 → 结果回传 → 继续循环     │
│  5. 达到 max_iter 或给出最终答案       │
└──────────────────────────────────────┘
         │
         ▼
    返回最终结果 / 写入 output_file
```

## 技术亮点

### 1. Role-Playing Prompt Engineering（角色扮演提示工程）

CrewAI 最核心的设计哲学是将每个 Agent 定义为「角色」而非「函数」。这不是简单的 system prompt 注入，而是一套完整的 persona 注入机制：

```python
# agents.yaml — 声明式角色定义
researcher:
  role: >
    {topic} Senior Data Researcher
  goal: >
    Uncover cutting-edge developments in {topic}
  backstory: >
    You're a seasoned researcher with a knack for uncovering the latest
    developments in {topic}. Known for your ability to find the most relevant
    information and present it in a clear and concise manner.
```

**为什么这很聪明？** CrewAI 的内部 prompt 模板会将 `role` + `goal` + `backstory` 组合成一个丰富的 system prompt，同时注入任务的 `description` 作为 user message。这种分离使得：
- 角色定义可复用（一个 Researcher 可以做不同 topic 的研究）
- 人格一致性由 backstory 保证（不只是"你是一个研究员"这种单薄指令）
- YAML 与代码解耦，非技术人员也能调整 Agent 行为

**对比 OpenClaw/楠楠：** 我们的 SOUL.md + IDENTITY.md 本质上在做同样的事——通过文本定义人格。但 CrewAI 将其工程化为框架级原语，支持多角色并存且各自独立。

### 2. 双范式架构：Crews（自治）+ Flows（控制）

这是 CrewAI 区别于大多数 Agent 框架的关键设计。大多数框架只选一种范式：

```python
# Crew 范例 — 自治多 Agent 协作
crew = Crew(
    agents=[researcher, writer, reviewer],
    tasks=[research_task, writing_task, review_task],
    process=Process.sequential,  # 或 hierarchical
)
result = crew.kickoff(inputs={'topic': 'AI Agents'})

# Flow 范例 — 精确的事件驱动控制
class ResearchFlow(Flow):
    model = "gpt-4o-mini"

    @start()
    def generate_queries(self):
        response = completion(model=self.model, messages=[...])
        self.state["queries"] = parse(response)
        return self.state["queries"]

    @listen(generate_queries)  # 监听上游方法完成
    def run_research(self, queries):
        # 可以在这里嵌入一个完整的 Crew！
        return research_crew.kickoff(inputs={'queries': queries})

    @listen(run_research)
    def write_report(self, findings):
        ...

flow = ResearchFlow()
flow.plot()  # 生成 DAG 可视化
result = flow.kickoff()
```

**设计洞察：**
- **Crew = 水平协作**：Agent 之间平等协商，适合探索性任务
- **Flow = 垂直编排**：开发者精确控制执行路径，适合生产环境
- 两者可以嵌套：Flow 的一个节点可以是一个完整 Crew
- 这解决了 Agent 框架长期存在的「 autonomy vs control 」矛盾

### 3. 声明式装饰器驱动开发（@CrewBase 模式）

```python
@CrewBase
class LatestAiDevelopmentCrew():
    agents: List[BaseAgent]
    tasks: List[Task]

    @agent  # 自动注册到 self.agents 列表
    def researcher(self) -> Agent:
        return Agent(
            config=self.agents_config['researcher'],  # 从 YAML 加载
            verbose=True,
            tools=[SerperDevTool()]
        )

    @task   # 自动注册到 self.tasks 列表
    def research_task(self) -> Task:
        return Task(config=self.tasks_config['research_task'])

    @crew    # 自动组装 agents + tasks
    def crew(self) -> Crew:
        return Crew(
            agents=self.agents,  # 装饰器自动收集
            tasks=self.tasks,
            process=Process.sequential,
        )
```

这个装饰器模式的精妙之处：
- **自动收集**：`@agent` 装饰的方法自动归入 `self.agents`，`@task` 归入 `self.tasks`
- **配置分离**：YAML 管理内容（prompt 文本），Python 管管逻辑（工具选择、参数传递）
- **类型安全**：`agents: List[BaseAgent]` 和 `tasks: List[Task]` 提供静态类型约束
- **IDE 友好**：装饰器不会破坏代码跳转和自动补全

### 4. 结构化状态管理与 Pydantic 集成

```python
from pydantic import BaseModel

class ResearchState(BaseModel):
    queries: list[str] = []
    findings: dict = {}
    report_draft: str = ""
    quality_score: float = 0.0
    iteration_count: int = 0

class ResearchFlow(Flow[ResearchState]):  # 泛型绑定状态类型！

    @start()
    def plan(self):
        self.state.queries = ["AI agents 2026", "multi-agent frameworks"]
        # self.state 有完整类型检查，写错字段名会报错

    @listen(plan)
    def execute(self):
        self.state.iteration_count += 1  # Pydantic 验证
```

关键点：
- `Flow[ResearchState]` 泛型语法让状态**强类型化**，不是裸 dict
- Pydantic 自动处理类型校验、默认值、序列化
- 状态在 Flow 的所有方法间共享，天然支持跨步骤数据传递
- 对比 LangGraph 的 annotated state，CrewAI 的方式更 Pythonic

### 5. Hierarchical Process 与 Manager Agent 模式

```python
# 层级式执行 — 一个经理 Agent 协调其他 Agent
crew = Crew(
    agents=[researcher, writer, reviewer],
    tasks=[...],
    process=Process.hierarchical,
    manager_llm=ChatOpenAI(model="gpt-4o", temperature=0),  # 经理用更冷静的模型
    manager_agent=Agent(
        role="Project Manager",
        goal="Coordinate team to produce high-quality output",
        backstory="Experienced tech lead who excels at delegation",
    ),
    verbose=True
)
```

工作原理：
1. Manager Agent 接收所有任务列表
2. Manager 分析每个任务，决定分配给哪个 Agent
3. Manager 审查 Agent 的输出质量
4. 不满意则打回重做（带有具体反馈）
5. 这本质上实现了一个 **Manager-Worker 多轮审查循环**

## 对 OpenClaw / 楠楠 的启发

### 🔍 可直接借鉴的点

**1. 角色扮演框架化**
- 当前楠楠的人格定义在 SOUL.md/IDENTITY.md 中是纯文本，加载后作为 context 注入
- CrewAI 的 `role + goal + backstory` 三元组可以形式化为一个 `Persona` 类，支持：
  - 多角色切换（桐桐 vs 楠楠有不同的 role/goal/backstory）
  - 动态人格微调（根据场景临时叠加 backstory 片段）
  - 人格版本管理和 A/B 测试

**2. Crew + Flow 双范式**
- OpenClaw 目前的 subagent 机制类似 Crew（派生子 agent 并行/串行执行）
- 但缺少 Flow 层的**声明式工作流编排**。如果引入类似 `@start()` / `@listen()` 的 DAG 编排能力：
  - 复杂任务可以可视化编排（`plot()` 生成 DAG 图）
  - 条件分支、循环、错误重试都可以声明式表达
  - 心跳任务可以用 Flow 重构，而不是硬编码在 HEARTBEAT.md 里

**3. YAML 配置 + 代码逻辑分离**
- CrewAI 的 `agents.yaml` / `tasks.yaml` 模式值得参考
- 当前 OpenClaw 的 skill 定义分散在 SKILL.md + 脚本中
- 如果将 skill 的「元信息」（描述、触发条件、所需权限）抽到 YAML，「执行逻辑」保留在 Python/脚本中，可以让非开发者也能创建和修改 skill

**4. Manager-Worker 审查模式**
- 当前的 evolve-review.py 已经实现了类似的评审循环
- 但它是手动的（脚本调用）。如果能做成框架级的 Manager Agent：
  - 每个 subagent 完成后自动触发 Manager 评审
  - 评分不通过则自动派生改进型 subagent
  - 形成 `Execute → Review → Improve → Re-review` 的自动闭环

**5. 结构化状态管理**
- 当前 OpenClaw 的 session state 较为隐式
- 引入 Pydantic BaseModel 约束的状态对象可以让：
  - 跨 subagent 的数据传递类型安全
  - 状态变更可追踪（Pydantic 的 field 监听）
  - 方便序列化到 memory 文件做持久化

### ⚠️ 需要注意的局限

- **CrewAI 本质仍是 LLM-centric**：每个 Agent 就是一个 LLM 循环，没有真正的持续学习或记忆进化
- **工具系统相对简单**：相比 OpenClaw 丰富的 tool 生态（browser/message/feishu 等），CrewAI 的工具偏基础
- **Flows 是新特性**（v2.0），成熟度和生产验证还不如 Crews
- **过度依赖 prompt engineering**：Agent 的行为质量高度依赖 YAML 中写的 prompt 文本，调优成本不低

---

## 总结

CrewAI 的核心价值在于它把「多 Agent 协作」这件事做得**极其简洁**。从安装到跑通第一个多 Agent 示例只需要 10 分钟，这降低了 AI Agent 的准入门槛。它的双范式设计（Crews + Flows）巧妙地平衡了自治与控制，Role-Playing 三元组（role/goal/backstory）虽然简单但效果出奇地好。

对于 OpenClaw 而言，最值得借鉴的不是照搬其实现，而是吸收其**设计思想**：声明式编排、角色形式化、状态类型安全、以及 Manager-Worker 审查闭环。这些概念可以无缝融入现有的 skill/subagent 体系。
