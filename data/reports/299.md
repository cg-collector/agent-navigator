# 第299份研究报告：OmegaUse — 通用跨平台GUI Agent

> **项目名称：** OmegaUse: Building a General-Purpose GUI Agent for Autonomous Task Execution
> **论文来源：** arXiv:2601.20380（2026年1月）
> **研究机构：** 百度（作者含Jingbo Zhou, Hua Wu, Haifeng Wang等）
> **报告日期：** 2026-04-12

---

## 一、项目背景与问题定义

当前GUI Agent领域面临三大核心痛点：

**1. 跨平台泛化能力不足。** 现有Agent大多针对单一平台优化——要么只做移动端（如MobileAgent系列），要么只做桌面端（如OSWorld生态）。一个能同时胜任手机和电脑操作的通用模型尚未成熟。

**2. 数据质量与规模的双重困境。** GUI交互数据天然稀缺：人工标注成本高昂，而自动采集的数据充满噪声（误对齐的边界框、模糊的指令描述、不完整的轨迹）。开源数据集（Aguvis、UI RefExp等）合计166万条实例，但经严格质检后仅11万条可用——质量衰减率高达93%。

**3. 训练范式效率低下。** 端到端训练将视觉感知与序列决策耦合在一起，导致模型在空间定位精度和长程规划能力之间难以兼顾。大参数量 Dense 模型推理成本过高，难以部署到实际场景。

OmegaUse 正是针对以上三个问题提出的系统性解决方案：通过**解耦式双模型架构**分离感知与规划，通过**精心设计的数据工程管线**解决数据质量问题，并通过**MoE架构+GRPO强化学习**实现效率与性能的平衡。

## 二、核心架构深度剖析

### 2.1 整体架构：解耦式双模型设计

OmegaUse 的核心创新在于将 GUI Agent 能力拆分为两个专业化模型：

| 组件 | 职责 | 输入 | 输出 |
|------|------|------|------|
| **OmegaUse-G（Grounding Model）** | 高精度视觉感知/元素定位 | 截图 + 自然语言描述 | 边界框坐标 [xmin,ymin,xmax,ymax] |
| **OmegaUse-N（Navigation Model）** | 序列化任务规划与导航 | 截图 + 任务指令 + 历史动作 | 动作三元组 (Observation, Thought, Action) |

这种解耦设计的优势在于：
- Grounding 模型可以专注于像素级精度的空间定位，不受规划逻辑干扰
- Navigation 模型可以聚焦于高层策略推理，不必同时学习细粒度坐标预测
- 两个模型可以独立迭代优化，降低联合训练的复杂度

### 2.2 数据工程管线

**Grounding 数据管线（从166万→11万）：**
1. 聚合6个公开数据集：Aguvis、UI RefExp、Widget Captioning、SeeClick、Uground、OS-Atlas
2. 规则过滤：去除边界框异常、描述过短等噪声样本
3. 下采样至30万条候选集
4. 人工校准：逐条检查边界框对齐度和描述准确性
5. 最终产出 **111K 高质量 grounding 样本**

**Navigation 数据管线（三管齐下）：**

*路径一：开源数据审计*
- 基于 AGUVIS（AITW + Mind2Web），两阶段质控：规则过滤（去除<3步轨迹、冗余动作）→ MLLM 审计（验证任务完成度与逻辑一致性）

*路径二：自动化轨迹合成（核心创新）*
- **Bottom-up 探索驱动：** DFS 策略探索单个应用，收集原始交互三元组 <state, action, next_state> → 构建状态转移图 → MLLM 语义聚类合并冗余UI状态 → 避环提取轨迹 → MLLM 将每条轨迹语义化为自然语言任务描述
- **Top-down 分类法引导：** 为桌面和移动环境分别设计分层任务分类体系 → 基于分类生成任务描述 → 专家模型在模拟环境中执行 → 成功轨迹记录 + 人工验证平台确保可靠性

*路径三：跨终端专家演示*
- LLM 从分类体系合成至少5步的任务指令 → 人工逐步执行 → 两层质量保证协议

### 2.3 统一动作空间

OmegaUse 设计了一套跨平台的统一动作原语：

**基础动作（全平台）：** Click, Drag, Scroll, Type, Wait, Finished

**桌面扩展：** Hotkey, LeftDouble, RightSingle, Hover

**移动扩展：** LongPress, PressBack, PressHome, PressEnter

**Web扩展：** Stop

这种标准化使得同一套模型权重可以在不同平台上复用，无需为每个平台单独训练。

### 2.4 MoE 骨干网络

OmegaUse 采用 Mixture-of-Experts 架构作为 backbone，在保持大模型推理能力的同时显著降低计算开销。MoE 的稀疏激活特性意味着每次前向传播只调用部分专家子网络，使得实际 FLOPs 远低于同等参数量的 Dense 模型。

## 三、技术亮点

### 亮点1：Bottom-up + Top-down 双向数据合成框架

这是 OmegaUse 最具创新性的技术贡献。传统的 GUI 数据采集方式存在根本性矛盾：
- **纯 Bottom-up（随机探索）：** 覆盖面广但缺乏目标导向，产生大量无意义轨迹
- **纯 Top-down（按任务执行）：** 目标明确但覆盖有限，容易遗漏边缘场景

OmegaUse 将两者有机结合：
1. DFS 探索收集底层 UI 状态转移图谱 → 保证**覆盖率**
2. 任务分类体系引导高层目标执行 → 保证**目的性**
3. MLLM 作为"翻译层"，将底层原子操作抽象为高层语义任务 → 实现**双向桥接**

原理上，这类似于程序分析中的 *symbolic execution + concrete testing* 组合策略——既穷举状态空间又保证路径有意义。

### 亮点2：GRPO 强化学习用于 GUI Agent 训练

Group Relative Policy Optimization（GRPO）最初由 DeepSeek 在数学推理中提出，OmegaUse 首次将其系统性应用于 GUI Agent 领域。

GRPO 的核心优势在于**无需独立的 Critic 模型**。传统 PPO 需要训练一个价值函数网络来估计 baseline，这在高维 GUI 状态空间中极不稳定。GRPO 通过同组内相对比较来估计优势函数：

$$\hat{A}_i = \frac{r_i - \text{mean}(\{r_1,...,r_G\})}{\text{std}(\{r_1,...,r_G\})}$$

对于 OmegaUse 的两个子模型，奖励函数设计也高度专门化：

**Grounding 奖励：**
$$R = R_{fmt} \cdot w_1 + R_{pos} \cdot w_2$$
其中 $R_{pos}$ 判断预测中心点是否落在 GT 边界框内——这是一个精确的空间约束。

**Navigation 奖励（分步）：**
$$R = R_{fmt} + R_{act}, \quad R_{act} = R_{type} + R_{coord}$$
其中 $R_{coord}$ 采用阶梯式坐标精度奖励——距离越近奖励越高，用阈值 $\theta_1, \theta_2$ 分档。这种细粒度的奖励 shaping 对 GUI 操作的学习至关重要。

### 亮点3：OS-Nav 跨终端离线评测基准

OmegaUse 自带了一套新的 benchmark suite —— OS-Nav，包含：
- **ChiM-Nav：** 中国 Android 移动环境，142条轨迹/69个应用/991步，聚焦日常使用场景
- **Ubu-Nav：** Ubuntu 桌面环境，101条轨迹/641步，评估多步推理能力

两条 benchmark 都提供**专家验证的推理轨迹 + MLLM 合成的中间 CoT 描述 + 人工精修**，质量远超现有自动生成的 benchmark。这填补了中文 GUI 环境和 Linux 桌面环境的评测空白。

## 四、基准测试表现

| Benchmark | OmegaUse 指标 | SOTA 对比 |
|-----------|--------------|-----------|
| **ScreenSpot-V2** | **96.3%** (SOTA) 🏆 | 超越 UI-Venus-Ground-72B、Seed1.5-VL |
| **ScreenSpot-Pro** | 55.47% | OS-Icon 子类 43.82%，以远小于72B的参数量达到竞争力水平 |
| **AndroidControl (offline)** | **79.1% Step SR** (SOTA) 🏆 | Type Accuracy 87.6% |
| **AndroidWorld (online)** | 55.7% Success Rate | 纯截图端到端模式，超越多个更大开源模型 |
| **ChiM-Nav** | **74.24% Step SR** (SOTA) 🏆 | Type Accuracy 87.78%，超越 UI-Venus-72b |
| **Ubu-Nav** | **55.9% Avg Success** (SOTA) 🏆 | 非坐标动作 48.6%，超越 Holo2-30B-A3B (50.0%) |

**关键观察：** OmegaUse 在4个 benchmark 上达到 SOTA 或领先地位，尤其在 grounding 精度和移动端导航方面优势明显。桌面端（Ubu-Nav）55.9%的成功率说明跨平台泛化仍有提升空间——桌面环境的复杂性显著高于移动端。

## 五、横向对比

| 维度 | OmegaUse | CUA (OpenAI) | OS-Copilot | Mobile-Agent-v3 |
|------|----------|-------------|------------|-----------------|
| **平台覆盖** | 移动+桌面+Web | 桌面为主 | 桌面(Windows/Mac/Linux) | 移动端 |
| **架构** | 解耦(Grounding+Navigation) | 端到端 | 端到端 | 端到端 |
| **骨干网络** | MoE（高效） | GPT-4o（Dense） | Qwen-VL（Dense） | Dense |
| **训练方法** | SFT + GRPO | RL + 蒙特卡洛树搜索 | SFT | SFT |
| **Grounding能力** | ScreenSpot-V2 96.3% | 强（未公开分数） | 中等 | 弱 |
| **开源程度** | 模型+数据+Benchmark | API闭源 | 部分开源 | 开源 |
| **中文支持** | ✅ ChiM-Nav专项优化 | ❌ | ❌ | 部分 |
| **最大优势** | 跨平台+SOTA grounding+MoE效率 | 综合能力强+生态好 | 多操作系统覆盖 | 移动端轻量 |

**OmegaUse 的核心优势：**
1. 解耦架构使各模块可独立优化，可解释性强
2. MoE 架构在实际部署中的推理成本可控
3. 中文 GUI 环境的专项优化是独有差异化
4. 完整的开源方案（模型+数据+Benchmark）

**OmegaUse 的劣势：**
1. 桌面端55.9%成功率仍偏低，复杂多应用工作流处理不足
2. 相比 CUA 的蒙特卡洛树搜索规划，缺少显式的搜索增强机制
3. 在线评测（AndroidWorld 55.7%）与顶级闭源方案仍有差距
4. 缺少错误恢复和自我纠错机制的设计讨论

## 六、对我们的实际启发

### 可落地想法1：解耦式 Agent 架构改造
我们当前的 Agent 是端到端的（VLM 直接输出动作）。参考 OmegaUse 的解耦思路，可以将视觉定位模块独立出来：
- 用小模型做专门的 UI 元素检测（类似 OmegaUse-G）
- 主模型专注于动作规划和推理
- 这与我们之前研究 OmniParser 的方向一致，但 OmegaUse 提供了更完整的工程实践

### 可落地想法2：GRPO 训练流程引入
如果我们在未来有条件进行模型微调，GRPO 比 PPO 更适合 GUI 场景：
- 不需要训练 Critic 网络，节省大量计算
- 组内相对比较在高维状态空间更稳定
- 可以直接用我们的交互日志构造 rollout 数据

### 可落地想法3：统一动作空间的标准化
OmegaUse 的跨平台动作原语设计值得借鉴。我们在构建自己的 Agent 时应提前定义标准化的动作 schema，而不是针对特定平台硬编码操作序列。

### 可落地想法4：Bottom-up + Top-down 数据合成
如果我们需要构建特定领域的 GUI 自动化能力（比如飞书操作自动化），可以采用 OmegaUse 的双向数据合成框架：
- 先用脚本遍历飞书界面获取状态转移图
- 再基于任务分类体系引导生成有目标的操作轨迹
- 用 VLM 做"翻译层"连接底层操作与高层语义
