## 一、项目定位

Instructor 是一个**专注于从 LLM 中提取结构化、类型安全数据的库**。它不是完整的 Agent 框架，而是 Agent 技术栈中**最关键的"数据层"基础设施**——解决的是所有 Agent 开发者都面临的根本问题：

> **LLM 输出是不可靠的自由文本，但你的程序需要可靠的结构化数据。**

Instructor 的定位可以概括为：
- **Pydantic + LLM 的桥梁**：用 Pydantic Model 定义期望的输出结构，Instructor 负责让 LLM 的输出严格符合该结构
- **多 Provider 统一抽象层**：一套 API 对接 OpenAI/Anthropic/Google/Mistral/Ollama/DeepSeek 等 15+ LLM 提供商
- **生产级可靠性保障**：自动重试、验证失败重询问（Reasking）、流式部分输出

官方给自己的定位很清晰："**Instructor for extraction, PydanticAI for agents**"——Instructor 专注做结构化提取这一件事，把 Agent 运行时的事情交给 PydanticAI。

## 二、核心架构

### 2.1 整体架构分层

```
┌─────────────────────────────────────────────┐
│              用户代码 (User Code)             │
│   client.create(response_model=User, ...)    │
├─────────────────────────────────────────────┤
│            Instructor Core Layer             │
│  ┌──────────┐ ┌──────────┐ ┌──────────────┐ │
│  │ Prompt   │ │ Parse    │ │ Validation   │ │
│  │ Injection│ │ Logic    │ │ & Reask      │ │
│  └──────────┘ └──────────┘ └──────────────┘ │
│  ┌──────────┐ ┌──────────┐ ┌──────────────┐ │
│  │ Retry    │ │ Streaming│ │ Multi-Provider│ │
│  │ (Tenacity)│ │ Support  │ │ Adapter      │ │
│  └──────────┘ └──────────┘ └──────────────┘ │
├─────────────────────────────────────────────┤
│              Pydantic V2                     │
│     (Schema Definition + Validation)         │
├─────────────────────────────────────────────┤
│           LLM Provider Layer                 │
│  OpenAI / Anthropic / Google / Ollama / ...  │
└─────────────────────────────────────────────┘
```

### 2.2 核心数据流

```
1. 用户定义 Pydantic Model (response_model)
       ↓
2. Instructor 将 JSON Schema 注入到 System Prompt 中
   （利用 LLM 原生 structured output 能力或 prompt engineering）
       ↓
3. 发送请求给 LLM Provider
       ↓
4. 解析 LLM 返回的 JSON
       ↓
5. Pydantic 验证 → 通过？→ 返回类型安全的实例
       ↓ 失败
6. Reask 机制：将验证错误反馈给 LLM 让其修正
       ↓ (可配置 max_retries)
7. 重试直到通过或达到上限
```

### 2.3 关键模块

| 模块 | 职责 |
|------|------|
| `instructor.from_provider()` | 统一的 Client 工厂，屏蔽底层 Provider 差异 |
| `client.create()` | 同步结构化提取入口 |
| `client.create_async()` | 异步版本 |
| `client.stream()` | 流式部分结果迭代 |
| `client.stream_partial()` | 流式增量输出（逐步 refinement） |
| `instructor.Patch` | 增量更新模式，只返回变化的字段 |

## 三、5 个技术亮点

### 亮点 1：Reask（验证失败自动重询）机制 —— 核心杀手锏

这是 Instructor 最具创新性的设计。当 Pydantic 验证 LLM 输出失败时，Instructor **不会直接抛异常**，而是将验证错误信息构造成自然语言反馈，追加到对话上下文中，要求 LLM 重新生成。

```python
class User(BaseModel):
    age: int = Field(gt=0, lt=120)  # 必须是 0-120 之间的整数
    email: str = Field(pattern=r".+@.+\..+")  # 必须是合法邮箱

# 如果 LLM 返回 age=-1 或 email="not_an_email"
# Instructor 自动构造类似这样的 reask prompt：
# "Your response failed validation: 
#  - field 'age': Input should be greater than 0 (got -1)
#  - field 'email': string does not match email pattern
#  Please fix these errors and return the corrected JSON."
```

**技术深度：**
- 利用 Pydantic 的 `ValidationError` 精确提取每个字段的错误信息
- 将机器可读的错误转化为 LLM 可理解的自然语言提示
- 支持嵌套模型的递归 Reask（深层字段错误也能精确反馈）
- 与 Tenacity 库集成实现指数退避重试策略
- 这本质上是一个 **LLM-as-a-Judge 的轻量闭环**：LLM 生成 → 规则验证 → 反馈修正

### 亮点 2：Patch 模式 —— 增量更新的优雅方案

Instructor 的 `Patch[T]` 类型解决了 Agent 场景中一个非常常见的问题：**当只需要更新对象的部分字段时，如何避免 LLM 重复生成全部内容？**

```python
from instructor import Patch

class User(BaseModel):
    name: str
    age: int
    email: str

# 只更新 email，其他字段保持不变
updated = client.create(
    response_model=Patch[User],
    messages=[{"role": "user", "content": "把邮箱改成 john@example.com"}]
)
# 返回的 Patch 对象只包含 changed fields
# 可以 .apply_to(original_user) 得到完整的新对象
```

**技术深度：**
- 基于 Pydantic 的 `model_copy(update=...)` 实现不可变更新语义
- 减少 token 消耗（不需要重复不变的字段）
- 对于多轮对话中的状态更新场景极其重要
- 类似 GraphQL 的 patch 思想应用到 LLM 结构化输出

### 亮点 3：Multi-Provider 统一抽象层

Instructor 通过 `from_provider()` 实现了一套统一的 Client 接口，底层适配了 15+ LLM 提供商的差异：

**关键差异处理：**
- **OpenAI**：使用原生 `response_format=json_schema`（最可靠的 structured output）
- **Anthropic**：使用 `tool_use` 格式模拟结构化输出（Claude 不支持原生 JSON mode）
- **Google Gemini**：使用 `json` controlled generation
- **Ollama/vLLM**：纯 prompt engineering + JSON parse fallback

这意味着切换 LLM Provider **只需改一行字符串**，业务代码完全不动。

**技术深度：**
- 每个 Provider Adapter 是一个独立的模块，遵循统一接口契约
- 自动检测 Provider 能力（是否支持 function calling / json mode / streaming）
- 根据能力选择最优的 extraction strategy
- 这种设计模式对楠楠的工作有直接参考价值——我们也在对接多个 API provider

### 亮点 4：Streaming Partial —— 流式增量结构化输出

Instructor 不仅支持一次性返回完整结构，还支持**流式逐步完善**的结构化输出：

```python
async for partial in client.stream_partial(
    response_model=User,
    messages=[...]
):
    print(partial)  # 每次 yield 一个逐渐完善的 User 实例
    # 第1次: User(name='John', age=None, email=None)
    # 第2次: User(name='John', age=30, email=None)
    # 第3次: User(name='John', age=30, email='john@...')
```

**技术深度：**
- 底层基于 LLM 的 streaming API（SSE/WebSocket）
- 每个 chunk 到达后尝试解析为 partial JSON
- 使用 Pydantic 的 `model_validate(..., strict=False)` 允许缺失字段
- 对于长列表提取场景特别有用（逐个元素出现而非等全部完成）
- UX 体验远好于等待完整响应后一次性渲染

### 亮点 5：多语言生态 + PydanticAI 战略协同

Instructor 已经从 Python 单语言扩展到了 **6 种编程语言**：Python、TypeScript、Go、Ruby、Elixir、Rust。每种语言的实现都遵循相同的核心设计哲学：

- 用该语言的 native 类型系统定义 schema（Python=Pydantic, TS=Zod, Go=struct tags）
- 统一的 Reask/Retry/Patch 语义
- 相同的多 Provider 适配模式

更值得注意的是 Instructor 团队的**产品战略**：他们明确将 Instructor 定位为"提取层"，同时推动 **PydanticAI** 作为上层的"Agent 运行时"。两者共享 Pydantic 生态，形成清晰的分层：

```
Instructor (结构化提取) → PydanticAI (Agent 编排) → 应用层
```

这种**做深不做宽**的产品哲学值得学习——与其做一个大而全的框架，不如把一件事做到极致再与生态伙伴协作。

## 四、对楠楠的启发（3 条具体可操作）

### 启发 1：引入 Instructor 式的 Reask 机制优化我们的 API 调用链路

**现状问题：** 我们调用 MiniMax/智谱 API 时，如果返回格式不符合预期（比如 JSON 解析失败、字段缺失），通常是直接报错或写死 fallback 逻辑，缺乏系统性的错误恢复。

**具体行动：**
- 为 workspace 的工具调用层封装一个 `validate_and_reask()` 函数
- 当 LLM 返回的工具调用参数不满足预期 schema 时，自动构造错误反馈 prompt 并重试
- 参考 Instructor 的 Reask 实现：捕获 ValidationError → 转自然语言 → 追加到 messages → 重新调用
- 预期效果：API 调用的成功率可以从当前的 ~95% 提升到 ~99%，减少大量因格式问题导致的无效调用浪费

### 启发 2：用 Pydantic Model 统一所有外部 API 的输入/输出类型定义

**现状问题：** 我们的脚本（web-search.py、evolve-review.py 等）的输入输出缺乏强类型约束，参数传递靠字典和字符串拼接，容易出错且难以调试。

**具体行动：**
- 为每个常用工具封装 Pydantic Model：`SearchQuery`, `SearchResult`, `ReviewRequest`, `ReviewScore`
- 在脚本入口处增加 `model_validate()` 校验，参数不对直接报错而不是跑到中间才崩溃
- 参考 Instructor 的 `from_provider()` 模式，为 MiniMax/智谱 API 封装统一的 typed client
- 长期来看，这能让楠楠的所有工具调用都有 IDE 自动补全和类型检查，大幅减少低级 bug

### 启发 3：借鉴 Patch 模式优化恋爱模拟器等状态更新场景

**现状问题：** love-sim.py 每次事件更新可能需要写入整个状态文件，而且多实例并发时容易丢失更新。

**具体行动：**
- 为 love-sim 的状态模型定义完整 Pydantic schema（属性、等级、成就等）
- 状态更新时采用 Patch 语义：只记录变更的 diff，而非全量覆盖
- 参考 Instructor 的 `Patch[T]` + `.apply_to()` 模式实现 immutable state updates
- 进一步可以考虑加入 Instructor 式的 validation：比如亲密度不能超过上限、等级不能降级等规则用 Pydantic validator 表达
- 这不仅适用于 love-sim，memory 文件的增量更新也可以用同样的思路
