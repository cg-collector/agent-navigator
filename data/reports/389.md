## 一、项目概述

OpenHands是Microsoft主导开发的**开源AI驱动软件工程平台**，定位为"AI-native软件工程基础设施"，覆盖从本地原型到企业级云端部署的全生命周期。其核心理念：**软件工程Agent不是辅助工具，而是能够独立承担真实工程任务的虚拟开发者。**

OpenHands与市面上其他AI编码助手（如Copilot）的本质区别在于：
- **Copilot**：IDE内嵌的智能补全，属于"人在回路"的辅助工具
- **OpenHands**：能够自主完成端到端工程任务（修复bug、升级依赖、处理PR）的独立Agent，运行在隔离环境中，有完整的生命周期管理

从产品形态看，OpenHands分为三层：
1. **SDK**：模块化Python库，定义Agent引擎的核心抽象（供其他产品二次开发）
2. **CLI**：本地命令行工具，对标Claude Code/Codex的本地开发体验
3. **GUI**（Local/Cloud）：可视化工作区，对标Devin/Jules的产品化界面

生态布局：
- `OpenHands` 主仓库（GUI + CLI + 核心）
- `software-agent-sdk` 独立仓库（V1架构的SDK核心，2025-2026重设计）
- `benchmarks` 评估基础设施
- `openhands-chrome-extension` 浏览器扩展
- `ToM-SWE` Theory-of-Mind模块

### 2.2 SDK V1 架构重设计（2025-2026核心演进）

OpenHands从V0单体架构演进到V1模块化SDK，解决了四大架构问题：

| 问题 | V0设计 | V1解决方案 |
|------|--------|-----------|
| 强制沙盒与本地灵活性不足 | 假设所有工具都在沙盒运行 | **可选隔离**：默认本地，需要时切换沙盒 |
| 可变配置与非确定性状态 | 配置系统混合多领域 | **无状态优先**：所有组件是不可变Pydantic模型 |
| 单体仓库与模块化不足 | Agent核心+评估+应用混在一起 | **四包分离**：SDK/Tools/Workspace/Server独立发布 |
| 单体逻辑与可扩展性差 | 新行为需要修改核心 | **两层可组合性**：部署包组合 + 类型化组件扩展 |

**四包设计：**
- `openhands.sdk`：核心抽象（Agent、Conversation、LLM、Tool、MCP）和事件系统
- `openhands.tools`：基于SDK抽象的具体工具实现
- `openhands.workspace`：执行环境（Docker/托管API），扩展SDK基类
- `openhands.agent_server`：暴露REST/WebSocket API的Web服务器

## 三、核心模块深度剖析

### 3.1 事件源状态管理（Event-Sourced State Management）

这是OpenHands SDK最独特的设计决策——将所有交互建模为不可变事件追加到日志中。

**事件层次结构：**
```
Event（基类）
├── LLMConvertibleEvent（可转为LLM消息格式）
│   ├── ActionEvent（工具调用）
│   └── ObservationBaseEvent（工具结果）
└── 内部事件（簿记，不暴露给LLM）
```

**ConversationState：单一状态真值源**

OpenHands的核心原则：**Agent、Tool、LLM等组件都是不可变、可序列化的Pydantic模型。唯一的有状态组件是ConversationState。**

```python
class ConversationState:
    metadata: dict              # Agent状态元数据
    event_log: list[Event]      # 仅追加的事件记录
    # FIFO锁确保线程安全更新
```

**持久化机制：**
- 元数据序列化到 `state.json`
- 事件作为独立JSON文件（实现增量持久化）
- 支持高效会话恢复

这与OpenClaw的状态管理有本质区别：
- OpenClaw：会话状态分散在内存和文件系统
- OpenHands：事件溯源（Event Sourcing），完整重建任意历史状态

### 3.2 LLM抽象层

**LiteLLM统一接口**：通过LiteLLM支持100+ LLM提供商，对上层隐藏provider细节。

**原生推理支持**：SDK透明处理前沿模型的原生推理字段：
- Anthropic的`ThinkingBlock`
- OpenAI的`ReasoningItemModel`
- OpenAI Responses API

**非函数调用模型适配**：对于不原生支持函数调用的模型（如某些开源模型），通过`NonNativeToolCallingMixin`将工具schema转换为文本提示指令，使用正则解析提取工具调用参数。这确保了工具调用机制的模型无关性。

**多LLM路由**：`RouterLLM`允许Agent根据不同请求选择不同模型。自定义实现可扩展`select_llm()`方法实现特定路由策略（按任务类型、按负载、按成本等）。

### 3.3 工具系统（Action-Execution-Observation模式）

OpenHands的工具定义采用三组件分离：

```
Tool = Action + Execution + Observation

Action（输入schema）
├── Pydantic模型验证LLM生成的参数
└── 定义工具调用的参数结构

Execution（实际逻辑）
├── ToolExecutor实现
└── 可能是本地shell、API调用、容器操作等

Observation（输出格式化）
├── 捕获执行输出
├── 定义返回schema
└── 转换为LLM兼容格式
```

**MCP集成**：MCP工具被视为SDK一等公民：
- MCP JSON Schema自动转换为Action模型
- 结果显示为结构化Observation
- `MCPToolDefinition`扩展标准`ToolDefinition`接口
- `MCPToolExecutor`将执行委托给FastMCP的`MCPClient`

**工具注册表与分布式执行**：
- 基于注册表的解析机制解耦工具规范与实现
- 工具规范可跨进程/网络边界序列化传输
- `register_tool(name, ToolDefinition)`绑定标识符到解析器
- 运行时根据会话上下文重构完整定义

### 3.4 Agent核心：无状态事件处理器

**架构原则**：Agent被定义为无状态、不可变的规范（包括LLM设置、工具规范、安全策略、Agent内容），与可变状态分离。

**事件驱动执行循环**：
```python
# Agent通过回调处理事件，发出结构化事件而非直接返回结果
on_event(event: Event) -> None
# 发出：MessageEvent / ActionEvent / ObservationEvent / ...
```

这种模式支持：
- 安全交织（安全检查可以插入任意两个事件之间）
- 增量执行（每个事件都是checkpoint）
- 事件流（支持WebSocket实时推送）

**子Agent委托**：通过委托工具（DelegateTool）支持分层Agent协调：
- 子Agent作为独立会话运行
- 继承父Agent的模型配置和工作区上下文
- 不需要修改核心SDK即可实现结构化并行

**AgentContext**：集中所有影响LLM行为的输入：
- 系统/用户消息的前缀/后缀
- 用户定义的Skill对象
- 技能可编程定义或从Markdown文件加载（与OpenClaw skill机制高度相似）

### 3.5 上下文窗口管理（Condenser系统）

**问题**：长时间会话中，LLM上下文历史持续增长，成本线性上升。

**解法**：Condenser系统在保持事件日志完整性的前提下，压缩历史：

```
原始事件流：[E1, E2, E3, E4, E5, ...]
     ↓ Condenser（删除事件，替换为摘要）
压缩后：[CondensationEvent(summary_of_E1-E3), E4, E5, ...]
```

- 凝结结果存储为`CondensationEvent`
- 在发送给LLM之前应用，保持事件日志完整性
- 默认`LLMSummarizingCondenser`可在不降低Agent性能的情况下将API成本降低**高达2倍**
- Condenser实现本身无状态（摘要存储在事件中）

### 3.6 本地会话（LocalConversation）

**设计目标**：提供最简单直接的SDK执行模式，用于快速迭代和调试。

**执行模型**：进程内运行完整Agent循环（LLM调用→工具调用→事件回调→状态更新），无需网络或容器开销。

**使用场景**：
- 本地开发调试
- 快速原型验证
- 无需隔离的简单任务

```python
conv = LocalConversation(agent=my_agent, workspace=LocalWorkspace("."))
conv.send_message("修复login函数的null pointer问题")
# 阻塞直到完成或出错
```

### 3.7 安全与确认机制

**SecurityAnalyzer**：根据风险将每个工具调用评级为低/中/高/未知，基于：
- 工具类型（文件操作 vs 网络请求 vs 系统命令）
- 操作参数（修改 vs 只读 vs 删除）
- 上下文（用户明确要求 vs Agent自主决策）

**ConfirmationPolicy**：根据行动风险和用户策略决定是否需要批准：
- WAITING_FOR_CONFIRMATION状态 → Agent暂停
- 用户批准后继续执行
- 被拒绝时Agent可能尝试更安全的替代方案
- 策略可在会话期间动态更新（自适应信任）

**SecretRegistry**：安全、后期绑定、可远程管理的凭据管理：
- 每个会话维护独立实例（严格隔离）
- 工具仅在执行时访问密钥
- 输出中所有密钥值被屏蔽
- 支持静态值或可调用对象
- 序列化期间加密

### 3.8 部署架构：从本地到远程

**Workspace统一抽象**：

```
BaseWorkspace（抽象基类）
├── LocalWorkspace：进程内执行，访问主机文件系统/shell
└── RemoteWorkspace：HTTP委托给Agent Server
    ├── DockerWorkspace：容器化服务器
    └── APIRemoteWorkspace：API管理运行时

Workspace(...) 工厂根据参数自动解析为本地或远程
→ Agent代码在不同环境中保持不变
```

**Agent Server**：用于远程Agent执行的API服务器：
- 暴露REST端点 + WebSocket（事件流）
- 每个Agent实例在独立Docker容器中运行
- 完整堆栈：API Server + VSCode Web + VNC Desktop + Chromium
- 支持水平扩展和多用户隔离

**Conversation工厂模式**：
```python
# 本地执行
conv = Conversation(workspace=LocalWorkspace("."))  # → LocalConversation

# 远程执行
conv = Conversation(workspace=RemoteWorkspace(server_url))  # → RemoteConversation

# API完全相同，从原型到部署无缝迁移
```

## 六、对OpenClaw的借鉴价值

### 可落地的想法

#### 💡 启发1：事件溯源状态管理

OpenHands的EventLog追加模式值得借鉴。当前OpenClaw的会话状态管理比较松散，如果引入事件溯源：
- 每个工具调用作为不可变事件记录
- 状态可完整重建和回放
- 会话恢复可靠性大幅提升

**实现路径**：设计`Event`基类 + `ConversationState`单一真值源 + 增量持久化。

#### 💡 启发2：MCP原生集成

OpenHands已将MCP视为一等公民，而OpenClaw目前主要通过HTTP直接调用。引入MCP协议层可以：
- 标准化工具定义
- 复用社区MCP Server生态
- 与Anthropic/OpenAI等厂商生态无缝对接

**实现路径**：参考OpenHands的`MCPToolDefinition` + `MCPToolExecutor`，将现有`web-search`等脚本封装为MCP Server。

#### 💡 启发3：多层部署架构（Local vs Remote Workspace）

OpenHands的Workspace抽象允许代码在本地和远程无缝切换。OpenClaw目前主要面向VPS部署，可以引入类似架构：
- `LocalWorkspace`：开发调试用
- `RemoteWorkspace`：生产运行用
- API兼容，本地测试→远程部署零迁移成本

#### 💡 启发4：Condenser上下文压缩

对于长时间运行的Agent任务（如楠楠的agent-study），Condenser可以将历史事件压缩为摘要，避免context窗口爆炸同时保留关键信息。

#### 💡 启发5：SecurityAnalyzer分级机制

OpenClaw在执行外部命令时可以引入类似的分级安全审查：
- 低风险操作（只读查询）直接执行
- 中高风险操作（文件修改、外部调用）触发确认
- 自适应策略（根据学长历史决策调整）
