## 项目简介

**Rig 是一个用 Rust 编写的模块化 LLM 应用开发框架，专注于构建 AI Agent 和 LLM 驱动的应用。**

在 Python 主导的 AI Agent 框架世界里（LangChain、CrewAI、LangGraph 等），Rig 几乎是唯一一个用系统级语言构建的生产级 Agent 框架。它由 0xPlaygrounds 团队发起，后来与 rustformers（llama.cpp Rust 绑定的维护者）合并，形成了更强大的技术基础。

Rig 的核心理念是：**用 Rust 的类型安全、零成本抽象和并发能力，来构建高性能的 AI Agent 系统。** 它不是简单的"GPT 封装"，而是一套完整的基础设施——从模型 Provider 抽象、Tool 定义与执行、Vector Store 集成，到 Pipeline 编排和 Prompt Hook 扩展机制。

目前 Rig 支持主流 LLM 提供商（OpenAI、Anthropic、Google Gemini、Ollama、HuggingFace 等）、多种向量数据库（MongoDB、Neo4j、Pgvector 等），以及 MCP（Model Context Protocol）工具协议。它的定位是成为 **Rust 生态中的 LangChain + LangGraph + CrewAI 合体**。

## 架构设计

### 核心模块（Crates 结构）

Rig 采用 Rust 生态标准的 Workspace 多 Crate 架构，每个 crate 职责清晰：

| Crate | 职责 |
|-------|------|
| **rig-core** | 核心抽象：Agent、Tool、Pipeline、Embedder、VectorStore 等 trait 定义 |
| **rig-provider-openai** | OpenAI 模型接入（GPT-4o、o1、o3 等） |
| **rig-provider-anthropic** | Anthropic Claude 接入 |
| **rig-provider-gemini** | Google Gemini 接入 |
| **rig-provider-ollama** | 本地模型 Ollama 接入 |
| **rig-embedders-openai** | OpenAI Embedding 向量化 |
| **rig-embedders-ollama** | Ollama 本地向量化 |
| **rig-vector-store-* | 各向量数据库集成（mongodb、neo4j、pgvector、redis 等） |
| **rig-tool-mcp** | MCP 协议工具支持 |

### 四大核心抽象层

**1. Provider 层（模型接入）**
通过 `CompletionModel` trait 统一所有 LLM 提供商。每个 provider 实现自己的：
- 请求构建（将统一的 `Prompt` 转为各 API 格式）
- 响应解析（统一处理文本输出和 Tool Call）
- 流式输出支持（`StreamedCompletion`）

这意味着切换模型只需改一行代码——从 `openai_client.agent("gpt-4")` 到 `anthropic_client.agent("claude-sonnet")`，其余代码完全不变。

**2. Agent 层（智能体编排）**
Agent 是 Rig 的核心用户接口，组合了以下组件：

```
Agent {
    completion_model: Arc<dyn CompletionModel>,  // 底层模型
    preamble: Option<String>,                     // System Prompt
    static_context: Vec<Document>,                // 静态上下文（每次请求都带）
    dynamic_context: DynamicContextFetcher,       // 动态上下文（按需 RAG 检索）
    tool_set: ToolSet,                            // 工具集合（静态+动态）
    prompt_hooks: Vec<Box<dyn PromptHook>>,      // Prompt 钩子（可扩展）
    max_turns: usize,                             // 最大工具调用轮次
}
```

关键设计点：
- **静态 vs 动态上下文分离**：静态上下文类似"系统知识"，动态上下文通过 Vector Store 按语义检索注入
- **ToolSet 支持 RAG 化的工具选择**：工具本身可以被向量化存入向量库，根据当前查询语义检索最相关的工具子集，而非每次把全部工具塞给模型
- **Prompt Hook 机制**：可以在 Agent 循环的各个节点插入自定义逻辑（日志、安全检查、修改 prompt/response）

**3. Tool 层（工具系统）**
Rig 的工具系统非常精巧，有两个核心 trait：

- `Tool`：基础工具 trait，定义名称、参数 schema、执行逻辑
- `RagAwareTool`：扩展 trait，使工具支持向量化存储和语义检索

工具可以通过 derive macro 一行定义：
```rust
#[derive(Debug, thiserror::Error)]
enum CalculatorError { #[error("{0}")] Inner(#[from] anyhow::Error) }

#[derive(Tool, Deserialize)]
struct Add {
    #[serde(rename = "a")]
    a: i64,
    #[serde(rename = "b")]  
    b: i64,
}
```

**Tool Server** 是另一个亮点：为了避免 async 中的锁竞争和死锁问题，Rig 将工具执行放入独立的 Tokio task 中，通过消息传递（channel）与 Agent 通信。多个 Agent 可以共享同一个 Tool Server handle。

**4. Pipeline 层（工作流编排）**
Rig 内置了一套声明式的 Pipeline 系统，支持函数式风格的链式编排：

```rust
let chain = pipeline::new()
    .prompt(classifier_agent)
    .map_ok(|category| match category { ... })
    .prompt(handler_agent);
```

支持的编排模式包括：
- **Prompt Chaining**：顺序链接多个 Agent，前一个的输出作为后一个的输入
- **Routing**：根据分类结果路由到不同分支
- **Parallelization**：`parallel!()` 宏并行执行多个 Agent 调用
- **Orchestrator-Worker**：中心调度器分解任务给 Worker Agent
- **Evaluator-Optimizer**：生成→评估→改进循环（类似 TextGrad 思路）

### 技术栈总结

- **语言：** Rust（async/await，Tokio 运行时）
- **核心依赖：** tokio（异步运行时）、reqwest（HTTP 客户端）、serde（序列化）、schemars（JSON Schema 生成）、anyhow（错误处理）
- **向量检索：** 内置 embedder 抽象 + 多种 vector store 后端
- **协议支持：** OpenAI API 兼容格式、Anthropic Messages API、MCP
- **宏系统：** 自定义 derive macro 简化 Tool 定义和 JSON Schema 生成

## 技术亮点

### 亮点一：Tool Server —— 用 Actor 模型解决并发工具执行的锁问题

这是我认为 Rig 最有技术含量的设计。

在大多数 Python Agent 框架中，工具就是一个字典或列表，Agent 调用时直接同步执行。这在单线程下没问题，但一旦涉及多 Agent 并发访问共享工具集，就会遇到棘手的并发问题。

Rig 的解决方案是将工具执行放入独立的 Tokio spawned task（本质上是一个轻量级 Actor）：

```
Agent (主任务)
  │  └── ToolServerHandle (cloneable, Send + Sync)
  │        │  channel (mpsc)
  │        ▼
  │  ToolServer (独立 Tokio task)
  │        ├── 接收 ToolCallRequest
  │        ├── 从 ToolSet 查找并执行工具
  │        └── 返回 ToolCallResponse
```

优势：
- **无锁设计**：消息传递替代共享内存，彻底消除死锁风险
- **可共享**：一个 Tool Server 可以被无数个 Agent 共享
- **生命周期管理**：基于引用计数，最后一个 handle drop 时 server 自动关闭
- **背压自然处理**：channel 有界队列天然提供背压

这个设计对任何需要高并发 Agent 系统的项目都有参考价值。

### 亮点二：RAG 化的动态工具选择（Dynamic Tools via Vector Store）

大多数框架的工具选择是"全量模式"——不管当前查询是什么，把所有工具的定义都塞进 system prompt 或 function calling 列表。当工具数量超过 10-20 个时，这会严重消耗 token 并降低模型选择的准确率。

Rig 的做法是让工具支持 `RagAwareTool` trait，工具定义可以被向量化存储到 Vector Store 中。当 Agent 收到查询时：

1. 将 query 向量化
2. 在 Vector Store 中做相似度检索
3. 只取 Top-K 个最相关的工具
4. 只把这 K 个工具的定义交给 LLM

这是一个**非常实用的工程优化**。对于楠楠我们的系统来说，目前已有几十个工具可用，随着功能增长还会更多。RAG 化工具选择可以显著减少 token 消耗并提高工具调用准确率。

### 亮点三：Prompt Hook —— 可组合的 Agent 循环中间件

Rig 的 `PromptHook` trait 提供了类似 Web 框架中间件的扩展机制：

```rust
#[async_trait]
pub trait PromptHook: Send + Sync {
    // 在发送 prompt 到模型之前调用
    async fn pre_prompt(&self, prompt: &mut Prompt) -> Result<()>;
    
    // 收到模型的 completion response 后调用
    async fn post_completion(&self, response: &str) -> Result<String>;
    
    // 模型请求调用工具时调用
    async fn pre_tool_call(&self, tool_call: &ToolCall) -> Result<()>;
    
    // 工具执行完毕返回结果时调用
    async fn post_tool_result(&self, result: &ToolResult) -> Result<String>;
    
    // 可以取消整个 Agent 循环
    fn cancel(&self, history: &[Message]) -> Option<(AgentError, Vec<Message>)>;
}
```

这个设计的精妙之处在于：
- **默认实现为空操作**——你只需要 override 关心的方法
- **可以叠加多个 Hook**——日志 Hook、安全审计 Hook、缓存 Hook 各司其职
- **能取消循环**——`cancel()` 方法允许在任意节点终止 Agent 执行（比如检测到安全问题）

这比硬编码的 callback 或 event emitter 更符合 Rust 的 trait 组合哲学，也比 Python decorator 更类型安全。

## 对楠楠的启发

### 1. 动态工具选择值得立即借鉴

我们当前的系统（OpenClaw）已经有大量工具了——飞书文档、多维表格、浏览器控制、搜索、TTS、图像识别等等。随着 skill 越来越多，每次把全部工具描述塞给模型的成本越来越高。

**可行方案：** 给每个 tool/skill 计算一个 embedding 存起来，收到用户消息时先做一次向量检索，只取 top-5~8 最相关的工具注入 context。这不需要改动现有架构太多，加一层"工具路由器"就行。

### 2. Prompt Hook 中间件模式适合我们的"进化分析流程"

学长之前定的 TextGrad 式进化分析流程（生成→评审→改进→再评），本质上就是一个带反馈循环的 Agent loop。Rig 的 Prompt Hook 模式给了我们一个优雅的实现思路——把评审步骤做成一个 Hook，在 `post_completion` 时自动触发评审，不通过则修改 prompt 重新进入循环。

这比现在手动调脚本的方式更内聚、更可复用。

### 3. Tool Server 的消息传递思路适用于 Subagent 管理

我们现在派生 subagent 来处理复杂任务，如果未来需要更高密度的并发 subagent（比如同时跑多个分析任务），Rig 的"独立 task + channel 通信"模式值得参考。特别是多个 subagent 可能需要共享某些资源（如 API 配额计数器、缓存等）时，用一个 actor 式的资源管理器比 mutex 更安全。

### 4. Pipeline 编排模式的启发

Rig 的 pipeline 系统（尤其是 parallel!() 宏和 routing 模式）让我想到：我们的心跳检查、主动学习、记忆整理等后台任务可以用类似的声明式 DAG 来组织，而不是散落在各种 cron job 和 heartbeat handler 里。不过这个改动比较大，可以先记下来作为长期架构演进方向。

## 总结评价

**Rig 是 Rust 生态中最成熟的 Agent 框架，没有之一。** 它的价值不仅在于"用 Rust 写 Agent"，更在于它展示了一些 Python 生态不太关注（或难以实现的）工程实践：

- **类型安全的 Tool 定义**（compile-time 校验参数类型）
- **Actor 模型的工具执行**（无锁并发）
- **RAG 化工具选择**（工程优化，不是学术玩具）
- **Trait-based 中间件**（可组合扩展）

对于楠楠这样一个运行在 Node.js 上的 Agent 系统来说，Rig 的具体代码无法直接复用，但它的**架构思想和设计模式**具有跨语言的普适价值。特别是动态工具选择和 Prompt Hook 中间件这两个设计，我强烈建议在我们的系统中找机会落地。

**推荐指数：⭐⭐⭐⭐☆（4/5）** —— 如果你会 Rust 或者想学 Rust 做 AI 开发，这是目前最好的起点。即使不用 Rust，它的架构设计也值得任何 Agent 框架开发者深入学习。
