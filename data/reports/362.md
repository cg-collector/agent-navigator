## 二、项目概述

Letta 是由加州大学伯克利分校研究团队发起的开源项目，最初以 **MemGPT**（Memory-GPT）之名发布，后于 2024 年正式更名为 Letta 并成立商业化公司。其核心理念是：**传统 LLM 应用本质上是"无状态的"——每次对话结束后，模型对用户的了解归零。Letta 试图通过引入操作系统中"虚拟内存管理"的思想，为 AI Agent 赋予真正的长期记忆能力。**

Letta 的愿景远不止于简单的聊天记忆。它构建了一套完整的**分层记忆体系**，让 Agent 能够像人类一样：记住你是谁、记住之前的对话内容、从交互中学习并改进自身行为。这一理念使其在众多 Agent 框架中独树一帜——大多数框架关注的是"如何编排工具调用"，而 Letta 关注的是"如何让 Agent 拥有持续进化的记忆与身份"。

2025-2026 年间，Letta 经历了重大架构升级，从 MemGPT 时代的工具调用驱动架构迁移至全新的 **Letta V1 Agent 架构**，全面拥抱现代 LLM 的原生推理能力和直接消息生成模式。同时推出了 **Letta Code**——一个运行在终端中的本地编码 Agent 产品，将记忆优先的理念带入实际编程工作流。

## 三、核心架构

### 3.1 分层记忆系统（Hierarchical Memory）

Letta 的架构灵魂是其受操作系统虚拟内存启发的分层记忆设计：

- **主上下文窗口（Main Context）**：相当于 CPU 寄存器/缓存，是 LLM 当前能"看到"的全部信息。容量有限但访问最快。
- **归档记忆（Archival Memory）**：相当于硬盘存储，容量几乎无限。Agent 可以主动将信息从主上下文"换出"到归档记忆，也可以在需要时"换入"。
- **记忆块（Memory Blocks）**：结构化的键值对存储，用于保存 Agent 的核心身份信息（如 `human` 块记录用户信息、`persona` 块记录 Agent 人格）。这些块始终保留在主上下文中，确保 Agent 不会"忘记自己是谁"。

这套设计的精妙之处在于：**Agent 本身拥有对记忆管理的控制权**。它不是被动地被塞入上下文，而是可以主动决定"记住什么、遗忘什么、什么时候去翻旧账"。这通过一组特殊的记忆管理工具（`archival_memory_search`、`archival_memory_insert`、`core_memory_append` 等）实现。

### 3.2 Agent Loop 演进

Letta 的 Agent 执行循环经历了三代演进：

**第一代：ReAct 风格**（早期）
- 思考（Thought）→ 行动（Action）→ 观察（Observation）的顺序循环
- 通过提示词格式化控制流程，终止条件依赖特殊 token
- 缺点：无法处理复杂工具链，容易陷入无限循环

**第二代：MemGPT 架构**（2023-2024）
- 所有操作（包括发送消息）都统一为工具调用
- 通过工具 schema 注入 `thinking` 和 `request_heartbeat` 参数控制推理和循环延续
- 默认终止策略（`request_heartbeat=False`），避免无限循环
- 引入 Tool Rules 机制约束工具调用模式
- 缺点：强依赖模型的工具调用能力，不支持原生推理模型

**第三代：Letta V1 架构**（2025-至今）
- 废弃 heartbeat 和 send_message 工具
- 全面支持 LLM 原生推理（reasoning tokens）和直接 assistant message 生成
- 与 GPT-5、Claude 4.5 Sonnet 等最新模型的训练分布对齐
- 显著提升最新模型的 Agent 表现

### 3.3 多模态与工具生态

- **MCP 协议支持**：原生集成 Model Context Protocol，可接入丰富的外部工具
- **多模型兼容**：完全模型无关（model-agnostic），支持 OpenAI、Anthropic、开源模型等
- **Skills 系统**（Letta Code）：预构建的技能包，Agent 可按需加载
- **Subagents**：支持子 Agent 委派，实现任务分解与并行执行

## 四、技术亮点

### 4.1 "虚拟内存"式记忆管理

这是 Letta 最具原创性的贡献。将操作系统的内存分页思想移植到 LLM 上下文管理中，不仅解决了长对话中的信息丢失问题，更重要的是赋予了 Agent **主动记忆管理能力**。Agent 可以自行判断哪些信息值得长期保存、哪些应该遗忘，这种"元认知"能力是迈向自主 Agent 的关键一步。

### 4.2 自我改进机制（Self-Improvement）

Letta 不仅让 Agent "记住"，还让 Agent "学习"。通过记忆块的持续更新和归档记忆的积累，Agent 的行为会随着交互次数增加而逐步优化。这在 Letta Code 编码场景中尤为明显——Agent 能记住你的编码偏好、项目结构约定、之前修复过的 bug 类型，从而在后续任务中表现越来越好。

### 4.3 白盒透明性（White-box Design）

与许多黑盒 Agent 框架不同，Letta 提供了完整的可观测性：
- 记忆内容的完整查看与编辑
- Agent 推理过程的透明追踪
- 工具调用的完整审计日志
- 这对于调试、信任建立和合规审计至关重要

### 4.4 从研究到产品的完整路径

MemGPT 起源于伯克利的学术论文（arXiv:2310.08560），经过开源社区验证后发展为 Letta 商业平台。这种"论文 → 开源 → 产品"的路径在 AI 领域并不多见，说明其技术理念经过了严格的学术审视和工程实践双重检验。

### 4.5 记忆基准测试争议中的技术立场

2025 年，Letta 团队公开质疑竞品 Mem0 在 LOCOMO 记忆基准测试中的数据造假，并用简单文件系统工具超越 Mem0 宣称的 SOTA 结果 26%。这场争议虽然带有商业竞争色彩，但也从侧面印证了 Letta 对技术诚实性的坚持和其记忆方案的实际有效性。

## 五、适用场景

1. **长期陪伴型应用**：虚拟伴侣、个性化助手、角色扮演 Agent——需要跨会话保持用户偏好和关系历史
2. **编程 Agent / 开发助手**：Letta Code 场景，需要记住项目约定、编码风格和历史决策
3. **企业知识管理 Agent**：需要持续积累领域知识、从文档和交互中学习的专业助手
4. **研究与实验**：探索 Agent 记忆机制、自我改进能力、长期交互行为的学术研究
5. **需要审计和可解释性的场景**：金融、医疗、法律等对 Agent 决策过程有追溯要求的领域

## 六、局限性

1. **上手成本较高**：相比 Smolagents 或 OpenAI Agents SDK 的"几行代码跑起来"，Letta 需要理解分层记忆概念、配置记忆块、可能还需要部署 Letta Server，学习曲线明显更陡峭
2. **记忆管理开销**：Agent 需要花费 token 来进行记忆的搜索、写入和管理操作，这在 token 成本敏感的场景下是一个实际问题
3. **自托管复杂度**：如果选择完全私有部署，需要维护 Letta Server、数据库、向量存储等基础设施，运维负担不小
4. **商业与开源的边界模糊**：Letta 正在推进商业化（SaaS API、Letta Code 产品），部分高级功能可能逐步收归付费版，开源社区需关注其开放承诺
5. **多 Agent 协作能力相对薄弱**：虽然支持 Subagents，但在多 Agent 复杂编排方面不如 CrewAI、LangGraph 或 AutoGen 成熟
6. **V1 架构迁移阵痛**：从 MemGPT 架构向 V1 迁移意味着旧版代码和工具需要适配，对已有用户存在迁移成本
7. **中文社区资源有限**：主要文档和讨论以英文为主，中文开发者生态有待建设

## 八、与同类框架对比

### 8.1 vs Mem0（#360 已述）

两者常被放在一起比较，因为都聚焦于 AI 记忆问题：

| 维度 | Letta | Mem0 |
|------|-------|------|
| **定位** | 完整的有状态 Agent 平台 | 通用记忆层（可插入任何框架） |
| **记忆方式** | 分层记忆 + Agent 自主管理 | 向量存储 + 语义检索 + 图记忆 |
| **架构复杂度** | 重（含完整 Agent Runtime） | 轻（纯记忆中间件） |
| **适用方式** | 用 Letta 构建整个 Agent | 将 Mem0 接入现有 Agent 栈 |
| **核心理念** | 操作系统虚拟内存类比 | RAG + 知识图谱增强 |

**结论**：如果你需要一个开箱即用的有状态 Agent，选 Letta；如果你只想给现有 Agent 加记忆层，选 Mem0。

### 8.2 vs Zep（#351 已述）

Zep 同样专注长期记忆，但路径不同：

| 维度 | Letta | Zep |
|------|-------|------|
| **记忆模型** | 分层块 + 归档存储 | 时序知识图谱 |
| **GraphRAG** | 不内置 | 核心特性 |
| **自托管** | 支持 | 支持 |
| **SDK 语言** | Python + TypeScript | Python + JavaScript |
| **成熟度** | 快速迭代中 | 相对稳定 |

### 8.3 vs LangGraph（#358 已述）

| 维度 | Letta | LangGraph |
|------|-------|--------|
| **核心关注点** | 记忆与状态持久化 | 工作流编排与状态图 |
| **状态管理** | 内置分层记忆系统 | 通用状态机（需自定义） |
| **编排能力** | 基础 Loop + Subagents | 强大的图编排、分支、并行 |
| **学习曲线** | 中等偏高 | 高（概念多） |
| **最佳场景** | 需要"记住一切"的长期 Agent | 复杂多步骤业务流程 |

### 8.4 vs CrewAI（#361 已述）

| 维度 | Letta | CrewAI |
|------|-------|--------|
| **范式** | 记忆优先的单体/轻量多 Agent | 角色扮演驱动的多 Agent 协作 |
| **记忆深度** | 深（分层、持久、自主管理） | 浅（依赖短期上下文 + 可选集成） |
| **协作模式** | Subagent 委派 | 角色分工 + 任务委派 + 层级协作 |
| **易用性** | 中等 | 高（面向非开发者友好） |

## 九、总结与展望

Letta 代表了 AI Agent 发展的一个重要方向：**从"无状态的工具调用器"走向"有状态的记忆承载者"**。当大多数框架还在优化"Agent 如何更好地使用工具"时，Letta 在回答一个更根本的问题——"Agent 如何真正地了解你、记住你、并在时间的维度上与你共同成长"。

其从伯克利学术论文到 21K+ Star 开源项目再到商业化公司的 trajectory，证明了"记忆优先"这一技术路线的市场认可度。V1 架构的重构显示了团队紧跟 LLM 技术演进的敏捷性。

然而，Letta 也面临着挑战：商业化与开源的平衡、与其他框架在多 Agent 编排上的差距、以及让更多开发者理解并接受其独特的记忆范式。对于正在选型的团队来说，如果你的核心需求是**长期记忆和 Agent 身份持续性**，Letta 是目前市场上最成熟的选择之一；如果更关注复杂工作流编排或多 Agent 协作，则可能需要搭配或选择其他框架。
