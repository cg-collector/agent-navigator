## 一、项目背景与问题定义

### 1.1 从"调用 API"到"操作计算机"：Agent 范式的根本转变

2024-2025 年间，AI 领域出现了一个关键认知转折：**LLM 的能力不应局限于文本生成和函数调用——它应该能够像人类一样直接操作计算机**。这个洞察催生了"Computer Use"这一全新的 Agent 交互范式。

传统 Agent 开发面临的核心困境是：
- **API 碎片化**：每个软件系统都有不同的 API，为每个应用编写集成代码成本极高
- **遗留系统无法接入**：大量企业软件只有 GUI 没有 API
- **Web 界面变化频繁**：网站 DOM 结构随时可能改变，基于选择器的自动化极其脆弱
- **跨应用工作流难以编排**：一个真实任务通常涉及浏览器、桌面应用、文件系统的协同操作

OpenAI 的回答是：**让模型"看"屏幕截图，然后像人一样点击、打字、滚动**。这就是 Computer Use Agent（CUA）的核心思想。

### 1.2 OpenAI 的双线布局：Operator 产品 + CUA API

OpenAI 在 Computer Use 方向采取了"产品+平台"的双线策略：

- **Operator（ChatGPT 内置）**：面向终端用户的 Computer Use 产品，$200/月订阅制，用户用自然语言描述任务，Operator 在独立浏览器环境中自动完成
- **CUA API（Responses API 集成）**：面向开发者的 `computer_use_preview` 工具类型，允许任何应用通过 API 调用获得 GUI 自动化能力

这种设计使 OpenAI 同时覆盖了消费级和企业级市场，也使得 CUA 成为目前**商业化程度最高、工程化最完整**的 Computer Use 解决方案之一。

### 1.3 为什么不是 Anthropic Computer Use？

Anthropic 在 2024 年率先推出了 Claude Computer Use（报告 #201 已详细分析），但两者的技术路线有本质差异：
- **Anthropic**：模型直接控制用户的真实桌面环境（Docker + Xvfb 虚拟显示），强调"给 Agent 一台真正的计算机"
- **OpenAI**：模型在托管容器/浏览器沙箱中运行，强调**安全隔离、托管执行、可控输出**

OpenAI 的方案更适合企业部署——开发者不需要自己管理 Docker 容器和虚拟显示环境，所有执行都在 OpenAI 托管的基础设施中进行。

### 2.2 CUA Loop：截图驱动的感知-决策-行动循环

CUA 的核心是一个紧密的反馈循环，每次迭代的流程如下：

```
步骤 1: 【感知】执行环境捕获当前屏幕截图（PNG，1024x768 标准分辨率）
         ↓
步骤 2: 【输入】截图 + 任务目标 + 历史动作记录 → 发送给 computer-use-preview 模型
         ↓
步骤 3: 【决策】模型分析截图内容，理解当前 UI 状态，决定下一步动作
         ↓
步骤 4: 【输出】模型返回结构化的 action 对象（JSON 格式）
         ↓
步骤 5: 【执行】Playwright 引擎将 action 翻译为实际的浏览器操作
         ↓
步骤 6: 【验证】再次截图确认操作结果 → 回到步骤 1
```

**关键设计决策：**
- **固定分辨率 1024x768**：这是 XGA 标准，确保模型训练和推理时的视觉一致性。更高分辨率会降低坐标精度和增加 token 消耗
- **结构化 action 输出**：模型不输出自由文本，而是返回预定义 schema 的 JSON 对象，包含 action type 和参数
- **隐式终止条件**：模型自行判断何时返回 `done` action，不需要外部循环控制器预设步数上限

### 2.3 Action Space 设计：七种原子操作的完备性论证

CUA 定义了七个原子操作类型，构成了 GUI 交互的最小完备集：

| Action Type | 参数 | 说明 |
|-------------|------|------|
| **`click`** | `x`, `y`, `button` | 在指定坐标点击（支持 left/right/middle） |
| **`type`** | `text` | 在当前焦点元素输入文本 |
| **`scroll`** | `direction`, `amount` | 向上/下/左/右滚动指定量 |
| **`key_press`** | `key` | 按下特殊键（Enter、Tab、Escape 等） |
| **`wait`** | `ms` | 等待指定毫秒数（页面加载等场景） |
| **`screenshot`** | 无 | 显式请求新的屏幕截图 |
| **`done`** | `output` | 宣布任务完成并返回结果 |

**为什么这七种操作是完备的？** 任何 GUI 操作都可以分解为以上原子的组合：
- 点击链接 = `click(x, y)`
- 填写表单 = `click`（聚焦）+ `type`（输入）
- 提交表单 = `type` + `key_press("Enter")`
- 下拉菜单 = `click` + `scroll` + `click`
- 切换标签页 = `key_press("Ctrl+Tab")` 或 `click` 标签栏

这种设计借鉴了 **GUI Testing** 领域的"最小动作集"理论——Selenium、Playwright 等工具都采用类似的原子操作抽象。

### 2.4 执行环境：托管容器 vs 本地 Playwright

CUA 支持两种执行模式：

**模式 A：托管浏览器环境（推荐）**
```python
tools=[{
    "type": "computer_use_preview",
    "display_width": 1024,
    "display_height": 768,
    "environment": "browser"  # OpenAI 托管的 Chromium 实例
}]
```
- 浏览器运行在 OpenAI 的服务器端
- 开发者无需安装任何浏览器或自动化工具
- 所有网络流量经过 OpenAI 的 sidecar egress proxy
- 适合纯 Web 任务

**模式 B：本地 Playwright 环境**
```python
tools=[{
    "type": "computer_use_preview",
    "display_width": 1024,
    "display_height": 768,
    "environment": "local-playwright",  # 连接本地 Playwright
    "computer_url": "ws://localhost:xxxx"
}]
```
- 使用本地安装的 Playwright + Chromium
- 可以控制本地桌面应用（通过 `environment: "mac"` / `"windows"` / `"ubuntu"`）
- 需要开发者自行管理安全沙箱
- 适合需要访问内部系统或桌面应用的场景

### 2.5 安全架构：三层防护体系

OpenAI 为 CUA 设计了三层安全机制，这在同类产品中最为完善：

**第一层：容器沙箱隔离**
- 每个 CUA 会话运行在独立的 Docker 容器中
- 容器内文件系统与宿主机完全隔离
- 进程权限受限，无法访问宿主机资源
- 支持 `mac`、`windows`、`ubuntu` 三种操作系统环境

**第二层：Sidecar Egress Proxy（侧车出口代理）**
- 所有容器的出站网络请求必须通过集中式策略代理
- 支持 allowlist 白名单机制：只允许访问预批准的域名
- 流量可审计、可监控
- 防止数据外泄到未授权的外部服务

**第三层：域作用域密钥注入（Domain-Scoped Secret Injection）**
- API 密钥等敏感凭证不在容器内明文存储
- 模型和容器只能看到凭证占位符
- 实际密钥值在 egress 代理处注入，且仅对白名单域名生效
- 即使模型被 prompt injection 攻击也无法读取原始密钥

这套安全体系解决了 Computer Use 最核心的安全焦虑：**如果 Agent 被诱导执行恶意操作怎么办？**

## 三、技术亮点

### 亮点一：Screenshot-Based GUI Grounding —— 视觉驱动的界面理解

**原理说明：**

CUA 的核心技术假设是：**现代多模态大模型已经具备了足够的视觉理解能力来解析 GUI 截图**。这意味着 Agent 不需要 DOM 树、可访问性树（accessibility tree）、或任何中间表示层——它直接"看"屏幕就能理解界面。

这与传统的 GUI 自动化形成鲜明对比：

```
传统 Selenium/Playwright:
  HTML/DOM → CSS Selector/XPath → 元素定位 → 操作
  ↑ 依赖页面结构，脆弱

CUA Screenshot-Based:
  PNG Screenshot → Vision Model → 坐标 + 动作类型 → 操作
  ↑ 只依赖视觉，对底层实现无关
```

**技术细节：**
- 输入：标准 PNG 截图（1024x768），作为图像 token 送入 vision encoder
- 模型需要同时完成两个任务：(1) 理解截图中显示了什么（视觉理解）(2) 决定在哪里点击/输入什么（空间推理）
- 训练时使用了大量人类操作计算机的轨迹数据（screenshots + actions pairs），让模型学会"看屏→操作"的映射关系
- 关键挑战是**像素级坐标精度**——OpenAI 在训练中特别强化了模型的"数像素"能力（pixel-counting），因为 GUI 操作对坐标精度要求极高（按钮可能只有 20x20 像素）

**为什么这种方式具有泛化优势？**
- 同一套模型可以操作 Web 应用、桌面应用、移动模拟器——只要能截图就能操作
- 不受前端框架（React/Vue/Angular）或 UI 工具包（Qt/GTK/Electron）的影响
- 即使网站改版导致 DOM 结构完全变化，只要视觉效果相似，Agent 仍能正常工作

### 亮点二：Context Compaction（上下文压缩）—— 长循环任务的内存管理

**原理说明：**

Computer Use 任务的一个固有问题是**上下文爆炸**：一个复杂任务可能需要几十甚至上百次截图-动作循环，每轮循环都要把之前的截图和动作历史加入上下文窗口。很快，context window 就会被填满。

OpenAI 的解决方案是 **Native Context Compaction**（原生上下文压缩）：

```
普通 Agent 循环的问题:
Turn 1: [用户指令] + [截图1] + [动作1]        = ~2000 tokens
Turn 2: 以上 + [截图2] + [动作2]               = ~4000 tokens
...
Turn 20: 以上 + [截图20] + [动作20]             = ~40000 tokens ← 接近上限!
Turn 21: 💥 Context Window Overflow

Compaction 解决方案:
Turn 1-10: 正常累积                               = ~20000 tokens
         ↓ 触发 compaction 阈值
Turn 11: [Compaction Item: 加密的紧凑状态摘要]      = ~3000 tokens
         + [最近几轮的高价值上下文]
Turn 11-20: 继续从压缩后的状态开始...               = 可持续运行
```

**关键技术特点：**
1. **模型原生训练**：compaction 不是简单的文本摘要，而是使用专门的加密 token-efficient 表示。GPT-5.2 及后续模型在训练阶段就被教会了如何产生高质量的 compaction item
2. **服务器端自动触发**：可以配置阈值，系统自动检测何时需要压缩，无需客户端维护复杂的逻辑
3. **保留高价值信息**：压缩后保留对话的开头（用户原始指令）和最近的几轮（最新的操作上下文），中间部分被压缩
4. **Codex 验证**：这套 compaction 系统首先在 Codex（OpenAI 的编程 Agent）中得到验证——Codex 的长编程任务经常需要数百轮 tool call，compaction 使其能够在不降低质量的情况下持续运行

这使得 CUA 理论上可以运行**任意长度**的任务——从简单的单步查询到需要数小时的多步骤工作流。

### 亮点三：并发 Shell 执行 + 输出封顶 —— Agent 循环的性能优化

**原理说明：**

虽然 CUA 主要面向 GUI 操作，但 OpenAI 的 Responses API 还引入了一个更通用的 **Shell Tool** 能力（与 Computer Use 并列的另一个内置工具）。Shell Tool 允许模型直接执行 Unix shell 命令，其设计中包含两个精巧的性能优化：

**优化一：并发命令执行**
```
串行执行（慢）:
grep "error" log.txt → 等待 → curl api/data → 等待 → python analyze.py → 等待
总耗时 = t1 + t2 + t3

并发执行（快）:
Session A: grep "error" log.txt  ─┐
Session B: curl api/data         ├→ 多路复用 → 合并输出
Session C: python analyze.py     ─┘
总耗时 ≈ max(t1, t2, t3)
```
模型可以在一次响应中提出多个独立的 shell 命令，Responses API 会在不同的容器 session 中并行执行它们，然后流式地将各路输出合并返回给模型。

**优化二：输出封顶（Output Capping）**
```
问题: curl 一个大 API 返回 50KB JSON → 全部塞进 context → 浪费 tokens 且淹没重要信息

解决: 模型指定 output_cap=1000
结果: "前500字符 ... [1000字符截断] ... 后500字符"
      ↑ 保留首尾，标记省略部分
```
模型可以为每个命令指定输出大小上限。API 强制执行该限制，同时保留输出的开头和结尾（通常包含最重要的信息），中间部分被截断并标记。这防止了大输出"污染"context window 的问题。

这两个优化组合起来，使得 Agent 循环既**快速**（并行执行独立子任务）又**经济**（不会因大输出浪费 context budget）。

## 四、基准测试表现

### 4.1 OSWorld Benchmark（核心评测基准）

OSWorld 是由香港大学团队开发的 Computer Use 领域最具权威性的基准测试，包含 369 个真实计算机任务（Web 应用、桌面软件、跨应用工作流、文件 I/O 等）。以下是截至 2026 年初的主要成绩：

| 智能体/模型 | OSWorld 得分 | 说明 |
|------------|-------------|------|
| **人类表现** | **72.36%** | 人工操作基线 |
| **Claude Opus 4.5+Computer Use** | **~72.5%** | 当前最强，接近人类水平 |
| **Coasty** | **~82%** | 专用 Computer Use 引擎（非通用模型） |
| **SimularAI Agent S3** | **~69.9%** | 接近人类水平 |
| **Qwen2.5-VL-72B** | **~25-30%** | 开源最佳之一 |
| **OpenAI CUA (computer-use-preview)** | **~32.6%-38.1%** | OpenAI 官方模型 |
| **GPT-4o**（早期版本） | **~15-20%** | 2024 年底水平 |
| **早期 VLM 基线** | **<12%** | OSWorld 论文原始数据 |

**数据分析：**
- OpenAI CUA 的 ~32-38% 得分处于行业中游水平，显著落后于 Claude Opus 4.5 的 ~72.5%
- 但需要注意的是，CUA 是**通用模型**（同一个 GPT-4o 衍生模型），而 Coasty/SimularAI 是**专用优化的 Computer Use 引擎**
- 从 2024 年底的 <15% 到 2026 年初的 ~38%，进步速度很快
- Operator 作为产品形态（$200/月）在某些特定任务上的实际体验可能优于裸 benchmark 分数

### 4.2 其他基准表现

| 基准 | CUA 表现 | 说明 |
|------|---------|------|
| **WebArena** | 中等偏上 | Web 导航和表单填写任务 |
| **WebVoyager** | 较好 | 信息检索类任务 |
| **MiniWoB++** | 优秀 | 简单 GUI 任务（但此基准已被认为过于简单） |

### 4.3 实际用户体验指标

根据社区反馈和第三方评测（如 Coasty.ai 的对比评测）：
- **简单任务成功率**（单步查询、导航）：70-85%
- **中等任务成功率**（多步表单、跨页面操作）：40-60%
- **复杂任务成功率**（多应用协作、异常处理）：15-30%
- **平均任务完成时间**：简单任务 30s-2min，复杂任务 5-15min
- **每步动作成本**：约 $0.01-0.05（取决于截图大小和模型版本）

## 五、与同类项目横向对比

### 5.1 三大 Computer Use 方案对比

| 维度 | **OpenAI CUA** | **Anthropic Computer Use** | **Browser-Use (开源)** |
|------|---------------|---------------------------|------------------------|
| **核心范式** | 截图 + 结构化 Action | 截图 + 危险动作列表 | 截图 + 元素提取 + LLM |
| **执行环境** | 托管容器 / 本地 Playwright | Docker + Xvfb 虚拟显示 | 本地 Playwright |
| **模型要求** | computer-use-preview（专用微调） | Claude 3.5 Sonnet+ | 任意 VLM（可切换） |
| **Action Space** | 7 种原子操作（固定 schema） | 类似 + 危险动作分级 | 可自定义扩展 |

### 5.3 差异化定位总结

OpenAI CUA 的独特价值主张在于：
1. **最低门槛**：不需要管理基础设施，一行 API 调用即可获得 Computer Use 能力
2. **最完善的安全模型**：三层防护在企业场景中无可替代
3. **原生集成**：与 Responses API 的其他工具（Web Search、File Search、Shell）无缝组合
4. **上下文压缩**：唯一内置 native compaction 的 Computer Use 方案

其短板也很明显：
- **Benchmark 表现不及 Claude Computer Use**：OSWorld 上差距接近一倍
- **闭源黑盒**：无法针对特定场景微调模型
- **灵活性有限**：固定的 action space 无法扩展自定义操作
- **成本较高**：Operator $200/月 + API 按量计费

## 六、对 OpenClaw / 我们的实际启发

### 启发一：截图驱动交互的普适性价值

OpenClaw 当前的 browser 工具主要依赖 **DOM 选择器 + aria ref** 来定位页面元素。这种方式在结构良好的网页上非常精确，但在以下场景会遇到困难：
- SPA 应用动态渲染的内容（ref 经常失效）
- Canvas/WebGL 渲染的界面（没有 DOM 元素可定位）
- 非 Web 的桌面应用（完全无法用 selector 定位）

可以考虑引入 **screenshot-based fallback**：当 snapshot/selector 失败时，降级为截图→VLM 分析→坐标点击的模式。我们的智谱 GLM-4V-Flash 图像识别能力正好可以用作 screenshot analyzer。

### 启发二：结构化 Action Space 的可靠性保障

CUA 将所有 GUI 操作约束在 7 种原子操作内，并通过 JSON Schema 强制校验输出格式。这种"受限输出空间"的设计显著提高了 Agent 操作的可靠性——模型不可能产生"非法动作"。

OpenClaw 的 browser act 目前接受自由格式的 request 对象（kind/ref/text 等），虽然灵活但也更容易出错。可以考虑为高频操作定义预校验的 action schema，减少格式错误导致的执行失败。

### 启发三：Context Compaction 思想用于长 Session

OpenClaw 的主会话在长时间对话后也会遇到 context 窗口膨胀问题。虽然我们不像 CUA 那样每轮都加入大尺寸截图，但心跳、多次 tool call、长报告输出等都会消耗 context。

可以借鉴的轻量方案：
- 定期将早期对话的关键信息压缩为摘要存储在 memory 文件中
- 主会话 context 中只保留最近 N 轮 + 压缩摘要
- 这比升级到更大的 context window 更经济

### 启发四：安全分层模型值得学习

OpenAI 的三层安全（容器隔离→网络代理→密钥注入）对 OpenClaw 有直接参考意义：
- 我们的 exec 工具已经有 security mode 和 elevated 权限控制
- 但对于 sub-agent 的网络访问、文件系统访问还缺乏细粒度策略
- 特别是当 sub-agent 被派去执行外部搜索和网页抓取时，egress control 可以防止意外数据泄露

### 启发五："托管执行"模式的商业启示

OpenAI 将 Computer Use 包装为托管服务（开发者只需调 API，不用管环境），大大降低了使用门槛。这提示我们：**Agent 能力的封装方向应该是"声明意图→自动执行"，而不是"提供工具→手动编排"。**

例如，当前的飞书文档操作需要先读 SKILL.md、再调 feishu_doc API、再处理错误。如果能封装成"帮我把这个分析写入飞书文档"的一步声明式操作，使用体验会大幅提升。

## 总结评价

OpenAI Computer Use Agent（CUA）代表了 **"GUI 自动化即服务"（GUI Automation as a Service）** 这一新兴范式的最成熟商业化尝试。它没有追求最高的 benchmark 分数，而是聚焦于**最低的使用门槛、最完善的安全模型、最深度的平台集成**这三者之间的平衡。

**核心优势：**
- **零基础设施启动**：一行 API 调用获得完整的 Computer Use 能力，不需要 Docker、Xvfb、Playwright 配置
- **三层安全体系**：容器隔离 + Egress Proxy + 密钥注入，是企业部署的关键保障
- **Native Context Compaction**：唯一内置上下文压缩的 Computer Use 方案，支持真正意义上的长任务
- **与 Responses API 生态深度整合**：Web Search、File Search、Shell、Computer Use 四大工具可无缝组合
- **标准化 Action Space**：7 种原子操作的固定 schema 保证了输出可靠性

**不足之处：**
- **Benchmark 表现中游**：OSWorld ~32-38%，显著落后于 Claude Opus 4.5 的 ~72.5%
- **闭源不可定制**：无法针对特定场景微调模型或扩展 action space
- **灵活性受限**：固定范式不适合需要特殊交互方式的场景
- **成本偏高**：Operator $200/月 + API 按量计费，大规模使用成本可观
- **环境局限**：托管模式仅支持浏览器，本地模式需要额外配置

**评分：7.8/10** — 扣分主要在 benchmark 表现和闭源限制。但在**"开箱即用的 GUI 自动化"**这个细分领域，CUA 的**工程完整性**和**安全设计**目前无人能及。它证明了 Computer Use 不只是学术玩具——它可以被包装成可靠的企业级服务。

**一句话总结：CUA 可能不是最强的 Computer Use Agent，但它是最容易上手、最安全、最适合生产环境的那个——OpenAI 用"服务化"思维重新定义了这个赛道。**
