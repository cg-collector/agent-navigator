# 📚 AI Agent 项目研究报告 #246 — Notte：面向生产环境的全栈 Web Agent 框架

## 一、背景与问题

随着 LLM 驱动的 Browser Agent 蓬勃发展（Browser-Use、Stagehand、Playwright + LLM 等方案层出不穷），一个核心矛盾日益凸显：**Agent 的探索能力与生产可靠性不可兼得**。

具体而言，当前 Web Agent 领域面临三大痛点：

1. **非确定性导致难以部署**：纯 LLM 驱动的浏览器操作是概率性的——每次执行可能选择不同的按钮、走不同的路径。在生产环境中这意味着不可复现的 bug 和无法预估的延迟。
2. **DOM 解析认知负载过高**：原始 DOM 树动辄数万个节点，直接喂给 LLM 既浪费 token 又容易让模型"迷路"。现有方案要么丢失语义（CSS selector），要么信息过载（完整 HTML）。
3. **从原型到生产的鸿沟**：大多数"prompt-to-automation"工具只能生成一次性执行的 Agent 工作流，无法转化为可维护、可调试的确定性代码。

**Notte**（由 Y Combinator S25 孵化的 nottelabs 团队开发）正是为解决这些问题而生的全栈 Web Agent 框架。它提出了一套"AI 探索 + 代码固化"的混合范式，在 Open Operator 基准测试中以 86.2% 的成功率大幅领先 Browser-Use（77.3%）和 Convergence（38.4%）。

## 二、核心架构

Notte 采用三层架构设计，自底向上分别为：

```
┌─────────────────────────────────────────────┐
│              Agent Layer (智能层)             │
│  Agent.run(task) → LLM 推理 → 动作决策        │
│  支持: Gemini/GPT/Claude + 结构化输出         │
├─────────────────────────────────────────────┤
│           Perception Layer (感知层) ⭐          │
│  DOM → 结构化自然语言地图 → 降维后的元素描述     │
│  核心: observe() / act() / extract()          │
├─────────────────────────────────────────────┤
│           Session Layer (基础设施层)           │
│  浏览器实例管理 / 代理 / Cookie / CDP / 录屏   │
│  基于 Patchright (反检测 Playwright fork)     │
└─────────────────────────────────────────────┘
```

### 三大组件：

- **`notte.Session`**：按需创建 headless/headful 浏览器实例，内置代理配置、CDP 协议支持、Cookie 集成和会话回放（Session Replay）。底层使用 Patchright（Playwright 的反检测分支），支持 Chrome/Firefox 等多种浏览器类型。
- **`notte.Vault`**：企业级凭证安全管理，与 Session 和 Agent 解耦，支持多站点凭据注入。
- **`notte.Agent`**：LLM 驱动的智能体执行引擎，支持配置推理模型、最大步数、结构化输出格式（Pydantic model），以及 `response_format` 约束返回结果。

## 三、技术亮点

### 亮点 1：感知层（Perception Layer）— DOM 到自然语言地图的结构化转换

这是 Notte 最核心的创新。传统 Browser Agent 要么将完整 DOM/HTML 塞给 LLM（token 爆炸、噪声极大），要么用 CSS/XPath selector（丢失语义、脆弱易碎）。

Notte 的感知层对 DOM 做**语义压缩**：
1. **元素过滤**：剔除不可见元素（`display:none`、`visibility:hidden`、零尺寸）、脚本标签、样式块等无交互价值的节点。
2. **语义标注**：将保留的交互元素转换为自然语言描述，例如 `<button id="submit" class="btn-primary">Submit Order</button>` → `"Submit Order button (primary action)"`。
3. **空间关系保留**：维护元素在页面中的相对位置和层级关系，使 LLM 能理解布局上下文。
4. **动态裁剪**：根据任务类型动态调整感知范围——搜索任务聚焦输入框和列表，表单任务聚焦 input 和 submit 按钮。

**原理**：这一过程类似于计算机视觉中的"感兴趣区域（ROI）提取"，只不过作用对象是 DOM 树而非像素。效果是将数万节点的 DOM 压缩到几十到几百个有意义的交互元素描述，LLM 的认知负载降低 1-2 个数量级，同时动作准确率显著提升。这也意味着可以使用更小更快的模型（如 Gemini 2.0 Flash）达到相同甚至更好的效果。

### 亮点 2：Agent Mode → Deterministic Script 的"探索-固化"工作流

Notte 提出了一个独特的工作流范式，解决了 Agent 从原型到生产的转化问题：

```
Step 1: Agent Mode（探索阶段）
   用户输入自然语言任务 → Agent 在真实浏览器中执行
   → LLM 通过感知层理解页面 → 自主决策每一步操作
   → 用户实时观察执行过程
   
Step 2: Convert to Script（固化阶段）
   将 Agent 的执行轨迹转化为确定性 Python/TypeScript 代码
   → 包含精确的 selector、等待条件、错误处理
   → 代码可编辑、可版本控制、可集成 CI/CD
   
Step 3: Refine & Scale（优化阶段）
   开发者审查生成的代码 → 优化 selector 策略
   → 添加业务逻辑 → 部署到生产环境
```

**原理**：这本质上是**程序综合（Program Synthesis）+ 执行追踪（Execution Tracing）**的结合。Agent Mode 中，系统记录每一步的：触发元素（XPath/CSS）、动作类型（click/type/select）、等待条件（element visible/text match）、页面状态快照。Convert to Script 时，将这些追踪记录编译为 Patchright/Playwright 的确定性 API 调用。关键洞察：LLM 擅长"探索未知路径"，而确定性代码擅长"重复已知路径"。两者各司其职。

### 亮点 3：Demonstrate Mode — 示范即代码（Show, Don't Prompt）

Notte 还提供了 Demonstrate Mode：用户在浏览器中手动执行一次操作流程，系统自动录制并生成自动化代码。

**原理**：通过 CDP（Chrome DevTools Protocol）监听浏览器的底层事件流——`Input.dispatchMouseEvent`、`Input.insertText`、`Page.frameNavigated` 等，结合感知层的 DOM 快照，将低级输入事件映射为高级语义动作（"点击提交按钮"而非"在坐标(342, 567)处 mouseDown/mouseUp"）。相比传统的宏录制工具，Notte 的优势在于它生成的不是坐标级脚本而是语义级代码，因此对 UI 变化具有天然的鲁棒性。

## 四、Benchmark 数据

Notte 团队在 [Open Operator Eval](https://github.com/nottelabs/open-operator-evals) 基准测试中发布了对比数据：

| 框架 | Agent 自报成功率 | LLM 评估成功率 | 平均任务耗时 | 任务可靠性 |
|------|------------------|----------------|-------------|-----------|
| **Notte** 🏆 | **86.2%** | **79.0%** | **47s** | **96.6%** |
| Browser-Use | 77.3% | 60.2% | 113s | 83.3% |
| Convergence | 38.4% | 31.4% | 83s | 50.0% |

关键发现：
- Notte 的**速度是 Browser-Use 的 2.4 倍**（47s vs 113s），主要归功于感知层降低了 LLM 需要处理的上下文量
- **任务可靠性 96.6%** 意味着同一任务重复执行时结果高度一致，这对生产部署至关重要
- LLM 评估成功率（79.0%）与自报成功率（86.2%）之间的差距小于竞品，说明 Notte 的自我评估更诚实/准确

## 六、启发与思考

### 对 GUI Agent 设计的启示

1. **感知层是最大的杠杆点**：Notte 的数据表明，如何在将页面信息呈现给 LLM 之前做预处理，比选择更强的 LLM 模型更能决定最终性能。OS-Atlas（报告 #245）走的也是类似路线——先训练基础动作模型，再让 Agent 调用。**"更好的表示 > 更大的模型"** 这一原则在 GUI Agent 领域尤为成立。

2. **混合范式是必然趋势**：纯 LLM 驱动（Browser-Use）适合探索性任务，纯代码驱动（Playwright）适合重复性任务，但生产环境需要两者的有机结合。Notte 的 "Agent bootstrap → Code deploy" 模式很可能成为未来 Browser Agent 的标准工作流。Stagehand V3 也朝着类似方向演进（hybrid approach）。

3. **可观测性是生产前提**：Notte 内置的 Session Replay 和 96.6% 的任务可靠性说明——如果无法复现和调试 Agent 的行为，就无法将其放入生产环境。这一点对所有 Agent 框架都是通用的启示。

### 对我们工作的参考价值

OpenClaw 的 browser 工具在处理网页交互时，也可以借鉴 Notte 的感知层思路：在将页面快照发送给模型之前，先做一轮语义过滤和压缩，减少无效信息干扰。此外，Demonstrate Mode 的"示范即代码"理念也值得在构建自动化工作流时参考。
