## 项目简介

Browser Use 是一个让 **AI Agent 能够直接操作浏览器** 的 Python 开源框架。它的核心理念是将任何网页变成 AI 可交互的接口——Agent 不再需要依赖 API，而是像人类一样通过浏览器完成点击、输入、导航等操作。

主要能力：
- **网页自动化任务**：自动填表、数据抓取、在线购物、申请职位等
- **多 LLM 支持**：原生支持 OpenAI、Anthropic Claude、Google Gemini，以及自研的 ChatBrowserUse 模型
- **云端 + 本地双模式**：提供 Browser Use Cloud（SaaS）和完全自托管的开源版本
- **100 个真实场景基准测试**：开源 benchmark 衡量 Agent 在真实浏览器任务上的表现
- **生产级部署**：支持沙箱环境、Cookie 同步、代理轮换、验证码处理

### 技术栈

| 层面 | 技术 |
|------|------|
| 语言 | Python 3.11+ |
| 浏览器引擎 | **Playwright**（Chromium） |
| LLM 接口 | LangChain ChatModel 兼容层 |
| 异步 | asyncio（全异步架构） |
| 包管理 | uv / pip |
| DOM 处理 | 自研元素选择器 + 可访问性树 |

### 核心运行流程

1. **DOM 快照** → 将当前页面解析为结构化的元素列表（标签、文本、XPath）
2. **Prompt 构建** → 将 DOM 元素 + 任务目标 + 历史动作注入 System Prompt
3. **LLM 决策** → 模型输出下一步动作（点击哪个元素、输入什么内容）
4. **动作执行** → 通过 Playwright 在浏览器中执行操作
5. **循环迭代** → 重复直到任务完成或达到最大步数

## 技术亮点

### 1. DOM 元素智能标注与可访问性树（Accessibility Tree）

这是 Browser Use 最核心的创新。它不是把原始 HTML 喂给 LLM（那会爆炸 token），而是：

- **元素过滤与优先级排序**：只保留可交互元素（按钮、输入框、链接），过滤掉 script/style 等噪声
- **智能编号**：给每个可交互元素分配简洁的 ID（如 `[123]`），LLM 只需输出 ID 即可定位
- **可访问性属性利用**：读取 `aria-label`、`role`、`alt` 等语义信息，让 LLM 更准确理解元素用途
- **动态裁剪**：根据页面复杂度自适应调整元素数量，控制 token 消耗

**为什么值得学习**：这是"让 LLM 操作 GUI"这个问题的最优解之一——如何在信息不丢失的前提下将 DOM 压缩到 LLM 能处理的规模。

### 2. 多模型兼容的自适应 Prompt 工程

Browser Use 针对不同 LLM 家族设计了不同的提示策略：

- **ChatBrowserUse**（自研）：专门为浏览器任务微调的模型，速度提升 3-5x
- **Claude 系列**：利用其强视觉能力，支持截图模式辅助决策
- **Gemini 系列**：利用长上下文窗口处理大型页面
- **GPT 系列**：标准 JSON 结构化输出模式

每种模型都有对应的 `output_parser` 和 prompt template，通过统一的 `ChatModel` 接口屏蔽差异。

**为什么值得学习**：这种"一套逻辑、多种后端"的抽象方式，是构建 LLM 应用时的最佳实践。

### 3. 生产级的容错与恢复机制

不同于实验室原型，Browser Use 为真实生产环境做了大量工程优化：

- **动作验证循环**：每次执行动作后会验证页面状态是否符合预期，不符合则重试或回退
- **智能等待**：不是固定 sleep，而是基于网络条件和页面加载状态的动态等待
- **错误分类与恢复**：区分网络错误、元素未找到、验证码弹出等不同错误类型，各有应对策略
- **Session 持久化**：支持保存/恢复浏览器状态（cookies、localStorage），适合长时间任务

**为什么值得学习**：从 demo 到生产的跨越，核心就是这些" boring "的工程细节。

## 对楠楠的启发

### 1. 浏览器操作 = Agent 的终极通用工具

Browser Use 的成功证明了一个重要趋势：**与其为每个网站写 API 集成，不如让 Agent 直接操作浏览器**。这对 OpenClaw 有直接启发——我们的 browser 工具已经在做类似的事，但可以参考 Browser Use 的 DOM 抽象层来增强能力，特别是：
- 更智能的元素定位（不依赖脆弱的 CSS selector）
- 动作前的意图确认机制
- 多步骤任务的自动分解与恢复

### 2. Token 效率是 Agent 工程的核心命题

Browser Use 在 DOM 压缩上的投入说明：**在 Agent 系统中，如何高效地将世界状态编码进 context window 是个关键问题**。我们在设计楠楠的记忆系统和工具调用时也应该借鉴这个思路——不是所有信息都需要完整传递，关键是传递"可行动的信息"。

### 3. 开源 + Cloud 的商业模式值得研究

Browser Use 的"开源核心 + 商业云服务"模式非常聪明：开源版吸引用户和贡献者，Cloud 版提供开箱即用的生产体验（反检测、代理池、CAPTCHA 解决）。这种模式对于任何 AI 工具项目都有参考价值。
