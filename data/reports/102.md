# AI Agent 学习报告 #102 — AutoGen（Microsoft）

## 项目信息

- **项目名：** AutoGen
- **GitHub 链接：** https://github.com/microsoft/autogen
- **一句话简介：** 微软开源的多智能体对话框架，支持事件驱动的 Agent 编排、分布式运行时和跨语言（Python/.NET）协作，是当前最成熟的企业级多 Agent 框架之一。
- **Stars：** 40k+ | **License：** MIT | **语言：** Python + C# (.NET)

---

## 架构设计

### 三层分层架构

AutoGen 采用清晰的**分层设计**，每层职责明确、可独立使用：

```
┌─────────────────────────────────────────────┐
│         AgentChat API (高层抽象)              │  ← 快速原型、常见多Agent模式
│   AssistantAgent / UserProxyAgent / Teams    │
├─────────────────────────────────────────────┤
│           Core API (核心引擎)                 │  ← 消息传递、事件驱动、分布式运行时
│   Message / Agent / Runtime / Subscription   │
├─────────────────────────────────────────────┤
│          Extensions API (扩展层)              │  ← LLM客户端、工具、代码执行
│   OpenAI / Azure / MCP / Code Execution      │
└─────────────────────────────────────────────┘
```

### 核心模块

| 模块 | 职责 |
|------|------|
| **autogen-core** | 消息传递总线、事件驱动 Agent 抽象、本地/分布式 Runtime、跨语言 gRPC 支持 |
| **autogen-agentchat** | 高层 API：`AssistantAgent`、`UserProxyAgent`、`Team` 编排模式（RoundRobin、Selector、Sequential） |
| **autogen-ext** | 扩展实现：OpenAI/Azure 客户端、MCP 工具集成、代码执行沙箱 |
| **autogen-studio** | 无代码 GUI，可视化搭建多 Agent 工作流 |
| **agbench** | Agent 基准测试套件 |
| **magentic-one-cli** | 基于 AutoGen 构建的 SOTA 多 Agent 团队（网页浏览+代码执行+文件处理） |

### 技术栈

- **语言：** Python 3.10+ / .NET (C#)
- **LLM 集成：** OpenAI、Azure OpenAI、通过扩展接口支持任意模型
- **协议：** gRPC 用于跨语言通信和分布式 Runtime
- **工具协议：** MCP (Model Context Protocol) 原生支持
- **异步模型：** 全面基于 `asyncio`（Python 端）

### 工作流程

1. **定义 Agent** → 每个 Agent 绑定 `model_client` + `system_message` + `tools`
2. **组织 Team** → 选择编排模式（RoundRobin 轮流发言、Selector 路由、Sequential 顺序执行）
3. **消息循环** → Core Runtime 的消息总线驱动 Agent 间通信，每个 Agent 收到消息后决定是否调用工具或回复
4. **工具调用** → 通过 Extensions 层连接 MCP Server 或内置工具，支持 `max_tool_iterations` 控制最大迭代次数
5. **流式输出** → `run_stream()` 支持实时流式返回 Agent 思考过程和中间结果

```python
# 典型使用模式
agent = AssistantAgent(
    "assistant",
    model_client=OpenAIChatCompletionClient(model="gpt-4.1"),
    tools=[math_agent_tool, chemistry_agent_tool],  # Agent作为工具嵌套
    max_tool_iterations=10,
)
result = await agent.run(task="计算 x^2 的积分")
```

## 技术亮点

### 1. Agent-as-Tool 嵌套模式（AgentTool）

这是 AutoGen 最精妙的设计之一——**将一个完整的 Agent 封装为另一个 Agent 的工具**：

```python
from autogen_agentchat.tools import AgentTool

# 专家 Agent 可以被包装成工具
math_agent = AssistantAgent("math_expert", model_client=client, system_message="你是数学专家")
math_agent_tool = AgentTool(math_agent, return_value_as_last_message=True)

# 主 Agent 像调用普通函数一样调用专家 Agent
general_agent = AssistantAgent("assistant", tools=[math_agent_tool, chemistry_agent_tool])
```

**为什么重要：** 这实现了**层级化的 Agent 分工**。主 Agent 负责"思考该找谁"，子 Agent 负责"专业领域推理"。对比 CrewAI 的 Role-based 扁平分工，AutoGen 的树状嵌套更灵活。对 OpenClaw 的启发：subagent 机制本质上就是 AgentTool 模式——可以把特定 skill 的执行封装为"工具 Agent"。

### 2. 事件驱动的 Core Runtime（发布-订阅模式）

Core 层不是简单的请求-响应，而是**完整的事件驱动架构**：

```python
# Agent 通过订阅消息类型来响应
@agent.on_message(lambda msg: isinstance(msg, TaskResult))
async def handle_task_result(agent, message):
    # 处理任务完成事件
    pass

# Runtime 作为消息总线
runtime = SingleThreadedRuntime()
await agent.send(message=target_agent)
```

**关键设计决策：**
- Agent 是**被动的消息处理器**（reactive），不是主动轮询
- 消息传递通过 Runtime 中间件解耦，天然支持分布式部署
- 订阅机制允许精细控制"什么消息触发什么行为"

这比 LangGraph 的图编排更底层、更灵活。LangGraph 定义的是"流程图"，AutoGen Core 定义的是"消息协议"。

### 3. MCP 工具原生集成

AutoGen 对 MCP (Model Context Protocol) 的一等公民支持非常彻底：

```python
from autogen_ext.tools.mcp import McpWorkbench, StdioServerParams

server_params = StdioServerParams(command="npx", args=["@playwright/mcp@latest", "--headless"])

async with McpWorkbench(server_params) as mcp:
    agent = AssistantAgent(
        "web_browsing_assistant",
        model_client=client,
        workbench=mcp,  # 直接注入 MCP 工具集
        max_tool_iterations=10,
    )
```

**技术细节：**
- `McpWorkbench` 管理 MCP Server 的生命周期（启动/停止/心跳）
- MCP 工具自动转换为 Agent 可调用的 function calling schema
- 支持多个 MCP Server 同时注入（传 list）
- 安全警告：只连接可信的 MCP Server（可能执行本地命令）

这对 OpenClaw 很有参考价值——我们的 skill/tool 系统如果也支持 MCP 协议，就能直接复用整个 MCP 生态的工具。

### 4. Team 编排模式（Sequencer/Selector/RoundRobin）

AgentChat 层提供了三种开箱即用的多 Agent 编排策略：

| 模式 | 适用场景 | 机制 |
|------|---------|------|
| **Sequential** | 流水线任务 | Agent A 输出 → Agent B 输入 → Agent C 输出 |
| **RoundRobin** | 讨论协商 | 所有 Agent 轮流发言，直到达成共识或达到轮次上限 |
| **Selector** | 路由分发 | 根据 task 内容选择最合适的 Agent 处理 |

```python
# Selector 模式示例：根据问题类型路由到对应专家
team = Selector(
    participants=[math_agent, coding_agent, writing_agent],
    model_client=client,
    # 模型自动判断应该把任务交给哪个 participant
)
result = await team.run(task="帮我写一首关于微积分的诗")
```

**与 CrewAI 对比：** CrewAI 用 `process=sequential/hierarchical` 做流程控制，但本质还是硬编码的顺序。AutoGen 的 Selector 用 LLM 做**语义路由**（semantic routing），更智能但也更耗 token。

### 5. 跨语言 gRPC Runtime

AutoGen Core 支持 Python 和 .NET 通过 gRPC 互操作：

```
Python Agent ←→ Core Runtime (gRPC) ←→ .NET Agent
     ↑                                    ↑
  autogen-core                      Microsoft.AutoGen.Core
```

这意味着：
- Python 写的 Agent 和 C# 写的 Agent 可以在同一个 Runtime 中协作
- 分布式部署时，不同节点可以用不同语言实现
- 企业环境中可以复用现有的 .NET 服务

这在多 Agent 框架中是**独一无二的**——LangGraph/CrewAI/MetaGPT 都是纯 Python。

## 对我们（OpenClaw/楠楠）的启发

### 🔍 可直接借鉴的设计

1. **AgentTool 嵌套模式 → subagent 优化**
   - 当前 OpenClaw 的 subagent 是"派生子进程执行任务后汇报"
   - 可以借鉴 AutoGen 将 subagent 包装为可调用的"工具"，让主 agent 在对话中按需调用，而不是每次都走完整的 spawn 流程
   - 特别是轻量级任务（查天气、搜资料），用 AgentTool 模式比 spawn subagent 更高效

2. **MCP 一等公民支持 → 工具生态打通**
   - AutoGen 的 `McpWorkbench` 设计很优雅：生命周期管理 + 自动 schema 转换
   - 如果 OpenClaw 的 tool 系统增加 MCP adapter 层，就能直接使用 Playwright MCP、Filesystem MCP 等数百个现成工具
   - 实现成本低：MCP 本身就是 JSON-RPC over stdio/SSE

3. **事件驱动 Core → 消息架构升级**
   - 当前 OpenClaw 的 agent 交互主要是 request-response
   - 如果引入发布-订阅消息总线，可以实现：agent A 完成任务后自动通知所有订阅者，而不是靠主 agent 轮询
   - 对 heartbeat 机制和多 channel 消息分发都有帮助

4. **Team 编排模式 → 多 skill 协作**
   - 当前的 skill 系统是"匹配一个 skill 执行"
   - 借鉴 Sequential/Selector 模式，可以实现"多个 skill 组成的流水线"（比如：搜索→分析→生成报告）
   - Selector 的语义路由思路可以用来做 skill dispatcher

### ⚠️ 需要注意的坑

1. **过度工程化风险：** AutoGen 的三层架构虽然优雅，但对简单场景来说太重了。OpenClaw 应保持"简单任务快车道"的原则，复杂编排才启用高级模式
2. **Token 消耗：** Selector 每次路由都要调 LLM 判断，成本不低。我们做类似功能时要考虑缓存/规则优先级
3. **Microsoft 方向转变：** AutoGen 正在与 Semantic Kernel 合并为统一的 Microsoft Agent Framework，项目未来有不确定性。学习其设计思想即可，不建议深度依赖
