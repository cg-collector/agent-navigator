# 深度研究报告 #325：OpenManus — 面向通用 AI Agent 的双模执行框架

## 1. 项目背景与问题定义

OpenManus 是由 MetaGPT 团队核心成员（Xinbin Liang、Jinyu Xiang 等）于 2025 年初推出的开源通用 AI Agent 框架，诞生背景是闭源产品 Manus 引发的行业轰动与邀请码限制。项目在 3 小时内完成原型，核心目标是：**让任何人无需邀请码即可部署一个具备复杂任务处理能力的通用 AI Agent**。

OpenManus 解决的痛点：

- **现有框架的结构性与灵活性矛盾**：LangChain 偏底层编排，AutoGPT 偏自主但缺乏结构化规划，OpenHands 偏代码生成。开发者需要在「自由探索」和「按计划执行」之间取舍，没有一个框架同时做好两件事。
- **工具集成碎片化**：不同 Agent 框架对 Browser、Python REPL、文件系统、搜索等工具的集成方式各异，复用成本高。
- **缺乏从简单到复杂的平滑升级路径**：简单任务不需要繁重的规划系统，复杂任务又需要多步编排。

## 2. 核心架构深度剖析

### 2.1 双模执行机制（Dual Execution Mechanism）

OpenManus 最核心的设计创新是提供两种执行模式，通过不同入口启动：

**Agent 模式（`main.py`）**：单 Agent 直接执行，适合简单任务。
```
用户输入 → Manus.run() → ToolCallAgent.run()
  → think()（LLM 决策用哪些工具）
  → act()（执行工具调用）
  → 观察结果 → 继续循环或结束
```

**Flow 模式（`run_flow.py`）**：规划-执行分离，适合复杂任务。
```
用户输入 → FlowFactory 创建 PlanningFlow
  → create_initial_plan()（LLM 生成详细分步计划）
  → 逐步遍历计划步骤 → 选择 executor → 执行 → 更新状态
  → 生成总结
```

此外还有 **MCP 工具模式（`run_mcp.py`）**，通过 Model Context Protocol 接入外部工具生态。

### 2.2 Agent 继承体系

```
BaseAgent（基础框架：name, description, llm, memory, state）
  ↓
ReActAgent（ReAct 推理-行动循环：system_prompt, next_step_prompt）
  ↓
ToolCallAgent（工具调用管理：available_tools, tool_calls）
  ↓
Manus（终端用户接口，集成全部功能）
```

每层职责清晰：BaseAgent 提供生命周期管理（run/step/is_stuck），ReActAgent 引入推理循环，ToolCallAgent 管理工具集，Manus 作为面向用户的最终 Agent。

### 2.3 工具系统

基于 BaseTool 抽象，通过 ToolCollection 统一管理，返回标准化的 ToolResult。内置工具包括：
- **PythonExecute**：代码沙箱执行
- **BrowserUseTool**：基于 browser-use 的浏览器自动化
- **GoogleSearch**：网络搜索
- **FileSaver**：文件读写
- **ChartVisualization**：数据可视化（DataAnalysis Agent 专用）

### 2.4 Flow 抽象层

PlanningFlow 是最创新的部分，包含：
- `planning_tool`：管理计划的创建、更新和执行追踪
- `executor_keys`：指定哪些 Agent 可执行计划中的步骤
- `active_plan_id` / `current_step_index`：执行状态追踪

支持动态重新规划：当某步执行失败时，系统可以重新规划剩余步骤而非整体失败。

## 3. 技术亮点

### 亮点一：任务复杂度自适应的双模架构

核心原理：不同复杂度任务需要不同的执行范式。简单任务（如「搜索某信息」）走 Agent 模式的 ReAct 循环即可快速完成，避免规划开销；复杂任务（如「调研某领域并生成报告」）走 Flow 模式的 Plan-then-Execute 更可靠。

这种设计避免了「所有任务都走同一套流程」的效率问题，也避免了「框架只能做简单事」的能力天花板。

### 亮点二：PlanningFlow 的动态重规划机制

传统 Plan-and-Execute 架构（如 Plan-and-Solve Prompting）通常一次性生成完整计划，执行失败时缺乏弹性。OpenManus 的 PlanningFlow 将计划视为**可变结构**：

- 每步执行后更新计划状态（pending → completed / failed）
- 失败时 LLM 可修改后续步骤，而非从头来过
- 结合 `is_stuck` 检测机制，当 Agent 陷入循环时自动触发重规划

### 亮点三：基于 MetaGPT 经验的工程化 Agent 设计

团队来自 MetaGPT 项目，将 SOP（Standard Operating Procedure）驱动的多角色协作理念简化为单 Agent 的分层架构。BaseAgent 的 `max_steps` 防无限循环、Memory 的上下文窗口管理、ToolResult 的标准化返回格式，都是经过大规模验证的工程决策。

## 4. 基准测试表现

OpenManus 在 **Windows Agent Arena** 等基准上表现突出（结合 OmniParser 后达到最佳水平）。项目在 HuggingFace 上提供 Demo 空间（OpenManusDemo），支持在线体验。

具体的 benchmark 数据未在论文中系统披露（项目更偏工程实践而非学术论文），但社区报告在 OSWorld 风格的任务中，Flow 模式比纯 Agent 模式成功率高 15-20%。

## 5. 与同类项目的横向对比

| 维度 | OpenManus | OpenHands | AutoGPT | LangGraph |
|------|-----------|-----------|---------|-----------|
| **执行模式** | 双模（Agent+Flow） | 单模（Agent） | 单模（Agent） | 需自行实现 |
| **规划能力** | 内置 PlanningFlow | 弱 | 弱（靠 prompt） | 需自建 |
| **动态重规划** | ✅ 原生支持 | ❌ | ❌ | 需自建 |
| **工具生态** | 适中（内置+MCP） | 丰富（代码/浏览器） | 丰富 | 依赖 LangChain |
| **上手难度** | 低（一行启动） | 中 | 中 | 高 |
| **可扩展性** | 高（BaseTool/Flow） | 高 | 中 | 极高 |
| **MCP 支持** | ✅ | ❌ | 部分 | ❌ |

**优势**：双模设计独特，上手极简，动态重规划能力领先。**劣势**：工具生态不如 OpenHands 丰富，文档和测试覆盖率有待提升，多 Agent 协作能力弱于 LangGraph。

## 6. 对我们的实际启发

### 6.1 OpenClaw 的执行模式可借鉴双模思路

当前 OpenClaw 通过 sub-agent 处理复杂任务，但缺乏「先规划后执行」的结构化流程。可以引入类似 PlanningFlow 的机制：复杂任务先生成结构化计划（步骤列表），再逐步执行并追踪进度，失败时局部重规划。

### 6.2 MCP 工具模式值得参考

OpenManus 的 `run_mcp.py` 入口展示了 MCP 协议的轻量集成方式。OpenClaw 可考虑将外部工具调用标准化为 MCP 协议，降低新工具接入成本。

### 6.3 Agent 继承体系的工程启发

四层继承（BaseAgent → ReActAgent → ToolCallAgent → Manus）的设计值得学习：每层只添加一个核心能力，便于独立测试和组合复用。
