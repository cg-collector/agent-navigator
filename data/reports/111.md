- **GitHub 链接**：https://github.com/mastra-ai/mastra

## 架构设计

### 整体架构：六大核心模块 + 统一运行时

```
┌─────────────────────────────────────────────────┐
│                   Mastra Platform                │
├──────────┬──────────┬──────────┬────────────────┤
│  Agents   │ Workflows │   RAG     │    Memory      │
│ (自主Agent)│(确定性流程)│(检索增强) │(持久化记忆)    │
├──────────┴──────────┴──────────┴────────────────┤
│              Evals (评估) | MCP (工具协议)         │
├─────────────────────────────────────────────────┤
│           Model Router (统一模型路由)             │
│   OpenAI · Anthropic · Gemini · DeepSeek · ...   │
│              (94+ Provider, 3300+ Models)        │
├─────────────────────────────────────────────────┤
│            Vercel AI SDK (底层 LLM 交互)          │
└─────────────────────────────────────────────────┘
```

### 核心模块详解

#### 1. Agent 系统（自主推理引擎）

Agent 是 Mastra 的核心抽象，基于 **ReAct（Reasoning + Acting）循环**：

```typescript
import { Agent } from '@mastra/core/agent'

export const testAgent = new Agent({
  id: 'test-agent',
  name: 'Test Agent',
  instructions: 'You are a helpful assistant.',
  model: 'openai/gpt-5.4',  // provider/model 格式
  tools: [searchTool, calculatorTool],  // 可选工具集
})
```

**工作流程**：
1. 接收用户输入 → 注入 system prompt（instructions）
2. LLM 推理 → 决定是直接回答还是调用工具
3. 若调用工具 → 执行工具 → 将结果回传 LLM
4. 循环步骤 2-3 直到 LLM 输出最终答案或触发停止条件
5. 返回结构化响应（text + toolCalls + toolResults + steps + usage）

**两种调用模式**：
- `.generate()` — 等待完整响应（所有 tool call 循环结束后返回）
- `.stream()` — 实时 token 流式输出（暴露 textStream / toolCalls / toolResults 等 Promise）

**多 Agent 模式**：支持 Supervisor Agent（管理者模式），一个 coordinator agent 调度多个子 agent，子 agent 可以作为 tool 注册到父 agent。

#### 2. Workflow 引擎（确定性编排）

当任务步骤明确时使用 Workflow 而非 Agent。采用 **Builder 模式 + 有向无环图(DAG)执行**：

```typescript
import { createWorkflow, createStep } from '@mastra/core/workflows'
import { z } from 'zod'

const step1 = createStep({
  id: 'step-1',
  inputSchema: z.object({ message: z.string() }),
  outputSchema: z.object({ formatted: z.string() }),
  execute: async ({ inputData }) => ({
    formatted: inputData.message.toUpperCase(),
  }),
})

const step2 = createStep({
  id: 'step-2',
  // ...
})

export const myWorkflow = createWorkflow({
  id: 'my-workflow',
  inputSchema: z.object({ message: z.string() }),
  outputSchema: z.object({ result: z.string() }),
})
  .then(step1)
  .then(step2)
  .commit()
```

**控制流原语**：
- `.then(step)` — 串行执行
- `.branch(condition, trueStep, falseStep)` — 条件分支
- `.parallel([stepA, stepB])` — 并行执行
- 支持 **Suspend/Resume** — 工作流可在任意 step 暂停，等待人工审批后恢复
- 支持 **State** — 跨 step 共享状态（`stateSchema` + `setState`）
- 支持 **嵌套** — Workflow 可作为另一个 Workflow 的 step
- 支持 **Time Travel** — 执行完成后可重放单个 step 进行调试

#### 3. RAG 管道（检索增强生成）

完整的 RAG pipeline：
- 文档分块（chunking）→ Embedding 生成 → 向量存储 → 相似性搜索 → 重排序（reranking）
- 支持的向量数据库：Pinecone, Qdrant, ChromaDB, pgvector, AstraDB, LanceDB 等
- 本地 embedding 通过 `@mastra/fastembed` 包支持

#### 4. Memory 系统（四层记忆架构）

| 层级 | 类型 | 说明 |
|------|------|------|
| Conversation History | 对话历史 | 原始消息存储 |
| Semantic Recall | 语义回忆 | 通过 embedding 找到相关历史消息 |
| Working Memory | 工作记忆 | 结构化事实和偏好（键值对） |
| Observational Memory | 观察记忆 | 压缩旧对话为稠密观察向量 |

#### 5. Evals（评估框架）

支持三种评估方式：
- **Model-Graded Evaluation** — 用 LLM 评估输出质量（相关性、忠实度、毒性等）
- **Rule-Based Evaluation** — 基于规则的自动检查
- **Statistical Evaluation** — 统计指标评估

#### 6. MCP 支持（Model Context Protocol）

双向 MCP：
- 作为 **MCP Client** — 连接外部 MCP Server 获取工具
- 作为 **MCP Server** — 将 Mastra 的 Agent/Tool 暴露给 Cursor、Claude Desktop、VS Code 等客户端

### 底层依赖

- **Vercel AI SDK** — 处理所有 LLM 交互和流式传输（Mastra 不重复造轮子）
- **Zod** — 所有 schema 定义（input/output/state 全部类型安全）
- **TypeScript 严格模式** — 全栈类型推断，IDE 完整 IntelliSense

## 技术亮点

### 亮点 1：TypeScript 原生 + Zod Schema 驱动的端到端类型安全

这是 Mastra 最核心的设计决策。不同于 Python 框架（LangChain/CrewAI）或 Python 移植版，Mastra 从零用 TypeScript 构建：

```typescript
// 每个 Step 的输入输出都有 Zod schema 编译期校验
const step = createStep({
  id: 'analyze',
  inputSchema: z.object({
    text: z.string().max(10000),
    mode: z.enum(['fast', 'thorough']),
  }),
  outputSchema: z.object({
    sentiment: z.enum(['positive', 'negative', 'neutral']),
    score: z.number().min(0).max(1),
    keywords: z.array(z.string()),
  }),
  execute: async ({ inputData }) => { /* ... */ }
})

// Workflow 组合后，整个链路的类型自动推导
const workflow = createWorkflow({ /* ... */ })
  .then(step)       // TypeScript 知道下一步的输入类型
  .then(anotherStep) // 类型检查跨 step 传递
  .commit()
```

**为什么重要**：在 Agent 开发中，tool call 的参数校验、workflow 步骤间的数据传递是最常见的 bug 来源。Zod schema 在编译期就能捕获大部分类型错误，而不是等到运行时 LLM 返回格式不对才报错。这对 OpenClaw 这种有大量 tool 定义的系统很有参考价值。

### 亮点 2：Agent 与 Workflow 的双轨设计（自主 vs 确定性）

Mastra 明确区分了两种执行范式：

| 维度 | Agent | Workflow |
|------|-------|----------|
| 控制方式 | LLM 自主决定 | 开发者显式定义 |
| 适用场景 | 开放式任务 | 多步骤确定流程 |
| 可预测性 | 低（每次执行路径可能不同） | 高（DAG 确定性执行） |
| 调试难度 | 高（非确定性循环） | 低（每步可重放） |
| 人工介入 | Approval 模式 | Suspend/Resume |

```typescript
// Agent 内部循环是非确定性的 — LLM 自己决定调什么工具
const response = await agent.generate('帮我分析这家公司的财务状况')
// 可能调用: searchCompany → getFinancialData → analyze → summarize
// 也可能直接回答，取决于 LLM 判断

// Workflow 是确定性的 — 每步都预定义好
const result = await run.start({ inputData: { company: 'AAPL' } })
// 必须按: fetch → parse → analyze → report 顺序执行
```

**关键洞察**：大多数框架只做其中一种。Mastra 的双轨设计让开发者可以根据任务特性选择合适的抽象——开放式问题用 Agent，流水线用 Workflow，两者还可以组合（Workflow 的某个 Step 内部调用 Agent）。

### 亮点 3：Model Router — 94+ Provider 的统一抽象层

```typescript
// 一行切换模型，代码零改动
const agent = new Agent({
  model: 'anthropic/claude-opus-4-6',  // 或 'openai/gpt-5.4' 或 'deepseek/deepseek-chat'
})

// 运行时动态切换
const router = mastra.getModel('openai/gpt-4o')
const fallback = mastra.getModel('anthropic/claude-sonnet-4')

// 自动 fallback — 主模型不可用时切到备用
agent.model = {
  primary: 'openai/gpt-4o',
  fallback: 'anthropic/claude-sonnet-4',
}
```

底层实现思路：每个 provider 封装为统一的 `ModelProvider` 接口，内部将 Mastra 的标准调用格式转换为各 provider 的 API 格式。通过 AI SDK 的 provider 体系实现，而非自己维护 adapter。

**对 OpenClaw 的启发**：OpenClaw 已经有 model routing（zai/glm-5V-turbo 等），但 Mastra 的 fallback 和 runtime switching 设计更优雅。

### 亮点 4：Workflow 的 Suspend/Resume + State 持久化

这是生产级 Agent 框架的关键能力：

```typescript
const approvalStep = createStep({
  id: 'approval',
  execute: async ({ suspend }) => {
    // 暂停执行，等待外部恢复
    await suspend({
      type: 'human-in-the-loop',
      payload: { action: 'delete-database', reason: 'destructive operation' },
    })
    // 用户审批后会从这里继续执行
    return { approved: true }
  },
})

// 恢复执行
const run = await workflow.createRun()
await run.resume({ suspensionId: 'xxx', inputData: { approved: true } })
```

**实现原理**：每次 suspend 时，当前 workflow state（包括所有 step 的输入输出、中间状态）被序列化到 storage（支持 SQLite/PostgreSQL/Redis）。恢复时从 storage 加载状态，找到 suspended 的 step 继续执行。这意味着即使服务器重启，挂起的 workflow 也不会丢失。

### 亮点 5：Processors — Agent 消息管道的中间件模式

```typescript
const agent = new Agent({
  model: 'openai/gpt-4o',
  processors: [
    // 前处理：注入上下文
    {
      beforeGeneration: async ({ messages, context }) => {
        const userContext = await getUserPreferences(context.userId)
        return [...messages, { role: 'system', content: `User prefers: ${userContext}` }]
      },
    },
    // 后处理：过滤敏感信息
    {
      afterGeneration: async ({ response }) => {
        return redactSensitiveInfo(response.text)
      },
    },
  ],
})
```

这本质上是一个 **洋葱模型中间件**（类似 Koa/Express middleware），允许在 LLM 调用的前后注入自定义逻辑：日志记录、上下文增强、输出过滤、prompt 注入检测等。

## 对我们（OpenClaw/楠楠）的启发

### 1. Zod Schema 驱动的 Tool 定义值得借鉴

OpenClaw 当前的 tool 定义是基于 JSON Schema 的（符合 MCP 协议规范），但缺少编译期类型推导。如果能在 skill/tool 注册层引入 Zod-like 的类型系统，让 tool 的 input/output 在注册时就完成类型检查，可以减少大量 "LLM 返回了错误参数格式" 的运行时错误。

**具体建议**：考虑在 OpenClaw 的 tool 注册机制中增加 schema validation 层，不一定要用 Zod（Node.js 生态），但可以用 JSON Schema + AJV 在 tool 调用前做一次验证。

### 2. Agent / Workflow 双轨设计的参考价值

OpenClaw 目前主要走 Agent 模式（LLM 自主决定调用什么 tool）。但对于一些固定流程的任务（比如"每日早报生成"= 搜索天气+搜索新闻+搜索股票+格式化输出），用 Workflow/DAG 模式会更可靠、更可调试。

**具体建议**：可以为 OpenClaw 增加"任务模板"概念——预定义的多步 task pipeline，不走 LLM 推理而是按固定顺序执行 tool，只在需要 LLM 的步骤才调用。这比纯 Agent 模式省 token、更快、更稳定。

### 3. Processor 中间件模式可直接应用

OpenClaw 的 AGENTS.md/SOUL.md 已经是一种"前处理器"（注入 system prompt），但缺乏结构化的后处理管道。引入 processor 模式可以让：
- 安全审计（检测 prompt injection）变成可插拔的 middleware
- 日志/观测变得标准化
- 不同 channel（飞书/Discord/Telegram）的输出适配变成独立的 processor

### 4. Suspend/Resume 对长任务的意义

OpenClaw 的 subagent 机制已经实现了异步任务，但缺少"暂停等待人工确认"的能力。对于需要审批的操作（删除文件、发外部消息、修改配置），引入 suspend/resume 模式可以显著提升安全性。

### 5. Memory 四层架构的参考

OpenClaw 目前的记忆系统是文件-based（daily notes + MEMORY.md）。Mastra 的四层设计提供了更细粒度的思路：
- **Working Memory** = 当前 session 的临时上下文（OpenClaw 已有）
- **Semantic Recall** = 用 embedding 检索历史记忆（OpenClaw 缺失，目前靠读文件）
- **Observational Memory** = 压缩旧对话（OpenClaw 的 MEMORY.md 是手动版的这个）

如果未来要扩展记忆系统，Semantic Recall 是最有价值的增量——让楠楠能根据当前对话语义自动检索相关历史记忆，而不是靠固定的文件读取顺序。

## 总结评价

**优势**：
- TypeScript 生态的最佳 Agent 框架，没有之一
- DX（开发体验）极佳，CLI scaffolding + Studio 可视化调试
- 双轨设计（Agent + Workflow）覆盖了更多场景
- 生产级特性完备（evals、observability、suspend/resume）

**劣势**：
- 相比 LangChain 生态，社区和第三方集成还比较年轻
- 企业功能（RBAC、SSO）闭源
- 文档虽好但部分高级功能（如 multi-agent 高级模式）文档还不够深入
- 对 Python 生态的工具/库集成需要通过 MCP bridge，多一层间接

**适合我们的点**：Mastra 的架构思想（特别是双轨设计、Processor 中间件、类型安全的 tool 定义）对 OpenClaw 的演进有直接的参考价值。虽然我们不会换成 Mastra（技术栈不同），但其设计模式可以借鉴。
