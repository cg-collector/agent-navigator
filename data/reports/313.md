# 📚 AI Agent 项目研究报告 #313 — HAR-GUI：面向GUI智能体的历史感知推理框架

## 一、项目背景与问题定义

**HAR-GUI（History-Aware Reasoning for GUI Agents）** 是浙江大学与蚂蚁集团联合提出的研究成果，发表于 **AAAI 2026**（Technical Track on Planning, Routing, and Scheduling）。论文核心作者包括 Ziwei Wang、Leyang Yang 等，DOI: 10.1609/aaai.v40i43.40966。

### 解决的核心痛点

当前基于多模态大语言模型（MLLM）的 GUI Agent 在长时序任务中存在一个根本性缺陷：**历史无关推理（History-Agnostic Reasoning）**。具体表现为：

1. **短期记忆薄弱**：Agent 将连续的交互序列视为离散的屏幕理解任务，无法将历史操作信息融入当前决策。每一步都"重新开始"，丢失了之前交互积累的上下文。
2. **缺乏情节推理能力**：在长 horizon 的 GUI 任务中，历史交互构成了面向目标的情节链（episodic chain），但现有 Agent 无法有效利用这些线索来指导当前动作选择。
3. **错误重复发生**：由于不具备从历史错误中反思学习的能力，Agent 在相似场景下会反复犯相同错误。

这一问题在移动端操作（如 AITW）、网页自动化（Mind2Web）和桌面 GUI 任务（GUI-Odyssey）等基准测试中普遍存在，严重制约了 GUI Agent 的实际可用性。

## 二、核心架构深度剖析

### 2.1 问题形式化

HAR 框架将 GUI 任务建模为马尔可夫决策过程（MDP）：

$$P(A_t | O_{\leq t}, A_{<t}, G)$$

其中 $D = (G, (O_1, A_1), \dots, (O_n, A_n))$ 是一个完整的情节链（episode chain），$G$ 为总目标，$O_t = (I_t, P)$ 表示第 $t$ 步的观察（屏幕图像 $I_t$ 和文本指令模板 $P$），$A_t \in \mathcal{A}$ 为该步的动作。关键设计在于：历史交互 $T_{<t}$ 通过文本模态传递——将过去的交互摘要整合到指令模板 $P$ 中。

### 2.2 两阶段训练框架

HAR 框架采用 **两阶段渐进式训练策略**：

#### 阶段一：GUI 场景预热（GUI Scenario Warm-up）

此阶段通过监督微调（SFT）向基础 MLLM 注入领域知识：

- **GUI 理解增强**：收集多源 GUI 数据进行多任务训练，涵盖：
  - 图形描述生成（Graphic Captioning）
  - UI 元素可点击性预测（Clickability Prediction）
  - 屏幕问答（Screen QA）
  - 屏幕摘要（Screen Summary）
  - 屏幕 Grounding（text-to-point 格式）
  
  数据覆盖移动端、桌面端和网站等多分辨率平台。

- **动作总结集成（Act2Sum）**：引入 teacher model $M_{teacher}$ 合成动作到摘要数据：
  
  $$O_t^{Act2Sum} = M_{teacher}(I_t, P_{sum}(G, A_t), \theta)$$
  
  其中 $P_{sum}$ 是指令模板，确保合成内容具有目标导向性和丰富语义，帮助 MLLM 生成全局感知的历史交互记录。

- **System-2 推理数据蒸馏**：$M_{teacher}$ 为每个情节实例合成 Chain-of-Thought（CoT）推理数据，经质量评估筛选后用于 SFT 训练，将高质量 GUI 推理知识注入基础模型。

#### 阶段二：从失败中学习（Learning From Failure）

预热后得到增强模型 $M_{warm-up}$，此阶段聚焦于通过自我演化从错误样本 $D_{his}$ 中学习：

- **错误纠正指南合成**：对每个错误实例 $\hat{x}_t = (G, T_{\leq t}, I_t)$，$M_{teacher}$ 生成不超过三条错误纠正指南 $G_t$：
  
  $$G_t = M_{teacher}(I_t, P_{guidance}(\hat{x}_t), \theta)$$
  
  这些指南作为外部 GUI 推理知识，协助 $M_{warm-up}$ 进行线索导向的推理增强。

- **第一轮强化学习（Round-1 RL）**：在反射场景中使用混合奖励函数训练，包含任务完成奖励、步骤效率奖励和指南遵循奖励三部分。

- **第二轮强化学习（Round-2 RL）**：移除外部指南，使执行格式与推理格式对齐。引入 **任务混合训练策略（TMTS, Task Mixing Training Strategy）**，将 grounding 任务和 GUI 情节推理任务联合训练，进一步提升屏幕视觉细节感知能力。

### 2.3 基础模型

HAR-GUI-3B 基于 **Qwen2.5-VL-3B-Instruct** 构建，是一个原生端到端的 GUI 专用模型，参数量仅 3B。

## 三、技术亮点

### 亮点一：历史感知推理范式转换

HAR 的核心创新在于将 GUI Agent 的推理模式从 **history-agnostic** 转换为 **history-aware**。这不是简单的上下文窗口扩展，而是通过结构化的训练流程让模型学会主动利用历史信息：

- Act2Sum 机制确保历史交互以**语义压缩**而非原始 token 形式传递，既节省了上下文空间又保留了关键决策线索
- 错误纠正指南以**外部知识注入**方式提供，避免了让小模型自行从混乱的长历史中提取有效信息的困难

这一设计的精妙之处在于：它承认了小参数量模型的上下文处理局限，用"教师引导+结构化摘要"的方式绕过了这个瓶颈。

### 亮点二：混合强化学习奖励函数

HAR 设计了一套创新的混合 RL 奖励函数，包含三个分量：

1. **任务完成奖励（Task Completion Reward）**：基于最终任务是否成功完成
2. **步骤效率奖励（Step Efficiency Reward）**：鼓励用更少步骤完成任务，避免冗余操作
3. **指南遵循奖励（Guidance Following Reward）**：Round-1 阶段专用，衡量模型是否正确利用了错误纠正指南

这种多维度奖励设计解决了传统 GUI Agent RL 训练中稀疏奖励导致的训练不稳定问题。特别是指南遵循奖励，为模型提供了密集的训练信号，显著加速了收敛。

### 亮点三：TMTS 任务混合训练策略

Round-2 RL 引入的 TMTS 是解决 **grounding 能力退化**问题的关键创新。经验表明，当模型过度专注于情节推理任务时，其细粒度的视觉定位（grounding）能力往往会下降。TMTS 通过在同一训练 batch 中混合 grounding 和情节推理两种任务，使用各自对应的奖励函数（Eq.7 用于 grounding，Eq.9 用于情节推理），迫使模型同时保持两类能力。实验证明 TMTS 使 ScreenSpot-V2 的平均准确率提升了数个百分点，有效缓解了任务间的负迁移效应。

## 四、基准测试表现

HAR-GUI-3B 在多个维度的 GUI 基准测试上取得了领先或极具竞争力的结果：

| 基准测试 | 类型 | HAR-GUI-3B 表现 | 对比说明 |
|---------|------|----------------|---------|
| **AITW** | 移动端情节推理 | **70.2% SSR** | 超越 MP-GUI (8B) 等更大模型 |
| **Mind2Web** | 网页自动化 | **44.0% SSR** | 同类最佳 |
| **GUI-Odyssey** | 桌面 GUI 推理 | 领先 | 显著优于 UI-R1-3B / GUI-R1-3B |
| **ScreenSpot** | 视觉 Grounding | **83.3% Avg.** | 领先结果 |
| **ScreenSpot-V2** | 视觉 Grounding | **86.2% Avg.** | 得益于 TMTS 策略 |
| **中文小程序基准** | OOD 泛化 | 显著优于同类 3B 模型 | 与更大模型竞争 |

**GUI 理解能力方面**，HAR-GUI-3B 在 ScreenQA、WebSRC、Screen2Words（S2W）等基准上超越多数 SOTA 方法，甚至在部分指标上超过 8B 参数量的 MP-GUI。

消融实验证实：TMTS 有效缓解了 grounding 退化；反射场景 + MAR（混合奖励）对于发展 history-aware 的严谨推理至关重要；Act2Sum 和 CoT 蒸馏分别贡献了显著的性能提升。

## 五、与同类项目的横向对比

### vs. CUA（Computer Use Agent，OpenAI）

| 维度 | HAR-GUI-3B | CUA |
|------|-----------|-----|
| 模型规模 | 3B（轻量级） | 依赖 GPT-4o（超大规模 API） |
| 推理模式 | History-Aware（内建历史感知） | 依赖 prompt 中的历史拼接 |
| 部署方式 | 本地端到端部署 | 云端 API 调用 |
| 训练策略 | 两阶段 SFT+RL | Prompt Engineering + Few-shot |
| 成本 | 极低（本地推理） | 高（API 调用费用） |
| 定制性 | 可针对特定场景微调 | 黑盒不可微调 |

**优势**：HAR-GUI 是真正可本地部署的开源方案，且通过结构化训练内建了历史感知能力，不依赖超大模型的通用推理能力来弥补 GUI 领域知识的不足。

**劣势**：CUA 背后的 GPT-4o 在纯语言理解和开放域推理上仍远强于 3B 模型，面对完全未见过的应用类型时泛化性可能不足。

### vs. PAL-UI（主动回溯框架，#309报告）

两者都关注 GUI Agent 的推理增强，但路径不同：
- **PAL-UI** 采用运行时回溯机制（Active Backtracking），在推理阶段动态修正错误路径
- **HAR-GUI** 采用训练时反思机制（Training-time Reflection），通过两阶段训练让模型学会正确推理

HAR-GUI 的优势在于推理时无需额外开销（不需要回溯搜索），但代价是需要精心设计的训练流程和数据合成管线。

### vs. AppAgentX（可进化框架，#310报告）

AppAgentX 强调运行时的动态进化和工具链扩展，而 HAR-GUI 更关注如何通过训练赋予模型固有的历史感知能力。两者互补：HAR-GUI 的训练方法论可以用于增强 AppAgentX 中基础模型的 GUI 推理能力。

## 六、对我们的实际启发

### 可落地想法 1：错误纠正指南的工程化复用

HAR 的错误纠正指南合成机制可以直接应用于我们的 GUI Agent 系统。具体做法：
- 收集 Agent 执行失败的历史案例
- 用强模型（如 GLM-5）为每个失败案例生成 2-3 条纠正指南
- 将（失败截图 + 指南）作为 few-shot 示例注入后续推理的 system prompt
- 这比单纯增加上下文长度更高效，因为指南是经过压缩的结构化知识

### 可落地想法 2：TMTS 多任务联合训练

如果我们计划训练或微调自己的 GUI Agent 模型，TMTS 提供了一个重要的训练策略参考：**不要只训练端到端的操作能力，必须同步保持 grounding 能力**。可以在训练 data loader 中按比例混合 grounding 数据（坐标预测）和操作轨迹数据（action sequence），避免模型"学会了操作但找不准按钮"的问题。

### 可落地想法 3：Act2Sum 历史压缩机制

对于我们的长对话/长任务场景，Act2Sum 的思想可以推广：不是简单截断或保留全部历史，而是用一个小模型将历史操作序列压缩为目标导向的语义摘要。这在 token 有限制的情况下尤其有价值——用 50 tokens 的摘要替代 500 tokens 的原始操作记录，且保留更多决策相关信息。
