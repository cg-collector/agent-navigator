# AI Agent 学习报告 #141 — OpenHands（原 OpenDevin）

> **研究日期：** 2026-04-05
> **GitHub：** https://github.com/OpenHands/OpenHands ⭐ 64k+ stars

## 项目简介

OpenHands 是目前开源社区中最受欢迎的 AI 软件开发 Agent 平台之一，前身为 OpenDevin，由 CMU 团队发起，现已发展为 All-Hands-AI 社区主导的大型开源项目。它旨在打造一个"AI 驱动的软件开发者"，能够像人类程序员一样自主完成编码、调试、测试、运行等完整开发流程。项目提供 SDK、CLI、本地 GUI 和云端服务四种使用方式，支持从个人实验到企业级部署的全场景覆盖。在 SWE-Bench Verified 等基准测试中表现优异，是目前代码执行型 Agent 的标杆项目。

## 架构设计

### 从 V0 到 V1 的架构演进

OpenHands 经历了一次重大的架构重构——从 V0 的单体架构升级为 V1 的模块化 SDK 架构。这次重构的背景非常值得借鉴：V0 在快速迭代中积累了严重的技术债务——140+ 配置字段散布在 15 个类中、强制沙箱假设导致本地执行需要大量 hack、单体仓库让 CLI/Web/GitHub App 三条产品线各自复制逻辑。

V1 的核心设计围绕**四个原则**展开：

1. **可选隔离**：Agent 默认本地运行，按需切换沙箱
2. **默认无状态，单一状态源**：所有组件不可变，唯一可变对象是 ConversationState
3. **严格关注点分离**：Agent 核心与下游应用解耦
4. **两层可组合性**：部署包可组合 + 组件可安全扩展

### 四包模块化架构

```
┌─────────────────────────────────────────────────┐
│              应用层 (Applications)                │
│   CLI │ Local GUI │ Cloud │ GitHub App           │
├─────────────┬───────────┬───────────────────────┤
│ agent_server│ workspace │      tools             │
│ REST/WS API │ Docker/   │  Editor/Bash/Browser/  │
│             │ Hosted    │  MCP Integration       │
├─────────────┴───────────┴───────────────────────┤
│              openhands.sdk (Core)                 │
│  EventSystem │ Agent │ LLM │ Tool │ Conversation │
└─────────────────────────────────────────────────┘
```

- **openhands.sdk**：核心抽象层，定义 Agent、Conversation、LLM、Tool、MCP 等接口和事件系统
- **openhands.tools**：基于 SDK 抽象的具体工具实现（编辑器、终端、浏览器等）
- **openhands.workspace**：执行环境抽象（Docker 沙箱、本地文件系统、托管 API）
- **openhands.agent_server**：REST/WebSocket API 服务器，支持远程执行

### 核心组件详解

**1. Event-Sourced State Management（事件溯源状态管理）**

这是整个架构最精妙的设计。所有交互都被建模为不可变事件追加到日志中：

- **Event** → 基类，提供 ID、时间戳、来源等不可变结构
- **LLMConvertibleEvent** → 可转换为 LLM 消息格式的事件（消息、动作、观察结果）
- **ActionEvent** → Agent 的工具调用（含思考和推理）
- **ObservationBaseEvent** → 工具执行的结果（成功/失败/拒绝）
- 内部事件（压缩请求、状态更新、暂停）不暴露给 LLM

**ConversationState** 是唯一的可变组件，维护两类状态：元数据字段（agent_status、stats 等）和 append-only 的 EventLog。FIFO 锁保证线程安全。

**2. Agent Loop（Agent 循环）**

Agent 实现标准的感知-推理-行动循环：
1. 从 ConversationState 获取当前上下文
2. 将相关事件转换为 LLM 消息（带上下文窗口管理）
3. 调用 LLM 获取响应（可能包含工具调用）
4. 执行工具调用，生成 Observation 事件
5. 追加到 EventLog，循环继续

**3. Tool System（工具系统）**

类型化的工具系统，原生集成 MCP（Model Context Protocol）。每个工具定义输入/输出 schema，支持异步执行和超时控制。

**4. Workspace Abstraction（工作空间抽象）**

统一的执行环境接口，同一个 Agent 可以在本地文件系统运行（开发调试），也可以无缝切换到 Docker 沙箱（生产部署），代码几乎不需要改动。

### 技术栈

- **语言**：Python（SDK 核心）、TypeScript（前端/CLI）
- **LLM 集成**：模型无关设计，支持 100+ 提供商（OpenAI、Anthropic、Google、Ollama 等）
- **容器化**：Docker（沙箱执行环境）
- **协议**：REST + WebSocket（实时通信）、MCP（工具扩展）
- **前端**：React SPA（GUI）、VSCode 浏览器版集成
- **数据验证**：Pydantic（全链路类型安全）

## 技术亮点

### 亮点一：Event-Sourced 状态管理 + 确定性回放

这是 OpenHands 最具架构深度的设计。传统 Agent 框架通常用可变字典或类属性来管理状态，导致调试困难、无法重现问题、崩溃后难以恢复。OpenHands 采用 CQRS/ES 领域的事件溯源模式：

**为什么这个设计如此重要？**

1. **完整的审计轨迹**：每一步操作都是不可变事件，可以精确回放任何时刻的 Agent 行为。这对调试"Agent 为什么在第 47 步做出了错误决策"这类问题至关重要。

2. **天然支持断点续传**：EventLog 持久化到磁盘后，Agent 崩溃可以从最后一条事件恢复，而不是丢失全部进度。对于可能运行数小时的复杂任务来说，这是生产级必备能力。

3. **LLM 上下文窗口管理**：当对话历史超出 LLM 上下文限制时，系统可以基于事件日志进行智能压缩（Condensation），丢弃细节事件但保留摘要——比简单的滑动窗口更聪明。

4. **多客户端同步**：Web UI、CLI、API 同时连接时，通过订阅 EventLog 实现实时状态同步，不需要复杂的 WebSocket 自定义协议。

**对我们的启发**：楠楠目前的记忆系统是文件-based 的（daily notes + MEMORY.md），如果引入事件溯源模式，可以让每次工具调用、每次学长交互都成为不可变事件，大幅提升记忆的可追溯性和可靠性。

### 亮点二：可选隔离的 Workspace 抽象

V0 的最大教训就是"强制沙箱化"。早期 OpenHands 假设所有工具调用都必须在 Docker 容器内执行，这导致了：
- 本地开发体验差（每次都要起容器）
- 与 MCP 协议冲突（MCP 假设本地执行）
- 多租户资源争抢（一个用户的浏览器截图撑爆容器）

V1 的解决方案优雅而实用：**Workspace 接口统一了本地和远程执行**。定义一个抽象的 Workspace 基类，LocalWorkspace 直接操作文件系统，DockerWorkspace 通过 gRPC/HTTP 与容器通信。Agent 层面完全不关心底层实现。

**关键洞察**：这不是简单的"加一层抽象"，而是重新思考了 Agent 与执行环境的关系。Agent 不应该知道自己在容器里还是裸机上——就像应用程序不应该知道自己在 KVM 还是物理机上一样。

**实际收益**：
- 开发时用 LocalWorkspace，秒级启动，无需 Docker
- 部署时切换 DockerWorkspace，自动获得隔离和安全
- 同一套测试可以同时在两种模式下跑

### 亮点三：Security Analyzer（安全分析器）

这是 OpenHands 区别于大多数开源 Agent 框架的独特功能。作为一个能执行任意代码和命令的 Agent，安全性是生死攸关的问题。OpenHands 内置了一个 Security Analyzer 组件：

- 在 Agent 执行每个 Action 前，分析该操作的潜在风险
- 支持多种策略模式：自动允许、自动拒绝、需人工确认、需审批流
- 可以检测危险操作（`rm -rf`、网络外连、敏感文件访问等）
- 与 Confirmation Policy 联动，形成完整的安全闭环

这个设计的精妙之处在于它不是事后审计，而是**事前拦截**——在 Action 变成实际的 Observation 之前就进行判断。而且安全策略是可配置的，不同部署场景可以用不同的安全级别。

## 对楠楠的启发

### 1. 引入事件溯源的记忆系统

当前楠楠的记忆是扁平的 markdown 文件，缺乏结构化的交互历史。可以借鉴 OpenHands 的 EventLog 设计：
- 每次工具调用、每次学长消息、每次 Agent 决策都记录为结构化事件
- 支持按时间线回放，方便 debug "为什么楠楠上次说错了话"
- 事件压缩机制可以在长期运行后保持记忆效率

### 2. 工具执行的沙箱化思路

虽然楠楠不像 OpenHands 那样需要执行任意代码，但随着能力扩展（比如 coding agent 子任务），工具执行的安全性会越来越重要。Workspace 抽象的思想可以提前引入——定义统一的工具执行接口，未来需要时可以无缝切换到隔离环境。

### 3. 安全确认机制

OpenHands 的 Security Analyzer + Confirmation Policy 模式可以直接借鉴。当前楠楠的"危险操作先问学长"是硬编码在 prompt 里的规则，如果改为结构化的安全分析器：
- 每个工具声明自己的风险等级
- 可配置的策略引擎决定是否需要确认
- 支持白名单/黑名单/正则匹配等多种规则

这样比 prompt engineering 更可靠，也更容易测试。

### 4. 配置系统的简化

OpenHands V0 的 140+ 配置字段的教训值得警惕。楠楠当前的配置分散在 AGENTS.md、TOOLS.md、SOUL.md、环境变量等多个地方，随着功能增长可能会面临同样的混乱。建议逐步收敛到一个结构化的配置源（比如 Pydantic Settings），各模块只读自己关心的部分。

> **一句话总结**：OpenHands 从 64k star 的成功项目中提炼出的架构智慧——事件溯源驱动状态、可选隔离的执行环境、内置安全分析——对任何想要从"能用的 Agent"进化到"生产级 Agent"的项目都有极高的参考价值。
