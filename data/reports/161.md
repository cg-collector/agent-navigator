## 📌 项目简介

Composio 是一个 **AI Agent 工具集成平台**，核心使命是解决 Agent 开发中最痛点的问题之一：**如何让 AI 安全、高效地调用真实世界的 API**。它提供 1000+ 预构建的工具包（Toolkit），覆盖 GitHub、Gmail、Slack、Notion、Jira、Salesforce 等几乎所有主流 SaaS 服务，同时内置认证管理、上下文管理和沙箱执行环境。

一句话概括：**Composio 是 AI Agent 的"工具层操作系统"——你只管写 Agent 逻辑，它负责让 Agent 能真正操作外部世界。**

### 五大核心模块

| 模块 | 职责 |
|------|------|
| **Toolkit System** | 1000+ 预构建 API 集成，每个 Toolkit 包含多个 Action/Trigger |
| **Auth Manager** | 统一管理 OAuth 流程、API Key、Token 刷新，Agent 无需关心认证细节 |
| **Connected Accounts** | 多用户多连接模型，每个用户独立管理自己的已授权账号 |
| **Context Manager** | 在 Agent 操作过程中维护和传递相关上下文信息 |
| **Sandbox Workbench** | 安全隔离的执行环境，用于测试和验证 Agent 的工具调用行为 |

### Provider 适配模式

Composio 通过 **Provider Pattern** 实现框架无关性：

```python
# Python: 三行代码给任何框架装上 1000+ 工具
from composio import Composio
from composio_openai_agents import OpenAIAgentsProvider

composio = Composio(provider=OpenAIAgentsProvider())
tools = composio.tools.get(user_id="user@acme.com", toolkits=["GITHUB", "SLACK"])
# → 直接塞给任何框架的 Agent 使用
```

支持的框架：OpenAI、OpenAI Agents、Anthropic、LangChain、LangGraph、LlamaIndex、Vercel AI SDK、Google Gemini、Google ADK、Mastra、Cloudflare Workers AI、CrewAI、AutoGen 等 **15+ 个**。

### Rube — MCP Server

Composio 还提供了 **Rube**，一个基于 MCP 协议的服务端，将 500+ 应用暴露为 MCP 工具。可以接入 Cursor、Claude Desktop、VS Code、Claude Code 等任何 MCP 兼容客户端。

## ✨ 技术亮点（5个）

### 1️⃣ 规模化的 Toolkit 生态：1000+ 预构建集成

这不是简单的 API wrapper。每个 Toolkit 包含：
- **Actions**（Agent 可主动调用的操作）：如 `create_issue`、`send_email`
- **Triggers**（可监听的事件）：如 `on_new_star`、`on_new_email`
- 自动从 OpenAPI Spec 生成 Schema
- 内置参数校验和错误处理

**为什么重要：** 大多数 Agent 框架的瓶颈不在推理能力，而在"能做什么"。Composio 直接把这个瓶颈干掉了。

### 2️⃣ 认证即服务（Auth-as-a-Service）

这是 Composio 最具差异化的能力：

- **OAuth 全流程托管**：授权跳转、Callback 处理、Token 存储、自动刷新
- **多认证模式**：OAuth2、API Key、Basic Auth、Bearer Token
- **连接状态管理**：用户授权了哪些应用、权限范围、过期时间
- **AgentID 机制**：每个 Agent 有独立证书，工具服务器可验证调用者身份

**为什么重要：** 让 Agent 调用需要认证的 API 从"工程噩梦"变成一行配置。开发者不需要自己实现 OAuth 流程。

### 3️⃣ 双语言 SDK + 15+ 框架的 Provider 矩阵

- TypeScript SDK（Node.js + 浏览器）
- Python SDK（3.10+）
- 每个主流框架都有专门的 Provider 包（`@composio/langchain`、`composio-crewai` 等）
- 支持自定义 Provider，可以对接任意框架

**为什么重要：** 这种"一个后端，N 个前端"的模式极大降低了采用门槛。无论你用什么框架开发 Agent，都能立刻获得 1000+ 工具能力。

### 4️⃣ 沙箱工作台（Sandbox Workbench）

- 隔离环境中执行工具调用
- 可在部署前安全测试 Agent 行为
- 防止 Agent 误操作生产环境数据
- 支持执行日志回放和分析

**为什么重要：** Agent 的工具调用是高风险操作（发邮件、创建工单、推送代码）。沙箱机制解决了"信任问题"，让开发者敢让 Agent 真正干活。

### 5️⃣ 工具语义搜索（Semantic Tool Search）

- Agent 不需要知道具体有哪些工具
- 通过自然语言意图匹配最相关的工具集
- 基于工具描述和参数 Schema 的语义理解
- 支持"模糊匹配"：Agent 说"帮我查一下代码问题"，自动映射到 `github_search_issues`

**为什么重要：** 当工具数量达到 1000+ 时，静态配置不现实。语义搜索让 Agent 能"按需发现"工具，类似人类在需要时才去找合适的工具。

## 💡 对楠楠的启发（3条具体可操作的）

### 启发一：为楠楠的工具体系引入"工具注册表"模式

**现状：** 楠楠目前通过 TOOLS.md 和 AGENTS.md 手动记录可用工具，新增工具靠改文件。
**借鉴 Composio：** 设计一个轻量版的 **工具注册表（Tool Registry）**：
- 每个工具用结构化 JSON 描述：名称、参数 Schema、适用场景、认证方式
- 新增工具只需注册一条记录，不需要改 prompt 或 skill 文件
- Agent 运行时根据任务意图搜索匹配的工具
- **第一步行动：** 在 `scripts/` 下建一个 `tool-registry.json`，把当前所有工具（飞书、智谱搜索、MiniMax API、TTS 等）结构化注册进去

### 启发二：实现"认证状态管理"的简化版

**现状：** API Key 散落在环境变量和 TOOLS.md 各处，Key 过期了经常才发现。
**借鉴 Composio 的 Connected Accounts 模型：**
- 建 `memory/auth-state.json`，集中记录所有凭证：
  ```json
  {
    "minimax": { "key_env": "MINIMAX_API_KEY", "status": "active", "quota_daily": {"image":120,"video":2}, "last_check": "2026-04-06" },
    "zhipu": { "key_env": "ZAI_API_KEY", "status": "active", "noted_limits": ["search_prime_5h_window"] },
    "feishu": { "type": "oauth", "account_id": "main", "status": "active" }
  }
  ```
- 心跳检查时顺带验证凭证有效性
- **第一步行动：** 创建 `auth-state.json` 并写一个 `scripts/check-auth.py` 脚本定期检测

### 启发三：学习 Provider Pattern 来解耦"能力"和"载体"

**现状：** 楠楠的能力（搜索、图像生成、语音合成等）和运行载体（OpenClaw）是强耦合的。
**借鉴 Composio 的 Provider 抽象：**
- 把核心能力抽象为独立的 **Capability 接口**（搜索、生成、识别、消息）
- 每个 Capability 可以有多种 **Backend 实现**（智谱搜索 / Brave搜索，MiniMax TTS / Edge TTS）
- 主系统只依赖 Capability 接口，不直接依赖具体 Backend
- 这意味着未来切换搜索提供商时，只需要换 Backend，不动任何业务逻辑
- **第一步行动：** 画一张楠楠的能力-后端映射图，标识哪些能力已经有多后端、哪些还是单点依赖
