# CoAct-1 — 将"写代码"作为核心动作的混合模态计算机操作多智能体系统

## 基本信息
- **来源:** CoAct-1: Computer-using Agents with Coding as Actions (arXiv preprint, v3 2026-02)
- **链接:** https://arxiv.org/abs/2508.03923 | GitHub: https://github.com/SalesforceAIResearch/CoAct-1
- **发布时间:** 2025年8月（v1），2026年2月更新至v3
- **研究机构:** Salesforce AI Research + USC + UW

## 解决什么痛点

传统 GUI Agent 面临一个根本性困境：**所有操作都必须通过点击、拖拽、键盘输入等低级 GUI 动作来完成**。即使引入了高层规划器（如 GTA-1 的 o3 规划器），底层执行仍然受限于 GUI 操作的固有缺陷——视觉定位模糊、长序列误差累积、单次误操作即可导致任务失败。举个具体例子：在 Excel 中筛选一个复杂表格、复制结果并保存为 CSV，纯 GUI Agent 可能需要 20+ 步精确操作；而用 Python 脚本只需几行 pandas 代码。同样，在嵌套目录中查找所有图片、调整尺寸、打包压缩——这类"后端型"任务通过 GUI 完成既脆弱又低效。

现有方案要么是纯端到端 VLM（如 UI-TARS、OpenCUA），要么是规划器+GUI执行器的两阶段架构（如 GTA-1、Agent-S2）。它们共同的问题是：**动作空间被锁死在 GUI 层面**，无法利用程序化操作的精度和效率优势。即使规划再好，底层仍需在密集 UI 元素中"盲人摸象"。CoAct-1 的核心洞察是：计算机操作不应只有一种模式——有些事该用眼睛和鼠标做，有些事该用代码直接做。

## 核心架构与技术

CoAct-1 采用**三智能体协作架构**：Orchestrator（编排器）、Programmer（程序员）、GUI Operator（GUI 操作员）。这是一个层次化的元策略系统 π = (π_orch, π_code, π_gui)，其中 Orchestrator 充当元策略 π_orch，负责将用户的高层目标分解为子任务，并动态判断每个子任务应该交给 Programmer 还是 GUI Operator 执行。

**Orchestrator（编排器）** 是系统的"大脑"，基于 LLM 实现（论文中使用 OpenAI o3）。它接收用户自然语言目标 G 和历史上下文 H_t，输出结构化的子任务分解和模态分配决策。关键设计在于其**动态路由能力**：不是预先固定哪些任务走代码路径，而是根据当前子任务的语义特征实时判断。例如"打开设置修改分辨率"→ 分配给 GUI Operator；"批量重命名文件夹中所有 .jpg 文件" → 分配给 Programmer。这种动态分配避免了硬编码规则带来的泛化瓶颈。

**Programmer（程序员）** 是 CoAct-1 最具创新性的组件。它能够编写并直接执行 Python 或 Bash 脚本与操作系统交互，形成一个 **coding-observation 循环**：编写代码 → 执行 → 观察输出/文件变化 → 根据反馈迭代修改。这与传统的"调用预定义 API 工具"有本质区别——Programmer 拥有自由编程能力，可以组合任意系统命令和 Python 库来解决开放性问题。例如处理 CSV 数据时可以直接 import pandas，操作文件系统时可以用 glob/shutil，甚至可以用 subprocess 调用其他 CLI 工具。代码执行环境与 GUI 环境共享同一台虚拟机，确保文件系统和状态的一致性。

**GUI Operator（GUI 操作员）** 则是基于 VLM 的传统 GUI Agent，负责需要视觉交互的前端操作：点击按钮、填写表单、导航菜单、拖拽元素等。它接收屏幕截图作为输入，输出坐标级或元素级的 GUI 动作。论文中基于 OpenAI Computer Use API (computer-use-preview) 实现，但架构上可替换为任意 VLM-based GUI Agent。

三者的协作流程如下：用户给出目标 → Orchestrator 分解为有序子任务列表 → 对每个子任务判断模态 → Programmer 用代码完成后端操作（如数据处理、文件管理）→ GUI Operator 接手前端交互（如确认弹窗、点击保存按钮）→ 两者交替推进直到任务完成。整个过程中，Orchestrator 持续监控进度，必要时重新规划。

## 技术亮点

### 1. 混合动作空间 A = A_GUI ∪ A_Code —— 范式层面的突破

这不是简单的"给 GUI Agent 加个代码工具"，而是从根本上扩展了 Computer Use Agent 的动作空间定义。传统 GUI Agent 的动作空间 A_GUI 仅包含 {click, type, scroll, drag, key_press} 等低级操作；CoAct-1 引入 A_Code = {Python script, Bash script} 作为一等公民。这意味着 Agent 不再被迫把所有问题都映射到"点哪里"——它可以选择"直接写段代码解决"。这种范式转换的价值在于：对于程序化可表达的任务（文件操作、数据转换、批量处理），代码路径的成功率接近确定性（只要 LLM 写对代码），而 GUI 路径则面临 O(n) 步骤中每一步都有失败概率的累积风险。

### 2. 动态子任务路由机制 —— 元策略层面的智能分发

Orchestrator 的路由决策不是基于规则的 if-else，而是由 LLM 基于子任务语义进行推理。论文展示了这一机制的精妙之处：同一类应用内的不同操作可能走不同路径。以 VS Code 为例——"安装某个 extension" 走 GUI 路径（需要在 Extension Marketplace 中搜索和点击），而"在项目中搜索所有 TODO 注释并统计数量" 则走 Code 路径（直接 grep + wc）。这种细粒度的动态分配使得系统能够在每个子任务上选择最优执行模态，而非一刀切地全部走 GUI 或全部走代码。

### 3. Coding-Observation 迭代循环 —— 自适应代码执行

Programmer 不是"一次性写完就跑"的模式，而是支持多轮编码-观察循环。第一轮代码执行后，Programmer 可以看到 stdout/stderr 输出、文件系统变化、甚至截图（如果代码触发了 GUI 变化），然后据此修正代码。这类似于人类程序员的 debug 过程：写代码 → 跑一下看结果 → 改 bug → 再跑。这种反馈机制大大提高了复杂编程任务的完成率，也使 Programmer 能够处理那些需要探索性编程的任务（如不确定文件结构的场景下先用 ls 探查）。

## 基准测试表现

CoAct-1 在两个主流基准测试上均取得 SOTA：

**OSWorld 基准测试：**
- 整体成功率：**60.76%**（此前 SOTA 为 Agent S2.5 的 55.98%）
- 平均完成步数：**10.15 步**（GTA-1 需要 ~15 步，提升约 32%）
- 分类别表现：
  - Calc（电子表格）：70.21%
  - VS Code：78.26%
  - 多应用协作任务：47.88%

**WindowsAgentArena 基准测试：**
- 整体成功率：**52.50%**

效率提升尤为显著——平均步数从 15 降至 10.15，意味着每个任务节省了近 1/3 的交互轮次。这在实际部署中直接转化为更低的 API 成本和更快的任务完成速度。值得注意的是，性能增益最明显的正是那些"程序化友好"的任务类别（Calc、VS Code、文件管理），验证了混合动作空间设计的有效性。

## 与同类对比

**vs. GTA-1（Google）：** GTA-1 同样使用 o3 作为规划器，但底层完全依赖 GUI 执行 + 测试时多候选采样投票。CoAct-1 在相同规划器基础上增加了 Code 路径，OSWorld 上领先约 5 个百分点且步数减少 32%。GTA-1 的优势是不需要代码执行环境，部署更简单；劣势是无法利用程序化操作的效率优势。

**vs. Agent S/S2/S2.5：** Agent S 系列采用 compositional generalist-specialist 架构，通过多个专家 Agent 协作完成任务，但所有专家都是 GUI-based。CoAct-1 的 Programmer 本质上是一个新的"专家角色"，专门处理后端操作。Agent S2.5 达到 55.98%，CoAct-1 达到 60.76%，差距主要来自程序化任务上的优势。

**vs. 纯 Code Agent（如 Open Interpreter）：** 纯代码 Agent 无法处理必须通过 GUI 完成的操作（如点击特定按钮、处理图形化向导）。CoAct-1 的混合架构保留了 GUI 能力作为必要补充，因此适用范围远大于纯代码方案。

**劣势分析：** (1) 需要维护代码执行沙箱环境，安全性和隔离成本更高；(2) Orchestrator 的路由决策本身可能出错——错误地将 GUI 任务分配给 Programmer 或反之；(3) 对不熟悉编程的 LLM 来说，Programmer 的代码质量可能不稳定。

## 对我们的启发

1. **Agent 动作空间应按需扩展：** 我们在设计 Agent 系统时，不应局限于单一交互模式。对于 OpenClaw 这类通用 Agent 平台，可以考虑让 Agent 在"浏览器操作"、"Shell 命令"、"API 调用"、"代码执行"等多种动作空间之间动态选择，而非强制所有任务都走同一条路。

2. **编排层的价值被低估：** CoAct-1 证明了一个好的 Orchestrator（即使是基于 prompt engineering 的 LLM 调度器）能带来显著收益。我们的多 Agent 系统可以借鉴这种"元策略 + 专业执行者"的分层设计，让调度层专注于"做什么"和"谁来做"，而非陷入具体执行的细节。

3. **代码即动作的实战可行性：** 对于文件处理、数据转换、批量操作等"脏活累活"，让 Agent 直接写脚本执行比模拟人类 GUI 操作高效得多。我们可以在工作流设计中内置"代码捷径"——当检测到任务适合程序化解决时，自动切换到代码执行模式。

## 综合评分: 8.5/10

**评分理由：** CoAct-1 在范式层面提出了"编码作为动作"这一简洁而有力的新思路，实验数据扎实（OSWorld 60.76% SOTA + 步数显著下降），工程实现完整且有开源代码。扣分点：(1) 依赖强模型（o3）作为 Orchestrator，成本较高；(2) 路由决策的失败模式和恢复策略讨论不足；(3) 安全性方面对代码执行的沙箱隔离要求严格，限制了部分场景的部署。总体而言，这是 2025 年 GUI Agent 领域最具实用价值的创新之一，为"如何让 Agent 更像人类专家而非只会点鼠标的机器人"提供了清晰的答案。
