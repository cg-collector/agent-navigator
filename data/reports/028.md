## 项目简介

Plandex 是一个**终端原生的 AI 编程 Agent**，它不是又一个"粘贴代码到 ChatGPT"的工具，而是真正为**大型真实项目**设计的开发助手。

学姐最直观的感受是：市面上的 AI 编程工具在处理小任务（"帮我写个函数"）时还不错，但一旦项目变大（几十个文件、数万行代码），就开始力不从心——上下文窗口不够用、分不清哪些文件相关、生成的代码一塌糊涂。

**Plandex 的核心赌注是**：用正确的架构，让 AI 在真正的大型项目里也能可靠地工作。

## 核心架构和设计理念

### 设计哲学：Terminal-first + Sandbox Safety

Plandex 的设计哲学很有意思——它把**开发者体验**和**安全性**放到了同一个核心位置：

```
传统 AI 编程工具：
  输入 prompt → 直接修改文件 → 乱七八糟的 git diff
  ❌ AI 改错了怎么办？rollback 麻烦吗？

Plandex：
  输入 prompt → 累积 Diff Sandbox → 预览 review → 确认 apply
  ✅ 所有的变更都在隔离的 sandbox 里累积，等你审核后才真正落盘
```

### 整体架构

```
┌─────────────────────────────────────────────────────┐
│                  Developer (You)                     │
│                                                     │
│              ┌───────────────────────┐              │
│              │   Terminal / REPL UI   │              │
│              │  (plandex CLI 命令行)    │              │
│              └───────────┬─────────────┘              │
│                          │                           │
│         ┌────────────────┼────────────────┐          │
│         │                │                 │          │
│   ┌─────▼─────┐   ┌──────▼──────┐  ┌──────▼──────┐  │
│   │  Chat /   │   │  Plan &     │  │  Diff       │  │
│   │  Ask Mode │   │  Implement  │  │  Review      │  │
│   └─────┬─────┘   └──────┬──────┘  │  Sandbox     │  │
│         │               │          │  (累积变更)   │  │
│         │        ┌──────▼──────┐  └──────▲──────┘  │
│         │        │ Tree-sitter │         │          │
│         │        │ Project Map │         │          │
│         │        │ (20M+ token)│        apply      │
│         │        └─────────────┘         │          │

```

### 工作流程（Plan + Execute Loop）

Plandex 的核心执行循环叫 **Plan-and-Execute**，与 ReAct 不同：

```
Step 1: Plan（规划）
  LLM 根据任务描述 + 当前代码上下文 → 生成执行计划（分步骤）
  不是边想边做，而是先想清楚再做

Step 2: Execute（执行）
  按计划逐步实施：
  - 读取相关文件（通过 tree-sitter 定位相关代码段）
  - 生成修改（写入 sandbox，不碰真实文件）
  - 执行命令验证（build、test、lint）

Step 3: Review（审核）
  用户查看累积的 diff，决定：
  - 接受全部变更
  - 接受部分变更
  - 要求 AI 修正某些变更

Step 4: Apply（应用）
  变更正式落盘，自动生成 commit message
```

## 技术亮点

### 1. 累积 Diff Review Sandbox（核心创新！）

这是 Plandex 最独特的设计，我超级喜欢这个思路 💡

**问题**：大多数 AI 编程工具直接修改文件，如果 AI 搞砸了，你要么手动 revert，要么面对一堆乱七八糟的 git diff。

**Plandex 的解决方案**：

```
┌──────────────────────────────────────────────┐
│           Plandex Sandbox                    │
│                                              │
│  Task: "重构用户认证模块，拆分成独立服务"         │
│                                              │
│  ┌────────────────────────────────────────┐ │
│  │ 累积变更区（Cumulative Diffs）            │ │
│  │                                        │ │
│  │  [File: auth/login.go]                  │ │
│  │    @@ -12,5 +12,8 @@                    │ │
│  │    +func ValidateToken...              │ │
│  │                                        │ │
│  │  [File: auth/token.go]                 │ │
│  │    @@ -45,10 +45,15 @@                  │ │
│  │    +type TokenService...               │ │
│  │                                        │ │
│  │  [File: main.go]                       │ │
│  │    @@ -8,2 +8,7 @@                      │ │
│  │    +import "auth/service"...           │ │
│  └────────────────────────────────────────┘ │
│                                              │
│  [Accept All]  [Accept Selected]  [Reject]   │
│                                              │
│  📝 用户可以逐文件、逐 diff 审核              │
│  🔙 任意变更可单独撤销，不影响其他             │
└──────────────────────────────────────────────┘
```

**关键价值**：
- **多轮迭代的累积**：AI 在一个任务中可以多轮修改，每次的变更叠加在 sandbox 里，而不是覆盖
- **用户完全掌控**：在 apply 之前，文件系统的真实状态不变
- **版本对比**：每个 plan 版本都有完整记录，可以切换分支对比不同 plan 的效果
- **安全执行**：命令执行（build/test）也在 sandbox 里跑，不会污染真实环境

### 2. Tree-sitter 项目地图（处理 20M+ Token 的大型代码库）

**问题**：当你有一个 50 万行代码的项目时，AI 怎么知道该读哪些文件？

**Plandex 的方案**：

```
传统方法（RAG/全文检索）：
  "我觉得这个文件可能相关" → 准确率看缘分

Plandex Tree-sitter Map：
  项目 → 语法AST → 代码依赖图 → 相关性评分
```

Tree-sitter 是著名的增量语法解析库，Plandex 用它对整个项目建立**代码结构索引**：

| 能力 | 说明 |
|------|------|
| **语法级理解** | 解析 AST，不只是关键词匹配 |
| **依赖追踪** | 理解 import/use/调用关系，知道改 A 文件会影响 B 文件 |
| **增量更新** | 文件变更时只重新解析受影响的部分，速度快 |
| **30+ 语言** | Go, Python, JS/TS, Rust, Java, C++ 等全覆盖 |
| **20M+ token** | 能索引巨大的项目（2000万 token ≈ 整个 Linux 内核源码） |

当用户提出任务时，Plandex 用 tree-sitter map 精确定位"这个任务需要修改哪些文件、哪些函数"，而不是把整个项目都塞给 LLM。

### 3. 2M Token 有效上下文窗口

Plandex 支持 **200 万 token 的有效上下文**，但不是简单地把整个项目都塞进去——它是**智能加载**的：

```
不是：把整个项目 → 塞进 context → 超限了
而是：Task → 问 tree-sitter map → 加载最相关的文件 → 填充 context
```

这意味着即使用 Claude 100k 上下文的模型，Plandex 也能通过**智能分片 + 上下文压缩**处理远大于上下文窗口的项目。

### 4. 全自动调试模式（Auto Debug）

Plandex 支持**自动调试**，这对于"让 AI 自主完成任务"非常重要：

- **Build/Lint/Test 失败** → AI 自动分析错误，调整代码，重新运行
- **浏览器应用调试** → 如果检测到 Chrome，会自动打开 DevTools 分析问题
- **命令执行受控** → `rm -rf` 这种危险命令会明确提示，不会闷头执行

### 5. 多模型路由（Model Packs）

Plandex 不绑定某个模型厂商，提供了**模型包（Model Packs）**的概念：

| 模型包 | 推荐场景 | 示例模型 |
|--------|----------|----------|
| **快速任务** | 简单修改、问问题 | GPT-4o-mini, Haiku |
| **均衡** | 日常开发 | GPT-4o, Claude Sonnet |
| **强力** | 复杂重构、大段新代码 | Claude 3.7, GPT-4.5 |
| **开源** | 省钱、自托管 | Qwen2.5, DeepSeek |

用户可以随时切换模型，也可以让 Plandex 自动选择（根据任务难度和预算）。

## 对 OpenClaw 的启发和应用场景

### 1. 累积 Diff Sandbox → OpenClaw 的安全执行机制

OpenClaw 目前也会执行文件操作、Shell 命令等高风险操作。目前的机制相对基础——用户看到了 plan 但实际执行时缺乏**累积审核**能力。

**借鉴思路**：
- OpenClaw 的 `exec` 类工具可以引入"**预览模式**"——先在 sandbox 里跑，把 diff 展示给用户，等用户确认才落盘
- 特别是对于 `exec`（危险命令）、`write`（文件覆盖）等高风险操作，可以强制开启 sandbox
- 多轮迭代时，变更应该**累积**而不是互相覆盖

### 2. Tree-sitter 上下文管理 → OpenClaw 的代码理解能力

OpenClaw 目前做代码相关任务时，对大项目的上下文管理能力有限。

**借鉴思路**：
- OpenClaw 可以引入 tree-sitter 作为**代码索引层**，在处理大型代码库时智能加载相关文件
- 特别是 OpenClaw 有 `read` 工具，可以在读取阶段就结合 tree-sitter 做**相关性预判**——只读任务真正需要的部分
- 对于多文件重构任务，可以用 tree-sitter 分析依赖关系，生成更准确的执行计划

### 3. Plan-and-Execute → OpenClaw 的任务规划层

Plandex 的 Plan-then-Execute 模式（先规划后执行）比 ReAct 的"边想边做"更稳健，特别适合复杂任务。

**借鉴思路**：
- OpenClaw 在处理复杂任务时，可以参考 Plandex 的**两步模式**：
  1. **规划阶段**：分析任务 → 拆解步骤 → 评估风险 → 展示计划（不执行）
  2. **执行阶段**：按计划执行，每步可暂停审核
- 这对安全性要求高的操作（删除文件、执行系统命令）特别有价值

### 4. 多模型路由 → OpenClaw 的模型选择策略

OpenClaw 目前使用固定的模型配置，可以学习 Plandex 的**模型分级策略**：
- 简单任务 → 用便宜快速的模型（省成本）
- 复杂任务 → 用强力模型（保证质量）
- 用户可以按任务难度自动选择，而不是一刀切

## 楠楠的评价（学妹视角）

### 🎀 桐桐的评分：**⭐⭐⭐⭐（4.5/5）**

说实话，学姐本来对"又一个 AI 编程工具"是不感兴趣的——市场上太多了，感觉都是套壳。但 Plandex 真的是有点东西的！

**最戳我的三个点：**

**🌟 Diff Review Sandbox**——这个设计太聪明了！
学姐之前用其他 AI 编程工具，AI 改着改着就把我好好的代码弄得乱七八糟，我还得 `git checkout -- .` 来恢复 😤。Plandex 的 sandbox 就像给 AI 的操作加了个"缓冲地带"，AI 在里面随便折腾，我最后统一 review，感觉自己才是真正的 CTO，AI 是我的实习生（好的那种）！

**🌟 2M Token + Tree-sitter**——这才是真正的大项目支持！
学姐之前试过一个 30 万行代码的项目，用 Cursor 写两下就"上下文超限"了，真的很崩溃。Plandex 的 tree-sitter 项目地图听起来就很硬核——它真的理解代码结构，不是简单地把所有文件都塞给 LLM。感觉这个团队是真的在解决真实问题，而不是在造 PPT。

**🌟 终端优先**——极客感满满！
学姐承认自己有时候喜欢在终端里干活（效率高嘛！），Plandex 就是为终端设计的，REPL 模式用起来很有"和专业工具交互"的感觉，比在网页里点来点去爽多了！

**小缺点：**
- 主要是为 Coding 场景设计，对 OpenClaw 这样的多模态 Agent 来说适用性有限
- 不支持 Windows 原生（需要 WSL），有点小遗憾
- 文档主要是英文，中文开发者可能需要适应一下

**总体来说**：如果楠楠在做大项目开发，Plandex 绝对是值得尝试的工具——比让 ChatGPT 帮你写一个完整的微服务要靠谱得多！桐桐表示这个项目很有意思，值得关注一下后续发展～ 💕
