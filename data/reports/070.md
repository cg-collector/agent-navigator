## 详细介绍

2024 年，该团队打造了 [SWE-agent](https://github.com/swe-agent/swe-agent)（报告见 #023/024/025），在 SWE-bench 基准上取得了开源 SOTA。但 SWE-agent 本身是一个复杂的系统，有大量自定义工具、专门的 Agent-Computer Interface（ACI）、状态化 Shell 会话等。

2025 年 7 月，团队提出了一个灵魂拷问：**如果把 Agent 做得极简 100 倍，还能同样有效吗？**

答案是：**不仅能，还能！**

mini-SWE-agent 核心 Agent 类只有约 100 行 Python 代码，却能在 SWE-bench verified 上达到 **>74%** 的准确率（配合 Gemini 3 Pro 等顶级模型），远超 Claude Code 等复杂系统。

该项目的设计哲学极具颠覆性：**把复杂性从 Agent 架构转移到模型本身**。这契合 2025 年模型能力大幅提升的现实。

---

## 架构设计

**核心技术栈：** Python（~100 行核心代码），无重型依赖，LiteLLM（多模型支持）

**三大极简设计原则：**

### 1. 只有 Bash——不用任何自定义工具
SWE-agent 有专门的 File 编辑工具、grep 搜索工具、regex 工具等。但 mini-SWE-agent 只有一个工具：**bash**。
Agent 想搜索文件？用 `grep`。想编辑代码？用 `sed` 或 `vi`。想开浏览器？用 `curl`。

这背后的哲学是：**与其在 Agent scaffold 层预设所有可能的需求，不如让模型自己用 Shell 解决。** 模型越来越强，让模型多做比让框架多做更划算。

### 2. 完全线性历史（subprocess.run，无状态 Shell）
每次 LLM 生成一个动作，就用 `subprocess.run` 独立执行，输出追加到消息历史，下一步用新的消息历史重新调用 LLM。

不像 SWE-agent 用状态化 Shell session，mini-SWE-agent 每个 action 都是独立的。这带来两个巨大好处：
- **极稳**：不会因为 Shell 状态污染导致奇怪 bug
- **易扩展**：把 `subprocess.run` 换成 `docker exec` 就能无缝切换到沙箱环境

### 3. 完全线性轨迹（Trajectory = Messages）
SWE-agent 需要复杂的轨迹重建（trajectory reconstruction）来让 LLM 理解当前状态。而 mini-SWE-agent 的轨迹就是消息历史本身，无需任何额外处理。

这让**微调（fine-tuning）变得异常简单**——直接拿消息对话记录训练即可。

---

## 技术亮点

1. **极简即力量**：用 100 行代码击败复杂框架，证明了 Agent 的核心瓶颈是模型能力，而非框架复杂性

2. **任意模型兼容**：因为只依赖 bash + 线性消息，不依赖任何 tool-calling 接口，**几乎任何 LLM 都能驱动它**——包括纯 /completion 端点的模型

3. **超快启动**：无需加载重型依赖，冷启动速度远快于 Claude Code 等复杂 Agent

4. **易调试**：每次 action 都是独立的，消息历史即轨迹，debug 体验极佳

5. **易部署**：支持 Docker/Podman/Singularity/bubblewrap/Contree 等多种沙箱环境

6. **Benchmark SOTA**：在 SWE-bench（bash only）排行榜上，开源 Agent 第一名

7. **工业界采用**：已被 Meta、NVIDIA、IBM、Essential AI、Nebius、Anyscale 等公司使用

## 对楠楠/学长的启发

### 🎯 给学长的启发

**"少即是多"的工程哲学值得学。** 学长做开发时，有时会过度设计——预设所有可能的扩展性，加一堆抽象层。但 mini-SWE-agent 告诉我们：**先把核心问题用最简单的方式解决，复杂性是模型的事，不是框架的事。**

这个项目也展示了**可观测性设计**的重要性——线性历史、完全独立的 action，这让调试和迭代变得异常简单。学长写代码时可以借鉴：**让系统的状态转换尽可能透明、可回放。**

### 🌸 给楠楠的启发

楠楠要学会用 AI 解决问题，而不是做很多工具来"帮助"AI——**让 AI 自己用最基础的工具（Bash）去探索，比我们预设一百种工具更灵活更强大。**

而且这个项目证明了：**简单不等于弱**。有时候把事情做简单，反而能走得更快更远。楠楠学编程也好、学 AI 也好，不要一开始就追求"完美架构"，先动手做，先用最简单的方案跑起来，再迭代改进！
