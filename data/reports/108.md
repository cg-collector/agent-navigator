## 架构设计

### 三层架构总览

Agno 采用 **Framework → Runtime → Control Plane** 三层分离架构：

```
┌─────────────────────────────────────────────┐
│           Control Plane (AgentOS UI)         │  ← 浏览器管理面板
│    Chat / Trace / Monitor / Evaluate         │
├─────────────────────────────────────────────┤
│            Runtime (FastAPI Backend)          │  ← 无状态生产服务
│   Stateless API / Session Scoping / Auth      │
├─────────────────────────────────────────────┤
│              Framework (SDK)                  │  ← 核心开发框架
│  Agent / Team / Workflow / Memory / Tools     │
└─────────────────────────────────────────────┘
```

### 核心模块

#### 1. Agent（智能体）— 最核心基类

```python
from agno.agent import Agent
from agno.models.anthropic import Claude
from agno.db.sqlite import SqliteDb
from agno.tools.mcp import MCPTools

agent = Agent(
    name="Agno Assist",
    model=Claude(id="claude-sonnet-4-6"),
    db=SqliteDb(db_file="agno.db"),          # 持久化层
    tools=[MCPTools(url="...")],               # 工具集
    knowledge=Knowledge("company_docs"),       # 知识库
    enable_memories=True,                      # 用户记忆开关
    add_history_to_context=True,               # 历史注入上下文
    num_history_runs=3,                        # 历史轮数限制
    input_guardrails=[...],                    # 输入护栏
    output_guardrails=[...],                   # 输出护栏
    markdown=True,
)
```

**Agent 执行流程（Run Loop）：**
1. **Context Assembly** — 组装完整上下文：system prompt + instructions + chat history + user memories + session state + knowledge retrieval results
2. **LLM Call** — 发送给模型，返回文本或 tool_call
3. **Tool Execution** — 如果是 tool_call，执行工具并将结果回注
4. **Loop** — 重复步骤 2-3 直到模型输出最终文本
5. **Memory Update** — 运行后自动提取用户记忆并持久化
6. **Session Persist** — 将整个会话状态写入数据库

#### 2. Memory（记忆系统）— 三层分离设计

这是 Agno 最精妙的设计之一，将"记忆"拆为三个独立概念：

| 层级 | Agno 名称 | 存储方式 | 用途 |
|------|-----------|----------|------|
| **Session Storage** | `Storage` | DB 表（sessions） | 短期：聊天历史 + 会话状态，跨 run 持久化 |
| **User Memories** | `Memory` | DB 表（memories） | 长期：用户偏好、事实（如"用户喜欢滑雪"），跨会话共享 |
| **Session Summaries** | `Summary` | DB 表（summaries） | 压缩：长对话的摘要表示，防止上下文爆炸 |

**关键实现细节：**
- User Memories 通过 LLM 自动从对话中提取（self-learning），不是简单的向量搜索
- 每个 memory 关联 `user_id`，支持多用户隔离
- Memory 在每次 run 时通过 relevance 检索注入 context，而非全量加载
- `add_history_to_context=True` 时自动将最近 N 轮对话加入 prompt

#### 3. Knowledge（知识库）— RAG 原生集成

```python
from agno.knowledge.knowledge import Knowledge
from agno.vectordb.lancedb import LanceDb
from agno.knowledge.embedder.openai import OpenAIEmbedder

knowledge = Knowledge(
    vector_db=LanceDb(
        uri="./lancedb",
        table_name="docs",
        search_type=SearchType.hybrid,       # 混合搜索（向量+关键词）
        embedder=OpenAIEmbedder(id="text-embedding-3-small")
    ),
)
knowledge.add_content(url="https://example.com/doc.pdf")
```

支持多种 Vector DB 后端：LanceDb（本地首选）、Milvus、Qdrant、Pgvector、Chroma 等。

#### 4. Team（多智能体协作）

```python
from agno.team import Team

team = Team(
    name="Research Squad",
    members=[web_researcher, finance_agent, coding_agent],
    model=Claude(id="claude-sonnet-4"),       # Team coordinator 模型
    db=Postgres(conn_string),
    instructions="Collaborate for deep research",
    enable_memories=True,
    mode="coordinate",                         # 协调模式
)
```

**Team 支持三种协调模式：**

| 模式 | 行为 |
|------|------|
| `route` | Leader 作为简单路由器，分析任务后分发给某个 member 执行 |
| `coordinate` | Leader 作为协调者，可以和多轮和多个 member 交互后再汇总 |
| `custom` | 自定义编排逻辑 |

#### 5. Workflow（工作流编排）

```python
from agno.workflow import Workflow
from agno.workflow.steps.router import Router

workflow = Workflow(
    name="Social Media Autopilot",
    description="Generate & publish engaging posts",
    db=Postgres(conn_string),
    steps=[
        Router(
            selector=select_channel,
            choices=[x_agent, linkedin_agent],  # 路由到不同 agent
        ),
        publish_post,                            # 函数步骤
    ],
)
```

Workflow 的 Step 可以是：Agent、Team、普通函数、Router（条件分支）。这提供了比纯 Agent 更确定性的执行路径。

#### 6. Guardrails（运行时护栏）

```python
from agno.guardrails.input_guardrail import InputGuardrail
from agno.guardrails.output_guardrail import OutputGuardrail

class PiiCheck(InputGuardrail):
    def validate(self, agent, input) -> GuardrailResult:
        if contains_pii(input):
            return GuardrailResult(blocked=True, msg="检测到PII信息")
        return GuardrailResult(blocked=False)

agent = Agent(
    ...,
    input_guardrails=[PiiCheck()],
    output_guardrails=[ToxicContentCheck()],
)
```

护栏在 Run Loop 内部执行，不是外部中间件——这意味着它可以访问完整的 agent context 做判断。

#### 7. AgentOS（生产运行时）

```python
from agno.os import AgentOS

agent_os = AgentOS(
    description="Powerful Agent System",
    agents=[knowledge_agent, support_agent],
    teams=[research_team],
    workflows=[sm_workflow],
    interfaces=[Slack(), AISdk(), AGUI()],    # 多渠道接入
    tracing=True,
)
app = agent_os.get_app()  # → FastAPI 应用
```

AgentOS 将所有 Agent/Team/Workflow 统一暴露为：
- RESTful API（50+ 端点）
- WebSocket（流式响应）
- 多渠道接口（Slack、Telegram、WhatsApp、Discord 等）

### 数据库 Schema（核心表）

```
sessions        — 会话历史 & 状态（session_type: agent/team/workflow）
memories        — 用户长期记忆（entity, memory, category）
summaries       — 会话摘要
runs            — 每次执行的追踪记录
traces          — 详细执行链路（用于调试和审计）
knowledge_entries — 向量化知识条目
```

所有数据存在**你自己的数据库**中（SQLite 开发 / PostgreSQL 生产），不经过第三方。

## 技术亮点

### 亮点 1：三层记忆架构 — 解决 Agent 记忆的根本性问题

大多数框架把"记忆"混为一谈。Agno 的 Session Storage / User Memories / Session Summaries 三分法直击痛点：

- **Session Storage** 解决多轮对话问题（类似 OpenClaw 的 daily notes）
- **User Memories** 解决个性化问题（类似楠楠的 MEMORY.md，但自动化）
- **Summaries** 解决上下文窗口限制

**代码级洞察：** Memory 的自动提取是在 Run Loop 的 post-run 阶段触发的，用轻量级 LLM 调用从当前对话中抽取 entity-fact 对，然后 upsert 到 memories 表。检索时不是暴力全量匹配，而是做 relevance scoring 后取 top-K 注入 context。这个设计比简单向量存储高明得多——它区分了"发生了什么"（session）和"学到了什么"（memory）。

**对我们的启发：** 楠楠目前的记忆系统是文件级的（daily notes + MEMORY.md），没有自动化的 user memory 提取。可以参考 Agno 的做法，在对话结束后自动抽取关键信息存入结构化记忆。

### 亮点 2：529× 实例化速度优势 — 极简设计哲学

Agno 官方 benchmark 数据：
- Agent 实例化速度：**529× 快于 LangGraph**，57× 快于 PydanticAI
- 内存占用：**24× 低于 LangGraph**

原因在于其**极度精简的类层次结构**。核心 `Agent` 类是一个扁平的 Pydantic model，不像 LangChain 有 AbstractChain → Chain → BaseChatModel 这样深的继承树。工具系统也是扁平的函数装饰器模式：

```python
from agno.tools.tool import tool

@tool
def get_weather(city: str) -> str:
    """Get current weather for a city."""
    return f"Weather in {city}: 22°C, sunny"

agent = Agent(tools=[get_weather])
```

任何 Python 函数加一个 `@tool` 装饰器就能变成 agent 工具，带自动类型推断和 JSON Schema 生成。同时原生支持 MCP 协议工具。

**对我们的启发：** OpenClaw 的 skill/tool 系统可以借鉴这种"函数即工具"的轻量模式。当前 SKILL.md 驱动的方式虽然灵活但偏重，对于简单工具可以更轻量。

### 亮点 3：Team 的三种协调模式 + 共享上下文

多数多智能体框架只提供一种协作模式（如 CrewAI 的 Process 类只有 sequential/hierarchical）。Agno 的 Team 提供 route/coordinate/custom 三种模式，覆盖了从简单路由到深度协作的全谱系：

- **Route 模式**：O(1) 单次分发，适合"专家路由"场景
- **Coordinate 模式**：Leader 可以多次和 member 交互，适合需要讨论和迭代的研究任务
- **Custom 模式**：完全自定义编排逻辑

关键是 Team 中**所有成员共享同一个 db 和 memory 系统**，member A 产生的记忆 member B 可以直接使用。这解决了多智能体系统中常见的"信息孤岛"问题。

**对我们的启发：** 当前的 subagent 机制是简单的派生-回收模式，没有 agent 间的协作能力。如果未来需要多 agent 协作（比如让"研究型楠楠"和"编码型楠楠"合作），Team 模式的思路值得参考。

### 亮点 4：Guardrails 作为一等公民嵌入执行循环

大多数框架的"安全检查"是外部中间件或 prompt 层面的。Agno 把 InputGuardrail 和 OutputGuardrails 直接嵌入 Agent 的 Run Loop：

```
User Input → InputGuardrails (可阻断) → Context Assembly → LLM Call
→ Tool Execution → OutputGuardrails (可阻断) → Response → Memory Update
```

护栏是 Python 类，有完整的 agent context（包括 user_id、session、history），可以做细粒度的权限控制、内容审查、PII 过滤等。而且支持 `human_in_the_loop`——当护栏不确定时可以暂停等待人工审批。

**对我们的启发：** OpenClaw 目前的安全机制主要在 AGENTS.md 的规则层面（软约束）。可以参考 Agno 做硬编码的 guardrail 机制，特别是在敏感操作（发消息、删除文件、外发数据）前增加可编程的安全检查点。

### 亮点 5：AgentOS — 从"写 Agent"到"运营 Agent 系统"的跨越

这不是一个框架特性而是一个**运维范式**。AgentOS 提供：
- 统一的 FastAPI 后端，所有 Agent/Team/Workflow 变成 API
- 内置 Tracing（每次执行的完整链路追踪）
- Web UI 用于 Chat、Monitor、Memory 管理、Evaluation
- 多渠道接入（Slack、Telegram、WhatsApp、Discord、AUI SDK）
- JWT + RBAC + request-level isolation

**核心差异点：** 大多数框架停留在"帮你写 Agent 代码"的阶段，Agno 往前走了一步——**帮你把 Agent 变成生产基础设施**。`AgentOS(agents=[...], teams=[...], workflows=[...]).get_app()` 一行代码就把整个系统变成可部署的 API 服务。

**对我们的启发：** OpenClaw 已经有了 gateway + channel 的运行时基础，但在可观测性（tracing）、评估（evaluation）、多租户隔离方面可以向 AgentOS 学习。特别是 tracing 和 evaluation 是目前完全缺失的能力。

## 对 OpenClaw / 楠楠 的启发总结

### 可直接借鉴的点

| 方面 | 现状 | Agno 启发 |
|------|------|-----------|
| **记忆自动化** | 手动维护 MEMORY.md | 自动从对话抽取 user memories |
| **知识库集成** | 无结构化 RAG | Knowledge + VectorDB 原生集成 |
| **护栏机制** | AGNS.md 软规则 | Run Loop 内嵌硬编码 Guardrails |
| **可观测性** | 基础日志 | 完整 Tracing + Evaluation |
| **工具注册** | SKILL.md 驱动 | @tool 装饰器轻量模式 |
| **多 Agent 协作** | Subagent 派生 | Team route/coordinate 模式 |

### 架构哲学层面的思考

Agno 最值得学习的不是某个具体功能，而是它的**设计哲学**：

1. **"Memory ≠ Storage ≠ Summary"** — 区分"记住对话"、"记住用户"、"压缩上下文"三个问题，各自用最优方案解决
2. **"Framework thin, Runtime thick"** — SDK 层保持极薄（一个 Agent 类搞定），重量级功能（部署、监控、多渠道）放在 Runtime 层
3. **"Your data, your cloud"** — 所有数据存用户自己的数据库，框架本身无状态，天然适合水平扩展
4. **"Guardrails in the loop"** — 安全不是事后审计，而是执行循环中的第一等公民

### 如果要落地，优先建议

1. **短期（1-2周）**：实现自动化的 user memory 抽取——每次对话结束后用轻量 LLM 调用抽取关键信息，存入结构化 JSON，替代/补充手动维护 MEMORY.md
2. **中期（1月）**：引入 Input/Output Guardrail 机制——至少覆盖外发消息和文件删除两个高风险操作
3. **长期（2-3月）**：考虑 Knowledge base 集成——将学长关心的领域知识（冰壶规则、股票分析方法等）向量化后作为知识库注入
