## 二、项目概述

Kong Agent Gateway 是 Kong AI Gateway 在 3.14 版本中引入的核心能力扩展，将 Kong 从传统的 API 网关和 LLM 网关，进一步升级为支持 **Agent-to-Agent（A2A）通信治理** 的全栈 AI 网关。它的核心主张是：**"Complete the AI Data Path"（完成 AI 数据路径）**——在一个统一平台上管理 API 调用、LLM 推理请求、MCP 工具访问以及 A2A 智能体间通信。

在 Agentic AI 快速落地的背景下，企业面临一个棘手问题：多个智能体之间的通信如何治理？没有统一的 A2A 网关，企业只能选择拼凑点对点方案、自建代理或接受基础设施盲区。Kong Agent Gateway 正是为此而生——它将 Kong 在 API 管理领域十余年的工程积累，延伸到了智能体交互这一新兴领域。

Gartner 在《Emerging Tech Adoption Radar 2026》中指出："Agent 之间的发现、协商和交易代表了下一个地平线——这是 AI 网关的一个新兴差异化能力类别。"Kong Agent Gateway 正是对这一判断的直接响应。

## 三、核心架构

### 3.1 三层 AI 流量统一模型

Kong Agent Gateway 的架构设计围绕三种 AI 流量模式展开：

```
┌─────────────────────────────────────────────┐
│              Kong AI Gateway                │
│                                             │
│  ┌──────────┐  ┌──────────┐  ┌───────────┐ │
│  │ LLM       │  │ MCP      │  │ Agent     │ │
│  │ Gateway   │  │ Gateway  │  │ Gateway   │ │
│  │ (已有)    │  │ (3.10+)  │  │ (3.14+)   │ │
│  └────┬─────┘  └────┬─────┘  └─────┬─────┘ │
│       │             │              │        │
│  应用→LLM    Agent→MCP工具    Agent↔Agent   │
│                                             │
│  统一策略层：认证 / 限流 / 审计 / 可观测性     │
└─────────────────────────────────────────────┘
```

**1. LLM Gateway 层**：管理应用与大语言模型之间的流量，支持多模型路由、负载均衡、语义缓存和基于 token 的速率限制。这是 Kong AI Gateway 的基础能力。

**2. MCP Gateway 层**（3.10 引入）：治理通过 Model Context Protocol 访问的工具和数据源，包括 MCP Server 的注册发现、访问控制和安全代理。支持将现有 API 编排声明式暴露为 MCP 工具。

**3. Agent Gateway 层**（3.14 引入）：新增的 A2A 通信治理能力，基于 Google 主导的 A2A（Agent-to-Agent）开放协议，提供智能体身份验证、通信策略执行和交互审计。

### 3.2 A2A 协议支持

Agent Gateway 原生实现 Google 的 A2A Protocol 规范，该协议定义了智能体之间通过 HTTP 进行 JSON-RPC 和 REST 绑定通信的标准方式。核心组件包括：

- **AI A2A Proxy 插件**：作为 A2A 通信的透明代理，拦截并检查所有智能体间消息
- **Agent 身份与认证**：确保只有授权的智能体能发起或参与 A2A 通信
- **Prometheus + OpenTelemetry 集成**：A2A 流量的可观测性指标采集
- **策略引擎**：基于 Kong 插件体系，支持自定义 A2A 通信规则

### 3.3 与 Kong Konnect 平台的集成

Agent Gateway 不是独立产品，而是 Kong Konnect（Kong 的云原生控制平面）的一部分。这意味着：

- 通过 Kong Manager Web UI 可视化管理 Agent 策略
- 与 Kong Event Gateway 联动，实现事件驱动的多智能体编排
- 支持 decK（Kong 的配置即代码工具）进行 GitOps 式的 Agent 策略管理
- 企业级功能（SSO、RBAC、细粒度审计）天然适用于 Agent 治理场景

## 四、技术亮点

### 4.1 "AI 数据路径全覆盖"的独特定位

这是 Kong Agent Gateway 最突出的差异化优势。目前市场上：
- 纯 LLM 网关（如 LiteLLM、Portkey）只管模型调用
- 纯 MCP 网关（如 agentgateway/agentgateway）只管工具访问
- 还没有任何其他产品能在一个平台内同时治理 API → LLM → MCP → A2A 这整条链路

对于已经在使用 Kong 做 API 网关的企业来说，Agent Gateway 是零额外基础设施成本的升级路径。

### 4.2 实时流量检测与安全防护

Agent Gateway 支持对 A2A 消息进行**飞行中内容检测（in-flight inspection）**，能够实时识别：
- Prompt 注入攻击跨智能体传播
- 异常行为模式（如某 Agent 异常高频调用）
- 数据越权访问尝试（敏感数据流向未授权 Agent）
- Token 预算失控（Rogue Agent 消耗大量资源）

这种"代理即防火墙"的能力在生产环境中至关重要——一个被攻破的子 Agent 不应成为横向移动的跳板。

### 4.3 Agent 级别的成本分摊

不同于传统 LLM 网关只能追踪到"哪个应用调用了哪个模型"，Agent Gateway 可以精确到**每个智能体的 token 消耗和资源占用**。在多 Agent 协作系统中，这意味着可以准确回答"规划 Agent 花了多少钱 vs 执行 Agent 花了多少钱"，为成本优化提供粒度数据。

### 4.4 基于 Lua/OpenResty 的高性能基础

Kong 底层基于 OpenResty（Nginx + LuaJIT），这是经过大规模生产验证的高性能网络框架。相比基于 Node.js 或 Python 构建的竞品网关，Kong 在高并发 A2A 场景下的延迟和吞吐量具有显著优势。对于需要处理大量智能体间实时通信的系统而言，这个底层选型差异不容忽视。

### 4.5 插件化扩展生态

Kong 拥有业界最丰富的 API 网关插件生态（社区插件数百个），这些插件可以直接应用于 Agent Gateway 场景。例如：
- AI Rate Limiting Advanced 插件的模型感知限流可直接用于 Agent 级别限流
- Request Transformer 插件可用于 A2A 消息的动态改写
- Key Auth / OAuth2 插件直接服务于 Agent 身份认证

## 五、适用场景

### 5.1 企业级多智能体生产环境

当企业从 PoC 阶段进入生产部署，拥有数十甚至上百个智能体服务时，Agent Gateway 提供了必要的治理基础设施。特别是金融、医疗等强监管行业，A2A 通信的审计日志是合规刚需。

### 5.2 混合架构中的智能体编排

企业通常不会全部采用单一厂商的 Agent 方案，而是混合使用 OpenAI Agents SDK、LangGraph、CrewAI 等不同框架构建的智能体。Kong Agent Gateway 作为框架无关的中间层，可以为异构 Agent 提供统一的通信治理。

### 5.3 已有 Kong 基础设施的企业

如果企业已经在使用 Kong 做 API 管理，引入 Agent Gateway 的边际成本极低——无需部署新的基础设施组件，只需升级版本并启用新插件即可获得完整的 AI 治理能力。

### 5.4 Agent 即服务（Agent-as-a-Service）平台

对于构建内部 Agent 平台或对外提供 Agent 服务的团队，Agent Gateway 可以作为平台的"前台"，统一处理多租户隔离、计费计量和安全防护。

## 六、局限性

### 6.1 A2A 协议本身尚处早期

Google 的 A2A Protocol 虽然有 Google 背书，但截至 2026 年初，生态成熟度仍远低于 MCP。主流 Agent 框架（OpenAI Agents SDK、LangGraph、CrewAI）尚未原生支持 A2A，这意味着实际落地往往需要额外的适配层。Kong Agent Gateway 的价值在很大程度上取决于 A2A 协议的普及速度。

### 6.2 开源版与企业版的功能差距

Kong 的典型商业模式是：核心网关功能开源（Apache 2.0），但高级 AI 能力（包括部分 Agent Gateway 特性）属于 Kong Plus/Enterprise 付费版。开源社区可以使用的 Agent 功能可能受限，企业在评估时需要注意这一点。

### 6.3 学习曲线陡峭

Kong 本身就是一个复杂的系统（数百个配置项、复杂的插件机制），加上 AI Gateway 和 Agent Gateway 的新概念，新手上手成本不低。相比之下，LiteLLM 或 Portkey 这类专注 AI 的网关更易于快速上手。

### 6.4 偏重"管道"而非"编排"

Kong Agent Gateway 定位是智能体通信的**治理层**（governance layer），而非**编排层**（orchestration layer）。它不负责决定"哪个 Agent 应该做什么"，而是确保"Agent 之间的通信符合规则"。如果你需要的是工作流编排能力，仍需配合 LangGraph、CrewAI 等框架使用。

### 6.5 对非 Kong 用户门槛较高

对于一个没有 Kong 使用背景的团队，为了 Agent Gateway 而引入整套 Kong 基础设施可能过于重量级。轻量级的替代方案如 `agentgateway/agentgateway`（GitHub 上新兴的开源项目）可能更适合从零开始的团队。

## 九、总结与展望

Kong Agent Gateway 代表了一个重要趋势：**AI 治理正在从"管模型调用"进化到"管智能体协作"**。随着 Agentic AI 从实验走向生产，企业需要的不再是一个能转发 HTTP 请求的代理，而是一个能理解智能体语义、执行细粒度策略、提供完整审计轨迹的治理中枢。

Kong 的战略判断是正确的——将 API 网关的成熟工程能力向上层延伸，覆盖新兴的 AI 交互模式。3.14 版本的 Agent Gateway 是这一战略的关键一步，但真正的考验在于 A2A 协议生态的成熟度和市场接受度。

对于技术决策者而言，如果你的组织已经在 Kong 投资上有所积累，或者你正在寻找一个能"一步到位"覆盖所有 AI 流量模式的治理方案，Kong Agent Gateway 值得认真评估。但对于初创团队或纯 AI Native 项目，更轻量的专用方案可能在现阶段更具性价比。
