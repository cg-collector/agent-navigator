# AI Agent 学习报告 #065

## 项目名称：Agency Swarm

## GitHub 链接
https://github.com/VRSEN/agency-swarm

## 一句话简介
基于 OpenAI Agents SDK 的多智能体协作框架，用企业组织结构思维构建层级式 AI Agent 团队。

## 核心架构

**技术栈：** Python 3.12+，OpenAI Agents SDK，LiteLLM（多模型支持），Pydantic（类型安全）

**设计模式：**
- **组织架构模式** — 用 `>` 运算符定义 Agent 间通信权限（如 `CEO > Developer` 表示 CEO 可向 Developer 发起对话），模拟企业层级
- **角色专业化** — 每个 Agent 有独立的指令文件、工具集、文件目录，职责明确
- **状态持久化** — 跨会话保存对话线程，支持外部存储集成

**项目结构：**
```
agency/
├── agency_manifesto.md  # 全局指令（团队行为准则）
├── CEO/                 # CEO 智能体
│   ├── tools/           # 自定义工具
│   ├── instructions.md  # 角色指令
│   └── files/           # 上传文件
├── Developer/           # 开发智能体
└── VirtualAssistant/    # 助理智能体
```

## 技术亮点

1. **通信流控制语法** — 用 `>` 运算符声明式定义 Agent 间通信权限，直观且安全，避免 Agent 间的无序交互
2. **OpenAPI 自动转工具** — 可将任意 OpenAPI 规范自动转换为 Agent 可调用的工具，零代码接入外部服务
3. **Pydantic 强类型工具** — 所有工具输入输出都有 Pydantic schema 验证，生产级可靠性
4. **多模型后端** — 通过 LiteLLM 支持 OpenAI、Anthropic、Gemini 等，不锁死单一供应商
5. **文件管理自动化** — 自动上传文件到 AI 运行环境，Agent 可直接操作文件

## 对楠楠/学长的启发

- **通信控制的思想值得借鉴** — OpenClaw 的多 Agent 协作也可以参考这种声明式权限模型，避免子 Agent 之间的混乱通信
- **角色专业化的实践** — 桐桐跑腿、楠楠聊天的分工其实就是这种模式的雏形，可以更正式化
- **manifesto 全局指令** — 类似 SOUL.md 的概念，但作用于整个 Agent 团队，适合多 Agent 系统统一风格
- **OpenAPI → Tools 的思路** — 如果学长要给楠楠接入新服务，可以参考这个自动转换方案

## 评分：⭐⭐⭐⭐

**理由：** 组织架构模式创新且实用，通信控制语法简洁优雅。扣一星是因为深度绑定 OpenAI 生态（虽通过 LiteLLM 解耦），且社区规模相对 CrewAI/LangGraph 较小。适合需要构建结构化多 Agent 团队的场景。
