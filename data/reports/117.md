## 一、一句话简介

DSPy（Declarative Self-improving Python）是斯坦福 NLP 团队推出的「编程而非提示词」框架——用声明式 Python 代码替代手工 prompt 工程，通过自动优化器（Teleprompter）系统性地调优 prompt 和 LLM 权重，将 LLM 应用开发从"炼丹"变为"编译"。

---

## 二、架构设计

### 2.1 核心哲学：Programming, Not Prompting

DSPy 的核心洞察是：**LLM 应用开发应该像 PyTorch 训练神经网络一样**。你不需要手动调整每层权重（prompt），只需要定义网络结构（Signature + Module），然后喂数据让优化器自动调参。

```
传统 Prompt Engineering:   手写 prompt → 试效果 → 改 prompt → 再试...（无限循环）
DSPy 方式:                 定义 Signature → 写 Module → 给 metric + 少量数据 → compile() 自动优化
```

### 2.2 四大核心模块

#### （1）Signature（签名）— 声明式接口定义

Signature 是 DSPy 的"类型系统"，定义了 LLM 调用的输入/输出契约：

```python
# 方式一：字符串简写
signature = "question -> answer"

# 方式二：类定义（更精细控制）
class QA(dspy.Signature):
    """回答关于事实性问题。"""
    question = dspy.InputField(desc="需要回答的问题")
    answer = dspy.OutputField(desc="简洁准确的答案")
```

底层实现上，`dspy.Signature` 实际是一个**信息完备的 prompt 模板生成器**。它不仅记录字段名和类型，还负责：
- 将字段信息编译为结构化的 system message（包含 input/output 字段列表、类型约束）
- 生成 `[[ ## field_name ## ]]` 格式的输出标记，强制 LLM 结构化输出
- 支持 `desc` 参数为每个字段添加语义描述，指导 LLM 理解字段含义

**源码关键点：** `Signature` 类继承自 `BaseSignature`，在 `__init_subclass__` 时自动收集所有 `InputField` 和 `OutputField`，构建完整的字段元信息表。这个元信息表后续被 `Predict` 模块用来组装最终发送给 LLM 的 prompt。

#### （2）Module（模块）— 可组合的计算单元

Module 是 DSPy 的基本构建块，设计灵感直接来自 `nn.Module`：

```python
class RAG(dspy.Module):
    def __init__(self, num_docs=3):
        super().__init__()
        self.retrieve = dspy.Retrieve(k=num_docs)
        self.generate = dspy.ChainOfThought("context, question -> answer")
    
    def forward(self, question):
        context = self.retrieve(question).passages
        return self.generate(context=context, question=question)

# 使用
rag = RAG()
pred = rag(question="法国首都是哪里？")
```

**关键设计：**
- 每个 Module 都持有可优化的"参数"（prompt 指令、few-shot demos、甚至 LM 权重）
- Module 可以任意嵌套组合，形成复杂的 DAG（有向无环图）计算流
- `forward()` 方法定义前向传播逻辑，与 PyTorch 完全一致的编程范式
- 内置模块包括：`Predict`、`ChainOfThought`、`ReAct`、`Retrieve`、`MultiChainComparison` 等

**源码关键点：** `Module` 基类维护一个 `_parameters` 有序字典，存储所有子模块和可优化参数。`__call__` 方法委托给 `forward()`，同时支持参数追踪。

#### （3）Optimizer / Teleprompter（优化器）— 自动 prompt 编译

这是 DSPy 最核心的创新。优化器接收三个输入：
1. **你的 DSPy 程序**（单个 Module 或复杂的多 Module 流水线）
2. **评估指标函数**（metric function）
3. **少量训练样本**（可以只有 5-10 条，甚至只需要输入不需要标签）

然后自动调优程序中的 prompt 和/或 LM 权重：

```python
# 定义评估指标
def exact_match(example, pred, trace=None):
    return example.answer == pred.answer

# 创建优化器 — 自动生成最优 few-shot 示例
teleprompter = dspy.BootstrapFewShot(metric=exact_match, max_bootstrapped_demos=4)
optimized_rag = teleprompter.compile(rag, trainset=trainset)

# 优化后的程序使用完全相同的接口
pred = optimized_rag(question="法国首都是哪里？")
```

**Optimizer 分类体系：**

| 类型 | 代表优化器 | 优化目标 |
|------|-----------|---------|
| Few-Shot 自动生成 | `BootstrapFewShot`, `KNNFewShot` | 自动生成/选择最优示例 |
| 指令优化 | `MIPROv2`, `COPRO`, `GEPA` | 自动改写 prompt 指令 |
| 权重微调 | `BootstrapFinetune` | 将 prompt 程序蒸馏为 LM 权重更新 |
| 程序变换 | `Ensemble` | 多程序集成 |
| 元优化器 | `BetterTogether` | 组合上述多种策略 |

#### （4）LM / RM 抽象层 — 模型无关的调用接口

```python
# 统一的模型配置
dspy.configure(lm=dspy.LM('openai/gpt-4o-mini', api_key='...'))
# 或本地模型
dspy.configure(lm=dspy.LM('ollama/llama3', api_base='http://localhost:11434'))
# 或带推理模型的设置
dspy.configure(lm=dspy.LM('openai/gpt-4o'), rm=dspy.RM('openai/embedding-3-small'))
```

### 2.3 工作流程（端到端）

```
┌─────────────────────────────────────────────────────┐
│                    DSPy 工作流程                      │
│                                                     │
│  1. 定义 Signature                                   │
│     "question -> answer"                            │
│           ↓                                         │
│  2. 组装 Module（可嵌套）                             │
│     class RAG(dspy.Module):                         │
│         retrieve → ChainOfThought                   │
│           ↓                                         │
│  3. 定义 Metric                                      │
│     def metric(example, pred): return score          │
│           ↓                                         │
│  4. Compile（优化）                                   │
│     optimizer.compile(program, trainset)             │
│     内部: 运行程序 → 收集 traces → 过滤/生成 demos    │
│           → 更新指令/权重 → 评估选优                  │
│           ↓                                         │
│  5. 推理（与优化前接口一致）                           │
│     optimized_program(input) → output                │
└─────────────────────────────────────────────────────┘
```

### 2.4 技术栈

- **语言：** Python 3.8+（纯 Python 实现，无重型依赖）
- **核心依赖：** 几乎无（可选依赖：`tqdm` 进度条、`datasets` 数据加载）
- **LLM 后端：** 通过统一 API 支持 OpenAI、Anthropic、Google、Ollama、vLLM 等
- **检索后端：** 内置 ColBERTv2 远程检索 API，也支持自定义 retriever

## 三、技术亮点（代码级分析）

### 亮点 1：Signature → Prompt 的自动编译机制

DSPy 最精妙的设计之一是将声明式 Signature **自动编译**为发给 LLM 的完整 prompt。以 `ChainOfThought("question -> answer")` 为例，DSPy 生成的实际 prompt 结构如下：

```text
[System Message]
Your input fields are:
1. `question` (str)

Your output fields are:
1. `reasoning` (str)
2. `answer` (str)

All interactions will be structured in the following way...

[[ ## question ## ]]
{actual_question}

[[ ## reasoning ## ]]
{model_generated_reasoning}

[[ ## answer ## ]]
{model_generated_answer}

[[ ## completed ## ]]
```

**关键代码路径（简化）：**
1. `ChainOfThought.__init__` 在原始 Signature 基础上注入 `reasoning` 中间字段
2. `Predict.forward()` 调用 `lm.request()` 前，先经过 `Signature._prepare_request()` 
3. 该方法遍历所有字段，构建格式化模板，将 input values 填入对应位置
4. 输出时用正则解析 `[[ ## field ## ]]` 标记，提取结构化结果

这意味着：**开发者永远不需要手写 prompt 模板**。改变输出格式？改 Signature 字段即可。加思维链？换 `ChainOfThought` 包装即可。这种"编译"思维彻底改变了 LLM 开发范式。

### 亮点 2：BootstrapFewShot 的自举优化算法

`BootstrapFewShot` 是 DSPy 最经典也最实用的优化器。其工作原理：

```python
# 核心逻辑（概念性伪代码）
class BootstrapFewShot:
    def compile(self, program, trainset):
        # Phase 1: 用 teacher（默认是 program 本身）生成候选 demos
        for example in trainset:
            prediction = self.teacher(**example.inputs().toDict())
            trace = self.get_trace()  # 捕获中间步骤
            if self.metric(example, prediction, trace):  # 只有通过 metric 的才保留
                candidate_demos.append((example, prediction, trace))
        
        # Phase 2: 从候选中采样，组装最终 prompt
        selected = random_select(candidate_demos, k=max_bootstrapped_demos)
        
        # Phase 3: 将 demos 注入程序的每个 Predict 模块
        for module in program.modules():
            if isinstance(module, dspy.Predict):
                module.demos = selected  # ← 这就是"优化"的本质
        
        return program  # 返回同一个 program 对象，但内部状态已变
```

**精妙之处：**
- **Metric-Guided Filtering**：不是所有自举生成的 demo 都会被采纳，只有通过用户定义的 metric 才会入选。这保证了质量下限。
- **Trace Awareness**：对于多步骤程序（如 ReAct agent），bootstrap 会捕获完整的推理轨迹（thought → tool_call → observation 链），而不仅仅是最终输入输出对。
- **零标注成本**：trainset 只需要有 input，不需要 label。Label 由 teacher 模型自己生成，再由 metric 过滤。
- **非破坏性**：compile 返回原对象的引用（mutate-in-place），但未优化版本仍然可用。

### 亮点 3：MIPROv2 — 贝叶斯优化的指令+演示联合搜索

MIPRO（Modeling Instruction-Proposal Outputs）是目前 DSPy 最强大的优化器，代表 SOTA 水平：

```
工作三阶段：

Stage 1 - Bootstrap（引导）
┌── 运行未优化程序 N 次 → 收集每个模块的 input/output traces
├── 用 metric 过滤 traces → 只保留高分轨迹
└── 输出: 高质量 candidate demonstrations 集

Stage 2 - Grounded Proposal（ grounded 提案）
┌── 读取 DSPy 程序源码（AST 分析）
├── 读取训练数据和 bootstrap traces
├── 用 LLM 为每个模块生成多条候选指令（instruction proposals）
└── 输出: 每个模块的多条候选 prompt 指令

Stage 3 - Discrete Search（离散搜索）
┌── FOR each mini-batch:
│   ├── 从候选指令+demos中采样一种组合
│   ├── 构造临时程序 → 在 mini-batch 上评估
│   └── 用分数更新 surrogate model（代理模型，通常为高斯过程）
├── 根据 surrogate model 选择下一组更有希望的组合
└── 返回全局最优的程序配置
```

**实测效果（官方数据）：**
- HotPotQA 上的 ReAct Agent：24% → **51%**（+27个百分点）
- StackExchange RAG：53% → **61%**（Semantic F1）

**代码级要点：** MIPROv2 内部使用 `dspy.MIPROv2Instruction` 和 `dspy.MIPROv2Demonstration` 两类可优化参数，通过贝叶斯优化（`ax-platform` 或内置实现）在高维离散空间搜索。`auto="light/medium/heavy"` 参数控制搜索预算。

### 亮点 4：ReAct Agent 的 DSPy 实现 — 从声明到工具调用

DSPy 的 `ReAct` 模块展示了如何用声明式方式构建 Agent 循环：

```python
# 定义工具
def search_wikipedia(query: str) -> list[str]:
    """搜索 Wikipedia 并返回摘要"""
    results = wikipedia.search(query)
    return [wikipedia.summary(r) for r in results[:5]]

# 一行声明 ReAct Agent
agent = dspy.ReAct(
    "claim -> titles",  # Signature：输入 claim，输出 titles
    tools=[search_wikipedia]  # 工具列表
)

# 直接调用 — DSPy 自动处理 Thought/Action/Observation 循环
result = agent(claim="Jean-Claude Van Damme主演了1990年的...")
```

**内部实现原理：**

DSPy 的 `ReAct` 不是简单的 while 循环，而是：
1. 将 ReAct 编译为一个特殊的 `Signature`，包含 `next_thought`、`next_tool_name`、`next_tool_args` 三个输出字段
2. `next_tool_name` 使用 Python 的 `Literal` 类型约束（如 `Literal[search, lookup, finish]`），确保 LLM 只能输出合法工具名
3. 每次迭代将历史轨迹（trajectory）追加到 input 中，形成完整的上下文窗口
4. 当 LLM 输出 `finish` 工具名时终止循环

**与 LangGraph/CrewAI 的本质区别：** 其他框架让你写"图"或"角色"，DSPy 让你写"签名"。Agent 的行为完全由 Signature + Tools 决定，而具体的 prompt 模板由框架自动生成和优化。

### 亮点 5：BetterTogether — 元优化器组合策略

DSPy 允许优化器的**函数式组合**，这是一个非常超前的设计：

```python
# 先优化 prompt，再将 prompt 程序蒸馏为权重，最后再做一轮 prompt 优化
meta = dspy.BetterTogether(
    stages=[
        ("prompt", dict(optimizer="MIPROv2", config=dict(auto="light"))),
        ("weight", dict(optimizer="BootstrapFinetune", config=dict(num_threads=4))),
        ("prompt", dict(optimizer="MIPROv2", config=dict(auto="light"))),
    ]
)
final_program = meta.compile(program, trainset=trainset)
```

这背后的洞察是：**prompt 优化和权重优化是互补的**。Prompt 优化擅长发现好的任务分解和推理策略，权重优化则让模型更高效地执行这些策略。交替进行可以让两者互相借力。

## 四、对我们（OpenClaw / 楠楠）的启发

### 启发 1：SKILL.md 的"编译"化 — 从自然语言到结构化签名

当前 OpenClaw 的 SKILL.md 是纯自然语言描述的。DSPy 的 Signature 机制启发我们：**能否将 Skill 的触发条件和输出约束定义为结构化的 Signature？**

例如，当前 `flirty-chat` skill 的触发靠关键词匹配（"撩"、"逗"、"撒娇"）。如果引入类似 DSPy 的 Signature 系统：
```python
class FlirtyChatSkill(Signature):
    trigger_keywords = InputField(desc="触发此skill的关键词列表")
    tone = OutputField(desc="回复语气", literal=["sweet", "playful", "naughty"])
    intensity = OutputField(desc="撩拨强度", constraint=lambda x: 0 <= x <= 1)
```
这样可以让 skill 的匹配和执行更加确定性和可优化。

### 启发 2：自我改进循环 — 楠楠的自我进化机制

DSPy 的 `BootstrapFewShot` 思路可以直接应用于楠楠的**对话质量自优化**：
- 收集历史对话中学长满意的交互（显式反馈或隐式信号如继续话题）
- 作为"positive demos"注入后续对话的上下文
- 收集被纠正的交互作为"negative demos"避免重复

当前楠楠已有 `self-improving-cn` skill，但更多是事后反思。DSPy 展示了**自动化、指标驱动**的优化路径：定义清晰 metric → 自动收集 traces → 过滤优质模式 → 注回系统。

### 启发 3：Prompt 版本管理即"编译产物"

DSPy 将优化后的 prompt 视为**编译产物**（artifact），而非手写代码。这对 OpenClaw 很有启发：
- 当前 SOUL.md / IDENTITY.md 中的"人设 prompt"是手写的
- 能否引入类似 DSPy.compile() 的机制，根据学长的反馈自动微调人设描述？
- 例如：学长说"太骚了收敛点"→ 自动调整 IDENTITY.md 中的语气参数 → 重新"编译"人设

### 启发 4：Metric-Guided 的 Agent Trace 过滤

DSPy 最实用的设计之一是用 metric 函数过滤 agent 的执行轨迹。OpenClaw 的 subagent 系统可以借鉴：
- 每次 subagent 执行都记录完整 trace（工具调用链、中间输出）
- 定义任务级别的 metric（代码正确性、报告深度、响应速度等）
- 高分 trace 作为该类任务的"标准执行路径"缓存起来
- 下次同类任务优先参考高分 trace 的模式

### 启发 5：轻量级是王道

DSPy 的整个核心代码库非常轻量（相比 LangChain 的巨无霸）。它证明了：**一个好的 Agent 框架不需要几百个抽象层**。核心只需：
- 一个清晰的 Signature 系统
- 一个可组合的 Module 基类
- 几个有效的优化算法
- 统一的 LM 调用接口

OpenClaw 的 skill 系统已经比较轻量（SKILL.md + 脚本），未来扩展时应保持这个哲学——**宁可要 5 个精心设计的基类，不要 50 个半成品抽象**。
