## 架构概述

Sim Studio 采用**微服务架构**设计，整体技术栈现代化且层次清晰：

```
前端层（Next.js + React 可视化画布）
    ↓（WebSocket 实时通信）
API 网关层（Next.js API Routes）
    ↓（RESTful API）
核心服务层（多模块分离）
├── 工作流引擎（Workflow Engine）— 拓扑排序+依赖调度
├── 智能体执行器（Agent Executor）— LLM/工具调用
├── 向量数据库服务（pgvector）— 内置 RAG
├── 身份认证服务（Better Auth）
└── 实时通信服务（Socket Server — Socket.io）
    ↓
数据存储层（PostgreSQL + pgvector 扩展）
```

**工作流执行流水线：**
1. 用户通过可视化画布拖拽 Block、连线设计工作流
2. 系统将节点拓扑结构转换为有向无环图（DAG）执行计划
3. 工作流引擎按依赖关系调度各节点执行
4. Agent Executor 调用对应 AI 模型或外部工具
5. 结果通过 Socket.io 实时推送给前端展示

**核心抽象概念：**
- **Block（块）：** 工作流基本执行单元，每个 Block 代表一个功能模块（LLM调用、数据处理、条件判断、HTTP请求等）
- **Canvas（画布）：** 可视化编排界面，支持拖拽、连线、缩放
- **Copilot：** 内置 AI 辅助，自然语言描述即可生成/修改节点

## 核心技术亮点

### 1. 可视化画布 + Copilot 双模式编排
这是 Sim Studio 最大的创新点。不仅提供拖拽式可视化画布降低门槛，还集成了 **AI Copilot** —— 用户可以用自然语言描述需求（如"帮我创建一个先分析用户意图再查询知识库的流程"），系统自动生成对应的节点和连线。这种"低代码 + AI辅助"的组合大幅提升了开发效率。

### 2. 内置向量数据库（pgvector），零配置 RAG
大多数 Agent 框架需要外接向量数据库（如 Pinecone、Milvus），而 Sim Studio **内置了 pgvector**，用户上传文档即可直接使用 RAG 能力。这对中小团队极其友好——不需要额外运维一套向量数据库服务，Docker 一键启动就全有了。

### 3. 多模型原生兼容，零锁定
紧贴各模型提供方的原生接口能力，支持 OpenAI、Anthropic (Claude)、Google Gemini、LLaMA 以及本地 Ollama/vLLM 模型。切换模型无需改动工作流逻辑，真正实现"一次构建，多模型运行"。同时完整暴露模型的原始参数（temperature、top-p 等），不搞黑箱封装。

### 4. 全链路可观测性
原生集成完整的可观测机制：
- **交互日志追踪：** 每次 Agent 与模型的对话内容完整记录
- **执行延迟可视化：** 快速识别性能瓶颈节点
- **成本追踪：** 每次模型调用的 token 消耗和费用
- **错误诊断报告：** 出错时精确定位上下游问题模块
这让调试 Agent 不再是"黑盒猜谜"，而是像调试普通代码一样直观。

### 5. 多部署模式 + 95% 镜像优化
提供 Docker Compose（生产）、NPM 包快速启动、Dev Container 三种部署方式。特别针对 Docker 做了 95% 的镜像体积优化，还区分 local-gpu / local-cpu 两种 Profile，对有无 GPU 的环境都友好。

## 对我们的启发

### 🎯 可以直接借鉴的点

1. **Block + Canvas 抽象模型非常适合 OpenClaw 的 Skill 编排**
   - 我们目前的 Skill 是文件驱动的，如果能引入类似 Block 的可视化编排概念，可以让非技术人员也能组合 Skill
   - 特别是 Copilot 自然语言生成节点的思路——用户说"帮我做个每天早上查天气发消息的流程"就能自动组装

2. **内置向量数据库的思路值得参考**
   - 我们的 memory 系统目前是基于文件的，如果未来要做语义检索记忆，pgvector 这种轻量方案比独立 Milvus 更合适
   - 文档上传即 RAG 的体验非常好

3. **全链路可观测性是我们的短板**
   - Sim Studio 的每次调用日志、延迟、成本追踪非常完善
   - OpenClaw 的 subagent 执行过程可以借鉴这个思路，给用户提供更透明的执行视图

4. **多模型零锁定的架构哲学**
   - 我们已经在做这件事（MiniMax + 智谱混用），但 Sim Studio 的"暴露原生参数"理念值得学习——不要过度封装，让高级用户能调一切可调的

5. **微服务模块分离的架构**
   - Workflow Engine 和 Agent Executor 分离的设计很干净
   - 我们在规划更复杂的 Agent pipeline 时可以参考这种分层

## 关键代码/配置片段

### 典型的 Block 定义结构（概念示意）

```typescript
// Sim Studio 中一个 Block 的核心接口定义
interface Block {
  id: string;
  type: BlockType;        // 'llm' | 'tool' | 'condition' | 'http' | 'rag' | ...
  inputs: Port[];          // 输入端口
  outputs: Port[];         // 输出端口
  config: BlockConfig;     // 类型特定的配置（如 LLM 的 temperature）
  position: { x, y };     // 画布位置
}

// 工作流 = Block 的有向图
interface Workflow {
  blocks: Block[];
  edges: Edge[];           // 连接关系（source block port → target block port）
  variables: Variable[];   // 全局变量
}
```

### 工作流引擎核心调度逻辑（概念）

```javascript
// DAG 拓扑排序 + 并行执行
async function executeWorkflow(workflow) {
  // 1. 构建 DAG 并拓扑排序
  const executionOrder = topologicalSort(workflow.blocks, workflow.edges);
  
  // 2. 按层级并行执行（同层级的无依赖节点并发）
  for (const layer of groupByLevel(executionOrder)) {
    await Promise.all(layer.map(block => executeBlock(block)));
  }
  
  // 3. 通过 Socket.io 推送实时状态
  socket.emit('block:completed', { blockId, output, latency });
}
```
