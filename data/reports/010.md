## 2. 项目简介

OpenHands 是目前最成功的开源软件工程 Agent 之一，目标是构建一个 **AI 驱动的自主开发平台**——Agent 可以像真正的开发者一样读代码、写代码、跑测试、提交 PR。

项目前身是 2024 年初出现的 OpenDevin，灵感来自 Cognition AI 推出的商业产品 Devin。作为开源替代方案，OpenDevin 迅速获得了大量社区关注，后更名为 OpenHands 并由 CMU 的 Graham Neubig 教授团队主导开发。

OpenHands 在 SWE-bench Verified（软件工程能力基准测试）上长期保持领先位置，并发表了正式的学术论文，是学术界和工业界都认可的生产级 Agent 框架。

## 3. 架构概述

### 3.1 V0 → V1 的架构演进

OpenHands 最有价值的学习点之一，是它从 V0（单体架构）到 V1（模块化 SDK 架构）的完整演进过程。这不是纸上谈兵的设计，而是从真实痛点中提炼出的架构哲学。

**V0 的问题：**
- 单体仓库，Agent 核心、评估套件、前端、后端、CLI 全混在一起
- 强制所有执行走 Docker 沙箱，本地开发体验极差
- 配置系统混乱——140+ 字段、15 个类、2800+ 行配置代码
- 不同的入口点（CLI / Web UI / GitHub App / SaaS）各自演进出了分叉的逻辑

**V1 的四大设计原则：**

| 原则 | 核心思想 |
|------|----------|
| **可选隔离** | 默认本地运行，需要安全控制时再切换沙箱 |
| **默认无状态** | 所有组件（Agent、Tool、LLM）不可变，唯一可变对象是 Conversation State |
| **严格关注点分离** | Agent 核心与下游应用完全解耦，各应用作为 SDK 的消费者 |
| **两层可组合性** | 部署层面可组合四个独立包；SDK 层面可替换/扩展工具和 Agent |

### 3.2 四包模块化架构

```
┌─────────────────────────────────────────────────┐
│                  应用层 (Applications)           │
│   CLI  ·  Web GUI  ·  GitHub App  ·  SaaS       │
├─────────────────────────────────────────────────┤
│            openhands-sdk (核心包)                │
│  Agent · Tool System · LLM · State · Security   │
├──────────┬──────────────────┬───────────────────┤
│ tools-*  │   workspace-*    │   agent-server    │
│ 工具实现  │  工作空间实现     │   REST/WS 服务    │
│ (本地/远程)│  (本地/沙箱)     │   (部署服务)       │
└──────────┴──────────────────┴───────────────────┘
```

四个独立 Python 包：

1. **`openhands-sdk`**：核心 Agent 框架，定义了 Agent、Tool、LLM、State 的抽象接口
2. **`openhands-tools-*`**：具体工具实现（文件编辑、终端执行、浏览器操作、MCP 集成等）
3. **`openhands-workspace-*`**：工作空间实现（本地文件系统 / Docker 沙箱 / 远程执行）
4. **`openhands-agent-server`**：REST + WebSocket 服务，支持远程部署和多租户

### 3.3 九大核心组件

1. **Event-Sourced State Management（事件溯源状态管理）**
2. **LLM 抽象层**：支持 100+ 模型提供商的统一接口
3. **Tool System（工具系统）**：类型化工具 + MCP 协议集成
4. **Agent 核心**：可组合的 Agent 实现
5. **Context Window Management（上下文窗口管理）**
6. **Local Conversation（本地会话）**
7. **Secret Registry（密钥管理）**
8. **Security & Confirmation（安全与确认）**
9. **Deployment Architecture（部署架构）**

## 4. 技术亮点

### 4.1 事件溯源（Event Sourcing）状态模型

这是 OpenHands V1 最有深度的设计决策。

**核心思想：** 所有 Agent 操作（包括 LLM 调用、工具执行、用户输入）都被记录为 **不可变事件（immutable events）**。Agent 的完整状态可以随时从事件流中 **确定性回放（deterministic replay）**。

```python
# 事件流示例
Event(action="user_message", content="修复这个 bug")
Event(action="llm_response", content="我来分析一下...")
Event(action="tool_call", tool="bash", command="pytest tests/")
Event(action="tool_result", output="3 passed, 1 failed")
Event(action="file_edit", path="src/fix.py", diff="...")
```

**好处：**
- **完美的可复现性**：给定相同的事件流，可以精确重现任何时刻的 Agent 状态
- **"时间旅行"调试**：可以回退到任意历史节点，查看 Agent 当时的决策上下文
- **可靠的会话恢复**：进程崩溃后可以从事件流重建状态，不丢失任何信息
- **可审计性**：每一步操作都有完整记录，适合生产环境的合规需求

### 4.2 可选沙箱 vs 本地执行的优雅切换

V0 强制所有操作走 Docker，导致本地开发极其痛苦。V1 的 workspace 抽象彻底解决了这个问题：

```python
# 同样的 Agent，不同的 workspace
from openhands.sdk import Agent, Conversation
from openhands.workspace.local import LocalWorkspace
from openhands.workspace.docker import DockerWorkspace

# 本地开发（快速迭代）
conv_local = Conversation(agent=agent, workspace=LocalWorkspace("/my/project"))
conv_local.run()

# 沙箱执行（安全隔离）
conv_sandbox = Conversation(agent=agent, workspace=DockerWorkspace(image="python:3.12"))
conv_sandbox.run()
```

**workspace 接口统一了：**
- 本地文件系统操作
- Docker 容器内执行
- 远程 Kubernetes Pod
- 浏览器环境（通过 VNC / Browser Use）

Agent 代码完全不需要改变，只是换了一个 workspace 实现。

### 4.3 不可变组件 + 单一可变状态

这是一个非常精妙的设计模式：

- **Agent**：构造时确定，之后不可变（model、tools、system prompt 都固定了）
- **LLM**：不可变（model name、API key、参数在构造时绑定）
- **Tool**：不可变（行为在注册时确定）
- **Conversation State**：唯一可变对象，记录所有历史事件

```python
# 不可变组件
llm = LLM(model="claude-sonnet-4-5", api_key="...")
agent = Agent(llm=llm, tools=[bash_tool, edit_tool])

# 唯一的可变状态
state = Conversation(agent=agent, workspace="/project")
state.send_message("修复测试失败")
state.run()  # 每一步都追加到事件流
```

**为什么这很重要？** 消除了 V0 中"配置被暗中修改"的 bug 类别。140+ 字段的配置系统被简化为构造时的一次性验证，运行时不再有任何可变配置。

### 4.4 多层工作空间接口

OpenHands 不只是"在容器里跑命令"，它提供了多种与 Agent 交互的方式：

| 接口 | 用途 |
|------|------|
| **VS Code (browser-based)** | 在浏览器中查看/编辑 Agent 修改的文件 |
| **VNC Desktop** | 远程桌面，Agent 可以操作 GUI 应用 |
| **Chromium Browser** | 持久化浏览器实例，Agent 可以浏览网页、调试前端 |
| **Terminal** | 直接查看 Agent 的终端输出 |
| **REST/WebSocket API** | 程序化调用 |

这意味着人可以在 Agent 工作时实时观察、介入甚至接管，极大提升了可控性。

### 4.5 内置安全分析器

OpenHands 的 security analyzer 会检查 Agent 的每一个动作：

- 文件操作是否涉及敏感路径
- 网络请求是否访问可疑域名
- 命令执行是否包含危险操作
- 可以配置为自动批准、自动拒绝、或请求人工确认

### 4.6 模型无关的多 LLM 路由

支持 100+ 模型提供商，可以在同一个 Agent 中混合使用不同模型：

```python
# 高级推理用 Claude Opus，日常任务用 GPT-4o-mini
llm = LLM(
    model="openhands/claude-opus-4",       # 复杂任务
    fallback_model="openhands/gpt-4o-mini",  # 简单任务
    api_key="..."
)
```

### 4.7 生命周期控制

完整的 Agent 生命周期管理：
- **暂停/恢复**：长时间任务可以暂停，下次从暂停处继续
- **子 Agent 委托**：主 Agent 可以派生子 Agent 处理子任务
- **历史恢复**：从任意历史点恢复会话
- **并发执行**：支持同时运行多个 Agent 实例

## 6. SWE-bench 竞赛表现

SWE-bench 是目前衡量 AI 软件工程能力的黄金标准。OpenHands 在该基准上的表现：

- 长期位居 **开源方案第一名**
- 在多个 LLM 后端上均展现出强劲表现
- 作为学术基线被广泛引用

SWE-bench 的核心挑战是：给定一个 GitHub issue 和对应的代码仓库，Agent 需要 **自主定位问题 → 修改代码 → 通过测试**。这要求 Agent 具备代码理解、问题定位、方案设计和代码实现的综合能力。

## 7. 最简使用示例

```python
from openhands.sdk import LLM, Conversation
from openhands.tools.preset.default import get_default_agent

# 三行代码启动一个完整的软件工程 Agent
llm = LLM(model="openhands/claude-sonnet-4-5-20250929", api_key="...")
agent = get_default_agent(llm=llm)
conversation = Conversation(agent=agent, workspace="/path/to/project")

# 像跟开发者对话一样
conversation.send_message("分析这个项目的架构，然后在 README 里补充架构文档")
conversation.run()
```

## 8. 启发与思考

### 8.1 架构演进的教科书案例

OpenHands V0 → V1 的演进是 **从原型到生产** 的完美教科书。很多 Agent 框架都处于 V0 阶段——快速迭代、功能堆叠、配置混乱。OpenHands 团队敢于做彻底的架构重构（而不是继续打补丁），这个决策本身就很有价值。

### 8.2 事件溯源是 Agent 状态管理的正确答案

传统的 CRUD 式状态管理在 Agent 场景下有天然缺陷——Agent 的状态本质上就是一个 **操作历史**。事件溯源不仅解决了状态持久化问题，还附带了可复现性、可审计性、可调试性三大好处。这个模式值得任何长期运行的 Agent 系统借鉴。

### 8.3 "不可变组件 + 唯一可变状态"的简洁之美

在复杂系统中，**减少可变性的范围**是降低 bug 的最有效手段。OpenHands 把所有组件都设计为不可变的，只留 Conversation State 作为唯一可变点，这个设计哲学可以推广到任何分布式系统。

### 8.4 可选隔离比强制隔离更务实

V0 的教训说明：**安全性不能以牺牲开发体验为代价**。如果安全机制太重，开发者就会找各种绕过方式（而这些绕过方式往往更危险）。OpenHands V1 的做法——默认本地执行，按需切换沙箱——是一种更务实的工程哲学。

### 8.5 对 OpenClaw 的借鉴

- **事件溯源**：OpenClaw 的会话历史记录可以借鉴事件溯源思想，实现更可靠的状态恢复
- **不可变配置**：OpenClaw 的配置管理可以从 OpenHands 的"构造时验证"模式中学习
- **workspace 抽象**：OpenClaw 的执行环境隔离可以参考 workspace 接口设计

### 8.6 开源 Agent 的商业化路径

OpenHands 展示了一条清晰的开源商业化路径：核心 MIT 开源 → 企业版 source-available（需许可证）→ SaaS 服务。这种"核心免费 + 企业付费"的模式已经在多个开源项目中得到验证。

## 9. 总结

OpenHands 不仅仅是一个"能让 AI 写代码"的工具，它是一个 **经过生产级验证的 Agent 架构参考实现**。从单体到模块化的重构、事件溯源状态管理、可选隔离的 workspace 抽象——这些设计决策背后的思考过程，比代码本身更有学习价值。

对于正在构建 AI Agent 系统的开发者来说，OpenHands V1 的架构论文（arXiv:2511.03690）是 2025 年最值得精读的 Agent 系统设计文献之一。
