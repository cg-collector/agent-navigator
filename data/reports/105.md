## 架构设计

### 核心模块

CAMEL 采用 **分层模块化架构**，核心代码位于 `camel/` 包下：

```
camel/
├── agents/          # Agent 实现（核心）
│   ├── base_agent.py       # BaseAgent 抽象基类：reset() + step()
│   ├── chat_agent.py       # ChatAgent — 主力 Agent
│   ├── critic_agent.py     # CriticAgent — 评审/批评 Agent
│   ├── task_agent.py       # TaskAgent — 任务分解
│   ├── embodied_agent.py   # 具身智能 Agent
│   └── ...                 # SearchAgent, KnowledgeGraphAgent 等
├── societies/        # 多智能体社会层
│   ├── role_playing.py      # 角色扮演框架（双 Agent 对话）
│   └── workforce/           # 工作力编排（多 Agent 协作）
├── models/          # 模型抽象层
│   └── model_factory.py     # ModelFactory 统一创建 20+ 平台模型
├── messages/        # 消息协议层
│   ├── base_message.py      # BaseMessage 类型系统
│   └── system_prompt.py     # System Prompt 管理
├── datagen/         # 合成数据生成引擎
│   ├── cot_datagen.py       # CoT 思维链数据生成
│   ├── self_instruct/       # Self-Instruct 自动指令生成
│   └── source2synth.py      # Source-to-Synth 数据转换
├── toolkits/        # 工具集成层
│   ├── search_toolkit.py    # 搜索工具集
│   ├── code_toolkit.py      # 代码执行工具
│   └── function_tool.py     # FunctionTool 装饰器
├── memory/          # 记忆系统
│   ├── context_memory.py    # 上下文记忆（滑动窗口）
│   └── chamel_memory.py     # CHAMEL 长期记忆（向量存储）
├── retrievers/      # RAG 检索器
├── storages/        # 持久化存储
├── benchmarks/      # 基准测试套件
└── runtime/         # 运行时环境管理
```

### 技术栈
- **语言**: Python 3.10+，纯 Python 实现（无 Rust/C++ 扩展）
- **LLM 后端**: OpenAI、Anthropic、Google、Mistral、Ollama、vLLM 等 20+ 平台
- **向量数据库**: ChromaDB、FAISS、Qdrant（通过 Storage 层抽象）
- **数据格式**: Pydantic v2（结构化输出）、OpenAI Chat Format / Hermes Format
- **异步**: asyncio 原生支持（`async_step()` 方法链）
- **协议**: OpenAI Function Calling 协议作为工具调用标准

### 工作流程

#### 1. 单 Agent 工作流（ChatAgent）
```
用户输入 → step(message)
  → Memory.push(user_msg)          # 存入记忆
  → ModelBackend.chat(messages)    # 调用 LLM（含 system prompt + history + tools）
  → 解析 response：
    ├─ 纯文本 → 直接返回
    ├─ function_call → 执行 Tool → 将结果回传 LLM 继续推理
    └─ 结构化输出 → Pydantic 解析验证
  → Memory.push(assistant_msg)     # 存入记忆
  → 返回 AgentResponse
```

#### 2. 角色扮演工作流（RolePlaying）
这是 CAMEL 最经典的设计——**双 Agent 角色扮演协作**：

```python
from camel.societies import RolePlaying
from camel.agents import ChatAgent

assistant_sys = "You are a Python programmer."
user_sys = "You are a stock trader needing a trading bot."

role_playing = RolePlaying(
    assistant_agent=ChatAgent(system_message=assistant_sys),
    user_agent=ChatAgent(system_message=user_sys),
    task_prompt="Build a stock trading bot in Python",
)

# 自动循环：assistant 提案 → user 反馈 → assistant 迭代
# 直到双方达成一致或达到最大轮次
info = role_playing.run()
```

核心机制：两个 Agent 各自持有 **角色定制的 System Prompt**，通过结构化的任务提示启动，自动进行多轮协商式对话。这比简单的"一个 Agent 干所有事"更接近真实人类协作模式。

#### 3. 数据生成工作流
```
种子问题集 → Self-Instruct 扩展 → CoT 推理增强 → 
质量过滤（CriticAgent 打分）→ 格式标准化 → 输出 SFT 数据集
```
CAMEL 用 Agent 自身来生成训练数据，形成 **"用 Agent 训练 Agent"** 的闭环。

## 技术亮点

### 1. BaseAgent 的 `step()` 统一接口 + 可组合性

所有 Agent 继承 `BaseAgent`，只暴露两个方法：

```python
# camel/agents/base_agent.py（简化）
class BaseAgent(ABC):
    @abstractmethod
    def reset(self) -> None:
        """重置 Agent 到初始状态"""
        
    @abstractmethod
    def step(self, input_message: BaseMessage) -> BaseResponse:
        """执行一步推理/行动"""
```

这个设计极其简洁但强大——任何复杂行为都被封装在 `step()` 内部。ChatAgent 的 `step()` 内部处理了：记忆管理、工具调用循环、多轮 function calling、结构化输出解析等全部逻辑。

**启发**: OpenClaw 的 skill 系统也可以考虑类似的统一接口——每个 skill 只需要实现 `execute(context) -> Result`，框架负责调度和上下文传递。

### 2. ModelFactory 多模型后端抽象

```python
# camel/models/model_factory.py（简化）
class ModelFactory:
    @staticmethod
    def create(model_platform: ModelPlatformType, 
               model_type: ModelType,
               model_config_dict: dict = None) -> BaseModelBackend:
        """统一接口创建任意平台的模型实例"""
        
        platform_handlers = {
            ModelPlatformType.OPENAI: _create_openai_model,
            ModelPlatformType.ANTHROPIC: _create_anthropic_model,
            ModelPlatformType.OLLAMA: _create_ollama_model,
            # ... 20+ 平台
        }
        return platform_handlers[model_platform](model_type, model_config_dict)
```

关键设计：
- **平台无关**: 切换模型只需改一个枚举值，Agent 代码零改动
- **配置透传**: `model_config_dict` 直接透传到底层 API（temperature、max_tokens 等）
- **支持模型调度策略**: 一个 Agent 可以挂载多个模型，按自定义策略选择每次 step 用哪个模型

```python
# 模型调度示例：简单任务用便宜模型，复杂任务用强模型
def cost_aware_strategy(models):
    if self.last_task_complexity == "high":
        return models[0]  # GPT-4o
    return models[1]      # GPT-4o-mini

agent.add_model_scheduling_strategy("cost_aware", cost_aware_strategy)
```

**启发**: OpenClaw 目前绑定单一模型提供商，可以借鉴这种多模型路由能力——不同任务类型自动选择最优模型，降低成本的同时保证质量。

### 3. RolePlaying 双 Agent 对话框架

CAMEL 的灵魂设计。核心实现在 `camel/societies/role_playing.py`：

```python
class RolePlaying:
    def __init__(self, assistant_agent, user_agent, task_prompt,
                 max_turns: int = 10, 
                 termination_condition=None):
        self.assistant_agent = assistant_agent
        self.user_agent = user_agent
        self.task_prompt = task_prompt
        
    def run(self) -> RolePlayingRecord:
        # Phase 1: 任务初始化
        assistant_msg = self.assistant_agent.step(self.task_prompt)
        
        # Phase 2: 角色协商循环
        for turn in range(self.max_turns):
            user_response = self.user_agent.step(assistant_msg.msgs[0])
            
            if self._check_termination(user_response):
                break
                
            assistant_response = self.assistant_agent.step(user_response.msgs[0])
            
        return RolePlayingRecord(conversation_history=...)
```

这不是简单的 A→B→A 对话，而是：
- **不对称角色**: Assistant 是主动提案方，User 是需求提出+反馈方
- **任务驱动**: 有明确的 `task_prompt` 作为共同目标
- **可终止**: 支持自定义终止条件（如"用户说满意了就停"）
- **可记录**: 完整对话记录可复用于数据生成

**启发**: OpenClaw 的 subagent 机制目前是主→子单向派发，缺少子 agent 之间的对等协商。可以引入类似的双 agent 协商模式用于方案评审、辩论式决策等场景。

### 4. 合成数据生成管线（Data Generation Pipeline）

CAMEL 不只是 Agent 框架，更是 **Agent-as-Data-Generator** 的典范：

```python
# camel/datagen/cot_datagen.py（简化）
class CoTDataGenerator:
    def __init__(self, model, critic_model=None):
        self.agent = ChatAgent(model=model)
        self.critic = CriticAgent(model=critic_model) if critic_model else None
        
    def generate(self, seed_questions: list[str]) -> list[DataRecord]:
        records = []
        for q in seed_questions:
            # Step 1: 让 Agent 生成 CoT 推理过程
            response = self.agent.step(f"Solve step by step: {q}")
            cot_text = response.msgs[0].content
            
            # Step 2: CriticAgent 评估质量
            if self.critic:
                score = self.critic.score(response)
                if score < threshold:
                    continue  # 质量不够则丢弃
                    
            records.append(DataRecord(question=q, reasoning=cot_text))
        return records
```

已生成的公开数据集包括 AI Society（50万+对话）、Code、Math、Physics、Chemistry、Biology 等，全部托管在 HuggingFace。

**启发**: 楠楠的日常交互数据可以用类似方式自动清洗和标注，构建个人化的微调数据集。比如把学长纠正过的错误回答标记为负样本，正确回答标记为正样本。

### 5. FunctionTool 装饰器 + 工具协议标准化

```python
# camel/toolkits/function_tool.py（简化）
def function_tool(func=None, *, name: str = None):
    """将普通 Python 函数转为 Agent 可调用的工具"""
    def decorator(f):
        # 自动从函数签名提取参数 schema（基于 inspect + typing）
        schema = _extract_function_schema(f)
        tool = FunctionTool(
            func=f,
            openai_tool_schema=schema,  # 标准 OpenAI function calling format
            name=name or f.__name__,
        )
        return tool
    return decorator(func) if func else decorator

# 使用方式极简
@function_tool
def calculator(a: int, b: int) -> int:
    """Calculate the sum of two numbers"""
    return a + b

agent = ChatAgent(tools=[calculator])
```

亮点在于 **零样板代码**——类型注解自动变成参数描述，docstring 变成工具说明。底层自动生成符合 OpenAI Function Calling 规范的 JSON Schema。

**启发**: OpenClaw 的 MCP 工具集成偏重量级，可以考虑提供类似的轻量级装饰器，让用户一行注解就能注册新工具给 Agent 使用。

## 对我们（OpenClaw/楠楠）的启发

### 可直接借鉴的点

| 方面 | CAMEL 做法 | OpenClaw 应用场景 |
|------|-----------|-----------------|
| **多模型路由** | ModelFactory + 调度策略 | 不同复杂度任务自动选模型，省钱 |
| **双 Agent 协商** | RolePlaying 框架 | 方案评审时让两个 subagent 辩论，提高决策质量 |
| **结构化输出** | Pydantic response_format | API 返回值强制 schema 校验，减少格式错误 |
| **工具装饰器** | @function_tool 零样板 | 快速注册自定义工具给楠楠使用 |
| **数据自生成** | Agent 生成训练数据 | 把日常纠错记录转化为微调样本 |

### 架构层面的思考

1. **CAMEL 的 Society 层概念值得学习**: 它把"单个 Agent 能做什么"和"多个 Agent 如何组织"彻底分开。OpenClaw 目前的 subagent 更像是临时任务分发，缺乏持久化的"社会关系"概念。如果楠楠未来要管理一组专业化的子 agent（代码专家、搜索专家、分析专家），Society 层的抽象会很有价值。

2. **step() 接口的纯粹性**: CAMEL 证明了一个好的 Agent 抽象不需要复杂接口——`reset()` + `step()` 就够了。OpenClaw 的 skill 系统如果能向这个方向简化，会更容易测试和组合。

3. **研究导向 vs 产品导向**: CAMEL 明确是研究框架（找 Scaling Law），所以它牺牲了一些工程便利性换取实验灵活性。OpenClaw 是产品级系统，需要在易用性和灵活性之间找到不同的平衡点。但 CAMEL 的 **Benchmark 和数据生成能力** 是产品系统通常忽视的——我们也应该建立自己的 eval 基准。

**优势**: 架构清晰、模块解耦好、研究深度顶尖、社区活跃（100+研究者）、合成数据生成是独门绝技
**不足**: 文档分散在 GitHub 和 docs.camel-ai.org 两处；部分高级功能（如 Workforce 编排）文档不够详细；生产部署案例相对较少
