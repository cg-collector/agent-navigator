## 架构概述

LangGraph 的核心思想是把**Agent 工作流建模为有向图（Directed Graph）**，每个节点（Node）是执行逻辑，每个边（Edge）定义状态转移。这与大多数把 Agent 视为"循环调用 LLM"的框架有本质区别。

### 核心灵感来源

LangGraph 的架构明确受到三个系统的深刻影响：

1. **Google Pregel** — Google 2010年提出的分布式图处理系统，核心是"BSP"（Bulk Synchronous Parallel）模型：计算分成多个超步（Superstep），每个超步中节点并行执行，然后全局同步。LangGraph 将这一思想引入 Agent 编排。
2. **Apache Beam** — Google 的另一项开源贡献，提出了统一批处理和流处理的编程模型，强调"一次编写，到处运行"。LangGraph 借鉴了其数据流处理的思想。
3. **NetworkX** — Python 的图论库，LangGraph 的公共 API 从 NetworkX 获得灵感，图结构的操作方式与 NetworkX 一脉相承。

### 核心架构层次

```
┌──────────────────────────────────────────────────────┐
│                    LangGraph Graph                    │
│  ┌────────────┐      ┌────────────┐      ┌─────────┐ │
│  │   Node A   │ ───→ │   Node B   │ ───→ │ Node C  │ │
│  │ (LLM Call) │      │(Tool Call) │      │ (End)   │ │
│  └────────────┘      └────────────┘      └─────────┘ │
│        ↑                    │                          │
│        └────── 边(Edge) ─────┘                          │
│                                                      │
│   状态(State)在整个图中流动，节点对其原子修改           │
├──────────────────────────────────────────────────────┤
│                   持久层 (Persistence)                 │
│  ┌──────────┐  ┌──────────┐  ┌──────────────────────┐ │
│  │ Checkpointer │ │ Memory  │  │  LangSmith Tracing  │ │
│  │  (状态快照)  │ │ (会话)  │  │    (可观测性)       │ │
│  └──────────┘  └──────────┘  └──────────────────────┘ │
├──────────────────────────────────────────────────────┤
│                   模型层 (Models)                      │
│    任意 LangChain 集成的 LLM：OpenAI / Anthropic /    │
│    Gemini / Ollama / Llama / DeepSeek / ...          │
└──────────────────────────────────────────────────────┘
```

### 两种编程 API

LangGraph 提供两种等价的 API：

**1. StateGraph（图形 API）** — 声明式定义图结构

```python
from langgraph.graph import StateGraph, MessagesState, START, END
from typing import TypedDict

class AgentState(TypedDict):
    messages: list
    current_step: str

def node_a(state: AgentState):
    # 节点逻辑：执行 LLM 调用
    return {"messages": [...], "current_step": "node_b"}

graph = StateGraph(AgentState)
graph.add_node("node_a", node_a)
graph.add_node("node_b", node_b)
graph.add_edge(START, "node_a")
graph.add_edge("node_a", "node_b")
graph.add_edge("node_b", END)
compiled = graph.compile()
```

**2. Pregel Functional API（函数式 API）** — 过程式风格，支持 Task 抽象

```python
from langgraph.func import entrypoint, task

@task
def my_task(input):
    # 任务逻辑，可自动重试和持久化
    return result

@entrypoint()
def my_graph(input):
    result = my_task.invoke(input)
    return result
```

## 技术亮点

### 1. 持久化执行（Durable Execution）— 核心差异化能力

这是 LangGraph 区别于几乎所有其他 Agent 框架的标志性能力。

**核心思想**：Agent 执行过程中，每个步骤完成后自动将状态快照（Checkpoint）写入持久化存储（内存、SQLite、PostgreSQL、Redis 等）。如果执行中断（LLM 超时、系统崩溃、用户主动中断），Agent 可以从上一个快照精确恢复，继续执行。

**与传统框架的对比**：

| 特性 | 传统 Agent 框架 | LangGraph |
|------|----------------|-----------|
| LLM 超时 | 丢失全部进度 | 从上一个 Checkpoint 恢复 |
| 用户中断 | 无法继续 | 可随时中断和恢复 |
| 长任务（>1小时） | 风险极高 | 可靠执行 |
| 崩溃恢复 | 完全重来 | 精确恢复 |

**实现机制**：通过 `checkpointer` 接口实现，支持多种存储后端：

```python
from langgraph.checkpoint.memory import MemorySaver
from langgraph.checkpoint.postgres import PostgresSaver

# 内存版（开发用）
checkpointer = MemorySaver()

# PostgreSQL 版（生产用，支持分布式）
checkpointer = PostgresSaver(conn)

graph = graph.compile(checkpointer=checkpointer)

# 执行时指定线程 ID，用于恢复
result = graph.invoke(
    {"messages": [{"role": "user", "content": "hi"}]},
    config={"configurable": {"thread_id": "user-123-session-1"}}
)
```

**三种持久化模式（Durability Modes）**：

- `"exit"`：只在图退出时保存（最快，中间崩溃不恢复）
- `"async"`：异步写入下一个节点开始前（较好性能，有小风险）
- `"sync"`：同步写入，下一步执行前必须落盘（最高可靠性）

### 2. Human-in-the-Loop（人在回路）

LangGraph 内置了 `interrupt` 机制，允许在执行过程中暂停，等待人类输入后再继续。这对于以下场景至关重要：

- **审批流程**：Agent 起草合同，人工审核后发出
- **敏感操作**：Agent 要执行删除操作前，需人工确认
- **实时修正**：Agent 方向错误，人工干预纠正

```python
from langgraph.types import interrupt

def human_review_node(state):
    # 中断执行，等待人工审批
    user_input = interrupt({
        "question": "是否继续发送这封邮件？",
        "email_content": state["draft"]
    })
    return {"approved": user_input["confirm"]}
```

恢复时可通过 API 注入人工决策：

```python
# 恢复执行，注入人工输入
result = graph.invoke(
    None,  # 无需重新输入，状态已保存
    config={"configurable": {"thread_id": "user-123", "checkpoint_id": "xxx"}},
    stream_mode="values"
)
```

### 3. 有状态内存（Stateful Memory）

LangGraph 的内存不是简单的"历史消息列表"，而是图级别的**状态管理**：

```python
class AgentState(TypedDict):
    messages: list                           # 短时记忆：当前对话
    session_memory: dict                     # 长时记忆：跨会话信息
    visited_nodes: set[str]                  # 追踪已访问节点
    retry_count: int                         # 重试计数器

graph = StateGraph(AgentState)
```

**短时记忆**：通过 Checkpoint 自动保存，图执行过程中在节点间流动。

**长时记忆**：通过 `store` 接口实现跨会话持久化：

```python
from langgraph.store.memory import MemoryStore

store = MemoryStore()  # 可替换为 PostgreSQL 等生产级存储

result = graph.invoke(
    {"messages": [...], "session_memory": {...}},
    config={"configurable": {"store": store}}
)
```

### 4. 图结构天然支持复杂工作流

有向图模型让复杂工作流的表达变得直观：

**条件分支**：

```python
from langgraph.graph import MessagesState

def should_continue(state):
    last_msg = state["messages"][-1]
    if "search" in last_msg.content.lower():
        return "search_node"
    return " END "

graph.add_conditional_edges(
    "llm_node",
    should_continue,
    {
        "search_node": "search_node",
        " END ": END
    }
)
```

**并行分支 + 汇聚（Fan-out / Fan-in）**：

```python
# 并行执行多个子任务
graph.add_edge("root", ["task_a", "task_b", "task_c"])

# 等待所有任务完成
graph.add_edge(["task_a", "task_b", "task_c"], "aggregator")
```

**子图（Subgraph）**：一个图可以嵌套另一个图，用于模块化和复用：

```python
subgraph = create_subgraph()
graph.add_node("subgraph_node", subgraph)
```

### 5. Task 抽象 — 处理非确定性操作

LangGraph 的 Task 是实现"一致性重放"（Consistent Replay）的关键。当工作流需要包含随机性或副作用时（如调用外部 API），Task 可以确保恢复时不重复执行：

```python
from langgraph.func import task

@task
def call_external_api(query: str):
    # 如果这个任务已执行过，结果从持久化存储读取
    # 如果未执行，正常调用
    return api_client.search(query)
```

## 启发与思考

### 1. "图"是表达 Agent 工作流的最佳抽象

大多数 Agent 框架把工作流隐式地编码为"LLM 循环 + if-else 判断"，这让复杂流程难以理解和维护。LangGraph 证明了**把工作流显式建模为有向图**具有巨大优势：

- 状态流转可视化
- 条件分支自然表达
- 并行/汇聚模式直观
- 局部修改不影响全局

这对 OpenClaw 的启发：我们的工作流引擎是否可以借鉴图的思维方式？

### 2. 持久化是生产级 Agent 的必要条件

目前大多数 Agent 框架（甚至 OpenClaw 的部分场景）在 LLM 超时或执行中断时只能完全重来。LangGraph 证明**持久化执行是 Agent 从 Demo 走向生产的关键**。

这对 OpenClaw 的启发：
- 为长时间运行的 Agent 任务添加 Checkpoint 机制
- 设计 `interrupt` + `resume` 的 API 模式
- 考虑在 Agent 执行中途支持"人工介入"

### 3. "低层次 + 可组合"比"高层次 + 固定模式"更有生命力

LangGraph 把自己定位为"低层次编排框架"（low-level orchestration framework），不提供现成的 Agent 架构，只提供构建 Agent 所需的基础设施（持久化、内存、人在回路）。这种设计哲学让 LangGraph 能适应各种各样的 Agent 架构，而不只是支持一种"正确的方式"。

这对 OpenClaw 的启发：在架构设计中，选择"提供 primitives + 灵活组合"而非"提供完整但僵化的方案"。

### 4. Pregel 模型的跨时代价值

Google 2010年的 Pregel 论文（用于 PageRank 等图算法）居然在2020年代被引入 Agent 领域并大放异彩。这提醒我们：**经典分布式系统思想在 AI 时代依然具有强大的生命力**。与其追逐最新的"Transformer 变体"，不如深入理解那些经得起时间考验的系统设计思想。

### 5. 与其他框架的定位差异

LangGraph 不是一个"完整的 Agent 方案"，而是"构建 Agent 的基础设施"。这与以下框架形成互补：

| 框架 | 定位 | LangGraph 能替代？ |
|------|------|-------------------|
| LangChain Agents | 高层次 Agent 抽象 | LangGraph 是其底层 |
| CrewAI | 角色驱动的多 Agent | 不直接替代，但可用 LangGraph 构建 |
| Pydantic AI | 类型安全的单 Agent | 不重叠，各有所长 |
| OpenHands | 软件工程 Agent | 不重叠，可用于构建其中某个工具 |
| AgentScope | 多 Agent 仿真 | 部分重叠，LangGraph 更通用 |

---

## 总结

LangGraph 是目前**架构最独特、生产可靠性最高**的 Agent 框架之一。它把 Google 内部验证过的分布式系统思想（Pregel + Beam）引入 Agent 领域，实现了传统 Agent 框架无法做到的能力：**持久化执行**和**人在回路**。

其核心价值主张是：**让 Agent 工作流像普通程序一样可靠运行，支持中断、恢复、崩溃重启**。这对构建生产级 AI 应用至关重要。

适合场景：复杂多步骤工作流、需要长时间运行的 Agent、需要人工审批的自动化流程、需要精确恢复能力的生产系统。

不适合场景：简单的一次性 LLM 调用、快速原型验证、非 Python 技术栈。
