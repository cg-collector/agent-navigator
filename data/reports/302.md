# 第302份研究报告：SeeClick — 基于GUI Grounding的通用视觉GUI Agent

> **项目名称：** SeeClick: Harnessing GUI Grounding for Advanced Visual GUI Agents
> **论文来源：** ACL 2024 (Findings) | arXiv:2401.10935
> **研究机构：** 南京大学 + 上海AI实验室
> **代码开源：** https://github.com/njucckevin/SeeClick
> **完成日期：** 2026-04-12

---

## 一、项目背景与问题定义

### 1.1 现有GUI Agent的核心困境

当前主流的GUI Agent（如Mind2Web、WebArena中的方法）普遍依赖**结构化文本**作为与环境的交互媒介——HTML DOM、Android View Hierarchy、Accessibility Tree等。这种范式存在三大固有缺陷：

- **不可达性**：iOS应用和桌面原生程序无法提供结构化文本，Agent天然被排除在这些场景之外
- **信息冗余与丢失并存**：HTML动辄数万token，但布局信息、图标语义、视觉样式等关键线索却完全丢失
- **平台碎片化**：不同平台的观察空间和动作空间格式各异，需要为每个平台定制适配层

### 1.2 SeeClick的定位：纯视觉范式的破局者

SeeClick提出一个根本性问题：**能否像人类一样，仅通过"看屏幕截图"来完成GUI操作？** 它是首个基于开源LVLM构建的**统一视觉GUI Agent**，不依赖任何平台特定的元数据，仅凭截图+指令即可在移动端、桌面端和网页端执行点击和输入操作。

### 1.3 核心挑战：GUI Grounding

作者通过预实验发现，视觉GUI Agent面临的一个基础性瓶颈是**GUI Grounding能力不足**——即根据自然语言指令准确定位屏幕元素坐标的能力。现有LVLM虽然在自然图像上具备一定的grounding能力，但在密集文字、大量图标和小控件的GUI截图中表现急剧下降。SeeClick将此作为核心突破口，通过专门的预训练策略来增强这一能力。

## 二、核心架构深度剖析

### 2.1 整体框架设计

SeeClick采用**两阶段架构**：

**阶段一：GUI Grounding基础模型（Foundation Model）**
- 基座模型：Qwen-VL（9.6B参数，448×448分辨率）
- 通过持续预训练（Continual Pre-training）注入GUI grounding能力
- 使用LoRA同时微调视觉编码器和语言模型
- 训练约10k步（约1个epoch），数据量约100万条样本

**阶段二：下游任务适配（Agent Adaptation）**
- 将基础模型适配到具体的Agent任务（MiniWob、AITW、Mind2Web）
- 采用In-Context Learning方式，通过精心设计的prompt模板引导模型输出动作
- 支持多轮交互：每一步将当前截图+历史操作+任务目标输入模型，预测下一步动作

### 2.2 数据流管道

```
┌─────────────────────────────────────────────────────┐
│                  数据构造管线                         │
├──────────┬──────────┬───────────────────────────────┤
│ Web数据   │ Mobile数据 │ 通用V-L数据（LLaVA）         │
│ ~300k页面 │ ~20k截图  │ 对话/描述/推理                │
│          │ ~40k控件  │                              │
│          │ ~100k描述 │                              │
├──────────┴──────────┴───────────────────────────────┤
│           混合 + 30种任务特定Prompt → 1M训练集       │
└─────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────┐
│              GUI Grounding 预训练                    │
│  输入：截图s + 指令x → 输出：坐标y = p(y|s,x)        │
│  坐标表示：直接生成自然语言数值（如"click (0.49,0.40)"）│
│  损失函数：交叉熵损失                                 │
│  LoRA微调：视觉编码器 + LLM                           │
└─────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────┐
│              下游Agent任务推理                        │
│  输入：[截图] + [任务描述] + [历史操作]               │
│  输出：动作类型 + 目标坐标 + 输入文本（如有）          │
│  执行：模拟器/浏览器执行动作 → 获取新截图 → 循环      │
└─────────────────────────────────────────────────────┘
```

### 2.3 数据构造的三大来源

**Web UI数据（自动化爬取）：**
- 从Common Crawl最新仓库爬取约30万个网页
- 自动提取两类可交互元素：(1) 含可见文本内容的元素；(2) 带"title"属性的悬停提示元素
- 同时构造两个方向的任务：Grounding（指令→坐标）和OCR（坐标→文本描述）

**Mobile UI数据（公开数据集重组）：**
- Widget Captioning数据集：近20k截图、40k控件、100k语言描述
- 反转captioning过程得到grounding数据（描述→控件位置）
- 补充RICO数据集中的自动采集元素以增加多样性
- 加入Mobile UI Summarization数据增强整体界面理解

**通用视觉-语言数据：**
- 来自LLaVA的对话、详细描述和复杂推理数据
- 用于保持LVLM在自然图像上的通用能力不退化

### 2.4 坐标表示的创新选择

不同于此前工作（如Pix2Act）将图像划分为1000个bin并创建特殊token词汇表的方法，SeeClick采用更直观的方式：**直接将坐标值作为自然语言生成**。例如模型直接输出`click (0.49, 0.40)`这样的归一化坐标。这种方式无需额外的tokenization或预处理/后处理步骤，利用了LVLM本身的数值生成能力。

## 三、技术亮点

### 亮点一：GUI Grounding作为独立能力的系统化探索

这是首个将GUI Grounding从Agent任务中解耦出来、作为**独立基础能力**进行系统性研究的论文。作者提出的核心论断——**GUI grounding能力的提升与下游Agent任务性能的提升存在强正相关**——通过三个基准的一致性实验得到了验证。这意味着：

1. Grounding不是Agent任务的副产品，而是可以也应当被单独优化的基础能力
2. 投资于grounding能力的提升会自动惠及多个下游任务
3. 这为GUI Agent的研究提供了一个新的评估维度和优化方向

### 亮点二：全自动化的Web Grounding数据构造流水线

SeeClick设计了一套巧妙的自动化数据构造方法，无需人工标注即可从网页中大规模提取grounding训练数据：

- 解析HTML DOM树，提取可见文本元素和title属性元素
- 利用浏览器渲染引擎获取元素的精确屏幕坐标
- 自动生成自然语言形式的指令描述
- 同时构造双向任务（指令→坐标 + 坐标→指令），形成自监督信号

这套方法的关键优势在于**零标注成本**和**规模可扩展**——理论上可以从整个互联网持续获取新鲜数据，使模型能跟上Web UI设计的演进趋势。

### 亮点三：ScreenSpot —— 首个跨平台GUI Grounding基准

ScreenSpot填补了GUI Agent领域的一个重要空白：

- **覆盖5大平台**：iOS、Android、macOS、Windows、Web
- **包含600+张真实截图**和**1200+条指令**
- **区分两种元素类型**：基于文本的元素 vs 图标/控件类元素（后者对grounding来说难度更高）
- 暴露了现有LVLM在GUI场景下的严重不足：GPT-4V平均准确率仅16.2%，而经过GUI grounding预训练后的SeeClick达到53.4%

## 四、基准测试表现

### 4.1 ScreenSpot（自建Grounding基准）

| 模型 | 参数量 | GUI专用 | 移动(文本) | 移动(图标) | 桌面(文本) | 桌面(图标) | Web(文本) | Web(图标) | 平均 |
|------|--------|---------|-----------|-----------|-----------|-----------|----------|----------|------|
| MiniGPT-v2 | 7B | ✗ | 8.4% | 6.6% | 6.2% | 2.9% | 6.5% | 3.4% | 5.7% |
| Qwen-VL | 9.6B | ✗ | 9.5% | 4.8% | 5.7% | 5.0% | 3.5% | 2.4% | 5.2% |
| GPT-4V | - | ✗ | 22.6% | 24.5% | 20.2% | 11.8% | 9.2% | 8.8% | 16.2% |
| Fuyu | 8B | ✓ | 41.0% | 1.3% | 33.0% | 3.6% | 33.9% | 4.4% | 19.5% |
| CogAgent | 18B | ✓ | 67.0% | 24.0% | 74.2% | 20.0% | 70.4% | 28.6% | 47.4% |
| **SeeClick** | **9.6B** | **✓** | **78.0%** | **52.0%** | **72.2%** | **30.0%** | **55.7%** | **32.5%** | **53.4%** |

关键发现：
- SeeClick以9.6B参数超越18B的CogAgent，证明预训练策略的高效性
- 图标/控件的grounding准确率显著低于文本元素，说明这是行业共性难题
- GPT-4V在GUI grounding上表现远低于预期（16.2%），说明通用VLM不能直接用于GUI场景

### 4.2 下游Agent任务表现

**MiniWob（网页操作基准）：**
- SeeClick达到与Pix2Act相当的性能
- 但仅使用了Pix2Act **0.3%的训练数据**，证明了数据效率的优势

**AITW（Android-in-the-Wild，安卓操作基准）：**
- 在Android设备上的多步任务中展现出竞争力
- 作为纯视觉方法，不需要Android View Hierarchy即可运行

**Mind2Web（网页理解基准）：**
- 在需要复杂网页理解的任务中表现良好
- 再次验证了GUI grounding提升→Agent性能提升的正相关关系

## 五、横向对比分析

### 与文本基Agent对比（Mind2Web / WebArena类）

| 维度 | 文本基Agent | SeeClick |
|------|------------|----------|
| 平台覆盖 | 仅限有结构化数据的平台 | 全平台（含iOS/桌面原生） |
| 信息完整性 | 有DOM/Accessibility Tree但缺视觉 | 完整视觉信息但缺语义层级 |
| 上下文效率 | HTML可能超长（数万token） | 截图固定大小，更紧凑 |
| 元素定位精度 | 精确（DOM节点级） | 近似（坐标级，依赖grounding精度） |
| 部署复杂度 | 需要平台特定适配器 | 统一接口，部署简单 |

### 与其他视觉Agent对比

| 维度 | Pix2Act | CogAgent | SeeClick |
|------|---------|----------|----------|
| 坐标表示 | 离散化bin token | 自然语言数值 | 自然语言数值 |
| 数据需求 | 大规模任务特定数据 | GUI专用预训练 | 混合数据+自动化构造 |
| 开源程度 | ✅ | 部分 | ✅ 完全开源 |
| Grounding专项优化 | 无 | 有 | 核心创新点 |

### SeeClick的劣势

1. **分辨率限制**：448×448的输入分辨率对于高密度UI（如Excel表格、复杂仪表盘）可能不够
2. **缺乏高层规划能力**：专注于单步grounding，多步任务规划依赖prompt工程而非显式规划模块
3. **桌面端grounding较弱**：桌面图标/控件准确率仅30%，明显低于移动端的72.2%
4. **无记忆机制**：不支持跨会话的学习或长期记忆

## 六、对我们的实际启发

### 6.1 可落地的想法

**1. Grounding-first的Agent开发策略**
我们在开发GUI相关功能时，可以借鉴SeeClick的思路：先建立一个高质量的grounding能力基线，再在其上构建Agent逻辑。具体可以：
- 收集学长常用应用的截图+操作记录，构造内部grounding数据集
- 用小规模LoRA微调增强特定场景的定位能力

**2. 自动化数据构造流水线**
SeeClick的Web数据自动化提取方案可以直接复用。我们可以：
- 针对飞书/钉钉等常用SaaS，用类似方法从DOM中自动提取元素-坐标对
- 构造飞书操作的grounding训练数据，让楠楠更精准地操作飞书界面

**3. 纯视觉方案的备选路径**
当某些平台无法提供API或结构化数据时（如某些企业内部系统），纯视觉方案可以作为fallback策略。SeeClick证明了这条路是可行的。

**4. ScreenSpot式评估体系**
建立我们自己的grounding评估集，定期测试模型在不同GUI场景下的定位准确率，作为Agent能力退化的早期预警指标。

### 6.2 与现有技术栈的结合点

- 我们的智谱GLM-4V-Flash可用于类似SeeClick的视觉理解pipeline
- MiniMax的图像理解能力可作为grounding模型的替代方案
- 飞书文档操作目前走的是API路线（正确选择），但视觉方案可作为补充通道
