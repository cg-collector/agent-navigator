## 二、项目概述

Zep 是一个面向生产环境的 AI Agent 记忆基础设施平台，其核心创新在于**用动态时序知识图谱（Temporal Knowledge Graph）替代传统向量检索**来管理 Agent 的长期记忆。与 Mem0、Letta 等主流方案不同，Zep 不把记忆视为"文档集合"，而是将其建模为一个**随时间演化的知识网络**——事实有生效和失效时间，关系有方向和权重，实体可以聚合成社区。

Zep 的核心组件叫 **Graphiti**——一个时序感知的知识图谱引擎。它同时摄入非结构化对话数据和结构化业务数据，自动构建三层子图：情节子图（Episodic）、语义实体子图（Semantic Entity）和社区子图（Community）。这种分层设计直接借鉴了认知心理学中人类记忆的"情景记忆 vs 语义记忆"双系统模型。

在学术评测方面，Zep 在 DMR（Deep Memory Retrieval）基准上以 **94.8% 的准确率超越 MemGPT 的 93.4%**；在更具挑战性的 LongMemEval 基准上，部分任务准确率提升高达 **18.5%**，同时检索延迟降低 **90%**。

值得注意的是，Zep 团队于 2025 年宣布**停止维护社区版（Community Edition）**，转向纯商业云服务模式。现有开源代码仍以 Apache 2.0 协议保留在 GitHub 上，但不再接收更新。这一战略转变值得开发者关注。

## 三、核心架构

### 3.1 三层知识图谱结构

Zep 的知识图谱 $\mathcal{G}=(\mathcal{N},\mathcal{E},\phi)$ 由三个层级子图组成：

**第一层：情节子图（Episode Subgraph, $\mathcal{G}_e$）**
- 节点 = Episodes（原始数据单元），支持三种类型：message、text、JSON
- 每个 Episode 携带引用时间戳 $t_{ref}$，用于时序推理
- 边连接 Episode 到其引用的语义实体
- 本质是一个**无损原始数据存储**，所有后续提取都基于此层

**第二层：语义实体子图（Semantic Entity Subgraph, $\mathcal{G}_s$）**
- 节点 = Entities（从 Episode 中提取并消解的实体）
- 边 = Semantic Edges（实体间的关系）
- 核心过程：LLM 从原始消息中抽取实体和关系 → 与已有实体做消解（Entity Resolution）→ 写入图谱
- 这一层对应人类认知中的**语义记忆**

**第三层：社区子图（Community Subgraph, $\mathcal{G}_c$）**
- 节点 = Communities（强关联实体的聚类）
- 包含聚类的高层摘要
- 边连接社区到其成员实体
- 借鉴了 Microsoft **GraphRAG** 的全局理解思想

### 3.2 双时间模型（Bi-Temporal Model）

这是 Zep 最具特色的设计之一：

- **Timeline T（事件时间线）**：事件发生的真实时间顺序，用于回答"用户什么时候说过 X"
- **Timeline T'（事务时间线）**：Zep 系统摄入数据的顺序，用于审计和数据版本管理

双时间模型使得 Zep 能够处理相对时间表达（如"下周四"、"去年夏天"），并能正确标记事实的**有效期区间**——当用户改变偏好时，旧事实被标记为过期而非删除。

### 3.3 数据流架构

```
用户消息/业务数据 → Episode Ingestion → LLM Extraction
                                            ↓
                    ← Context Retrieval ← Graph Search ← Entity Resolution
                                            ↓
                              Temporal Fact Tracking
                                            ↓
                         Community Detection & Summarization
```

典型使用流程：
```python
# 1. 添加聊天消息到图谱
await zep_client.thread.add_messages(thread_id, messages=[
    Message(name="Jane", role="user", content="我穿 Adidas 的鞋")
])

# 2. 添加结构化业务数据
await zep_client.graph.add(user_id=user_id, type="json", data={
    "account_status": "suspended"
})

# 3. 获取当前对话相关的上下文记忆
context = zep_client.thread.get_user_context(
    thread_id="thread_id", mode="basic"
)
```

### 3.4 自定义本体（Custom Ontology）

Zep 允许通过 Pydantic（Python）或 Zod（TypeScript）定义领域特定的图实体类型，使记忆提取更精准：

```python
class MediaPreference(BaseModel):
    genre: str
    title: str
    publisher: str
    app: str
```

这种设计让企业可以将自身的数据模型直接映射到知识图谱中，大幅提升记忆的相关性和准确性。

## 四、技术亮点

### 4.1 时序感知的事实追踪

传统 RAG 方案将所有文档一视同仁地做向量相似度匹配，无法区分"用户上周说的偏好"和"用户今天改口的新偏好"。Zep 通过为每个事实附加有效时间区间 `[t_start, t_end]`，天然解决了**信息时效性**问题。当用户说"我以后改穿 Nike 了"，Zep 会将旧事实 `prefers(Adidas)` 标记为失效，同时创建新事实 `prefers(Nike)`，两段历史都保留可查。

### 4.2 非损失性记忆存储

Episode 层完整保存原始输入数据，不依赖 LLM 提取的准确性。即使初期提取有误，后续也可以重新运行提取流程修正图谱。这种**原始数据 + 派生知识**的双轨存储是工程上的重要考量。

### 4.3 亚 200ms 检索延迟

Zep 官方宣称 P95 检索延迟低于 200ms，这对语音 AI 等实时交互场景至关重要。相比需要将大量历史上下文塞入 LLM context window 的方案，Zep 的图谱检索在长程记忆场景下延迟优势显著（论文报告降低 90%）。

### 4.4 对话 + 业务数据融合

大多数记忆框架只处理对话消息。Zep 支持同时摄入 JSON 格式的结构化业务数据（如 CRM 记录、订单状态），并将其与对话中提取的知识统一到同一张图中。这意味着 Agent 可以同时知道"用户喜欢什么"（来自对话）和"用户的账户状态是什么"（来自业务系统）。

### 4.5 社区检测与高层摘要

借助社区发现算法（类似 Louvain 方法），Zep 自动识别图中高度关联的实体簇，并为每个社区生成摘要。这使得 Agent 在面对宽泛问题时可以获得**领域级概览**而非零散的事实列表。

## 五、适用场景

1. **个性化对话助手**：需要跨会话记住用户偏好的客服/陪伴型 Agent
2. **语音 AI / 车载助手**：对延迟敏感（<200ms）、需要长期记忆的实时交互场景
3. **企业知识管理 Agent**：需要融合内部业务数据（CRM/ERP）与员工对话记录
4. **医疗/法律垂直 Agent**：信息时效性至关重要（患者病史变化、合同条款更新）
5. **游戏 NPC / 虚拟角色**：需要持续学习玩家行为并演化人格的角色系统
6. **多轮任务型 Agent**：复杂工作流中需要在步骤间传递和维护状态

## 六、局限性

### 6.1 开源策略转向

**最关键的局限**：Zep 已于 2025 年停止维护社区版。虽然源码仍以 Apache 2.0 开放，但不再修复 bug 或接受 PR。对于坚持全栈开源的技术团队来说，这是一个重大风险信号。未来功能迭代仅限于商业云服务。

### 6.2 运维复杂度

自部署 Zep 需要 PostgreSQL（图谱存储）+ Redis（缓存/队列）+ 向量数据库（可选）+ 图谱构建服务，基础设施门槛高于简单的向量数据库方案（如 Mem0 + ChromaDB）。

### 6.3 LLM 提取成本

每次消息摄入都需要调用 LLM 进行实体和关系提取，这带来了持续的 API 成本和延迟开销。在高并发场景下，提取管道可能成为瓶颈。

### 6.4 冷启动问题

新用户或新会话的图谱几乎是空的，需要积累足够多的 Episode 后才能发挥知识图谱的优势。在冷启动阶段，Zep 的表现未必优于简单的历史消息窗口方案。

### 6.5 图谱质量依赖提取模型

知识图谱的质量上限由负责实体/关系提取的 LLM 决定。如果提取模型产生幻觉或遗漏关键信息，下游检索质量会受影响。虽然有 Episode 层作为纠错基础，但重新提取的成本不低。

### 6.6 缺乏标准化评估

尽管 Zep 引用了 DMR 和 LongMemEval 两个基准，但整个 Agent 记忆领域尚缺乏公认的工业级评估标准。不同框架在不同基准上的表现差异较大，选型时需结合实际场景验证。

### 选型建议

- **选 Zep 如果你**：需要亚秒级检索延迟、有复杂的业务数据融合需求、愿意使用商业云服务、对信息时效性要求极高
- **选 Mem0 如果你**：想要快速集成、全栈开源活跃、团队规模较小、以对话记忆为主
- **选 Letta 如果你**：构建角色驱动型 Agent（NPC/虚拟伴侣）、需要类操作系统的内存管理范式
- **选 LangMem 如果你**：做学术研究、需要灵活组合不同记忆策略、探索认知科学启发的记忆架构

## 九、总结与展望

Zep 代表了 Agent 记忆系统的一个重要方向：**从"检索文档"到"维护一个随时间变化的世界模型"**。其时序知识图谱的设计理念是当前最接近人类记忆运作机制的工程实现之一——我们确实记得事件（情景记忆），也拥有关于世界的抽象知识（语义记忆），并且这些知识会随着新经验的获取而更新。

然而，从开源社区的视角看，Zep 的商业化转向令人遗憾。Agent 基础设施层的开源竞争正在加剧，Mem0 凭借极低的集成门槛获得了更大的社区 momentum，而 Letta 在角色扮演赛道建立了差异化优势。Zep 如果想保持影响力，或许需要考虑开放核心（Open Core）模式而非完全收归闭源。

对于 OpenClaw 这样的 Agent 平台而言，Zep 的**时序事实追踪**和**双层存储（原始+派生）**设计思路值得借鉴——特别是在改进当前基于文件的记忆系统时，引入事实有效期概念和结构化实体提取能力，可以显著提升长程记忆的准确性和相关性。
