# AI Agent 学习报告 #098 — OpenAI Agents SDK (openai-agents-python)

## 项目名
**OpenAI Agents SDK**（Python 版）

## GitHub 链接
https://github.com/openai/openai-agents-python

## 一句话简介
OpenAI 官方出品的轻量级多 Agent 工作流框架，以极简原语（Agent / Handoff / Guardrail）构建生产级 Agentic 应用，月下载量超 1000 万次。

## 架构设计

### 核心模块

项目源码结构清晰，核心位于 `src/agents/` 下：

| 模块 | 文件 | 职责 |
|------|------|------|
| **Agent 核心** | `agent.py` | Agent 定义：instructions、tools、model、handoffs |
| **运行引擎** | `run.py` + `run_internal/` | Runner 调度循环：LLM调用→工具执行→结果回传→终止判断 |
| **Handoff 机制** | `handoffs/` | Agent 间委托转移，支持 input_filter / on_handoff 回调 |
| **Guardrails** | `guardrail.py` + `tool_guardrails.py` | 输入输出安全校验，并行执行 fail-fast |
| **Tracing** | `tracing/` | 内置 Trace/Span 追踪，BatchTraceProcessor 批量上报 |
| **Memory** | `memory/` | Session 持久化记忆层（支持 Redis 后端） |
| **MCP 集成** | `mcp/` | 原生 MCP Server 工具调用，与 function tool 统一接口 |
| **Realtime/Voice** | `realtime/` + `voice/` | 基于 gpt-realtime-1.5 的语音 Agent 管线 |
| **工具系统** | `tool.py` + `function_schema.py` | Python 函数 → Tool 的自动 schema 生成（Pydantic 驱动） |

### 技术栈
- **语言**: Python 3.10+（利用 `contextvars` 实现并发安全 tracing）
- **依赖**: Pydantic（schema 校验）、MCP Python SDK、any-llm / LiteLLM（多模型支持）
- **构建**: uv + ruff（lint）、mypy + Pyright（双类型检查）
- **可选**: websockets（实时通信）、SQLAlchemy / Redis（持久化）

### 工作流程

```
用户输入 → Runner.run()
    ↓
Agent Loop 开始
    ↓
┌─ LLM 调用（generation_span 记录）
│   ├─ 模型返回文本 → 检查是否需要 tool_call
│   └─ 模型返回 tool_call → 进入工具执行
│       ├─ Function Tool → 本地函数执行（function_span）
│       ├─ MCP Tool → 远程 MCP Server 调用
│       ├─ Handoff Tool → 委托给目标 Agent（handoff_span）
│       │   └─ on_handoff() 回调 → input_filter 过滤历史 → 新 Agent 接管
│       └─ Agent-as-Tool → 子 Agent 独立运行后返回结果
│           ↓
│       结果回传 LLM（带 guardrail 校验）
│       ↓
│   循环直到模型输出纯文本（无更多 tool_call）
    ↓
输出 Result（final_output + items 列表）
    ↓
Tracing 数据批量上报 OpenAI Backend
```

关键设计：**Runner 是单线程事件循环**，不是异步图编排。每个 Agent turn 是同步的 `LLM call → tool exec → LLM call` 循环。多 Agent 通过 Handoff 实现"控制权转移"而非并行调度。

## 技术亮点

### 1. Handoff 作为 Tool 的统一抽象 — 极致简洁的多 Agent 编排

这是整个框架最精妙的设计。Handoff 不是独立的编排概念，而是**被实现为一种特殊的 Tool**：

```python
# handoff 在内部被注册为 function tool
# 默认 tool_name = "transfer_to_{agent.name}"
# LLM 自己决定何时调用哪个 handoff tool

triage_agent = Agent(
    name="Triage agent",
    instructions="根据用户问题路由到专业 Agent",
    handoffs=[
        billing_agent,                    # 直接传 Agent 实例
        handoff(refund_agent,             # 用 handoff() 包装，可定制
            on_handoff=log_escalation,     # 回调钩子
            input_type=EscalationData,     # 让 LLM 传结构化元数据
            input_filter=filter_history,   # 控制下个 Agent 看到什么历史
        )
    ]
)
```

**为什么这个设计好？**
- 不需要引入 DAG / 图编排等重量级概念
- LLM 天然具备"选择调用哪个函数"的能力，Handoff 复用了这个能力
- `input_filter` 参数让开发者精确控制上下文传递——可以摘要、裁剪、或完全替换对话历史
- `on_handoff` 回调支持副作用（如日志记录、数据预取），不阻塞主流程
- `is_enabled` 支持动态启停，可以用函数在运行时决定某个 handoff 是否可用

对比 CrewAI 的 Process（顺序/层级）和 AutoGen 的 GroupChat（轮转发言），OpenAI 的 Handoff 方式更**声明式**且**LLM-native**。

### 2. 内置 Tracing 系统 — 基于 contextvars 的零侵入追踪

Tracing 是默认开启的，架构设计非常优雅：

```python
# 使用 Python contextvar 跟踪当前 trace，天然支持 async 并发
class Trace:
    _current: ContextVar[Optional["Trace"]] = ContextVar("trace_current", default=None)

# 用户代码只需一行 with 语句
with trace("Customer support workflow"):
    result1 = await Runner.run(triage_agent, user_message)
    result2 = await Runner.run(resolution_agent, result1.final_output)
# 两次 run() 自动归入同一个 trace

# Span 层级自动嵌套（也是 contextvar 驱动）
# trace() → agent_span() → generation_span() / function_span()
```

Span 类型覆盖完整：
- `generation_span` — LLM 输入/输出（可配置是否捕获敏感数据）
- `function_span` — 工具调用的参数和返回值
- `handoff_span` — Agent 转移事件及元数据
- `guardrail_span` — 安全校验结果
- `transcription_span` / `speech_span` — 音频处理

**自定义 Processor 扩展能力**：
```python
# 可添加额外 processor 或完全替换默认的
add_trace_processor(my_custom_processor)  # 追加
set_trace_processors([my_processor])      # 替换（不再发到 OpenAI）
```

这个设计对 OpenClaw 有直接参考价值——我们的 subagent 调用链路如果也有类似的 trace/span 埋点，调试会方便很多。

### 3. Guardrails 并行校验 — Fail-Fast 安全网

Guardrail 可以同时作用于输入和输出，并且**与 Agent 主循环并行执行**：

```python
from agents import Agent, GuardrailResult, RunContextWrapper

async def pii_guardrail(ctx: RunContextWrapper, agent_input: str):
    if contains_pii(agent_input):
        return GuardrailResult(
            output_message="检测到敏感信息，请重新描述",
        )
    return GuardrailResult(output_message=None)  # 放行

agent = Agent(
    name="Safe agent",
    input_guardrails=[pii_guardrail],   # 输入校验
    output_guardrails=[toxicity_check], # 输出校验
)
```

关键特性：
- 多个 guardrail **并行执行**（非串行），任一失败立即 fail-fast
- `triage_guardrail` 支持：输入和输出用同一套规则
- `tool_guardrails` 可以为每个 tool 单独设置校验逻辑
- 返回 `GuardrailResult.output_message` 时，该消息会作为最终输出返回给用户（不会送入 LLM）

这比 LangChain 的 output parser 异常处理要系统化得多。

### 4. Agent-as-Tool 模式 — 组合优于继承

一个 Agent 可以被包装成另一个 Agent 的 Tool：

```python
researcher = Agent(
    name="Researcher",
    tools=[web_search],
    instructions="搜索并总结信息"
)

writer = Agent(
    name="Writer",
    tools=[researcher.as_tool(  # 将 researcher 变为 writer 的工具
        name="research",
        description="研究某个主题"
    )],
    instructions="基于研究结果写文章"
)
```

这与 Handoff 的区别：
- **Handoff** = 控制权永久转移，新 Agent 接管整个对话
- **Agent-as-Tool** = 子 Agent 独立运行后返回结果，父 Agent 继续工作

两种模式覆盖了多 Agent 协作的两大场景：**流水线**（Handoff 链式传递）和**主从**（Manager 调用 Worker）。

### 5. Model Provider Agnostic 设计 — 通过抽象层解耦

虽然叫 "OpenAI" Agents SDK，但它通过 `ModelProvider` 抽象支持 100+ 模型：

```python
# 内部使用 any-llm 和 LiteLLM 做 provider 适配
# 核心接口定义在 models/ 目录

from agents import Agent, OpenAIChatCompletionsModel
from openai import AsyncOpenAI

# 非 OpenAI 模型示例
client = AsyncOpenAI(base_url="http://localhost:11434/v1", api_key="ollama")
model = OpenAIChatCompletionsModel(chat_completion_client=client, model="llama3")

agent = Agent(name="Local", model=model, model_settings=...)
```

Responses API（新版 function calling 原生支持）和 Chat Completions API（兼容性）双模式并存，通过 `model_settings` 统一配置。

## 对我们（OpenClaw/楠楠）的启发

### 🔍 直接可借鉴的点

1. **Handoff 思想用于 Subagent 路由**
   - 当前 OpenClaw 的 subagent 是 spawn-and-forget 模式。可以借鉴 Handoff 的 `input_filter` + `on_handoff` 模式，让 main agent 能更精细地控制传递给 subagent 的上下文（截断、摘要、注入特定指令），并在 subagent 启动时触发回调做日志/监控。
   
2. **Tracing 值得引入**
   - OpenAI SDK 的 contextvar-based tracing 几乎零侵入。我们可以为 OpenClaw 的 agent run 链路加上类似的结构化 trace：每次 tool call、subagent spawn、message send 都是一个 span。这对调试长链路任务（如股票分析 multi-turn）极有价值。

3. **Guardrails 的并行 Fail-Fast 模式**
   - 我们的飞书消息发送、API 调用等操作可以加类似的安全检查层。比如发消息前并行跑：内容长度检查 + 敏感词过滤 + 频率限制，任一失败立刻拦截。

4. **Agent-as-Tool vs Handoff 的二分法**
   - 这个分类很清晰。OpenClaw 目前的 coding-agent spawn 类似 Agent-as-Tool（子 agent 干完活返回结果）。如果我们以后要做多 agent 对话场景（比如楠楠+桐桐同时在线），Handoff 的控制权转移模型更适合。

### 💡 更宏观的启发

5. **"少即是多"的原语设计哲学**
   - 整个 SDK 核心就三个概念：Agent、Handoff、Guardrail。没有 Chain、PromptTemplate、OutputParser、MemoryBackend 等一堆中间抽象。但通过组合这三个原语就能表达复杂的工作流。这对我们设计 skill 系统和工具接口有启发——**保持原语少而强，靠组合而非堆砌功能**。

6. **内置 Tracing = 降低采用门槛**
   - 大多数框架把可观测性当插件。OpenAI 把它做成默认开启的一等公民。结果是开发者第一次运行就能看到完整的执行流程可视化。这种"开箱即用的可观测性"对用户体验的提升是质的。

7. **MCP 原生集成**
   - SDK 直接内置了 MCP client，不需要额外适配器。这意味着任何 MCP server 暴露的工具都能直接变成 Agent 的 tool。OpenClaw 如果未来要接入 MCP 生态，可以参考其 `mcp/` 模块的封装方式。
