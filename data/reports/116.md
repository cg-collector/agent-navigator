## 架构设计

### 整体架构：Harness 而非 Framework

DeerFlow 2.0 的定位不是"给你工具让你拼"的 Framework（如 LangChain/LangGraph），而是**开箱即用的 Harness**——内置最佳实践、batteries included、更有主见（opinionated）。

### 核心模块

#### 1. Lead Agent（单一主 Agent）

整个系统只有一个 `lead_agent`，通过 `make_lead_agent(config)` 工厂函数创建：

```python
def make_lead_agent(config):
    return create_agent(
        model=...,          # 动态选择的大语言模型
        tools=...,          # 动态组装的工具集
        middleware=...,     # 14层中间件链
        system_prompt=...,  # 动态生成的系统提示词
    )
```

**设计哲学：不靠多角色分工，靠工具集扩展 + 中间件基础设施 + 子Agent动态调度。**

#### 2. Middleware 链（14层洋葱模型）

这是 DeerFlow 最核心的设计创新。14个 Middleware **严格有序**，采用洋葱模式（进入时由外到内、退出时由内到外），顺序写死在代码中：

| 层级 | Middleware | 核心职责 |
|------|-----------|---------|
| 1 | DanglingToolCallMiddleware | 修补缺失的 ToolMessage（修复 LangChain 历史污染） |
| 2 | SandboxMiddleware | 注入沙盒状态到 ThreadState |
| 3 | ThreadDataMiddleware | 提取 thread_id，创建线程隔离目录 |
| 4 | UploadsMiddleware | 处理文件上传注入上下文 |
| 5 | SummarizationMiddleware | 上下文超长时自动摘要压缩 |
| 6 | TodoListMiddleware | Plan 模式下提供 write_todos 工具 |
| 7 | TokenUsageMiddleware | Token 计量统计 |
| 8 | TitleMiddleware | 首次对话后自动生成标题 |
| 9 | MemoryMiddleware | 对话入队，异步更新长期记忆 |
| 10 | ViewImageMiddleware | 视觉模型图片内容注入 |
| 11 | DeferredToolFilterMiddleware | 工具过多时延迟暴露 + tool_search |
| 12 | SubagentLimitMiddleware | 截断超并发的 task() 调用 |
| 13 | LoopDetectionMiddleware | 滑动窗口 hash 检测重复工具调用 |
| 14 | ClarificationMiddleware | 拦截澄清请求（**必须最后执行**） |

**关键工程决策：顺序依赖是显式的，不是约定。** 比如 SandboxMiddleware(2) 需要 thread_id，而 ThreadDataMiddleware(3) 才负责注入它——洋葱模型的退出阶段让外层能读到内层数据。

#### 3. Sub-Agent 编排系统

```python
# task_tool 内部轮询逻辑（简化）
async def task_tool(...):
    task_id = executor.execute_async(prompt)
    
    while True:
        result = get_background_task_result(task_id)
        
        if result.status == COMPLETED:
            writer({"type": "task_completed", ...})
            return f"Task Succeeded. Result: {result.result}"
        elif result.status == FAILED:
            return f"Task failed. Error: {result.error}"
        
        await asyncio.sleep(5)  # 5秒轮询间隔
        
        if poll_count > max_poll_count:  # 超时保护
            return "Task polling timed out..."
```

**关键设计：**
- **LLM 无感知轮询** —— 对 LLM 来说 `task()` 就是同步调用，不会消耗 token 做轮询
- **双线程池架构**：调度线程池（排队编排）+ 执行线程池（实际运行，支持超时）
- **硬编码防递归**：子 Agent 的 `subagent_enabled=False`，拿不到 `task()` 工具
- **并发上限**：默认单次最多 3 个子 Agent 并行
- **实时进度推送**：通过 LangGraph `stream_writer` 推送 SSE 事件到前端

#### 4. Sandbox 沙盒系统

```python
class AioSandbox(Sandbox):
    def __init__(self, id: str, base_url: str, ...):
        self._client = AioSandboxClient(base_url=base_url, timeout=600)

    def execute_command(self, command: str) -> str:
        result = self._client.shell.exec_command(command=command)
        return result.data.output
```

三种模式：
- **LocalSandboxProvider**：本地文件系统（开发用）
- **AioSandboxProvider**：Docker 容器隔离（生产推荐）
- **K8s Provider**：Kubernetes Pod 级别隔离（大规模部署）

虚拟路径映射：
```
/mnt/user-data/{workspace,uploads,outputs} → thread-specific 物理目录
/mnt/skills → deer-flow/skills/ 目录
```

**双重安全防护：**
1. 容器级别隔离（防止攻击扩散到宿主机）
2. 命令审计正则匹配（`SandboxAuditMiddleware`）：

### 工作流程

```
用户输入 → Nginx路由 → Gateway预处理
    ↓
Lead Agent 接收请求
    ↓
[洋葱式 Middleware 链处理]
  ├─ ThreadDataMiddleware: 创建线程隔离工作区
  ├─ UploadsMiddleware: 注入上传文件
  ├─ SandboxMiddleware: 分配沙盒环境
  ├─ SummarizationMiddleware: 检查是否需要压缩上下文
  ├─ DeferredToolFilter: 决定全量/延迟暴露工具
  └─ ... (共14层)
    ↓
LLM 推理 → 工具调用 / 子Agent派发 / 直接回复
    ↓
[退出 Middleware 链（逆序）]
  ├─ MemoryMiddleware: 异步队列提取记忆
  ├─ LoopDetectionMiddleware: 检测循环
  └─ ClarificationMiddleware: 拦截澄清请求
    ↓
流式响应返回前端/IM渠道
```

## 技术亮点

### 亮点1：14层有序 Middleware 洋葱模型

这不是简单的 AOP 面向切面编程，而是一套**有严格依赖关系的管道架构**：

```python
# 中间件链组装（伪代码）
middlewares = [
    DanglingToolCallMiddleware(),    # 1: 最外层，修复历史数据
    SandboxMiddleware(),             # 2: 需要thread_id，但依赖第3层的注入
    ThreadDataMiddleware(),          # 3: 内层先注入thread_id，外层退出时读到
    UploadsMiddleware(),             # 4: 依赖thread_id做文件路由
    SummarizationMiddleware(),       # 5: 上下文压缩
    TodoListMiddleware(),            # 6: 计划模式
    TokenUsageMiddleware(),          # 7: 计量
    TitleMiddleware(),               # 8: 标题生成
    MemoryMiddleware(),              # 9: 异步记忆入队
    ViewImageMiddleware(),           # 10: 条件触发
    DeferredToolFilterMiddleware(),  # 11: 工具延迟加载
    SubagentLimitMiddleware(),       # 12: 并发控制
    LoopDetectionMiddleware(),       # 13: 循环检测
    ClarificationMiddleware(),       # 14: 必须最内层（最后拦截）
]
```

**为什么这个设计值钱？** 大多数框架的 Middleware 是"顺序无关"的平铺列表。DeerFlow 把横切关注点（线程隔离、沙盒分配、上下文压缩、记忆更新、循环检测……）全部解耦成独立中间件，新加能力只需插入一层，不影响现有逻辑。这是**真正的关注点分离**。

### 亮点2：LoopDetectionMiddleware —— 滑动窗口 Hash 防循环

```python
def _hash_tool_calls(tool_calls: list[dict]) -> str:
    # 归一化：只保留 name + args
    normalized = [{
        "name": tc.get("name", ""),
        "args": tc.get("args", {}),
    } for tc in tool_calls]

    # 排序使得顺序无关（先A后B ≡ 先B后A）
    normalized.sort(key=lambda tc: (
        tc["name"],
        json.dumps(tc["args"], sort_keys=True, default=str),
    ))

    blob = json.dumps(normalized, sort_keys=True, default=str)
    return hashlib.sha256(blob.encode()).hexdigest()[:16]
```

**两档响应机制：**
- 连续相同 hash **3次** → 注入警告提示（"你似乎在重复相同的操作"）
- 连续相同 hash **5次** → 强制剥离所有 tool_calls，逼 LLM 输出最终答案

**成本几乎为零**（一次 sha256），但能有效阻断 Agent stuck 场景。滑动窗口只跟踪最近 N 次（默认20），不会被早期重复"冤枉"。这个设计可以直接移植到任何基于工具调用的 Agent 系统。

### 亮点3：DeferredToolFilter —— RAG 思想应用在工具层

当 MCP 工具数量膨胀到几十上百个时，全部塞进 system prompt 会：
1. 稀释模型注意力 → 选工具准确率下降
2. Token 成本蹭蹭上涨

DeerFlow 的解法：

```python
class DeferredToolFilterMiddleware:
    def __init__(self, tools, threshold=20):
        self.all_tools = tools
        self.threshold = threshold
    
    async def process(self, state):
        if len(self.all_tools) <= self.threshold:
            # 工具少：直接全量注入，零额外开销
            state.tools = self.all_tools
        else:
            # 工具多：只暴露 tool_search，按需检索
            state.tools = [tool_search_tool]
            state.tool_registry = self.all_tools  # 供检索使用
```

本质是把 **RAG（检索增强生成）的思想从知识域迁移到了工具域**——工具也是一种"知识"，按需检索比全量注入更高效。自适应阈值（默认20）让小场景零负担。

### 亮点4：Sub-Agent 内置轮询 —— 不让 LLM 做它不擅长的事

传统 multi-agent（如 AutoGPT 时代）的做法：
```
LLM: 调task("调研X") → 返回"任务进行中"
LLM: 调check_status() → 返回"还在跑"
LLM: 调check_status() → 返回"还在跑"
LLM: （开始走神，忘记原始目标...）
```

DeerFlow 的做法：
```
LLM: 调task("调研X") → [内部轮询5s间隔] → 直接返回最终结果
```

**LLM 完全无感知**。轮询逻辑封装在 `task_tool` 内部，对 LLM 来说就是一个同步调用。同时通过 `stream_writer` 向前端推送 `task_running` / `task_completed` 事件，用户体验不打折。

配合 **硬编码防递归**（子 Agent 拿不到 task 工具）和 **并发上限**（默认3个），形成完整的子任务安全编排体系。

### 亮点5：结构化记忆的时效分层 + 置信度衰减

```json
{
  "user": {
    "workContext": {},      // 当前工作——最高优先级
    "personalContext": {},   // 个人背景——中等优先级  
    "topOfMind": {}         // 当前关注——动态更新最快
  },
  "history": {
    "recentMonths": {},     // 近期——时效性强
    "earlierContext": {},    // 较早——背景参考
    "longTermBackground": {} // 长期——低频访问
  }
}
```

对比大多数 Agent 框架的"对话历史全部塞 prompt"或"纯文本摘要文件"，DeerFlow 的设计优势在于：
- **语义分层 > 时间排序**：三个月前的技术选型比昨天的闲聊更值得记住
- **置信度连续谱 > 二值记住/忘记**：从观察到确信是一个渐进过程
- **per-agent 隔离**：多 Agent 场景下记忆互不污染
- **MemoryStorage 抽象类**：可替换为数据库存储，接口稳定

## 对 OpenClaw / 楠楠 的启发

### 🔥 高价值可借鉴

#### 1. 循环检测机制（LoopDetection）—— 建议直接引入

楠楠在做复杂任务（研究、分析、代码生成）时确实可能出现"反复调用同一个工具但参数差不多"的情况。DeerFlow 的滑动窗口 hash 方案实现成本极低（<100行），效果显著。可以在 OpenClaw 的 agent 执行层加一个类似的中间件。

#### 2. 记忆时效分层思想

我们目前的记忆系统是 Markdown 平铺文件（daily notes + MEMORY.md）。可以借鉴 DeerFlow 的分层思路：
- **当前活跃层**：最近1-2天的详细记录
- **近期参考层**：近1-2周的摘要
- **长期沉淀层**：经过提炼的关键事实（类似 facts with confidence）

这样 heartbeat 整理记忆时可以有明确的归档策略，而不是每次都扫一遍所有 daily notes。

#### 3. DeferredToolFilter（工具延迟加载）

随着楠楠接入的 MCP 工具和 Skills 越来越多，system prompt 里的工具描述会越来越长。可以借鉴"工具数量 < 阈值时全量注入，超过阈值时提供 tool_search 索引"的自适应策略。

#### 4. Middleware 管道化架构

OpenClaw 目前的横切关注点（恋爱模拟器事件触发、心跳检查、记忆维护等）散落在各处。DeerFlow 的洋葱模型提供了一个清晰的思路：把每个横切关注点做成独立的、有序的中间件，进入时初始化、退出时清理。

#### 5. Sub-Agent 轮询透明化

目前我们的 subagent 是 push-based（完成后自动通知），这其实已经比 DeerFlow 的轮询方案更优雅。但 DeerFlow 的**防递归**（子 Agent 不能再派生子 Agent）和**并发上限**机制值得参考——防止意外的 subagent 爆炸。

### ⚠️ 不建议照搬的

| 设计 | 原因 |
|------|------|
| Docker 沙盒依赖 | 我们是轻量级 VPS 部署，Docker 开销太大；本地 exec 更适合我们的场景 |
| 部署复杂度 | Docker + Python 3.12 + Node.js 22 + K8s 可选，对个人开发者门槛太高 |
| Skills 必须绑沙盒 | 我们的 Markdown-driven Skills 更灵活，不需要 Python 脚本执行环境 |
| 无 cron 支持 | 我们已有的 cron 系统反而是优势 |

### 📊 总结评价

| 维度 | 评分 | 说明 |
|------|------|------|
| 架构设计 | ⭐⭐⭐⭐⭐ | 14层Middleware + 单Agent + 动态编排，工程设计水准极高 |
| 工程完整度 | ⭐⭐⭐⭐⭐ | 从沙盒到记忆到IM渠道，端到端完整 |
| 创新程度 | ⭐⭐⭐⭐ | 循环检测、工具RAG、记忆分层都是实用创新 |
| 部署友好度 | ⭐⭐ | 依赖重，启动链路长 |
| 与我们相关性 | ⭐⭐⭐⭐ | 中间件思想、循环检测、记忆分层高度相关可借鉴 |

**一句话总结：DeerFlow 2.0 是目前开源 Agent 框架中工程设计最完整的项目之一。它最大的贡献不是某个单一功能，而是一套"让 Agent 能真正跑完复杂长任务"的系统方法论——Middleware 管道化、子任务编排透明化、记忆结构化、执行隔离化。这些设计思想的借鉴价值远大于直接使用这个框架。**
