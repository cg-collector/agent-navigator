## 一、项目背景与问题定义

GUI Agent 领域存在三大矛盾：**准确率 vs 延迟 vs 跨平台兼容性**。当前方法大致分为三类：

1. **语言方法**（基于 HTML/XML）：准确率高但仅限 Web，存在 HTML 注入安全风险，跨平台兼容性差
2. **视觉+语言方法**（结合截图与 HTML）：同样依赖 HTML 获取，Web 外场景受限
3. **纯视觉方法**（仅截图输入）：跨平台兼容性好、延迟低，但**元素定位准确率显著落后**

SpiritSight 要解决的核心痛点：**纯视觉 GUI Agent 的元素定位（element grounding）能力不足**。现有纯视觉方法依赖额外工具（OCR、图标识别模型）来辅助定位，增加了系统复杂度和推理延迟。SpiritSight 旨在构建一个**端到端、纯视觉、无需额外工具**的 GUI Agent，实现"One Look"——看一眼就能准确定位并操作。

## 二、核心架构深度剖析

### 2.1 整体架构

SpiritSight 基于 InternVL2 VLM 架构，采用端到端设计：输入 GUI 截图 + 用户指令，直接输出操作动作（点击坐标 + 操作类型）。发布三个尺寸版本：**2B / 8B / 26B**。

关键流程：
1. **预训练阶段**：在自建的 GUI-Lasagne 数据集（573万样本）上预训练，获得三级 GUI 理解能力
2. **推理阶段**：使用 UBP（Universal Block Parsing）方法处理动态分辨率输入，输出 (x, y) 坐标 + 操作类型

### 2.2 GUI-Lasagne 数据集（核心数据工程）

采用**分层、低成本、可扩展**的数据收集策略，共 573 万样本，分三级：

| 层级 | 任务 | 数据量 | 数据来源 | 收集成本 |
|------|------|--------|----------|----------|
| Level 1 | 视觉-文本对齐（text2bbox, bbox2text, bbox2dom） | 300万 | CommonCrawl网页 + AitW移动端 | 免费（自动化） |
| Level 2 | 视觉-功能对齐（function2bbox） | 150万 | VLM 自动标注 + LLM 增强 | 极低（无人工） |
| Level 3 | GUI 导航（轨迹数据） | 123万 | 公开数据集（AitW, Mind2Web等） | 零成本（复用） |

**Level 1 数据构造细节：**
- 使用 Playwright 从 CommonCrawl 收集 755K 真实网页截图 + DOM 数据
- 自训练 InternVL-Icon 模型（基于 3万图标-描述对微调）标注网页图标
- 设计三种双向任务：text→bbox（定位）、bbox→text（识别）、bbox→DOM（布局理解）
- **数据打包策略**：每个训练样本塞入多个数据对，充分利用模型上下文长度

**Level 2 数据构造细节：**
- 借鉴机器翻译中 back-translation 思想：用 InternVL2-26B 理解页面，自动生成元素的"功能描述"
- 将截图分为 3×3 网格，提供粗略位置信息 + 精确 bbox，让 VLM 推断元素功能
- 用 InternLM2.5-20B 增强生成质量，人工验收通过率 90.9%
- **关键洞察**：前两级占总量 90%，且完全免费收集，大幅降低训练成本

### 2.3 Universal Block Parsing（UBP）方法

这是 SpiritSight 的核心技术贡献。

**问题背景：** 现代 VLM（如 InternVL2, Qwen2-VL）采用动态高分辨率策略——将高分辨率图像切成多个 tile 分别编码。但 SpiritSight 发现这会引入**定位歧义**：同一个 bbox 坐标可能出现在不同 tile 中，模型无法区分该坐标对应哪个 tile，导致训练时学到的定位信号混乱。

**UBP 解决方案：**
- 在推理和训练时，将输出坐标除以对应 tile 的分辨率比例，**统一归一化到全局坐标系**
- 具体而言：对每个 tile 内的检测结果，根据该 tile 在原图中的位置偏移和缩放比例，将局部坐标转换为全局坐标
- 这样模型输出的坐标直接对应原始截图坐标，消除多 tile 切分带来的歧义

## 三、技术亮点

### 亮点一：分层数据工程——90% 免费、零人工的 573 万数据集

传统 GUI 数据集要么依赖人工标注（成本高、规模受限），要么是合成数据（不够真实）。SpiritSight 的分层策略巧妙之处在于：
- Level 1-2（共 450 万）完全自动化收集，利用 VLM 自身能力生成标注，形成"自举循环"
- Level 3 直接复用现有公开数据集，零成本
- 仅用少量人工验证质量（90.9% 通过率），而非大规模人工标注

这种"用 AI 收集训练数据给 AI 用"的数据飞轮模式，对其他垂直领域也有启发。

### 亮点二：UBP 消除动态分辨率歧义

动态高分辨率是现代 VLM 的标配能力，但 SpiritSight 首次发现它在 GUI 定位场景中的歧义问题并提出简洁解决方案。UBP 的优雅之处在于：
- **不改变模型架构**，仅调整坐标解析逻辑
- **不增加推理成本**，只是一个简单的坐标变换
- 效果显著：在 Mind2Web 上 Step SR 提升约 6-10 个百分点

### 亮点三：End-to-End 纯视觉设计，无需额外 grounding 工具

传统两阶段方法（如 MobileAgent）需要 OCR + 图标识别 + 候选元素筛选等多个外部工具链，SpiritSight 将所有能力内化到单个 VLM 中：
- 文本识别 → 内化在 Level 1 训练中
- 图标识别 → 通过 InternVL-Icon 标注数据内化
- 功能理解 → 通过 Level 2 训练内化
- DOM 布局理解 → 通过 bbox2dom 任务内化

**单次前向传播即可完成"看图→理解→定位→操作"全流程**，真正实现 One Look。

## 四、基准测试表现

### Multimodal-Mind2Web（核心 benchmark，无候选元素设置）

| 模型 | 尺寸 | Cross-Task Step SR | Cross-Website Step SR | Cross-Domain Step SR |
|------|------|-------------------|----------------------|---------------------|
| OmniParser | - | 39.4% | 36.5% | 42.0% |
| SeeClick | 9.6B | 25.5% | 16.4% | 20.8% |
| CogAgent | 18B | - | - | 59.4%（有候选） |
| HTML-T5-XL | 3B | 71.5% | 62.2% | 67.1%（有候选） |
| SeeAct | - | 40.2% | 32.4% | 36.8% |
| **SpiritSight-2B** | **2B** | **44.9%** | **37.8%** | **36.9%** |
| **SpiritSight-8B** | **8B** | **52.7%** | **44.0%** | **44.4%** |
| **SpiritSight-26B** | **26B** | **54.7%** | **48.1%** | **49.2%** |

在无候选元素（open-set）设置下，SpiritSight 全面超越所有纯视觉方法，甚至超越部分有候选的方法。

### 其他 benchmark

- **AitW（移动端）**：在 OS, Android 两平台均达到 SOTA
- **OSWorld**：在纯视觉方法中领先
- **ScreenSpot**：Grounding 准确率大幅领先，8B 版本即超越 18B 的 CogAgent

**关键观察**：SpiritSight-8B（8B 参数）在很多场景下超越了 18B 的 CogAgent 和 340B 的 ReadAgent，展示了数据工程和架构优化的杠杆效应。

## 五、与同类项目横向对比

| 维度 | SpiritSight | CogAgent | MobileAgent-v2 | SeeAct | UI-TARS |
|------|------------|----------|----------------|--------|---------|
| 输入模态 | 纯视觉 | 纯视觉 | 纯视觉+OCR | 视觉+HTML | 纯视觉 |
| 架构 | 单阶段端到端 | 单阶段 | 两阶段（GPT-4V+工具） | 单阶段 | 单阶段 |
| 定位方式 | bbox坐标 | bbox坐标 | 工具辅助 | HTML候选 | bbox坐标 |
| 跨平台 | Web+Mobile+Desktop | Web为主 | Mobile | Web | Web+Mobile |
| 开源 | 是（计划） | 是 | 否（GPT-4V） | 否 | 是 |
| 主要优势 | 高精度+低延迟 | 成熟稳定 | 利用闭源模型 | 结合HTML信息 | 持续学习 |

**SpiritSight 的核心优势**：在纯视觉方法中，以更小的模型参数实现了更高的准确率，真正做到了"one look"的体验。

**劣势**：
- 依赖大规模预训练数据，复现成本较高
- 目前尚未完全开源（论文承诺发布后开源）
- 在有候选元素的设置下，仍不如 HTML-T5 等利用结构信息的方法

## 六、对我们的实际启发

### 1. 分层数据飞轮策略适用于 OpenClaw Browser Agent

OpenClaw 的 browser 工具目前依赖 snapshot（DOM 文本）进行元素定位。可以借鉴 SpiritSight 的思路：
- **Level 1**：自动收集 Playwright 截图 + DOM 对，构建视觉-文本对齐数据
- **Level 2**：用现有 VLM 自动标注功能描述，增强 grounding 能力
- 如果未来想构建自己的视觉 GUI Agent，这套数据工程管线可以直接复用

### 2. UBP 坐标归一化思想

在处理高分辨率截图时，tile 切分的坐标歧义问题是通用的。如果我们自己的 agent 需要处理高分辨率 GUI 截图，UBP 的坐标归一化思路值得直接采用。

### 3. "One Look" 设计哲学

SpiritSight 证明了：与其堆叠外部工具链（OCR → 候选筛选 → 排序 → 选择），不如把能力内化到一个模型中。对于我们的 agent 设计也有启发：**减少工具调用轮次，提升单次决策质量**，可以显著降低延迟和出错率。

### 4. 数据 > 模型规模

SpiritSight-8B 超越 18B/340B 模型的核心原因不是更好的架构，而是 **573 万高质量 GUI 数据**。这对我们的启示：在垂直场景（如飞书操作、股票分析），投入数据工程（收集、标注、构造）的 ROI 远大于直接换更大的模型。
