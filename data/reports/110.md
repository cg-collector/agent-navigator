## 架构设计

### 核心模块（6 大子系统）

```
┌─────────────────────────────────────────────────────┐
│                    CAMEL Framework                   │
├──────────┬──────────┬──────────┬──────────┬─────────┤
│  Agents   │ Societies│ Models   │ Memory   │ Tools   │
│          │          │          │          │         │

├──────────┴──────────┴──────────┴──────────┴─────────┤
│              DataGen │ Benchmarks │ Runtime           │
└─────────────────────────────────────────────────────┘
```

### 技术栈
- **语言**: Python 3.10+（纯 Python 实现，无 DSL/自定义语法）
- **LLM 后端**: OpenAI、Anthropic、Google Vertex、Mistral、Ollama、vLLM 等 15+ 平台
- **数据层**: ChromaDB / Qdrant / FAISS 向量存储 + HuggingFace 数据集托管
- **协议**: 原生 Function Calling / Tool Use 协议适配
- **扩展**: PyPI `camel-ai` 包，可选 `[all]` / `[web_tools]` / `[agents]` 等特性包

### 工作流程（Role-Playing 模式为例）

```
1. 定义角色对 → AssistantRole + UserRole（含系统提示词）
2. ModelFactory.create() 创建模型实例（可多模型调度）
3. ChatAgent(model, tools, memory) 初始化 Agent
4. agent.step(user_input) 执行单步推理：
   ├── 构建 messages 列表（system + history + user）
   ├── 调用 LLM API（含 tool_definitions）
   ├── 解析 response：text / function_call / structured output
   ├── 若有 tool_call → 执行工具 → 将结果回传 LLM 继续推理
   └── 更新 memory（ConversationMemory / ChunkedMemory）
5. 多 Agent 场景：通过 Society 编排 Agent 间消息传递
```

### 关键抽象层次

| 层次 | 类 | 职责 |
|------|-----|------|
| 消息层 | `BaseMessage` / `AgentMessage` | 类型安全的消息封装（system/user/assistant/function） |
| 模型层 | `ChatModel` / `ModelFactory` | 统一的多平台 LLM 调用接口 |
| Agent 层 | `BaseAgent` → `ChatAgent` | step()/reset() 接口，工具绑定，记忆管理 |
| 社会层 | `RolePlaying` / `Workforce` | 多 Agent 编排与通信模式 |
| 工具层 | `FunctionTool` / `Toolkit` | Python 函数 → JSON Schema 自动转换 |

## 技术亮点

### 1. Role-Playing 双 Agent 对话引擎（核心创新）

CAMEL 的灵魂是 **角色扮演对话系统**——让两个 LLM 分别扮演不同角色进行开放式协作：

```python
from camel.societies import RolePlaying
from camel.types import RoleType

# 定义两个角色的系统提示
assistant_sys_msg = """You are a Python programmer. 
You write clean, well-documented code."""

user_sys_msg = """You are a stock trader.
You need a trading bot for algorithmic trading."""

# 创建角色扮演社会
society = RolePlaying(
    assistant_role_name="Python Programmer",
    assistant_agent_kwargs=dict(
        system_message=assistant_sys_msg,
        model="gpt-4o"
    ),
    user_role_name="Stock Trader",
    user_agent_kwargs=dict(
        system_message=user_sys_msg,
        model="gpt-4o"
    ),
    task_prompt="Build a trading bot that uses moving average crossover strategy.",
    with_task_specify=True,  # 让 Agent 先协商明确任务需求
    chat_turn_limit=10,
)

# 运行对话
response = society.run()
```

**为什么这个设计重要？**
- 不是简单的 prompt template，而是让两个 Agent 通过自然语言协商来细化任务（`with_task_specify=True`）
- 产生的对话本身就是高质量的 **合成训练数据** —— CAMEL 的 AI Society 数据集就是用这个方式生成的，已被下载数万次
- 这比单 Agent 自我反思更接近真实的人类协作模式

### 2. ModelFactory 统一模型网关（15+ 平台）

```python
from camel.models import ModelFactory
from camel.types import ModelPlatformType, ModelType

# 方式1: 枚举指定
model = ModelFactory.create(
    model_platform=ModelPlatformType.OPENAI,
    model_type=ModelType.GPT_4O,
    model_config_dict={"temperature": 0.0}
)

# 方式2: 字符串简写（最新优化）
agent = ChatAgent("You are helpful.", model=("anthropic", "claude-haiku-4-5"))

# 方式3: 多模型调度策略
def cost_aware_strategy(models):
    return sorted(models, key=lambda m: m.cost_per_token)[0]

agent.add_model_scheduling_strategy("cheapest", cost_aware_strategy)
```

**设计亮点：**
- 用 `ModelPlatformType` 枚举统一了 15+ LLM 提供商的 API 差异
- 支持运行时动态切换模型（`add_model_scheduling_strategy`），实现成本/质量/速度的自动权衡
- 每个平台的 API key、rate limit、token 计费逻辑全部封装在 `ChatModel` 子类中

### 3. 结构化输出（Pydantic 原生集成）

```python
from pydantic import BaseModel, Field
from typing import List

class TradingSignal(BaseModel):
    action: str = Field(description="BUY or SELL")
    symbol: str = Field(description="Stock ticker")
    confidence: float = Field(description="0.0 to 1.0")
    reasoning: str = Field(description="Why this signal")

agent = ChatAgent(model="gpt-4o-mini")
response = agent.step(
    "Analyze AAPL's current trend.",
    response_format=TradingSignal
)

# 直接拿到解析好的 Pydantic 对象
signal = response.msgs[0].parsed
print(signal.action)       # "BUY"
print(signal.confidence)   # 0.85
print(signal.reasoning)    # "Moving average golden cross..."
```

**关键点：**
- 不是后处理正则提取，而是通过 LLM 的 `response_format` / `tool` 参数原生约束输出
- 支持嵌套对象和 List 字段
- 返回的 `.parsed` 属性直接是类型安全的 Python 对象

### 4. 合成数据生成管线（DataGen）

这是 CAMEL 区别于其他框架的最大特色——它不只是运行 Agent 的框架，更是**用 Agent 生成训练数据的工厂**：

```python
from camel.datagen import CoTDataGenerator
from camel.models import ModelFactory

generator = CoTDataGenerator(
    model=ModelFactory.create(ModelPlatformType.OPENAI, ModelType.GPT_4O),
)

# 自动生成带思维链的数学推理数据
dataset = generator.generate(
    topic="Solve quadratic equations using the formula method",
    num_samples=100,
    language="english"
)
# 输出: [{"question": "...", "reasoning": "...", "answer": "..."}, ...]
```

已生成的公开数据集：
- **AI Society**: 50K+ 角色扮演对话（11 种语言翻译版）
- **Code / Math / Physics / Chemistry / Biology**: 各学科 CoT 数据
- **Misalignment**: 用于安全对齐研究的对抗性数据

### 5. Memory 分层架构

```python
from camel.memories import (
    ContextMemory,        # 滑动窗口上下文
    ChunkedMemory,        # 分块记忆（长文档）
    VectorMemory,         # 向量检索记忆
    CombinedMemory,       # 组合多种记忆策略
)

# 使用组合记忆：短期上下文 + 长期向量检索
memory = CombinedMemory([
    ContextMemory(window_size=10),
    VectorMemory(dimension=1536, store="chromadb"),
])

agent = ChatAgent(memory=memory)
```

**分层设计：**
- `ContextMemory`: 维护最近 N 轮对话（滑动窗口）
- `ChunkedMemory`: 将长文本分块存储，按相关性检索
- `VectorMemory`: 语义向量检索（支持 ChromaDB/Qdrant/FAISS）
- `CombinedMemory`: 可组合多个记忆后端，查询时合并返回

## 对我们（OpenClaw/楠楠）的启发

### 🔥 可直接借鉴的点

**1. Role-Playing 社会编排模式 → 多人格系统增强**

CAMEL 的 `RolePlaying` 设计给了我们一个思路：OpenClaw 的 subagent 机制可以更结构化。目前我们派 subagent 是临时性的，但可以引入「角色注册表」概念——预定义一组专业角色（研究员、程序员、审计员），让主 agent 像 CAMEL 一样编排它们之间的对话流程。这对复杂任务（如进化分析流程）特别有价值。

**2. ModelFactory 多模型网关 → 成本优化**

我们的 TOOLS.md 已经有 MiniMax 主力 + 智谱搜索/视觉的分工。CAMEL 的 `add_model_scheduling_strategy` 启发我们可以做得更智能：根据任务复杂度自动选择模型（简单问候用便宜模型，深度分析用强模型），而不是手动指定。

**3. 合成数据生成能力 → 自我改进闭环**

CAMEL 最独特的价值是用 Agent 对话生成训练数据。我们可以借鉴这个思路：
- 让楠楠和桐桐之间的「高质量对话」被收集起来作为 Few-shot 示例
- 用 `evolve-review.py` 评分 ≥ 85 的回复自动存入优质样本库
- 这些样本可以作为后续会话的 in-context 参考，形成自我改进的正向循环

**4. 结构化输出的 Pydantic 集成 → 工具调用规范化**

CAMEL 把 Pydantic model 直接传给 `step(response_format=...)` 的做法很优雅。我们在定义 skill 的输入输出时可以借鉴这种模式——每个 skill 的参数和返回值都用 Pydantic model 约束，减少解析错误。

**5. Message 类型安全体系 → 通信可靠性**

CAMEL 的 `BaseMessage` 层次结构（SystemMessage / UserMessage / AssistantMessage / FunctionMessage）确保了消息在多 Agent 传递时不会丢失元信息。OpenClaw 在 subagent 通信中也可以引入类似的消息类型标注。

### ⚠️ 不适合直接照搬的

- CAMEL 的百万级 Agent 模拟是面向学术研究的，生产环境不需要
- 它的 DataGen 管线依赖 GPT-4 级别模型大量调用来生成数据，成本较高
- 文档虽全但有些模块成熟度不均（如 EmbodiedAgent 相对实验性）
