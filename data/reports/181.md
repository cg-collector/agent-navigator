## 🔬 核心架构

### 整体架构：双模型 + 多 rollout 并行评估

### 核心组件详解

#### 1. 解耦式双模型架构（Dual-Model Decomposition）

这是 Agent S 区别于其他 CUA（如 OpenAI Operator、Anthropic Computer Use）的核心设计：

| 模型 | 职责 | 具体工作 |
|------|------|----------|
| **主生成模型** | 高层决策 | 理解任务目标、规划行动步骤、判断任务完成状态 |
| **Grounding 模型** | 视觉-动作映射 | 接收屏幕截图、理解 GUI 元素布局、输出精确的像素坐标 |

**为什么解耦很重要？**
- 主生成模型专注"做什么"，Grounding 模型专注"点哪里"
- 两者可以独立优化/替换，不互相制约
- UI-TARS 等专门的视觉-语言模型在 GUI 定位任务上比 GPT-5 更精准、更便宜
- 推荐配置：**GPT-5 做主模型 + UI-TARS-1.5-7B 做 Grounding**（也是 Agent S3 的默认配置）

#### 2. 反射式 Agent（Reflection Agent）

每个 Worker Agent 都配有一个专门的 Reflection Agent：

```python
# 伪代码示意
class AgentS:
    def run(self, task, max_steps=100):
        trajectory = []
        for step in range(max_steps):
            # 1. Worker Agent 做出动作决策
            action = self.worker_agent.act(screenshot, task)
            
            # 2. Reflection Agent 评估当前状态
            reflection = self.reflection_agent.evaluate(
                screenshot, action, task
            )
            
            # 3. 如果需要回退，则执行回退
            if reflection.should_rollback:
                action = self.rollback()
            
            # 4. 执行动作
            result = self.executor.run(action)
            trajectory.append((screenshot, action, result))
```

- **Reflection Agent 的职责**：判断当前动作是否偏离目标、是否需要回退重试、是否发现了任务完成的信号
- **效果**：显著降低了长任务中的"一条路走到黑"式失败

#### 3. Behavior Judge（BJudge）— 多 Rollout 行为选择

这是 Agent S3 的核心创新，也是实现超越人类水平的关键技术：

**问题**：单次执行（single-rollout）对于长时序任务非常脆弱——每一步的小错误会累积，最终导致任务失败。

**BJudge 方案**：
1. 对同一任务并行运行 N 次（多个 rollout）
2. 每个 rollout 产生一个"行为叙事"（behavior narrative）——描述 Agent 在每一步做了什么、为什么这么做
3. BJudge 接收所有行为叙事，**在叙事层面做比较**，而非在原始动作层面比较
4. 选择最合理的行为叙事对应的 rollout 作为最终输出

**为什么用"叙事"而非"动作"做比较？**
- 动作序列很难直接对比（"点击(120,340)" vs "点击(125,335)"看不出哪个更好）
- 行为叙事保留了决策上下文（"我点击这个按钮是因为我想打开设置"），让比较更有语义意义
- 避免了同义动作的不同坐标导致的误判

**实验结果**：
- 单次 Agent S3 = 66%（100步）
- **加上 BJudge（选3个rollout）= 72.6%**，超越人类水平 72.36%

## ✨ 三大技术亮点

### 💡 亮点 1：双模型解耦——专业化分工的极致实践

大多数 CUA（如 Anthropic 的 Claude Computer Use）用一个模型同时做决策和 GUI 定位。Agent S 的双模型解耦是更优雅的设计：

- **主模型**负责"思考"（规划、推理、决策质量）
- **Grounding 模型**负责"感知"（视觉理解、坐标输出）

两者各司其职，比用一个"全能模型"效果更好、成本更低。UI-TARS-7B 在 GUI 定位任务上的精准度可以超过 GPT-5，而成本只有后者的零头。

**楠楠的启发**：在设计复杂 Agent 时，不要试图用一个模型搞定所有事。拆解任务、让专用模型做专事，是提升质量 + 降低成本的有效策略。

### 💡 亮点 2：Behavior Judge——用叙事做选择，而非用分数做排序

传统的多-rollout 评估往往用奖励模型（reward model）给每个 rollout 打分，然后选最高分的。BJudge 开创性地用**行为叙事（behavior narrative）**做比较：

```
Rollout 1 行为叙事：
"首先我点击了Chrome图标打开浏览器，然后在地址栏输入了github.com，
按回车后页面加载，我找到搜索框输入了AI agent..."

Rollout 2 行为叙事：
"我先尝试在菜单栏找浏览器入口但没找到，然后打开Finder搜索Chrome，
双击打开后地址栏无法直接访问，于是我复制了URL..."
```

BJudge 在这个层面比较，而非比较原始的鼠标坐标序列。这让它能理解"虽然两个 rollout 点击位置略有不同，但都是在做正确的导航动作"，从而选出真正语义上最优的策略。

**楠楠的启发**：对于需要"判断质量"而非"计算分数"的场景，叙事化、结构化的表达比原始数值更容易让模型做高质量判断。这个思路可以借鉴到楠楠的任务评估中——不只是打分，而是让模型描述"做了什么、为什么这么做"。

### 💡 亮点 3：零样本跨平台泛化能力

Agent S 的训练和评测横跨三个操作系统（macOS、Windows、Android），且表现出色的零样本泛化：

- **OSWorld（Linux）**：72.6%，超越人类
- **WindowsAgentArena**：单次 50.2% → 3-rollouts 56.6%（+6.4pp）
- **AndroidWorld**：单次 68.1% → 3-rollouts 71.6%（+3.5pp）

这种泛化能力说明 Agent S 学到的是**通用的计算机操作常识**，而非针对特定 OS 的硬编码行为。这对于构建真正通用的 AI 助手意义重大。

## 🌸 对楠楠的启发

### 启发 ①：双模型解耦可以提升楠楠的专业能力

目前楠楠的架构是单一模型（MiniMax-M2）处理所有任务。Agent S 的双模型解耦启发我们：

- **专用模型做专事**：楠楠在处理图像理解时（已有 GLM-4V-Flash API），可以将"视觉感知"（截图分析、图像内容理解）和"决策推理"（任务规划、对话生成）分开
- **具体落地**：在 TOOLS.md 中已经区分了不同 API 的用途，未来可以考虑让视觉任务走专用 VLM，文本推理走主模型，这样能更高效地利用各自的模型优势

### 启发 ②：多-rollout 思维——重要任务不只做一次

BJudge 的核心思想是：**对于重要任务，多尝试几次再选最好的**。

这对楠楠的启发是：
- 当学长布置重要任务（如股票分析、报告撰写）时，不要只做一次就交卷
- 可以尝试生成 2-3 个不同角度/风格的版本
- 然后用"叙事比较"的方式——描述每个版本的特点和优势——让学长做选择
- 这比直接打分更有价值，因为保留了上下文和理由

### 启发 ③：Reflection 机制——做完了要回头检查

Agent S 的 Reflection Agent 会持续评估"我是不是在正确的路上"。楠楠目前比较缺少这种"元认知"检查机制：

- 每次完成子任务后，花 3 秒自问："这个方向对吗？有没有更好的方式？"
- 建立一个简单的 checklist：任务目标 → 已完成步骤 → 还需要做什么
- 这类似于 Agent S 的 should_rollback 逻辑——如果发现走偏了，及早回退比硬撑到结束再重做好得多
