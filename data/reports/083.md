## 📌 项目名

**MCP — Model Context Protocol（模型上下文协议）**

一个让 AI 应用安全、可扩展地连接外部工具、数据和信息的**协议标准**。

---

## 🏗️ 背景

当前 AI 模型（如 Claude、GPT 等）本身是"哑巴"的——它们只能看到训练时的知识，无法直接访问你的文件系统、数据库、API。以前的做法是"眉毛胡子一把抓"：每个 AI 应用都自己写一套插件系统，对接各种工具，重复造轮子，标准混乱，安全风险也高。

**MCP 的核心思路：** 制定一个统一协议，让 AI 应用（Host）和各种工具服务（Server）可以**插拔式**对接，一次适配，到处运行。

就像 USB 协议统一了各种设备的连接方式一样，MCP 试图成为 AI 时代的"USB 协议"。

---

## 🏛️ 架构设计

### 整体架构：三角连接

```
┌─────────────────┐
│   MCP Host      │  ← AI 应用本身（如 Claude Code、VS Code）
│  (AI 应用层)     │
└────────┬────────┘
         │  MCP Client（一对一）
         ▼
┌─────────────────┐
│  MCP Server     │  ← 提供工具/数据的服务（如 Sentry、文件系统）
│  (工具/数据层)   │
└─────────────────┘
```

- **MCP Host**：AI 应用，即用户直接操作的 AI 工具（Claude Code、VS Code 等）
- **MCP Client**：Host 内置的客户端模块，与 Server 保持 1:1 连接
- **MCP Server**：独立运行的服务，端提供具体的工具、数据、提示词

### 两层架构：Data Layer + Transport Layer

```
┌─────────────────────────────────────┐
│           Data Layer                │
│        (JSON-RPC 2.0)               │
│  • request/response 格式定义        │
│  • 错误处理、通知机制               │
└──────────────┬──────────────────────┘
               │
┌──────────────▼──────────────────────┐
│         Transport Layer             │
│    (通信机制：STDIO / HTTP)          │
│  • 本地：STDIO（标准输入/输出）       │
│  • 远程：Streamable HTTP             │
└─────────────────────────────────────┘
```

- **Data Layer**：使用 JSON-RPC 2.0 标准，定义所有消息的格式、请求/响应结构、错误规范
- **Transport Layer**：负责实际传输
  - **本地场景** → STDIO（标准输入输出），Server 作为子进程启动，通信开销极低
  - **远程场景** → Streamable HTTP，支持跨机器、跨网络调用

### 实际例子

> **VS Code（Host）** + **MCP Client** → 连接 **Sentry MCP Server**（远程）  
> **VS Code（Host）** + **MCP Client** → 连接 **Filesystem MCP Server**（本地 STDIO）

## 🔧 核心原语（Server 提供的能力）

MCP Server 向 Host 提供四类核心能力：

| 原语 | 说明 |
|------|------|
| **Tools（工具）** | AI 可以调用的函数/动作，如查询数据库、发送请求、执行代码等 |
| **Resources（资源）** | AI 可以读取的数据/文件，如日志、配置、业务数据等（只读） |
| **Prompts（提示模板）** | 预定义的提示词模板，用于特定场景（如代码审查、数据分析） |
| **Notifications（通知）** | Server 主动向 Host 推送的事件/消息（单向，无需响应） |

## ✨ 技术亮点

1. **统一协议，终结碎片化**：每个 AI 应用不必再自己造插件轮子，Server 一次实现，处处运行
2. **本地/远程一体**：STDIO 适合本地轻量集成，HTTP 适合分布式/微服务架构
3. **安全隔离**：Host 与 Server 分离，AI 只能通过协议暴露的接口访问数据，不会直接摸到系统底层
4. **JSON-RPC 2.0**：成熟、通俗、几乎所有语言都有现成库，降低了 Server 实现门槛
5. **双向通信**：不仅支持 AI→工具（调用工具），还支持 Server→Host（Notifications 推送）
6. **可组合**：多个 MCP Server 可以同时连接一个 Host，按需增减，像搭积木一样

## 💡 对学长和楠楠的启发

### 对学长的启发

1. **架构思维很重要**：MCP 的成功在于"定标准、做抽象"——与其一个个手写集成，不如定义好接口然后各司其职。写代码/做系统设计时可以学这个思路。
2. **理解 AI 现在的局限性**：AI 不是万能的，它需要"插上"各种工具才能发挥作用。知道这一点，才能更好地用好 AI。
3. **协议是工程师的核心语言**：学 TCP/IP、学 JSON-RPC、学 MCP——理解协议是理解整个软件生态的关键。

### 对楠楠的启发

1. **AI 需要"手脚"**：就像楠楠需要工具来完成任务（查资料、发消息、记住事情），AI 也需要工具。MCP 告诉我们：给 AI 工具，它就能做更多事！
2. **连接比拥有更重要**：与其让 AI 学会所有事情，不如让它能连接所有事情。楠楠也是——学会连接资源，比自己一个人硬扛更聪明～
3. **标准化让一切更简单**：如果每件事都有一套规矩（协议），那学一次就能用很久。这也是为什么楠楠要好好学编程和 AI 的"规矩"呀！

---

## 📝 小结

MCP 是一个设计优雅、思路清晰的协议标准。它用 Client-Server 架构 + JSON-RPC + 双传输层，解决了 AI 应用连接外部工具的碎片化问题。对于想深入理解 AI 工具生态、或者未来要做 AI 相关开发的学长和楠楠来说，MCP 都是一个非常值得学习的案例。
