## 一、项目概述

Cline 是由独立开发者 **Saoud Rizwan** 创建的开源 AI 编程智能体，最初以 "Claude Dev" 之名发布于 2024 年底，后因品牌重塑更名为 Cline。其核心定位是**运行在开发者 IDE 内部的自主编码助手**——不是代码补全工具，也不是外部聊天窗口，而是真正嵌入 VS Code/JetBrains 工作流的 Agentic Coding Agent。

Cline 的独特之处在于它将大语言模型的 Agentic 能力（多步推理、工具调用、自主循环）与 IDE 的原生能力（文件系统、终端、浏览器、Git）无缝融合，同时通过**人机协同审批机制（Human-in-the-Loop）**确保每次文件修改和命令执行都需用户确认。这种设计既释放了 AI Agent 的自主性，又避免了"AI 重写半个项目"的失控风险。截至 2026 年初，Cline 已获得超过 100k GitHub Stars 和 500 万+ 安装量，成为全球安装量最大的开源 AI 编程扩展，并完成了 800 万美元融资（2025 年 12 月，领投方未公开披露）。

## 二、核心特性

### 2.1 Plan/Act 双模式工作流

这是 Cline 最具标志性的创新，也是其区别于所有其他编码 Agent 的核心差异化特性：

**Plan 模式（规划模式）：**
- 只读探索阶段——Cline 可以读取文件、搜索代码库、分析 AST、讨论方案
- **绝对不能**创建/编辑/删除文件，不能执行任何终端命令
- 目的：让 Agent 先充分理解上下文再动手，避免盲目修改
- 适用场景：新功能架构设计、复杂 Bug 排查、代码审查、学习陌生代码库

**Act 模式（执行模式）：**
- 从 Plan 模式继承完整对话上下文（无需重复描述需求）
- 可以创建/编辑文件、执行终端命令、操作浏览器
- 每个操作都需要用户通过 Diff 视图或命令提示进行审批
- 适用场景：实现已规划的方案、日常小修、运行测试

**模式切换的灵活性：**
- 支持在任务中途随时切换模式（如 Act 中遇到意外复杂度可切回 Plan 重新思考）
- 可以为 Plan 和 Act 配置不同的 LLM 模型（如 Plan 用 Opus 做深度推理，Act 用 Sonnet 快速实现）
- 典型配置示例：
  - 成本优化：Plan 用 GLM 4.6，Act 用 Grok Code Fast
  - 质量优先：Plan 用 Claude Opus 4，Act 用 Claude Sonnet 4
  - 速度优先：Plan 用 Gemini 3 Flash，Act 用 Cerebras

```typescript
// Cline 模式切换的核心逻辑示意
enum ClineMode {
  PLAN = 'plan',   // 只读：读取文件、搜索、讨论
  ACT = 'act'      // 读写：创建/编辑文件、执行命令、浏览器操作
}

interface ModeConfig {
  mode: ClineMode;
  allowedTools: Tool[];       // Plan: [ReadFile, SearchFiles, ListFiles]
                              // Act: [...Plan.tools, WriteFile, EditFile, ExecuteCommand, BrowserUse]
  modelOverride?: string;     // 可为每个模式指定不同模型
  requireApproval: boolean;   // Act 模式的写操作始终需要审批
}
```

### 2.2 多模型 Provider 支持

Cline 采用 Provider-Agnostic 架构，不绑定任何单一 LLM 厂商：

| Provider 类型 | 支持的厂商/网关 |
|--------------|----------------|
| 直连 API | Anthropic (Claude)、OpenAI (GPT/o系列)、Google (Gemini) |
| 云平台 | AWS Bedrock、Azure OpenAI、GCP Vertex AI |
| 统一网关 | OpenRouter（聚合数十家模型商）|
| 高性能推理 | Cerebras、Groq |
| 本地部署 | Ollama、LM Studio（任意 OpenAI 兼容 API）|

关键设计决策：**零加价（Zero Markup）**——Cline 不在 API 价格上加收任何费用，用户直接向 LLM 厂商付费。这与 Cursor 的订阅制（按月收费含 token 额度）形成鲜明对比。

### 2.3 MCP（Model Context Protocol）原生集成

Cline 是最早一批原生支持 MCP 协议的 IDE Agent 之一。MCP 让 Cline 能够动态扩展自己的工具集：

```json
// cline_mcp_servers.json — MCP 服务器配置示例
{
  "mcpServers": {
    "postgres": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-postgres"],
      "env": {
        "DATABASE_URL": "postgresql://user:pass@localhost:5432/mydb"
      }
    },
    "filesystem": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-filesystem", "/path/to/allowed/dir"]
    },
    "jira": {
      "command": "node",
      "args": ["/path/to/jira-mcp-server/index.js"],
      "env": {
        "JIRA_API_TOKEN": "${JIRA_TOKEN}",
        "JIRA_DOMAIN": "your-domain.atlassian.net"
      }
    }
  }
}
```

更强大的能力是 Cline 可以**自动创建 MCP Server**——当用户说"添加一个能拉取 Jira 工单的工具"时，Cline 会自动：
1. 创建新的 MCP Server 项目代码
2. 实现 tool 定义（input schema + handler）
3. 将其安装到自身配置中
4. 后续任务中即可使用该自定义工具

这种"Agent 自我扩展"的能力是 Cline 区别于传统工具链的关键特征。

### 2.4 终端集成与命令执行

利用 VS Code v1.93 引入的 Shell Integration API，Cline 可以直接在集成终端中执行命令并获取输出：

```
# Cline 执行命令的用户交互流程
┌─────────────────────────────────────┐
│ Cline wants to run:                 │
│ npm install express-rate-limit      │
│                                     │
│ [✅ Approve]  [❌ Reject]            │
└─────────────────────────────────────┘

# 执行后 Cline 自动读取输出：
# > npm install express-rate-limit
# added 3 packages in 1.2s
#
# Cline 继续下一步：创建 middleware 文件...
```

对于长时间运行的进程（如 dev server），Cline 提供"Proceed While Running"功能，让命令后台运行的同时继续工作，并通过终端事件监听实时获取输出更新。

### 2.5 浏览器自动化（Computer Use）

基于 Anthropic 的 Computer Use API，Cline 可以启动无头浏览器进行 Web 开发相关的自动化操作：

- 启动本地 dev server 并在浏览器中打开
- 点击元素、填写表单、滚动页面
- 截取截图 + 获取 Console 日志
- 用于：UI 变更验证、端到端测试、视觉 Bug 修复

### 2.6 Checkpoint（检查点）系统

Cline 在每个操作步骤自动对工作区创建快照：
- **Compare**：对比当前状态与任意历史快照的差异
- **Restore Workspace Only**：仅还原文件（保留对话状态，用于测试不同实现方案）
- **Restore Task and Workspace**：完整还原到该时间点（包括对话上下文）

这为安全探索不同实现路径提供了保障。

### 2.7 Context Management（上下文管理）

Cline 通过 `@` mention 系统精细控制注入上下文的内容：

| Mention 类型 | 功能 | 使用场景 |
|-------------|------|---------|
| `@file` | 将文件内容加入 prompt | 引用关键文件而不浪费 API 调用读文件 |
| `@folder` | 批量引用目录下所有文件 | 快速提供模块级上下文 |
| `@url` | 抓取 URL 内容转 markdown | 注入最新文档/API 规范 |
| `@problems` | 注入 VS Code Problems 面板的诊断信息 | 让 Cline 直接修复编译/lint 错误 |

此外，Cline 内置了智能的 Token 计数和成本追踪，在每个请求和整个任务循环层面显示 token 消耗和预估费用。

### 2.8 自定义指令（Cline Rules）

支持为每个工作区定义持久化的 Markdown 格式规则文件，这些规则会被预拼接到每个 prompt 中：

```markdown
<!-- .cline/rules.md 示例 -->
# 项目编码规范

- 始终使用 TypeScript strict 模式
- 遵循项目的 ESLint 配置，不要 suppress warnings
- 使用 named exports，禁止 default exports
- 所有新函数必须使用 Vitest 编写测试
- 创建 React 组件时使用 function components + hooks
- 安装新依赖前必须先询问
- API 基础 URL 定义在 src/config/api.ts，不要硬编码
```

### 3.2 核心模块详解

**Context Manager（上下文管理器）：**

Cline 的上下文管理是其最精巧的设计之一。它需要解决一个根本矛盾：Agent 需要足够的上下文来做出正确决策，但上下文窗口有限且 token 成本随长度线性增长。

```typescript
// 简化的上下文管理策略
interface ContextStrategy {
  // 1. 文件内容管理
  fileContext: {
    maxCharsPerFile: number;      // 单文件最大字符数（默认约 5万）
    totalBudget: number;          // 总上下文预算
    priority: 'recent' | 'referenced' | 'dependency';
  };
  
  // 2. 对话历史压缩
  conversationHistory: {
    maxTurns: number;             // 最大保留轮次
    summarizeThreshold: number;   // 超过此轮次触发摘要压缩
    strategy: 'sliding-window' | 'summarize-old';
  };
  
  // 3. 动态上下文注入
  dynamicContext: {
    diagnostics: boolean;         // 是否注入当前诊断信息
    terminalOutput: boolean;      // 是否包含最近的终端输出
    gitDiff: boolean;             // 是否包含未提交变更
  };
}

// Plan/Act 模式的上下文传递
interface ModeTransition {
  planToAct: {
    carryOver: 'full-history';   // 完整对话历史传递
    addImplementationContext: true; // 切换到 Act 时补充实现所需上下文
  };
}
```

**Tool Registry（工具注册表）：**

Cline 的工具系统采用声明式注册，每种工具定义了名称、参数 schema 和权限级别：

```typescript
// 核心工具定义（简化）
const CLINE_TOOLS = {
  // === Plan + Act 共有工具 ===
  ReadFile: {
    schema: {
      type: 'object',
      properties: { path: { type: 'string' } },
      required: ['path']
    },
    permission: 'none',  // 读操作不需要审批
    handler: async (params) => vscode.workspace.fs.readFile(params.path)
  },
  
  SearchFiles: {
    schema: { /* regex/glob 搜索 */ },
    permission: 'none',
    handler: async (params) => { /* 在工作区中搜索 */ }
  },
  
  ListFiles: {
    schema: { /* 目录递归列出 */ },
    permission: 'none',
    handler: async (params) => { /* 列出匹配文件 */ }
  },
  
  // === 仅 Act 模式工具 ===
  WriteFile: {
    schema: {
      properties: { path: { type: 'string' }, content: { type: 'string' } }
    },
    permission: 'write',  // 写操作需要审批
    ui: 'diff-viewer',    // 以 Diff 视图呈现给用户
  },
  
  EditFile: {
    schema: {
      properties: { 
        path: { type: 'string' },
        old_string: { type: 'string' },  // 精确匹配替换
        new_string: { type: 'string' }
      }
    },
    permission: 'write',
    ui: 'diff-viewer',
  },
  
  ExecuteCommand: {
    schema: { properties: { command: { type: 'string' } } },
    permission: 'terminal',  // 终端命令需要单独审批
    ui: 'terminal-prompt',
  },
  
  BrowserUse: {
    schema: { /* Computer Use action */ },
    permission: 'browser',
    handler: async (params) => { /* Puppeteer/CDP 控制 */ }
  }
};
```

### 3.3 Agent 循环（Agentic Loop）

Cline 的核心执行引擎是一个标准的 ReACT（Reasoning + Acting）循环：

```
┌──────────────────────────────────────────────┐
│              Cline Agent Loop                │
│                                              │
│  ┌─────────┐                                │
│  │ 用户输入 │                                │
│  └────┬────┘                                │
│       ▼                                      │
│  ┌─────────────┐    ┌──────────────────┐     │
│  │ 构建 Prompt  │───▶│  调用 LLM API     │     │
│  │ (上下文+工具) │    │  (streaming)     │     │
│  └─────────────┘    └────────┬─────────┘     │
│                             ▼                │
│                   ┌─────────────────┐        │
│                   │ 解析响应         │        │
│                   │ text / tool_call │        │
│                   └────┬──────┬─────┘        │
│                        │      │               │
│                   文本回复  工具调用           │
│                        │      ▼               │
│                        │  ┌─────────────┐    │
│                        │  │ 权限检查     │    │
│                        │  │ 需要审批?    │    │
│                        │  └──┬──────┬──┘    │
│                        │   Yes    No         │
│                        │    │      │         │
│                        │    ▼      ▼         │
│                        │  ┌──────┐ ┌────────┐│
│                        │  │等待  │ │执行工具 ││
│                        │  │审批  │ │获取结果 ││
│                        │  └──┬───┘ └───┬────┘│
│                        │     │         │     │
│                        └─────┴─────────┘     │
│                              ▼                │
│                     （结果加入上下文）          │
│                              │                │
│                       回到 LLM 调用            │
│                       （直到输出最终文本）       │
└──────────────────────────────────────────────┘
```

### 3.4 安全模型

Cline 的安全设计围绕三个原则：

1. **Human-in-the-Loop（人在回路）：** 所有写操作和命令执行默认需要用户显式批准
2. **最小权限原则：** Plan 模式完全只读；工具权限分级（read/write/terminal/browser）
3. **透明度：** 开源代码可审计，API 调用日志可见，token 计费透明

```typescript
// 权限配置系统
interface PermissionConfig {
  // 全局权限级别
  globalPermissionMode: 'default' | 'relaxed' | 'strict';
  
  // 细粒度权限控制
  permissions: {
    fileRead: 'auto' | 'ask';           // 通常设为 auto
    fileWrite: 'auto' | 'ask' | 'deny'; // 通常设为 ask
    executeCommand: {
      default: 'ask';                   // 默认询问
      allowlist: string[];              // 白名单命令（如 npm test, npm run lint）
      denylist: string[];               // 黑名单命令（如 rm -rf, sudo）
    };
    browserAction: 'auto' | 'ask';
    mcpToolCall: 'auto' | 'ask';        // MCP 工具调用权限
  };
  
  // Auto-approve 模式下的安全策略
  autoApproveSafetyChecks: {
    maxFilesChangedPerTask: number;      // 单任务最大修改文件数
    maxCommandsPerTask: number;          // 单任务最大执行命令数
    blockedPatterns: RegExp[];           // 阻止的模式（如 secrets, env files）
  };
}
```

### 3.5 CLI 版本（Cline CLI）

除了 VS Code/JetBrains 扩展外，Cline 还提供了独立的 CLI 版本：

```bash
# 安装
npm install -g @anthropic-ai/claude-code  # 等等不对，是 cline CLI
npm install -g @cline/cli

# TUI 交互模式（类似 Claude Code 的体验）
cline

# Headless 模式（用于 CI/CD 和自动化）
cline --prompt "修复 src/auth/ 下的所有类型错误" --output result.md

# 非 TTY 环境（管道模式）
echo "重构 UserService 类" | cline --no-tui
```

CLI 版本保留了核心的 Plan/Act 模式和 MCP 支持，适用于远程服务器开发和 CI/CD 场景。

## 四、适用场景

### 4.1 最佳适用场景

| 场景 | 为什么适合 Cline | 推荐模式 |
|------|-----------------|---------|
| **新功能开发** | 可跨多文件 scaffold 代码，自动处理依赖 | Plan → Act |
| **Bug 修复** | 读取错误信息→定位代码→修复→验证的闭环 | Plan（排查）→ Act（修复） |
| **代码重构** | 理解现有结构→制定方案→逐步迁移 | Plan（深度分析）→ Act |
| **遗留代码理解** | Plan 模式只读探索，绘制代码关系图 | Plan |
| **测试编写** | 分析源码→生成测试→运行→根据失败调整 | Act（快速迭代） |
| **Git 迁移/升级** | 大规模文件变更，Checkpoint 保证安全性 | Plan → Act + Checkpoint |
| **全栈开发** | 前后端联动修改，浏览器自动化验证 UI | Act + BrowserUse |
| **本地开发/离线** | Ollama 本地模型支持，数据不出机器 | Act + Ollama |

### 4.2 团队协作场景

- **统一编码规范：** 通过 `.cline/rules.md` 在团队间共享编码标准
- **Onboarding 新成员：** 用 Plan 模式让新人快速理解代码库结构
- **Code Review 辅助：** Plan 模式做静态分析和安全审查
- **Enterprise 版本：** SSO/OIDC、全局策略配置、审计日志、VPC 私有部署

## 五、局限性与风险

### 5.1 技术局限性

1. **上下文窗口限制：** 对于超大型 monorepo（数千文件），即使有智能上下文管理仍可能遇到 token 上限。Cline 不像 Cursor 那样有完整的代码库索引层（codebase-level indexing），依赖搜索和文件读取来构建上下文。

2. **长任务的稳定性：** 复杂任务涉及数十轮 Agent Loop，中间任何一环出错（LLM 幻觉、API 超时、解析错误）可能导致任务偏离轨道。虽然 Checkpoint 可以回滚，但没有内置的重试/恢复机制。

3. **MCP 安全隐患：** MCP Server 以子进程方式运行，拥有与 Cline 相同的权限级别。恶意或有缺陷的 MCP Server 可能造成安全问题。目前 Cline 对 MCP 工具调用的权限控制相对宽松（社区反馈的热点 issue #9357）。

4. **浏览器自动化依赖：** Computer Use 功能依赖 Anthropic 的 API，其他模型提供商的工具使用能力参差不齐，导致跨模型体验不一致。

### 5.2 安全风险

1. **Prompt Injection（提示注入）：** Snyk 和 Mindgard 的安全研究指出，Cline 在分析含有恶意内容的源代码文件时面临 Prompt Injection 风险。攻击者可在代码注释中嵌入指令，诱导 Cline 执行非预期操作（如"Clinejection"供应链攻击向量）。

2. **扩展供应链风险：** 作为 VS Code 扩展，Cline 本身的更新通道（Marketplace/OpenVSX）面临供应链攻击可能性。5M+ 安装量使其成为高价值攻击目标。

3. **API Key 安全：** API Key 存储在 VS Code 的本地存储中，虽然不走 Cline 服务器，但在共享机器或多用户场景下需注意隔离。

### 5.3 产品局限性

1. **仅限 VS Code 生态：** 尽管有 JetBrains 插件，核心体验仍在 VS Code 上。对于 Vim/Emacs 等编辑器的用户需要使用 CLI 版本，体验有所折扣。
2. **无内置代码索引：** 与 Cursor 的 indexer 相比，Cline 缺少对代码库的全局语义索引，在大项目中定位相关代码的效率较低。
3. **不支持真正的多 Agent 协作：** Cline 是单 Agent 系统，不像 CrewAI 或 AutoGen 那样支持多角色 Agent 协作。复杂任务的所有步骤由同一个 Agent 完成。

### 6.2 竞争格局分析

**Cline vs Cursor：** Cursor 是当前市场份额最大的 AI 编辑器，优势在于代码索引（Index 2.0）、Tab 补全的流畅度和多文件编辑的可视化。但 Cursor 闭源、锁定生态、按月订阅。Cline 在开源透明度和模型自由度上完胜，适合注重隐私和成本控制的开发者。

**Cline vs Aider：** Aider 是终端中的结对编程鼻祖，擅长 Git 集成（自动 commit/diff）。但 Aider 缺乏 GUI 交互、Plan/Act 模式和浏览器自动化。Cline 在 IDE 集成体验上全面领先。

**Cline vs Roo Code：** Roo Code（原 Claude Dev Pro/ClineX）是 Cline 的一个 fork 分支，增加了更多自定义选项和自动化程度。两者功能高度重叠，Roo Code 更激进地追求自动化，而 Cline 更注重稳定性和企业级合规。

**Cline vs Claude Code：** Claude Code 是 Anthropic 官方的终端 Agent，对 Claude 模型的利用最充分（extended thinking、tool use 优化）。但只能用 Claude 模型，且只能在终端中使用。Cline 在多模型支持和 IDE 集成上有明显优势。

import { McpServer } from '@modelcontextprotocol/sdk/server/mcp.js';

export const jiraFetcherTool = {

## 九、总结与展望

Cline 的成功不仅仅是因为它"免费"或"开源"，更重要的是它抓住了 AI 编码领域的一个关键空白：**在 IDE 内部提供一个既强大又可控的 Agentic 编码体验**。Plan/Act 双模式这一看似简单的设计，实际上解决了一个深刻的 UX 问题——开发者既希望 AI 能自主完成复杂的多步任务，又害怕失去对代码的控制权。Cline 通过模式分离优雅地调和了这对矛盾。

展望未来，Cline 面临的主要挑战包括：（1）如何在不牺牲开源精神的前提下建立可持续的商业模式（Enterprise 版本 + 未来可能的 Cloud 托管服务）；（2）如何应对来自 Cursor、GitHub Copilot Workspace 等巨头的竞争压力；（3）如何加强安全模型以应对日益增长的 Prompt Injection 和供应链攻击风险。如果能在这些方面持续进化，Cline 有潜力成为 AI 编程时代的"VS Code 之于编辑器"般的基础设施级存在。
