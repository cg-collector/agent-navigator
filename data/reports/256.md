# 📚 AI Agent 项目研究报告 #256 — Gorilla：让大模型学会调用API的工具使用先驱

## 一、项目背景与问题定义

Gorilla 是 UC Berkeley 于 2023 年 5 月发布的开创性项目，由 Shishir G. Patil 团队主导。它解决的核心问题是：**如何让 LLM 准确地调用海量 API（工具）来完成复杂任务？**

### 1.1 问题本质

在 Agent 领域，"工具调用"（Tool Use / Function Calling）是连接 LLM 与真实世界的关键桥梁。然而 2023 年初的 LLM 在这方面表现堪忧：

- **参数幻觉（Parameter Hallucination）**：模型会编造不存在的 API 参数名和值。比如调用 `torch.nn.Conv2d` 时，GPT-4 可能编造 `kernel_shape` 这个根本不存在的参数
- **语义错误（Semantic Error）**：选对了 API 但参数含义理解错误
- **文档滞后性**：API 文档频繁更新，而模型的训练数据是静态的，导致过时知识
- **规模瓶颈**：现实世界中存在数以万计的 API，无法全部塞进 prompt 或训练数据

Gorilla 的论文发表于 NeurIPS 2024，截至目前已被引用超过 **1090 次**，是该领域最具影响力的工作之一。

## 二、核心架构深度剖析

### 2.1 整体架构：Retriever + LLM 的 RAG 式工具调用

Gorilla 的架构可以用一句话概括：**基于检索增强生成（RAG）范式的 API 调用系统**。

```
用户自然语言查询
       │
       ▼
┌──────────────┐     ┌─────────────────────┐
│ 文档检索器     │────▶│  Gorilla (LLaMA微调)   │
│ (Retriever)  │     │                     │
│              │     │  输入: 查询+检索到的    │
│ 从API文档库中  │     │        API文档片段      │
│ 检索相关文档   │     │                     │
└──────────────┘     │  输出: 结构化API调用    │
                      └─────────────────────┘
                               │
                               ▼
                        可执行的API调用
```

与传统 prompt engineering 不同，Gorilla 不是把所有 API 文档塞进上下文窗口，而是：
1. 先通过 retriever 从文档库中找到最相关的 API 文档片段
2. 将查询 + 检索结果一起喂给微调后的 LLaMA 模型
3. 模型输出结构化的 API 调用代码

### 2.2 基础模型与训练策略

- **基座模型**：LLaMA-7B（初始版本），后续扩展到更大规模
- **训练方式**：在自建的 APIBench 数据集上进行监督微调（SFT）
- **关键创新**：引入 **Retrieval Aware Training（RAT）**——训练时模拟检索过程，随机抽取不同数量的相关/无关文档作为上下文，让模型学会从噪声中提取有效信息

### 2.3 APIBench 数据集

这是 Gorilla 项目的重要基础设施贡献：

| 来源 | API 数量 | 样本数 | 特点 |
|------|---------|--------|------|
| HuggingFace Transformers | ~925 | ~4,700 | ML 模型推理 API |
| TorchHub | ~476 | ~2,900 | PyTorch 预训练模型 |
| TensorHub | ~198 | ~1,200 | TensorFlow Hub 模型 |
| **合计** | **~1,600+** | **~11,000+** | 覆盖三大主流 ML 框架 |

每个样本格式为：(自然语言查询, 对应的正确 API 调用代码, 相关 API 文档)

## 三、技术亮点深度解析

### 亮点一：Retrieval Aware Training（RAT）— 解决文档动态更新难题

**原理说明：**

传统方案将 API 文档硬编码到 prompt 中或直接注入训练数据，一旦 API 更新就完全失效。Gorilla 的 RAT 策略的核心洞察是：**让模型学会"阅读文档再调用"，而不是"记住 API 再调用"。**

具体做法：
- 训练时，对每个样本随机抽取 k 个相关文档（k ∈ {0, 1, 5}）
- 同时混入一定比例的**干扰文档**（irrelevant documents）
- 模型必须学会：从混合文档中识别正确的 API → 提取参数 → 生成调用

这带来两个关键优势：
1. **泛化能力**：面对新 API 时，只要提供文档就能正确调用（无需重新训练）
2. **抗幻觉**：模型被训练为依赖提供的文档而非内部记忆，大幅减少编造参数的情况

实验数据显示，Gorilla 的幻觉率相比 GPT-4 降低了约 **70%**。

### 亮点二：AST 约束评估 — 可执行性验证

**原理说明：**

Gorilla 引入了一种创新的评估方法：不只看文本相似度，而是**解析生成的 API 调用是否可以实际执行**。

评估分为三个层级：

1. **语法正确性（Syntax Validity）**：生成的代码能否通过 Python AST 解析器
2. **语义正确性（Semantic Correctness）**：函数名、参数名是否匹配真实 API signature
3. **可执行性（Executable）**：在实际运行环境中能否成功执行并返回合理结果

这种"可执行即正确"的评估标准比传统的 BLEU/ROUGE 分数更有意义——因为一个 token 之差可能导致整个 API 调用失败，但 BLEU 可能仍然给出高分。

### 亮点三：Berkeley Function Calling Leaderboard（BFCL）

Gorilla 团队后续推出了 **BFCL**（目前已迭代至 V4），成为业界公认的 Function Calling 评测基准：

- 覆盖 **Simple / Multi-turn / Parallel / Concurrent** 等多种调用场景
- 支持 **REST API / SDK / SQL / 类自然语言**等多种工具类型
- 包含 **Java / JavaScript / Python / C#** 等多语言支持度评测
- 已有 50+ 个主流模型参与排名（GPT-4o、Claude 3.5、Gemini 等）

BFCL 的出现统一了此前各家用不同标准各自为战的混乱局面，推动了整个领域的标准化。

## 四、基准测试表现

### 4.1 APIBench 上的核心结果

| 模型 | APIBench 得分 | 幻觉率 |
|------|-------------|--------|
| **Gorilla (7B + Retriever)** | **85.0%** | **~5%** |
| GPT-4 (2023.03) | 75.0% | ~18% |
| Claude | 72.0% | ~20% |
| ChatGPT (GPT-3.5) | 65.0% | ~25% |
| LLaMA-7B (原始) | 38.0% | ~45% |

**关键发现：**
- 一个 7B 参数的微调模型在特定任务上超越了当时最强的闭源模型 GPT-4
- 加入 Retriever 后性能提升显著（+10~15%），验证了 RAG 方向的有效性
- 在 API 文档版本变更场景下，Gorilla 的适应能力远超纯参数化方法

### 4.2 BFCL V4 最新榜单趋势（2025）

根据 BFCL V4 排行榜，当前 Function Calling 领域呈现以下格局：
- **闭源领先组**：GPT-4o、Claude 3.5 Sonnet、Gemini 2.5 Pro 占据前列
- **开源追赶组**：Qwen2.5、DeepSeek、Llama 3.x 等快速缩小差距
- **专业优化组**：Gorilla OpenFunctions-v2 在特定场景仍有优势

## 六、对实际工作的启发

### 6.1 工具调用系统的设计原则

Gorilla 给我们的第一课是：**不要试图让模型"记住"所有工具，而是让它学会"查阅"工具文档。**

在实际构建 Agent 系统时，这意味着：
1. 维护一个结构化的工具注册表（tool registry），包含每个工具的名称、描述、参数 schema
2. 用户查询到来时，先做 tool retrieval（可用 embedding 相似度或 LLM-based routing）
3. 将检索到的工具定义注入 prompt，让模型基于此生成调用
4. 这套模式已被 OpenAI Function Calling、Anthropic Tool Use 等主流 API 采用

### 6.2 "可执行性优先"的评估思维

Gorilla 的 AST 约束评估启发我们：**对于工具调用任务，文本相似度指标具有欺骗性。**

在实际项目中应建立：
- 语法校验层（JSON schema validation / AST parse）
- 沙箱执行层（安全环境下试运行生成的调用）
- 结果校验层（返回值的类型和合理性检查）

### 6.3 RAT 思想的泛化应用

Retrieval Aware Training 的思想不仅适用于 API 调用，可推广到任何需要处理动态知识的场景：
- **RAG 系统**：训练时混入干扰文档提升鲁棒性
- **代码生成**：让模型学会参考文档写代码而非死记 API
- **多跳问答**：训练模型从多个检索结果中综合信息

### 6.4 局限性与未来方向

Gorilla 也存在明显局限：
- 主要聚焦 ML 框架 API，未覆盖通用 Web API / 企业内部 API
- 基于 LLaMA-7B 的基础能力上限较低（推理、指令遵循等弱于大模型）
- 未涉及多步工具链式调用（Tool Chaining）和多 Agent 协作

后续演进方向（已在 Gorilla OpenFunctions 和 GoEx 执行引擎中部分实现）：
- 支持更广泛的工具类型（REST API、数据库查询、GUI 操作等）
- 引入执行引擎提供安全性保障
- 从单次调用进化到多步骤工作流编排

---

## 七、总结

Gorilla 是 LLM 工具调用领域的**奠基性工作**。它的三大贡献——APIBench 数据集、Retrieval Aware Training 方法论、以及 BFCL 评测标准——深刻影响了后续几乎所有工具调用相关的研究和产品。虽然今天 GPT-4o/Claude 等闭源模型在通用 Function Calling 上已超越早期 Gorilla，但 Gorilla 所确立的 **"检索+生成+可执行验证"** 范式，依然是构建可靠 Agent 工具调用系统的最佳实践之一。

> **一句话总结**：Gorilla 证明了——在工具调用这件事上，**让模型学会查文档比让模型背文档更重要**。
